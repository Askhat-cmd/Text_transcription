## 🎯 Суть проекта

Telegram-бот психолог на основе LLM (GPT-5-mini / OpenAI API). Бот работает как **AI-сопровождающий** — не даёт советов напрямую, а помогает пользователю осознать своё состояние, предлагает практики (ACT, IFS, Polyvagal Theory, AQAL) и ведёт через трансформационный процесс.

**Целевая аудитория:** люди начиная с «зелёного мема» по спиральной динамике — осознанные, рефлексирующие, готовые к внутренней работе. Классификация пользователей по SD **не нужна** — вся база знаний доступна всем.

**Текущее состояние:** система работает, но перегружена техдолгом — противоречивые промты, тройной routing, SD-классификатор на каждый запрос, устаревшие фильтры. Задача — **рефакторинг под новую упрощённую архитектуру**.

## 🏗️ Существующая архитектура (что есть сейчас)

### Пайплайн обработки запроса

```
User Input
  → [StateClassifier] + [SDClassifier] (параллельно, оба LLM-вызовы)
  → [DecisionGate #1] pre_routing → Fast Path?
      ↓ Нет:
  → [HybridQueryBuilder] → [Retriever] (с SD-фильтром по уровню!)
  → [VoyageReranker]
  → [DecisionGate #2] pre_rerank_routing
  → [DecisionGate #3] routing_result (финальный)
  → block_cap → [UserLevelAdapter] (beginner/intermediate/advanced)
  → [ResponseGenerator] → LLM → Answer
```

## 🧠 Концептуальная архитектура бота (из документации Neo_mindbot.md)

### Приоритеты блоков (AA > A > Core)

```
{{{AA}}} — Safety Layer (абсолютные запреты, кризисные состояния)
{{{A}}}  — Seasonal Filter (сейчас: Spring Consciousness), Anti-Bland Rule, Emoji Rule
Core     — Identity, Methodology, Style, Formatting
```


### Диагностический алгоритм (3 состояния нервной системы)

```
HYPER (гипервозбуждение) → только телесный канал, замедление
WINDOW (окно толерантности) → полная работа, все каналы
HYPO (гиповозбуждение) → осторожно, контейнирование, активация
```


### Reflective Method (4 шага)

```
1. Recognition   — что происходит? (AI-зеркало)
2. Decentering   — отстранение от мысли (ACT)
3. Perspective   — переосмысление (IFS + AQAL)
4. Micro-action  — маленький шаг (CTA ≤10 слов)
```


### 6 функций запроса пользователя

```
1. Discharge/Vent    — выговориться, эмоциональная разрядка
2. Understand        — понять что происходит
3. Solution/Directive — "скажи что делать"
4. Validation        — подтверди что я прав
5. Explore/Depth     — исследовать глубже
6. Just Contact      — просто присутствие
```


### Структура памяти (JSON-снэпшот)

```json
{
  "nervous_system_state": "window_of_tolerance",
  "request_function": "understand_system",
  "dominant_part": "curious_explorer",
  "active_quadrant": "it_system",
  "core_theme": "technical_architecture_analysis",
  "last_state_snapshot": {...},
  "current_summary": "...",
  "active_track": null,
  "insights_log": [...]
}
```


### Библиотека практик (3 канала × триггеры)

- **Канал BODY** (Polyvagal/Somatic): 5-4-3-2-1, заземление, дыхание
- **Канал THINKING** (ACT/CBT): когнитивное разделение, факт vs интерпретация
- **Канал ACTION** (BJ Fogg Tiny Habits): поведенческий эксперимент, микрошаг


### Трек из 7 практик

Автоматически генерируется по `core_theme` + `nervous_system_state`. Прогрессия: body → thinking → action → интеграция. Поддерживает 3 режима запуска: автоматический (после инсайта), пользовательский (запросил сам), feedback-based (адаптация по откликам).

## 🔴 Диагностированные проблемы (из анализа кода)

### Проблема 1 — SD-классификатор лишний

`sd_classifier.classify_user()` запускается на **каждый запрос** параллельно с StateClassifier — двойной LLM-вызов, лишняя задержка, результат используется только для фильтрации БД.

### Проблема 2 — SD-фильтр в retrieval режет базу

```python
sd_level_int = SD_LEVELS_ORDER.index(sd_result.primary) + 1
retriever.retrieve(..., sd_level=sd_level_int)  # ← часть БД недоступна!
```

Пользователь с низким SD-confidence не получает часть материалов. Решение: `sd_level=0` или убрать параметр.

### Проблема 3 — Тройной DecisionGate

`pre_routing` → `pre_rerank_routing` → `routing_result` — три разных «мнения», LLM получает финальное от третьего, но блоки уже отфильтрованы первыми двумя. Источник противоречивых указаний.

### Проблема 4 — UserLevelAdapter лишний

```python
adapted_blocks = level_adapter.filter_blocks_by_level(blocks)
```

Третий слой фильтрации поверх SD. Наследие системы beginner/intermediate/advanced — не нужен при единой аудитории.

### Проблема 5 — 10 UserState vs 3 состояния нервной системы

В концепции (Neo_mindbot.md) — 3 состояния (HYPER/WINDOW/HYPO). В коде — 10 состояний UserState. Рассогласование приводит к противоречивым промтам.

## 🎯 Следующие шаги (план рефакторинга)

**Шаг 1 — Чистка лишних модулей**

- Отключить `sd_classifier.py` через feature flag (не удалять сразу — сначала тест)
- Убрать `sd_level` из `retriever.retrieve()`
- Убрать `user_level_adapter.filter_blocks_by_level()`

**Шаг 2 — Упрощение routing**

- Оставить один `DecisionGate.route()` — после retrieval+rerank
- Убрать `pre_routing` и `pre_rerank_routing`

**Шаг 3 — Переосмысление StateClassifier**

- Смаппировать 10 UserState на 3 состояния нервной системы (HYPER/WINDOW/HYPO) + функцию запроса (6 типов)
- Или: оставить 10 состояний, но переписать маппинг на режимы ответа под новую концепцию

**Шаг 4 — Новая система промтов**

- Один чистый системный промт вместо многослойных оверлеев
- Чёткая иерархия: Safety AA → Seasonal A → Core методология

**Шаг 5 — Тестирование**

- Прогнать 20-30 тестовых диалогов через новый пайплайн
- Проверить что retrieval отдаёт адекватные блоки без SD-фильтра
