# PRD-045.0 — Quality Baseline Audit Harness

**Project:** Bot Psychologist / Neo MindBot  
**Repository:** `Askhat-cmd/Text_transcription`  
**Target area:** `bot_psychologist`  
**Priority:** High  
**Risk level:** Low  
**Type:** evaluation / observability foundation  
**Runtime impact:** No intended production behavior change  
**Authoring context:** This PRD starts the quality-improvement roadmap after migration to multiagent-only runtime.

---

## 0. Executive Summary

The current project already has a working multiagent-only runtime:

```text
State Analyzer → Thread Manager → Memory Retrieval → Writer → Validator → Final Answer + Debug Payload
```

The next stage must not start with major refactors or “smarter agents”.  
The next stage must first create a **quality baseline**: a small, repeatable evaluation harness that lets us see when answers are too generic, too shallow, disconnected from the thread, repetitive, or poorly matched to the user state.

This PRD creates:

1. A small versioned evaluation dataset.
2. A baseline runner script.
3. A structured report format.
4. A minimal safe log/report capture workflow.
5. A `TO_DO_LIST` project folder for PRDs and implementation reports.
6. A commit/push discipline so future audits can inspect results from GitHub.

This PRD must not change the live response behavior of the bot.

---

## 1. Guiding Principles

### 1.1 Simplicity first

Do not build a complex evaluation platform yet.  
Create a simple, inspectable baseline harness that is easy to run, debug, and extend.

### 1.2 Modularity

The evaluation dataset, runner, report writer, and future quality flags must be separated.  
Do not hide evaluation logic inside the orchestrator.

### 1.3 Flexibility

The harness must support at least two modes:

- `dry-run` / structural mode: works without live LLM access where possible.
- `live` mode: runs real cases through the current runtime when API keys and services are available.

### 1.4 Trace-friendly design

The output report must help answer:

- What did the bot answer?
- Which state/thread/memory/debug fields were produced?
- Was the answer specific or generic?
- Did it preserve continuity?
- Did it repeat already closed ideas?
- Did the system use memory/RAG?
- Which model/settings were active?

### 1.5 Safety and privacy

Do not commit `.env`, API keys, local databases, raw private chat dumps, vector stores, or any secrets.  
If logs are captured, they must be sanitized or placed in a clearly safe snapshot.

## 2. Current Architecture Assumptions

The implementation must assume:

- Active runtime is `multiagent_adapter`.
- Active pipeline is `multiagent_v1`.
- Core agents:
  - State Analyzer
  - Thread Manager
  - Memory Retrieval
  - Writer
  - Validator
- `answer_adaptive.py` is compatibility shim only.
- Legacy cascade runtime is removed.
- Web trace/debug payload already exists and should be used as the main evidence source.
- Thread Manager is currently heuristic.
- Writer usually uses `gpt-5-mini`.
- State Analyzer usually uses `gpt-5-nano`.
- Production-hardening is not part of this PRD.

The script must support:

```bash

2. Run each case turn-by-turn.

```

### 5.4 Runner modes

#### 5.4.1 `dry-run`

Purpose: allow the command to run without external LLM/API dependencies.

Acceptable implementation options:

- Validate dataset schema only.
- Run deterministic parts if safe.
- Produce a report with `"mode": "dry-run"` and `"live_runtime_executed": false`.

Dry-run must not fail just because OpenAI credentials are missing.

#### 5.4.2 `live`

Purpose: run the current real runtime.

Preferred options, in order:

1. Directly call the multiagent orchestrator if project dependencies allow.
2. Otherwise call the existing adaptive API endpoint if backend is running.
3. If neither is available, fail with a clear message explaining how to run backend and required services.

Live mode must capture:

```json
{
  "case_id": "QB-010",
  "turn_index": 3,
  "user_message": "...",
  "answer": "...",
  "debug": {
    "pipeline_version": "...",
    "nervous_state": "...",
    "intent": "...",
    "safety_flag": false,
    "confidence": 0.81,
    "thread_id": "...",
    "phase": "...",
    "relation_to_thread": "...",
    "continuity_score": 0.74,
    "response_mode": "...",
    "context_turns": 6,
    "semantic_hits_count": 0,
    "model_used": "...",
    "model_temperature": 0.7,
    "model_max_tokens": 600,
    "validator_blocked": false,
    "validator_quality_flags": [],
    "timings": {}
  }
}
```

If a field is unavailable, set it to `null` rather than failing.

### 5.5 Basic heuristic quality checks

Do not add an LLM judge yet.

Add simple heuristic flags in the report only:

```json
{
  "heuristic_quality": {
    "too_short": false,
    "too_long": false,
    "too_many_questions": false,
    "contains_numbered_list": false,
    "generic_phrase_risk": true,
    "mentions_user_context": false,
    "has_practice_step_when_expected": true
  }
}
```

Suggested generic phrase markers:

- "я понимаю"
- "это нормально"
- "важно помнить"
- "попробуй разобраться"
- "дай себе время"
- "всё индивидуально"
- "тебе стоит"
- "просто"

These markers must not automatically mean failure.  
They are audit signals only.
