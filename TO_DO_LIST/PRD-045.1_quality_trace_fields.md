# PRD-045.1 — Расширение trace quality fields для multiagent runtime

**Проект:** Bot Psychologist / Neo MindBot  
**Репозиторий:** `Askhat-cmd/Text_transcription`  
**Рабочая зона:** `bot_psychologist`  
**Приоритет:** высокий  
**Риск:** низкий  
**Тип:** observability / trace / evaluation foundation  
**Влияние на поведение бота:** не должно менять финальные ответы пользователя  
**Предыдущий этап:** PRD-045.0 — Quality Baseline Audit Harness  
**Главный принцип:** простота, модульность, гибкость, быстрые настройки, понятный аудит через trace.

---

## 0. Краткое резюме

PRD-045.0 создал базовую инфраструктуру оценки качества: `TO_DO_LIST`, baseline dataset, dry-run/live runner, JSON/Markdown отчёты, implementation report и безопасные command/log snapshots.

Но live-baseline не был фактически выполнен, потому что backend на `localhost:8001` не был запущен. Это нормально для PRD-045.0: он был фундаментальным.

Следующий шаг — не улучшать ответы и не менять логику агентов, а расширить `debug/trace` так, чтобы будущий live-прогон мог показать **почему ответ получился сильным или слабым**.

Этот PRD добавляет в multiagent trace компактный блок:

```json
{
  "quality_trace_version": "quality_trace_v1",
  "quality_trace": {
    "answer": {},
    "state": {},
    "thread": {},
    "memory": {},
    "continuity": {},
    "response_mode": {},
    "validator": {},
    "summary_flags": []
  }
}
```

Важно: `quality_trace` — это не LLM-оценщик и не новая логика ответа. Это только deterministic/heuristic audit-сигналы.

---

## 1. Почему это нужно

Сейчас у проекта уже есть rich debug payload, но он больше показывает техническую картину: модель, prompt, response, tokens, timings, state fields, thread fields, memory hits, validator flags.

Для развития качества нужен слой, который быстро отвечает на вопросы:

- ответ слишком короткий или слишком длинный?
- в ответе слишком много вопросов?
- есть риск generic-фраз?
- ответ связан с текущей нитью или звучит как новый старт?
- использована ли память?
- есть ли semantic hits?
- были ли open loops / closed loops?
- есть ли риск повторения закрытой петли?
- соответствует ли ответ `response_mode`?
- есть ли практический шаг, если mode = `practice`?

На этом PRD мы не делаем “идеальную оценку”. Мы создаём первый простой audit-блок, который потом будет виден в trace, baseline report и будущей Web Trace системе.

## 2. Принципы реализации

### 2.1 Не менять поведение бота

Финальный ответ пользователя не должен измениться из-за этого PRD.

Запрещено:

- менять Writer prompt;
- менять State Analyzer prompt;
- менять Thread Manager decision logic;
- менять Memory Retrieval ranking;
- блокировать ответы по новым quality-сигналам;
- добавлять LLM judge;
- добавлять Response Planner.

### 2.2 Только deterministic signals

Все новые поля считаются кодом:

- длина ответа;
- количество вопросов;
- наличие numbered list;
- generic phrase markers;
- наличие practice step markers;
- количество open/closed loops;
- наличие semantic hits;
- наличие user profile;
- confidence bucket;
- continuity bucket.

### 2.3 Не утекать приватный текст

`quality_trace` не должен добавлять новые raw user messages, full answers, full prompts, full memories или raw semantic chunks.

Разрешены:

- числа;
- boolean;
- enum;
- короткие технические флаги;
- счётчики.

Запрещены внутри `quality_trace`:

- полный текст пользователя;
- полный ответ бота;
- полный conversation context;
- полные semantic hits;
- секреты;
- API keys;
- session tokens.

Ответ и prompt уже могут существовать в других debug-полях, но `quality_trace` должен быть безопасным compact audit layer.

### 2.4 Модульность

Не расписывать всё в `orchestrator.py`.

Создать отдельный модуль:

```text
bot_psychologist/bot_agent/multiagent/quality_trace.py
```

## 5. Требуемые новые поля trace

В `debug` ответа orchestrator должны появиться:

```json
{
  "quality_trace_version": "quality_trace_v1",
  "quality_trace": {}
}
```

### 5.1 `quality_trace.answer`

```json
{
  "chars": 312,
  "sentences": 4,
  "questions": 1,
  "too_short": false,
  "too_long": false,
  "too_many_questions": false,
  "contains_numbered_list": false,
  "generic_phrase_risk": true,
  "generic_markers_count": 1
}
```

Рекомендованные thresholds:

```python
TOO_SHORT_CHARS = 40
TOO_LONG_CHARS = 1400
TOO_MANY_QUESTIONS = 3
```

Generic phrase markers:

```python
[
  "я понимаю",
  "это нормально",
  "важно помнить",
  "дай себе время",
  "всё индивидуально",
  "тебе стоит",
  "просто"
]
```

Важно: generic marker — это не ошибка, а сигнал для аудита.

### 5.2 `quality_trace.state`

```json
{
  "nervous_state": "window",
  "intent": "explore",
  "openness": "open",
  "ok_position": "I+W+",
  "safety_flag": false,
  "confidence": 0.81,
  "confidence_bucket": "high"
}
```

Buckets:

```text
low: confidence < 0.6
medium: 0.6 <= confidence < 0.8
high: confidence >= 0.8
unknown: confidence is None
```

### 5.3 `quality_trace.thread`

```json
{
  "thread_id_present": true,
  "phase": "explore",
  "relation_to_thread": "continue",
  "continuity_score": 0.74,
  "continuity_bucket": "high",
  "open_loops_count": 1,
  "closed_loops_count": 0,
  "has_open_loops": true,
  "has_closed_loops": false,
  "turns_in_phase": 2
}
```

Continuity buckets:

```text
low: score < 0.25
medium: 0.25 <= score < 0.65
high: score >= 0.65
unknown: score is None
```

### 5.4 `quality_trace.memory`

```json
{
  "context_turns": 6,
  "semantic_hits_count": 2,
  "has_relevant_knowledge": true,
  "rag_query_present": true,
  "conversation_context_present": true,
  "user_profile_non_empty": true,
  "retrieval_status": "has_hits"
}
```

`retrieval_status`:

```text
has_hits
no_hits
no_context
unknown
```

Пример логики:

- `has_hits`: `semantic_hits_count > 0`
- `no_hits`: `semantic_hits_count == 0`, но есть `rag_query_present`
- `no_context`: нет context и нет hits
- `unknown`: невозможно определить

### 5.5 `quality_trace.continuity`

```json
{
  "answer_mentions_recent_context": true,
  "answer_mentions_open_loop": false,
  "possible_closed_loop_repetition": false,
  "context_overlap_terms_count": 4,
  "open_loop_overlap_terms_count": 0,
  "closed_loop_overlap_terms_count": 0
}
```

Важно:

- не записывать сами overlap terms;
- только counts/booleans;
- simple token overlap достаточно;
- русский/английский unicode tokenization.

### 5.6 `quality_trace.response_mode`

```json
{
  "mode": "practice",
  "goal_present": true,
  "practice_step_present": true,
  "mode_risk_flags": []
}
```

Примеры `mode_risk_flags`:

```text
practice_without_step
regulate_too_long
validate_too_many_questions
safe_override_too_long
explore_no_question
```

Эти флаги должны быть простыми и не блокировать ответ.

### 5.7 `quality_trace.validator`

```json
{
  "blocked": false,
  "block_reason_present": false,
  "quality_flags_count": 1,
  "quality_flags": ["too_long: 1500 chars"]
}
```

### 5.8 `quality_trace.summary_flags`

```json
[
  "generic_phrase_risk",
  "practice_without_step"
]
```

`summary_flags` — компактный список ключевых сигналов, чтобы Web Trace мог быстро подсвечивать turn.

## 6. Рекомендуемая архитектура модуля

Создать:

```text
bot_psychologist/bot_agent/multiagent/quality_trace.py
```

Рекомендуемый публичный API:

```python
def build_quality_trace(
    *,
    final_answer: str,
    writer_contract: WriterContract,
    validation_result: ValidationResult,
) -> dict:
    ...
```

Рекомендуемые helper-функции:

```python
_tokenize(text: str) -> set[str]
_count_sentences(text: str) -> int
_count_questions(text: str) -> int
_contains_numbered_list(text: str) -> bool
_confidence_bucket(value: float | None) -> str
_continuity_bucket(value: float | None) -> str
_build_answer_quality(...)
_build_state_quality(...)
_build_thread_quality(...)
_build_memory_quality(...)
_build_continuity_quality(...)
_build_response_mode_quality(...)
_build_validator_quality(...)
```

Все helper-функции должны быть deterministic и легко тестируемыми.

## 7. Интеграция в orchestrator

Файл:

```text
bot_psychologist/bot_agent/multiagent/orchestrator.py
```

После получения `final_answer` и `validation_result`, но перед `return`, собрать:

```python
quality_trace = build_quality_trace(
    final_answer=final_answer,
    writer_contract=writer_contract,
    validation_result=validation_result,
)
```

В debug добавить:

```python
"quality_trace_version": "quality_trace_v1",
"quality_trace": quality_trace,
```

### 7.1 Защитный fallback

Ошибка в `quality_trace` не должна ломать ответ пользователю.

Рекомендуемый fallback:

```python
try:
    quality_trace = build_quality_trace(...)
    quality_trace_error = None
except Exception as exc:
    quality_trace = {
        "version": "quality_trace_v1",
        "error": "quality_trace_failed"
    }
    quality_trace_error = str(exc)
```

В debug:

```python
"quality_trace_error": quality_trace_error,
```

Не логировать sensitive text в error.

## 10. Тесты

### 10.1 Новый unit test для quality trace

Создать:

```text
bot_psychologist/tests/multiagent/test_quality_trace.py
```

Покрыть:

1. `build_quality_trace` возвращает dict.
2. Есть все верхнеуровневые ключи:
   - `version`
   - `answer`
   - `state`
   - `thread`
   - `memory`
   - `continuity`
   - `response_mode`
   - `validator`
   - `summary_flags`
3. `too_many_questions=True`, если вопросов больше 3.
4. `contains_numbered_list=True`, если ответ содержит numbered list.
5. `generic_phrase_risk=True`, если есть marker.
6. `confidence_bucket` считается верно.
7. `continuity_bucket` считается верно.
8. `practice_without_step` появляется для `response_mode=practice`, если нет marker практического шага.
9. Нет raw user text в `quality_trace`.

### 10.2 Обновить orchestrator e2e test

Файл:

```text
bot_psychologist/tests/multiagent/test_orchestrator_e2e.py
```

Добавить проверку:

```python
assert "quality_trace_version" in result["debug"]
assert result["debug"]["quality_trace_version"] == "quality_trace_v1"
assert "quality_trace" in result["debug"]
assert isinstance(result["debug"]["quality_trace"], dict)
```

Важно: существующие тесты не должны ломаться.

### 10.3 Проверить evaluation tests

```bash
python -m pytest bot_psychologist/tests/evaluation -q
```

### 10.4 Проверить baseline runner dry-run

```bash
python bot_psychologist/scripts/run_quality_baseline.py --mode dry-run
```

Dry-run не обязан иметь real `quality_trace`, но runner не должен падать.

### 10.5 Live baseline attempt

Попробовать:

Если backend/API недоступны — не считать PRD проваленным, но обязательно документировать причину.
