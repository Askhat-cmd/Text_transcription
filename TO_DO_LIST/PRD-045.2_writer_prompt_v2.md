# PRD-045.2 — Writer Prompt v2: конкретность, continuity и анти-общие ответы

**Проект:** Bot Psychologist / Neo MindBot  
**Репозиторий:** `Askhat-cmd/Text_transcription`  
**Рабочая зона:** `bot_psychologist`  
**Приоритет:** высокий  
**Риск:** низкий / средний  
**Тип:** prompt quality / answer quality improvement  
**Влияние на архитектуру:** не менять архитектуру  
**Влияние на ответы бота:** да, ожидается улучшение качества формулировок  
**Предыдущий этап:** PRD-045.1.1 — Доставка `quality_trace` до API, debug endpoints и baseline report  
**Главный принцип:** простота, модульность, гибкость, быстрые настройки, понятный trace-аудит.

---

## 0. Краткое резюме

После PRD-045.0–045.1.1 у проекта появилась минимальная измеримость качества:

- baseline dataset;
- baseline runner;
- live baseline report;
- `quality_trace`;
- доставка `quality_trace` до API/debug/baseline.

Теперь можно сделать первый аккуратный шаг в улучшении качества ответа.

Этот PRD улучшает только Writer prompt.

Цель: уменьшить поверхностные, слишком общие и слишком длинные ответы, улучшить continuity и точность следующего хода.

Запрещено менять:

- multiagent architecture;
- State Analyzer;
- Thread Manager;
- Memory Retrieval;
- Validator logic;
- Response Planner;
- модельные настройки;
- Web UI;
- Web Admin.

---

## 1. Почему этот PRD нужен

Live baseline после PRD-045.1.1 показал, что технически trace уже работает, но первые кейсы всё ещё дают признаки слабости:

- QB-001 нормальный короткий контакт.
- QB-002: пользователь просит «пару спокойных слов», а бот даёт довольно длинное отражение и вопрос выбора.
- QB-003: пользователь говорит «всё как-то не так», а бот делает достаточно сильную интерпретацию про «не соответствуешь ожиданиям» и «не на своём месте».
- `Quality trace summary` уже фиксирует `generic_phrase_risk` в части ответов.
- Heuristic flags также показывают generic risk.

Значит, следующий шаг — не усложнять систему, а уточнить Writer Prompt:

1. Ответ должен быть конкретным относительно сообщения пользователя.
2. Ответ не должен добавлять лишнюю психологическую интерпретацию.
3. Ответ должен уважать ресурс пользователя.
4. Ответ должен делать один следующий микро-шаг, если режим это подразумевает.
5. Ответ должен не повторять закрытые петли.
6. Ответ должен лучше исполнять `response_mode`.

---

## 2. Главная цель

Сделать Writer более точным и менее общим за счёт prompt v2.

Ожидаемый эффект:

- меньше длинных ответов в `validate` и `regulate`;
- меньше generic phrase risk;
- меньше лишних интерпретаций;
- лучшее соответствие `response_mode`;
- лучшее удержание continuity;
- больше «один точный следующий ход», а не лекция/теория.

## 5. Что нужно изменить в Writer prompt

### 5.1 Добавить общий принцип «один ответ = один точный ход»

В `WRITER_SYSTEM` добавить правило:

```text
Один ответ = один точный следующий ход.
Не пытайся за один ответ:
- объяснить всю ситуацию;
- открыть новую теорию;
- дать несколько направлений;
- задавать несколько вопросов;
- исправить всё сразу.
```

### 5.2 Добавить анти-общие правила

Добавить блок:

```text
АНТИ-ОБЩИЕ ОТВЕТЫ:
- Не отвечай универсальными фразами, если можно опереться на конкретные слова пользователя.
- Не пиши «это нормально», «важно помнить», «дай себе время», если это не добавляет смысла.
- Не пересказывай очевидное длинно.
- Не используй психологические клише.
- Не делай выводы сильнее, чем позволяет сообщение пользователя.
- Не добавляй диагнозы, скрытые мотивы или глубокие интерпретации без явной опоры в тексте.
```

Важно: эти фразы не должны быть абсолютным ban, но prompt должен снижать их использование.

### 5.3 Добавить правило точной опоры на сообщение пользователя

Добавить:

```text
Сначала найди в сообщении пользователя 1 главный смысловой фокус.
Ответ должен быть вокруг него.
Если сообщение короткое или неопределённое — не достраивай за пользователя. Лучше мягко уточни.
```

### 5.4 Добавить правило resource-fit

Добавить:

```text
Если пользователь устал, перегружен, просит коротко или «без советов»:
- сокращай ответ;
- не задавай сложных вопросов;
- не давай теорию;
- не превращай поддержку в анализ.
```

### 5.5 Уточнить response modes

Текущие режимы уже есть. Нужно сделать их более конкретными.

#### validate

Текущий риск: может быть слишком длинным или превращаться в reflect.

Обновить:

```text
validate — дай пространство и признание
- 2–4 коротких предложения.
- Не анализируй.
- Не объясняй причину состояния.
- Не задавай вопрос, если пользователь явно просит просто поддержку.
- Лучше простое присутствие, чем глубокая формулировка.
```

#### reflect

Текущий риск: может делать слишком сильную интерпретацию.

Обновить:

```text
reflect — помоги человеку сформулировать суть происходящего
- Отрази только то, что реально есть в сообщении.
- Добавь лёгкую глубину, но не делай сильных выводов.
- Один уточняющий вопрос в конце.
- Не превращай ответ в диагноз или объяснение личности.
```

#### explore

```text
explore — расширь перспективу
- Один новый угол зрения как гипотеза, не как истина.
- Используй слова: «возможно», «может быть», «похоже».
- Заверши одним открытым вопросом.
```

#### regulate

```text
regulate — помоги стабилизировать состояние
- Очень коротко.
- Одно простое телесное/ориентировочное действие.
- Без анализа.
- Без длинных объяснений.
```

#### practice

```text
practice — дай один конкретный шаг
- Один микрошаг.
- Указать: что сделать, когда, сколько времени.
- Без списка.
- Без мотивационной лекции.
```

#### safe_override

```text
safe_override — кризисный режим
- Коротко и прямо.
- Приоритет немедленной безопасности.
- Проси обратиться к живому человеку/экстренной помощи, если есть риск вреда себе.
- Не задавай исследовательских вопросов.
```

### 5.6 Добавить правило open_loops / must_avoid

Добавить:

```text
Если есть open_loops:
- выбери максимум одну открытую петлю;
- не пытайся закрыть все сразу;
- если текущий запрос важнее open_loop, отвечай на текущий запрос.

Если есть must_avoid:
- не повторяй эти темы даже другими словами;
- не возвращай уже закрытую петлю как новый инсайт.
```

### 5.7 Добавить правило длины

Добавить ориентиры:

```text
Ориентиры длины:
- validate: 2–4 коротких предложения.
- regulate: 2–4 коротких предложения.
- reflect: 4–7 предложений.
- explore: 4–7 предложений.
- practice: 3–6 предложений.
- safe_override: максимально коротко, но с безопасностью.
```

Не добавлять жёсткий кодовый лимит. Только prompt.

### 5.8 Добавить правило вопросов

```text
Не задавай больше одного вопроса.
Если пользователь просит поддержку без вопросов — не задавай вопрос.
Если пользователь просит конкретный шаг — дай шаг, а не вопрос.
```

## 6. Требования к Writer User Template

Файл:

```text
writer_agent_prompts.py
```

`WRITER_USER_TEMPLATE` сейчас содержит:

- user_message;
- response_mode;
- response_goal;
- phase;
- nervous_state;
- ok_position;
- openness;
- safety_active;
- open_loops;
- must_avoid;
- conversation_context;
- user_profile;
- semantic_hits.

Не менять структуру радикально.

Но можно добавить короткий блок в конец перед “Напиши ответ”:

```text
ПЕРЕД ОТВЕТОМ ВНУТРИ СЕБЯ ВЫБЕРИ:
- главный фокус пользователя;
- нужную глубину;
- один следующий ход.

В ответе не показывай этот анализ.
```

Важно: не просить модель выводить скрытый анализ. Только использовать как внутреннюю инструкцию.
