# PRD-045.3.1 — Точечная калибровка State Analyzer и response mode routing v1

**Проект:** Bot Psychologist / Neo MindBot  
**Репозиторий:** `Askhat-cmd/Text_transcription`  
**Рабочая зона:** `bot_psychologist`  
**Приоритет:** высокий  
**Риск:** средний  
**Тип:** runtime calibration / routing fix / quality improvement  
**Влияние на ответы бота:** да, ожидается изменение `response_mode` и, как следствие, качества ответов  
**Предыдущий этап:** PRD-045.3 — State Analyzer Calibration Dataset + Response Mode Routing Audit  
**Главный принцип:** простота, модульность, гибкость, быстрые настройки, понятный trace-аудит.

---

## 0. Краткое резюме

PRD-045.3 создал calibration harness и показал важную проблему:

```text
nervous_state_accuracy: 0.8889
intent_accuracy: 0.2639
openness_accuracy: 0.4444
ok_position_accuracy: 0.5556
response_mode_accuracy: 0.5694
all_fields_exact_match: 0.125
high_severity_mismatches: 40
```

Главные кластеры ошибок:

```text
intent: contact -> explore (23)
response_mode_new_thread: validate -> reflect (15)
intent: clarify -> explore (12)
intent: solution -> explore (10)
response_mode_new_thread: practice -> reflect (10)
nervous_state: hypo -> window (8)
openness: defensive -> open (8)
openness: collapsed -> open (8)
```

Самые вредные ошибки для качества ответа:

1. Пользователь просит короткий контакт/поддержку, а система маршрутизирует в `reflect`.
2. Пользователь просит один конкретный шаг, а система маршрутизирует в `reflect`, потому что новый thread всегда стартует с phase=`clarify`.
3. Низкий ресурс (`hypo`) часто классифицируется как `window`.
4. Defensive/collapsed часто сглаживается до `open`.

Этот PRD делает **только точечную калибровку v1**:

- расширяет deterministic hints State Analyzer;
- исправляет routing в Thread Manager для `solution -> practice`;
- добавляет targeted regression tests;
- запускает calibration before/after;
- не трогает Writer prompt, Web UI, Admin, Memory Retrieval, Validator.

---

## 1. Почему это нужно

PRD-045.2 улучшил Writer prompt, но baseline показал частичное улучшение. Причина стала яснее после PRD-045.3: Writer часто получает неверный `response_mode`.

Пример вредного маршрута:

```text
User: "Я очень устал. Не хочу советов, просто скажи пару спокойных слов."

Expected:
intent=contact
nervous_state=hypo или window
response_mode=validate / regulate

Actual:
intent=explore
response_mode=reflect
```

В таком случае Writer даже с хорошим prompt будет делать reflect, потому что так велит upstream pipeline.

---

## 2. Главная цель

Улучшить самые частые и вредные ошибки Analyzer/Routing без большой перестройки архитектуры.

Целевые улучшения после PRD:

- `contact -> explore` mismatch должен заметно снизиться.
- `validate -> reflect` mismatch должен заметно снизиться.
- `solution -> explore` mismatch должен заметно снизиться.
- `practice -> reflect` mismatch должен заметно снизиться.
- Low-resource/contact cases больше не должны массово уходить в reflect/explore.
- Solution/practice cases должны маршрутизироваться в `practice` на новом thread.

## 5. Важные ограничения

### 5.1 Не подгонять датасет

Calibration dataset из PRD-045.3 может быть улучшен позже, но в этом PRD нельзя просто менять expected labels, чтобы метрики стали лучше.

Разрешено:

- исправить явные опечатки;
- добавить отдельные targeted regression tests;
- добавить небольшое число реалистичных cases, если это не ломает coverage;
- документировать искусственность части dataset.

Запрещено:

- массово переписывать expected labels;
- удалять сложные cases;
- делать метрики красивыми за счёт dataset manipulation.

### 5.2 Не лечить все поля сразу

В этом PRD не нужно полностью исправлять `openness` и `ok_position`.

Приоритет:

1. `intent`
2. `response_mode_new_thread`
3. `nervous_state` для low-resource
4. `openness` только для defensive/collapsed явных фраз

## 11. Task 6 — Исправить Thread Manager response mode routing

Файл:

```text
bot_psychologist/bot_agent/multiagent/agents/thread_manager.py
```

Сейчас `_mode_and_goal(phase, state_snapshot)` даёт `practice` только если:

```python
phase == "explore" and state_snapshot.intent == "solution"
```

Но новый thread стартует с phase=`clarify`, поэтому `solution` на новом thread часто превращается в `reflect`.

Нужно изменить routing так:

```python
def _mode_and_goal(phase: str, state_snapshot: StateSnapshot) -> tuple[str, str]:
    if state_snapshot.safety_flag:
        return "safe_override", "stabilize and reduce overload"

    if state_snapshot.intent == "contact":
        return "validate", "hold contact and avoid overload"

    if state_snapshot.nervous_state in {"hyper", "hypo"}:
        return "regulate", "softly stabilize state"

    if state_snapshot.intent == "solution":
        return "practice", "provide one realistic next step"

    if phase == "explore":
        return "explore", "expand perspective"

    if phase == "clarify":
        return "reflect", "clarify key point"

    return "validate", "keep safe contact"
```

Обоснование:

- safety всегда выше всего;
- explicit contact выше nervous_state, потому что “устал + без анализа + пару слов” должен быть validate, а не длинный regulate;
- nervous hyper/hypo без contact должен stabilize/regulate;
- solution должен давать practice даже на новом thread;
- clarify остаётся reflect.
