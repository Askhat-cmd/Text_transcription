# PRD-045.3.2 — Runtime Freshness Guard и direct baseline smoke

**Проект:** Bot Psychologist / Neo MindBot  
**Репозиторий:** `Askhat-cmd/Text_transcription`  
**Рабочая зона:** `bot_psychologist`  
**Приоритет:** высокий  
**Риск:** низкий  
**Тип:** evaluation reliability / diagnostics / baseline runner hardening  
**Влияние на ответы бота:** не должно менять финальные ответы пользователя  
**Предыдущий этап:** PRD-045.3.1 — Точечная калибровка State Analyzer и response mode routing v1  
**Следующий этап после аудита:** PRD-045.3.3 или PRD-045.4, в зависимости от подтверждённых smoke-результатов  
**Главный принцип:** простота, модульность, гибкость, быстрые настройки, понятный trace-аудит.

---

## 0. Краткое резюме

PRD-045.3.1 дал сильное улучшение в calibration harness:

```text
intent_accuracy: 0.25 -> 0.75
response_mode_accuracy: 0.5556 -> 0.8889
high_severity_mismatches: 42 -> 10
```

Но quality baseline smoke по `QB-002` всё ещё показал:

```text
response_mode: reflect
answer length: 369 -> 393
```

При этом baseline report был с `Git SHA: c93d61c`, то есть с предыдущего HEAD до коммита PRD-045.3.1. Это может означать одно из двух:

1. baseline запускался через уже старый backend-процесс, который не был перезапущен после изменений;
2. изменения реально не исправили `QB-002`, потому что этот кейс идёт по другому пути.

Сейчас нельзя уверенно делать новый behavioral fix, пока не будет устранена диагностическая неопределённость.

Этот PRD не меняет поведение бота.  
Он делает baseline smoke более надёжным:

- добавляет direct mode, который вызывает текущий local orchestrator без HTTP backend;
- добавляет runtime freshness metadata;
- добавляет targeted smoke report по `QB-002`, `QB-003`, `QB-005`, `QB-010`;
- добавляет явное сравнение API mode vs direct mode;
- защищает нас от ситуации “тестируем старый запущенный сервер”.

---

## 1. Почему это нужно

Сейчас `run_quality_baseline.py --mode live` идёт через HTTP API. Это хорошо для end-to-end проверки, но плохо для проверки свежих code changes, если backend уже был запущен раньше.

В PRD-045.3.1 были изменены `state_analyzer.py` и `thread_manager.py`, но baseline smoke report всё ещё показал старое поведение `QB-002`.

Нужно ответить на вопрос:

```text
QB-002 остаётся reflect потому что:
A) код не помогает,
или
B) baseline бился в старый backend?
```

Без этого следующий PRD может исправлять несуществующую проблему или дублировать уже сделанную работу.

---

## 2. Цель PRD

Создать надёжный targeted smoke harness, который показывает:

1. что текущий код действительно используется;
2. какой путь прошёл каждый кейс:
   - State Analyzer output;
   - Thread Manager output;
   - response_mode;
   - writer model;
   - quality_trace;
3. совпадает ли поведение direct mode и API mode;
4. стал ли `QB-002` реально `validate/regulate` на свежем runtime.

## 7. Task 3 — Добавить direct mode

Сейчас baseline runner поддерживает:

```text
--mode dry-run
--mode live
```

Добавить:

```text
--mode direct
```

Пример запуска:

```bash
python bot_psychologist/scripts/run_quality_baseline.py --mode direct --limit 3
python bot_psychologist/scripts/run_quality_baseline.py --mode direct --case-id QB-002
```

### 7.1 Что делает direct mode

Direct mode должен:

1. Загружать dataset.
2. Для каждого case выполнять turns через текущий Python-код.
3. Вызывать `bot_agent.multiagent.orchestrator.orchestrator.run(...)` напрямую.
4. Не использовать HTTP API.
5. Сохранять тот же формат report, что live mode.
6. Обязательно сохранять `debug` и `quality_trace`.

Псевдокод:

```python
from bot_agent.multiagent.orchestrator import orchestrator

async def _run_case_direct(case):
    user_id = f"baseline_direct_{case_id}"
    for turn in case["user_turns"]:
        result = await orchestrator.run(query=turn, user_id=user_id)
    return result
```

Важно:

- direct mode может использовать real OpenAI API внутри агентов, если ключ доступен;
- если ключа нет, должен быть понятный error;
- не подменять ответы fake-ответами;
- не писать raw private logs;
- использовать synthetic dataset.

### 7.2 Session isolation

Каждый case должен получать уникальный `user_id`, например:

```text
baseline_direct_QB-002_<timestamp>
```

Чтобы thread storage/memory не смешивали разные кейсы.
