# PRD-045.3 — State Analyzer Calibration Dataset + Response Mode Routing Audit

**Проект:** Bot Psychologist / Neo MindBot  
**Репозиторий:** `Askhat-cmd/Text_transcription`  
**Рабочая зона:** `bot_psychologist`  
**Приоритет:** высокий  
**Риск:** низкий  
**Тип:** evaluation / calibration / routing diagnostics  
**Влияние на ответы бота:** не должно менять финальные ответы пользователя  
**Предыдущий этап:** PRD-045.2 — Writer Prompt v2  
**Следующий этап после аудита:** PRD-045.4 или PRD-045.3.1 — точечная калибровка State Analyzer / response mode routing  
**Главный принцип:** простота, модульность, гибкость, быстрые настройки, понятный trace-аудит.

---

## 0. Краткое резюме

PRD-045.2 улучшил Writer prompt, но baseline показал только частичное улучшение:

- `quality_trace` виден;
- prompt guardrails добавлены;
- тесты прошли;
- но `generic-risk flags` на live `--limit 3` не снизились;
- `QB-002` всё ещё слишком длинный для low-resource поддержки;
- `QB-003` всё ещё может уходить в чрезмерную интерпретацию.

Главная гипотеза: проблема не только в Writer prompt.

Writer исполняет `response_mode`, который приходит из связки:

```text
State Analyzer → Thread Manager → response_mode → Writer
```

Если пользователь просит «пару спокойных слов», но State Analyzer / Thread Manager дают `response_mode=reflect`, Writer будет отражать и углублять, даже если prompt просит быть аккуратнее.

Этот PRD ничего не меняет в runtime-логике.  
Он создаёт calibration dataset и runner, чтобы измерить:

1. как State Analyzer определяет `nervous_state`;
2. как он определяет `intent`;
3. как он определяет `openness`;
4. как он определяет `ok_position`;
5. какой `response_mode` получается на выходе Thread Manager для нового сообщения;
6. где именно возникают ошибки: Analyzer или mode routing.

---

## 1. Зачем это нужно

Сейчас у нас есть baseline качества финального ответа, но нет отдельной диагностики State Analyzer.

Это опасно, потому что можно бесконечно улучшать Writer prompt, хотя реальная причина плохого ответа может быть раньше:

```text
Пользователь: «я очень устал, скажи пару спокойных слов»
State Analyzer: intent=clarify, nervous_state=window
Thread Manager: response_mode=reflect
Writer: делает длинное reflect
```

Правильная диагностика должна показать:

```text
Ожидалось:
nervous_state=hypo или window
intent=contact
openness=mixed/open
response_mode=validate или regulate

Получилось:
intent=clarify
response_mode=reflect
```

И тогда следующий PRD сможет безопасно исправлять не Writer, а классификацию/маршрутизацию.

## 4. Связь с референс-диалогом Wake_up

Пользователь приложил диалог с другим ботом как референс удержания линии разговора.

Из этого референса полезны не конкретные формулировки, а принципы:

- бот держит одну смысловую линию через много ходов;
- возвращает пользователя к уже найденному паттерну;
- использует предыдущие ответы пользователя как материал следующего хода;
- не начинает каждый ответ заново;
- задаёт один сильный вопрос;
- предлагает один конкретный шаг;
- завершает диалог мягкой фиксацией сдвига.

Но этот стиль не нужно копировать напрямую.  
В нём есть риск чрезмерной драматизации, сильных интерпретаций и давления на пользователя.  
Для Bot Psychologist эти идеи нужно использовать аккуратно: как материал для future continuity/pattern-core PRD, а не как немедленное изменение Writer.

В рамках PRD-045.3 можно добавить несколько synthetic calibration cases, вдохновлённых такими ситуациями:

- страх оценки проекта;
- вечная доработка вместо выхода к людям;
- страх быть невостребованным;
- выбор действия вместо поиска уверенности;
- пользователь просит не анализ, а короткое подтверждение.

Не копировать диалог дословно в датасет.

Всего: 60–80 cases.

1. `contact` — 6 cases
2. `low_resource_support` — 8 cases
3. `clarify` — 8 cases
4. `solution` — 8 cases
5. `vent` — 6 cases
6. `explore` — 6 cases
7. `defensive` — 6 cases
8. `collapsed` — 5 cases
9. `hyper` — 5 cases
10. `hypo` — 5 cases
11. `safety` — 4 cases
12. `ambiguous_short` — 4 cases
13. `continuity_followup` — 4 cases
14. `project_fear_evaluation` — 4 cases
15. `practice_request` — 4 cases

Dataset must include at least:

- 10 cases where expected intent is `contact`;
- 10 cases where expected intent is `clarify`;
- 10 cases where expected intent is `solution`;
- 8 cases where expected nervous_state is `hypo`;
- 8 cases where expected nervous_state is `hyper`;
- 6 defensive cases;
- 5 collapsed cases;
- 4 safety cases;
- 8 cases where expected response mode is `validate`;
- 8 cases where expected response mode is `regulate`;
- 8 cases where expected response mode is `practice`;
- 8 cases where expected response mode is `reflect`;
- 4 cases where confidence should be intentionally low/medium due ambiguity.

Use synthetic messages only.  
Do not use raw private user conversations.

## 9. Calibration runner

Создать:

```text
bot_psychologist/scripts/run_state_analyzer_calibration.py
```

CLI:

```bash
python bot_psychologist/scripts/run_state_analyzer_calibration.py --mode dry-run
python bot_psychologist/scripts/run_state_analyzer_calibration.py --mode live
python bot_psychologist/scripts/run_state_analyzer_calibration.py --limit 10
python bot_psychologist/scripts/run_state_analyzer_calibration.py --case-id SA-LOW-001
python bot_psychologist/scripts/run_state_analyzer_calibration.py --output TO_DO_LIST/reports/PRD-045.3_STATE_ANALYZER_CALIBRATION_REPORT.json
python bot_psychologist/scripts/run_state_analyzer_calibration.py --markdown-output TO_DO_LIST/reports/PRD-045.3_STATE_ANALYZER_CALIBRATION_REPORT.md
```

### 9.1 `dry-run`

`dry-run` должен:

- загрузить dataset;
- проверить schema;
- посчитать coverage;
- создать JSON/Markdown report;
- не требовать OpenAI API key.

### 9.2 `live`

`live` должен:

- импортировать `state_analyzer_agent`;
- вызвать `analyze(...)` по каждому case;
- собрать `StateSnapshot`;
- для `response_mode_new_thread` получить ThreadState через `thread_manager_agent.update(...)` с:
  - `current_thread=None`;
  - `archived_threads=[]`;
  - уникальный `user_id` на case;
- сравнить actual vs expected;
- записать mismatches.

Важно: не запускать весь bot/orchestrator.  
Нужен быстрый изолированный analyzer calibration.

## 12. Severity rules

Простая логика:

`high` если:

- expected `safety` / `safe_override`, но actual safety не сработал;
- expected `contact|low_resource_support`, а actual response_mode=`reflect|explore`;
- expected `hypo|collapsed`, а actual `window` + response_mode=`explore`;
- expected `solution/practice`, а actual response_mode не `practice`;
- expected defensive/collapsed, а actual openness=`open` с confidence >= 0.75.

`medium` если:

- один важный field не совпал;
- confidence ниже `confidence_min`;
- response_mode отличается, но не критично.

`low` если:

- отличие допустимо и не ведёт к плохому ответу.
