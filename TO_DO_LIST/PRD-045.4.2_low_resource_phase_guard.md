# PRD-045.4.2 — Low-resource phase guard v1: не переводить слабый ресурс в explore

**Проект:** Bot Psychologist / Neo MindBot  
**Репозиторий:** `Askhat-cmd/Text_transcription`  
**Рабочая зона:** `bot_psychologist`  
**Приоритет:** высокий  
**Риск:** средний  
**Тип:** targeted runtime calibration / phase transition fix  
**Влияние на ответы бота:** да, ожидается более мягкое поведение в low-resource продолжениях  
**Предыдущий этап:** PRD-045.4.1 — Low-resource continuity calibration v1  
**Следующий этап после аудита:** PRD-045.2.1 Writer brevity/completion guard или PRD-045.5 Pattern Core / Active Frame preparation  
**Главный принцип:** простота, модульность, гибкость, быстрые настройки, понятный trace-аудит.

## 0. Краткое резюме

PRD-045.4.1 успешно исправил главную проблему `QB-010`:

```text
Before:
relation=new_thread
relation_reason=continuity_below_threshold
response_mode=reflect
thread_action=new_thread

After:
relation=continue
relation_reason=low_resource_continuation_marker
thread_action=continue_thread
response_mode=validate
mode_reason=explicit_contact_validate
```

То есть low-resource continuation больше не разваливается на новый thread.

Но diagnostics теперь показывает следующий точечный дефект:

```text
QB-010 after:
phase=explore
phase_reason=clarify_to_explore
nervous_state=hypo
intent=contact
response_mode=validate
```

Формально `response_mode=validate`, но фаза стала `explore`. Для low-resource состояния это риск: система может в следующих ходах начать углублять/исследовать слишком рано, хотя пользователь просит очень простую поддержку.

Этот PRD делает маленькую правку:

```text
Если nervous_state=hyper/hypo или intent=contact,
Thread Manager не должен продвигать фазу clarify -> explore только потому, что turns_in_phase >= 2.
```

Цель: удерживать low-resource продолжения в `clarify` или `stabilize`, не переводить их в `explore`.

---

## 1. Почему это нужно

Thread Manager сейчас имеет правило:

```python
elif current_phase == "clarify" and relation == "continue" and turns_in_phase >= 2:
    selected_phase = "explore"
    phase_reason = "clarify_to_explore"
```

Это полезно для обычного диалога, но слишком грубо для low-resource сценариев.

Психологически:

- если человек пишет “сил почти нет”;
- потом “даже отвечать тяжело”;
- потом “можно что-то совсем простое на сейчас?”;

то это не готовность к `explore`. Это запрос на удержание, стабилизацию, простое действие, минимум нагрузки.

После PRD-045.4.1 relation уже исправлен. Теперь нужно не дать фазе преждевременно перескочить в `explore`.

---

## 2. Главная цель

Сделать так, чтобы low-resource/contact continuation не переводился в `explore`.

Ожидаемый результат:

```text
QB-010 final turn:
relation_to_thread=continue
relation_reason=low_resource_continuation_marker
phase != explore
phase_reason=low_resource_hold_phase или contact_hold_phase
response_mode=validate или regulate
```

## 7. Task 3 — Добавить phase guard в Thread Manager

Файл:

```text
bot_psychologist/bot_agent/multiagent/agents/thread_manager.py
```

Изменить `_resolve_phase_with_debug(...)`.

Текущее problematic rule:

```python
elif current_phase == "clarify" and relation == "continue" and turns_in_phase >= 2:
    selected_phase = "explore"
    phase_reason = "clarify_to_explore"
```

Добавить guard перед этим правилом.

Рекомендуемая логика:

```python
elif (
    current_phase == "clarify"
    and relation == "continue"
    and (
        state_snapshot.nervous_state in {"hyper", "hypo"}
        or state_snapshot.intent == "contact"
    )
):
    selected_phase = "clarify"
    phase_reason = (
        "low_resource_hold_phase"
        if state_snapshot.nervous_state in {"hyper", "hypo"}
        else "contact_hold_phase"
    )
elif current_phase == "clarify" and relation == "continue" and turns_in_phase >= 2:
    selected_phase = "explore"
    phase_reason = "clarify_to_explore"
```

### Важные нюансы

1. Safety всё равно должен идти выше всех.
2. `stabilize_to_clarify` для `current_phase=stabilize` и `nervous_state=window` должен остаться.
3. High continuity keep phase можно оставить выше или ниже guard — но если high continuity keep phase возвращает `clarify`, это безопасно.
4. Обычный `clarify -> explore` должен продолжать работать для `window + clarify/explore` случаев.
5. Нельзя ломать `solution -> practice`: phase может остаться clarify, но mode всё равно должен быть `practice`.
