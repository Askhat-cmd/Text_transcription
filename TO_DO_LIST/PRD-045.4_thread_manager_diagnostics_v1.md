# PRD-045.4 — Thread Manager Diagnostics v1: причины relation / phase / response_mode в trace

**Проект:** Bot Psychologist / Neo MindBot  
**Репозиторий:** `Askhat-cmd/Text_transcription`  
**Рабочая зона:** `bot_psychologist`  
**Приоритет:** высокий  
**Риск:** низкий  
**Тип:** observability / diagnostics / thread continuity audit  
**Влияние на ответы бота:** не должно менять финальный текст ответа  
**Предыдущий этап:** PRD-045.3.2 — Runtime Freshness Guard и direct baseline smoke  
**Следующий этап после аудита:** PRD-045.5 Pattern Core / Active Frame preparation или PRD-045.4.1 точечная калибровка continuity  
**Главный принцип:** простота, модульность, гибкость, быстрые настройки, понятный trace-аудит.

---

## 0. Краткое резюме

PRD-045.3.2 подтвердил, что свежий direct runtime работает правильно:

```text
QB-002: direct=response_mode validate, live=response_mode reflect
QB-005: direct=response_mode practice, live=response_mode reflect
```

Значит, behavioral fix PRD-045.3.1 действительно сработал в локальном runtime. Расхождение с live API объясняется stale/другим backend runtime path.

Теперь можно идти дальше по изначальному плану: улучшать наблюдаемость Thread Manager.

Сейчас Thread Manager уже принимает решения:

```text
relation_to_thread: continue | branch | new_thread | return_to_old
phase: stabilize | clarify | explore | integrate
response_mode: validate | reflect | explore | regulate | practice | safe_override
```

Но trace не объясняет, почему эти решения были приняты.

Этот PRD добавляет `thread_diagnostics_v1` — компактный безопасный debug-блок с причинами:

- почему выбрана relation;
- почему выбрана phase;
- почему выбран response_mode;
- какие markers сработали;
- какой continuity score и threshold;
- были ли open/closed loop changes;
- был ли new thread / restore / archive / safety patch.

Важно: PRD не должен менять поведение Thread Manager. Только диагностика.

---

## 1. Почему это нужно

Следующий большой риск качества — потеря линии разговора.

Сейчас Thread Manager использует:

- token overlap между `user_message` и `core_direction`;
- markers возврата;
- branch markers;
- resolution markers;
- phase transitions;
- response mode routing.

Но в trace мы видим в основном итог:

```json
{
  "relation_to_thread": "new_thread",
  "continuity_score": 0.18,
  "phase": "clarify",
  "response_mode": "reflect"
}
```

И не видим причины:

```text
new_thread потому что continuity_score < 0.20?
branch потому что сработал marker "кстати"?
continue потому что сработал followup marker?
reflect потому что phase=clarify?
practice потому что intent=solution?
```

Без этого нельзя безопасно улучшать continuity и удержание линии.

## 6. Privacy and safety

`thread_diagnostics` не должен содержать:

- полный текст пользователя;
- полный `core_direction`;
- token strings;
- semantic hits;
- memory content;
- prompt text;
- secrets.

Разрешены только:

- counts;
- booleans;
- enum values;
- thresholds;
- short reason codes.

## 7. Архитектура реализации

### 7.1 Thread Manager `last_debug`

Файл:

```text
bot_psychologist/bot_agent/multiagent/agents/thread_manager.py
```

Добавить в `ThreadManagerAgent`:

```python
self.last_debug: dict[str, Any] = {}
```

Каждый вызов `update(...)` должен выставлять `last_debug`.

### 7.2 Helper functions

Добавить helpers:

```python
def _relation_diagnostics(...): ...
def _phase_diagnostics(...): ...
def _mode_and_goal(...): ...
def _mode_and_goal_with_reason(...): ...
```

Лучше не ломать публичное поведение `_mode_and_goal`, а сделать внутренний helper:

```python
def _mode_and_goal_with_reason(phase, state_snapshot) -> tuple[str, str, str]:
    ...
```

А `_mode_and_goal(...)` может вызывать его и возвращать только `(mode, goal)` для обратной совместимости:

```python
def _mode_and_goal(phase, state_snapshot):
    mode, goal, _reason = _mode_and_goal_with_reason(phase, state_snapshot)
    return mode, goal
```

### 7.3 Relation diagnostics

Сейчас `_resolve_relation(...)` возвращает:

```python
relation, continuity
```

Можно оставить это поведение, но добавить helper:

```python
def _resolve_relation_with_debug(...) -> tuple[str, float, dict]:
    ...
```

А старый `_resolve_relation(...)` может вызывать новый и возвращать первые два значения.

### 7.4 Phase diagnostics

Сейчас `_resolve_phase(...)` возвращает phase.

Добавить:

```python
def _resolve_phase_with_debug(...) -> tuple[str, dict]:
    ...
```

И сохранить старую `_resolve_phase(...)` для совместимости.

### 7.5 Action diagnostics

В `update(...)` на каждом пути:

- current_thread is None;
- safety patch;
- return_to_old;
- new_thread;
- continue/branch;
- fallback;

нужно заполнить `self.last_debug`.

## 11. Тесты Thread Manager diagnostics

Создать:

```text
bot_psychologist/tests/multiagent/test_thread_manager_diagnostics.py
```

### 11.1 New thread diagnostics

Case: `current_thread=None`.

Проверить:

```python
debug["version"] == "thread_diagnostics_v1"
debug["action"]["thread_action"] == "new_thread"
debug["relation"]["relation_reason"] == "no_current_thread"
debug["phase"]["phase_reason"] == "new_thread_default_clarify"
```

### 11.2 Solution mode reason

StateSnapshot `intent=solution`.

Проверить:

```python
debug["mode"]["selected_mode"] == "practice"
debug["mode"]["mode_reason"] == "solution_practice"
```

### 11.3 Contact mode reason

StateSnapshot `intent=contact`, `nervous_state=hypo`.

Проверить:

```python
debug["mode"]["selected_mode"] == "validate"
debug["mode"]["mode_reason"] == "explicit_contact_validate"
```

### 11.4 Low continuity new thread

Current thread с `core_direction="project criticism fear"`, user_message unrelated.

Проверить:

```python
debug["relation"]["relation_reason"] == "continuity_below_threshold"
debug["summary_flags"] includes "low_continuity_new_thread"
```

### 11.5 Followup marker override

Current thread есть, message содержит followup marker.

Проверить:

```python
debug["relation"]["relation_reason"] == "followup_continue_marker"
```

### 11.6 Branch marker

Message содержит `кстати` или `by the way`.

Проверить:

```python
debug["relation"]["branch_marker_hit"] is True
```

Если branch не срабатывает из-за low continuity precedence, тест должен учитывать текущую логику и не менять её.
