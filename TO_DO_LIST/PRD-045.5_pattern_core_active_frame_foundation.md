# PRD-045.5 — Pattern Core / Active Frame v1: подготовка смысловой линии без нового агента

**Проект:** Bot Psychologist / Neo MindBot  
**Репозиторий:** `Askhat-cmd/Text_transcription`  
**Рабочая зона:** `bot_psychologist`  
**Приоритет:** высокий  
**Риск:** средний  
**Тип:** architecture foundation / continuity semantics / trace foundation  
**Влияние на ответы бота:** минимальное, контролируемое  
**Предыдущий этап:** PRD-045.4.2 — Low-resource phase guard v1  
**Следующий этап после аудита:** PRD-045.5.1 Pattern Core usage in Writer / Planner-lite  
**Главный принцип:** простота, модульность, гибкость, быстрые настройки, понятный trace-аудит.

---

## 0. Краткое резюме

После PRD-045.0–045.4.2 проект получил хороший диагностический фундамент:

- baseline dataset и runner;
- `quality_trace`;
- `thread_diagnostics`;
- direct baseline mode;
- калибровку State Analyzer / response routing;
- low-resource continuity guard;
- phase guard для low-resource.

Теперь техническая линия разговора стала стабильнее:

```text
QB-002: validate / clarify / explicit_contact_validate
QB-005: practice / clarify / solution_practice
QB-010: continue / clarify / low_resource_hold_phase
```

Но текущий Thread Manager всё ещё в основном держит continuity через:

- token overlap;
- marker rules;
- phase counters;
- response mode routing.

Это полезно, но недостаточно для “умного” психологического диалога.

Следующий шаг — добавить минимальный смысловой слой:

```text
pattern_core
active_frame
last_user_need
last_supportive_move
```

Важно: этот PRD не добавляет нового LLM-агента, не добавляет Response Planner и не переписывает Thread Manager.  
Он только создаёт поля, storage, trace и prompt-context foundation, чтобы следующие PRD могли использовать смысловую линию безопасно.

---

## 1. Почему это нужно

Ранее пользователь дал референс-диалог другого бота, где сильная сторона — удержание линии:

- бот не начинает каждый ответ заново;
- возвращает пользователя к одному паттерну;
- использует предыдущие ответы пользователя;
- задаёт один следующий вопрос/шаг;
- фиксирует смысловой сдвиг.

Наш текущий бот уже лучше держит техническую нить, но пока не имеет явного поля:

```text
“О чём на самом деле эта линия разговора?”
```

Сейчас есть `core_direction`, но это скорее срез первого/последнего пользовательского сообщения. Нужно добавить более устойчивую и компактную рамку:

```json
{
  "pattern_core": "страх оценки и уход в бесконечную подготовку",
  "active_frame": {
    "current_need": "короткая поддержка без давления",
    "last_supportive_move": "дали разрешение на паузу",
    "next_recommended_direction": "удержать контакт, не углублять"
  }
}
```

В этом PRD мы не пытаемся идеально вычислять эти поля.  
Мы создаём безопасную v1-структуру и простые deterministic правила.

## 5. Новые поля ThreadState

Файл:

```text
bot_psychologist/bot_agent/multiagent/contracts/thread_state.py
```

Добавить в `ThreadState` поля с безопасными default values:

```python
pattern_core: str = ""
active_frame: dict[str, str] = field(default_factory=dict)
```

### 5.1 Почему `dict[str, str]`, а не отдельный dataclass

Для v1 лучше не усложнять схему.  
`active_frame` пока compact dict.

Рекомендуемые ключи:

```python
{
    "current_need": "",
    "last_supportive_move": "",
    "next_recommended_direction": "",
}
```

Можно хранить только непустые ключи.

### 5.2 Backward compatibility

`from_dict(...)` должен безопасно читать старые сохранённые thread states, где этих полей нет.

Убедиться:

```python
payload.setdefault("pattern_core", "")
payload.setdefault("active_frame", {})
```

Также защититься, если `active_frame` не dict:

```python
if not isinstance(payload.get("active_frame"), dict):
    payload["active_frame"] = {}
```

### 5.3 `to_dict()`

Добавить:

```python
"pattern_core": self.pattern_core,
"active_frame": self.active_frame,
```

## 7. Deterministic Pattern Core v1

Файл:

```text
bot_psychologist/bot_agent/multiagent/agents/thread_manager.py
```

Добавить helpers:

```python
def _derive_pattern_core_v1(
    *,
    user_message: str,
    state_snapshot: StateSnapshot,
    previous_core: str = "",
) -> str:
    ...
```

### 7.1 Не делать LLM summarization

Никаких API-вызовов.  
Только простые deterministic правила.

### 7.2 Правила v1

Если есть `previous_core`, по умолчанию сохранять его.

Если `previous_core` пустой, создать из первого сообщения:

```python
_clean_core = user_message.strip()[:120]
```

Но можно сделать чуть лучше:

- если `intent=contact` и `nervous_state=hypo`:  
  `pattern_core = "low-resource support / короткая поддержка при низком ресурсе"`
- если `intent=solution`:  
  `pattern_core = "request for concrete next step / запрос на конкретный следующий шаг"`
- если `intent=clarify`:  
  `pattern_core = "clarifying inner pattern / прояснение внутреннего паттерна"`
- если `intent=vent`:  
  `pattern_core = "emotional venting / выражение сильного чувства"`
- иначе fallback: first message slice.

Важно: это не должно быть “умнее”, чем данные позволяют.

### 7.3 Когда обновлять pattern_core

В v1:

- `new_thread`: создать pattern_core.
- `continue`: сохранять existing pattern_core.
- `branch`: можно сохранить existing pattern_core или слегка добавить suffix? Лучше сохранить existing.
- `return_to_old`: восстановить archived pattern_core.
- `safety_patch`: сохранить existing pattern_core.

Не пытаться динамически переписывать pattern_core на каждом ходе.

## 8. Active Frame v1

Добавить helper:

```python
def _build_active_frame_v1(
    *,
    user_message: str,
    state_snapshot: StateSnapshot,
    thread_state: ThreadState | None,
    relation: str,
    response_mode: str,
    phase: str,
) -> dict[str, str]:
    ...
```

### 8.1 Поля

Рекомендуемые ключи:

```python
{
    "current_need": "...",
    "last_supportive_move": "...",
    "next_recommended_direction": "..."
}
```

### 8.2 Deterministic rules

`current_need`:

```text
safety_flag -> "immediate safety and stabilization"
intent=contact + hypo -> "short support without pressure"
intent=contact -> "warm contact and validation"
intent=solution -> "one concrete next step"
intent=clarify -> "gentle clarification"
intent=vent -> "emotional validation"
intent=explore -> "explore perspective carefully"
```

`last_supportive_move`:

- для нового thread: `"started contact and matched response mode"`
- для continue: `"continued current thread without restarting"`
- для branch: `"acknowledged branch while preserving thread"`
- для safety: `"safety override"`
- для low_resource continuation: `"held low-resource thread and avoided exploration"`

`next_recommended_direction`:

```text
safety -> "prioritize safety"
hypo/contact -> "keep answer short and low pressure"
hyper -> "stabilize before analysis"
solution -> "offer one executable micro-step"
clarify -> "reflect one key point and ask at most one question"
explore -> "expand carefully without strong interpretation"
```

### 8.3 Не хранить raw user text

`active_frame` не должен содержать полный user message.

## 9. Thread Manager integration

### 9.1 New thread

В `_new_thread(...)` добавить:

```python
pattern_core = _derive_pattern_core_v1(...)
active_frame = _build_active_frame_v1(...)
```

Передать в `ThreadState(...)`.

### 9.2 Continue thread

В `_continue_thread_with_debug(...)`:

```python
pattern_core=current_thread.pattern_core or _derive_pattern_core_v1(...)
active_frame=_build_active_frame_v1(...)
```

### 9.3 Restore archived

В `_restore_archived(...)`:

```python
pattern_core=best.pattern_core
active_frame=best.active_frame
```

Если archived не расширен — использовать fallback.

### 9.4 Safety patch

Сохранять existing `pattern_core`, но active_frame обновить:

```python
current_need="immediate safety and stabilization"
next_recommended_direction="prioritize safety"
```

- API keys;
- bearer/session tokens;
- invite keys;
- HMAC/internal keys;
- raw private conversations;
- local DB;
- `.sqlite`;
- `.cache_bot_agent`;
- `node_modules`;
- `.venv`;
- `.cursor`.
