# PRD-045.6.1 — Context Assembly Calibration / Micro-Summary Quality Guard v1

## 0. Краткое резюме

**Цель PRD-045.6.1:** довести слой `Context Assembly / Turn Micro-Summary v1` после реализации PRD-045.6 до более надёжного качества, не переходя сразу к Diagnostic Center поверх слишком грубого сжатия.

PRD-045.6 успешно добавил:

- `ContextAssemblyPackage`;
- `TurnMicroSummary`;
- `ContextBudget`;
- `ContextAssemblyTrace`;
- интеграцию в `MemoryRetrievalAgent`, `WriterContract`, `orchestrator`, baseline runner;
- smoke/debug отчёты.

Но аудит показал важный хвост:

```text
micro-summary v1 сейчас слишком грубый:
- threshold для full-turn слишком низкий;
- summary строится в основном как first sentence + last sentence;
- смысловые маркеры почти не учитываются;
- important_quote часто отсутствует;
- dedup слишком буквальный и может не ловить реальные дубли semantic hit vs recent turns.
```

Этот PRD — маленькая калибровочная итерация перед PRD-045.7.

Главный принцип:

```text
Не менять архитектуру PRD-045.6.
Не менять State Analyzer / Thread Manager / Writer prompt radically.
Только улучшить качество context assembly и добавить тесты, которые ловят потерю смысла.
```

---

## 1. Почему нужно

PRD-045.6 решил архитектурную проблему: Writer теперь получает управляемый context package и trace.

Но качество пакета зависит от двух деталей:

1. Какие turns сохраняются full.
2. Как сжимаются длинные turns.

Если обычные средние сообщения пользователя слишком рано уходят в summary, Writer снова теряет живую ткань диалога. Если summary берёт только начало и конец, смысл из середины может исчезнуть.

Для Bot Psychologist / Neo MindBot это особенно опасно, потому что важный психологический материал часто находится не в первой и не в последней фразе, а в середине:

```text
"Сначала я думал, что проблема в работе...
потом понял, что сильнее всего меня задевает чувство, что я никому не нужен...
и теперь не знаю, что с этим делать."
```

Если summary возьмёт только начало и конец, ключевой паттерн потеряется.

---

## 2. Цель

### 2.1. Основная цель

Улучшить `Context Assembly v1` так, чтобы:

1. Короткие и средние recent turns чаще оставались full.
2. Длинные turns сжимались через deterministic marker-scored summary, а не просто first+last.
3. `important_quote` выбирался даже без кавычек, если есть короткая эмоционально значимая фраза.
4. Dedup лучше ловил реальные дубли между recent turns и semantic memory hits.
5. Trace показывал метод и причины summary/dedup без raw private logs.
6. Tests проверяли не только counts, но и сохранение ключевой смысловой фразы из середины длинного turn.

### 2.2. Архитектурная цель

Сделать Context Assembly достаточно надёжным основанием для следующего слоя:

```text
PRD-045.7 — Diagnostic Center Contract / WriterContract v2
```

Diagnostic Center не должен строиться на слишком случайном сжатии контекста.

## Task 2 — Implement marker-scored deterministic micro-summary

Заменить текущий слишком простой алгоритм summary на deterministic scoring.

### 2.1. Требуемый helper

Сохранить существующий публичный API, если он уже используется:

```python
build_turn_micro_summary_v1(
    *,
    turn_id: str,
    role: str,
    content: str,
    max_summary_chars: int = MICRO_SUMMARY_MAX_CHARS,
) -> TurnMicroSummary
```

### 2.2. Алгоритм scoring

1. Normalize whitespace.
2. Split into sentence/fragments.
3. Для каждого фрагмента рассчитать score.

User markers:

```text
я
мне
меня
моё
хочу
не хочу
не могу
не получается
сил нет
устал
устала
страшно
боюсь
стыдно
злюсь
не знаю
важно
задевает
больно
один
одна
никому не нужен
никому не нужна
должен
надо
```

Assistant markers:

```text
можно попробовать
первый шаг
микро-шаг
на 1 минуту
практика
заметь
проверь
вопрос
если коротко
смысл
важно
```

Universal markers:

```text
?
!
но
при этом
самое важное
главное
```

4. Добавить небольшой positional boost:
   - first meaningful fragment: +1;
   - last meaningful fragment: +1;
   - fragments with question mark: +1;
   - fragments length 20–220 chars: +1.
5. Выбрать 2–4 фрагмента с максимальным score.
6. Сохранить порядок выбранных фрагментов как в исходном тексте.
7. Уложить в `max_summary_chars`.
8. Не добавлять психологические интерпретации от себя.

### 2.3. Пример правильного summary

Input:

```text
Сначала я думал, что дело просто в усталости. На работе всё вроде нормально, но внутри постоянно есть чувство, что я никому не нужен. Я пытаюсь быть полезным, но чем больше стараюсь, тем сильнее пустота. И теперь не знаю, с чего начать.
```

Ожидаемый summary должен сохранить смысловую середину:

```text
... внутри постоянно есть чувство, что я никому не нужен. Я пытаюсь быть полезным, но чем больше стараюсь, тем сильнее пустота. И теперь не знаю, с чего начать.
```

Не обязательно дословно в таком виде, но фраза про `никому не нужен` / `пустота` / `не знаю, с чего начать` должна сохраниться.

## Task 3 — important_quote fallback

Сейчас quote может появляться только если пользователь поставил кавычки. Нужно добавить fallback.

Алгоритм:

1. Если есть quoted text — оставить текущую логику.
2. Если quoted text нет:
   - взять самый короткий high-score фрагмент user turn;
   - длина желательно 20–160 chars;
   - не брать слишком технические фразы;
   - не брать пустые/служебные фрагменты.
3. Если подходящего нет — `important_quote=None`.

Примеры хорошего `important_quote`:

```text
"внутри постоянно есть чувство, что я никому не нужен"
"сил почти нет"
"я не знаю, с чего начать"
```

Trace не должен выводить raw quote в sanitized logs. В package quote допустим, потому что это часть prompt context, но в safe logs — только факт наличия quote и длина.

## Task 4 — Better dedup for real semantic hits

Сейчас exact normalized text может не ловить реальный дубль, потому что semantic hit часто выглядит так:

```text
User: <полный user_input>
Assistant: <bot_response_preview>
```

А recent full содержит отдельно:

```text
<полный user_input>
<полный bot_response>
```

Нужно добавить более практичный v1 dedup.

### 4.1. Required behavior

Если semantic hit содержит recent user text или recent assistant text как подстроку, и длина совпадения значимая, semantic hit удаляется.

Пример:

```text
recent user: "мне очень тяжело сегодня, сил почти нет"
semantic hit: "User: мне очень тяжело сегодня, сил почти нет\nAssistant: я рядом..."
```

Ожидание:

```text
semantic hit removed as duplicate
```

### 4.2. Simple rule

Нормализовать тексты.

Считать duplicate, если:

```text
recent_text in hit_text
```

и

```text
len(recent_text) >= 40
```

или если similarity по token overlap выше простого порога:

```text
common_tokens / min(len(tokens_a), len(tokens_b)) >= 0.75
```

Не нужно подключать внешние библиотеки.

### 4.3. Trace

Добавить safe reason:

```text
semantic_duplicate_recent_substring
knowledge_duplicate_recent_substring
semantic_duplicate_token_overlap
```

В trace counts можно показать:

```json
{
  "duplicates_removed": 2,
  "reasons": ["semantic_duplicate_recent_substring"]
}
```
