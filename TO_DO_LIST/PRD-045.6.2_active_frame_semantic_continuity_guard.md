# PRD-045.6.2 — Active Frame Semantic Continuity Guard / QB-015 v1

## 0. Краткое резюме

**Цель PRD-045.6.2:** закрыть следующий слабый шов после PRD-045.6 / PRD-045.6.1: ложный `new_thread` в ситуациях, где пользователь явно продолжает прежнюю смысловую линию, но lexical overlap с `core_direction` низкий.

Главный индикатор — baseline case:

```text
QB-015 — Continuation without restarting frame
```

После PRD-045.6.1 `Context Assembly` уже работает лучше: recent turns остаются full, duplicates удаляются, context package собран. Но `Thread Manager` всё ещё принимает решение до Writer и до Diagnostic Center. В `QB-015` он технически выбирает:

```text
relation_to_thread = new_thread
relation_reason = continuity_below_threshold
```

При этом содержательно это продолжение той же линии: пользователь снова хочет отложить публикацию/действие. Значит, следующий маленький шаг — не Diagnostic Center, а аккуратный continuity guard на уровне Thread Manager.

**Принцип:** не добавлять бесконечный список маркеров, а использовать уже созданные `pattern_core`, `active_frame`, `open_loops` и `core_direction` как semantic frame для решения “это продолжение или новая тема?”.

---

## 1. Почему нужно

Сейчас relation routing в Thread Manager в основном опирается на:

- explicit return markers;
- follow-up markers;
- branch markers;
- low-resource markers;
- token overlap между `user_message` и `current_thread.core_direction`.

Это хорошо работает для простых случаев, но ломается на коротких продолжениях с низким overlap:

```text
И сейчас снова хочу отложить.
```

Такая реплика может содержать мало общих токенов с изначальным `core_direction`, но слово `снова` и действие `отложить` явно указывают на продолжение старого паттерна.

Если Thread Manager выбирает `new_thread`, система теряет техническую нить:

- старый thread архивируется;
- `pattern_core` может стать слишком узким raw slice текущего сообщения;
- `active_frame` перезапускается как новый контакт;
- будущий Diagnostic Center будет строиться поверх искажённой thread relation;
- trace показывает `continuity_below_threshold`, хотя для пользователя это продолжение.

Поэтому до PRD-045.7 нужен маленький continuity calibration PRD.

## 2. Цель

### 2.1. Основная цель

Добавить в Thread Manager deterministic guard:

```text
Active Frame Semantic Continuity Guard v1
```

Он должен срабатывать, когда:

1. Есть `current_thread`.
2. Нет safety override.
3. Нет явного branch / return_to_old / resolution scenario, который должен иметь приоритет.
4. Raw lexical continuity ниже threshold.
5. В текущем сообщении есть recurrence / continuation cue: `снова`, `опять`, `всё ещё`, `по-прежнему`, `again`, `still` и т.п.
6. Сообщение имеет хотя бы минимальную смысловую связь с current semantic frame:
   - `current_thread.core_direction`;
   - `current_thread.pattern_core`;
   - `current_thread.open_loops`;
   - values из `current_thread.active_frame`.

Если условия выполнены, relation должен стать:

```text
relation_to_thread = continue
relation_reason = active_frame_semantic_continuity
continuity_score = max(raw_score, 0.25)
```

### 2.2. Цель по baseline

Для `QB-015` ожидаемый результат после PRD-045.6.2:

```text
relation_to_thread = continue
relation_reason = active_frame_semantic_continuity
phase = clarify или текущая корректная phase по existing phase logic
response_mode = reflect или другой existing mode без деградации
pattern_core preserved from previous thread
active_frame present
context_assembly_trace present
```

`QB-010` не должен сломаться:

```text
relation_to_thread = continue
relation_reason = low_resource_continuation_marker
phase = clarify
response_mode = validate
```

## Task 2 — Add semantic continuity helpers

Добавить маленькие deterministic helper functions рядом с relation logic.

Рекомендуемые constants:

```python
_RECURRENCE_CONTINUITY_MARKERS = (
    "снова",
    "опять",
    "всё ещё",
    "все еще",
    "по-прежнему",
    "как тогда",
    "то же самое",
    "та же история",
    "again",
    "still",
    "same thing",
)
```

Рекомендуемый helper:

```python
def _active_frame_continuity_signal(
    *,
    user_message: str,
    current_thread: ThreadState,
) -> tuple[bool, dict[str, Any]]:
    ...
```

Он должен вернуть:

```python
(
    True/False,
    {
        "recurrence_marker_hit": bool,
        "semantic_frame_token_overlap": int,
        "semantic_frame_token_count": int,
        "message_token_count": int,
        "matched_sources": ["core_direction", "pattern_core", "open_loops", "active_frame"],
    }
)
```

### 2.1. Semantic frame text

Собрать safe semantic frame text из:

```python
current_thread.core_direction
current_thread.pattern_core
" ".join(current_thread.open_loops)
" ".join(str(v) for v in current_thread.active_frame.values())
```

Не использовать raw archived thread bodies.

### 2.2. Minimal overlap rule

Guard срабатывает, если:

```text
recurrence_marker_hit == True
AND semantic_frame_token_overlap >= 1
```

или более надёжный вариант:

```text
recurrence_marker_hit == True
AND token overlap ratio between user_message and semantic_frame >= 0.08
```

Для коротких сообщений лучше использовать `>= 1 meaningful token`, чтобы не потерять `снова хочу отложить`.

### 2.3. Meaningful tokens

Использовать уже существующую `_normalize_tokens(...)`, но при необходимости добавить stopwords для слишком общих слов:

```text
сейчас, хочу, как, это, тоже, опять, снова, ещё, еще, still, again
```

Важно: recurrence marker сам по себе не должен считаться semantic overlap. Иначе фраза `снова хочу купить велосипед` может ошибочно продолжить старый thread про публикацию.

## Task 3 — Integrate guard into `_resolve_relation_with_debug`

Добавить guard в relation priority.

Рекомендуемый порядок:

```text
return_to_old marker with archived threads
followup_continue_marker
branch_marker
low_resource_continue_marker
active_frame_semantic_continuity
continuity_below_threshold -> new_thread
else continue
```

### 3.1. Почему после branch marker

Если пользователь явно говорит `кстати`, `ещё вопрос`, `другая тема`, это может быть branch. Guard не должен перехватывать branch.

### 3.2. Почему после low-resource marker

Low-resource contact continuation уже откалиброван и важен для `QB-010`. Его reason должен остаться прежним:

```text
low_resource_continuation_marker
```

### 3.3. Expected override

Если raw continuity ниже threshold, но semantic guard hit:

```python
relation = "continue"
continuity = max(continuity_raw, 0.25)
relation_reason = "active_frame_semantic_continuity"
```

1. Добавлен deterministic `active_frame_semantic_continuity` guard.
2. Guard использует recurrence marker + semantic frame overlap, а не просто отдельный keyword.
3. Branch markers всё ещё имеют приоритет над semantic continuity guard.
4. Low-resource continuation не сломан.
5. `QB-015` в direct smoke больше не уходит в `new_thread`.
6. `QB-010` остаётся `continue / clarify / validate`.
