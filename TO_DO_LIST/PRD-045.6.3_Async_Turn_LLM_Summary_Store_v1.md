# PRD-045.6.3 — Async Turn LLM Summary Store v1

## 0. Статус документа

- **Проект:** Bot Psychologist / Neo MindBot
- **PRD:** `PRD-045.6.3`
- **Название:** Async Turn LLM Summary Store v1
- **Язык:** русский
- **Тип:** context-quality / memory additive layer
- **Исполнитель:** IDE-агент
- **Архитектор:** ChatGPT / проектный архитектор
- **Цель:** добавить безопасный слой LLM-суммаризации завершённых turns для улучшения recent dialogue context, сохранив deterministic micro-summary как fallback и не блокируя текущий ответ пользователя.

---

## 1. Почему этот PRD нужен сейчас

После `PRD-046.0.5-APPLY1` Knowledge Base получила controlled real LLM enrichment overlay:

```text
real LLM enrichment metadata applied
→ Chroma refreshed
→ /api/query отдаёт governance + enrichment metadata
→ bot retrieval path использует botdb_api
```

Это усиливает качество knowledge context.

Но recent dialogue context пока опирается на `PRD-045.6`:

```text
Context Assembly
→ recent turns full if short
→ deterministic extractive micro-summary if long
```

Это хороший baseline, но для психологического бота возникает риск асимметрии:

```text
Knowledge context становится LLM-связным,
а живой диалог пользователя остаётся extractive-фрагментарным.
```

Если сейчас пойти сразу в retrieval eval / Diagnostic Center, Writer может начать сильнее полагаться на красиво сформулированный KB context, чем на живую линию диалога. Поэтому следующий безопасный шаг — добавить additive async LLM summary для завершённых exchanges.

## 2. Текущая архитектурная база

`PRD-045.6` уже создал основу:

```text
bot_psychologist/bot_agent/multiagent/contracts/context_package.py
bot_psychologist/bot_agent/multiagent/context_assembly.py
MemoryRetrievalAgent structured recent_turns
WriterContract.context_package
context_assembly_trace_v1
build_turn_micro_summary_v1(...)
```

Важно: этот PRD не переписывает Context Assembly. Он добавляет новый summary source:

```text
full turn
llm_abstractive summary if ready and valid
deterministic_fallback if llm summary missing/pending/failed/hash mismatch
```

---

## 3. Главная цель

Добавить в систему хранение и использование LLM-summary завершённых диалоговых обменов:

```text
user message + bot response saved raw
→ enqueue / mark pending turn summary
→ summary job creates compact JSON summary
→ summary stored next to raw turn
→ next Context Assembly uses llm summary if ready
→ otherwise deterministic fallback
```

При этом:

- текущий ответ пользователя не должен ждать LLM-summary;
- raw turn history всегда сохраняется полностью;
- deterministic summary остаётся fallback;
- summary-model не ставит диагнозы, не даёт советы, не принимает safety decisions;
- feature flag должен позволять выключить весь слой.

## 5. Архитектурное решение

### 5.1. Что такое turn summary

В v1 summary создаётся для завершённого exchange:

```text
user_input + assistant_response + minimal metadata → TurnLLMSummary
```

Summary не является ответом пользователю. Это внутренний context artifact для следующего хода.

### 5.2. Хранимый контракт

Добавить contract, например:

```text
bot_psychologist/bot_agent/multiagent/contracts/turn_llm_summary.py
```

Рекомендуемый contract:

```json
{
  "status": "pending|ready|failed|disabled",
  "summary": "2-3 предложения",
  "important_quote": "короткая цитата пользователя или пусто",
  "open_loop": "какой вопрос/нить осталась открытой или пусто",
  "user_need": "главная потребность пользователя",
  "assistant_move": "что бот уже сделал/предложил",
  "emotional_tone": "краткое описание тона",
  "summary_version": "turn_llm_summary_v1",
  "summary_method": "llm_abstractive_v1",
  "model": "configured_model",
  "provider": "openai|mock|disabled",
  "source_hash": "sha256(user_input + assistant_response)",
  "created_at": "ISO-8601",
  "error": null
}
```

Минимально обязательные поля:

```text
status
summary
summary_version
summary_method
source_hash
created_at
```

Остальные поля полезны для Writer / Thread Manager / будущего Diagnostic Center.

### 5.3. Source hash

Обязательно вычислять:

```text
source_hash = sha256(normalized_user_input + "\n---assistant---\n" + normalized_assistant_response)
```

Context Assembly может использовать LLM-summary только если:

```text
summary.status == ready
summary.source_hash == current_turn_source_hash
summary.summary_version == turn_llm_summary_v1
summary.summary not empty
```

Иначе fallback.

- тесты должны работать без реального API через mock provider.

Цель: не тащить повреждённую документацию дальше.

### Workstream B — Contract + validation

Добавить contract для LLM turn summary.

Добавить validator:

```text
validate_turn_llm_summary(summary, source_hash)
```

Проверки:

- status in allowlist;
- ready summary has non-empty summary;
- summary length <= configured max;
- source_hash matches;
- no explicit diagnosis language;
- no advice/action commands in summary-model voice;
- no internal technical labels exposed as user-facing content;
- no raw full turn dump.

Validator должен быть conservative. Failed summary не используется в context assembly.

### Workstream C — Summary prompt

Добавить prompt file:

```text
bot_psychologist/bot_agent/multiagent/prompts/turn_llm_summary_v1.md
```

Prompt должен требовать строгий JSON:

```json
{
  "summary": "...",
  "important_quote": "...",
  "open_loop": "...",
  "user_need": "...",
  "assistant_move": "...",
  "emotional_tone": "..."
}
```

Prompt requirements:

```text
Сожми завершённый обмен пользователь+бот в 2–3 предложения.

Сохрани:
- эмоциональный тон пользователя;
- главный запрос или напряжение;
- что бот уже сделал или предложил;
- открытую петлю, если есть;
- важную короткую цитату пользователя, если она помогает.

Запрещено:
- диагнозы;
- новые интерпретации;
- советы;
- усиление драматизации;
- technical labels;
- пересказ всего текста;
- длинные цитаты.
```

### Workstream D — Summary service

Добавить service, например:

```text
bot_psychologist/bot_agent/multiagent/turn_summary_service.py
```

Функции:

```python
compute_turn_source_hash(user_input: str, assistant_response: str) -> str

build_pending_turn_summary_record(...) -> TurnLLMSummary

generate_turn_llm_summary_v1(
    user_input: str,
    assistant_response: str,
    metadata: dict | None = None,
    provider: str = "mock/openai"
) -> TurnLLMSummary

should_summarize_turn(user_input, assistant_response, thresholds) -> bool
```

Правила:

- Не суммаризировать очень короткие turns.
- Не суммаризировать пустые turns.
- Обрезать input до safe max chars.
- Не сохранять raw prompt в artifacts.
- При ошибке вернуть status=`failed`, error=safe_preview.
- Mock provider должен быть deterministic и покрывать тесты.

### Workstream E — Memory update integration

Найти фактическую точку `Memory Update` / сохранения turn.

После сохранения raw user + assistant turn:

```text
if TURN_LLM_SUMMARY_ENABLED:
    attach pending summary metadata
    enqueue / mark for async processing
```

В v1 допустимы два безопасных режима:

#### Режим 1 — Pending-only in runtime

Runtime после ответа только записывает:

```json
"turn_llm_summary": {
  "status": "pending",
  "source_hash": "...",
  "summary_version": "turn_llm_summary_v1"
}
```

#### Режим 2 — Separate processor

Отдельный CLI/tool обрабатывает pending summaries:

```bash
python bot_psychologist/tools/process_turn_llm_summaries.py --limit 10 --mock
python bot_psychologist/tools/process_turn_llm_summaries.py --limit 10 --provider openai --confirm
```

Это безопаснее, чем пытаться внедрить полноценную background task infrastructure сразу.

Важно: текущий пользовательский ответ не ждёт этот processor.

### Workstream G — Eval set

Создать небольшой eval set:

```text
bot_psychologist/tests/fixtures/turn_summary_eval_cases.json
```

Минимум 5 кейсов:

1. Страх оценки проекта:
   - проект → откладывание → страх оценки → желание не показывать.
2. Самокритика после несделанного обещания:
   - обещание → срыв → злость → самокритика.
3. Low-resource контакт:
   - “нет сил, не хочу анализа” сохраняется как просьба о простом присутствии.
4. Бот уже дал практику:
   - summary фиксирует, что практика уже была предложена.
5. Открытый вопрос:
   - summary сохраняет open_loop.

Eval может быть deterministic/mock на этом PRD. Real LLM eval не обязателен.


