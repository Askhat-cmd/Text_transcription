# PRD-045.6 — Context Assembly / Turn Micro-Summary v1

## 0. Краткое резюме

**Цель PRD-045.6:** ввести управляемую сборку контекста перед Writer, чтобы Writer получал не случайно обрезанные фрагменты истории через `...`, а понятный `context package`:

```text
current_user_message
recent_turns_full
recent_turns_summarized
active_frame
pattern_core
personal_history_context / semantic_memory_hits
knowledge_rag_hits
context_budget
context_assembly_trace_v1
```

Этот PRD должен исправить текущую проблему: Writer видит обрывки истории и дублирующиеся куски контекста, из-за чего даже хороший `pattern_core` и `active_frame` не всегда дают устойчивую смысловую линию.

**Главный принцип:**  
не “запихнуть весь чат в prompt”, а собрать компактный, проверяемый, traceable context package.

**Тип изменения:** маленькая инфраструктурная итерация.  
**Риск:** средний, потому что меняется путь контекста в Writer.  
**Ограничение:** не менять поведение State Analyzer / Thread Manager, не переписывать Writer prompt радикально, не менять Knowledge Base chunking.

---

## 1. Почему нужно

После PRD-045.5 в системе появились `pattern_core` и `active_frame`, но остаётся проблема выше Writer:

```text
Writer получает историю и релевантные обмены как обрезанные куски,
часть контекста дублируется,
часть живой линии теряется,
trace не объясняет, что было включено, что сжато, что отброшено.
```

Это ограничивает качество ответов:

- Writer может не видеть последнюю важную реплику целиком.
- Длинные turn’ы превращаются в `content[:N] + "..."`, и смысл может ломаться.
- Одно и то же может попасть одновременно как recent history и как relevant exchange.
- RAG/knowledge chunks могут конкурировать с личной историей пользователя.
- Debug показывает результат, но не показывает механику сборки контекста.

Правильный следующий слой:

```text
Context Assembly
```

Он должен стать отдельным наблюдаемым шагом между Memory Retrieval / Thread Manager и Writer.

---

## 2. Цель

### 2.1. Основная цель

Создать **Context Assembly / Turn Micro-Summary v1**, который:

1. Сохраняет raw history полностью в storage.
2. Передаёт Writer последние важные короткие/средние turn’ы полностью.
3. Для длинных turn’ов создаёт deterministic extractive micro-summary.
4. Разделяет источники контекста:
   - recent conversation;
   - personal history / semantic memory;
   - knowledge RAG;
   - active frame / pattern core.
5. Удаляет простые дубли.
6. Соблюдает общий context budget.
7. Добавляет `context_assembly_trace_v1` в debug / baseline reports.
8. Не ломает существующие baseline cases и routing calibration.

### 2.2. Концептуальная цель

Writer должен видеть не “обрывки прошлого”, а собранную рабочую сцену:

```text
Что пользователь сказал сейчас?
Что было в последних репликах?
Какой смысловой frame уже активен?
Какая личная память релевантна?
Какие knowledge chunks реально нужны?
Что было сжато и почему?
```

## 3. Non-goals

В рамках PRD-045.6 **нельзя** делать следующее:

1. **Не добавлять полноценный LLM Summarizer.**  
   В v1 используется deterministic / extractive micro-summary. LLM summarizer можно вынести в отдельный будущий PRD.

2. **Не менять State Analyzer behavior.**  
   Никакой перекалибровки `intent`, `nervous_state`, `openness`, `ok_position`.

3. **Не менять Thread Manager behavior.**  
   Никакой новой логики `relation_to_thread`, `phase`, `response_mode`.

4. **Не переписывать Writer prompt радикально.**  
   Можно только аккуратно изменить формат входного context block, чтобы Writer получил новый `context package`.

5. **Не менять Knowledge Base chunking.**  
   Проблема generic `book` chunks решается позже в PRD-046.0.

6. **Не делать Retrieval Governance полностью.**  
   В PRD-045.6 только подготовить naming и минимальное разделение:
   - `personal_history_context`;
   - `semantic_memory_hits`;
   - `knowledge_rag_hits`.

7. **Не сохранять raw private logs в репозиторий.**

8. **Не коммитить базы, `.env`, ключи, runtime cache.**

## Task 3 — Добавить deterministic extractive micro-summary v1

Добавить helper, например:

```python
build_turn_micro_summary_v1(turn: ConversationTurn, max_summary_chars: int = 360) -> TurnMicroSummary
```

### 3.1. Что это НЕ должно делать

Нельзя просто заменить:

```python
content[:N] + "..."
```

на другое место в коде.

Нужно создать структурированное сжатие:

```json
{
  "turn_id": "...",
  "role": "user",
  "summary": "Пользователь описывает ...",
  "important_quote": "Сил почти нет.",
  "was_truncated": true,
  "raw_chars": 2400,
  "summary_chars": 280,
  "summary_method": "deterministic_extractive_v1"
}
```

### 3.2. Алгоритм v1

LLM не использовать.

Сделать deterministic extractive algorithm:

1. Normalize whitespace.
2. Split text into candidate sentences / fragments.
3. Score fragments by markers:
   - для user:
     - `я`, `мне`, `меня`, `хочу`, `не могу`, `страшно`, `устал`, `сил нет`, `не знаю`, `важно`, `боюсь`, `надо`, `должен`;
   - для assistant:
     - фразы с выводом, микро-действием, вопросом, practice CTA;
   - универсально:
     - вопросительные предложения;
     - последние 1–2 meaningful sentences;
     - короткие эмоционально насыщенные фразы.
4. Select 2–4 fragments within `max_summary_chars`.
5. Build `summary` as extractive compressed text, not invented interpretation.
6. Build `important_quote` as one short quote, preferably from user message.
7. If no good fragments found, use first meaningful sentence + last meaningful sentence.
8. Mark:
   - `was_truncated=True`;
   - `summary_method="deterministic_extractive_v1"`.

### 3.3. Safety

Micro-summary must not invent diagnosis.

Bad:

```text
Пользователь в травматической реакции из-за детской раны.
```

Good:

```text
Пользователь пишет, что сил почти нет, и просит короткую поддержку без давления.
```

### 3.4. Char limits

Рекомендуемые defaults:

```python
RECENT_FULL_TURN_MAX_CHARS = 1200
MICRO_SUMMARY_MAX_CHARS = 360
IMPORTANT_QUOTE_MAX_CHARS = 180
```

Можно настроить иначе, если существующий prompt budget требует другого.

## Task 4 — Добавить Context Assembler

Добавить функцию/класс:

```python
assemble_context_package_v1(
    current_user_message: str,
    recent_turns: list[Any],
    memory_bundle: MemoryBundle | None,
    thread_state: ThreadState | None,
    *,
    max_context_chars: int = 8000,
    recent_full_turn_limit: int = 6,
    recent_full_turn_max_chars: int = 1200,
    micro_summary_max_chars: int = 360,
) -> ContextAssemblyPackage
```

### 4.1. Приоритеты budget

Context package должен собираться в таком порядке:

1. `current_user_message` — всегда полностью, никогда не drop.
2. `pattern_core` + `active_frame` — если есть.
3. Последние 3–6 recent turns:
   - короткие/средние — full;
   - длинные — micro-summary.
4. `personal_history_context` / `semantic_memory_hits`.
5. `knowledge_rag_hits`.

Если budget заканчивается:

- сначала урезать/ограничивать knowledge hits;
- затем semantic hits;
- затем older summarized turns;
- recent full turns сохранять максимально;
- current user message не трогать.

### 4.2. Recent turns full

Последние turn’ы должны передаваться полностью, если:

```text
raw_chars <= RECENT_FULL_TURN_MAX_CHARS
```

и если они помещаются в budget.

### 4.3. Recent turns summarized

Если turn длиннее threshold:

```text
raw_chars > RECENT_FULL_TURN_MAX_CHARS
```

то в Writer идёт `TurnMicroSummary`, а не обрезанный raw text.

### 4.4. Dropped turns

Если turn не помещается даже как summary, он может быть dropped, но trace обязан показать:

```text
dropped_count
reason: "budget_limit"
```

### 4.5. Не ломать raw storage

Context Assembler работает только на уровне prompt/context package.

Он не должен перезаписывать raw history.

## Task 5 — Simple de-duplication

Добавить простую de-duplication v1.

Проблема:

```text
один и тот же обмен может попасть в recent history и semantic memory hits
```

### 5.1. Dedup keys

Использовать один или несколько признаков:

```text
turn_id / message_id
normalized text hash
source + timestamp
```

Если нет id, использовать normalized text hash:

```python
normalize_for_dedup(text):
    lower
    strip
    collapse whitespace
    remove trivial punctuation variance
```

### 5.2. Правило приоритета

Если дубль найден:

```text
recent_turns_full / recent_turns_summarized выигрывают над semantic_memory_hits
semantic_memory_hits выигрывают над knowledge_rag_hits
```

### 5.3. Trace

`context_assembly_trace_v1` должен включать:

```json
{
  "duplicates_removed": 2,
  "reasons": [
    "removed semantic duplicate already present in recent_turns_full"
  ]
}
```

Не записывать raw duplicate content в trace.

## 9. Acceptance criteria

PRD-045.6 считается выполненным, если:

1. Создан `ContextAssemblyPackage` или эквивалентный contract.
2. Есть deterministic extractive `TurnMicroSummary`.
3. Raw history не перезаписывается и не теряется.
4. Recent short/medium turns идут в Writer полностью.
5. Long turns идут как structured micro-summary, а не silent `content[:N] + "..."`.
6. Есть context budget и приоритеты drop.
7. Есть simple de-duplication.
8. Личные/семантические memory hits и knowledge RAG хотя бы минимально разведены по naming.
9. WriterContract использует context package.
10. Debug содержит `context_assembly_trace_v1`.
11. Baseline runner показывает context assembly summary.
12. Targeted tests проходят.
13. Direct baseline smoke по `QB-002`, `QB-005`, `QB-010`, `QB-015` выполнен.
14. Созданы safe logs:
    - `test_command_output.txt`;
    - `sanitized_runtime_logs.txt`.
15. Implementation report обновлён **после финального commit/push** и содержит commit hash.
16. Изменения закоммичены и запушены в `main`.
