# PRD-045.7 — Diagnostic Center Contract / WriterContract v2

## 0. Краткое резюме

**Цель PRD-045.7:** добавить первый контрактный слой Diagnostic Center перед Writer, но без полноценной методологической диагностики и без LLM-классификатора.

После PRD-045.6 / 045.6.1 / 045.6.2 в системе уже есть:

```text
State Analyzer
Thread Manager
Pattern Core / Active Frame
Context Assembly Package
Context Assembly Trace
Semantic Continuity Guard
```

Теперь нужно собрать эти сигналы в один компактный, наблюдаемый объект:

```text
DiagnosticCard v1
```

Он должен стать рабочей «карточкой ситуации» для Writer:

```text
Что сейчас происходит с пользователем?
Какая линия диалога продолжается?
Какая текущая потребность?
Какой безопасный следующий ход?
Чего Writer должен избегать?
На какую часть context package нужно опереться?
```

Главный принцип:

```text
Diagnostic Center Contract сейчас НЕ диагностирует человека.
Он собирает уже существующие сигналы в ясную рабочую карточку для Writer.
```

Это маленький инфраструктурный шаг перед настоящим Diagnostic Center v1 в PRD-046.1.

---

## 1. Почему нужно

Сейчас Writer получает много отдельных полей:

```text
state_snapshot
thread_state
pattern_core
active_frame
memory_bundle
context_package
context_assembly_trace
quality_trace
thread_diagnostics
```

Это уже лучше, чем было раньше, но проблема остаётся:

```text
Writer видит набор сигналов, а не собранное рабочее понимание ситуации.
```

Пример:

- State Analyzer говорит: `intent=explore`, `nervous_state=window`.
- Thread Manager говорит: `relation=continue`, `phase=explore`.
- Active Frame говорит: `current_need=explore perspective carefully`.
- Context Assembly говорит: `recent_full=4`, `summarized=0`.
- Semantic continuity говорит: `active_frame_semantic_continuity`.

Writer должен сам склеить эти сигналы в ответ. Иногда он справляется, иногда уходит в слишком длинный разбор, слишком быстрое углубление или не держит главный микро-ход.

Нужен слой:

```text
Diagnostic Card
```

Он не заменяет Writer, но даёт Writer короткую сцену:

```text
"Пользователь продолжает прежнюю линию избегания.
Состояние стабильное.
Нужно мягко отразить повторяющийся паттерн и предложить один безопасный микро-шаг.
Не давить, не ставить диагноз, не превращать ответ в лекцию."
```

## 2. Цель

### 2.1. Основная цель

Добавить deterministic `DiagnosticCard v1`, который:

1. Собирает данные из:
   - `StateSnapshot`;
   - `ThreadState`;
   - `ContextAssemblyPackage`;
   - `thread_diagnostics`;
   - `context_assembly_trace`.

2. Формирует компактные поля:
   - `situation_label`;
   - `user_state_summary`;
   - `thread_line_summary`;
   - `current_need`;
   - `suggested_writer_move`;
   - `avoid_list`;
   - `evidence_refs`;
   - `confidence`;
   - `risk_flags`.

3. Передаётся в `WriterContract`.

4. Видим в debug / baseline summary.

5. Не меняет State Analyzer / Thread Manager / Context Assembly behavior.

6. Не добавляет LLM diagnostic reasoning.

7. Не вводит клинические диагнозы.

### 2.2. Цель по качеству

Writer должен получать не только context, но и рабочую ориентацию:

```text
Не просто "есть context package",
а "какой следующий безопасный ход здесь нужен".
```

### 2.3. Цель по baseline

Direct smoke cases:

```text
QB-002
QB-005
QB-010
QB-015
```

должны показывать:

```text

diagnostic_card.version = diagnostic_card_v1
suggested_writer_move соответствует response_mode/current_need
risk_flags не содержат ложных high-risk флагов
```

## Task 2 — Add DiagnosticCard contract

Добавить contract в стиле проекта, например dataclass:

```python
@dataclass
class DiagnosticEvidenceRef:
    source: str
    key: str
    value_preview: str | None = None


@dataclass
class DiagnosticCardTrace:
    version: str
    builder: str
    rules_applied: list[str]
    risk_flags: list[str]
    evidence_sources: list[str]


@dataclass
class DiagnosticCard:
    version: str
    situation_label: str
    user_state_summary: str
    thread_line_summary: str
    current_need: str
    suggested_writer_move: str
    avoid_list: list[str]
    evidence_refs: list[DiagnosticEvidenceRef]
    confidence: float
    risk_flags: list[str]
    trace: DiagnosticCardTrace
```

Можно адаптировать названия под стиль проекта, но смысл сохранить.

### 2.1. Required fields

`version`:

```text
diagnostic_card_v1
```

`situation_label` — короткая machine-readable метка:

```text
low_resource_contact
concrete_next_step
semantic_continuation
new_exploration
safety_override
generic_support
```

`user_state_summary` — короткая безопасная формулировка:

```text
hypo/contact: user has low resource and asks for contact
window/solution: user asks for one concrete next step
window/explore: user continues previous exploration line
```

`thread_line_summary` — опора на thread:

```text
new thread started
continuing active thread
semantic continuity preserved by active_frame guard
low-resource continuation preserved
```

`current_need` — из `active_frame.current_need`, fallback по response_mode.

`suggested_writer_move` — deterministic suggestion:

```text
validate_briefly
offer_one_micro_step
reflect_pattern_once
regulate_first
safe_override
clarify_one_point
```

`avoid_list` — объединение:
- `thread_state.must_avoid`;
- safety-derived avoid rules;
- response-mode derived avoid rules.

`evidence_refs` — ссылки на безопасные источники без raw private text:
- `state.intent`;
- `state.nervous_state`;
- `thread.relation_reason`;
- `thread.phase_reason`;
- `thread.pattern_core_present`;
- `context.recent_full_count`;
- `context.summarized_count`.

`confidence` — deterministic approximate float:
- high: 0.8–0.95, если state/thread/context согласованы;
- medium: 0.55–0.75, если есть смешанные сигналы;
- low: 0.3–0.5, если fallback.

`risk_flags` — безопасные флаги:
- `low_resource`;
- `safety_active`;
- `continuity_low_confidence`;
- `new_thread_after_continuation_cue`;
- `context_sparse`;
- `avoid_over_exploration`;
- `avoid_directive_advice`.

### 2.2. Serialization

Все структуры должны иметь:

```text
to_dict/from_dict
```

или Pydantic equivalent.

Debug/baseline должны работать без custom encoder errors.

## Task 3 — Add deterministic Diagnostic Center v1 builder

Добавить файл:

```text
bot_psychologist/bot_agent/multiagent/diagnostic_center.py
```

С функцией:

```python
build_diagnostic_card_v1(
    *,
    state_snapshot: StateSnapshot,
    thread_state: ThreadState,
    context_package: ContextAssemblyPackage | None,
    thread_diagnostics: dict[str, Any] | None = None,
) -> DiagnosticCard:
    ...
```

### 3.1. No LLM

Никаких API calls.

### 3.2. Situation label rules

Рекомендуемые правила:

```python
if state_snapshot.safety_flag or thread_state.safety_active:
    situation_label = "safety_override"

elif state_snapshot.nervous_state in {"hypo", "hyper"} and state_snapshot.intent == "contact":
    situation_label = "low_resource_contact"

elif thread_state.response_mode == "practice" or state_snapshot.intent == "solution":
    situation_label = "concrete_next_step"

elif thread_state.relation_to_thread == "continue" and relation_reason == "active_frame_semantic_continuity":
    situation_label = "semantic_continuation"

elif thread_state.relation_to_thread == "continue":
    situation_label = "thread_continuation"

elif thread_state.relation_to_thread == "new_thread":
    situation_label = "new_exploration"

else:
    situation_label = "generic_support"
```

### 3.3. Suggested writer move rules

```python
if safety:
    suggested_writer_move = "safe_override"

elif nervous_state in {"hypo", "hyper"}:
    suggested_writer_move = "regulate_first" or "validate_briefly"

elif response_mode == "practice":
    suggested_writer_move = "offer_one_micro_step"

elif relation_reason == "active_frame_semantic_continuity":
    suggested_writer_move = "reflect_pattern_once"

elif response_mode == "reflect":
    suggested_writer_move = "clarify_one_point"

elif response_mode == "explore":
    suggested_writer_move = "explore_carefully"

else:
    suggested_writer_move = "validate_briefly"
```

### 3.4. Avoid list rules

Base avoid list:

```text
do_not_give_directive_life_advice
do_not_diagnose
do_not_over_explain
ask_at_most_one_question
```

If low-resource:

```text
avoid_deep_analysis
avoid_long_explanations
avoid_multiple_steps
```

If solution/practice:

```text
avoid_abstract_theory
offer_only_one_next_step
```

If semantic continuation:

```text
avoid_restarting_context
avoid_generic_advice
name_the_repetition_gently
```

If safety:

```text
avoid_exploration
prioritize_safety_protocol
```

### 3.5. Evidence refs

Evidence refs must be safe. Example:

```json
[
  {"source": "state", "key": "nervous_state", "value_preview": "window"},
  {"source": "state", "key": "intent", "value_preview": "explore"},
  {"source": "thread", "key": "relation_reason", "value_preview": "active_frame_semantic_continuity"},
  {"source": "context", "key": "recent_full_count", "value_preview": "4"}
]
```

Do not include raw user message.

## 14. Что дальше

После PRD-045.7 следующий вероятный шаг:

```text
PRD-045.8 — Validator Quality Flags v1
```

Почему:

```text
Когда Writer начнёт получать diagnostic card,
Validator должен научиться проверять не только safety/format,
но и соответствие ответа diagnostic card:
- не слишком общий;
- не потерял линию;
- не слишком длинный;
- не давит;
- не задаёт слишком много вопросов;
- не ушёл в лекцию вместо suggested_writer_move.
```

Настоящий методологический Diagnostic Center с линзами из “Кузницы Духа” и Neo MindBot материалов лучше оставить на PRD-046.1, после contract layer, validator quality flags и retrieval governance.
