# PRD-046.0.10 — BotDB Legacy SD Cleanup / Documentation Alignment v1

## 0. Назначение PRD

Этот PRD закрывает два технических долга в `Bot_data_base` перед переходом к Diagnostic Center:

1. Убрать остатки старого SD-слоя из активного ingestion/runtime пути.
2. Обновить внутреннюю документацию `Bot_data_base`, чтобы она соответствовала текущей архитектуре после `PRD-046.0.7.2-HF4`.

Это **не PRD на Diagnostic Center** и не PRD на новую функциональность базы знаний. Это cleanup / documentation / readiness шаг.

## 2. Почему нужен этот PRD

### 2.1. SD — старый слой

SD / Spiral Dynamics была ранним слоем смысловой разметки. Сейчас основной runtime уже опирается не на SD, а на:

```text
governance
chunk_type
allowed_use
safety_flags
metadata.llm_enrichment
retrieval policy
Writer KB policy
Diagnostic Card / Writer Move Compliance
```

Остатки SD не должны быть authority для будущей системы.

### 2.2. Главный риск

SD не является главным runtime-блокером для поиска, но может мешать ingestion/reprocess:

```text
новая загрузка материала
→ PipelineRunner
→ старый SDLabeler
→ лишние LLM-вызовы
→ задержки, API-расходы, дополнительная точка отказа
```

Нужно сделать так, чтобы старый SD-слой:

```text
не вызывался по умолчанию;
не требовал OpenAI API;
не влиял на ingestion;
не считался active semantic layer;
не мешал новым источникам;
оставался только как legacy/backward-compatible metadata, если уже есть в старых JSON.
```

### 2.3. Документация Bot_data_base отстаёт

`Bot_data_base/README.md` сейчас выглядит как минимальный стартовый файл, но проект уже сильно изменился:

- port фактически `8003`;
- требуется Python 3.10 для `Bot_data_base`;
- Chroma/reindex/recovery gate уже есть;
- source hygiene есть;
- focus source protected;
- enrichment/review/apply pipeline уже есть;
- legacy SD не должен быть описан как активная часть системы;
- нужны внутренние docs по аналогии с `bot_psychologist/docs`.

## 3. Scope

### Входит в PRD

1. Audit всех SD-остатков в `Bot_data_base`.
2. Отключение SD из активного ingestion пути.
3. Перевод SD в legacy/deprecated режим.
4. Обновление конфигурации.
5. Обновление README и создание `Bot_data_base/docs`.
6. Тесты, подтверждающие, что SD больше не вызывает LLM и не участвует в retrieval/runtime.
7. Сохранение backward compatibility для старых `sd_level` / `sd_distribution` данных.
8. Обновление living docs: `PROJECT_STATE.md`, `ROADMAP.md`, `PRD_INDEX.md`, при необходимости `DECISIONS.md`.

### Не входит в PRD

1. Не запускать Diagnostic Center.
2. Не менять production KB content.
3. Не менять `all_blocks_merged.json`.
4. Не делать reprocess `Кузницы`.
5. Не делать Chroma reindex.
6. Не удалять насильно legacy `sd_level` / `sd_distribution` из уже существующих артефактов.
7. Не менять governance authority поля:
   - `chunk_type`
   - `allowed_use`
   - `safety_flags`
   - `source_id`
   - `block_id`
   - `text`
8. Не вызывать LLM provider.

### 4.1. Добавить audit tool

Добавить инструмент:

```text

```

Отчёт:

```text
TO_DO_LIST/reports/PRD-046.0.10_LEGACY_SD_USAGE_AUDIT.md
```

### 4.2. Отключить SD из ingestion по умолчанию

Найти все места, где ingestion/reprocess вызывает старую разметку SD, например:

```python
self.sd_labeler.label_blocks(blocks)
```

И изменить поведение:

```text
по умолчанию SD не вызывается;
LLM provider не вызывается;
новые блоки не получают active sd_level как обязательное поле;
pipeline не падает, если OPENAI_API_KEY отсутствует только из-за SD;
SD может быть включён только явным legacy флагом.
```

Рекомендуемая логика:

```yaml
legacy_sd_labeling:
  enabled: false
```

или, если сохранён старый ключ:

```yaml
sd_labeling:
  enabled: false
  deprecated: true
```

Если старый ключ остаётся для compatibility, в коде нужно явно проверять:

```text
enabled == true AND explicit_legacy_mode == true
```

Иначе SDLabeler не вызывается.

### 4.3. Не ломать старые данные

Если в старых registry/blocks уже есть:

```text
sd_level
sd_distribution
complexity
```

Не удалять их насильно.

Но в API/документации пометить их как:

```text
legacy metadata
not used by current retrieval/runtime authority
not used by Diagnostic Center
```

### 4.4. API compatibility

Если в API есть старый фильтр `sd_level`, поведение должно быть безопасным:

```text
sd_level accepted for backward compatibility
sd_level ignored by retrieval
response/debug/warning: legacy sd_level ignored
```

Тест должен подтверждать, что SD-фильтр не влияет на retrieval results.

### 4.5. Dashboard / Registry

Dashboard не должен визуально подавать SD как актуальный смысловой слой.

Допустимые варианты:

1. Не показывать SD вообще в основных карточках.
2. Показывать только в техническом legacy/debug разделе.
3. Переименовать в `legacy_sd_distribution`.

Не использовать SD в readiness score.
