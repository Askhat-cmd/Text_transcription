# PRD-046.0.1 — Bot_data_base Integration / Admin Upload Governance Adapter v1

## 0. Краткое резюме

**Цель PRD-046.0.1:** связать уже созданный `knowledge_governance` слой из PRD-046.0 с реальной рабочей системой базы знаний проекта:

```text
C:\My_practice\Text_transcription\Bot_data_base
```

`Bot_data_base` должен остаться **главным ingestion/storage/Web Admin сервисом** для базы знаний бота.  
`knowledge_governance` не должен стать параллельной системой рядом с ним.

Главная задача этого PRD:

```text
Bot_data_base upload / processing / export / query
    +
knowledge_governance metadata
    =
единый управляемый путь подготовки базы знаний
```

После выполнения PRD-046.0.1 новый материал, загруженный через Web Admin/API, должен проходить через backward-compatible governance adapter и получать безопасную машиночитаемую разметку:

```text
chunk_type
allowed_use
safety_flags
lens_family
practice_metadata
governance_notes
chunking_quality
```

При этом:

```text
Writer / DiagnosticCenter / live runtime filtering пока НЕ подключаются к этим знаниям напрямую.
```

Runtime filtering будет отдельным PRD-046.0.2.

## 1. Архитектурный контекст

### 1.1. Что уже есть

В проекте уже есть два слоя, которые нужно соединить.

#### A. Реальный сервис базы знаний

```text
Bot_data_base/
```

Сейчас он отвечает за:

```text

```

Ключевой поток:

```text

    ↓

    ↓
BookChunker / SemanticChunker
    ↓
SDLabeler
    ↓

    ↓

    ↓

    ↓

    ↓

```

#### B. Новый governance foundation из PRD-046.0

```text
bot_psychologist/bot_agent/knowledge_governance/
```

Он уже добавил:

```text

```

Но PRD-046.0 был только foundation:

```text
source documents → governed chunks → metadata → safety/use policy → dry-run reports
```

Он не подключал governance к реальному `Bot_data_base` upload/query path.

### 1.2. Главный риск

Нельзя допустить две параллельные системы:

```text
1. Bot_data_base — реальная база, Web Admin, Chroma, registry, query.
2. knowledge_governance — отдельный красивый слой подготовки chunks.
```

Правильная архитектура:

```text
Bot_data_base остаётся главным телом базы знаний.
knowledge_governance становится governance-мозгом подготовки и разметки.
Web Admin/API запускают pipeline, который добавляет governance metadata.
bot_psychologist потом сможет фильтровать retrieved chunks по governance.
```

### 1.3. Важное ограничение

`Bot_data_base` сейчас сырой в части подготовки материала:

```text
- chunking в основном token/chapter-based;
- практики могут резаться неидеально;
- heading_path не всегда сохраняется;
- governance metadata отсутствует;
- Chroma metadata сейчас flat и не хранит policy fields;
- /api/query/ не отдаёт allowed_use / safety_flags;
- bot_psychologist DBApiClient не принимает governance metadata.
```

Этот PRD не должен пытаться идеально решить всю нарезку.  
Он должен:

```text
1. встроить governance pass-through;
2. сохранить backward compatibility;
3. добавить минимальный governance adapter;
4. сделать audit качества chunking;
5. подготовить основу для следующего PRD по runtime filtering и будущего улучшения chunker.
```

## 2. Основная цель

Добавить в `Bot_data_base` интеграционный слой:

```text
Admin Upload Governance Adapter v1
```

который:

1. После текущего chunking/normalizing добавляет governance metadata к каждому `UniversalBlock`.
2. Сохраняет governance в JSON export.
3. Сохраняет безопасные flat governance поля в Chroma metadata.
4. Возвращает governance summary через `/api/query/`.
5. Позволяет `bot_psychologist` принять governance fields через `DBApiClient`.
6. Создаёт chunking/governance audit report.
7. Не меняет user-facing ответы и не подключает governed chunks к Writer без фильтра.

## Task 3 — Add governance adapter

Создать:

```text
Bot_data_base/governance/governance_adapter.py
```

Основная функция:

```python
def apply_governance_to_blocks_v1(
    *,
    blocks: list[UniversalBlock],
    source_id: str,

    source_type: str,
    source_kind: str | None = None,
    governance_profile: str | None = None,
    default_allowed_use: list[str] | None = None,
    default_safety_flags: list[str] | None = None,
) -> list[UniversalBlock]:
    ...
```

### 3.1. Import strategy

Preferred:

```text
Reuse bot_psychologist/bot_agent/knowledge_governance classifiers/contracts where feasible.
```

But do not create brittle runtime dependency. If import path is unstable:

```text
- create local Bot_data_base adapter with the same output schema;
- document why;
- add schema compatibility test.
```

### 3.2. Governance output schema

Every block should receive:

```python
block.governance = {
    "schema_version": "governance_v1",
    "chunk_type": "...",
    "allowed_use": [...],
    "safety_flags": [...],
    "lens_family": [...],
    "tags": [...],
    "practice_metadata": {...},
    "governance_notes": [...],
    "source_trace": {
        "source_id": "...",
        "source_title": "...",
        "source_type": "...",
        "source_kind": "...",
        "chunk_index": 0,
        "adapter": "bot_data_base_governance_adapter_v1",
    },
}
```

### 3.3. Minimal deterministic classification

Do not use LLM.

Rules can be simple, but must be safe.

#### Practice detection

If title/text contains:

```text
практика
упражнение
техника
шаг
цель:
время:
минут
дыхание
заземление
```

then:

```text
chunk_type = practice
allowed_use includes practice_suggestion
practice_metadata extracted if possible
```

If practice is long / journaling / deep analysis:

```text
low_resource_safe = false
safety_flags includes practice_requires_low_resource_check
```

If practice is short body-based breathing/grounding:

```text
low_resource_safe = true
lens_family includes somatic
```

#### Lens/theory detection

If text contains:

```text
паттерн
программа
автоматизм
слепая зона
защита
избегание
страх
идентичность
самоценность
```

then:

```text
chunk_type = lens
allowed_use includes diagnostic_lens
safety_flags includes requires_grounding
```

If strong statements are present:

```text
safety_flags includes not_for_direct_quote
```

#### Architecture/style detection

If source_kind/profile is `architecture_notes` or source title contains `Neo MindBot`:

```text
allowed_use includes internal_only
allowed_use includes style_guidance
safety_flags includes source_style_not_user_facing
safety_flags includes not_for_direct_quote
```

#### Safety/protocol detection

If text contains:

```text
кризис
суицид
самоповреждение
не заменяет специалиста
экстренная помощь
безопасность
```

then:

```text
chunk_type = safety or protocol
allowed_use includes safety_protocol
```

### 3.4. Governance profiles

Support simple profiles:

```text
general_book
practice_manual
architecture_notes
transcript
```

Defaults:

#### `practice_manual`

```text
allowed_use: diagnostic_lens, writer_context, practice_suggestion
default_safety_flags: requires_grounding, not_for_direct_quote
```

#### `architecture_notes`

```text
allowed_use: internal_only, style_guidance, safety_protocol, diagnostic_lens
default_safety_flags: source_style_not_user_facing, not_for_direct_quote
```

#### `general_book`

```text
allowed_use: writer_context
default_safety_flags: not_for_direct_quote
```

#### `transcript`

```text
allowed_use: writer_context, diagnostic_lens
default_safety_flags: requires_grounding, not_for_direct_quote
```
