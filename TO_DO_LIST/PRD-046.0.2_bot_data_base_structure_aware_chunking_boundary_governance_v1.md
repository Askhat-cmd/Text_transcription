# PRD-046.0.2 — Bot_data_base Structure-Aware Chunking / Boundary Governance v1

## 0. Краткое резюме

**Цель PRD-046.0.2:** улучшить качество подготовки материалов в `Bot_data_base` до уровня, при котором practices / lenses / safety / architecture notes не растворяются в общих `theory` chunks.

После PRD-046.0.1 governance уже проходит через реальный путь:

```text
Web Admin / API upload
↓
Bot_data_base PipelineRunner
↓
chunking
↓
sd_labeling
↓
normalizing
↓
governance adapter
↓
JSON export + Chroma indexing
↓
/api/query governance summary
↓
bot_psychologist DBApiClient pass-through
```

Но аудит PRD-046.0.1 показал главный дефект:

```text
Synthetic practice_manual source → 3 blocks → chunk_type distribution: theory=3
```

То есть pipeline уже умеет хранить governance, но текущая нарезка остаётся token/chapter-driven и может смешивать практики, линзы, safety и теорию в один chunk. Если прямо сейчас включить runtime filtering, бот будет фильтровать плохо нарезанные блоки.

Главный принцип этого PRD:

```text
Сначала — качественные границы chunks.
Потом — runtime filtering.
Потом — Diagnostic Center / Lens Selection.
```

Этот PRD не делает полноценный Diagnostic Center и не подключает знания в Writer. Он делает `Bot_data_base` более пригодным для будущей управляемой базы знаний.

---

## 1. Почему нужно

`Bot_data_base` уже является главным ingestion/storage/Web Admin путём базы знаний. Новые материалы должны добавляться через Web Admin и проходить подготовку внутри этой системы.

Но текущий `BookChunker` работает в основном так:

```text
parse chapter headings
↓
split by paragraphs / token budget
↓
overlap tail
```

Это нормально для общего RAG, но недостаточно для материалов типа:

- `КУЗНИЦА ДУХА`;
- методические конспекты;
- практики;
- safety-протоколы;
- architecture notes;
- будущие материалы с линзами и упражнениями.

Для таких источников важны не только размер и similarity, а **семантическая роль блока**:

```text
practice ≠ theory
safety ≠ quote
diagnostic lens ≠ user-facing instruction
architecture notes ≠ direct answer text
```

Сейчас есть риски:

1. Практика может быть разрезана посередине шагов.
2. Теория и практика могут попасть в один chunk.
3. Safety/disclaimer может смешаться с обычным текстом.
4. `heading_path` не хранится как structured hierarchy.
5. `governance_adapter` классифицирует уже готовый block, но не видит нормальных section boundaries.
6. Runtime filtering будет работать формально, но некачественно.

Для концепции Neo MindBot это критично: база должна быть библиотекой линз, практик и формулировок, а не случайным набором цитат. Бот должен помогать видеть механизм, а не транслировать книгу.

## 2. Цель

### 2.1. Основная цель

Добавить в `Bot_data_base` structure-aware chunking для book/practice_manual/architecture_notes sources:

1. Парсить markdown-like heading hierarchy:
   - `#`
   - `##`
   - `###`
   - `Глава ...`
   - numbered headings
   - explicit practice/safety/lens headings.
2. Сохранять `heading_path` в `UniversalBlock`.
3. Разделять section units до token-budget chunking.
4. Не смешивать разные section roles в одном chunk, если это можно избежать.
5. Сохранять practices целиком, если они не превышают max budget.
6. Если practice слишком длинная — split только по step boundaries, а не по случайным paragraphs.
7. Улучшить governance adapter так, чтобы он использовал:
   - `heading_path`;
   - `chapter_title`;
   - explicit section role;
   - block title;
   - content markers.
8. Расширить `chunking_quality`:
   - `heading_path_present`;
   - `section_role`;
   - `mixed_intent_risk`;
   - `boundary_confidence`;
   - `split_reason`;
   - `practice_steps_preserved`.
9. Обновить JSON export / Chroma metadata / query summary минимально и backward-compatible.
10. Добавить smoke, который доказывает:
    - synthetic practice section становится `practice`, не `theory`;
    - safety section становится `safety`;
    - architecture notes становятся `style/internal_only`;
    - mixed theory/practice не превращается в один слепой `theory` chunk.

### 2.2. Quality target

До PRD-046.0.2 было:

```text
synthetic practice_manual: theory=3
```

После PRD должно быть примерно:

```text
synthetic practice_manual:
- practice >= 1
- lens >= 1
- safety >= 1 where safety markers exist
- theory may exist, but not 100% if source contains explicit practice/lens/safety sections
```

### 2.3. Architectural target

`Bot_data_base` должен стать не просто “splitter + embeddings”, а подготовителем управляемых knowledge units:

```text
source document
↓
structured sections
↓
boundary-aware chunks
↓
governance metadata
↓
quality trace
↓
query-visible governance summary
```

## Task 4 — Make BookChunker structure-aware

Update:

```text
Bot_data_base/chunkers/book_chunker.py
```

### 4.1. New internal flow

For markdown-like text, flow should become:

```text
clean_text
↓
parse_markdown_like_sections_v1
↓
for each StructuredSection:
    chunk section using role-aware rules
↓
make UniversalBlock with heading_path / section_role_hint / boundary trace
```

### 4.2. Preserve existing behavior as fallback

If structure parser returns weak sections or errors:

```text
fallback to current chapter/token behavior
```

Do not break non-markdown books.

### 4.3. Role-aware splitting

Rules:

#### Practice sections

If section role is `practice`:

- keep full practice in one block if <= max_tokens;
- if > max_tokens, split by:
  - `Шаг 1`, `Шаг 2`, ...
  - numbered/bulleted steps;
  - subheadings inside practice;
- do not split after `Цель:` without `Шаги`;
- mark:
  - `split_reason="practice_preserved"` or `split_reason="practice_step_split"`;
  - `practice_steps_preserved=True` in chunking quality.

#### Safety sections

If `safety`:

- keep compact;
- do not merge with theory;
- `split_reason="safety_boundary"`.

#### Lens sections

If `lens`:

- keep lens explanation with its key examples if reasonable;
- do not merge with following practice unless same section explicitly includes practice;
- `split_reason="lens_boundary"`.

#### Architecture sections

If `architecture`:

- keep as internal/style chunk;
- do not merge with user-facing practices;
- `split_reason="architecture_boundary"`.

#### Theory sections

Can still use current token/paragraph budget.

### 4.4. Parent section id

Create deterministic `parent_section_id`, e.g.:

```text
<source_id>::section::<index>::<short_hash>
```

or inside BookChunker if source_id not available yet:

```text
<section_index>::<short_hash>
```

## Task 5 — Improve governance adapter with structure hints

Update:

```text
Bot_data_base/governance/governance_adapter.py
```

Current adapter uses only title/text. Extend it to use:

```python
block.heading_path
block.section_role_hint
block.boundary_confidence
block.chapter_title
```

### 5.1. Role precedence

Rules:

```text
section_role_hint=safety       -> chunk_type=safety
section_role_hint=practice     -> chunk_type=practice
section_role_hint=architecture -> chunk_type=style or principle depending profile
section_role_hint=lens         -> chunk_type=lens
```

Only override role hint if very strong contrary evidence exists.

### 5.2. Practice metadata extraction

Improve `_extract_practice_metadata`:

Extract where possible:

```json
{
  "goal": "...",
  "duration": "...",
  "steps_count": 0,
  "requires_journaling": false,
  "body_based": false,
  "low_resource_safe": false,
  "channel": "body|thinking|action|relationship|mixed|unknown"
}
```

Rules:

- `дых`, `тело`, `зазем`, `ощущ` -> channel `body`
- `запиши`, `дневник`, `ответь письменно` -> channel `thinking`
- `сделай`, `отправь`, `выбери`, `начни` -> channel `action`
- if body-based and <= 3 steps and duration <= 10 min -> `low_resource_safe=true`
- journaling / deep analysis -> `low_resource_safe=false`

### 5.3. Architecture notes

For `architecture_notes`:

- default `allowed_use` must include `internal_only`, `style_guidance`, optionally `diagnostic_lens`;
- must include `source_style_not_user_facing`, `not_for_direct_quote`;
- must not include `practice_suggestion` unless explicit structured practice section exists.

### 5.4. Mixed intent

If a chunk contains multiple strong role markers:

```text
practice + safety
lens + practice
architecture + user-facing practice
```

Add safety/quality note:

```text
mixed_intent_risk
```

Do not fail ingestion, but make it visible.

## 6. Acceptance criteria

PRD is accepted if:

1. `Bot_data_base` has deterministic structure parser.
2. `BookChunker` uses structure-aware section units for markdown-like sources.
3. Existing plain book chunking remains backward-compatible.
4. `UniversalBlock` stores `heading_path`, `section_role_hint`, `boundary_confidence`, `split_reason`, `parent_section_id`.
5. Practice sections are not randomly split when within max token budget.
6. Long practice sections split by step/subsection boundaries.
7. Safety sections are not merged with ordinary theory when identifiable.
8. Architecture notes are marked internal/style and not user-facing by default.
9. Governance adapter uses section role hints.
10. `chunking_quality` exposes boundary quality and mixed intent risk.
11. JSON export preserves structure/governance/chunking quality.
12. Chroma metadata includes safe flat structure fields.
13. Query API can return governance/chunking summary.
14. `bot_psychologist` pass-through remains compatible.
15. Synthetic smoke no longer returns all `theory` for explicit practice/lens/safety source.
16. Bot_data_base tests pass.
17. bot_psychologist multiagent regression passes.
18. Writer/DiagnosticCard/runtime filtering are unchanged.
19. Reports and sanitized logs are created.
20. Commit and push to `main` completed.
21. Implementation report updated after push.
