# PRD-046.0.3-HF1 — Live Runtime Restore / LLM Client Compatibility + BotDB Health Guard

## 0. Статус и приоритет

**Тип:** hotfix PRD  
**Приоритет:** critical / blocker  
**Причина:** живой Web UI отвечает fallback-фразами вместо полноценного ответа. Перед `PRD-046.0.3.1 — Debug Knowledge Redaction / Safe Trace Guard v1` нужно восстановить live runtime.

Этот PRD временно останавливает feature-движение по Knowledge Base / Diagnostic Center и чинит текущую работоспособность бота.

---

## 1. Наблюдения из live-проверки

Пользователь запустил Web UI и отправил:

1. `привет`
2. `Что такое самореализация? И помогает ли в этом самоконтроль или лучше быть осознанным?`
3. `меня зовут Асхат`

В UI бот ответил fallback-фразами:

- `Я слышу тебя. Расскажи больше, если хочешь.`
- `Я слышу тебя.`
- `Сделай медленный вдох. Я рядом.`

По скриншотам видно:

- pipeline визуально получает `OK`, но ответ не содержательный;
- `LLM response | gpt-5-mini` пустой / `—`;
- `tokens` не отображаются;
- Memory stage занимает почти всё время запроса;
- в логах присутствуют ошибки LLM client и проблемы Bot_data_base retrieval.

---

## 2. Root cause audit

### 2.1. Критическая причина №1 — LLM client вызывает Responses API, но установленный SDK его не поддерживает

Из `bot.log`:

```text
[WRITER] write failed: 'AsyncOpenAI' object has no attribute 'responses'
AttributeError: 'AsyncOpenAI' object has no attribute 'responses'
```

Та же ошибка есть в `STATE_ANALYZER`:

```text
[STATE_ANALYZER] llm fallback failed: 'AsyncOpenAI' object has no attribute 'responses'
```

Текущий `agent_llm_client.py` делает model-family routing:

- для моделей, которые поддерживают custom temperature → `client.chat.completions.create(...)`;
- для reasoning / gpt-5 family → `client.responses.create(...)`.

Но live environment имеет `AsyncOpenAI`, у которого нет `.responses`.

Результат:

```text
Writer не получает LLM output → падает → UI показывает fallback canned answer.
```

Это главный blocker.

### 2.2. Критическая причина №2 — Bot_data_base API отвечает 503 ChromaDB unavailable

Из `retrieval.log` и `bot.log`:

```text
[RETRIEVAL] API fallback: kind=http_status status=503 message=HTTP 503 from Bot_data_base: {"detail":"ChromaDB unavailable"}
```

После этого включается fallback:

```text
[RETRIEVAL] semantic fallback found 4 blocks
```

То есть retrieval не полностью сломан, но каждый запрос ждёт API timeout/retry и только потом падает в local semantic fallback. Это даёт большую задержку Memory stage.

### 2.3. Non-blocker observations

Эти факты не являются главным blocker:

- `GET /api/debug/session/.../multiagent-trace | status=404` до появления session trace — ожидаемо для ранней polling-фазы UI.
- `identity.legacy_user_id_used` — важно для будущей чистки identity/session, но не причина fallback-ответов.

---

## 3. Цель PRD

Восстановить live runtime так, чтобы:

```text
пользовательский запрос
  → StateAnalyzer не падает на LLM client
  → Writer не падает на LLM client
  → retrieval не ждёт каждый раз медленный BotDB 503
  → Web UI получает содержательный ответ, а не fallback
```

---

1. Делать `PRD-046.0.3.1` debug redaction.
2. Делать Diagnostic Center v1.
3. Менять Writer prompt по смыслу.
4. Менять governance policy rules, если это не нужно для восстановления live runtime.

9. Скрывать ошибки за “OK”, если LLM реально не отработал.

## 5. Workstream A — LLM Client Compatibility Guard

### 5.1. Файл

Обновить:

```text
bot_psychologist/bot_agent/multiagent/agents/agent_llm_client.py
```

### 5.2. Проблема

Сейчас код напрямую вызывает:

```python
response = await client.responses.create(**request_kwargs)
```

Но live SDK не имеет `client.responses`.

### 5.3. Требуемое поведение

Добавить runtime capability detection:

```python
has_responses = hasattr(client, "responses") and hasattr(client.responses, "create")
has_chat = hasattr(client, "chat") and hasattr(client.chat, "completions")
```

Если policy route выбрал Responses API, но `client.responses` отсутствует:

1. Не падать с `AttributeError`.
2. Переключиться на compatibility path через `chat.completions.create(...)`.
3. Сохранить корректные параметры для gpt-5/reasoning моделей:
   - не передавать `temperature`, если модель его не поддерживает;
   - использовать правильный token param (`max_completion_tokens` для gpt-5/o* family, `max_tokens` для обычных моделей);
   - если `response_format` нужен для JSON и chat completions его поддерживает — передать;
   - если `response_format` недоступен в fake/test client — тест должен покрыть fallback без зависимости от реального API.
4. Вернуть `AgentLLMResult.api_mode`:

```text
responses
chat_completions
chat_completions_compat
```

`chat_completions_compat` должен означать: модель хотела Responses API, но SDK/runtime умеет только Chat Completions.

### 5.4. Startup diagnostics

Добавить безопасный лог при первом LLM вызове или при startup:

```text
[AGENT_LLM] sdk_capabilities has_responses=false has_chat_completions=true model=gpt-5-mini selected_mode=chat_completions_compat
```

Не логировать API key.

### 5.5. Error handling

Если нет ни Responses API, ни Chat Completions:

- выбросить понятную ошибку `LLMClientCapabilityError` или `RuntimeError` с текстом:

```text
OpenAI client has neither responses.create nor chat.completions.create
```

Но эта ошибка должна быть отдельной и тестируемой, а не `AttributeError`.

## 7. Workstream C — Bot_data_base 503 circuit breaker / fast local fallback

### 7.1. Файлы

Обновить:

```text
bot_psychologist/bot_agent/retriever.py
bot_psychologist/bot_agent/db_api_client.py
bot_psychologist/bot_agent/config.py
```

### 7.2. Проблема

При `KNOWLEDGE_SOURCE=api|chromadb` retriever каждый запрос делает HTTP call к Bot_data_base, ждёт timeout/retry, получает `503 ChromaDB unavailable`, и только затем использует local semantic fallback.

Это делает Memory stage очень долгим.

### 7.3. Требуемое поведение

Добавить lightweight circuit breaker для Bot_data_base API retrieval.

Минимальный контракт:

```python
self._bot_db_circuit_open_until: float = 0.0
self._bot_db_last_error_kind: str = ""
self._bot_db_last_error_message: str = ""
```

Если Bot_data_base вернул:

- `connect`
- `timeout`
- `http_status=503`
- body содержит `ChromaDB unavailable`

то открыть circuit на TTL:

```text
BOT_DB_CIRCUIT_BREAKER_TTL_SECONDS = 60 default
```

Пока circuit открыт:

- не вызывать Bot_data_base API на каждый пользовательский запрос;
- сразу идти в semantic fallback / TF-IDF fallback;
- логировать один короткий warning, не спамить traceback.

Пример лога:

```text
[RETRIEVAL] BotDB circuit open: skip api for 60s reason=http_503_chromadb_unavailable → semantic fallback
```

### 7.4. Config

Добавить env-config:

```text
BOT_DB_CIRCUIT_BREAKER_ENABLED=True
BOT_DB_CIRCUIT_BREAKER_TTL_SECONDS=60
BOT_DB_FAST_FAIL_ON_503=True
```

Не менять existing defaults так, чтобы production path сломался.

### 7.5. Debug trace

Добавить в retriever-level debug/log или memory debug summary безопасные поля:

```json
{
  "retrieval_source_attempted": "api",
  "retrieval_source_used": "semantic_fallback",
  "bot_db_circuit_open": true,
  "bot_db_last_error_kind": "http_status",
  "bot_db_last_status_code": 503,
  "bot_db_last_error_class": "ChromaDB unavailable"
}
```

Если полноценный debug plumbing слишком большой — достаточно логов + smoke report, но не менять Writer prompt.

## 12. Acceptance criteria

PRD is accepted only if all are true:

1. Live Writer no longer fails with `AsyncOpenAI has no attribute responses`.
2. StateAnalyzer no longer logs the same LLM client AttributeError.
3. `agent_llm_client.py` supports both Responses API and Chat Completions compatibility path.
4. `writer_api_mode` correctly reports actual path.
5. UI no longer returns canned fallback for normal prompts when API key/model are valid.
6. BotDB 503 no longer causes repeated long wait on every request.
7. Local fallback retrieval remains available.
8. No feature movement into Diagnostic Center / debug redaction PRD.
9. Existing multiagent regression passes.
10. Implementation report is synced after push.
