# PRD-046.0.3.1 — Debug Knowledge Redaction / Safe Trace Guard v1

## 0. Краткое резюме

**Цель:** закрыть безопасностный хвост после PRD-046.0.3: runtime governance уже фильтрует knowledge chunks перед Writer, но debug-поля всё ещё могут содержать raw content исходных retrieved hits.

PRD-046.0.3 корректно сделал:

```text
retrieved hits → knowledge policy → sanitized writer-facing knowledge_rag_hits → Context Assembly / Writer
```

Но в `MultiAgentOrchestrator` debug сейчас формирует `semantic_hits_detail` из `memory_bundle.semantic_hits` и кладёт туда `content_full`. Эти hits идут **до writer-facing policy filtering**, поэтому туда потенциально может попасть raw content из:

- `internal_only`;
- `source_style_not_user_facing`;
- `not_for_direct_quote`;
- `do_not_use`;
- high mixed-intent chunks;
- architecture notes / internal methodology.

Перед настоящими source dry-runs и Diagnostic Center это нужно исправить.

Главный принцип:

```text
Policy trace/debug must explain decisions without exposing raw governed content.
```

---

## 1. Почему это важно

После PRD-046.0.3 Writer уже не получает raw internal-only content. Это хорошо.

Но debug payload может быть использован:

- в baseline reports;
- в smoke logs;
- в UI/админке;
- в trace-отчётах;
- при анализе ошибок;
- как artifact для агента IDE или архитектора.

Если debug содержит `content_full`, то получается обход governance:

```text
Writer защищён,
но debug всё ещё может раскрыть то, что Writer видеть не должен.
```

Это особенно рискованно перед обработкой материалов:

- `КУЗНИЦА ДУХА`;
- `Neo MindBot КОНСПЕКТ`;
- будущих внутренних архитектурных/методологических заметок.

---

## 2. Цель PRD

Добавить безопасную redaction-политику для knowledge debug:

```text
raw SemanticHit content
    ↓
safe debug serializer
    ↓
content_preview only / sanitized preview / no content_full
    ↓
knowledge_policy_trace without raw text
```

PRD должен гарантировать:

1. Raw internal-only / do_not_use / source_style_not_user_facing content не попадает в orchestrator debug.
2. `content_full` для retrieved knowledge hits удалён или всегда redacted.
3. Debug сохраняет полезность: chunk_id, source, score, governance summary, policy action, reasons, risk flags.
4. Existing Writer behavior не меняется.
5. Existing policy filtering не меняется по смыслу.

## 5. Required behavior

### 5.1. Remove raw `content_full` from knowledge debug

Current risky pattern in `orchestrator.py`:

```python
semantic_hits_detail.append({
    ...
    "content_preview": content_full[:200],
    "content_full": content_full,
})
```

Change this so debug never exposes full retrieved knowledge hit content.

Recommended output shape:

```python
{
    "chunk_id": "...",
    "source": "...",
    "score": 0.91,
    "content_preview": "...",       # max 120 chars, sanitized
    "content_len": 1234,
    "content_redacted": True,
    "governance_summary": {...},
    "policy_action": "...",
    "policy_reasons": [...],
    "risk_flags": [...],
}
```

Do **not** include:

```text
content_full
raw_content
full_text
unredacted_content
```

### 5.2. Use policy decisions when possible

Best option: build debug details from `knowledge_policy_trace.decisions` + writer-facing `knowledge_rag_hits`, not from raw `semantic_hits`.

If using `memory_bundle.semantic_hits` remains necessary for count/debug, then apply strict redaction.

### 5.3. Safe preview rules

Create deterministic helper:

```python
safe_knowledge_debug_preview_v1(text: str, governance: dict, max_chars: int = 120) -> str
```

Rules:

- normalize whitespace;
- max 120 chars;
- if `internal_only` or `source_style_not_user_facing` or `do_not_use`, return empty string;
- if `not_for_direct_quote`, return max 80 chars;
- never return text inside quote marks as a quote;
- never include content if policy action is `drop` or `internal_only`.

### 5.4. Policy trace remains raw-free

`knowledge_policy_trace_v1` should continue to include only:

```text
chunk_id
source
score
action
reasons
risk_flags
```

No raw content.

Add explicit test to prevent future regressions.

### 5.5. Debug should still be useful

The debug payload should preserve:

- counts;
- which chunk was included/dropped/internal;
- why;
- risk flags;
- whether content was redacted;
- sanitized preview only where allowed.

## 6. Integration details

### 6.1. In `KnowledgePolicyDecision`

Optional but recommended:

Add method:

```python
def to_safe_debug_dict(self) -> dict[str, Any]:
    ...
```

It must not include `sanitized_content` if action is not `include_writer_context`.

Example:

```python
{
    "chunk_id": self.chunk_id,
    "source": self.source,
    "score": self.score,
    "action": self.action,
    "allowed_for_writer": self.allowed_for_writer,
    "allowed_for_diagnostic": self.allowed_for_diagnostic,
    "allowed_for_practice": self.allowed_for_practice,
    "reasons": self.reasons,
    "risk_flags": self.risk_flags,
    "sanitized_content_len": len(self.sanitized_content or ""),
}
```

### 6.2. In `orchestrator.py`

Replace `semantic_hits_detail` raw construction.

Minimum acceptable:

```python
semantic_hits_detail = []
for hit in memory_bundle.semantic_hits:
    ...
    semantic_hits_detail.append({
        "chunk_id": ...,
        "source": ...,
        "score": ...,
        "content_preview": safe_preview,
        "content_len": len(content_full),
        "content_redacted": True,
        "governance_summary": safe_governance_summary,
    })
```

Preferred:

```python
knowledge_debug_detail = build_safe_knowledge_debug_detail_v1(
    semantic_hits=memory_bundle.semantic_hits,
    knowledge_policy_trace=memory_bundle.knowledge_policy_trace,
    knowledge_rag_hits=memory_bundle.knowledge_rag_hits,
)
```

Then debug contains:

```python
"semantic_hits_detail": knowledge_debug_detail
```

or, better but backward-compatible enough:

```python
"semantic_hits_detail": knowledge_debug_detail,
"semantic_hits_raw_redacted": True,
```

### 6.3. Avoid raw leakage in sanitized logs

Update `sanitized_runtime_logs.txt` generation for this PRD to prove no raw internal marker appears.

Use synthetic marker:

```text
INTERNAL_ONLY_RAW_TEXT_SHOULD_NOT_LEAK_046031
```

This marker must not appear in:

- orchestrator debug JSON;
- sanitized runtime logs;
- smoke report body except in a statement like `marker leaked: no`.

Do not include the marker in a way that makes the test ambiguous. In test code, the marker can exist as test input, but assert it is not present in serialized debug output.

5. Writer-facing knowledge remains sanitized by PRD-046.0.3 policy.
6. No Writer prompt / DiagnosticCard behavior changes.
7. Targeted redaction tests pass.
8. Existing knowledge policy tests pass.
