# PRD-046.0.4.1 — BotDB Chroma Recovery / Governed Index Consistency Gate v1

## 0. Status

**Type:** infrastructure + quality gate PRD  
**Priority:** high / blocker before retrieval evaluation and Diagnostic Center  
**Scope:** Bot_data_base Chroma health, index consistency, and governance readiness gate  
**Runtime stance:** do not improve Writer, do not change DiagnosticCard, do not change response style.

This PRD follows `PRD-046.0.4 — Real Source Knowledge Base Quality & Chroma Readiness Audit v1`.

The audit showed two simultaneous problems:

1. **Chroma query path is unhealthy**.
2. **The current real source export appears stale / not governance-ready**: `КУЗНИЦА ДУХА` has 229 chunks, but all 229 lack summaries, allowed_use, safety_flags, lens_family, and boundary_confidence.

The goal is not to “make Chroma green at any cost.”  
The goal is to restore Chroma safely **without allowing stale ungoverned chunks to become production-quality retrieval context**.

## 2. Architectural decision

Do **not** simply reset/reindex Chroma and call it fixed.

A healthy Chroma collection containing stale ungoverned chunks would be dangerous because:

```text
BotDB /api/query becomes available
→ bot_psychologist uses API retrieval path
→ chunks have no governance metadata
→ runtime policy treats them as legacy_no_governance
→ Writer may receive generic writer_context from material that should be internal/no-citation/practice-gated
```

Therefore, this PRD must implement a **governed index consistency gate**:

```text
Chroma can be repaired technically,
but BotDB query path must be marked production-ready only if indexed blocks pass minimal governance readiness thresholds.
```

1. Diagnose and fix the Chroma embedding/index consistency path.

## 6. Workstream A — Fix embedding model consistency

### 6.1. Problem

`Bot_data_base/config.yaml` declares:

```yaml
embedding:
  model: intfloat/multilingual-e5-large
```

But `ChromaManager._init_embedding_model()` currently reads:

```python
os.getenv("SENTENCE_TRANSFORMERS_MODEL", "sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2")
```

This can create mismatches between:

```text
config.yaml expectation
runtime env
existing Chroma collection dimension
new embeddings dimension
test dummy embeddings
```

### 6.2. Required change

Update `ChromaManager` so the embedding model name is explicit and comes from the pipeline config.

Recommended constructor:

```python
class ChromaManager:
    def __init__(self, db_path: str, collection_name: str, embedding_model_name: str | None = None) -> None:
        ...
        self.embedding_model_name = embedding_model_name or os.getenv(
            "SENTENCE_TRANSFORMERS_MODEL",
            "sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2",
        )
```

Update `PipelineRunner`:

```python
self.chroma_manager = ChromaManager(
    self._resolve_path(self.config["storage"]["chroma_db_path"]),
    self.config["storage"]["collection_name"],
    embedding_model_name=self.config.get("embedding", {}).get("model"),
)
```

### 6.3. Test

Add/extend a test proving:

- config embedding model is passed to `ChromaManager`;
- env fallback still works if config missing;
- dummy embeddings mode still works for tests.

## 8. Workstream C — Safe Chroma reset/reindex CLI

Create:

```text
Bot_data_base/tools/chroma_recovery.py
```

CLI should support:

```bash
python Bot_data_base/tools/chroma_recovery.py --probe-only
python Bot_data_base/tools/chroma_recovery.py --dry-run
python Bot_data_base/tools/chroma_recovery.py --reset --confirm
python Bot_data_base/tools/chroma_recovery.py --reindex-from-json --confirm
python Bot_data_base/tools/chroma_recovery.py --reset --reindex-from-json --confirm
```

### 8.1. Safety rules

- Default mode must be safe / non-mutating.
- Any destructive reset requires `--confirm`.
- If `--confirm` missing, tool must exit with a clear message and no mutation.
- Tool must write JSON probe artifact to logs.
- Tool must not log full chunk content.
- Tool must report whether indexed blocks are governance-ready.

### 8.2. Reindex source

Use:

```text
Bot_data_base/data/processed/all_blocks_merged.json
```

or allow explicit:

```bash
--blocks-path C:\...\all_blocks_merged.json
```

But reindexing from JSON must not imply quality acceptance.

## 9. Workstream D — Governed Index Gate

Add a deterministic readiness gate.

### 9.1. Gate inputs

For blocks being indexed, calculate:

```text
total_blocks
governance_present_count
allowed_use_present_count
safety_flags_present_count
lens_family_present_count
summary_present_count
boundary_confidence_present_count
not_for_direct_quote_count
chunk_type_distribution
```

### 9.2. Minimum thresholds v1

For `КУЗНИЦА ДУХА`, current thresholds should be intentionally strict but not impossible:

```text
governance_present_ratio >= 0.95
allowed_use_present_ratio >= 0.95
safety_flags_present_ratio >= 0.95
not_for_direct_quote_ratio >= 0.95
summary_present_ratio >= 0.60
boundary_confidence_present_ratio >= 0.80
```

`lens_family_present_ratio` can be lower for v1, but must be reported:

```text
lens_family_present_ratio >= 0.50 desired, not blocker in this PRD
```

### 9.3. Gate output

```json
{
  "version": "governed_index_gate_v1",
  "status": "ready | not_ready | degraded",
  "blocker_reasons": [
    "allowed_use_missing",
    "safety_flags_missing",
    "summary_missing",
    "boundary_confidence_missing"
  ],
  "metrics": {...},
  "recommended_next_prd": "PRD-046.0.4.2 — Source Reprocess / Governance Backfill v1"
}
```

### 9.4. Important behavior

If Chroma becomes technically healthy but governance gate fails, report:

```text
Chroma technical health: restored
Governed index readiness: not_ready
Runtime recommendation: keep fallback/circuit or treat hits as legacy until source reprocess/backfill.
```

Do not silently mark the knowledge base as production-ready.

## 13. Acceptance criteria

PRD is accepted only if:

1. `ChromaManager` uses the embedding model from config or explicitly reports fallback.
2. Chroma probe no longer crashes with ambiguous errors like `object of type 'int' has no len()`.
3. Safe reset/reindex CLI exists and is non-mutating by default.
4. Destructive reset requires `--confirm`.
5. Reindex from JSON can run in dry-run mode.
6. Governance readiness gate exists.
7. Current `КУЗНИЦА` stale export is not falsely marked production-ready if governance fields are missing.
8. `/api/query/` health is probed and reported.
9. Reports do not include raw full chunk content.
10. Existing Bot_data_base and multiagent tests pass.
11. Implementation report is synced after push.

## 15. Important architectural note

Quality is more important than making the status indicator green.

A green `/api/query/` with ungoverned stale chunks is not success.  
Success is:

```text
Chroma technically healthy
+ indexed blocks consistent with local processed blocks
+ governance readiness status explicit
+ unsafe/stale knowledge not silently promoted into runtime.
```
