# PRD-046.0.4.2.1 — Practice Classification Precision / Manual Review Reduction v1

## 0. Статус и приоритет

**Тип:** data quality / deterministic classification calibration PRD  
**Приоритет:** высокий, блокер перед Chroma production reindex  
**Область:** `Bot_data_base`, deterministic backfill/classification, manual review queue  
**Язык документа:** русский  
**Runtime-позиция:** не менять Writer, DiagnosticCard, Thread Manager, State Analyzer, runtime policy и стиль ответа.

Этот PRD следует после:

```text
PRD-046.0.4.2 — Source Reprocess / Governance Backfill v1
```

`PRD-046.0.4.2` успешно сделал deterministic backfill и формально провёл базу через governed index gate:

```text
governance_present_ratio = 1.0
allowed_use_present_ratio = 1.0
safety_flags_present_ratio = 1.0
not_for_direct_quote_ratio = 1.0
boundary_confidence_present_ratio = 1.0
summary_present_ratio = 1.0
lens_family_present_ratio = 0.7424
gate_after = ready
```

Но audit/reports показали качественный хвост:

```text
manual_review_items = 172
p0_items = 91
practice chunks = 113
practice_suggestion = 113
practice_requires_low_resource_check = 113
```

В sample manual review queue видны ложные `practice`-классификации:

```text
- Q&A / диалоговые фрагменты
- “Из сессии” case fragments
- теоретические объяснения
- разделители "***"
- авторские/стилевые фрагменты
- фрагменты вида “Теория без практики...”
```

Это значит: gate стал зелёным технически, но перед Chroma reindex нужно повысить precision `chunk_type=practice`, иначе retrieval начнёт возвращать слишком много “практик”, которые на самом деле являются theory/case/lens/style.

---

## 1. Главная цель

Цель PRD:

```text
снизить ложные practice-классификации,
отделить настоящие практики от theory/case/Q&A/lens/style,
уменьшить P0 manual review queue,
и сохранить governed gate в рабочем состоянии.
```

Это не PRD про новые знания и не PRD про LLM.

Это PRD про качество deterministic классификации после backfill.

---

## 2. Архитектурное решение

Не идти сразу в:

```text
PRD-046.0.4.3 — Chroma Reindex from Governed Blocks / API Query Restore
```

потому что Chroma начнёт индексировать governed, но семантически шумные `practice` chunks.

Нужно сначала сделать:

```text
classification precision pass
```

Особенно важно:

```text
practice_suggestion должен появляться только у chunks,
которые действительно являются самодостаточной практикой.
```

Не все фрагменты, где есть слова:

```text
Цель
Время
Шаг
Практика
```

должны автоматически становиться `practice`.

Например:

```text
Q&A про практику → case или lens
объяснение “почему практика важна” → lens/theory
разделитель "***" → style/noise/internal
из сессии → case
предупреждение о практике → safety/lens
```

## 6. Workstream B — Более строгий practice classifier

### 6.1. Настоящая практика должна иметь self-contained признаки

`chunk_type=practice` разрешён только если есть минимум 2 из 4 сильных признаков:

```text
1. goal_marker:
   - "Цель:"
   - "Задача:"
   - "Для чего:"
   - heading содержит "Практика" / "Упражнение"

2. duration_marker:
   - "Время:"
   - "минут"
   - "секунд"
   - "в течение дня"
   - "один раз как исследование"

3. instruction_marker:
   - повелительные глаголы: "возьми", "запиши", "сядь", "закрой", "попробуй", "заметь", "выбери", "назови", "остановись"
   - numbered steps
   - bullet steps

4. integration_marker:
   - "после выполнения"
   - "в конце"
   - "что изменилось"
   - "заметь"
   - "запиши вывод"
   - "смыкание"
```

И при этом не должно быть сильных override-маркеров case/Q&A/style/noise.

### 6.2. Практика без шагов

Если есть `goal_marker` + `duration_marker`, но нет инструкций:

```text
chunk_type не должен автоматически становиться practice.
```

Возможные решения:

```text
- chunk_type=lens
- allowed_use=["writer_context", "diagnostic_lens"]
- safety_flags += ["needs_review"]
- manual_review reason = "practice_candidate_missing_steps"
```

### 6.3. Длинные практики

Если практика длится:

```text
20–30 минут
30–40 минут
```

или содержит сложное погружение:

```text
тень
глаза закрыты
сильные эмоции
регрессия
травма
пустота
```

то она может оставаться `practice`, но:

```text
practice_metadata.low_resource_safe = false
safety_flags += ["practice_requires_low_resource_check"]
manual_review priority >= P1
```

Не нужно автоматически делать P0, если практика самодостаточная и безопасно ограничена.

## 7. Workstream C — Override rules: case / Q&A / theory / style должны побеждать practice

### 7.1. Case override

Если chunk содержит маркеры:

```text
"Из сессии"
"> **Имя,"
"клиент"
"пример:"
"история:"
"диалог:"
```

и нет явно отдельного блока инструкции, то:

```text
chunk_type = "case"
allowed_use = ["writer_context"] или ["internal_only"] по safety
practice_suggestion запрещён
manual_review reason = "case_fragment_not_practice" если раньше был practice
```

Если case содержит практику внутри, но текст в основном состоит из истории клиента:

```text
chunk_type = "case"
chunking_quality.mixed_intent_risk = true
mixed_intent_severity = "medium" или "high"
manual_review reason = "case_practice_mixed"
```

### 7.2. Q&A override

Если chunk содержит диалоговую структуру:

```text
"— Я ..."
"— Почему ..."
"— Как ..."
"Вопрос:"
"Ответ:"
"Q:"
"A:"
```

и не содержит самодостаточную инструкцию, то:

```text
chunk_type = "case" или "lens"
practice_suggestion запрещён
```

Правило выбора:

```text
Q&A объясняет механизм → lens
Q&A описывает конкретного человека/сессию → case
Q&A задаёт границы/риски → safety
```

### 7.3. Theory/lens override

Если chunk содержит:

```text
"три причины"
"почему"
"механизм"
"нейропластичность"
"кортизол"
"дофамин"
"это знание"
"программа держится"
"как это работает"
```

и нет явной инструкции к действию, то:

```text
chunk_type = "lens" или "theory"
practice_suggestion запрещён
```

### 7.4. Style/source override

Если chunk содержит:

```text
"НейроСталкинг"
"программа"
"духовн"
"спиральная динамика"
"оранжевый / зелёный"
"авторская методология"
```

и звучит как стиль/концепт источника, а не user-safe практика:

```text
chunk_type = "lens" или "style"
safety_flags += ["source_style_not_user_facing"]
practice_suggestion запрещён, кроме явно самодостаточной практики
```

### 7.5. Noise override

Если chunk text после очистки:

```text
- "***"
- "---"
- пустой
- только markdown-разделители
- только заголовок
```

то:

```text
chunk_type = "style" или "noise"
allowed_use = ["internal_only"]
safety_flags += ["do_not_use"] или ["source_style_not_user_facing"]
manual_review reason = "noise_chunk"
```

Если текущие enums не поддерживают `noise`, использовать:

```text
chunk_type = "style"
allowed_use = ["internal_only"]
safety_flags += ["do_not_use"]
```

## 8. Workstream D — Исправить allowed_use после переклассификации

После изменения `chunk_type` обязательно пересчитать:

```text
allowed_use
safety_flags
practice_metadata
summary
manual_review reasons
```

Правила:

```text
chunk_type=practice:
  allowed_use = ["practice_suggestion"]
  + writer_context только если практика self-contained и не high-risk

chunk_type=lens:
  allowed_use = ["writer_context", "diagnostic_lens"]

chunk_type=theory:
  allowed_use = ["writer_context"]

chunk_type=case:
  allowed_use = ["writer_context"] если sanitized, иначе ["internal_only"]

chunk_type=safety:
  allowed_use = ["internal_only", "safety_protocol"]

chunk_type=style:
  allowed_use = ["internal_only", "style_guidance"]
```

Жёсткое правило:

```text
practice_suggestion может быть только у chunk_type=practice.
```

## 9. Workstream E — Manual review queue reduction

### 9.1. Не цель: убрать все review items

Manual review queue должна остаться.  
Она нужна для качества и будущей Web Admin review.

Цель — убрать ложные P0 из-за плохой классификации.

### 9.2. New review reasons

Добавить/уточнить reasons:

```text
false_practice_candidate
practice_candidate_missing_steps
case_fragment_not_practice
qa_fragment_not_practice
theory_fragment_not_practice
noise_chunk
source_style_not_user_facing
long_practice_requires_review
practice_requires_low_resource_check
lens_family_low_confidence
summary_placeholder_low_quality
```

### 9.3. Priority recalibration

P0 должен оставаться только для реально критичных случаев:

```text
P0:
- unsafe practice
- practice_suggestion + incomplete/self-unsafe practice
- high mixed_intent
- clinical/safety risk
- do_not_use/noise accidentally writer-facing
- empty allowed_use/safety after calibration

P1:
- long but self-contained practice
- low lens confidence
- source_style_not_user_facing
- medium mixed_intent
- summary placeholder weak

P2:
- enrichment candidate
- optional style cleanup
```

### 9.4. Target reduction

Ожидаемый минимум:

```text
P0 manual review items должны снизиться минимум на 30% относительно 91.
```

То есть:

```text
P0 <= 63
```

Желательный результат:

```text
P0 <= 50
```

Если не удалось снизить P0, report должен объяснить почему.


## 18. Финальная архитектурная цель

После этого PRD база должна стать не только формально governed, но и менее шумной:

```text
меньше ложных practice chunks
меньше случайных practice_suggestion
меньше P0 review из-за practice_incomplete
лучше различение practice / lens / case / theory / style
без Chroma reindex до завершения калибровки
```

Только после этого можно безопасно перейти к Chroma reindex и retrieval eval.
