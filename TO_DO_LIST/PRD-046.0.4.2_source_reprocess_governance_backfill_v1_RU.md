# PRD-046.0.4.2 — Source Reprocess / Governance Backfill v1

## 0. Статус и приоритет

**Тип:** quality / data pipeline / deterministic backfill PRD  
**Приоритет:** высокий, блокер перед LLM enrichment, retrieval eval и Diagnostic Center  
**Область:** `Bot_data_base`, processed export `КУЗНИЦА ДУХА`, governance/backfill, quality gate  
**Runtime-позиция:** не менять Writer, DiagnosticCard, Thread Manager, State Analyzer и стиль ответа.

Этот PRD следует после:

```text
PRD-046.0.4 — Real Source KB Quality & Chroma Readiness Audit v1
PRD-046.0.4.1 — BotDB Chroma Recovery / Governed Index Consistency Gate v1
```

`PRD-046.0.4.1` показал:

```text
Chroma tooling: реализован
Chroma /api/query: всё ещё 503
Governed index gate: not_ready
all_blocks_merged.json: 229 chunks
governance_present_ratio: 0.0
allowed_use_present_ratio: 0.0
safety_flags_present_ratio: 0.0
not_for_direct_quote_ratio: 0.0
summary_present_ratio: 0.0
boundary_confidence_present_ratio: 0.0
lens_family_present_ratio: 0.0
```

Вывод: текущий processed export `КУЗНИЦЫ ДУХА` устарел или был создан до свежих governance/chunking PRD. Его нельзя индексировать в Chroma как production-ready knowledge base.

---

## 1. Главная цель

Цель PRD — не “сделать Chroma зелёным”.

Цель:

```text
пересобрать или backfill-нуть processed blocks КУЗНИЦЫ ДУХА так,
чтобы all_blocks_merged.json стал минимально governance-ready
и прошёл governed index gate хотя бы до статуса degraded/ready,
не используя LLM enrichment и не меняя runtime Writer.
```

То есть нужно восстановить качество данных до индексации:

```text
raw markdown
→ structure-aware chunking
→ governance adapter
→ allowed_use
→ safety_flags
→ not_for_direct_quote
→ heading_path / boundary_confidence
→ chunking_quality
→ deterministic summary placeholder
→ manual_review markers
→ all_blocks_merged.json
→ governed_index_gate
```

## 2. Архитектурный контекст

### 2.1. Почему это нужно

Если просто переиндексировать текущие 229 chunks в Chroma, бот начнёт получать из `/api/query` неразмеченный материал:

```text
allowed_use отсутствует
safety_flags отсутствуют
summary отсутствует
lens_family отсутствует
boundary_confidence отсутствует
not_for_direct_quote отсутствует
```

Даже при runtime policy это приведёт к деградации:

```text
- хорошие chunks могут быть отброшены как legacy_no_governance;
- опасные или source-style chunks могут выглядеть как обычный writer_context;
- Diagnostic Center позже не сможет выбрать линзы;
- retrieval eval будет тестировать мусорную базу;
- качество ответов Writer будет зависеть от случайных фрагментов.
```

### 2.2. Роль `КУЗНИЦА ДУХА`

`КУЗНИЦА ДУХА` — пока единственный основной knowledge source.

Использовать как:

```text
- внутренние диагностические линзы;
- практики;
- мягкие mirror-фразы;
- use_when / avoid_when материал;
- источники смысловых моделей;
- материал для будущего Diagnostic Center.
```

Не использовать как:

```text
- прямой источник цитат пользователю;
- книжную консультацию;
- духовный авторитет;
- готовые директивы;
- user-facing “как написано в Кузнице...”.
```

Поэтому для всех chunks `КУЗНИЦЫ` базовый safety flag должен включать:

```text
not_for_direct_quote
```

Если фрагмент звучит особенно авторским/стилистически директивным, дополнительно:

```text
source_style_not_user_facing
```

## 6. Workstream B — Source reprocess CLI

Создать CLI:

```text
Bot_data_base/tools/source_reprocess.py
```

Поддержать режимы:

```bash
python Bot_data_base/tools/source_reprocess.py --source "КУЗНИЦА ДУХА" --dry-run
python Bot_data_base/tools/source_reprocess.py --source "КУЗНИЦА ДУХА" --reprocess --confirm
```

### 6.1. Safety rules

- По умолчанию CLI не должен мутировать processed файлы.
- Любая запись в `data/processed` требует `--confirm`.
- Перед заменой `all_blocks_merged.json` создать backup:

```text
Bot_data_base/data/processed/backups/all_blocks_merged__before_PRD-046.0.4.2__YYYYMMDD_HHMMSS.json
```

- Отчёты не должны содержать full raw chunk text.
- В logs можно хранить safe preview до 160 символов.
- CLI должен явно писать:

```text
mutation_performed: true/false
backup_created: true/false
blocks_before
blocks_after
gate_before
gate_after
```

## 7. Workstream C — Deterministic governance backfill

Если полный reprocess невозможен или нестабилен, должен быть fallback path:

```text
backfill current 229 blocks deterministically
```

### 7.1. Минимальные поля, которые должны появиться

Для каждого block:

```text
metadata.governance.schema_version
metadata.governance.source_profile
metadata.governance.chunk_type
metadata.governance.allowed_use
metadata.governance.safety_flags
metadata.governance.lens_family
metadata.governance.practice_metadata
metadata.chunking_quality
metadata.heading_path
metadata.section_role_hint
metadata.boundary_confidence
metadata.split_reason
metadata.parent_section_id
summary
```

### 7.2. Базовый source profile

Для `КУЗНИЦА ДУХА` использовать профиль:

```text
source_profile = "practice_manual"
source_kind = "knowledge_source"
source_visibility = "internal_lens"
```

### 7.3. Safety defaults

Для всех chunks `КУЗНИЦЫ`:

```text
safety_flags включает "not_for_direct_quote"
```

Если chunk содержит выраженные авторские/духовные/директивные формулировки, добавить:

```text
source_style_not_user_facing
```

Если chunk содержит потенциально опасные советы, clinical/medical claims, кризисные темы или практики глубокой регрессии без safety boundaries, добавить:

```text
needs_review
```

Не использовать `do_not_use`, если нет явного критичного риска. `do_not_use` должен быть редким и обоснованным.

### 7.4. allowed_use rules

Минимальные deterministic rules:

```text
chunk_type=practice:
  allowed_use = ["practice_suggestion"]
  + possibly ["writer_context"] only if practice is self-contained and low-risk

chunk_type=lens:
  allowed_use = ["writer_context", "diagnostic_lens"]

chunk_type=theory:
  allowed_use = ["writer_context"]

chunk_type=safety:
  allowed_use = ["internal_reasoning"]

chunk_type=case:
  allowed_use = ["writer_context"] only if sanitized; otherwise internal_reasoning

chunk_type=style/architecture/internal:
  allowed_use = ["internal_reasoning"]
  safety_flags += ["source_style_not_user_facing"]
```

Важно:

```text
practice_suggestion нельзя давать theory/lens/case/style chunks.
```

### 7.5. chunk_type inference

Использовать существующие deterministic rules из governance adapter, но усилить для реального markdown.

Примерные маркеры:

```text
practice:
- "Практика"
- "Практикум"
- "Шаги"
- "Время:"
- "Цель:"
- numbered instructions
- "сделай", "запиши", "остановись", "попробуй"

lens:
- "механизм"
- "как это работает"
- "почему"
- "схема"
- "паттерн"
- "драйвер"
- "программа"
- explanatory conceptual section

safety:
- "важно"
- "если тяжело"
- "не форсируй"
- "сбавь темп"
- "не нужно"
- "если совсем тяжело"
- "обратись за помощью"
- boundaries

case:
- "Из сессии"
- named example
- dialogue quote blocks
- client story

theory:
- long explanation without steps
```

Не стремиться к идеальной классификации. Цель v1 — минимально надёжная deterministic разметка + manual review markers.

## 8. Workstream D — Deterministic summary placeholder

Поскольку LLM enrichment ещё не разрешён в этом PRD, нужен deterministic summary placeholder.

### 8.1. Требования

Для каждого chunk создать `summary`, если оно пустое.

Summary должен быть:

```text
- коротким;
- безопасным;
- не user-facing;
- без прямых цитат;
- без мистификации;
- годным для retrieval;
- не более 240 символов.
```

### 8.2. Простая формула v1

Пример:

```text
"{chunk_type}: {heading_path_tail}. Темы: {lens_family/tags}. Роль: {section_role_hint}. Использование: {allowed_use}."
```

Если lens_family пустой:

```text
"{chunk_type}: {heading_path_tail}. Роль: {section_role_hint}. Нужна ручная проверка темы."
```

### 8.3. Запрет

Не создавать красивый пересказ через LLM.  
Не писать summary от лица автора.  
Не копировать длинные предложения из source text.

## 10. Workstream F — Manual review queue v2

Обновить manual review queue после reprocess/backfill.

Формат JSON:

```json
{
  "version": "manual_review_queue_v2",
  "source": "КУЗНИЦА ДУХА",
  "items": [
    {
      "priority": "P0 | P1 | P2",
      "block_id": "...",
      "heading_path": ["..."],
      "chunk_type": "...",
      "reasons": [
        "summary_missing",
        "lens_family_low_confidence",
        "mixed_intent_high",
        "practice_incomplete",
        "source_style_not_user_facing",
        "boundary_confidence_low"
      ],
      "safe_preview": "до 160 символов",
      "recommended_action": "review_chunk_type | review_lens_family | review_practice_completeness | split_source_section"
    }
  ]
}
```

### 10.1. Priority rules

```text
P0:
- practice chunk without clear steps;
- high mixed_intent;
- safety risk;
- empty allowed_use/safety_flags after backfill;
- boundary_confidence very low;
- raw/source style likely to leak.

P1:
- lens_family low confidence;
- summary placeholder low quality;
- medium mixed_intent;
- long chunk.

P2:
- stylistic cleanup;
- optional future LLM enrichment.
```

## 16. Acceptance criteria

PRD принимается, если:

1. Есть `source_reprocess.py` или эквивалентный CLI.
2. По умолчанию CLI не мутирует данные.
3. Mutation требует `--confirm`.
4. Backup создаётся перед заменой processed export.
5. `all_blocks_merged.json` после reprocess/backfill содержит governance metadata.
6. `allowed_use` заполнен минимум у 95% chunks.
7. `safety_flags` заполнены минимум у 95% chunks.
8. `not_for_direct_quote` есть минимум у 95% chunks `КУЗНИЦЫ`.
9. `summary` заполнен минимум у 95% chunks хотя бы deterministic placeholder.
10. `boundary_confidence` или эквивалентный boundary marker есть минимум у 80% chunks.
11. `practice_suggestion` есть только у `chunk_type=practice`.
12. Manual review queue создан и не содержит raw full content.
13. Governed index gate показывает `degraded` или `ready`, либо если `not_ready`, причины стали существенно уже и понятнее.
14. Chroma production reset/reindex не выполнен без gate readiness.
15. Тесты проходят.
16. Implementation report обновлён после push.
17. В отчётах нет секретов, raw full chunks, `.env` или приватных runtime данных.

## 18. Важное правило качества

Не пытаться “нарисовать готовность” искусственно.

Плохой outcome, который нельзя принимать:

```text
Все chunks получили одинаковые пустые governance-поля,
gate стал формально зелёным,
но manual review и metrics показывают мусорную разметку.
```

Лучше честный `degraded` с понятными manual_review items, чем фальшивый `ready`.
