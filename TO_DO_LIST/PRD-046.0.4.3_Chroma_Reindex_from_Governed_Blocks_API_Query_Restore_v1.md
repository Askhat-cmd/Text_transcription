**Статус:** ready for IDE-agent implementation

**Тип:** controlled recovery + smoke/audit  
**Язык PRD:** русский

Этот PRD продолжает линию `PRD-046.0.4.x` и закрывает следующий слабый шов: Chroma/API retrieval path всё ещё не подтверждён как здоровый production-ready путь после governance backfill и practice classification precision.

Главный критерий успеха — не просто “Chroma зелёный”, а **Chroma/API возвращает governed knowledge hits с корректной metadata, а `bot_psychologist` может использовать API retrieval path без ухода в fallback/circuit breaker**.

## 1. Контекст

### 1.1. Архитектурная рамка

Проект сейчас находится в блоке подготовки базы знаний. Качество ответов бота зависит от цепочки:

```text
исходный материал
→ markdown-разметка
→ chunking
→ governance-классификация
→ summary / metadata
→ retrieval
→ runtime policy
→ Context Assembly
→ Diagnostic Card
→ Writer
```

Текущая цель — восстановить governed retrieval path, а не строить новые “умные” слои поверх нестабильной базы.

### 1.2. Роль источников

`КУЗНИЦА ДУХА v.2.md` — основной knowledge source на текущем этапе, но он используется как **внутренняя линза**, а не как текст для прямого цитирования пользователю.

`Neo MindBot КОНСПЕКТ.md` — internal architecture/style material, не обычный user-facing knowledge source.

Поэтому восстановленный `/api/query` должен отдавать governance metadata, чтобы downstream runtime policy мог отличать:

- `writer_context`;
- `diagnostic_lens`;
- `practice_suggestion`;
- `safety_protocol`;
- `style_guidance`;
- `internal_only`;
- `not_for_direct_quote`;

## 2. Root cause / почему нужно

После governance backfill и practice precision governed blocks готовы для индексации, но Chroma/API path ещё не восстановлен как основной путь retrieval.

Текущие риски:

1. **Бот продолжает жить через fallback/circuit breaker.**  
   Это скрывает проблему Chroma и не проверяет реальное качество governed retrieval.

2. **Можно случайно “позеленить” Chroma старым индексом.**  
   Если collection содержит stale/ungoverned chunks, `/api/query` технически заработает, но качество и safety ухудшатся.

3. **API может не отдавать governance metadata.**  
   Тогда `bot_psychologist` не сможет корректно применить runtime policy и отличить internal-only chunks от writer-facing chunks.

4. **Debug/report surfaces могут случайно сохранить raw full chunk text.**  
   Даже если runtime policy защищает Writer, отчёты и smoke logs не должны становиться обходом governance.

Поэтому нужен controlled reindex именно из governed `all_blocks_merged.json`, с preflight gate, safe reset/recreate collection, API smoke и bot-side smoke.
