# PRD-046.0.4 — Real Source Knowledge Base Quality & Chroma Readiness Audit v1

## 0. Краткое резюме

**Тип:** audit / quality checkpoint  
**Приоритет:** high  
**Статус:** готово для IDE-агента  
**Основной принцип:** качество ответов бота теперь напрямую зависит от качества базы знаний.

После `PRD-046.0.3.1` runtime governance и debug redaction закрыли важные safety-хвосты. Теперь можно безопасно перейти к реальному аудиту базы знаний.

Этот PRD не улучшает Writer, не строит Diagnostic Center и не меняет runtime-поведение. Его задача — честно измерить качество реальной БД и состояние Chroma, чтобы следующий PRD был точным, а не предположительным.

Главный источник на этом этапе:

```text
КУЗНИЦА ДУХА
```

Это пока единственный реальный источник базы знаний. Новые материалы добавляются позже, когда текущая база выйдет на устойчивое плато качества.

## 1. Контекст

Проект уже прошёл цепочку:

```text
PRD-046.0       — Knowledge Governance foundation
PRD-046.0.1     — Bot_data_base Integration / Admin Upload Governance Adapter
PRD-046.0.2     — Structure-Aware Chunking / Boundary Governance
PRD-046.0.2.1   — Boundary Calibration / Allowed-Use Precision Guard
PRD-046.0.3     — Runtime Governance Filtering / Retrieval Policy
PRD-046.0.3-HF1 — Live Runtime Restore / LLM Client Compatibility + BotDB Health Guard
PRD-046.0.3.1   — Debug Knowledge Redaction / Safe Trace Guard
```

Сейчас система уже умеет:

- пропускать governance metadata через `Bot_data_base`;
- хранить `governance`, `chunking_quality`, `heading_path`, `boundary_confidence`;
- применять runtime policy перед Writer;
- скрывать raw knowledge content из debug;
- работать в live UI даже при нездоровом BotDB Chroma через fallback.

Но всё ещё неизвестно:

- насколько качественно реально нарезалась `КУЗНИЦА ДУХА`;
- достаточно ли `lens_family`;
- не слишком ли много `theory`;
- корректно ли выделяются `practice`, `lens`, `safety`;
- есть ли плохие summary;
- где mixed-intent;
- где source markdown требует ручной подготовки;
- почему BotDB `/api/query/` всё ещё даёт `503 ChromaDB unavailable`;
- готова ли Chroma collection к нормальному retrieval.

## 2. Архитектурное решение

Следующий шаг — не Diagnostic Center и не blind Chroma fix.

Следующий шаг:

```text
Real Source Knowledge Base Quality & Chroma Readiness Audit
```

Причина:

```text
Если сначала просто починить Chroma, он начнёт быстрее находить chunks,
но мы всё ещё не будем знать, хорошие ли это chunks.

Если сейчас построить Diagnostic Center,
он будет выбирать линзы из неподтверждённой базы.
```

Поэтому PRD-046.0.4 должен дать измеримую картину:

```text
Что есть в базе?
Как оно размечено?
Как оно нарезано?
Что опасно?
Что не готово?
Что нужно чинить следующим PRD?
```

## 3. Scope

### 3.1. Основной источник

Аудитировать в первую очередь:

```text
КУЗНИЦА ДУХА
```

Если в `Bot_data_base` есть другие источники, их можно показать в inventory, но detailed quality audit делать только по `КУЗНИЦЕ`.

### 3.2. Neo MindBot КОНСПЕКТ

`Neo MindBot КОНСПЕКТ` не является обычным user-facing knowledge source.

В рамках этого PRD:

- не индексировать его как обычную базу для Writer;
- не смешивать его с `КУЗНИЦЕЙ`;
- если он уже есть в processed sources, пометить как `architecture/internal` risk item;
- указать в отчёте, что для него нужен отдельный governance profile:
  - `architecture_notes`;
  - `style_guidance`;
  - `internal_only`;
  - `source_style_not_user_facing`.

## 4. Non-goals

В рамках этого PRD нельзя:

1. Менять Writer prompt.
2. Менять DiagnosticCard.
3. Делать Diagnostic Center v1.
4. Менять runtime retrieval policy.
5. Делать LLM enrichment.
6. Делать Chroma reset/reindex.

8. Автоматически переписывать source markdown.
9. Массово исправлять chunks.

11. Скрывать плохие результаты аудита.
12. Цитировать или вставлять raw content из источников в отчёты сверх коротких safe previews.

## 5. User/product decisions to preserve

### 5.1. Источник

```text
КУЗНИЦА ДУХА — пока единственный основной источник.
```

Новые источники добавляются позже.

### 5.2. Термины автора

Термины вроде:

```text
программа
нейросталкинг
присутствие
```

не нужно специально использовать в ответах пользователю.

Но их не нужно и искусственно запрещать, если они уместны контексту.

В рамках аудита:

- посчитать такие source-specific terms;
- отметить, где они могут требовать neutral mapping;
- не переписывать их автоматически.

### 5.3. Принцип использования источника

Источник остаётся внутренней линзой:

```text
без прямого цитирования пользователю
```

В рамках аудита:

- проверить, что chunks имеют `not_for_direct_quote`, если source используется как внутренняя линза;
- отметить chunks, которые потенциально могут попасть в Writer слишком прямо;
- не менять runtime policy.

### 5.4. Ручной review

Сейчас не делать полноценный Admin Review Workflow.

Но можно и нужно создать:

```text
manual review queue
```

То есть audit report должен содержать список конкретных chunks/sections, которые стоит вручную проверить.

Не нужно пока делать UI для review.

### 5.5. Offline LLM enrichment

Offline LLM enrichment разрешён позже и может использовать API-затраты.

Важно:

```text
LLM enrichment делается при подготовке/загрузке материала,
а не на каждом пользовательском запросе.
```

В этом PRD LLM enrichment не выполнять. Только подготовить выводы, где он нужен.

## 6. Требуемые результаты

PRD должен создать read-only audit tooling и отчёты.

### 6.1. Source inventory

Найти и описать:

```text
Bot_data_base/data/processed/all_blocks_merged.json
Bot_data_base registry
processed book exports
Chroma collection metadata
```

Собрать:

- список sources;
- source_id;
- title;
- author;
- source_type;
- blocks count;
- registry status;
- export files;
- processed paths;
- является ли source `КУЗНИЦА ДУХА`;
- есть ли подозрительные architecture/internal sources в общей базе.

### 6.2. Chunk distribution metrics

Для `КУЗНИЦЫ` посчитать:

```text
total_chunks
chunk_type distribution:
- theory
- practice
- lens
- safety
- style
- architecture
- excluded
- unknown/empty

allowed_use distribution:
- writer_context
- diagnostic_lens
- practice_suggestion
- safety_protocol
- internal_only
- do_not_use

safety_flags distribution:
- not_for_direct_quote
- source_style_not_user_facing
- too_strong_claim
- spiritual_authority_risk
- clinical_risk
- legacy_no_governance
- other

lens_family distribution:
- shame
- guilt
- anger
- grief
- procrastination
- perfectionism
- evaluation_fear
- loneliness
- attachment
- self_criticism
- avoidance
- burnout
- low_resource
- unknown/empty
```

Не нужно ограничиваться этим списком: если в данных есть другие `lens_family`, показать их тоже.

### 6.3. Structure / boundary metrics

Посчитать:

```text
heading_path present / missing
boundary_confidence buckets:
- high >= 0.80
- medium 0.50–0.79
- low < 0.50
split_reason distribution
section_role_hint distribution
parent_section_id present / missing
```

Найти:

- sections without heading path;
- long chunks with low boundary confidence;
- practice chunks split into suspicious fragments;
- lens chunks without meaningful heading;
- safety chunks mixed with practice/theory.

### 6.4. Mixed-intent metrics

Посчитать:

```text
mixed_intent_risk true/false
mixed_intent_severity:
- none
- low
- medium
- high
mixed_intent_primary_role
secondary_role_markers
mixed_intent_reason
```

Создать список top mixed chunks для manual review:

```text
TO_DO_LIST/reports/PRD-046.0.4_MANUAL_REVIEW_QUEUE.md
```

### 6.5. Summary readiness audit

Проверить, насколько chunks готовы к retrieval по summary/metadata.

Если у blocks есть summary/title/metadata:

- пустые summary;
- summary слишком короткий;
- summary совпадает с первым декоративным предложением;
- summary слишком общий;
- summary не содержит смысловых маркеров;
- summary не помогает отличить practice/lens/theory.

Если summary как отдельного поля нет — зафиксировать это как gap.

Создать метрики:

```text
summary_missing_count
summary_too_short_count
summary_too_generic_count
summary_duplicates_count
summary_needs_llm_enrichment_count
```

Важно: не делать LLM enrichment сейчас. Только выявить необходимость.

### 6.6. Practice completeness audit

Для chunks с `chunk_type=practice` проверить эвристически:

- есть ли цель/назначение;
- есть ли шаги;
- есть ли длительность или формат выполнения;
- есть ли low-resource safety marker;
- есть ли contraindication / avoid_when / caution;
- practice не обрезана ли на середине;
- practice не смешана ли с теорией слишком сильно.

Создать список:

```text
practice_chunks_needing_review
```

### 6.7. Internal lens / no-citation audit

Поскольку источник используется как внутренняя линза, проверить:

- chunks без `not_for_direct_quote`;
- chunks с `writer_context`, но без `not_for_direct_quote`;
- chunks с author/source-specific language, которые могут попасть пользователю слишком прямо;
- chunks, где `allowed_use` слишком широкий;
- chunks, которые должны быть `diagnostic_lens`, но сейчас `theory`.

Это не исправлять автоматически. Только отметить.

### 6.8. Source markdown readiness audit

Если исходный markdown/text source доступен, проверить:

- есть ли H1/H2/H3;
- средняя длина section;
- есть ли слишком длинные sections без подзаголовков;
- есть ли слитые абзацы;
- есть ли оглавление, номера страниц, колонтитулы;
- есть ли markers вроде `практика`, `упражнение`, `вопрос`, `шаг`;
- есть ли sections, где practice/lens/safety можно разметить вручную.

Если исходный markdown недоступен, отчёт должен явно сказать:

```text
raw source markdown not found
```

и не считать это successful audit.

### 6.9. Chroma readiness audit

Проверить BotDB Chroma без мутаций.

Проверить:

```text
GET /api/status/
GET /api/registry/
POST /api/query/
```

Если сервис не запущен — зафиксировать.

Если `/api/query/` возвращает `503 ChromaDB unavailable`, собрать:

- endpoint;
- error body;
- Chroma db path from config;
- collection name;
- collection exists yes/no, если можно проверить;
- collection count, если можно проверить;
- local `all_blocks_merged.json` exists yes/no;
- parsed blocks count;
- registry count vs local blocks count vs Chroma count;
- embedding model configured for BotDB;
- possible embedding dimension mismatch;
- whether safe reset/reindex is needed.

Запрещено в этом PRD:

```text
reset Chroma
delete collection
reindex production collection
```

Можно только рекомендовать следующий PRD:

```text
PRD-046.0.4.1 — BotDB Chroma Recovery / Index Consistency v1
```

### 6.10. Retrieval probe snapshot

Сделать лёгкий read-only retrieval probe, не полноценный eval set.

Проверить 5–7 запросов:

```text
я злюсь на себя, потому что опять не сделал обещанное
я боюсь показать проект и снова хочу отложить
нет сил, не хочу анализа, просто поддержи
я всё время стараюсь быть лучше и не могу остановиться
я чувствую вину, когда выбираю себя
что делать, если я снова прокрастинирую
мне стыдно за то, что я ничего не сделал
```

Для каждого запроса собрать:

- source path used:
  - Chroma API;
  - local fallback;
  - TF-IDF;
  - semantic fallback;
- top chunk ids;
- top chunk types;
- lens_family;
- allowed_use;
- safety_flags;
- policy actions if available;
- whether result seems plausible by metadata.

Не нужно оценивать deep quality вручную. Это snapshot, а не full eval.

### 9.3. Manual review queue

Manual review queue must be practical, not huge.

Include top 30–50 items max, grouped by severity:

```text

```

Each item:

```text

heading_path

allowed_use
safety_flags
lens_family
boundary_confidence
mixed_intent_severity
issue_reason
recommended_manual_action
safe_preview max 160 chars
```

No raw long content.

### 9.4. Next PRD recommendation

The agent must recommend next PRD based on actual findings:

Possible next PRDs:

```text
PRD-046.0.4.1 — BotDB Chroma Recovery / Index Consistency v1
PRD-046.0.4.2 — Source Markdown Readiness / Manual Boundary Prep v1
PRD-046.0.5   — Offline LLM Summary + Lens Enrichment v1
PRD-046.0.6   — Knowledge Retrieval Eval Set v1
```

Decision rule:

- If `/api/query/` still 503 or Chroma count mismatch → recommend Chroma Recovery first.
- If markdown/source structure is very poor → recommend Source Markdown Readiness first.
- If chunks are structurally okay but summaries/lens_family weak → recommend Offline LLM Enrichment.
- If retrieval probe already returns plausible governed chunks → recommend Retrieval Eval Set.

## 12. Acceptance criteria

PRD accepted only if:

1. Real processed source inventory is produced.
2. `КУЗНИЦА ДУХА` is identified as focus source, or report clearly says not found.
3. `kb_quality_metrics.json` exists.
4. `KB_QUALITY_AUDIT_REPORT.md` exists and contains distributions for:
   - chunk_type;
   - allowed_use;
   - safety_flags;
   - lens_family;
   - boundary_confidence;
   - mixed_intent.
5. Summary readiness is audited.
6. Practice completeness is audited.
7. No-citation/internal-lens readiness is audited.
8. Manual review queue exists with prioritized candidates.
9. Chroma readiness report exists and explicitly states whether `/api/query/` works.
10. Chroma is not reset/reindexed in this PRD.
11. Retrieval probe snapshot exists.
12. Report does not leak long raw source text.
13. Tests for audit logic pass.
14. Existing key regressions pass.
15. Implementation report includes:
    - files changed;
    - commands run;
    - metrics summary;
    - next PRD recommendation;
    - commit hash and push status after push.

## 13. Definition of done

The PRD is done when architect can answer:

```text
1. Насколько готова КУЗНИЦА как база знаний?
2. Что именно плохо: markdown, chunking, governance, summary, lens_family, Chroma или retrieval?
3. Какие chunks надо вручную проверить первыми?
4. Можно ли чинить Chroma следующим?
5. Нужен ли LLM enrichment?
6. Можно ли после этого двигаться к Diagnostic Center?
```

If answers are vague, PRD is not done.
