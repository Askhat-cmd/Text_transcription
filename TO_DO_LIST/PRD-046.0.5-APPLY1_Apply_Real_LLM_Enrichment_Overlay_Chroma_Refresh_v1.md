# PRD-046.0.5-APPLY1 — Apply Real LLM Enrichment Overlay + Chroma Refresh v1

## 0. Статус документа

- **Проект:** Bot Psychologist / Neo MindBot
- **PRD:** `PRD-046.0.5-APPLY1`
- **Название:** Apply Real LLM Enrichment Overlay + Chroma Refresh v1
- **Язык:** русский
- **Тип:** маленький controlled apply / infrastructure update
- **Исполнитель:** IDE-агент
- **Архитектор:** ChatGPT / проектный архитектор
- **Цель:** безопасно применить real LLM enrichment overlay-кандидат после `PRD-046.0.5-RUN1-HF3`, обновить Chroma/API metadata и подтвердить, что runtime retrieval может видеть enrichment metadata без нарушения deterministic governance.

---

## 1. Контекст

Проект прошёл длинную цепочку подготовки Knowledge Base:

```text
PRD-046.0.4.x
→ governed blocks
→ Chroma/API restore
→ real LLM enrichment pipeline
→ mock calibration
→ real run
→ prompt/validator HF
→ low-resource avoid_when HF
```

После `PRD-046.0.5-RUN1-HF3` controlled real LLM batch достиг состояния:

```text
validation_passed = 60
validation_failed = 0
hard_validation_failed = 0
unknown_lens_candidates = 0
summary_direct_quote_risk_count = 0
safety_governance_invariant_violations = 0
raw_text_leak_check = pass
production_candidate_ready = true
promotion_allowed = false
promotion_reasons = ["requires_separate_apply_prd"]
```

Это означает: overlay-кандидат достаточно качественный для controlled apply, но его нельзя применять “вручную” или неявно. Требуется отдельный безопасный PRD, который:

1. проверит readiness;
2. применит overlay только как advisory enrichment metadata;
3. не изменит deterministic governance authority;
4. пересоберёт Chroma collection;
5. подтвердит, что API retrieval отдаёт enrichment metadata;
6. обновит living docs.

---

## 2. Главная цель

Безопасно применить production-candidate overlay из `PRD-046.0.5-RUN1-HF3` к governed processed blocks и выполнить controlled Chroma refresh так, чтобы:

```text
all_blocks_merged.json
+ enrichment overlay
→ governed enriched blocks
→ Chroma reindex
→ /api/query returns governance + enrichment metadata
→ bot retrieval path remains botdb_api, no fallback
```

Важно: enrichment metadata **не становится safety/governance authority**.

---

## 3. Non-goals

В этом PRD запрещено:

1. Менять Writer.
2. Менять DiagnosticCard.
3. Менять Thread Manager.
4. Менять State Analyzer.
5. Запускать Diagnostic Center.
6. Менять user-facing tone или runtime prompt Writer.
7. Делать enriched KB прямым источником цитат.
8. Заменять deterministic governance решения LLM-данными.
9. Удалять raw block text.
10. Переписывать весь KB pipeline.
11. Применять overlay без backup / diff / validation gate.
12. Включать Async Turn LLM Summary. Это отдельный следующий PRD.
13. Делать retrieval eval set. Это отдельный следующий PRD после context balance шага.

## 4. Архитектурное решение

### 4.1. Что именно можно применять

Применять можно только advisory поля из real enrichment candidate overlay:

```json
{
  "summary_candidate": "...",
  "lens_family_candidates": ["..."],
  "tags": ["..."],
  "use_when": ["..."],
  "avoid_when": ["..."],
  "self_contained_score": 0.0,
  "self_contained_reason": "...",
  "split_merge_suggestion": {
    "action": "...",
    "reason": "..."
  },
  "confidence": 0.0,
  "needs_human_review": true,
  "review_reasons": ["..."],
  "llm_metadata": {
    "provider": "...",
    "model": "...",
    "prompt_version": "...",
    "mock": false
  }
}
```

### 4.2. Что нельзя мутировать

Следующие поля нельзя менять из overlay:

```text
text
id / block_id
source
title
source metadata identity
governance.chunk_type
governance.allowed_use
governance.safety_flags
governance.not_for_direct_quote semantic meaning
chunking_quality
boundary_confidence
split_reason
parent_section_id
```

### 4.3. Где хранить applied enrichment

Рекомендуемый формат внутри каждого block:

```json
{
  "metadata": {
    "llm_enrichment": {
      "schema_version": "kb_llm_enrichment_v1",
      "applied_from_prd": "PRD-046.0.5-APPLY1",
      "source_overlay": "PRD-046.0.5-RUN1-HF3",
      "status": "applied_candidate",
      "review_status": "machine_candidate_needs_human_review",
      "summary": "...",
      "lens_family_candidates": ["..."],
      "tags": ["..."],
      "use_when": ["..."],
      "avoid_when": ["..."],
      "self_contained_score": 0.0,
      "self_contained_reason": "...",
      "split_merge_suggestion": {
        "action": "...",
        "reason": "..."
      },
      "confidence": 0.0,
      "needs_human_review": true,
      "review_reasons": ["..."],
      "llm_metadata": {
        "provider": "...",
        "model": "...",
        "prompt_version": "kb_enrichment_v1_3",
        "mock": false
      },
      "applied_at": "ISO-8601"
    }
  }
}
```

Разрешается также обновить top-level `summary`, но только если это уже существующий canonical summary slot для retrieval/admin, и только при соблюдении правил:

```text
- original summary сохранить в metadata.llm_enrichment.previous_summary или backup artifact;
- summary_candidate не должен быть direct quote;
- deterministic governance не зависит от summary;
- diff report должен явно показать summary_changed_count.
```

Если есть сомнение, предпочтительнее не перезаписывать top-level summary, а хранить enrichment summary только в `metadata.llm_enrichment.summary`.

### Workstream B — Safe apply tool

Добавить или расширить CLI-инструмент:

```text
Bot_data_base/tools/apply_enrichment_overlay.py
```

Минимальные режимы:

```bash
python Bot_data_base/tools/apply_enrichment_overlay.py --dry-run
python Bot_data_base/tools/apply_enrichment_overlay.py --apply --confirm
```

Опции:

```bash
--blocks-path Bot_data_base/data/processed/all_blocks_merged.json
--overlay-path TO_DO_LIST/logs/PRD-046.0.5-RUN1-HF3/real_enrichment_candidates.jsonl
--output-path Bot_data_base/data/processed/all_blocks_merged.json
--backup-dir TO_DO_LIST/logs/PRD-046.0.5-APPLY1/backups
--source-label "КУЗНИЦА ДУХА"
```

Требования:

- `--apply` требует `--confirm`.
- В dry-run нельзя мутировать файлы.
- Перед apply создать backup текущего `all_blocks_merged.json`.
- Apply должен быть idempotent: повторный запуск не должен дублировать enrichment.
- Apply должен сопоставлять candidates по `block_id`.
- Apply должен пропускать overlay rows без matching block и фиксировать это в отчёте.
- Apply не должен падать от `needs_human_review=true`, но обязан сохранить этот статус.
- Apply должен валидировать no raw full text leak в generated artifacts.


```text

```
