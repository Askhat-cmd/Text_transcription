# PRD-046.0.5-HF1 — Enrichment Prompt / Validator Calibration v1

## 0. Назначение PRD

Этот PRD является коротким hotfix/calibration-шагом после `PRD-046.0.5 — Offline LLM Summary + Lens Enrichment v1`.

`PRD-046.0.5` успешно создал инфраструктуру offline enrichment:

- enrichment contracts;
- prompt;
- validators;
- CLI;
- overlay output;
- safety invariant checks;
- tests.

Но acceptance-run показал, что текущая калибровка ещё не готова к следующему этапу:

```text
chunks_selected: 30
chunks_enriched: 30
validation_passed: 2
validation_failed: 28
needs_human_review: 28
validation_reasons_distribution:
  summary_direct_quote_risk: 28
real_llm_run: false
mock_llm_run: true
raw_text_leak_check: pass
safety_governance_invariant_violations: 0
```

Главная проблема: mock/enrichment path создаёт `summary_candidate`, слишком близкий к исходному тексту. Валидатор правильно ловит это как `summary_direct_quote_risk`.

Цель этого PRD — **не ослабить validator**, а откалибровать prompt/mock-generation/acceptance gate так, чтобы enrichment-кандидаты были смысловыми переформулировками, а не цитатными фрагментами.

---

## 1. Архитектурное решение

Следующий большой PRD (`PRD-046.0.6 — Knowledge Retrieval Eval Set v1`) пока НЕ начинать.

Причина:

```text
Retrieval eval должен проверять качество поиска по enriched summaries/lens/tags.
Но enrichment пока не прошёл quality gate:
2/30 passed, 28/30 failed из-за direct quote risk.
```

Правильный маленький шаг:

```text
PRD-046.0.5-HF1
  ↓
calibrate prompt / mock LLM / validators / overlay gate
  ↓
получить validation_passed >= целевого порога на calibration batch
  ↓
только потом PRD-046.0.5-RUN1 или PRD-046.0.6
```

---

## 2. Цель

Откалибровать offline enrichment pipeline так, чтобы:

1. `summary_candidate` был retrieval-friendly смысловой переформулировкой, а не началом/куском исходного chunk.
2. `summary_direct_quote_risk` резко снизился.
3. Validator сохранил защиту от прямого цитирования.
4. Mock LLM стал полезным deterministic test double, а не источником заведомо провальных summaries.
5. Overlay не считался production-ready, если run был mock или validation не прошёл acceptance gate.
6. Все артефакты оставались sanitized: без raw full chunk text, без `content_full`, без raw prompt with full text.
7. Runtime бота не менялся.

---

## 3. Non-goals

В рамках этого PRD НЕ делать:

- не менять `Writer`;
- не менять `DiagnosticCard`;
- не менять `Thread Manager`;
- не менять `State Analyzer`;
- не менять runtime retrieval behavior;
- не делать Chroma production reindex;
- не мутировать `all_blocks_merged.json`;
- не применять enrichment к production blocks;
- не ослаблять deterministic governance/safety;
- не делать Diagnostic Center;
- не запускать full enrichment на все 229 chunks как обязательный шаг;
- не требовать обязательного real LLM run, если нет ключа/API окружения;
- не коммитить `.env`, ключи, `.venv`, `node_modules`, sqlite/db/cache, raw private logs.

## 6. Workstream B — Prompt v1.1: anti-quote enrichment contract

Создать новую версию prompt:

```text
Bot_data_base/knowledge_governance/prompts/kb_enrichment_v1_1.md
```

Не удалять `kb_enrichment_v1.md`.

Prompt v1.1 должен явно требовать:

```text
summary_candidate:
- не начинать с фразы исходного текста;
- не повторять предложение из исходного текста;
- не использовать длинные авторские формулировки;
- быть сжатым смысловым описанием “для чего этот chunk нужен retrieval/context assembly”;
- не быть user-facing советом;
- не ссылаться на книгу/автора/Кузницу;
- не содержать “в данном тексте”, “автор говорит”, “книга рассказывает”, “согласно...”;
- 120–500 символов;
- русский UTF-8.
```

Добавить 2–3 встроенных мини-примера:

### Плохой пример

```json
{
  "summary_candidate": "Три причины, по которым программа «несовершенное Я» держится так крепко..."
}
```

Проблема: начинается как исходный chunk.

### Хороший пример

```json
{
  "summary_candidate": "Фрагмент помогает распознать устойчивый внутренний механизм самокритики: почему реакция закрепляется через ранний опыт, эмоциональную значимость и автоматичность, и когда эту линзу уместно использовать для мягкого отражения паттерна."
}
```

Пример не должен быть зашит как универсальный шаблон для всех chunks.

## 7. Workstream C — Mock LLM calibration

Обновить `_MockLLMClient` в:

```text
Bot_data_base/tools/kb_llm_enrichment.py
```

Требования:

1. Mock LLM больше не должен строить `summary_candidate` как `_safe_preview(excerpt)`.
2. Он должен deterministic-образом собирать paraphrased summary из:
   - `chunk_type_original`;
   - `title`;
   - `chapter_title`;
   - `lens_family_current`;
   - keyword/lens inference;
   - `mixed_intent_severity`;
   - safety flags;
   - current summary only как weak hint, не как копия.
3. Summary должен быть 120–500 символов.
4. Summary не должен начинаться с первых 80 символов исходного chunk.
5. Summary не должен быть слишком generic:
   - плохо: “Фрагмент описывает психологический паттерн...”
   - лучше: “Фрагмент помогает распознать связь между самокритикой, страхом оценки и попыткой контролировать переживание через достижение...”
6. Mock должен оставаться deterministic и быстрым.

Добавить отдельную helper-функцию, например:

```python
_build_mock_paraphrase_summary(context: dict[str, Any]) -> str
```

или аналогичную.

## 8. Workstream D — Validator calibration without weakening safety

Файл:

```text
Bot_data_base/knowledge_governance/enrichment_validators.py
```

Требования:

1. Не удалять `summary_direct_quote_risk`.
2. Не снижать защиту до бессмысленного уровня.
3. Сделать validator более объяснимым и пригодным для prompt calibration.

Рекомендация:

Сохранить текущую prefix-проверку:

```python
summary[:80] in source
```

Но вынести параметры в именованные константы:

```python
SUMMARY_DIRECT_QUOTE_PREFIX_CHARS = 80
SUMMARY_DIRECT_QUOTE_MIN_LENGTH = 80
```

Добавить sanitized reason detail только в локальную структуру тестов/логов без raw text, например:

```text
summary_direct_quote_risk: prefix_overlap
```

Если добавляется n-gram/overlap logic, она должна:

- работать только in-memory;
- не сохранять raw source text в artifacts;
- не считать короткие общие фразы direct quote;
- иметь unit tests.

Важно: если validator меняется так, что старые copy-like summaries вдруг проходят — PRD не принят.

## 9. Workstream E — Overlay readiness gate

Сейчас `PRD-046.0.5` записал overlay:

```text
Bot_data_base/data/processed/enrichment_overlays/kuznitsa_llm_enrichment_v1.jsonl
```

Но run был mock и 28/30 failed. Значит overlay нельзя считать production-ready.

Добавить overlay readiness gate в enrichment CLI.

Создать артефакт:

```text
TO_DO_LIST/logs/PRD-046.0.5-HF1/overlay_readiness_report.json
```

Минимальные поля:

```json
{
  "overlay_path": "...",
  "run_kind": "mock|real",
  "validation_passed": 0,
  "validation_failed": 0,
  "validation_failed_ratio": 0.0,
  "summary_direct_quote_risk_count": 0,
  "unknown_lens_candidates": 0,
  "safety_governance_invariant_violations": 0,
  "raw_text_leak_check": "pass|fail",
  "production_ready": false,
  "promotion_allowed": false,
  "reasons": []
}
```

Production readiness rules:

```text
production_ready = true only if:
- real_llm_run = true
- validation_failed_ratio <= 0.20
- summary_direct_quote_risk_count <= 3 for 30-chunk calibration batch
- unknown_lens_candidates = 0
- safety_governance_invariant_violations = 0
- raw_text_leak_check = pass
```

Для mock-run:

```text
production_ready must be false
promotion_allowed must be false
```

Даже если mock-run проходит validation, он может быть `calibration_passed=true`, но не production-ready.

## 11. Acceptance criteria

PRD считается выполненным, если:

### 11.1. Calibration quality

На 30-chunk mock calibration batch:

```text
chunks_selected = 30
chunks_enriched = 30
validation_passed >= 24
validation_failed <= 6
summary_direct_quote_risk <= 3
unknown_lens_candidates = 0
safety_governance_invariant_violations = 0
raw_text_leak_check = pass
```

Желательный результат:

```text
validation_passed >= 27
summary_direct_quote_risk = 0
```

### 11.2. Safety invariants

Должно остаться true:

```text
allowed_use mutated: false
safety_flags mutated: false
chunk_type mutated: false
runtime behavior changed: false
Writer changed: false
DiagnosticCard changed: false
Thread Manager changed: false
State Analyzer changed: false
Chroma production reindex performed: false
all_blocks_merged mutated: false
```

### 11.3. Overlay gate

Для mock-run:

```text
calibration_passed: true
production_ready: false
promotion_allowed: false
```

Если overlay write выполняется для теста, он должен быть явно помечен как calibration/mock, а не production-ready.

### 11.4. Tests

Запустить и сохранить вывод:

```bash
python -m pytest Bot_data_base/tests -q
python -m pytest bot_psychologist/tests/test_knowledge_policy.py -q
python -m pytest bot_psychologist/tests/multiagent -q
```

Дополнительно добавить/обновить unit tests:

```text
Bot_data_base/tests/test_kb_llm_enrichment_prompt_calibration.py
Bot_data_base/tests/test_kb_llm_enrichment_overlay_gate.py
Bot_data_base/tests/test_kb_llm_enrichment_validation.py
```

Минимальные тесты:

1. Copy-like summary fails with `summary_direct_quote_risk`.
2. Paraphrased summary passes direct quote guard.
3. Mock LLM summary does not start with source text.
4. Mock calibration produces mostly valid candidates.
5. Mock overlay is never production-ready.
6. Governance invariants cannot be mutated by enrichment candidate.
