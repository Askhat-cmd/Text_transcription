## 2. Цель PRD

Цель: откалибровать real LLM enrichment так, чтобы модель возвращала `lens_family_candidates` только в допустимых canonical enum-значениях, а validator различал:

```text
hard validation failures
soft review warnings
LLM free-form notes
```

При этом нельзя ослаблять safety/governance правила.

### 2.1. Главный целевой результат

После HF2 real-run должен показать:

```text
unknown_lens_candidates = 0
summary_direct_quote_risk_count = 0
safety_governance_invariant_violations = 0
raw_text_leak_check = pass
validation_failed_ratio <= 0.10
```

И overlay gate должен уметь явно сказать:

```text
production_candidate_ready: true/false
promotion_allowed: true/false
promotion_reasons: [...]
```

### 2.2. Что считается успехом

Успех HF2 — это не “любой ценой сделать 60/60 passed”.

Успех — это:

```text
real LLM outputs стали совместимы с controlled taxonomy;
unknown lens перестали ломать batch;
hard blockers остались hard blockers;
сомнительные элементы уходят в review warnings, а не подменяют governance;
production promotion разрешается только при прохождении gate.
```

## 4. Важный архитектурный guardrail: turn LLM summarization

Пользователь передал отдельную исследовательскую записку:

```text
LLM_turn_summarization_recommendations_for_architect_agent.md
```

Вывод из неё принят:

```text
Нельзя допустить, чтобы Knowledge Base context стал LLM-связным и semantic-rich,
а recent dialogue context остался deterministic/extractive и фрагментарным.
```

Это создаёт риск, что Writer начнёт сильнее опираться на “красивый” knowledge context, чем на живую линию пользователя.

Однако в рамках текущего состояния проекта следующий немедленный шаг всё ещё должен закрыть blocker `unknown_lens_candidates` в KB enrichment.

### 4.1. Когда внедрять turn LLM summary

Рекомендуемый момент внедрения:

```text
после закрытия текущего KB real-enrichment blocker:
  PRD-046.0.5-RUN1-HF2
  → при необходимости повторный real batch / RUN2
до:
  - активного runtime-использования enriched KB overlays;
  - полноценного Retrieval Eval как основы Diagnostic Center;
  - Diagnostic Center v1.
```

Предварительная нумерация будущего PRD:

```text
PRD-045.6.3 — Async Turn LLM Summary Store v1
```

Почему именно так:

- это продолжение Context Assembly / Turn Micro-Summary линии;
- deterministic summary остаётся fallback;
- LLM-summary должна быть async after-response, не блокируя текущий ответ;
- raw turns сохраняются полностью;
- trace должен показывать `summary_source`.

Этот PRD НЕ реализует turn summarization, но должен оставить в отчётах архитектурную заметку:

```text
Next architecture checkpoint after KB real-enrichment stabilization:
consider PRD-045.6.3 Async Turn LLM Summary Store v1 before enriched KB runtime promotion.
```


Нельзя перезаписывать `kb_enrichment_v1_1.md`; он должен остаться как история.

### 5.1. Требования к prompt v1_2

Prompt должен явно включать controlled enum для `lens_family_candidates`.

Использовать текущий allowlist из validator:

```text

```

Обязательное правило:

```text
lens_family_candidates MUST contain only exact strings from this allowlist.
Do not translate them.
Do not invent new lens names.
If no exact lens is suitable, return fewer candidates and set needs_human_review=true.
```

### 5.2. Review reasons

Prompt должен ограничить `review_reasons` controlled enum-значениями.

Рекомендуемый enum:

```text
low_confidence
insufficient_context
mixed_intent_unclear
split_merge_unclear
lens_mapping_uncertain
practice_context_unclear
safety_context_unclear
summary_quality_uncertain
```

Запретить свободные длинные русские формулировки в `review_reasons`.

Если нужна свободная заметка, она не должна попадать в `review_reasons`. Для v1 можно вообще не просить free-form note.

### 5.3. Good/bad examples

Добавить 2–3 коротких примера:

1. Хороший пример lens candidates:

```json
"lens_family_candidates": ["self_criticism", "achievement", "shame"]
```

2. Плохой пример:

```json
"lens_family_candidates": ["самокритика", "страх оценки", "внутренняя программа"]
```

Причина: значения не из allowlist.

3. Хороший пример review reasons:

```json
"review_reasons": ["lens_mapping_uncertain"]
```

4. Плохой пример:

```json
"review_reasons": ["Необходима проверка на соответствие контексту и точности интерпретации."]
```

Причина: free-form review reason ломает машинную агрегацию.

Обновить:


### 6.1. Нельзя делать

Нельзя просто расширить allowlist всеми словами, которые придумала LLM.

Почему:

- allowlist должен быть controlled taxonomy;
- Diagnostic Center позже будет опираться на эти значения;
- хаотическое расширение lens_family разрушит управляемость.

### 6.2. Нужно сделать

Добавить deterministic post-processing перед validation:

```text
raw LLM lens candidates

→ apply synonym map
→ keep only allowlist values

→ candidate.lens_family_candidates contains only allowlist values
```

## Workstream D — Review reason handling

Сейчас real LLM может возвращать свободные длинные review reasons, например:

```text
Необходима проверка на соответствие контексту и точности интерпретации.
```

Это плохо для машинной агрегации.

### 7.1. Требования

Добавить controlled review reason normalization:

```text
raw review_reasons
→ keep only controlled enum values
→ unknown/free-form review strings store as sanitized llm_review_notes / review_reason_unmapped
→ do not treat free-form note as hard validation failure by itself
```

Hard validation failure должны давать только deterministic validators:

```text
summary_empty
summary_generic_prefix
summary_direct_quote_risk
unknown_lens_candidate_after_normalization_if_no_valid_lens_or_strict_mode
low_resource_avoid_when_missing
governance invariant violations
raw text leak
invalid schema / malformed JSON
```

### 7.2. Gate logic

`needs_human_review` может быть true, но это не всегда равно `validation_failed`.

Разделить в reports:

```text
hard_validation_failed
soft_review_warnings
llm_requested_review
unmapped_review_notes_count
```

В текущем PRD можно оставить старое поле `needs_human_review`, но добавить новые поля, чтобы не смешивать блокеры и soft warnings.
