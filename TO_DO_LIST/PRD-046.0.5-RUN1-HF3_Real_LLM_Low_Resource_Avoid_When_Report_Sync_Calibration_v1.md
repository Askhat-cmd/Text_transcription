## 2. Архитектурное решение

Нельзя идти дальше к `PRD-046.0.6` или Diagnostic Center.

Нужно сделать маленький hotfix:

```text
1. Исправить report sync / пустые отчёты HF2.
2. Закрыть hard blocker low_resource_avoid_when_missing.
3. Сохранить strict validator — не ослаблять safety.
4. Повторить real-run на той же controlled выборке 60 chunks.
5. Разрешить только production-candidate readiness, но не мутировать all_blocks_merged и не делать Chroma reindex.
```

---

## 3. Цель PRD

Сделать real LLM enrichment batch production-candidate-ready на controlled выборке 60 chunks за счёт:

- исправления post-push/report sync хвостов `HF2`;
- усиления prompt `kb_enrichment_v1_3`;
- deterministic post-validation / candidate normalization для `avoid_when` в safety-critical случаях;
- сохранения strict safety validator;
- повторного real-run;
- генерации полных, непустых отчётов.

---

## 4. Non-goals

В этом PRD запрещено:

- менять Writer;
- менять DiagnosticCard;
- менять Thread Manager;
- менять State Analyzer;
- менять runtime user-facing behavior;

- писать raw full chunk text в отчёты или debug artifacts.

## 6. Workstream B — prompt v1_3: canonical low-resource avoid_when

Создать новый prompt:

```text
Bot_data_base/knowledge_governance/prompts/kb_enrichment_v1_3.md
```

На базе `kb_enrichment_v1_2.md`.

Добавить жёсткое правило:

```text
Если в safety_flags_original есть practice_requires_low_resource_check,
то avoid_when ОБЯЗАТЕЛЬНО должен содержать один из canonical phrases:
- "при низком ресурсе, истощении или острой нестабильности"
- "когда у пользователя мало сил, кризис или выраженная перегрузка"
```

Важно:

- это не user-facing совет;
- это enrichment metadata;
- фраза нужна, чтобы validator мог надёжно проверить safety condition;
- summary, tags, lens candidates не должны ухудшаться.

Также добавить bad/good пример.

Плохой пример:

```json
{
  "avoid_when": ["если практика не подходит"]
}
```

Хороший пример:

```json
{
  "avoid_when": [
    "при низком ресурсе, истощении или острой нестабильности",
    "когда нужна срочная кризисная поддержка вместо исследования"
  ]
}
```

## 7. Workstream C — deterministic avoid_when normalization без ослабления validator

### 7.1 Проблема

Real LLM может смыслово написать корректный avoid_when, но без слов, которые validator распознаёт. Однако safety-critical chunks с `practice_requires_low_resource_check` должны иметь явный low-resource guard.

### 7.2 Решение

Добавить deterministic normalization step **до финального validation result**, но после получения LLM candidate:

```text
если safety_flags_original содержит practice_requires_low_resource_check
и avoid_when не содержит canonical low-resource phrase,
то добавить canonical avoid_when phrase автоматически
и записать trace:
avoid_when_low_resource_guard_autofilled=true
```

Это не ослабляет safety, а усиливает её: у всех risk-chunks появляется явный avoid guard.

### 7.3 Ограничения

- Не удалять исходные avoid_when от LLM.
- Не менять `allowed_use`.
- Не менять `safety_flags`.
- Не менять `chunk_type`.
- Не превращать internal_only/safety chunks в writer-facing.
- Добавить trace fields в candidate artifact, но без raw full text.

Рекомендуемые поля:

```json
{
  "normalization_trace": {
    "avoid_when_low_resource_guard_autofilled": true,
    "lens_normalization_applied": true,
    "review_reason_normalization_applied": true
  }
}
```

## 13. Связь с LLM turn summarization

Этот PRD не внедряет turn summarization.

Но после успешного HF3 следующий правильный архитектурный шаг — `PRD-045.6.3 — Async Turn LLM Summary Store v1`.

Причина:

```text
Knowledge Base уже становится LLM-enriched и связной.
Recent dialogue context пока остаётся deterministic/extractive.
Для психологического бота живой контекст пользователя должен иметь приоритет над книжной базой.
```

Поэтому после закрытия текущего KB blocker нельзя сразу идти в Diagnostic Center и нежелательно сразу усиливать Writer через enriched KB. Нужно сначала подтянуть Context Assembly.
