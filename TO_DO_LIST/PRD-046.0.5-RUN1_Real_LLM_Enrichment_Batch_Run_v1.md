# PRD-046.0.5-RUN1 — Real LLM Enrichment Batch Run v1

## 0. Мета

**Статус:** новый PRD для IDE-агента  
**Язык реализации отчётов:** русский  
**Тип шага:** controlled offline real LLM batch-run  
**Главная цель:** впервые прогнать `КУЗНИЦУ ДУХА` через реальный LLM enrichment на ограниченной репрезентативной выборке, подтвердить качество prompt `kb_enrichment_v1_1`, валидаторов и overlay readiness gate на настоящих LLM-кандидатах, не изменяя runtime и production knowledge index.

---

## 1. Контекст

После `PRD-046.0.4.3` восстановлен governed retrieval path:

- Chroma collection пересобрана из governed `all_blocks_merged.json`;
- `collection_count_after=229`;
- `/api/status`, `/api/registry`, `/api/query` подтверждены smoke-проверками;
- `bot_psychologist` использует `botdb_api`, fallback/circuit breaker в smoke не используется.

После `PRD-046.0.5` создан offline LLM enrichment pipeline, но acceptance-run был mock-only и показал проблему:

- `validation_passed=2`;
- `validation_failed=28`;
- основная причина: `summary_direct_quote_risk=28`.

После `PRD-046.0.5-HF1` калибровка mock/prompt/validator прошла успешно:

- добавлен prompt `Bot_data_base/knowledge_governance/prompts/kb_enrichment_v1_1.md`;
- mock summary перестал строиться как `_safe_preview(excerpt)`;
- validator не ослаблен, а диагностически уточнён;
- `validation_passed_after=30`;
- `validation_failed_after=0`;
- `summary_direct_quote_risk_after=0`;
- `raw_text_leak_check=pass`;
- overlay gate корректно заблокировал promotion для mock-run: `production_ready=false`, `promotion_allowed=false`, reason `mock_run_not_production_eligible`.

Следующий шаг — не retrieval eval и не Diagnostic Center, а **real LLM batch-run**. Нельзя применять mock overlay как production enrichment.

---

## 2. Цель PRD

Сделать безопасный controlled real-run:

```text
Governed blocks
→ prompt kb_enrichment_v1_1
→ real LLM candidates
→ deterministic validators
→ overlay readiness gate
→ reports
→ next decision
```

Нужно подтвердить, что реальный LLM:

1. создаёт нецитатные summary;
2. не предлагает lens вне allowlist;
3. не меняет governance authority;
4. не приводит к raw text leaks в артефактах;
5. корректно маркируется как `real_llm_run=true`, `mock_llm_run=false`;
6. может дать overlay-кандидат, пригодный для следующего controlled apply-step.

## 4. Основной принцип безопасности

LLM в этом PRD — **offline semantic enrichment assistant**, а не authority.

LLM может предложить:

- `summary_candidate`;
- `lens_family_candidates`;
- `tags`;
- `use_when`;
- `avoid_when`;
- `self_contained_score`;
- `split_merge_suggestion`;
- `confidence`;
- `needs_human_review`.

LLM не имеет права финально менять:

- `chunk_type`;
- `allowed_use`;
- `safety_flags`;
- `not_for_direct_quote`;
- `source_style_not_user_facing`;
- clinical/safety authority.

## 6. Workstream B — Real LLM hard requirement

В `Bot_data_base/tools/kb_llm_enrichment.py` нужно добавить или проверить наличие режима, который **запрещает silent fallback на mock**.

Требуемый CLI-флаг:

```bash
--require-real-llm
```

Поведение:

- если `--require-real-llm` включён и `OPENAI_API_KEY` отсутствует — run должен завершиться как blocked/failed preflight;
- если `--require-real-llm` включён и реальный LLM вызов не удался — нельзя подменять результат mock-кандидатом;
- `mock_llm_run` должен быть `false`;
- `real_llm_run` должен быть `true`;
- каждый candidate должен иметь `llm_metadata.mock=false`;
- в report явно указывать provider/model/prompt_version без API key.

Если сейчас инструмент автоматически делает fallback на `_MockLLMClient`, это допустимо для dev/mock режимов, но **не допустимо** для RUN1 acceptance.

## 8. Workstream D — Representative real batch selection

Запустить real-run на репрезентативной выборке из `КУЗНИЦЫ`.

Минимальные требования:

```text
limit: 60 chunks
source: КУЗНИЦА ДУХА
prompt: kb_enrichment_v1_1.md
run_kind: real
```

Выборка должна быть deterministic и stratified:

- `case`;
- `lens`;
- `practice`;
- `safety`;
- `style`;
- `theory`.

Дополнительно в выборке должны быть представлены:

- chunks с `mixed_intent_severity=medium/high`, если есть;
- practice chunks с `practice_requires_low_resource_check`;
- chunks с пустым или слабым `lens_family`, если есть;
- минимум несколько `internal_only` / `safety_protocol` chunks, чтобы проверить constraints;
- разные главы/разделы, а не только первые 60 chunks.

Если существующий selector уже stratified, расширить/параметризовать его так, чтобы в отчёте была видна distribution.

## 11. Success criteria

RUN1 считается успешным для перехода к следующему apply-step, если:

- `real_llm_run=true`;
- `mock_llm_run=false`;
- `prompt_version=kb_enrichment_v1_1`;
- `chunks_selected >= 60`;
- `chunks_enriched == chunks_selected`;
- `validation_failed=0`;
- `summary_direct_quote_risk_count=0`;
- `unknown_lens_candidates=0`;
- `safety_governance_invariant_violations=0`;
- `raw_text_leak_check=pass`;
- `needs_human_review=0` или строго обоснованный малый tail с `production_ready=false`;
- `all_blocks_merged mutated=false`;
- `Chroma production reindex performed=false`;
- runtime behavior changed=false;
- Writer/DiagnosticCard/Thread Manager/State Analyzer unchanged.

Если хотя бы один production-ready критерий не выполнен, PRD всё равно может быть implementation-done, но следующий PRD должен быть targeted hotfix/calibration, а не full apply.

## 18. Критерий архитектурного успеха

После этого PRD у нас должен быть не “моковый красивый enrichment”, а доказательство, что настоящий LLM в offline режиме может безопасно и стабильно обогащать governed knowledge без:

- цитатного leakage;
- governance mutation;
- runtime side effects;
- Chroma/index side effects;
- hidden fallback на mock.

Только после этого можно переходить к full apply/refresh, а затем к retrieval eval.
