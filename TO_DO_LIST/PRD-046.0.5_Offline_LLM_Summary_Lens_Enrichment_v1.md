После `PRD-046.0.4.3` governed Chroma/API retrieval path восстановлен:

- governed gate: `ready`;
- Chroma reindex выполнен из governed `all_blocks_merged.json`;
- indexed blocks: `229`;
- collection count: `229`;
- `/api/status`, `/api/registry`, `/api/query` validated `200` на чистом локальном BotDB API instance;
- `bot_psychologist` smoke подтвердил `retrieval_source=botdb_api`, `fallback_used=false`, `circuit_breaker_open=false`;
- Writer / DiagnosticCard / Thread Manager / State Analyzer не менялись.

Следующий безопасный шаг — не Diagnostic Center, а offline enrichment качества knowledge chunks:

```text
governed chunk
→ offline LLM semantic enrichment candidate
→ deterministic validation
→ safe overlay / report
→ human/architect review
→ только потом apply/full refresh/retrieval eval
```

`PRD-046.0.5` должен добавить offline LLM enrichment pipeline для улучшения:

- `summary`;
- `lens_family` candidates;
- tags;
- `use_when`;
- `avoid_when`;
- self-contained оценки;
- split/merge suggestions.

Важно: LLM **не получает финальную власть над safety/governance**.  
LLM не меняет runtime.  
LLM не переписывает `allowed_use`, `safety_flags`, `chunk_type` как источник истины.

## 1. Контекст и причина PRD

### 1.1 Что уже завершено

Линия `046.0.4.x` закрыла важные инфраструктурные проблемы:

1. Старый processed export `КУЗНИЦЫ` был stale / ungoverned.
2. Deterministic governance backfill довёл gate до `ready`.
3. Practice classification precision снизил шум false practice.
4. `PRD-046.0.4.3` восстановил governed Chroma/API retrieval path.

Теперь база технически управляемая и доступна через API.

### 1.2 Что всё ещё не решено

Даже при рабочем Chroma/API deterministic metadata остаётся семантически ограниченной:

- summary может быть формальным или не поисково-полезным;
- `lens_family` покрыта не полностью и местами грубо;
- tags отсутствуют или не стандартизированы;
- у chunk нет `use_when` / `avoid_when`;
- нет оценки self-contained качества;
- split/merge suggestions пока не формируются semantic-level способом.

Это влияет на качество будущего retrieval eval и Diagnostic Center.

### 1.3 Почему не Diagnostic Center сейчас

Diagnostic Center будет выбирать линзы и практики из governed retrieval. Если enrichment и retrieval eval ещё не готовы, Diagnostic Center начнёт выбирать материал по слабым summary/lens metadata.

Правильный порядок:

```text
Chroma/API restored
→ offline LLM enrichment
→ retrieval eval set
→ admin review workflow
→ Diagnostic Center
```

## 2. Цель PRD

Добавить offline enrichment pipeline, который создаёт validated enrichment candidates для governed chunks `КУЗНИЦА ДУХА`, без изменения runtime поведения.

Целевой результат PRD:

```text
Bot_data_base/tools/kb_llm_enrichment.py
  ↓
читает governed all_blocks_merged.json
  ↓
вызывает LLM offline или mock client
  ↓
получает strict JSON enrichment candidate
  ↓
прогоняет deterministic validators
  ↓
пишет safe overlay/report/logs
  ↓
не ломает gate и не меняет safety/governance authority
```

---

## 3. Главный архитектурный принцип

LLM в этом PRD — **семантический помощник**, не governance-судья.

Разрешено LLM:

```text
summary_candidate
lens_family_candidates
tags
use_when
avoid_when
self_contained_score
split_merge_suggestion
enrichment_notes
```

Запрещено LLM как финальное решение:

```text
chunk_type
allowed_use
safety_flags
do_not_use
internal_only
clinical/safety risk
source_style_not_user_facing
not_for_direct_quote
```

LLM может отметить “possible_safety_note”, но deterministic validators не должны автоматически переносить это в `safety_flags`.

### Workstream E — Prompt v1 для enrichment

Создать prompt template, например:

```text
Bot_data_base/knowledge_governance/prompts/kb_enrichment_v1.md
```

Требования к prompt:

- язык ответа: русский;
- output: strict JSON only;
- не цитировать chunk;
- summary — paraphrase, не direct quote;
- не делать медицинских/клинических утверждений;
- не давать пользовательских советов;
- не менять governance authority;
- не использовать “КУЗНИЦА говорит...”;
- не превращать материал в книжную рекомендацию;
- `КУЗНИЦА` используется как внутренняя линза;
- для low-resource/safety-sensitive chunks добавлять `avoid_when`.

Prompt должен давать LLM только необходимый sanitized context:

```text
block_id
title/source_title/chapter_title
chunk_type
allowed_use
safety_flags
lens_family current
summary current
content excerpt or bounded text for enrichment
```

Допускается передавать full chunk text в offline LLM call, но:

- не сохранять full chunk text в output artifacts;
- не писать raw prompt с full text в logs;
- не коммитить raw LLM request dumps.

### Workstream F — Deterministic validators

Добавить validators для enrichment candidates.

Минимальные проверки:

#### Summary validation

- not empty;
- recommended length: 120–500 chars;
- не начинается с пустых клише:
  - “В данном тексте...”
  - “Автор говорит...”
  - “Книга рассказывает...”
  - “Согласно Кузнице...”
- не содержит прямого длинного цитирования из chunk;
- содержит смысловые слова, полезные для retrieval;
- не звучит user-facing как инструкция бота.

#### Lens validation

- every lens in allowlist;
- unknown values -> `needs_human_review=true`;
- max recommended lens count: 3–8;
- do not delete existing deterministic lens automatically.

#### Tags validation

- lowercase / snake_case or normalized text convention;
- max 12 tags;
- no huge phrases.

#### use_when / avoid_when validation

- 1–4 items each;
- short, concrete;
- no direct advice to user;
- must respect safety flags;
- if `practice_requires_low_resource_check` present, `avoid_when` should mention low resource / crisis / unstable state.

#### Safety/governance invariants

Validators must prove:

```text
allowed_use_original unchanged
safety_flags_original unchanged
chunk_type_original unchanged
internal_only remains internal
do_not_use remains do_not_use if present
not_for_direct_quote remains preserved
source_style_not_user_facing remains preserved
```

#### Redaction validation

Artifacts must not contain:

```text
content_full
raw_full_text
full_chunk_text
raw_llm_prompt_with_text
api_key
.env content
```
