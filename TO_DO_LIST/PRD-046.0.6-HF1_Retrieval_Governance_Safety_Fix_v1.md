# PRD-046.0.6-HF1 — Retrieval Governance Safety Fix v1

## 0. Статус документа

- **Проект:** Bot Psychologist / Neo MindBot
- **PRD:** `PRD-046.0.6-HF1`
- **Название:** Retrieval Governance Safety Fix v1
- **Язык:** русский
- **Тип:** targeted hotfix / retrieval policy calibration
- **Исполнитель:** IDE-агент
- **Архитектор:** ChatGPT / проектный архитектор
- **Цель:** закрыть safety-gate после `PRD-046.0.6`: убрать появление `internal_only` hits в non-safety retrieval top-k, не ломая semantic quality, governance metadata pass-through и raw-leak защиту.

---

## 1. Контекст

`PRD-046.0.6 — Knowledge Retrieval Eval Set v1` успешно создал deterministic retrieval eval:

```text
dataset: 24 cases
categories: 12
/api/status ok
/api/registry ok
/api/query 24/24 ok
top5_semantic_match_rate = 1.0
top5_governance_present_rate = 1.0
raw_full_text_leak_detected = false
```

Но safety threshold не пройден:

```text
internal_only_unsafe_exposure_count = 4
weak_cases_count = 3
```

Weak cases:

```text
KR-002 self_criticism_achievement
KR-013 body_awareness_freeze
KR-022 internal_style_not_quote
```

Dominant failure mode:

```text
internal_only exposure in non-safety categories
```

Это означает, что retrieval семантически работает хорошо, но policy-aware ranking/filter недостаточно строгий: chunks с `allowed_use=["internal_only", "safety_protocol"]` могут попадать в top-k там, где запрос не является safety/crisis/internal-policy запросом.

Следующий шаг не `Admin Review`, а этот targeted HF.

---

## 2. Главная цель

Сделать retrieval policy-aware:

```text
normal / non-safety user query
→ internal_only hits excluded or demoted out of returned top-k

safety / crisis / explicit safety-boundary query
→ safety_protocol/internal_only may be allowed
→ still marked as internal/not-for-direct-quote
```

После fix нужно rerun `PRD-046.0.6` dataset unchanged и получить:

```text
internal_only_unsafe_exposure_count = 0
queries_failed = 0
raw_full_text_leak_detected = false
top5_governance_present_rate >= 0.95
top5_semantic_match_rate >= 0.70
```

## 4. Архитектурное решение

### 4.1. Retrieval policy layer

Добавить или усилить policy layer между raw vector hits и API/bot returned hits.

Примерное место:

```text
Bot_data_base API query handler
или retrieval service / Chroma manager query wrapper
или bot-side BotDB API result filtering layer
```

Предпочтительно фиксить на BotDB/API side, чтобы:

```text
/api/query
bot retrieval path
eval runner
```

видели одинаковое поведение.

### 4.2. Policy categories

Нужно lightweight классифицировать retrieval request:

```text
safety_context = true/false
```

#### safety_context=true, если query содержит явные маркеры:

```text
мне очень плохо
страшно оставаться одному
я на грани
не хочу жить
хочу исчезнуть
не выдерживаю
опасно
кризис
паника
самоповреждение
причинить вред себе
причинить вред другому
суицид
```

Также можно учитывать API request metadata, если уже есть field типа:

```text
intent=safety
include_internal=true
```

Но если такого поля нет — не добавлять сложную новую public API surface без необходимости.

#### safety_context=false

Все остальные запросы по умолчанию считаются non-safety.

## 5. Policy rule

### 5.1. Normal/non-safety query

Для `safety_context=false`:

```text
if hit.allowed_use contains "internal_only":
    remove from returned user-facing top-k
```

Допускается не hard-remove, а сильный demotion, но acceptance требует, чтобы в final returned top5 для non-safety не было `internal_only`.

Лучше правило:

```text
raw_top_n = query Chroma with expanded n (например top_k * 3 или минимум 15)
filter out internal_only for non-safety
return first top_k after filtering
```

Так мы не теряем top5 quality при фильтрации.

### 5.2. Safety query

Для `safety_context=true`:

```text
internal_only + safety_protocol hits may remain
```

Но response metadata must preserve:

```text
allowed_use
safety_flags
not_for_direct_quote
source_style_not_user_facing
internal_use_only=true or equivalent marker
```

### 5.3. No raw leak

Filtering/reranking не должен менять redaction policy:

```text
content_preview <= existing safe limit
no full text
no content_full
no full_text
```

## 9. Failure branches

### Branch A — internal_only leakage fixed, semantic quality drops

If:

```text
internal_only_unsafe_exposure_count = 0
top5_semantic_match_rate < 0.70
```

Then:

```text
increase raw fetch size
add metadata-aware reranking
do not weaken dataset
```

### Branch B — non-safety fixed but safety query loses safety_protocol

If safety query cannot retrieve safety_protocol/internal_only when appropriate:

```text
improve safety_context classifier
add explicit safety query tests
do not globally remove internal_only from index
```

### Branch C — API path hard to change safely

If BotDB API policy is too risky:

```text
implement bot-side filter only as temporary guard
write limitation clearly
next PRD must move filter into BotDB API
```

But preferred fix remains API-side.
