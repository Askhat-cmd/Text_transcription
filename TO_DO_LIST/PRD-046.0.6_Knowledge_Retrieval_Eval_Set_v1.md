# PRD-046.0.6 — Knowledge Retrieval Eval Set v1

## 0. Статус документа

- **Проект:** Bot Psychologist / Neo MindBot
- **PRD:** `PRD-046.0.6`
- **Название:** Knowledge Retrieval Eval Set v1
- **Язык:** русский
- **Тип:** evaluation / quality gate / no-runtime-change
- **Исполнитель:** IDE-агент
- **Архитектор:** ChatGPT / проектный архитектор
- **Цель:** создать воспроизводимый eval-набор и runner для проверки качества Knowledge Base retrieval после governed Chroma/API restore, LLM enrichment apply и Async Turn LLM Summary stabilization.

---

## 1. Контекст

Проект прошёл ключевые readiness-этапы:

```text
PRD-046.0.4.3
→ Chroma/API retrieval restored from governed blocks

PRD-046.0.5-APPLY1
→ real LLM enrichment overlay applied as advisory metadata
→ Chroma refreshed
→ /api/query returns governance + enrichment metadata
→ bot retrieval path uses botdb_api

PRD-045.6.3 + HF1
→ Async Turn LLM Summary Store added
→ fallback behavior validated
→ recent dialogue context no longer сильно слабее enriched KB context
```

Теперь нужно не добавлять новые runtime-функции, а измерить: **достаёт ли BotDB/API retrieval правильные governed chunks на реальные психологические запросы**.

Без retrieval eval нельзя безопасно переходить к Diagnostic Center, потому что Diagnostic Center будет выбирать линзы и next-focus на основе retrieval-context. Если retrieval возвращает случайные, слишком книжные или небезопасные blocks, диагностический слой будет строиться на плохом основании.

---

## 2. Главная цель

Создать воспроизводимую систему оценки retrieval качества:

```text
eval queries
→ /api/query
→ sanitized top-k hits
→ expected governance/lens/use_when checks
→ scorecard
→ weak queries report
→ next action recommendation
```

Eval должен ответить на вопросы:

1. Находятся ли релевантные chunks по реальным user-like запросам?
2. Возвращается ли governed metadata?
3. Видна ли applied enrichment metadata там, где она есть?
4. Не попадают ли unsafe/internal chunks в user-facing role без policy markers?
5. Не доминирует ли `КУЗНИЦА` как “цитатник”, вместо internal lens library?
6. Какие запросы слабые и требуют calibration/admin review?

---

## 3. Non-goals

В этом PRD запрещено:

1. Менять Writer.
2. Менять DiagnosticCard.
3. Менять State Analyzer.
4. Менять Thread Manager.
5. Менять Context Assembly runtime behavior.
6. Менять Async Turn LLM Summary runtime behavior.
7. Менять Knowledge Base chunks или governance metadata.
8. Делать Chroma reindex, кроме read-only smoke verification.
9. Запускать Diagnostic Center.
10. Делать Admin Review UI.
11. Использовать raw full chunk text в отчётах.
12. Подгонять retrieval качество путём ручной мутации результатов.
13. Делать real LLM в eval обязательным.

---

## 4. Архитектурное решение

### 4.1. Eval должен быть offline/read-only

Runner должен только читать:

```text
/api/status
/api/registry
/api/query
all_blocks_merged.json metadata if needed for validation
```

И писать только:

```text
TO_DO_LIST/logs/PRD-046.0.6/*
TO_DO_LIST/reports/PRD-046.0.6_*
docs/*
```

### 4.2. Eval не должен оценивать Writer

Это не conversational quality eval. Это retrieval eval:

```text
query → hits → metadata quality → expected semantic coverage
```

### 4.3. Eval должен быть deterministic

В v1 не использовать LLM-as-judge. Проверки должны быть rule-based:

```text
expected_any_lens_family
expected_any_chunk_type
expected_any_allowed_use
expected_safety_flags
expected_enrichment_presence
must_not_include flags
```

### 4.4. Eval должен быть tolerant but useful

Не нужно требовать идеального top-1. Достаточно оценивать top-k:

```text
top_1
top_3
top_5
```

## 5. Eval dataset

Создать:

```text
Bot_data_base/eval/retrieval_eval_v1.json
```

Минимум 24 кейса.

Каждый кейс:

```json
{
  "id": "KR-001",
  "query": "я всё время стараюсь быть лучше и не могу остановиться",
  "category": "self_criticism_achievement",
  "expected_any_lens_family": ["self_criticism", "achievement", "shame", "perfectionism"],
  "expected_any_chunk_type": ["lens", "case", "theory"],
  "expected_any_allowed_use": ["writer_context", "diagnostic_lens"],
  "expected_safety_flags_any": ["not_for_direct_quote"],
  "expected_enrichment_optional": true,
  "must_not_allowed_use": ["internal_only"],
  "notes": "achievement/self-worth loop; should not return only generic theory"
}
```

### 5.1. Required categories

Минимум по 2 кейса на категорию:

1. `self_criticism_achievement`
   - “я всё время стараюсь быть лучше и не могу остановиться”
   - “если я не лучший, я будто никто”

2. `guilt_boundaries`
   - “я чувствую вину, когда выбираю себя”
   - “мне трудно сказать нет, потом злюсь на себя”

3. `low_resource_support`
   - “нет сил, не хочу анализа, просто поддержи”
   - “я выжат, не хочу практик и объяснений”

4. `avoidance_procrastination`
   - “я боюсь показать проект и снова хочу отложить”
   - “я всё откладываю, потому что страшно получить оценку”

5. `anger_relationships`
   - “я раздражаюсь на близкого человека сильнее, чем ситуация того стоит”
   - “я вспыхиваю, а потом чувствую вину”

6. `fear_rejection`
   - “мне страшно, что меня отвергнут, если я скажу честно”
   - “я подстраиваюсь, чтобы меня не бросили”

7. `body_awareness_freeze`
   - “я замираю и будто не чувствую тело”
   - “когда надо действовать, тело выключается”

8. `meaning_identity`
   - “я не понимаю, кто я без достижений”
   - “вроде всё есть, но внутри пусто”

9. `practice_safe_use`
   - “дай маленький шаг, но у меня мало сил”
   - “можно что-то простое, без глубокого анализа”

10. `safety_boundary`
   - “мне очень плохо и страшно оставаться одному”
   - “я на грани, не хочу разбирать причины”

11. `internal_style_not_quote`
   - запросы, где KB может быть полезна как стиль/линза, но нельзя цитировать напрямую.

12. `enrichment_visibility`
   - кейсы, где expected top-k должен хотя бы иногда включать `llm_enrichment_*` fields после APPLY1.

## 6. Eval runner

Добавить tool:

```text
Bot_data_base/tools/run_retrieval_eval.py
```

CLI:

```bash
python Bot_data_base/tools/run_retrieval_eval.py \
  --dataset Bot_data_base/eval/retrieval_eval_v1.json \
  --api-base-url http://127.0.0.1:8013 \
  --top-k 5 \
  --output-dir TO_DO_LIST/logs/PRD-046.0.6
```

Опции:

```text
--api-base-url
--top-k
--timeout-seconds
--dataset
--output-dir
--fail-on-api-error
--include-sanitized-previews
```

Runner должен:

1. проверить `/api/status`;
2. проверить `/api/registry`;
3. для каждого query вызвать `/api/query`;
4. сохранить sanitized results;
5. посчитать метрики;
6. сформировать weak-case report.

## 7. Metrics

Минимальные метрики:

```json
{
  "cases_total": 24,
  "api_status_ok": true,
  "api_registry_ok": true,
  "queries_ok": 24,
  "queries_failed": 0,
  "top1_semantic_match_rate": 0.0,
  "top3_semantic_match_rate": 0.0,
  "top5_semantic_match_rate": 0.0,
  "top5_governance_present_rate": 1.0,
  "top5_allowed_use_present_rate": 1.0,
  "top5_safety_flags_present_rate": 1.0,
  "raw_full_text_leak_detected": false,
  "internal_only_unsafe_exposure_count": 0,
  "enrichment_metadata_seen_count": 0,
  "weak_cases_count": 0
}
```

### 7.1. Semantic match v1

A hit counts as semantic match if any of these are true:

```text
hit.lens_family intersects expected_any_lens_family
hit.chunk_type in expected_any_chunk_type AND allowed_use intersects expected_any_allowed_use
hit.llm_enrichment_tags/use_when/summary contains expected category markers
```

В v1 это rule-based approximate. Важно не переусложнять.

### 7.2. Governance pass

Every returned hit in top-k must have:

```text
chunk_type
allowed_use
safety_flags
not_for_direct_quote field or equivalent safety marker
```

If hit lacks governance fields, mark as failure.

### 7.3. Internal-only safety

If case is not explicitly safety/internal, top-k should not expose `allowed_use=["internal_only"]` as normal writer context.

If `internal_only` appears, it must be accompanied by appropriate safety/internal policy and should be flagged for review.

### 7.4. Raw leak check

Reports/logs must not contain:

```text
raw full chunk text
content_full
full_text
unredacted source block
large content dump
```

Only allowed:

```text
content_preview <= 240 chars
id
score
source_title
chapter_title
chunk_type
allowed_use
safety_flags
lens_family
llm_enrichment_* metadata
```

## 11. Acceptance criteria

PRD считается выполненным, если:

1. `Bot_data_base/eval/retrieval_eval_v1.json` создан.
2. Dataset содержит минимум 24 кейса.
3. Все required categories присутствуют.
4. Runner `Bot_data_base/tools/run_retrieval_eval.py` создан.
5. Runner работает против mocked API in tests.
6. Runner может работать против live local `/api/query`.
7. Scorecard создаётся.
8. Weak cases report создаётся.
9. Governance safety report создаётся.
10. No raw full text leak detected.
11. API smoke ok.
12. Regression tests pass.
13. Living docs updated.
14. Commit and push to `main`.
15. Implementation report updated after final push with commit hash and push status.
