**Проект:** Bot Psychologist / Neo MindBot  
**Репозиторий:** `Askhat-cmd/Text_transcription`  
**Рабочая директория:** `C:\My_practice\Text_transcription\`

## 1. Контекст

После `PRD-046.0.7` в проект добавлен безопасный CLI/report workflow для human review enrichment metadata:

- review queue создана;
- decisions overlay валидируется;
- production KB не мутируется;
- Chroma reindex не выполняется;
- runtime/Writer/Diagnostic Center не менялись.

По отчёту `PRD-046.0.7`:

```text
total_blocks_scanned: 229
review_items_count: 87
priority_counts: P0=66, P1=21, P2=0
recommended_action_counts: needs_edit=87
decisions_count: 0
source_mutated=false
raw_full_text_leak_detected=false
```

Это значит, что review infrastructure готова, но реального human decision apply ещё не было.

Пользователь провёл ручной осмотр Web Admin и прислал скриншоты. По ним выявлены новые архитектурные риски:

1. В `Bot_data_base` registry/UI есть много старых тестовых источников:
   - `test123`;
   - `Книга`;
   - YouTube/book entries с `0` blocks;
   - зависшие или бессмысленные `processing`;
   - общее количество источников в dashboard — `41`, при этом рабочий indexed corpus фактически `229` blocks от `Кузница Духа`.

2. В UI всё ещё отображается legacy `SD-распределение`:
   - `GREEN`, `BLUE`, `YELLOW`, `RED`, `ORANGE`, `BEIGE`, `PURPLE`, `TURQUOISE`, `UNCERTAIN`.

3. В API/query layer ранее сохранялась SD-логика:
   - `sd_level`;
   - `SD_LEVELS_ORDER`;
   - SD where-filter.

4. Текущие chunks/review items показывают следы старых алгоритмов:
   - practice-like chunks, начинающиеся с `**Цель:** / **Время:**`, иногда классифицированы как `lens`, `case`, `theory`;
   - это нельзя исправлять в этом PRD, но надо зафиксировать как blocker для будущего clean reprocess.

Главный вывод:

```text
Нельзя сразу переходить к Controlled Review Decision Apply или Diagnostic Center.
Сначала нужно очистить registry/admin слой, деактивировать legacy SD в UI/API и подготовить readiness gate для будущего полного reprocess.
```

## 2. Цель PRD

Создать безопасный hygiene/readiness слой для `Bot_data_base`, который:

1. Показывает реальное состояние источников:
   - active;
   - empty;
   - stale;
   - processing_stale;
   - archived;
   - orphaned.

2. Позволяет безопасно архивировать/скрывать мусорные источники через CLI и/или API, не затрагивая рабочую `КУЗНИЦУ ДУХА`.

3. Убирает legacy SD-разметку из Web Admin как активную пользовательскую/production-концепцию.

4. Деактивирует SD-фильтр в API/retrieval path или переводит его в явно deprecated compatibility mode.

5. Создаёт `reprocess_readiness_gate`, который отвечает на вопрос:

```text
Можно ли безопасно запускать future clean source reprocess?
```

6. Готовит отчёты и JSON-артефакты для архитектурного аудита следующего шага.

## 3. Non-goals

В этом PRD строго запрещено:

1. Не делать reprocess `КУЗНИЦЫ`.
2. Не делать Chroma reset/reindex.
3. Не применять review decisions к production KB.
4. Не менять `all_blocks_merged.json` как processed KB corpus.
5. Не менять Writer.
6. Не менять DiagnosticCard.
7. Не менять Thread Manager.
8. Не менять State Analyzer.
9. Не запускать Diagnostic Center.
10. Не удалять физически рабочий source raw markdown `КУЗНИЦА ДУХА v.2.md`.
11. Не удалять source files без backup/snapshot.
12. Не переписывать governance authority:
    - `chunk_type`;
    - `allowed_use`;
    - `safety_flags`;
    - `not_for_direct_quote`;
    - `source_style_not_user_facing`.
13. Не делать hard-delete источников с `blocks_count > 0`, кроме отдельного future PRD.
14. Не считать legacy SD metadata production-сигналом.

## 4. Архитектурное решение

### 4.1. Registry hygiene вместо ручного удаления файлов

Удаление/очистка не должна выполняться руками через файловую систему.

Нужно создать контролируемый механизм:

```text
registry inventory
→ candidate classification
→ dry-run archive/delete report
→ explicit confirm archive/delete
→ post-operation audit
```

Для источников с `0` blocks допустим safe-delete/soft-delete, но только после snapshot.

Для источников с `blocks_count > 0` в этом PRD допустим только `archive` / `quarantine`, не hard delete.

### 4.2. Active source allowlist

Для текущего этапа единственный production-ready KB-source:

```text
Кузница Духа
source_id примерно: 123__кузница_духа
blocks_count: 229
```

Все остальные источники должны быть классифицированы как:

```text
empty_test
stale_test
processing_stale
orphaned
unknown_review_required
```

Но не удаляться без явного режима.

### 4.3. SD decommission

SD/Spiral Dynamics больше не является активной системой классификации проекта.

Требование:

- убрать SD-график с dashboard;
- убрать SD-колонку из registry table;
- убрать SD как production-filter из API query route;
- если backward compatibility нужен, оставить только deprecated no-op / ignored path с trace warning;
- документационно зафиксировать: SD legacy fields больше не участвуют в retrieval, governance, review, Diagnostic Center readiness.

### 4.4. Reprocess readiness gate

До будущего reprocess система должна уметь ответить:

```json
{
  "ready_for_clean_reprocess": false,
  "blockers": [],
  "warnings": [],
  "active_source_count": 1,
  "active_source_id": "...",
  "active_source_title": "Кузница Духа",
  "legacy_sd_active": false,
  "registry_stale_sources_count": 0,
  "empty_sources_count": 0,
  "processing_stale_count": 0,
  "raw_source_found": true,
  "processed_blocks_count": 229,
  "chroma_count": 229,
  "recommendation": "..."
}
```

В этом PRD не нужно добиваться `ready=true` любой ценой. Нужно честно показать состояние и закрыть очевидные hygiene blockers.

### A2. Критерии classification

`is_focus_source=true`, если:

- title/source_id содержит `кузниц` + `дух`;
- blocks_count близок к `229`;
- source_type = `book`;
- raw markdown существует.

`is_test_like=true`, если:

- title/source_id содержит `test`, `test123`, `Книга` без нормального title;
- blocks_count = `0`;
- статус `processing` слишком долго;
- source_type book/youtube, но нет raw/processed данных;
- author = generic `Автор`.

`processing_stale=true`, если:

- status = `processing`;
- нет новых blocks;
- нет признаков активной обработки;
- created_at/updated_at старше разумного порога, если такие поля есть;
- если timestamp отсутствует, пометить `processing_unknown_age`.

### A3. Не ломать, если структура registry отличается

Если registry schema отличается от ожиданий, CLI не должен падать.

Он должен:

- записать degraded report;
- указать missing fields;
- сохранить safe summary.

## Workstream E — Reprocess Readiness Gate

### E1. Создать CLI

```bash
python Bot_data_base/tools/reprocess_readiness_gate.py --output-json TO_DO_LIST/logs/PRD-046.0.7-HF1/reprocess_readiness_gate.json --output-md TO_DO_LIST/reports/PRD-046.0.7-HF1_REPROCESS_READINESS_GATE_REPORT.md
```

### E2. Что проверять

Gate должен проверить:

```text
registry has exactly one production/focus source OR clearly identifies one focus source
focus source raw markdown exists
focus source processed blocks exist
processed blocks count == expected around 229
Chroma count == processed blocks count, если Chroma доступен
legacy SD UI/API active usage removed/disabled
test/empty/stale sources archived or at least classified
source_hygiene_plan exists
no pending unsafe hard delete
review queue exists
decisions_count may be 0 and это не blocker для reprocess readiness
```

### E3. Ready status

Gate должен иметь статусы:

```text
ready
not_ready
degraded
```

Пример blockers:

```text
focus_source_not_found
multiple_active_sources_without_allowlist
legacy_sd_filter_still_active
registry_has_unclassified_zero_block_sources
raw_markdown_missing
processed_blocks_missing
chroma_count_mismatch
```

Важно: если в этом PRD мы только классифицировали мусор, но не архивировали его, gate может вернуть `not_ready` с чёткой рекомендацией. Это допустимо, если apply не был безопасен.

14. Writer не изменён.
15. Diagnostic Center не запущен.
16. Chroma не reset/reindex в этом PRD.

- SD-график больше не отображается.

- Chroma count понятно.

### Upload book

- Нет SD-поля.
- Governance Profile есть.
- Source Kind есть.

### Upload YouTube

- Нет SD-поля.
- Governance Profile есть.
- Source Kind есть.

### Export merged JSON

- Экспорт работает.
- Экспорт не создаёт новый мусорный source.
- Экспорт не ломает registry.

## 11. Next PRD decision logic

После выполнения этого PRD следующий шаг выбирать по outcome.

### Outcome A — hygiene clean, SD decommission complete, readiness ready

Следующий PRD:

```text
PRD-046.0.8 — Clean Source Reprocess from Single Active Source v1
```

Цель:

- безопасно перепроцессить `КУЗНИЦУ` через актуальный pipeline;
- получить новые chunks/governance/boundaries/enrichment readiness;
- не включать Diagnostic Center.

### Outcome B — hygiene частично clean, но остались blockers

Следующий PRD:

```text
PRD-046.0.7-HF2 — Source Hygiene Blocker Fix v1
```

Цель:

- закрыть конкретные blockers;
- не делать reprocess до ready gate.

### Outcome C — SD decommission сломал API/UI

Следующий PRD:

```text
PRD-046.0.7-HF1-FIX — Legacy SD Decommission Regression Fix
```

Цель:

- восстановить API/UI без возврата SD как active signal.

### Outcome D — registry мутировать небезопасно

Следующий PRD:

```text
PRD-046.0.7-HF2 — Registry Storage Contract Stabilization v1
```

Цель:

- формализовать registry schema;
- добавить безопасные archive/delete contracts.

## 12. Важное правило для IDE-агента

Не пытайся "сделать красиво" через большой рефакторинг.

Главная задача этого PRD:

```text
сделать BotDB/Admin управляемым и готовым к будущему clean reprocess,
а не переписать всю базу знаний.
```

Любое сомнение трактовать в сторону безопасности:

```text
archive вместо delete
dry-run вместо apply
warning вместо silent mutation
degraded report вместо падения
```

## 13. Definition of done

Готово, когда:

1. Архитектор может открыть GitHub и увидеть:
   - что мусорные источники обнаружены/классифицированы;
   - что SD больше не active UI/API concept;
   - что есть readiness gate для reprocess;
   - что `Кузница` защищена;
   - что никакой reprocess/Chroma reset не был сделан случайно.

2. Пользователь может открыть Web Admin и визуально подтвердить:
   - SD исчезла из интерфейса;
   - registry стал понятнее;
   - тестовый мусор не маскируется под production KB.

3. Следующий PRD можно писать на основании конкретного readiness outcome, а не предположений.
