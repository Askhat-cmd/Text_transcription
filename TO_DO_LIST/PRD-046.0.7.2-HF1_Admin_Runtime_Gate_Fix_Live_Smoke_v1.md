# PRD-046.0.7.2-HF1 — Admin Runtime Gate Fix / Live Smoke v1

## 0. Назначение документа

Этот PRD описывает маленький hotfix после `PRD-046.0.7.2`.

Цель HF1 — закрыть единственный оставшийся blocker post-apply quality gate: недоступность live BotDB Admin API во время проверки.

Текущий статус проекта: `post-PRD-046.0.7.2-with-admin-api-blocker`.

`PRD-046.0.7.2` уже подтвердил:

- `data_consistency_passed=true`;
- `apply_route_consistency_passed=true`;
- `retrieval_quality_passed=true`;
- `writer_kb_policy_passed=true`;
- no-mutation proof чистый;
- production apply не выполнялся;
- Chroma reindex не выполнялся;
- provider не вызывался.

Но live admin API был недоступен:

- `/api/status` — connection refused;
- `/api/registry` — connection refused;
- `/api/dashboard` — connection refused;
- `/api/dashboard/` — connection refused.

Поэтому итоговый gate корректно вернул:

- `admin_runtime_status=blocked_admin_api_unavailable`;
- `admin_consistency_passed=false`;
- `quality_gate_passed=false`;
- `diagnostic_center_ready=false`.

HF1 должен проверить live API правильно: либо поднять его в controlled subprocess, либо использовать уже запущенный сервер, дождаться readiness и только после этого запускать smoke.

Разрешено:

## 2. Важная договорённость по запуску админки

HF1 должен поддерживать два режима.

### Режим A — админка уже запущена пользователем

Если пользователь заранее запустил BotDB Admin API на `http://127.0.0.1:8003`, агент должен:

1. Проверить readiness endpoint.
2. Не стартовать второй сервер на том же порту.
3. Выполнить live smoke.
4. Зафиксировать:
   - `server_launch_mode=external_existing`;
   - `server_started_by_hf1=false`.

### Режим B — агент сам пытается поднять админку

Если API не запущен, агент должен:

1. Найти канонический способ запуска admin backend в репозитории.
2. Запустить его как subprocess.
3. Дождаться readiness.
4. Выполнить live smoke.
5. Корректно остановить subprocess после проверки.
6. Зафиксировать:
   - `server_launch_mode=hf1_subprocess`;
   - `server_started_by_hf1=true`;
   - command;
   - stdout/stderr sanitized logs;
   - startup timeout;
   - shutdown result.

Если канонический запуск найти нельзя или сервер не стартует, HF1 не должен притворяться успешным. В этом случае статус:

```text
final_status=done_with_admin_launch_blocker
quality_gate_passed=false
diagnostic_center_ready=false
```

## 3. Ответ пользователю: нужно ли заранее запускать админку

Желательно — да.

Если админка будет заранее запущена, HF1 пройдёт проще и быстрее: агент просто проверит live endpoints.

Но PRD должен быть написан так, чтобы IDE-агент мог попробовать поднять backend сам. Поэтому отсутствие заранее запущенной админки не должно ломать проект, а должно давать честный blocker с понятными логами.

Практическое правило:

- если пользователь хочет максимально надёжный проход HF1 — перед запуском IDE-агента нужно поднять админку;
- если пользователь не хочет запускать вручную — IDE-агент должен попытаться сделать это сам;
- если не получилось — это не failure проекта, а `admin_launch_blocker`, который нужно исправить отдельно.

## 8. Endpoint validation contract

### `/api/status`

Acceptance:

- HTTP 200;
- JSON body;
- body содержит признаки живого backend:
  - `status`, `ok`, `service`, `version`, `ready`, `database`, `chroma`, или другой documented equivalent.

Если текущая схема отличается, агент должен не ломать backend ради теста, а документировать фактическую схему и валидировать минимально надёжный contract:

- endpoint доступен;
- JSON parse OK;
- нет error payload;
- backend готов отвечать.

### `/api/registry`

Acceptance:

- HTTP 200;
- JSON body;
- можно извлечь source rows;
- focus source `123__кузница_духа` найден;
- focus source status `done` или documented equivalent;
- `blocks_count = 247`.

### `/api/dashboard` and `/api/dashboard/`

Acceptance:

- оба endpoint возвращают HTTP 200;
- оба возвращают JSON;
- payload не пустой;
- dashboard содержит или позволяет вывести:
  - production/focus blocks count = `247`;
  - registry/focus status;
  - Chroma/admin visible count = `247`, если поле есть;
  - enrichment/review/apply visible status, если поле уже поддерживается;
  - нет silent empty card для ошибок.

Если `/api/dashboard` и `/api/dashboard/` возвращают разные, но совместимые схемы, агент должен описать это в отчёте.

## 12. Acceptance criteria

### Обязательные acceptance criteria

1. Live smoke больше не выполняет пустые вызовы в неизвестное состояние.
2. Если API выключен и agent не может его поднять, статус остаётся blocker, а не success.
3. Если API включён или успешно поднят agentом:
   - `/api/status` возвращает HTTP 200;
   - `/api/registry` возвращает HTTP 200;
   - `/api/dashboard` возвращает HTTP 200;
   - `/api/dashboard/` возвращает HTTP 200;
   - все payload JSON-валидны;
   - focus source `123__кузница_духа` найден;
   - focus blocks count = `247`;
   - dashboard не пустой;
   - admin runtime status = `passed`.
4. `quality_gate_passed=true` только если `admin_consistency_passed=true`.
5. `diagnostic_center_ready=true` только если весь quality gate passed.
6. Production data не мутируется.
7. Chroma reindex не выполняется.
8. Provider не вызывается.
9. Tests проходят.
10. docs/reports обновлены.

### Negative acceptance criteria

Следующие состояния запрещены:

```json
{
  "admin_runtime_status": "blocked_admin_api_unavailable",
  "admin_consistency_passed": true
}
```

```json
{
  "quality_gate_passed": true,
  "admin_consistency_passed": false
}
```

```json
{
  "diagnostic_center_ready": true,
  "quality_gate_passed": false
}
```
