# PRD-046.0.7.2-HF2 — Admin API Launch/Schema Fix / Live Gate Closure v1

## 0. Краткий смысл

После `PRD-046.0.7.2-HF1` проект остановился не из-за качества базы знаний, а из-за инфраструктурного blocker: Admin API не вышел в readiness при запуске через subprocess.

Пользователь сообщил, что после перезапуска IDE админка снова заработала. Поэтому этот hotfix должен **не выполнять новый apply**, **не менять KB**, **не делать Chroma reindex**, а повторить live admin gate на уже работающем backend и закрыть blocker, если `/api/status`, `/api/registry`, `/api/dashboard`, `/api/dashboard/` действительно доступны и возвращают ожидаемые данные.

Главная цель:

```text
external existing Admin API
→ live smoke
→ schema/count validation
→ no-mutation proof
→ quality gate closure
```

---

## 1. Текущий контекст

Текущий stage по `docs/PROJECT_STATE.md`:

```text
post-PRD-046.0.7.2-HF1-with-admin-launch-blocker
```

Состояние:

- controlled apply уже выполнен в `PRD-046.0.7.1`;
- production KB содержит `247` governed blocks для `123__кузница_духа`;
- Chroma collection также ожидается `247`;
- `PRD-046.0.7.2` подтвердил:
  - `data_consistency_passed=true`;
  - `apply_route_consistency_passed=true`;
  - `retrieval_quality_passed=true`;
  - `writer_kb_policy_passed=true`;
- `PRD-046.0.7.2-HF1` добавил live smoke tooling, но зафиксировал:

```text
admin_runtime_status=blocked_admin_launch_failed
quality_gate_passed=false
diagnostic_center_ready=false
```

Причина HF1 blocker:

```text
canonical launch command найден;
server_started_by_hf1=true;
readiness_passed=false;
/api/status, /api/registry, /api/dashboard, /api/dashboard/ недоступны.
```

Теперь пользователь сообщает, что IDE перезапущена и админка работает. Значит, HF2 должен сначала проверить именно режим:

```text
server_launch_mode = external_existing
```

## 2. Что НЕ делать

Строго запрещено в рамках HF2:

1. Не выполнять production apply.
2. Не изменять `all_blocks_merged.json`.
3. Не изменять registry, кроме документационных отчётов/логов PRD.
4. Не делать Chroma reindex.
5. Не вызывать LLM provider.
6. Не менять Writer, Context Assembly, Diagnostic Card, Thread Manager, State Analyzer.
7. Не запускать Diagnostic Center.
8. Не пытаться “исправлять” KB, если live admin smoke не прошёл.
9. Не считать gate пройденным, если хотя бы один обязательный endpoint недоступен.
10. Не выставлять `diagnostic_center_ready=true`, если admin runtime status не `passed`.

## 3. Основной сценарий HF2

### 3.1. Проверка уже работающей админки

IDE-агент должен выполнить live smoke против:

```text
http://127.0.0.1:8000
```

Обязательные endpoints:

```text
/api/status
/api/registry
/api/dashboard
/api/dashboard/
```

На первом шаге HF2 агент должен **не поднимать subprocess**, а проверить существующий backend:

```text
try_start_server=false
require_admin_api=true
allow_offline_admin_checks=false
```

Ожидаемый успешный режим:

```text
detected_existing_server=true
server_launch_mode=external_existing
server_started_by_hf1=false
readiness_passed=true
```

Если существующий сервер не найден, HF2 может сделать одну controlled попытку запуска через уже найденную canonical command, но это должен быть fallback, а не основной путь.

---

## 4. Что считать успешным live smoke

HF2 считается успешным только если:

```text
/api/status      -> HTTP 200 + валидный JSON
/api/registry    -> HTTP 200 + содержит source_id=123__кузница_духа, status=done, blocks_count=247
/api/dashboard   -> HTTP 200 + валидный dashboard payload
/api/dashboard/  -> HTTP 200 + валидный dashboard payload
```

Также должны быть подтверждены:

```text
focus_source_id = 123__кузница_духа
focus_blocks_count = 247
dashboard_blocks_count = 247 или совместимое поле active_source/production/focus blocks = 247
chroma_count = 247, если поле присутствует
```

Если `chroma_count` отсутствует в payload, это может быть warning, но не blocker, если:

- `/api/registry` корректен;
- dashboard payload валиден;
- data consistency из предыдущего gate сохранена;
- отдельный локальный Chroma count report показывает `247`.

## 5. Schema validation требования

HF2 должен не просто проверить HTTP 200, а проверить смысл payload.

### `/api/status`

Минимально допустимо:

```json
{
  "ok": true
}
```

или совместимый статусный payload, где явно видно, что backend жив.

### `/api/registry`

Должен позволять извлечь:

```text
source_id = 123__кузница_духа
status = done
blocks_count = 247
```

Допустимы оба формата:

```json
{
  "sources": [...]
}
```

или прямой список:

```json
[
  {...}
]
```

### `/api/dashboard` и `/api/dashboard/`

Оба endpoint должны быть доступны. Они могут возвращать не полностью идентичный payload, но оба должны быть валидными JSON и содержать данные, достаточные для admin overview.

Проверить минимум:

```text
payload is dict
payload не пустой
есть блоки/метрики/registry/dashboard sections или совместимые поля
нет error-only payload
```

---

## 6. Если API работает, но schema отличается

Если endpoints доступны, но schema не совпадает с тем, что ожидает checker, агент должен:

1. Сначала записать фактический payload shape в sanitized artifact.
2. Не печатать raw/full text из KB.
3. Не раскрывать секреты/ключи.
4. Различить:

```text
schema checker слишком строгий
```

и

```text
API реально отдаёт неправильные данные
```

5. Если проблема только в checker, исправить checker/test так, чтобы он поддерживал текущий валидный dashboard contract.
6. Если проблема в API, сделать минимальный backend fix только для admin endpoints.

Backend fix допускается только в read-only API layer. Нельзя менять production KB/Chroma/retrieval logic.

## 9. Acceptance criteria

### 9.1. Полный success

HF2 считается успешно закрытым, если:

```text
admin_runtime_status=passed
admin_consistency_passed=true
quality_gate_passed=true
diagnostic_center_ready=true
```

И при этом:

```text
/api/status reachable
/api/registry reachable
/api/dashboard reachable
/api/dashboard/ reachable
focus_blocks_count=247
dashboard_blocks_count=247 или accepted-compatible dashboard count proof
chroma_count=247 или accepted-compatible local Chroma count proof
all_blocks_merged_mutated=false
registry_mutated=false
production_apply_performed=false
chroma_reindex_performed=false
provider_called=false
```

Важно: `diagnostic_center_ready=true` здесь означает только, что можно переходить к **проектированию Diagnostic Center PRD**, а не что Diagnostic Center уже запущен.

### 9.2. Если снова blocker

Если endpoints недоступны, статус должен быть честным:

```text
final_status=done_with_admin_api_blocker
```

или:

```text
final_status=done_with_admin_launch_blocker
```

Если endpoints доступны, но payload/schema некорректны:

```text
final_status=done_with_admin_schema_blocker
```

В любом blocker-сценарии:

```text
quality_gate_passed=false
diagnostic_center_ready=false
```

## 11. Implementation notes для IDE-агента

1. Сначала проверить живой сервер:

```text
GET http://127.0.0.1:8000/api/status
```

2. Если ответ `200`, работать в режиме:

```text
external_existing
```

3. Не убивать существующий сервер после smoke, если он был запущен пользователем/IDE вручную.

4. Если сервер не жив, можно fallback:

```text
python -m uvicorn api.main:app --host 127.0.0.1 --port 8000
```

из директории:

```text
Bot_data_base
```

5. Если fallback не стартует, обязательно сохранить sanitized launch diagnostics:

```text
launch command
cwd
readiness attempts
last error
sanitized stdout/stderr tail
```

6. Не писать в отчёты секреты, ключи, raw/full KB text.
