# PRD-046.0.7.2-HF3 — Dashboard Chroma Count Reconciliation / Strict Gate v1

## 0. Краткий смысл

После `PRD-046.0.7.2-HF2` Admin API наконец стал доступен на актуальном порту `8003`, endpoints вернули `200`, и gate был формально переведён в `passed`.

Но в live dashboard обнаружено важное расхождение:

```text
production blocks / registry / dashboard blocks = 247
live dashboard chroma.count = 229
```

При этом HF2 засчитал gate как passed, потому что принял старый local proof `247` из предыдущего артефакта. Это небезопасно: live dashboard не должен считаться зелёным, если сам показывает `chroma.count=229`.

Цель HF3:

```text
live Admin API на 8003
→ строгая проверка live dashboard/chroma count
→ запрет pass через stale local proof
→ либо честный passed при live chroma=247,
   либо честный blocker при chroma mismatch
```

HF3 не должен запускать Diagnostic Center. HF3 только закрывает или честно блокирует readiness gate.

## 2. Главная проблема

Сейчас readiness gate может быть ложнозелёным:

```text
admin_runtime_status = passed
quality_gate_passed = true
diagnostic_center_ready = true
```

при том, что live dashboard сообщает:

```text
chroma.count = 229
```

А expected state после reprocess/apply:

```text
production blocks = 247
registry focus blocks = 247
expected chroma count = 247
```

Такой gate нельзя считать достаточным для перехода к Diagnostic Center.

---

## 3. Цель HF3

Сделать strict live gate, в котором:

```text
quality_gate_passed = true
diagnostic_center_ready = true
```

разрешены только если live runtime сам подтверждает:

```text
/api/status 200
/api/registry 200
/api/registry/ 200 или совместимый fallback
/api/dashboard 200
/api/dashboard/ 200

registry focus source blocks_count = 247
dashboard blocks.production_total = 247
dashboard blocks.active_source_blocks = 247
dashboard blocks.registry_total = 247

dashboard chroma.status = ok
dashboard chroma.count = 247
dashboard chroma.indexed_source_ids contains only/at least 123__кузница_духа
```

Если live dashboard всё ещё показывает `229`, результат должен быть:

```text
final_status = done_with_chroma_count_blocker
quality_gate_passed = false
diagnostic_center_ready = false
admin_runtime_status = blocked_chroma_count_mismatch
```

## 5. Обязательная логика strict gate

### 5.1. Запуск

HF3 должен использовать актуальный URL:

```text
http://127.0.0.1:8003
```

CLI должен иметь параметр:

```text
--admin-base-url http://127.0.0.1:8003
```

Default для HF3 должен быть `8003`, а не `8000`.

Если API недоступен:

```text
final_status = done_with_admin_api_blocker
quality_gate_passed = false
diagnostic_center_ready = false
```

### 5.2. Endpoints

Проверить:

```text
GET /api/status
GET /api/registry
GET /api/registry/
GET /api/dashboard
GET /api/dashboard/
```

`/api/registry/` добавлен специально, потому что актуальная пользовательская проверка использует именно trailing slash.

Допустимо, если `/api/registry` и `/api/registry/` оба работают.
Если один из них не поддерживается, но второй работает, это не обязательно blocker, но должно быть явно зафиксировано как compatibility warning.
Для `/api/dashboard` и `/api/dashboard/` оба должны работать, потому что ранее оба endpoint были частью acceptance.

### 5.3. Count contract

Строгий contract:

```text
expected_source_id = 123__кузница_духа
expected_blocks = 247
expected_chroma_count = 247
```

Проверить:

```text
registry.focus_source.blocks_count == 247
dashboard.blocks.production_total == 247
dashboard.blocks.active_source_blocks == 247
dashboard.blocks.registry_total == 247
dashboard.chroma.count == 247
dashboard.chroma.indexed_source_ids includes 123__кузница_духа
```

Если `dashboard.chroma.count != 247`:

```text
issues += ["dashboard_chroma_count_mismatch"]
admin_runtime_status = "blocked_chroma_count_mismatch"
quality_gate_passed = false
diagnostic_center_ready = false
```

### 5.4. Запрет stale proof override

Удалить/запретить логику:

```text
dashboard_chroma_count_mismatch_accepted_with_local_proof
```

HF3 может использовать old/local proof только как diagnostic evidence, но не как pass condition.

Правило:

```text
historical proof cannot override live dashboard mismatch
```

Если old proof говорит `247`, а live dashboard говорит `229`, результат должен быть blocker:

```text
live_chroma_count = 229
historical_chroma_count = 247
chroma_reconciliation_status = "live_historical_mismatch"
strict_gate_passed = false
```

Добавить read-only artefact:

```text

```

Он должен содержать:

```json
{
  "schema_version": "dashboard_chroma_count_reconciliation_v1",
  "source_prd": "PRD-046.0.7.2-HF3",
  "admin_base_url": "http://127.0.0.1:8003",
  "expected_source_id": "123__кузница_духа",
  "expected_blocks": 247,
  "registry_focus_blocks": 247,
  "dashboard_blocks_count": 247,
  "dashboard_chroma_count": 229,
  "direct_chroma_count": null,
  "historical_chroma_count": 247,
  "historical_proof_path": "...",
  "reconciliation_status": "blocked_live_dashboard_chroma_mismatch",
  "strict_gate_passed": false,
  "issues": ["dashboard_chroma_count_mismatch"],
  "warnings": []
}
```

Если direct read-only Chroma count доступен:

```text
direct_chroma_count = actual direct count
```

Тогда логика:

```text
case A:
dashboard_chroma_count = 247
direct_chroma_count = 247 или null
→ passed

case B:
dashboard_chroma_count = 229
direct_chroma_count = 247
→ dashboard_count_source_bug
→ можно исправить read-only dashboard count source
→ повторить live smoke
→ passed только если live dashboard стал 247

case C:
dashboard_chroma_count = 229
direct_chroma_count = 229
→ actual_chroma_stale_or_not_reindexed
→ no mutation in HF3
→ recommend next PRD for controlled Chroma reindex/recovery

case D:
dashboard_chroma_count = null
→ dashboard_chroma_schema_missing
→ blocker
```

## 9. Acceptance criteria

### 9.1. Passed case

HF3 считается полностью passed только если:

```text
implementation = done
final_status = passed
admin_runtime_status = passed
quality_gate_passed = true
diagnostic_center_ready = true

admin_base_url = http://127.0.0.1:8003
server_launch_mode = external_existing или valid_live_existing

/api/status = 200
/api/registry = 200
/api/registry/ = 200 или compatibility warning при рабочем /api/registry
/api/dashboard = 200
/api/dashboard/ = 200

registry_focus_blocks = 247
dashboard_blocks_count = 247
dashboard_chroma_count = 247
strict_chroma_count_passed = true

issues = []
no stale proof override warnings
```

### 9.2. Blocker case

Если live dashboard продолжает показывать `chroma.count=229`, HF3 всё равно может быть реализован корректно, но статус должен быть blocker:

```text
implementation = done_with_chroma_count_blocker
final_status = done_with_chroma_count_blocker
admin_runtime_status = blocked_chroma_count_mismatch
quality_gate_passed = false
diagnostic_center_ready = false

issues includes dashboard_chroma_count_mismatch
next_prd = PRD-046.0.7.2-HF4 — Chroma Count Recovery / Controlled Reindex v1
```

Это не провал IDE-агента, если он честно зафиксирует mismatch и не будет делать reindex в HF3.
