# PRD-046.0.7.2-HF4 — Chroma Count Recovery / Controlled Reindex v1

## 0. Краткий смысл

После `PRD-046.0.7.2-HF3` live Admin API доступен на актуальном порту `8003`, но strict gate честно заблокирован из-за расхождения:

```text
registry_focus_blocks = 247
dashboard_blocks_count = 247
dashboard_chroma_count = 229
expected_blocks = 247
strict_chroma_count_passed = false
diagnostic_center_ready = false
```

Цель HF4 — не «сделать зелёную цифру», а безопасно выяснить источник расхождения и восстановить согласованность Chroma/Dashboard/Registry.

Итоговое целевое состояние:

```text
live /api/dashboard chroma.count = 247
direct_chroma_count = 247
registry_focus_blocks = 247
dashboard_blocks_count = 247
indexed_source_ids = ["123__кузница_духа"]
source set = only focus source
quality_gate_passed = true
diagnostic_center_ready = true
```


## 3. Что НЕ делать

Строго запрещено в HF4:

1. Не менять `all_blocks_merged.json`.
2. Не менять `chunk_type`, `allowed_use`, `safety_flags`, `source_id`, `block_id`, `text`.
3. Не выполнять LLM enrichment.
4. Не вызывать LLM provider.
5. Не менять Writer, Context Assembly, Diagnostic Card, Diagnostic Center.
6. Не запускать Diagnostic Center.
7. Не считать gate пройденным, если live `/api/dashboard` показывает `chroma.count != 247`.
8. Не подменять live Chroma count старым historical proof.
9. Не делать reindex без backup.
10. Не reindex-ить тестовые/архивные/zero-block источники.

---

## 4. Главные сценарии HF4

HF4 должен различить три ситуации.

### Сценарий A — Dashboard считает не тот Chroma

Если direct Chroma diagnostic показывает:

```text
direct_chroma_count = 247
dashboard_chroma_count = 229
```

то проблема не в Chroma, а в dashboard/API count source.

Тогда нужно:

```text
исправить dashboard/API источник подсчёта
не делать reindex
перезапустить strict live gate
добиться dashboard_chroma_count = 247
```

### Сценарий B — Chroma реально содержит 229

Если direct Chroma diagnostic показывает:

```text
direct_chroma_count = 229
dashboard_chroma_count = 229
expected_blocks = 247
```

то нужна controlled Chroma recovery / reindex.

Тогда нужно:

```text
backup Chroma
preflight authoritative blocks = 247
reindex только focus source
post-reindex direct_chroma_count = 247
live dashboard_chroma_count = 247
retrieval smoke
strict gate pass
```

### Сценарий C — direct Chroma count недоступен

Если direct Chroma diagnostic снова даёт:

```text
direct_chroma_count = null
```

то нельзя выполнять recovery вслепую.

Тогда итог:

```text
final_status = done_with_direct_chroma_diagnostic_blocker
quality_gate_passed = false
diagnostic_center_ready = false
```

Отчёт обязан объяснить, почему direct diagnostic недоступен: import error, path error, missing chromadb, wrong collection path, config mismatch, locked SQLite, etc.

## 5. Source hygiene preflight

Перед любым reindex/recovery нужно проверить live registry:

```text
/api/registry
/api/registry/
```

Acceptance для целевого состояния:

```text
sources_total = 1
focus_source_id = 123__кузница_духа
focus_author = Саламат Сарсекенов
focus_title = Кузница Духа
focus_status = done
focus_blocks = 247
focus_protected = true
non_focus_sources_count = 0
non_focus_blocks_count = 0
```

Если registry всё ещё содержит test/failed/processing rows, HF4 должен:

- не reindex-ить их;
- отметить `source_hygiene_warning` или `source_hygiene_blocker`;
- если non-focus rows имеют `blocks_count > 0`, hard-block recovery;
- если non-focus rows zero-block/test-like, можно продолжить reindex focus-only, но итоговый dashboard должен явно показывать, что Chroma содержит только focus source.

Поскольку пользователь уже удалил лишние источники вручную, ожидаемое live-состояние — один источник.

## 6. Необходимые новые/обновлённые инструменты

### 6.1 Direct Chroma diagnostic

Создать или расширить tool:

```text
Bot_data_base/tools/diagnose_chroma_runtime_count.py
```

Он должен:

1. Использовать тот же Chroma persist path / collection name, что live API.
2. Считать:
   - total count;
   - indexed source ids;
   - count by source id;
   - collection name;
   - persist directory;
   - embedding dimension, если доступно;
   - sample ids, безопасно без raw text.
3. Не выводить raw/full text.
4. Не мутировать Chroma.
5. Писать отчёт:

```text
TO_DO_LIST/logs/PRD-046.0.7.2-HF4/direct_chroma_diagnostic.json
```

Ожидаемая структура:

```json
{
  "schema_version": "direct_chroma_diagnostic_v1",
  "source_prd": "PRD-046.0.7.2-HF4",
  "status": "ok",
  "collection_name": "...",
  "persist_directory": "...",
  "total_count": 247,
  "source_ids": ["123__кузница_духа"],
  "count_by_source_id": {
    "123__кузница_духа": 247
  },
  "sample_ids_count": 10,
  "raw_text_leak_detected": false,
  "errors": [],
  "warnings": []
}
```

### 6.2 Controlled Chroma reindex

Создать или расширить tool:

```text
Bot_data_base/tools/reindex_focus_source_chroma_controlled.py
```

Только если direct count подтвердил actual mismatch.

Preflight:

```text
all_blocks_total = 247
focus_source_blocks = 247
source_ids = ["123__кузница_духа"]
registry_focus_blocks = 247
dashboard_blocks_count = 247
direct_chroma_count_before = 229
```

Backup:

```text
TO_DO_LIST/backups/PRD-046.0.7.2-HF4/chroma_before_reindex/
TO_DO_LIST/logs/PRD-046.0.7.2-HF4/chroma_backup_manifest.json
```

Reindex:

- удалить/пересобрать только focus source в целевой Chroma collection;
- не добавлять test/failed/processing sources;
- использовать production embedding pipeline, а не mock/test vectors;
- не создавать коллекцию с неверной dimensionality;
- не трогать registry и all_blocks.

Post-check:

```text
direct_chroma_count_after = 247
dashboard_chroma_count_after = 247
indexed_source_ids_after = ["123__кузница_духа"]
retrieval_smoke_passed = true
```

### 6.3 Strict gate rerun

Создать runner:

```text
Bot_data_base/tools/run_chroma_recovery_hf4.py
```

Он должен объединять:

1. source hygiene preflight;
2. direct chroma diagnostic;
3. decision:
   - dashboard count source fix;
   - controlled reindex;
   - diagnostic blocker;
4. live admin smoke after fix/reindex;
5. retrieval smoke;
6. no-mutation proof for `all_blocks_merged.json` and `registry.json`;
7. reports/docs update.




Если source hygiene не focus-only и есть non-focus blocks > 0:

