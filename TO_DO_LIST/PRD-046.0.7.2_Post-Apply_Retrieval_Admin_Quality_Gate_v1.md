# PRD-046.0.7.2 — Post-Apply Retrieval/Admin Quality Gate v1

## 1. Назначение PRD

Этот PRD нужен как **финальный контроль качества после controlled apply** из `PRD-046.0.7.1`.

Важно: это **не новый apply**, не reprocess, не enrichment-run и не запуск Diagnostic Center.

Задача PRD — честно проверить, что после применения validated review decisions:

- production KB осталась согласованной;
- `metadata.llm_enrichment` применена только там, где было разрешено;
- `defer/rejected` решения не были применены;
- retrieval не начал вытаскивать опасные/цитатные/сырьевые фрагменты;
- practice guardrails сохранились;
- admin/API smoke не даёт ложный `passed=true`, если сервер не запущен;
- проект действительно готов к следующему архитектурному этапу.

```text

```

Это недопустимо как финальный readiness gate. Если API не поднят, gate не должен считаться зелёным.

## 3. Главная проблема

Сейчас технически apply прошёл корректно, но перед переходом к Diagnostic Center нельзя опираться на неполный smoke.

Нужно отделить:

```text
data-level consistency passed
```

от

```text
admin/API runtime consistency passed
```

Если локальный API недоступен, это не обязательно означает, что система сломана. Но это означает, что **admin/runtime gate не подтверждён**.

Правильные статусы:

```text
passed
blocked_admin_api_unavailable
skipped_offline_explicit
failed
```

А не:

```text
connection refused + passed=true
```

---

## 4. Цель

Сделать post-apply quality gate, который:

1. Не меняет production data.
2. Не делает Chroma reindex.
3. Не вызывает LLM provider.
4. Проверяет data consistency.
5. Проверяет retrieval safety.
6. Проверяет Writer KB policy smoke.
7. Проверяет Admin/API runtime честно.
8. Обновляет project docs только по фактическому статусу.
9. Блокирует Diagnostic Center, если quality gate не прошёл полностью.

## 5. Scope

### 5.1 Новый post-apply quality gate модуль

Добавить модуль, например:

```text
Bot_data_base/review/post_apply_quality_gate.py
```

или другое имя в существующей структуре `Bot_data_base/review/`.

Он должен уметь:

- читать `all_blocks_merged.json`;
- читать `registry.json`;
- читать artifacts из `PRD-046.0.7.1`;
- читать decisions overlay из `PRD-046.0.9.3`;
- проверять apply routes;
- проверять no-authority mutation proof;
- проверять retrieval smoke;
- проверять admin/API endpoints;
- формировать итоговый quality gate snapshot.

### 5.2 CLI

Добавить CLI, например:

```text
Bot_data_base/tools/run_post_apply_quality_gate.py
```

Ожидаемый пример запуска:

```bash
python Bot_data_base/tools/run_post_apply_quality_gate.py ^
  --source-prd PRD-046.0.7.2 ^
  --blocks Bot_data_base/data/processed/all_blocks_merged.json ^
  --registry Bot_data_base/data/registry.json ^
  --apply-result TO_DO_LIST/logs/PRD-046.0.7.1/apply_result.json ^
  --overlay TO_DO_LIST/logs/PRD-046.0.9.3/architect_auto_decisions_overlay.json ^
  --review-queue TO_DO_LIST/logs/PRD-046.0.9-RUN1/review_queue_after_real_enrichment.json ^
  --out-dir TO_DO_LIST/logs/PRD-046.0.7.2
```

Дополнительные флаги:

```text
--admin-base-url http://127.0.0.1:8000
--require-admin-api
--allow-offline-admin-checks
```

Правило:

- по умолчанию admin/API strict mode;
- если API недоступен — `admin_runtime_status=blocked_admin_api_unavailable`;
- `quality_gate_passed=false`;
- `diagnostic_center_ready=false`;
- если специально передан `--allow-offline-admin-checks`, тогда admin status может быть `skipped_offline_explicit`, но итоговый readiness всё равно не должен открывать Diagnostic Center.

### 5.3 Data consistency gate

Проверить:

```text
blocks_total = 247
registry_focus_blocks = 247
source_ids = ["123__кузница_духа"]
governance_present_rate = 1.0
allowed_use_present_rate = 1.0
safety_flags_present_rate = 1.0
```

Проверить, что после `PRD-046.0.7.1`:

```text
text_changed_count = 0
chunk_type_changed_count = 0
allowed_use_changed_count = 0
safety_flags_changed_count = 0
source_id_changed_count = 0
block_id_changed_count = 0
governance_invariant_violations = 0
```

### 5.4 Apply route consistency

Проверить expected route counts:

```text
safe_non_review_apply = 160
review_approved_apply = 28
review_needs_edit_apply = 12
review_rejected_skip = 1
review_defer_skip = 46
updated_blocks = 200
```

Проверить, что:

```text
approved -> applied
needs_edit -> edited_fields applied
rejected -> not applied
defer -> not applied
```

Если блоки `rejected/defer` имеют старую enrichment metadata от более раннего apply, это не должно автоматически считаться ошибкой. Ошибка — если они были изменены именно текущим controlled apply route.

Поэтому желательно проверять не просто наличие `metadata.llm_enrichment`, а route/provenance markers, например:

```text
metadata.llm_enrichment.applied_by_prd
metadata.llm_enrichment.apply_route
metadata.llm_enrichment.review_decision
metadata.llm_enrichment.source_overlay_prd
```

Если таких markers нет — добавить warning и предложить HF, но не мутировать данные в этом PRD.

### 5.5 Retrieval smoke

Проверить retrieval по нескольким темам:

```text
страх
тень
границы
практика
внутренний голос
```

Минимальные условия:

```text
retrieval_smoke_passed=true
hits_count > 0 для ключевых тем
quote_violation_ids=[]
practice_guardrail_missing_ids=[]
internal_only_unsafe_exposure_count=0
raw_full_text_leak_detected=false
```

Также проверить, что retrieval не подсовывает пользователю `КУЗНИЦУ` как прямой цитатник.

### 5.6 Writer KB policy smoke

Добавить лёгкий smoke, который проверяет политику использования KB в Writer/knowledge policy:

- Writer не должен прямо цитировать `КУЗНИЦУ`;
- Writer не должен ссылаться на источник как на авторитет;
- practice-блоки не должны предлагаться без low-resource check;
- snippets не должны обрезаться посреди слова;
- internal lens должен использоваться как мягкий контекст, а не как директива.

Это может быть deterministic/unit-style check без LLM provider.

### 5.7 Admin/API runtime gate

Проверить endpoints:

```text
/api/status
/api/registry
/api/dashboard
/api/dashboard/
```

Ожидаемое поведение:

- если сервер доступен:
  - `/api/status` возвращает 200;
  - `/api/registry` возвращает focus source с `247`;
  - `/api/dashboard` и `/api/dashboard/` возвращают согласованные counters;
  - dashboard не скрывает enrichment/apply state;
  - оба slash/no-slash варианта работают.
- если сервер недоступен:
  - `admin_runtime_status=blocked_admin_api_unavailable`;
  - `admin_consistency_passed=false`;
  - `quality_gate_passed=false`;
  - в отчёте явно написать команду/подсказку для запуска локального API, если такая команда известна проекту.

Запрещено:

```text
connection refused + admin_consistency_passed=true
```

## 8. Acceptance criteria

### 8.1 Если API доступен

PRD считается `done`, если:

```text
quality_gate_passed=true
data_consistency_passed=true
apply_route_consistency_passed=true
retrieval_quality_passed=true
writer_kb_policy_passed=true
admin_consistency_passed=true
diagnostic_center_ready=true
```

При этом:

```text
all_blocks_merged_mutated=false
registry_mutated=false
chroma_reindex_performed=false
provider_called=false
```

### 8.2 Если API недоступен

PRD считается не полностью закрытым, а:

```text
done_with_admin_api_blocker
```

Допустимый итог:

```text
data_consistency_passed=true
apply_route_consistency_passed=true
retrieval_quality_passed=true
writer_kb_policy_passed=true
admin_consistency_passed=false
admin_runtime_status=blocked_admin_api_unavailable
quality_gate_passed=false
diagnostic_center_ready=false
```

Это не провал кода, но это блокер перехода к Diagnostic Center.

### 8.3 Недопустимый итог

Нельзя принимать PRD как зелёный, если:

```text
connection refused
admin_consistency_passed=true
quality_gate_passed=true
```

## 12. Короткая инструкция IDE-агенту

Работай как инженер качества, а не как разработчик новой фичи.

Твоя задача — не “сделать зелёный отчёт”, а честно показать состояние системы.

Если API недоступен — это должен быть blocker, а не passed.

Не мутируй базу.

Не запускай Diagnostic Center.

Не делай reindex.

Не вызывай LLM.

Главный результат этого PRD — надёжный quality gate, которому можно доверять перед следующим крупным этапом.
