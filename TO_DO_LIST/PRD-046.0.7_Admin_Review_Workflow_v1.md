## 1. Контекст

После цепочки `PRD-046.0.5 → PRD-046.0.5-HF1 → PRD-046.0.5-RUN1 → HF2 → HF3 → APPLY1` в проекте появился controlled LLM enrichment overlay.

Ключевое состояние после APPLY1:

```text
overlay_rows_total = 60
matched_blocks = 60
updated_blocks = 60
text_changed_count = 0
chunk_type_changed_count = 0
allowed_use_changed_count = 0
safety_flags_changed_count = 0
governance_invariant_violations = 0
metadata_llm_enrichment_present_count = 60
needs_human_review_count = 60
```

Это означает:

```text
LLM enrichment уже применён как advisory metadata,
но все элементы controlled batch всё ещё требуют human review.
```

После `PRD-046.0.6-HF1` retrieval safety gate стабилизирован:

```text
internal_only_unsafe_exposure_count = 0
queries_failed = 0
top5_semantic_match_rate = 1.0
top5_governance_present_rate = 1.0
raw_full_text_leak_detected = false
```

Следовательно, текущий bottleneck проекта — не retrieval и не Writer, а отсутствие управляемого human-review процесса для KB enrichment и reviewable governance artifacts.

---

## 2. Архитектурное решение

Следующий слой должен быть:

```text
PRD-046.0.7 — Admin Review Workflow v1
```

Это **не Diagnostic Center** и **не Web Admin UI**.

Цель этого PRD — создать минимальный, безопасный, CLI/report-based workflow, который позволяет:

```text
1. собрать очередь review items из KB/enrichment metadata;
2. показать человеку только sanitized preview и review metadata;
3. принять review decisions в отдельный overlay-файл;
4. провалидировать decisions;
5. ничего не применять к production KB в рамках этого PRD.
```

Ключевой принцип:

```text
Review workflow сначала должен научиться безопасно показывать,
валидировать и хранить решения.

Применение этих решений к KB/Chroma — отдельный будущий PRD.
```

---

## 3. Главная цель

Создать минимальный управляемый workflow для human review KB enrichment artifacts.

В результате PRD должен появиться слой, который:

- строит read-only review queue;
- формирует `ReviewItem` contract;
- формирует `ReviewDecision` contract;
- валидирует review decisions;
- запрещает raw full text leaks;
- создаёт Markdown/JSON отчёты;
- не мутирует `all_blocks_merged.json`;
- не меняет Chroma;
- не меняет runtime бота;
- не включает Diagnostic Center.

---

### 5.2. ReviewItem contract

В `Bot_data_base/review/review_contracts.py` описать минимальные dataclass/Pydantic-like контракты.

Можно использовать `dataclasses` + explicit validation, чтобы не добавлять новые зависимости.

Обязательная структура `ReviewItem`:

```json
{
  "review_item_id": "string",
  "block_id": "string",
  "source_id": "string",
  "source_title": "string",
  "chunk_type": "lens|practice|case|theory|style|safety|architecture|unknown",
  "allowed_use": ["writer_context", "diagnostic_lens"],
  "safety_flags": ["not_for_direct_quote"],
  "lens_family": ["..."],
  "content_preview": "sanitized preview <= 240 chars",
  "llm_enrichment": {
    "schema_version": "string",
    "summary": "string",
    "lens_family_candidates": ["..."],
    "tags": ["..."],
    "use_when": ["..."],
    "avoid_when": ["..."],
    "self_contained_score": 0.0,
    "self_contained_reason": "string",
    "split_merge_suggestion": "string",
    "confidence": 0.0,
    "needs_human_review": true,
    "review_status": "string",
    "review_reasons": ["..."],
    "llm_metadata": {
      "provider": "string",
      "model": "string",
      "prompt_version": "string",
      "mock": false
    }
  },
  "review_priority": "P0|P1|P2",
  "review_reasons": ["..."],
  "recommended_action": "approve|reject|needs_edit|split_merge_review|defer",
  "source_prd": "PRD-046.0.7"
}
```

Требования:

- `review_item_id` должен быть deterministic.
- Рекомендуемый алгоритм:

```text
review_item_id = "kbri_" + sha1(source_id + "|" + block_id + "|" + schema_version)[:16]
```

- `content_preview` не длиннее 240 символов.
- `content_preview` должен быть sanitized:
  - whitespace collapsed;
  - без markdown blockquote больших фрагментов;
  - без raw full text;
  - без секретов / ключей / `.env`-подобных строк;
  - без переносов большого текста.

### 5.4. Sanitizer

Создать `Bot_data_base/review/review_sanitizer.py`.

Функции минимум:

```python
sanitize_preview(text: str, limit: int = 240) -> str
contains_forbidden_review_key(payload: Any) -> bool
find_forbidden_review_keys(payload: Any) -> list[str]
contains_secret_like_value(text: str) -> bool
assert_review_artifact_is_sanitized(payload: Any) -> None
```

Требования:

- `sanitize_preview` должен:
  - схлопывать whitespace;
  - обрезать до `limit`;
  - возвращать строку без многострочных raw fragments;
  - не возвращать `None`.
- `find_forbidden_review_keys` должен рекурсивно проверять dict/list.
- `contains_secret_like_value` должен искать хотя бы базовые паттерны:
  - `OPENAI_API_KEY=`;
  - `sk-`;
  - `api_key`;
  - `password`;
  - `token=`;
  - `.env`.
- Review queue generator обязан прогонять output через sanitizer assertion перед записью файла.

## 6. CLI: Review Queue Builder

Создать CLI:

```text
Bot_data_base/tools/kb_review_queue.py
```

Назначение:

```text
Собрать read-only queue из processed KB blocks и enrichment metadata.
```

Рекомендуемые аргументы:

```bash
python Bot_data_base/tools/kb_review_queue.py \
  --output-json TO_DO_LIST/logs/PRD-046.0.7/review_queue.json \
  --output-md TO_DO_LIST/reports/PRD-046.0.7_REVIEW_QUEUE_REPORT.md \
  --template-json TO_DO_LIST/logs/PRD-046.0.7/review_decisions_template.json \
  --max-items 200 \
  --source-prd PRD-046.0.7
```

CLI должен иметь default discovery входного файла.

Рекомендуемый порядок поиска:

```text
Bot_data_base/data/processed/all_blocks_merged.json
Bot_data_base/data/processed/*all_blocks*.json
Bot_data_base/data/**/all_blocks_merged.json
```

Если файл не найден:

- не падать с traceback;
- выдать понятное сообщение;
- записать degraded report;
- вернуть non-zero exit code.

### 6.1. Источники попадания в queue

Review item должен попадать в queue, если выполняется хотя бы одно условие:

```text
metadata.llm_enrichment.needs_human_review = true
metadata.llm_enrichment.review_status contains "review"
metadata.llm_enrichment.review_reasons not empty
metadata.llm_enrichment.confidence < 0.80
metadata.llm_enrichment.self_contained_score < 0.75
metadata.llm_enrichment.split_merge_suggestion not empty and not "keep"
chunking_quality.mixed_intent_severity in ["medium", "high"]
retrieval weak-case marker exists, если такой marker уже хранится в metadata
```

Если в текущих данных есть 60 элементов APPLY1 с `needs_human_review=true`, queue должна их обнаружить.

### 6.2. Priority policy

Реализовать deterministic priority:

#### P0

```text
- safety_flags contain clinical/safety risk
- allowed_use contains safety_protocol
- split_merge_suggestion требует split/merge
- confidence < 0.65
- self_contained_score < 0.60
- mixed_intent_severity = high
- review_reasons contain quote/safety/low_resource/hard_validation
```

#### P1

```text
- needs_human_review = true
- confidence < 0.80
- self_contained_score < 0.75
- review_reasons not empty
- avoid_when missing or suspicious
- use_when missing
```

#### P2

```text
- soft warning only
- confidence >= 0.80
- no hard safety concern
```

### 6.3. Recommended action policy

Deterministic `recommended_action`:

```text
split_merge_review:
  если split_merge_suggestion suggests split/merge или mixed_intent_severity=high

needs_edit:
  если summary/use_when/avoid_when/tags неполные или review_reasons требуют правки

reject:
  если есть hard quote/safety/unsafe/directive risk в enrichment metadata

approve:
  если confidence high, review_reasons empty, no safety/mixed issue

defer:
  если недостаточно данных для решения
```

Важно: `recommended_action` — рекомендация, не решение.

## 7. CLI: Review Decisions Validator

Создать CLI:

```text
Bot_data_base/tools/kb_review_decisions.py
```

Назначение:

```text
Проверить decision overlay на валидность и безопасность.
```

Рекомендуемый запуск:

```bash
python Bot_data_base/tools/kb_review_decisions.py \
  --queue-json TO_DO_LIST/logs/PRD-046.0.7/review_queue.json \
  --decisions-json TO_DO_LIST/logs/PRD-046.0.7/review_decisions_template.json \
  --output-json TO_DO_LIST/logs/PRD-046.0.7/review_decisions_validation.json \
  --output-md TO_DO_LIST/reports/PRD-046.0.7_REVIEW_DECISIONS_VALIDATION_REPORT.md \
  --validate-only
```

### 7.1. Validator должен проверять

1. `review_item_id` существует в queue.
2. `block_id` соответствует queue item.
3. `decision` входит в allowed enum.
4. `reviewer` входит в allowed enum.
5. `reason` не пустой для `rejected`, `needs_edit`, `defer`.
6. `approved_fields`, `rejected_fields`, `edited_fields` содержат только allowed advisory fields.
7. Нет forbidden keys.
8. Нет secret-like values.
9. Нет raw full text.
10. `created_at` валидный ISO-8601 или автоматически normalizable.
11. `source_prd = PRD-046.0.7`.

### 7.2. Decision overlay не применяется

В этом PRD validator только валидирует.

Он не должен:

- писать в KB;
- менять `all_blocks_merged.json`;
- менять Chroma;
- менять metadata blocks;
- запускать reindex.

```text
LLM enrichment exists as advisory metadata.
Human review workflow exists.
Review decisions can be represented safely.
No production KB mutation from review decisions yet.
Diagnostic Center still deferred.
```

Формула:

```text
Сначала безопасно увидеть и проверить machine-candidate metadata.
Потом отдельным PRD — применить проверенные решения.
Только потом — строить Diagnostic Center поверх проверенного knowledge layer.
```

---

## 16. Инструкция IDE-агенту

Работай маленькими шагами:

```text
1. Создай task list.
2. Реализуй contracts.
3. Реализуй sanitizer.
4. Реализуй queue builder.
5. Реализуй decision validator.
6. Напиши unit tests.
7. Запусти обязательные тесты.
8. Сгенерируй review artifacts.
9. Проверь no-mutation.
10. Создай reports/logs.
11. Обнови docs.
12. Commit + push.
13. Post-push sync implementation report with final commit hash.
```

Не проектируй самостоятельно Diagnostic Center.  
Не меняй runtime.  
Не применяй decisions к KB.  
Не коммить raw full text.  
Если обнаружишь, что текущие processed KB files отсутствуют локально — создай degraded report и объясни, какой входной файл не найден, но не подменяй задачу другой архитектурой.

## 17. Definition of Done

Готово только когда:

```text
review queue существует,
decision overlay contract существует,
validator работает,
artifacts sanitized,
source KB не мутирован,
tests passed,
reports/logs созданы,
docs обновлены,
commit pushed,
post-push report sync выполнен.
```
