# PRD-046.0.8-HF1 — Candidate Governance / Practice Classification Calibration v1

## 1. Назначение PRD

Этот PRD является hotfix-калибровкой после `PRD-046.0.8`.

`PRD-046.0.8` успешно собрал clean reprocess candidate из единственного active source `123__кузница_духа` без мутаций production-данных. Однако governance gate показал статус:

`candidate_needs_governance_calibration`

Основная причина — неточная классификация practice-like chunks:

- `practice_like_candidates_count = 77`
- `practice_like_as_practice_count = 66`
- `practice_like_misclassified_count = 11`
- `mixed_intent_alerts = 2`

Цель этого PRD — откалибровать candidate governance / practice classification так, чтобы перед production apply/reindex candidate-набор был безопаснее и точнее.

Этот PRD НЕ применяет candidate в production.

## 3. Главная проблема

В текущем candidate-наборе часть chunks содержит маркеры практики, но classified не как `practice`.

Важно: не все chunks со словом `практика` должны автоматически стать `practice`.

Нужна более точная таксономия:

1. **direct_practice_protocol** — явная инструкция / упражнение / протокол.
2. **practice_context_or_theory** — текст рассуждает о практиках, но не является самостоятельным упражнением.
3. **case_or_dialogue_about_practice** — сессия, кейс, диалог или пример, где практики упоминаются внутри истории.
4. **quote_or_source_fragment_with_practice_terms** — фрагмент источника с маркерами, но не пригодный как user-facing практика.

Ошибкой считается не любое несовпадение маркеров, а именно ситуация, когда `direct_practice_protocol` не classified как `practice`.

## 4. Цели

### 4.1. Функциональные цели

1. Добавить/откалибровать детектор practice-like chunks в candidate pipeline.
2. Разделить practice-like findings на:
   - `direct_practice_protocol`
   - `practice_context_or_theory`
   - `case_or_dialogue_about_practice`
   - `quote_or_source_fragment_with_practice_terms`
3. Откалибровать governance assignment для direct practice chunks:
   - `chunk_type = practice`
   - `allowed_use` включает `practice_suggestion`, если практика безопасна для low-resource проверки
   - `safety_flags` обязательно включают `not_for_direct_quote`
   - для практик обязательно наличие low-resource guard flag / caution marker
4. Сохранить conservative behavior для cases/quotes/theory.
5. Повторно собрать candidate.
6. Повторно прогнать governance gate.
7. Подготовить отчёты и next PRD recommendation.

### 4.2. Архитектурные цели

1. Candidate governance должен быть детерминированным.
2. LLM не должен становиться authority для `chunk_type/allowed_use/safety_flags`.
3. Все изменения должны быть применены только к candidate pipeline / classifier / reports.
4. Production KB не должна мутироваться.

### 7.2. Direct practice detection

Direct practice detector должен учитывать не один marker, а комбинации:

Strong markers:

- `**Цель:**`
- `Цель:`
- `**Время:**`
- `Время:`
- `Шаг 1`
- `Шаг 2`
- `Упражнение`
- `Практика:`
- императивные инструкции, если они находятся в structured protocol

Правила:

- `Цель + Время` почти всегда direct practice.
- `Цель + Шаг` direct practice.
- `Время + Шаг` direct practice.
- `Шаг 1 + Шаг 2` direct practice.
- Одиночное слово `практика` НЕ делает chunk practice.
- Диалоговый блок `> ...` или сессия не становится practice автоматически.
- Теоретическое рассуждение о практиках не становится practice автоматически.

### 7.3. Governance calibration

Для direct practice chunks:

- `chunk_type = practice`
- `allowed_use` должен включать `practice_suggestion`
- `allowed_use` может включать `writer_context`, если это безопасно
- `safety_flags` должны включать:
  - `not_for_direct_quote`
  - `practice_requires_low_resource_check`
- если есть признаки потенциально сильной эмоциональной/телесной практики:
  - добавить/сохранить conservative flag `needs_review` или эквивалентный review flag
- не удалять `source_style_not_user_facing`, если он уже применим

Для non-direct practice mentions:

- не принудительно менять `chunk_type` на `practice`
- если это quote/case/theory/lens, оставить соответствующий type
- сохранить `not_for_direct_quote`
- не добавлять `practice_suggestion`, если chunk не является самостоятельным упражнением

## 8. Acceptance Criteria

### 8.1. Candidate completeness

Обязательно:

- candidate собран успешно
- candidate blocks count > 0
- source id consistency rate = `1.0`
- governance present rate = `1.0`
- chunking quality present rate = `1.0`
- allowed use present rate = `1.0`
- safety flags present rate = `1.0`
- sd labeling active = `false`

### 8.2. Practice calibration

Обязательно:

- `direct_practice_misclassified_count = 0`
- `unsafe_practice_suggestion_count = 0`
- все direct practice chunks имеют `chunk_type=practice`
- все direct practice chunks имеют `not_for_direct_quote`
- все direct practice chunks имеют low-resource practice guard
- одиночные contextual mentions слова `практика` не считаются hard misclassification

Допустимо:

- `contextual_false_positive_count > 0`, если они явно classified as contextual/theory/case/quote и не являются blocker.
- `mixed_intent_alerts > 0`, если они не high severity и попали в review queue / warning.

### 8.3. Gate

Успешный вариант:

- `governance_gate.status = passed`

Допустимый вариант для этого PRD:

- `governance_gate.status = candidate_needs_governance_calibration`
- но только если:
  - нет blockers
  - `direct_practice_misclassified_count = 0`
  - remaining warnings объяснены как non-critical contextual/mixed-intent review candidates

Недопустимо:

- `governance_gate.status = failed`
- source_id mismatch
- missing governance
- missing allowed_use/safety_flags
- SD labeling active
- production mutation

### 8.4. No mutation

Обязательно:

- `all_blocks_merged_mutated = false`
- `registry_mutated = false`
- `chroma_mutated = false`
- `review_queue_mutated = false`
- `production_apply_performed = false`
