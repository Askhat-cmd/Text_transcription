## 3. Главная проблема

После исправления practice taxonomy остались 2 mixed-intent cases.

Mixed-intent в этом контексте означает, что один block может одновременно содержать несколько ролей:

- lens + practice
- theory + practice
- case/dialogue + practice
- quote/source fragment + practice terms
- explanation + instruction
- source narrative + standalone exercise

Не всякий mixed-intent является ошибкой.

Нужно различить:

1. **critical_mixed_intent_requires_split**  
   Один block реально содержит два самостоятельных смысловых блока, которые должны быть split в candidate.

2. **governance_mixed_intent_resolved_by_primary_role**  
   Block смешанный, но primary role можно определить безопасно. Split не нужен, но secondary roles должны быть отражены в `chunking_quality`.

3. **review_only_mixed_intent**  
   Block допустим для candidate, но требует review flag / warning metadata, без блокировки apply.

4. **false_positive_mixed_intent**  
   Detector ошибочно поднял warning из-за слов-маркеров, но block не является mixed-intent.

Цель HF2 — не скрыть warning, а объяснить его детерминированно и убрать unresolved high/medium alerts.

### 6.2. Mixed-intent resolution taxonomy

- `primary_role_resolved`
- `review_only`
- `false_positive`

Для каждого case нужно определить:

- primary role
- secondary roles
- severity after calibration
- whether split is required
- whether review flag is required
- whether candidate apply can proceed

### 6.3. Candidate governance calibration

Для candidate blocks нужно применить только candidate-level calibration.

Разрешено:

- менять candidate block `metadata.governance.chunk_type`
- менять candidate block `metadata.governance.allowed_use`
- менять candidate block `metadata.governance.safety_flags`
- менять candidate block `metadata.chunking_quality`
- добавлять candidate-only diagnostic fields:
  - `mixed_intent_resolution`
  - `primary_role_after`
  - `secondary_role_markers_after`
  - `split_merge_suggestion`
  - `review_reasons`

Запрещено:

- менять production files
- менять raw text
- удалять safety flags
- превращать quote/source fragments в direct user-facing text
- убирать `not_for_direct_quote` с внутренних материалов
- делать Chroma reindex

### 6.4. Split decision policy

Если mixed-intent case реально требует split:

В рамках HF2 допустимы два варианта:

1. **Candidate split**  
   Если split можно сделать deterministic и безопасно в candidate-only mode.

2. **Apply-blocking review flag**  
   Если split небезопасен автоматом, оставить block как есть, но candidate gate должен не pass, а вернуть `candidate_needs_manual_split_review`.

Для перехода к `PRD-046.0.8.1` нельзя иметь unresolved `split_required`.
