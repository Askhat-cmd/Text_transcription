# PRD-046.0.8.1 — Controlled Candidate Apply + Chroma Reindex + KB Quality Re-Audit v1

## 1. Назначение PRD

Этот PRD переводит успешно откалиброванный clean reprocess candidate из candidate-only состояния в production knowledge base.

После цепочки:

- `PRD-046.0.8` — candidate clean reprocess
- `PRD-046.0.8-HF1` — practice taxonomy calibration
- `PRD-046.0.8-HF2` — mixed-intent warning calibration

текущий candidate готов к controlled apply:

- `governance_gate.status = passed`
- `candidate_ready_for_apply = true`
- `direct_practice_misclassified_count = 0`
- `unsafe_practice_suggestion_count = 0`
- `mixed_intent_unresolved_count = 0`
- `mixed_intent_split_required_count = 0`
- `candidate_blocks_count = 247`
- production blocks currently = 229
- legacy SD inactive
- source_id consistency = 1.0
- governance completeness = 1.0

Цель PRD — безопасно применить candidate в production, переиндексировать Chroma и заново проверить качество KB/retrieval после reprocess.

---

## 2. Почему этот PRD нужен сейчас

До этого момента все reprocess-шаги были candidate-only. Это было правильно, потому что нужно было сначала:

1. очистить source registry;
2. убрать legacy SD active path;
3. собрать clean candidate;
4. исправить practice classification;
5. закрыть mixed-intent warnings.

Теперь governance gate прошёл, поэтому можно перейти к production apply.

Но это high-risk операция, потому что меняется production KB:

- `all_blocks_merged.json` изменится с 229 blocks на 247 candidate blocks;
- registry для focus source должен быть обновлён;
- Chroma должна быть переиндексирована;
- старые review queue / enrichment artifacts могут стать устаревшими относительно новых block ids.

Поэтому PRD должен быть строго staged и проверяемым.

## 5. LLM enrichment and review queue policy

This PRD does not apply old LLM enrichment overlay.

Reason:

- block ids and block boundaries may have changed;
- old review queue was built against old production blocks;
- applying old decisions to new candidate blocks would risk governance drift.

Required behavior:

1. Old `TO_DO_LIST/logs/PRD-046.0.7/review_queue.json` is considered pre-reprocess historical artifact.
2. Do not mutate old queue.
3. Create `review_queue_staleness_report.json`.
4. Rebuild review queue against new 247-block production KB.
5. Store new queue as:
   - `TO_DO_LIST/logs/PRD-046.0.8.1/review_queue_new_baseline.json`
6. Add next recommendation for fresh enrichment/review cycle if needed.

This may change next roadmap after `PRD-046.0.8.1`.

Do not assume `PRD-046.0.7.1 Controlled Review Decision Apply` can apply old decisions after reprocess.
