## 1. Краткое назначение PRD

Цель этого PRD — **безопасно выполнить staged clean reprocess единственного активного источника `КУЗНИЦА ДУХА` из raw markdown** и получить новый проверяемый candidate-артефакт обработанных блоков.

Этот PRD **не должен сразу заменять production KB**, **не должен делать Chroma reset/reindex** и **не должен менять runtime бота**.

Результат PRD — не “новая база уже в бою”, а **чистый reprocess candidate + отчёты**, на основании которых можно принять решение:

1. candidate готов к controlled apply/reindex в следующем PRD;
2. candidate требует hotfix-калибровки chunking/governance;
3. reprocess небезопасен и должен быть остановлен.

---

## 2. Почему этот шаг нужен сейчас

После `PRD-046.0.7-HF2` source hygiene blocker закрыт:

- мусорные/test/stale источники архивированы guarded workflow;
- единственный active source — `123__кузница_духа`;
- focus source protected;
- legacy SD active path отключён;
- readiness gate перешёл в `ready`.

При этом текущие processed blocks и review queue всё ещё несут следы старых этапов:

- часть practice-like chunks с маркерами `**Цель:**`, `**Время:**` ранее классифицировалась как `lens`, `case` или `theory`;
- старые blocks могли быть созданы до последних governance/chunking calibration;
- review queue была построена поверх текущей старой processed-базы.

Поэтому перед дальнейшим review/apply/Diagnostic Center нужно построить **чистый candidate reprocess** из одного активного источника и проверить его качество.

## 6. Архитектурное решение

### 6.1. Candidate-first reprocess

До этого база была очищена на уровне registry readiness, но старые processed blocks всё ещё остаются production baseline.

Этот PRD вводит staged candidate reprocess:

```text
raw markdown
   ↓
current deterministic chunking
   ↓
normalization
   ↓
deterministic governance
   ↓
chunking_quality
   ↓
candidate JSON + reports
   ↓
quality gates
   ↓
следующий PRD решает apply/reindex или hotfix
```

### 6.2. Почему нельзя сразу reindex

Если сразу заменить processed blocks и Chroma, то:

- можно потерять старый working retrieval baseline;
- можно получить новую misclassification волну;
- review queue станет невалидной относительно новых blocks;
- будет трудно понять, ухудшился ли retrieval из-за reprocess, governance или Chroma.

Поэтому этот PRD делает только candidate build.

### 6.3. SD decommission правило

`SDLabeler` нельзя вызывать как active classification pass.

Если current classes требуют `sd_level` field для совместимости, поле может быть empty/default/deprecated, но:

- SD не должен влиять на chunking;
- SD не должен влиять на governance;
- SD не должен влиять на retrieval;
- SD distribution не должен быть operational signal.

## 11. Governance gate

`governance_gate.json` должен вернуть один из статусов:

- `passed`
- `candidate_needs_governance_calibration`
- `failed`

### 11.1. Passed

Можно поставить `passed`, если:

- preflight passed;
- candidate blocks count > 0;
- source_id consistency rate = 1.0;
- governance_present_rate = 1.0;
- chunking_quality_present_rate = 1.0;
- allowed_use_present_rate = 1.0;
- safety_flags_present_rate = 1.0;
- `sd_labeling_active=false`;
- no mutation check passed;
- no raw full text in markdown reports;
- no secret-like values.

### 11.2. Candidate needs governance calibration

Ставить этот статус, если:

- reprocess технически успешен;
- no mutation passed;
- но есть:
  - practice-like misclassification;
  - missing lens_family for important lens chunks;
  - unclear chunk_type;
  - suspicious allowed_use;
  - mixed_intent severity high/medium above threshold.

В этом случае следующий PRD должен быть hotfix calibration, а не Chroma reindex.

### 11.3. Failed

Ставить `failed`, если:

- preflight failed;
- raw source missing;
- active source not unique;
- source_id mismatch;
- candidate blocks count = 0;
- production artifacts mutated;
- Chroma count changed;
- registry changed;
- SD active path returned;
- tests failed.

Если обнаружится, что current pipeline не позволяет безопасно staged reprocess без Chroma/registry mutation, нужно:

3. не менять production data ради удобства;
4. зафиксировать ограничение в report.

Если reprocess candidate показывает плохую классификацию, не исправлять всё хаотично в этом же PRD. Нужно:

1. зафиксировать проблему;
2. поставить gate `candidate_needs_governance_calibration`;
