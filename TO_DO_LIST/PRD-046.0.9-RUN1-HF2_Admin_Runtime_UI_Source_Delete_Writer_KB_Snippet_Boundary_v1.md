# PRD-046.0.9-RUN1-HF2 — Admin Runtime UI Fix + Source Delete Hygiene + Writer KB Snippet Boundary Audit v1

## 1. Назначение PRD

Этот PRD является hotfix после `PRD-046.0.9-RUN1-HF1`.

`PRD-046.0.9-RUN1-HF1` добавил read-only dashboard endpoint и frontend-логику, а тесты показали зелёный статус. Но ручная проверка Web Admin показала, что фактический UI всё ещё не прошёл acceptance:

1. Dashboard по-прежнему показывает `—` вместо реальных метрик.
2. В Registry кнопка удаления ведёт себя непонятно:
   - protected source корректно нельзя удалить;
   - но test/archived/failed sources тоже не удаляются или показывают непонятные alert-сообщения;
   - источник `Книга` с `blocks_count=1` остаётся в реестре как мусор и визуально помечен как protected/blocked, хотя не является ценным источником.
3. Dashboard/API smoke считает `blocks_total=248`, хотя рабочая production KB должна быть `247` blocks по `Кузнице Духа`. Это значит, что dashboard смешивает production blocks и registry мусор.
4. В реальном чате с ботом в секции `ЗНАНИЯ ИЗ БАЗЫ` видны обрезанные фразы:
   - `Добро пожаловат`
   - `Ешь на бе`

Это не обязательно ломает итоговый ответ, но это явный signal, что Writer получает некачественно обрезанный retrieval context/snippet. Нужно диагностировать и закрыть это отдельным guard.

Главная цель PRD — привести Web Admin и runtime context visibility к фактическому состоянию проекта перед Human Review.

## 3. Root cause hypotheses to verify

IDE agent must not guess. It must verify.

### 3.1. Dashboard still blank

Possible causes:

1. Browser is serving cached `/static/app.js`.
2. Frontend requests a different endpoint than tested.
3. `/api/dashboard/` works in TestClient but fails in real local server context.
4. Static mount/working directory mismatch.
5. JS crashes before rendering.
6. HTML updated, but old JS still cached.
7. Dashboard endpoint returns `blocks_total=248`, causing mismatch but not blank.
8. CSS/card layout hides meta rows.

### 3.2. Delete button not working

Possible causes:

1. JS does not use `encodeURIComponent(source_id)`.
2. UI labels `Удалить` and `Защищено` are not aligned with backend policy.
3. Backend forbids all sources with `blocks_count > 0`, including test-like registry-only garbage with `1` block.
4. Backend deletes only zero-block sources but UI does not explain why.
5. Archived zero-block delete path has no actual smoke test.
6. Alert messages are too technical and not mapped to user-facing Russian explanations.

### 3.3. Dashboard blocks count mismatch

Possible causes:

1. Registry stats count archived/test-like blocks.
2. Dashboard uses `registry.get_statistics().total_blocks`, not active/focus production KB count.
3. Old test source `Книга` with `blocks_count=1` is still counted, causing `248` instead of production `247`.

### 3.4. Writer receives cut KB phrases

Possible causes:

1. Retrieval context uses fixed char truncation without word/sentence boundary.
2. Prompt builder truncates block text after selecting top-k.
3. Safe preview logic is reused for Writer context.
4. Context Assembly budget trims KB snippets without ellipsis/boundary guard.
5. Chunk text in production file is already corrupted.
6. Chroma metadata contains clipped text.

This PRD should diagnose and fix the first obvious boundary issue if safe, but should not redesign retrieval.

## 6. Scope B — Source delete hygiene UX and backend policy

### 6.1. Delete policy

Implement explicit source delete policies:

#### Protected source

For `123__кузница_духа`:

- hard delete forbidden
- archive forbidden unless special override, not in this PRD
- UI button label: `Защищено`
- tooltip/reason: `Основной источник базы`

#### Archived zero-block test sources

For archived/test/failed sources with `blocks_count=0`:

- hard delete allowed
- must delete registry entry
- must remove associated upload/json files if present
- must not mutate Chroma except safe delete attempt for that source id
- UI button label: `Удалить`
- success message in Russian

#### Registry-only test-like sources with `blocks_count <= 1`

For non-focus, test-like, failed/archived sources with `blocks_count=1`:

- do not call it protected
- UI label should be `Удалить после проверки` or `Очистить тестовый`
- backend must allow deletion only if:
  - source is not focus source
  - source is classified as `test_like` or `registry_only_blocks_test_like`
  - source_id is not present in production active source set
  - Chroma contains 0 records for that source id, or delete_source(source_id) removes 0/known test entries safely
  - backup/snapshot is created before mutation
- If not safe, return clear Russian reason.

### 6.2. Encode source ids

Registry JS must call:

```js
fetch(`/api/registry/${encodeURIComponent(id)}`, { method: 'DELETE' })
```

### 6.3. Clear UI states

Do not show disabled `Защищено` for every non-deletable source.

Use states:

- `Защищено` — only focus source.
- `Удалить` — safe zero-block deletable.
- `Очистить тестовый` — test-like <=1 block cleanup path.
- `Недоступно` — blocked with visible reason.
- `Архив` — already archived but retained.

## 8. Scope D — Writer KB snippet boundary audit/fix

### 8.1. Evidence

Uploaded real prompt shows `ЗНАНИЯ ИЗ БАЗЫ` contains cut fragments:

- `Добро пожаловат`
- `Ешь на бе`

This means Writer receives snippets cut mid-word/mid-sentence.

### 8.2. Audit

Create:

`TO_DO_LIST/logs/PRD-046.0.9-RUN1-HF2/writer_kb_snippet_boundary_audit.json`

Must check:

1. Are production block texts already corrupted?
2. Are Chroma documents corrupted?
3. Are retrieval API snippets corrupted?
4. Is Context Assembly truncating snippets?
5. Is Writer prompt builder truncating snippets?
6. Does truncation happen at fixed char limit?
7. Are ellipses added consistently?

### 8.3. Fix boundary if localized

If issue is localized to snippet/preview truncation, implement safe truncation:

- trim on sentence boundary where possible;
- otherwise trim on word boundary;
- add `…` when truncated;
- never cut in the middle of a Cyrillic word;
- keep max budget limits;
- do not increase prompt size dangerously.

Example:

Bad:

```text
Добро пожаловат
Ешь на бе
```

Good:

```text
Добро пожаловать…
Ешь на бегу, думаешь на бегу…
```

or if full sentence unavailable:

```text
Знакомо? Добро пожаловать…
```

### 8.4. Tests

Add tests that fail on mid-word Cyrillic truncation in KB snippets sent to Writer.

Suggested tests:

- `bot_psychologist/tests/multiagent/test_writer_kb_snippet_boundaries.py`
- `Bot_data_base/tests/test_retrieval_snippet_boundaries.py`, if relevant.

Acceptance:

- no Writer prompt KB snippet ends with suspicious unfinished Cyrillic token unless followed by ellipsis;
- snippets use boundary-aware truncation;
- production block text remains unchanged;
- retrieval budget remains bounded.
