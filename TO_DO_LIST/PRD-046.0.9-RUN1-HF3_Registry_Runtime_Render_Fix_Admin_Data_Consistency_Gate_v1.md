# PRD-046.0.9-RUN1-HF3 — Registry Runtime Render Fix + Admin Data Consistency Gate v1

## 1. Назначение

Hotfix после `PRD-046.0.9-RUN1-HF2`.

HF2 частично успешен: Dashboard в браузере ожил, показывает production blocks `247`, Enrichment/Review `87`, Writer snippet boundary guard внедрён. Но ручная проверка показала новый blocker:

- Registry page открывается, но таблица источников пустая.
- Dashboard показывает `sources=32`, значит источники в системе есть.
- Dashboard показывает `ChromaDB = 0`, `status: unavailable`, хотя ожидаемое runtime состояние после apply/reindex — `247`.

Это не ожидаемое поведение. Реестр не должен “обновиться только после новых загрузок”. Он обязан показывать уже существующие источники.

Цель PRD — восстановить runtime-visible Registry и согласовать Dashboard/Registry/Chroma состояние перед переходом к Human Review.

## 5. Scope A — Registry endpoint runtime hardening

### 5.1. Safe Chroma source existence helper

Replace any direct policy call like:

```python
runner.chroma_manager.source_exists(source_id)
```

with guarded helper:

```python
def _safe_chroma_source_exists(runner, source_id) -> tuple[bool | None, str | None]:
    ...
```

Rules:

- If Chroma manager has `source_exists`, use it safely.
- If not available, use safe fallback if current project has source-id scan support.
- If impossible to know, return `(None, "chroma_source_exists_unavailable")`, not exception.
- For delete policy:
  - unknown Chroma existence must block risky `blocks_count > 0` delete with clear reason;
  - unknown Chroma existence must not crash listing;
  - zero-block listing must not crash.

### 5.2. Row-level isolation

`_build_sources_payload()` must never fail the whole registry list because one source has bad metadata or policy error.

For each source:

- wrap classification and delete policy;
- include row anyway;
- set `delete_policy.state = "unavailable"` on row-level error;
- set visible reason: `Ошибка политики: ...`;
- collect row-level warning.

Endpoint must still return HTTP 200 with rows.

## 6. Scope B — Registry frontend error visibility

Registry page must not show an empty table silently.

Add:

- loading state: `Загрузка реестра...`
- error state: `Ошибка загрузки реестра: ...`
- empty state: `Источники не найдены` only when API returns 200 and total=0
- normal state: rows rendered

Add visible error block:

- `id="registry-errors"`

Registry JS must:

- catch errors from `loadRegistry()`;
- display the error;
- keep table state understandable;
- use `encodeURIComponent(source_id)`;
- render at least focus source when payload contains it.

Bump cache:

```html
<script src="/static/registry.js?v=046_0_9_run1_hf3"></script>
```

## 9. Acceptance Criteria

PRD accepted if:

### Registry

- `/api/registry/` returns HTTP 200 in runtime smoke.
- Registry page shows rows, not empty table.
- `Кузница Духа` visible with 247 blocks.
- Archived/test sources visible or explicitly deleted via safe hygiene policy.

- No single source row can crash the whole registry payload.

### Dashboard / Chroma

- Dashboard still shows blocks 247 and review 87.
- Dashboard Chroma shows 247, or explicit diagnostic explains real local unavailability.
- No silent mismatch.

### Delete policy

- Focus source delete forbidden.
- Safe zero-block delete still supported.
- Test-like one-block cleanup does not crash list rendering.
- Missing/failed `source_exists` does not crash registry.

### Safety

- production KB unchanged.
- focus source unchanged.
- Chroma not reindexed.
- enrichment not applied.
- review decisions not applied.
