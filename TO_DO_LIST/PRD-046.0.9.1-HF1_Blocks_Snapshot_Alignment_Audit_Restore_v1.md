# PRD-046.0.9.1-HF1 — Blocks Snapshot Alignment Audit / Restore v1

## 0. Контекст

После выполнения `PRD-046.0.9.1 — Human Review Decisions for Post-Reprocess Enrichment v1` workflow был реализован, но завершился со статусом:

```text
done_with_data_alignment_blocker
```

Причина blocker:

```text
fresh RUN1 review queue содержит 87 items,
но текущий Bot_data_base/data/processed/all_blocks_merged.json содержит 229 blocks,
и 0 из 87 queue block ids найдено в текущем blocks snapshot.
```

При этом актуальное runtime/admin состояние после `PRD-046.0.9-RUN1-HF3` должно быть:

```text
Dashboard blocks: 247
Registry focus blocks: 247
Chroma count: 247
focus source: 123__кузница_духа
review queue: 87 items
```

Значит, текущий workspace snapshot `all_blocks_merged.json` рассинхронизирован с post-reprocess baseline. Переходить к `PRD-046.0.7.1 Controlled Review Decision Apply` нельзя, пока alignment не восстановлен.

---

## 1. Цель PRD

Восстановить согласование между:

```text
fresh RUN1 review queue
current all_blocks_merged.json
registry focus source
Chroma/admin runtime count
```

так, чтобы `PRD-046.0.9.1` review decisioning workflow мог быть повторно запущен с результатом:

```text
blocks_total = 247
queue_items_count = 87
queue_block_ids_missing_in_blocks_count = 0
queue_block_ids_present_in_blocks_count = 87
```

---

## 2. Главный принцип безопасности

Этот PRD не является apply review decisions.

Запрещено:

```text
- применять human review decisions;
- мутировать llm_enrichment на основе review decisions;
- менять Writer;
- менять Context Assembly;
- менять Diagnostic Card;
- запускать Diagnostic Center;
- делать Chroma reindex без отдельной необходимости и явного отчёта;
- менять deterministic governance authority вручную;
- подменять current production KB произвольным candidate-файлом без provenance proof.
```

Разрешено только:

```text
- провести alignment audit;
- найти authoritative 247-block snapshot;
- при наличии proof восстановить stale all_blocks_merged.json до текущего 247 baseline;
- пересчитать manifest/review workflow;
- усилить alignment gate, чтобы подобный mismatch больше не проходил как обычный success.
```

## 4. Требуемые изменения

### 4.1. Alignment audit tool

Добавить инструмент:

```text
Bot_data_base/tools/audit_blocks_snapshot_alignment.py
```

Он должен:

1. Прочитать fresh review queue.
2. Прочитать `all_blocks_merged.json`.
3. Прочитать registry.
4. Прочитать доступные post-apply/reprocess manifests/logs.
5. Посчитать:

```json
{
  "blocks_total": 229,
  "expected_blocks_total": 247,
  "queue_items_count": 87,
  "queue_block_ids_present_count": 0,
  "queue_block_ids_missing_count": 87,
  "chroma_count": 247,
  "registry_focus_blocks": 247,
  "status": "not_aligned"
}
```

6. Найти возможный authoritative 247 snapshot в известных местах:

```text
Bot_data_base/data/processed/all_blocks_merged.json
TO_DO_LIST/logs/PRD-046.0.8.1/**
TO_DO_LIST/logs/PRD-046.0.9/**
TO_DO_LIST/logs/PRD-046.0.9-RUN1/**
Bot_data_base/data/backups/**
```

7. Для каждого кандидата вывести:

```json
{
  "path": "...",
  "blocks_total": 247,
  "sha256": "...",
  "queue_block_ids_present_count": 87,
  "queue_block_ids_missing_count": 0,
  "source_ids": ["123__кузница_духа"],
  "governance_present_rate": 1.0,
  "candidate_is_authoritative": true/false,
  "reasons": []
}
```

### 4.2. Conditional restore tool

Добавить инструмент:

```text
Bot_data_base/tools/restore_blocks_snapshot_alignment.py
```

Он должен работать только через явный staged режим:

```bash
python Bot_data_base/tools/restore_blocks_snapshot_alignment.py --dry-run
python Bot_data_base/tools/restore_blocks_snapshot_alignment.py --apply
```

`--apply` разрешён только если найден authoritative candidate:

```text
blocks_total = 247
queue_block_ids_missing_count = 0
source_ids = [123__кузница_духа]
governance_present_rate = 1.0
allowed_use_present_rate = 1.0
safety_flags_present_rate = 1.0
raw_full_text_leak_detected = false
```

Перед apply обязательно создать backup:

```text
TO_DO_LIST/logs/PRD-046.0.9.1-HF1/backups/all_blocks_merged.before_restore.json
TO_DO_LIST/logs/PRD-046.0.9.1-HF1/backups/registry.before_restore.json
```

Если authoritative candidate не найден, apply запрещён. В этом случае PRD завершается отчётом `blocked_no_authoritative_snapshot_found` без мутаций.

### 4.3. Alignment gate для human review workflow

Усилить `prepare_human_review_decisions.py` или общий модуль так, чтобы при запуске с флагом:

```bash
--require-aligned
```

скрипт завершался non-zero, если:

```text
queue_block_ids_missing_in_blocks_count > 0
```

По умолчанию можно оставить audit/report mode, но для pre-apply workflow должен быть строгий режим.

### 4.4. Повторный запуск PRD-046.0.9.1 artifacts

После restore или после подтверждения alignment IDE-агент обязан повторно запустить:

```bash
python Bot_data_base/tools/prepare_human_review_decisions.py --require-aligned
python Bot_data_base/tools/validate_human_review_decisions.py
```

И получить:

```text
queue_block_ids_missing_in_blocks_count = 0
queue_block_ids_present_in_blocks_count = 87
validation valid = true
no_mutation_proof passed
```

## 8. Критерии приёмки

PRD считается выполненным успешно, если:

```text
1. Причина 229/247 mismatch явно диагностирована.
2. Найден authoritative 247 snapshot или явно доказано, что он отсутствует.
3. Если snapshot найден — all_blocks_merged восстановлен до 247 через backup + restore workflow.
4. После restore review workflow показывает queue_block_ids_missing_in_blocks_count = 0.
5. Fresh RUN1 review queue остаётся source_prd = PRD-046.0.9-RUN1.
6. No human review decisions applied.
7. No llm_enrichment apply performed.
8. No Chroma reindex performed, если Chroma already 247.
9. Tests passed.
10. docs updated.
11. commit + push completed.
12. implementation report содержит commit hash и push status.
```

Если authoritative 247 snapshot не найден, acceptable result:

```text
status = blocked_no_authoritative_snapshot_found
production mutation = false
next recommendation = restore/rebuild 247 snapshot from clean source reprocess or rerun controlled apply with backups
```
