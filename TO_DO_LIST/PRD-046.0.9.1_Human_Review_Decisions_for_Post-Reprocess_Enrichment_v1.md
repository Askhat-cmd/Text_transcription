## 0. Короткое резюме

После `PRD-046.0.9-RUN1` выполнен реальный post-reprocess LLM enrichment run по текущим `247` block ids. Enrichment завершился успешно (`247/247`, validation errors `0`), но сформировал fresh review queue на `87` элементов.

После `PRD-046.0.9-RUN1-HF3` Admin runtime consistency восстановлена: Dashboard / Registry / Chroma согласованы на `247/247`, Registry больше не рендерится пустым, row-level isolation добавлен.

Следующий безопасный шаг — **не применять enrichment в production**, а создать слой human review decisions поверх fresh post-reprocess review queue.

Этот PRD должен:

```text
fresh review queue
→ human-readable review workbench
→ review decisions template
→ decisions validator
→ validation/report artifacts
→ no-mutation proof
```

Этот PRD **не должен**:

```text
мутировать production KB,
делать Chroma reindex,
применять enrichment overlay,
менять Writer,
менять Context Assembly,
запускать Diagnostic Center.
```

## 2. Главная цель PRD

Создать безопасный, воспроизводимый и проверяемый workflow для human review decisions по fresh post-reprocess enrichment queue.

Цель не в том, чтобы IDE-агент “сам решил”, какие enrichment-поля правильные. Цель — дать человеку/админу удобный и безопасный слой принятия решений, а системе — строгую валидацию этих решений.

Формула:

```text
LLM enrichment предлагает.
Human review принимает / редактирует / отклоняет / откладывает.
Deterministic validator проверяет.
Production KB пока не мутируется.
```

В scope входит:

## 5. Архитектурные ограничения

### 5.1. Deterministic governance остаётся authority

Human review в рамках этого PRD может работать только с advisory enrichment metadata.

Нельзя менять через review decisions:

```text
text
content
content_full
full_text
source_raw
chunk_type
allowed_use
safety_flags
not_for_direct_quote
source_style_not_user_facing
internal_only
do_not_use
governance
embedding
vector
source_id
block_id
```

### 5.2. Разрешённые advisory-поля

В decisions допускаются только поля, уже разрешённые review contract или явно добавленные как advisory-only:

```text
summary
lens_family_candidates
tags
use_when
avoid_when
self_contained_score
self_contained_reason
split_merge_suggestion
confidence
```

Если IDE-агент считает нужным добавить новое advisory-only поле, он должен:

1. не добавлять его молча;
2. описать причину в implementation report;
3. добавить тесты;
4. не смешивать это с governance authority fields.

По умолчанию новые advisory fields не добавлять.

### 5.3. Никаких raw leaks

Review workbench и JSON artifacts не должны содержать:

```text
content_full
raw_text
full_text
chapter_text
source_raw
embedding
vector
private logs
.env
api keys
tokens
passwords
```

`content_preview` допускается только sanitized и ограниченный.

Рекомендуемое ограничение:

```text
content_preview_max_chars: 240
```

### 5.4. Старый review queue нельзя применять

Review queue / decisions от `PRD-046.0.7` относятся к старым `229` blocks и не могут применяться напрямую после reprocess.

Можно переиспользовать:

```text
review contracts
sanitizer
validator concepts
CLI patterns
tests approach
```

Нельзя переиспользовать:

```text
старые decisions;
старый review queue как источник текущих решений;
старые block ids без explicit remap+hash proof.
```

## 10. Дополнительные validation rules

### 10.1. Duplicate decisions

Если один `review_item_id` встречается больше одного раза:

```text
valid = false
error: duplicate_review_item_id
```

### 10.2. Needs edit без edits

Если:

```text
decision = needs_edit
```

то должен быть хотя бы один из:

```text
edited_fields not empty
rejected_fields not empty
```

Иначе:

```text
valid = false
error: needs_edit_without_changes
```

### 10.3. Approved с edited_fields

Если:

```text
decision = approved
edited_fields not empty
```

то это должно быть error или warning. Рекомендуется error:

```text
approved_must_not_have_edited_fields
```

### 10.4. Rejected без reason

Если:

```text
decision = rejected
reason empty
```

то:

```text
valid = false
error: reason_required_for_decision
```

### 10.5. Defer без reason

Если:

```text
decision = defer
reason empty
```

то:

```text
valid = false
error: reason_required_for_decision
```

### 10.6. Избыточно длинные advisory поля

Рекомендуемые ограничения:

```text
summary: <= 600 chars
tags: <= 20 items, each <= 64 chars
lens_family_candidates: <= 10 items, each <= 64 chars
use_when: <= 12 items, each <= 180 chars
avoid_when: <= 12 items, each <= 180 chars
self_contained_reason: <= 500 chars
split_merge_suggestion: <= 500 chars
confidence: number between 0 and 1
self_contained_score: number between 0 and 1
```

Если превышено — validator должен выдавать warning или error. Для первого PRD допустимо warning, кроме явно опасных/raw/private значений.

1. Fresh review queue от `PRD-046.0.9-RUN1` найдена и проверена.

8. Empty template проходит как template, но не помечается как production-ready apply decisions.

11. Unknown review_item_id блокируется.
12. Block id mismatch блокируется.
13. Secret-like values блокируются.

15. Tests запущены и вывод сохранён.
16. `docs/PROJECT_STATE.md`, `docs/ROADMAP.md`, `docs/PRD_INDEX.md` обновлены.
17. Implementation report содержит commit hash и push status.
