# PRD-046.0.9.2 — Architect Semantic Review Pass / Fill Real Decisions v1

## 0. Назначение PRD

Этот PRD заменяет идею ручной проверки пользователем на **архитекторское смысловое ревью**.

Пользователь не будет самостоятельно проверять качество блоков. Ответственность за смысловую оценку берёт архитектурный ChatGPT-агент проекта, опираясь на базовые материалы:

- `Концептуальная структура бота.md`;
- `Neo MindBot КОНСПЕКТ.md`;
- `КУЗНИЦА ДУХА v.2.md`;
- текущие architecture decisions и living docs проекта.

IDE-агент не должен сам принимать смысловые решения за архитектора. Его задача — подготовить безопасные review-пакеты, формат для решений и проверочные инструменты, чтобы архитектор мог принять решения по `87` aligned review items без доступа к production mutation.

---

## 1. Текущий контекст

После `PRD-046.0.9.1-HF1` data-alignment blocker закрыт:

```text
all_blocks_merged.json = 247 blocks
RUN1 review queue = 87 items
queue coverage = 87/87
missing = 0
registry_focus_blocks = 247
chroma_count = 247
```

Реальные human/architect decisions пока не заполнены:

```text
decisions_count = 0
validation = valid=true для пустого template
```

Следовательно, к `PRD-046.0.7.1 Controlled Review Decision Apply` переходить нельзя.

---

## 2. Главная цель

Создать управляемый процесс, в котором:

```text
aligned review queue
→ sanitized architect review batches
→ architect semantic decisions
→ validated decisions overlay
→ no production mutation
```

Цель PRD — не применить решения, а подготовить и зафиксировать **реальные смысловые решения архитектора** по 87 элементам review queue.

---

## 3. Роль архитектора и IDE-агента

### 3.1. Архитектор ChatGPT

Архитектор принимает смысловые решения:

```text
approved
needs_edit
rejected
defer
```

Архитектор оценивает не “нравится / не нравится”, а соответствие блока назначению Neo MindBot:

```text
бот как зеркало / навигатор / мягкий катализатор;
KB как внутренняя библиотека линз, а не цитатник;
КУЗНИЦА как internal lens/practice source;
Writer не должен становиться книжным консультантом;
никаких диагнозов, директив и духовного авторитета;
низкий ресурс пользователя важнее глубины;
практики только при resource-fit и safety-fit.
```

### 3.2. IDE-агент

IDE-агент не принимает финальные смысловые решения.

IDE-агент должен:

- подготовить review batches;
- проверить, что batches безопасны и не содержат raw/full leaks;
- создать файл для решений архитектора;
- добавить validator для заполненного decisions overlay;
- не мутировать KB, registry, Chroma;
- не запускать apply.

## 4. Non-goals

В этом PRD запрещено:

1. Применять decisions к `all_blocks_merged.json`.
2. Делать Chroma reindex.
3. Менять production KB metadata.
4. Менять `chunk_type`, `allowed_use`, `safety_flags` через LLM или через review decisions.
5. Менять Writer, DiagnosticCard, State Analyzer, Thread Manager, Context Assembly.
6. Запускать Diagnostic Center.
7. Просить пользователя вручную оценивать блоки.
8. Автоматически approve все элементы.
9. Делать бота цитатником `КУЗНИЦЫ ДУХА`.

## 5. Review rubric архитектора

Каждый review item оценивается по следующим критериям.

### 5.1. Соответствие идентичности Neo MindBot

Блок пригоден, если помогает боту быть:

```text
зеркалом;
навигатором;
мягким катализатором осознавания;
живым откликом;
трабл-шутером только там, где это безопасно и не директивно.
```

Блок не должен превращать бота в:

```text
духовного учителя;
психотерапевта с диагнозами;
книжного консультанта;
автора прямых жизненных советов;
морализатора;
носителя “истины”.
```

### 5.2. Роль `КУЗНИЦЫ ДУХА`

Материал из `КУЗНИЦЫ` должен использоваться как:

```text
internal lens;
диагностическая оптика;
источник мягких mirror-фраз;
источник практик;
источник use_when / avoid_when;
источник предупреждений.
```

Материал не должен использоваться как:

```text
прямая цитата пользователю;
авторитетная доктрина;
объяснение “как всё устроено”;
эзотерическая терминология без адаптации;
замена психологической безопасности.
```

### 5.3. Качество advisory enrichment

Проверяются только advisory-поля:

```text
summary
lens_family_candidates
tags
use_when
avoid_when
self_contained_score
self_contained_reason
split_merge_suggestion
confidence
```

Критерии:

- summary кратко и точно передаёт функцию блока;
- tags не декоративные, а помогают retrieval;
- lens_family_candidates соответствуют смыслу;
- use_when описывает реальные пользовательские состояния;
- avoid_when обязательно учитывает low-resource/hypo/hyper/safety cases;
- self_contained_score не завышен;
- split_merge_suggestion не предлагает опасных мутаций без отдельного PRD.

### 5.4. Safety

Отклонять или править, если блок / enrichment:

- звучит как диагноз;
- даёт директивный жизненный совет;
- усиливает вину/стыд;
- тащит пользователя глубже, чем он может выдержать;
- предлагает практику без low-resource check;
- романтизирует страдание;
- обещает исцеление / трансформацию;
- делает бота духовным авторитетом;
- может попасть Writer’у как user-facing цитата.

## 6. Решения и их смысл

### `approved`

Использовать только если advisory enrichment безопасен и достаточно точен.

Условия:

```text
нет raw/direct quote риска;
summary точный;
use_when/avoid_when уместны;
теги помогают retrieval;
нет клинической/духовной директивности;
не требуется ручная правка.
```

### `needs_edit`

Использовать, если блок полезен, но одно или несколько advisory-полей нужно исправить.

Обязательно:

```text
reason не пустой;
edited_fields содержит только advisory поля;
rejected_fields указывает, что именно исправляется.
```

### `rejected`

Использовать, если advisory enrichment непригоден:

```text
искажает смысл;
опасен;
слишком общий;
делает бота цитатником;
подталкивает к директивности;
не помогает retrieval/context assembly.
```

### `defer`

Использовать, если данных недостаточно для честного решения.

Например:

```text
safe preview слишком короткий;
непонятен контекст раздела;
нужна split/reprocess проверка;
есть сомнение в source boundary.
```

Разделить `87` review items на batches по 10–12 элементов.

Каждый item в batch должен содержать только sanitized information:

```text

source_id
heading_path если есть
chunk_type
allowed_use
safety_flags
lens_family
review_priority

lens_family_candidates
use_when
avoid_when
self_contained_score
self_contained_reason
confidence

```

Запрещённые ключи в batch-файлах:


## 10. Важное workflow-правило

IDE-агент не должен сам заполнять смысловые решения за архитектора.

Допустимый вариант:

1. IDE-агент создаёт batches и пустой overlay.
2. Пользователь сообщает в чат: “готово”.
3. Архитектор ChatGPT заходит на GitHub, читает batch-файлы и формирует решения.
4. IDE-агент переносит решения архитектора в overlay и валидирует.

Если IDE-агент всё же создаёт draft decisions автоматически, они должны быть помечены как:

```text
reviewer = machine_draft
status = not_final
```

И не могут быть применены в `PRD-046.0.7.1`.

## 11. Validation rules

Validator должен проверять:

- `queue_items_count = 87`;
- все decisions ссылаются на known review_item_id;
- block_id совпадает с queue;
- нет duplicate review_item_id;
- нет unknown review_item_id;
- нет forbidden keys;
- нет secret-like values;
- нет authority field mutation attempts;
- `needs_edit/rejected/defer` имеют непустой reason;
- `approved` не содержит edited_fields;
- `needs_edit` содержит edited_fields или rejected_fields;
- edited_fields только из advisory fields;
- decisions coverage report показывает, сколько items закрыто.

Для завершения PRD допустимы два режима:

### Режим A — batch preparation only

Если архитектор ещё не заполнил решения:

```text
batches_created = true
decisions_count = 0
ready_for_architect_review = true
apply_ready = false
```

### Режим B — decisions filled

Если архитектор уже дал решения и IDE-агент перенёс их в overlay:

```text
decisions_count > 0
validation_valid = true
coverage_report создан
apply_ready = true только если все 87 items имеют final decision
```

## 14. Acceptance criteria

PRD считается выполненным, если:

1. Review batches созданы и покрывают все `87` items.
2. Batches не содержат forbidden/raw/full/private keys.
3. Alignment остаётся чистым:

```text
blocks_total = 247
queue_block_ids_present = 87
queue_block_ids_missing = 0
chroma_count = 247
registry_focus_blocks = 247
```

4. Production KB не мутировалась.
5. Chroma reindex не выполнялся.
6. Writer/runtime layers не менялись.
7. Есть no-mutation proof.
8. Есть validation report.
9. Если decisions заполнены — validator проходит без errors.
10. Если decisions не заполнены — project state явно показывает `ready_for_architect_review`, а не `apply_ready`.

Только после заполнения и валидации decisions overlay можно переходить к:

```text

```
