# PRD-046.0.9.3 — Review Automation Policy / Auto-Decisions v1

## 0. Короткое резюме

Текущий проект дошёл до точки, где ручной разбор всех `87` review items становится неправильным направлением. Цель этого PRD — заменить ручной проход по batch-файлам на безопасную автоматизированную политику принятия review-решений.

Этот PRD должен:

1. Сгенерировать валидный `architect_decisions_overlay` по всем `87` item.
2. Не применять решения в production KB.
3. Не менять `text`, `chunk_type`, `allowed_use`, `safety_flags` и другие governance authority поля.
4. Не запускать Chroma reindex.
5. Перевести процесс из ручной проверки блоков в воспроизводимую review policy.
6. Подготовить систему к будущему сценарию: пользователь загружает материалы через веб-админку, а обработка, enrichment, review routing и safe apply проходят автоматически.

Важно: автоматизация не означает “одобрить всё”. Автоматизация означает: по каждому item должно быть принято безопасное решение — `approved`, `needs_edit`, `rejected` или `defer`.

## 2. Продуктовая цель

Сделать первый автоматизированный слой принятия review-решений для KB enrichment metadata.

В зрелой системе после загрузки материала через веб-админку процесс должен выглядеть так:

```text
upload source
→ structure-aware chunking
→ deterministic governance
→ LLM enrichment
→ validation
→ review policy
→ auto-decisions overlay
→ controlled apply gate
→ Chroma refresh/reindex when allowed
→ admin status
```

Пользователь не должен каждый раз вручную проверять блоки. Архитектор не должен вручную читать сотни фрагментов, если система уже может безопасно маршрутизировать решения.

1. Не менять production KB напрямую, кроме создания/обновления overlay artifacts.

3. Не менять deterministic authority поля:

8. Не писать пользовательские ответы на основе KB.
9. Не считать `approved` единственным успешным исходом.
10. Не удалять review batches.

## 5. Основной принцип auto-decision policy

Policy должна быть conservative-first:

```text
Очевидно безопасное → approved
Безопасное, но требует guardrail → needs_edit
Сомнительное / мало данных / чувствительное → defer
Непригодное или опасное для advisory metadata → rejected
```

Цель — не максимальный процент approve, а безопасное закрытие coverage по всем `87` item.

## 6. Decision values

Использовать только существующие значения:

```text
approved
needs_edit
rejected
defer
```

### 6.1 approved

Применять только если enrichment metadata достаточно безопасна и полезна.

Пример условий:

- item aligned with current block id;
- нет forbidden/private/raw keys;
- нет secret-like values;
- нет authority mutation attempt;
- summary не делает диагноз;
- summary не звучит как прямой совет;
- summary не превращает `КУЗНИЦУ` в авторитет;
- fields are advisory only;
- item не имеет высокорисковых reasons;
- item не относится к direct practice без low-resource guard.

### 6.2 needs_edit

Использовать, когда metadata в целом полезна, но нужно добавить или заменить безопасные advisory поля.

Пример:

- practice/case with `practice_requires_low_resource_check`;
- missing `avoid_when` для практики;
- summary слишком директивный, но можно заменить на более нейтральный;
- use_when/avoid_when нужно сделать безопаснее.

`needs_edit` может менять только advisory fields:

```text
summary
tags
lens_family_candidates
use_when
avoid_when
self_contained_score
self_contained_reason
split_merge_suggestion
confidence
```

### 6.3 rejected

Использовать, если enrichment metadata лучше не применять вообще.

Примеры:

- metadata содержит диагноз;
- summary искажает смысл блока;
- summary делает прямое обещание эффекта;
- metadata предлагает опасную практику;
- metadata превращает внутренний материал в user-facing doctrine;
- metadata содержит pseudo-clinical certainty.

### 6.4 defer

Использовать для всего, что нельзя безопасно auto-approve.

Важно: `defer` — нормальный outcome. Он закрывает coverage, но не применяет metadata.

## 8. Review policy categories

### 8.1 Hard reject / hard defer signals

Если обнаружено что-либо из этого, нельзя auto-approve:

- review priority `P0`;
- missing block id;
- block id mismatch;
- forbidden/raw/private keys;
- secret-like values;
- authority field mutation attempt;
- summary includes diagnostic assertions;
- summary includes direct therapeutic promise;
- summary includes imperative direct advice without low-resource check;
- practice item without safety guard;
- content about panic, psychosis, suicidality, severe dissociation, self-harm, abuse, trauma without proper deferral;
- sexual instinct / sexuality content with insufficient neutral framing;
- fear/lissophobia content framed too clinically;
- spiritual authority framing.

Outcome:

```text
rejected
```

or

```text
defer
```

depending on whether metadata is unsafe or merely insufficient.

### 8.2 Practice/case with low-resource guard

If item has:

```text
practice_requires_low_resource_check
```

or `chunk_type=practice`, then do not auto-approve as-is unless metadata contains safe `use_when` and `avoid_when`.

Recommended v1 behavior:

```text
needs_edit
```

with edited advisory fields like:

```json
{
  "use_when": [
    "использовать только как мягкую внутреннюю подсказку, когда пользователь в устойчивом состоянии и просит практическое самонаблюдение"
  ],
  "avoid_when": [
    "не предлагать как упражнение при остром дистрессе, панике, суицидальных мыслях, психозе, сильной диссоциации или когда пользователь просит срочную помощь",
    "не подавать как терапевтическую методику или замену специалисту"
  ]
}
```

Можно также заменить summary на более нейтральный:

```json
{
  "summary": "Внутренняя practice-подсказка для мягкого самонаблюдения; не является инструкцией, диагнозом или терапевтической интервенцией."
}
```

### 8.3 Quote chunks

`chunk_type=quote` нельзя превращать в user-facing quote source.

Default v1 behavior:

```text
defer
```

или `approved` только для very safe summary, если:

- summary clearly paraphrases;
- no doctrine tone;
- no direct quote use;
- no clinical/safety-sensitive topic.

Conservative default: `defer`.

### 8.4 Theory/lens chunks

May be `approved` if metadata is:

- neutral;
- non-diagnostic;
- not directive;
- useful as internal lens;
- not safety-sensitive;
- has summary and at least one helpful tag/lens/use_when signal.

If metadata is too generic or empty, use:

```text
defer
```

If metadata is useful but missing guardrails, use:

```text
needs_edit
```

### 8.5 Generic or hallucinated summary

If summary is generic and not grounded in the block preview, use:

```text
defer
```

Examples of generic summary patterns:

- “может быть использовано для обогащения консультирования”;
- “важно соблюдать безопасность и конфиденциальность” when the block is not about that;
- summary talks about “активное слушание” while block is about an internal writing practice;
- summary adds concepts not visible from heading/preview.

### 8.6 Sensitive themes

Default to `defer` unless metadata is obviously safe.

Sensitive terms include:

```text
страх
паника
лоссофобия / лиссофобия
сойти с ума
потерять контроль
травма
насилие
стыд
сексуальность
слияние
тень
диссоциация
депрессия
суицид
самоповреждение
```

## 9. Required decision payload shape

Each decision must include:

```json
{
  "review_item_id": "...",
  "block_id": "...",
  "decision": "approved | needs_edit | rejected | defer",
  "reviewer": "architect",
  "reason": "...",
  "approved_fields": [],
  "rejected_fields": [],
  "edited_fields": {},
  "created_at": "ISO-8601",
  "source_prd": "PRD-046.0.9.3"
}
```

For `approved`, reason may be short but should exist for audit clarity.

For `defer`, reason must explain why auto-policy did not approve.

For `needs_edit`, reason must explain what is being corrected.

For `rejected`, reason must explain the safety or quality issue.
