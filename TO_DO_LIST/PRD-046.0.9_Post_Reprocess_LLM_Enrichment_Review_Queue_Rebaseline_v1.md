# PRD-046.0.9 — Post-Reprocess LLM Enrichment + Review Queue Rebaseline v1

## 1. Назначение PRD

Этот PRD запускает новый enrichment/review baseline после успешного controlled apply в `PRD-046.0.8.1`.

После `PRD-046.0.8.1` production knowledge base была пересобрана и применена:

- production blocks: `229 -> 247`
- Chroma reindex: `247`
- active source: `123__кузница_духа`
- governance completeness: `1.0`
- allowed_use completeness: `1.0`
- safety_flags completeness: `1.0`
- direct practice misclassification: `0`
- unsafe practice suggestion: `0`
- mixed intent unresolved: `0`
- internal_only unsafe exposure: `0`
- retrieval eval status: `ok`

Но старые enrichment/review артефакты больше нельзя использовать напрямую, потому что после reprocess изменились block boundaries и block ids.

Главная цель PRD — заново построить advisory LLM enrichment candidate overlay и fresh review queue для текущих 247 production blocks.

Этот PRD НЕ применяет enrichment в production.

---

## 2. Почему этот PRD нужен сейчас

До reprocess уже существовал LLM enrichment APPLY1 для 60 старых блоков. Но теперь:

- old review queue из `PRD-046.0.7` помечена как stale;
- новая baseline review queue после `PRD-046.0.8.1` содержит `0` items;
- retrieval eval показывает `enrichment_metadata_seen_count = 0`;
- текущие 247 blocks имеют governance, но ещё не имеют свежего advisory enrichment под новые block ids.

Это значит:

```text
База теперь структурно чистая,
но semantic enrichment / use_when / avoid_when / tags / lens candidates
нужно построить заново поверх новой production KB.
```

Важно: этот PRD не должен ломать deterministic governance authority. LLM enrichment остаётся advisory.

### 4.4. Provider / credits handling

The tool must support three modes:

```bash
--mode dry-run
--mode real
--mode validate-existing
```

Rules:

1. `dry-run`
   - builds inventory and empty candidate template;
   - does not call LLM;
   - useful for preflight.

2. `real`
   - calls configured LLM provider;
   - must be resumable;
   - must support batch size;
   - must write partial progress safely.

3. `validate-existing`
   - validates a previously generated overlay without calling LLM.

If provider credits/API are unavailable:

- do not mutate production;
- write `provider_unavailable` status;
- preserve inventory and dry-run artifacts;
- implementation status may be `blocked_by_provider`, not `done`;
- next recommendation should be to rerun `real` mode after credits are available.

- dry-run overlay template generated;
- provider failure is clearly reported;
- no production mutation;
- no Chroma mutation;
- next recommendation says to rerun real enrichment after credits/provider restore.

Do not fake enrichment results.

### 6.3. Quality gates

Blocker if:

- overlay references old block ids;
- overlay contains raw full text;
- overlay mutates governance authority fields;
- validator detects clinical diagnosis or directive life advice;
- review queue builder uses stale PRD-046.0.7 queue as current;
- production `all_blocks_merged.json` changes;
- registry changes;
- Chroma changes;
- retrieval eval regresses due to accidental apply.
