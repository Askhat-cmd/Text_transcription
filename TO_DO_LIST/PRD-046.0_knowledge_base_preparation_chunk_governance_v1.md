# PRD-046.0 — Knowledge Base Preparation / Chunk Governance v1

## 0. Краткое резюме

**Цель PRD-046.0:** подготовить безопасный и управляемый слой Knowledge Base для будущих материалов проекта, прежде всего:

- `КУЗНИЦА ДУХА v.2.md`;
- `Neo MindBot КОНСПЕКТ.md`.

Это **не PRD про подключение полноценного Diagnostic Center** и не PRD про активное использование “Кузницы” в ответах пользователю.  
Это подготовительный слой:

```text
source documents → governed chunks → metadata → safety/use policy → dry-run reports
```

Главная задача — не “нарезать текст как попало”, а создать правила, по которым база знаний будет безопасно попадать в RAG и будущий Diagnostic Center.

После PRD-045.x система получила:

```text
Context Assembly
DiagnosticCard
Writer Move Compliance
Quality Trace
```

Теперь можно аккуратно готовить внешний материал как библиотеку:

```text
линз / практик / принципов / протоколов / формулировок
```

а не как случайные чанки, которые Writer будет цитировать без контроля.

---

## 1. Почему нужно

Сейчас в проекте уже есть retrieval-инфраструктура:

- `bot_agent/retriever.py` умеет искать по блокам через API / ChromaDB / TF-IDF fallback;
- `bot_agent/data_loader.py` умеет читать `db_json` / `api` / legacy JSON;
- `MemoryRetrievalAgent` уже разделяет:
  - recent turns;
  - personal history;
  - semantic memory;
  - knowledge RAG hits.

Но для материалов вроде `КУЗНИЦА ДУХА` и `Neo MindBot КОНСПЕКТ` нельзя просто сделать markdown split по 1000 символов.

Причины:

1. **Разные типы знания.**
   - принципы;
   - практики;
   - диагностические линзы;
   - safety-правила;
   - стилистические правила;
   - примеры;
   - спорные/сильные утверждения.

2. **Разные режимы использования.**
   - одни фрагменты можно использовать как user-facing практику;
   - другие — только как internal lens;
   - третьи — только как архитектурный principle;
   - некоторые должны быть excluded или flagged.

3. **Риск авторитетности.**
   Бот не должен превращаться в духовного наставника, диагноста или “носителя истины”.

4. **Риск клинических/жёстких формулировок.**
   Материал может содержать сильные утверждения о психике, теле, инстинктах, программах, зависимости, страхе. Их нельзя без фильтра отдавать пользователю.

5. **Будущий Diagnostic Center зависит от качества чанков.**
   Если база будет нарезана плохо, Diagnostic Center начнёт выбирать неправильные линзы или слишком сильные интервенции.

Главный принцип:

```text
Сначала governance.
Потом ingestion.
Потом retrieval evaluation.
Потом Diagnostic Center v1.
```

## 2. Цель

### 2.1. Основная цель

Добавить первый слой Knowledge Base Preparation / Chunk Governance v1:

1. Описать источник знания через manifest.
2. Добавить governed chunk schema.
3. Добавить deterministic markdown chunker/dry-run для `.md` файлов.
4. Разметить chunks по типам:
   - `principle`
   - `lens`
   - `practice`
   - `protocol`
   - `safety`
   - `style`
   - `case`
   - `quote`
   - `excluded`
5. Разметить allowed use:
   - `internal_only`
   - `writer_context`
   - `diagnostic_lens`
   - `practice_suggestion`
   - `safety_protocol`
   - `do_not_use`
6. Добавить safety/governance flags:
   - `clinical_risk`
   - `directive_risk`
   - `spiritual_authority_risk`
   - `too_strong_claim`
   - `requires_grounding`
   - `practice_requires_low_resource_check`
   - `source_style_not_user_facing`
7. Создать dry-run reports по двум материалам.
8. Не подключать эти chunks к live retrieval без отдельного PRD.

### 2.2. Архитектурная цель

Подготовить основу для:

```text
PRD-046.1 — Diagnostic Center v1
```

чтобы Diagnostic Center мог выбирать не “любой похожий кусок текста”, а governed knowledge unit:

```text
lens + allowed_use + safety_flags + contraindications + source_trace
```

## 3. Non-goals

В рамках PRD-046.0 нельзя:

1. Подключать `КУЗНИЦА ДУХА` к live Writer RAG без фильтра.
2. Менять `Writer` prompt.
3. Менять `DiagnosticCard` логику.
4. Делать Diagnostic Center v1.
5. Делать LLM-based chunking.
6. Делать embeddings или Chroma ingestion как обязательный шаг.
7. Коммитить raw private/source documents, если они не предназначены для репозитория.
8. Коммитить `.env`, ключи, базы, `.sqlite`, `.cache_bot_agent`, `.venv`, `node_modules`, raw private logs.
9. Делать user-facing цитирование материалов как авторитетных истин.
10. Давать клинические диагнозы, духовные ярлыки или жёсткие интерпретации.

## Task 3 — Add deterministic markdown chunker

Создать `chunker.py`.

Функция:

```python
chunk_markdown_document_v1(
    *,
    text: str,
    manifest: KnowledgeSourceManifest,
    max_chars: int = 1800,
    soft_min_chars: int = 400,
) -> list[GovernedKnowledgeChunk]
```

### 3.1. Rules

1. Сохранять heading hierarchy.
2. Не разрывать практику внутри `Цель / Время / Шаги`, если можно.
3. Не делать chunks слишком маленькими, если это не safety/protocol item.
4. Не делать chunks слишком большими.
5. Если heading начинается с `Практика`, chunk_type должен быть `practice`.
6. Если section содержит “запрещено / безопасность / кризис / не заменяет специалиста”, chunk_type может быть `safety` или `protocol`.
7. Если section содержит архитектурные правила, chunk_type может быть `principle` или `style`.

### 3.2. Heading path

Для каждого chunk сохранять:

```json
"heading_path": ["Глава 1", "Практика 2: Дофаминовый аудит"]
```

### 3.3. Chunk id

Детерминированный id:

```text
<source_id>::chunk::<0001>::<short_hash>
```

Hash от:

```text
source_id + heading_path + normalized_text
```

## Task 4 — Rule-based classification

Создать `classifiers.py`.

Функция:

```python
classify_chunk_governance_v1(chunk_text: str, heading_path: list[str], manifest: KnowledgeSourceManifest) -> dict
```

Возвращает:

```python
{
    "chunk_type": "...",
    "allowed_use": [...],
    "safety_flags": [...],
    "tags": [...],
    "lens_family": [...],
    "practice_metadata": {...},
    "governance_notes": [...]
}
```

### 4.1. Lens family tags

Поддержать v1 tags:

```text
act
ifs
somatic
aqal
mct
neurobiology
instincts
fear
avoidance
inner_program
self_worth
dopamine
attention
values
boundaries
relationship
safety
style
architecture
```

### 4.2. Practice metadata

Для practice chunks попытаться извлечь:

```json
{
  "goal": "...",
  "duration": "...",
  "steps_count": 0,
  "requires_journaling": true,
  "body_based": false,
  "low_resource_safe": false
}
```

Если practice содержит длинное письменное исследование, по умолчанию:

```text
low_resource_safe = false
practice_requires_low_resource_check
```

Если practice короткая телесная/дыхательная и без глубокого анализа:

```text
low_resource_safe = true
```

### 4.3. Allowed use policy

Правила:

- `Neo MindBot КОНСПЕКТ`:
  - mostly `internal_only`, `style_guidance`, `safety_protocol`, `diagnostic_lens`;
  - не использовать как user-facing quote.
- `КУЗНИЦА ДУХА`:
  - `practice` может быть `practice_suggestion`, но только если safety flags позволяют;
  - `theory/lens` — в основном `diagnostic_lens` или `writer_context`;
  - сильные утверждения — `not_for_direct_quote`, `too_strong_claim`.
- Любой section с директивными формулировками:
  - добавить `directive_risk`;
  - не отдавать напрямую без Writer moderation.
- Любой section с “программа”, “инстинкт”, “зависимость”, “нейрохимия”:
  - добавить `requires_grounding`;
  - если формулировка звучит как диагноз — `not_for_direct_quote`.

## Task 5 — Governance validator

Создать `validators.py`.

Функция:

```python
validate_governed_chunks_v1(chunks: list[GovernedKnowledgeChunk]) -> dict
```

Проверяет:

1. `chunk_id` unique.
2. `chunk_type` allowed.
3. `allowed_use` non-empty.
4. Если `chunk_type=practice`, есть `practice_metadata`.
5. Если `practice_suggestion`, должен быть либо `low_resource_safe=true`, либо safety flag `practice_requires_low_resource_check`.
6. Если `clinical_risk` или `spiritual_authority_risk`, direct quote/use запрещён.
7. Нет пустых text/summary.
8. Нет raw secrets/path leaks в exported chunks.

Возвращает report:

```json
{
  "chunks_total": 120,
  "errors": [],
  "warnings": [],
  "counts_by_type": {...},
  "counts_by_allowed_use": {...},
  "counts_by_safety_flag": {...}
}
```

## 7. Safety / privacy

Не коммитить:

```text
.env
.env.*
*.sqlite
*.db
.cache_bot_agent/
.venv/
node_modules/
__pycache__/
.pytest_cache/
raw private logs
raw source docs unless explicitly intended
API keys
tokens
```

Особое правило для `КУЗНИЦА ДУХА` и подобных материалов:

```text
Материал может использоваться как источник линз/практик,
но бот не должен звучать как духовный авторитет,
не должен выдавать его положения как диагноз,
и не должен цитировать сильные утверждения напрямую без moderation.
```

Особое правило для `Neo MindBot КОНСПЕКТ`:

```text
Это architecture/source-of-principles материал.
Он mostly internal-only и не должен попадать пользователю как цитата.
```
