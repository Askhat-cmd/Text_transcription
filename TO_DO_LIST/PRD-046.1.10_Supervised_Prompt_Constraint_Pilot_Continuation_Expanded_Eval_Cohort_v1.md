# PRD-046.1.10 — Supervised Prompt-Constraint Pilot Continuation / Expanded Eval Cohort v1

## 0. Краткое название

**Supervised Prompt-Constraint Pilot Continuation / Expanded Eval Cohort v1**

## 1. Контекст

После `PRD-046.1.9` выполнен один controlled supervised execution/observability cycle для Prompt-Constraint Pilot.

Подтверждённое состояние после PRD-046.1.9:

```text
final_status=passed
decision=continue_supervised
source_plan_gate_passed=true
cohort_size=3
cohort_limit_respected=true
test_apply_applied_count=3
normal_user_apply_count=0
rollback_failure_count=0
stale_apply_after_force_disabled_count=0
candidate_weaker_than_baseline_count=0
safety_regression_count=0
kb_policy_regression_count=0
prompt_bloat_regression_count=0
constraint_conflict_regression_count=0
raw_kb_text_exposure_count=0
provider_called_by_execution_count=0
production_mutation_detected=false
artifact_encoding_hygiene_passed=true
```

Это даёт право не на broad rollout, а только на **следующий supervised continuation cycle** с чуть более широким, но всё ещё test-only / allowlisted eval cohort.

Текущий runtime baseline остаётся:

```text
PROMPT_CONSTRAINT_PILOT_ENABLED=false
PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=true
PROMPT_CONSTRAINT_PILOT_MODE=shadow|test_apply
```

Архитектурная позиция не меняется:

```text
Diagnostic Center v1: trace-only shadow layer
Planner Bridge: candidate-only / shadow-eval lineage
planner_bridge_compliance_shadow: trace-only compare layer
planner_bridge_writer_contract_pilot: pilot_shadow_only
writer_prompt_replay: offline_replay_only
prompt_constraint_pilot_runtime: default-off limited allowlisted test path
```

## 2. Цель PRD

Провести **второй supervised continuation cycle** для Prompt-Constraint Pilot на расширенном eval cohort.

Цель — не “запустить на пользователей”, а проверить, сохраняются ли инварианты качества и безопасности при более разнообразном test cohort:

```text
- больше сценариев;
- больше комбинаций user_state / intent / openness / ok_position;
- повторная проверка rollback;
- повторная проверка normal-user no-effect;
- baseline vs test_apply comparison;
- observability scorecard;
- no-mutation proof;
- решение: continue_supervised / stay_limited / hotfix_required / stop.
```

## 3. Главный принцип

```text
PRD-046.1.10 расширяет только eval cohort внутри supervised harness.
Он не расширяет production rollout и не меняет default runtime flags.
```

После PRD обычный пользовательский путь должен остаться неизменным:

```text
normal_user_apply_count=0
default_off_user_path_effect_count=0
config_mutated=false
production_apply_performed=false
```

## 4. Non-goals

В рамках PRD запрещено:

1. Включать broad rollout.
2. Включать pilot для всех пользователей.
3. Менять default flags в config/code.
4. Делать `PROMPT_CONSTRAINT_PILOT_ENABLED=true` новым baseline default.
5. Делать `PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=false` новым baseline default.
6. Использовать реальных production-пользователей.
7. Снимать allowlist/test-prefix ограничения.
8. Увеличивать cohort выше лимита PRD.
9. Вызывать provider для acceptance.
10. Мутировать KB, registry, Chroma, config.
11. Менять deterministic governance authority.
12. Делать Diagnostic Center runtime authority.
13. Считать успешный second supervised cycle разрешением на broad rollout.
14. Делать следующий переход без отдельного PRD.

## 5. Scope

### 5.1. Разрешённый cohort

Второй supervised eval cohort:

```text
max_cohort_size=6
allowed user ids:
- pilot_alpha
- pilot_beta
- pilot_gamma
- pilot_delta
- pilot_epsilon
- pilot_zeta
```

Важно:

```text
- cohort должен быть test-only;
- normal users не входят в cohort;
- user_id должен начинаться с `pilot_` или быть явно в allowlist;
- обычный user control case обязателен отдельно.
```

### 5.2. Минимальное разнообразие cases

Cohort должен покрывать минимум 6 типов сценариев:

```text
1. low_resource / support-first
2. reflective / open exploration
3. directive-seeking / wants answer
4. defensive / validation-seeking
5. KB-lens relevant but no direct quote
6. safety-sensitive but non-crisis boundary case
```

Если в текущих fixtures нет такого покрытия, добавить synthetic deterministic cases, но без provider calls и без raw private data.

## 8. Decision rules

### 8.1. `continue_supervised`

Разрешено только если:

```text
source_execution_gate_passed=true
cohort_size<=6
scenario_coverage_passed=true
test_apply_applied_count>0
normal_user_apply_count=0
default_off_user_path_effect_count=0
rollback_failure_count=0
stale_apply_after_force_disabled_count=0
candidate_weaker_than_baseline_count=0
safety_regression_count=0
kb_policy_regression_count=0
prompt_bloat_regression_count=0
constraint_conflict_regression_count=0
raw_kb_text_exposure_count=0
internal_only_exposure_count=0
not_for_direct_quote_violation_count=0
provider_called_by_continuation_count=0
production_mutation_detected=false
artifact_encoding_hygiene_passed=true
```

### 8.2. `stay_limited`

Использовать, если:

```text
hard blockers отсутствуют,
но evidence недостаточно для следующей supervised-фазы
или scenario coverage неполный
или test_apply_applied_count=0.
```

### 8.3. `hotfix_required`

Использовать, если:

```text
есть non-critical issue, который требует HF,
но нет hard safety/rollback/Kb/privacy blocker.
```

### 8.4. `stop`

Использовать при hard abort:

```text
normal_user_apply_count > 0
allowlist_violation_count > 0
rollback_failure_count > 0
stale_apply_after_force_disabled_count > 0
raw_kb_text_exposure_count > 0
internal_only_exposure_count > 0
not_for_direct_quote_violation_count > 0
safety_regression_count > 0
kb_policy_regression_count > 0
production_mutation_detected=true
```
