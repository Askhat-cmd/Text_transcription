# PRD-046.1.13 — Production-Limited Prompt-Constraint Pilot Execution / Monitoring Gate v1

## 0. Краткое название

**Production-Limited Prompt-Constraint Pilot Execution / Monitoring Gate v1**

## 1. Контекст

После `PRD-046.1.12` проект получил production-limited rollout plan для Prompt-Constraint Pilot.

Подтверждённое состояние после `PRD-046.1.12`:

```text
final_status=passed
decision=ready_for_production_limited_execution_prd
source_consolidation_passed=true
source_decision=prepare_production_limited_rollout_plan
rollout_plan_ready=true
cohort_policy_ready=true
preflight_gates_ready=true
operator_checklist_ready=true
monitoring_plan_ready=true
rollback_plan_ready=true
abort_criteria_ready=true
execution_performed=false
provider_called_by_plan=false
production_mutation_detected=false
default_flags_changed=false
artifact_encoding_hygiene_passed=true
```

Это разрешает выполнить **один короткий production-limited execution / monitoring gate**, но не разрешает broad rollout.

Текущий baseline остаётся conservative:

```text
PROMPT_CONSTRAINT_PILOT_ENABLED=false
PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=true
PROMPT_CONSTRAINT_PILOT_MODE=shadow|test_apply
```

Архитектурная позиция не меняется:

```text
Diagnostic Center v1: trace-only shadow layer
Planner Bridge: candidate-only / shadow-eval lineage
planner_bridge_compliance_shadow: trace-only compare layer
planner_bridge_writer_contract_pilot: pilot_shadow_only
writer_prompt_replay: offline_replay_only
prompt_constraint_pilot_runtime: default-off limited allowlisted/test path
```

## 2. Цель PRD

Выполнить **один строго ограниченный production-limited execution cycle** для Prompt-Constraint Pilot с полным monitoring gate.

Смысл PRD:

```text
не расширить rollout,
а проверить, что production-limited execution path можно безопасно включить,
наблюдать,
откатить,
и доказать отсутствие влияния на обычный пользовательский путь.
```

PRD должен:

1. Прочитать production-limited plan из `PRD-046.1.12`.
2. Проверить preflight gates.
3. Выполнить один короткий controlled production-limited execution window.
4. Ограничить execution максимум одним allowlisted operator/user id.
5. Выполнить normal-user control cases.
6. Собрать sanitized trace samples.
7. Сравнить baseline vs production-limited `test_apply`.
8. Проверить rollback через `PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=true`.
9. Проверить no-mutation.
10. Сформировать monitoring scorecard.
11. Принять решение:

```text
continue_limited
stay_limited
hotfix_required
stop
```

## 3. Главный принцип

```text
PRD-046.1.13 выполняет только один production-limited execution cycle.
Он не включает broad rollout и не меняет default runtime flags.
```

После PRD обычный пользовательский путь должен остаться неизменным:

```text
normal_user_apply_count=0
default_off_user_path_effect_count=0
normal_user_prompt_changed_by_pilot_count=0
normal_user_final_answer_changed_by_pilot_count=0
```

## 4. Важное ограничение по “реальному пользователю”

Этот PRD не должен автоматически выбирать настоящего пользователя из базы, логов или истории.

Разрешён только один из двух безопасных вариантов:

### Вариант A — operator-provided allowlisted id

Если в локальном окружении есть явно подготовленный operator-approved user id, runner может использовать его.

Требования:

```text
- user_id явно передан через CLI или allowlist artifact;
- user_id не содержит персональных данных;
- user_id sanitizable;
- user_id не берётся из raw private logs;
- user_id не выбирается автоматически.
```

### Вариант B — synthetic production-limited operator user

Если operator-approved id не предоставлен, использовать controlled synthetic id:

```text
prod_limited_operator_001
```

Это считается проверкой production-limited execution path, но не customer rollout.

Оба варианта остаются production-limited по runtime path, потому что используют production-limited флаги, allowlist, operator checklist, monitoring, rollback и no-mutation gates.

## 5. Non-goals

В рамках PRD запрещено:

1. Broad rollout.
2. Автоматическое включение pilot для всех.
3. Изменение default flags в config/code.
4. Делать `PROMPT_CONSTRAINT_PILOT_ENABLED=true` новым baseline default.
5. Делать `PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=false` новым baseline default.
6. Включать больше одного production-limited user id.
7. Выбирать реальных пользователей автоматически.
8. Читать raw private logs для выбора cohort.
9. Сохранять raw prompt, raw KB text, private user text.
10. Вызывать provider для acceptance.
11. Мутировать KB, registry, Chroma, config.
12. Менять deterministic governance authority.
13. Делать Diagnostic Center runtime authority.
14. Удалять rollback-first policy.
15. Считать один production-limited run разрешением на broad rollout.

## 6. Разрешённый результат PRD

Успешный PRD может завершиться одним из решений:

```text
continue_limited
stay_limited
```

`continue_limited` означает, что можно готовить следующий PRD на post-run quality gate / limited continuation planning.

`stay_limited` означает, что execution безопасен, но evidence недостаточно для продолжения.

Если есть проблемы:

```text
hotfix_required
stop
```

## 9. Decision rules

### 9.1. `continue_limited`

Разрешено только если:

```text
source_plan_gate_passed=true
preflight_passed=true
execution_window_count=1
target_user_count=1
target_user_limit_respected=true
production_limited_apply_count>=1
normal_user_apply_count=0
default_off_user_path_effect_count=0
normal_user_prompt_changed_by_pilot_count=0
normal_user_final_answer_changed_by_pilot_count=0
rollback_failure_count=0
stale_apply_after_force_disabled_count=0
candidate_weaker_than_baseline_count=0
safety_regression_count=0
kb_policy_regression_count=0
prompt_bloat_regression_count=0
constraint_conflict_regression_count=0
raw_kb_text_exposure_count=0
internal_only_exposure_count=0
not_for_direct_quote_violation_count=0
provider_called_by_execution_count=0
trace_sanitization_failed=false
production_mutation_detected=false
artifact_encoding_hygiene_passed=true
```

### 9.2. `stay_limited`

Использовать, если:

```text
hard blockers отсутствуют,
но production_limited_apply_count=0
или evidence недостаточно для следующей continuation фазы.
```

### 9.3. `hotfix_required`

Использовать при non-critical issue:

```text
operator notes missing
trace_samples_count lower than expected
minor monitoring artifact gap
non-blocking warning in runbook/checklist
```

### 9.4. `stop`

Использовать при hard abort:

```text
target_user_count > 1
normal_user_apply_count > 0
default_off_user_path_effect_count > 0
rollback_failure_count > 0
stale_apply_after_force_disabled_count > 0
safety_regression_count > 0
kb_policy_regression_count > 0
raw_kb_text_exposure_count > 0
internal_only_exposure_count > 0
not_for_direct_quote_violation_count > 0
trace_sanitization_failed=true
production_mutation_detected=true
```
