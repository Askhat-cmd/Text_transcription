# PRD-046.1.14 — Production-Limited Prompt-Constraint Pilot Results / Rollback & Quality Gate v1

## 0. Краткое название

**Production-Limited Results / Rollback & Quality Gate v1**

## 1. Контекст

После `PRD-046.1.13` проект впервые выполнил один controlled production-limited execution / monitoring gate для Prompt-Constraint Pilot.

Подтверждённое состояние после `PRD-046.1.13`:

```text
final_status=passed
decision=continue_limited
execution_window_count=1

target_source=synthetic_operator
real_user_count=0
synthetic_operator_user_count=1
target_user_limit_respected=true
production_limited_apply_count=1
normal_user_cases_total=2
normal_user_apply_count=0
default_off_user_path_effect_count=0
normal_user_prompt_changed_by_pilot_count=0
normal_user_final_answer_changed_by_pilot_count=0
rollback_failure_count=0
stale_apply_after_force_disabled_count=0
force_disabled_absolute_priority=true
candidate_weaker_than_baseline_count=0
safety_regression_count=0
kb_policy_regression_count=0
prompt_bloat_regression_count=0

provider_called_by_execution_count=0
trace_sanitization_failed=false
production_mutation_detected=false
artifact_encoding_hygiene_passed=true
```

Это даёт право не на broad rollout, а на **post-run results / rollback / quality gate**: зафиксировать результат production-limited цикла, проверить rollback и quality evidence, обновить риск-карту и принять решение о завершении текущей pilot-цепочки или необходимости hotfix.

Текущий baseline должен остаться conservative:

```text
PROMPT_CONSTRAINT_PILOT_ENABLED=false
PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=true
PROMPT_CONSTRAINT_PILOT_MODE=shadow|test_apply
```

Архитектурная позиция не меняется:

```text
Diagnostic Center v1: trace-only shadow layer
Planner Bridge: candidate-only / shadow-eval lineage
planner_bridge_compliance_shadow: trace-only compare layer
planner_bridge_writer_contract_pilot: pilot_shadow_only
writer_prompt_replay: offline_replay_only
prompt_constraint_pilot_runtime: default-off limited allowlisted/test path
```

## 2. Цель PRD

Создать **post-run quality gate** по результатам `PRD-046.1.13`.

PRD должен:

1. Прочитать production-limited execution artifacts из `PRD-046.1.13`.
2. Проверить source gate.
3. Сконсолидировать monitoring metrics.
4. Подтвердить rollback proof.
5. Подтвердить normal-user no-effect.
6. Подтвердить baseline vs test_apply quality.
7. Проверить trace sanitization.
8. Проверить no-mutation.
9. Создать post-run risk register.
10. Сформировать финальный quality decision.
11. Подготовить следующий шаг: `PRD-046.1.15 — Diagnostic Center stabilization / cleanup / eval harness consolidation`.
12. Не запускать новый production execution.
13. Не расширять rollout.
14. Не менять runtime defaults.

## 3. Главный принцип

```text
PRD-046.1.14 = post-run results gate only.
Он не делает новый rollout, не добавляет пользователей и не включает Diagnostic Center на всех.
```

После PRD обычный пользовательский путь должен остаться неизменным:

```text
normal_user_apply_count=0
default_off_user_path_effect_count=0
config_mutated=false
production_apply_performed=false
```

## 4. Non-goals

В рамках PRD запрещено:

1. Запускать новый production execution cycle.
2. Расширять cohort.
3. Включать broad rollout.
4. Включать pilot для обычных пользователей.
5. Менять default flags в config/code.
6. Делать `PROMPT_CONSTRAINT_PILOT_ENABLED=true` новым baseline default.
7. Делать `PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=false` новым baseline default.
8. Вызывать provider для acceptance.
9. Мутировать KB, registry, Chroma, config.
10. Менять deterministic governance authority.
11. Делать Diagnostic Center runtime authority.
12. Удалять rollback-first policy.
13. Удалять eval/trace artifacts до отдельного cleanup PRD.
14. Выполнять stabilization cleanup в этом PRD.

## 5. Разрешённый результат PRD

Успешный PRD может завершиться одним из решений:

```text
ready_for_stabilization_cleanup
stay_limited
hotfix_required
stop
```

### 5.1. `ready_for_stabilization_cleanup`

Использовать, если production-limited result прошёл все gates и проект готов к следующему этапу:

```text
PRD-046.1.15 - Diagnostic Center stabilization / cleanup / eval harness consolidation
```

### 5.2. `stay_limited`

Использовать, если hard blockers нет, но evidence недостаточно для stabilization.

### 5.3. `hotfix_required`

Использовать, если есть non-critical issue, который нужно закрыть до stabilization.

### 5.4. `stop`

Использовать, если найден hard safety/privacy/rollback/governance blocker.

## 8. Decision rules

### 8.1. `ready_for_stabilization_cleanup`

Разрешено только если:

```text
source_execution_gate_passed=true
source_final_status=passed
source_decision=continue_limited
source_execution_window_count=1
source_target_user_count=1
source_production_limited_apply_count>=1
quality_gate_passed=true
rollback_gate_passed=true
normal_user_gate_passed=true
trace_sanitization_gate_passed=true
risk_register_has_blockers=false
new_execution_performed=false
provider_called_by_results_gate=false
production_mutation_detected=false
artifact_encoding_hygiene_passed=true
```

### 8.2. `stay_limited`

Использовать, если:

```text
hard blockers отсутствуют,
но evidence недостаточно для stabilization cleanup
или risk register содержит medium risks, требующие дополнительного limited cycle.
```

### 8.3. `hotfix_required`

Использовать, если:

```text
есть non-critical artifact/report/risk issue,
но нет hard safety/privacy/rollback/governance blocker.
```

### 8.4. `stop`

Использовать при hard abort:

```text
normal_user_apply_count > 0
default_off_user_path_effect_count > 0
rollback_failure_count > 0
stale_apply_after_force_disabled_count > 0
safety_regression_count > 0

raw_kb_text_exposure_count > 0
```

2. Не начинать кодинг до task list.
3. Не запускать новый production execution.
4. Не менять baseline defaults.
5. Не включать broad rollout.
6. Не использовать новых пользователей.
7. Не коммитить `.env`, ключи, `.venv`, `node_modules`, sqlite/db/cache.
8. Все artifacts должны быть sanitized.
9. Raw private prompts, raw KB text и raw user private logs не коммитить.
10. Все изменения должны быть provider-free.
11. Все mutation-sensitive files должны иметь no-mutation proof.
12. Запустить тесты и CLI.
13. Сохранить command output.
14. Запустить artifact encoding hygiene gate.
15. Обновить reports и living docs.
16. Сделать commit и push в `main`.
17. После push обновить implementation report: commit hash + push status.
