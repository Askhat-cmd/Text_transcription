# PRD-046.1.15 — Diagnostic Center Stabilization / Cleanup / Eval Harness Consolidation

## 0. Краткое название

**Diagnostic Center Stabilization / Cleanup / Eval Harness Consolidation**

## 1. Контекст

После `PRD-046.1.14` проект завершил production-limited Prompt-Constraint Pilot цепочку post-run results gate.

Подтверждённое состояние после `PRD-046.1.14`:

```text
final_status=passed
decision=ready_for_stabilization_cleanup
source_execution_gate_passed=true
quality_gate_passed=true
rollback_gate_passed=true
normal_user_gate_passed=true
trace_sanitization_gate_passed=true
risk_register_has_blockers=false
new_execution_performed=false
provider_called_by_results_gate=false
production_mutation_detected=false
artifact_encoding_hygiene_passed=true
```

Перед этим были выполнены:

```text
PRD-046.1.9  — supervised execution, cohort=3
PRD-046.1.10 — supervised continuation, cohort=6
PRD-046.1.11 — supervised consolidation decision gate
PRD-046.1.12 — production-limited rollout plan
PRD-046.1.13 — one production-limited execution/monitoring gate
PRD-046.1.14 — post-run results / rollback / quality gate
```

Сейчас проекту нужен не новый rollout, а стабилизация: отделить runtime-critical код от eval/PRD harness, зафиксировать, какие gates должны остаться permanent regression gates, привести артефакты в порядок и подготовить transfer brief для перехода в новый чат.

Текущий baseline остаётся conservative:

```text
PROMPT_CONSTRAINT_PILOT_ENABLED=false
PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=true
PROMPT_CONSTRAINT_PILOT_MODE=shadow|test_apply
```

Архитектурная позиция на входе в PRD:

```text
Diagnostic Center v1: trace-only shadow layer
Planner Bridge: candidate-only / shadow-eval lineage
planner_bridge_compliance_shadow: trace-only compare layer
planner_bridge_writer_contract_pilot: pilot_shadow_only
writer_prompt_replay: offline_replay_only
prompt_constraint_pilot_runtime: default-off limited allowlisted/test path
```

## 2. Цель PRD

Выполнить **stabilization / cleanup / eval harness consolidation** после успешной pilot-цепочки.

PRD должен:

1. Создать инвентаризацию модулей Diagnostic Center / Prompt-Constraint Pilot.
2. Классифицировать файлы на категории:

```text
production_runtime
runtime_contracts
regression_gates
eval_harness
prd_artifacts
archive_candidates
docs_living_state
do_not_touch
```

3. Отделить runtime-critical код от временных PRD/eval harness слоёв на уровне документации и manifest-ов.
4. Сформировать список permanent regression gates, которые должны остаться в проекте.
5. Сформировать non-destructive cleanup / archive plan.
6. Убрать или пометить только явно временные debug/test artifacts, если это безопасно.
7. Не удалять production code.
8. Не мутировать KB, registry, Chroma, config.
9. Не менять runtime flags.
10. Подготовить transfer brief для перехода в новый чат.
11. Обновить living docs.

## 3. Главный принцип

```text
PRD-046.1.15 стабилизирует и наводит порядок.
Он не запускает новый rollout, не расширяет Diagnostic Center authority и не меняет production defaults.
```

## 4. Non-goals

В рамках PRD запрещено:

1. Запускать новый supervised или production-limited execution.
2. Расширять cohort.
3. Делать broad rollout.
4. Менять default flags.
5. Делать `PROMPT_CONSTRAINT_PILOT_ENABLED=true` новым baseline.
6. Делать `PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=false` новым baseline.
7. Давать Diagnostic Center прямую authority-роль.
8. Удалять runtime-critical код.
9. Удалять regression gates.
10. Мутировать KB, registry, Chroma, config.
11. Делать Chroma reindex.
12. Вызывать provider.
13. Удалять исторические PRD reports без archive manifest.
14. Переносить raw private logs в архив.
15. Архивировать файлы без явного manifest, reason и rollback path.

## 5. Что разрешено менять

Разрешено:

```text
- добавить stabilization/inventory modules;
- добавить CLI для inventory/consolidation;
- добавить manifest artifacts;
- добавить/обновить docs;
- добавить tests;
- создать archive plan;
- переместить только явно временные debug artifacts, если они:
  1) не runtime-critical,
  2) не нужны как permanent regression evidence,
  3) не содержат raw private data,
  4) имеют manifest,
  5) имеют rollback path.
```

Если есть сомнения, файл не перемещать, а только пометить как `archive_candidate`.

## 7. Обязательные категории классификации

### 7.1. `production_runtime`

Сюда попадают файлы, которые участвуют в runtime path или могут быть runtime-critical.

Примеры:

```text
orchestrator/runtime integration
writer prompt optional apply path
prompt_constraint_pilot_runtime builder/formatter
knowledge policy guardrails
context assembly integration
diagnostic center shadow integration
```

Правила:

```text
- не удалять;
- не перемещать;
- не архивировать;
- только документировать.
```

### 7.2. `runtime_contracts`

Схемы и dataclass/Pydantic contracts, необходимые для trace/runtime compatibility.

Правила:

```text
- не удалять;
- не переименовывать без отдельного PRD;
- включить в regression catalog.
```

### 7.3. `regression_gates`

Тесты и CLI, которые должны остаться как permanent regression gates.

Минимум сохранить gates для:

```text
default-off no effect
FORCE_DISABLED rollback priority
allowlist enforcement
normal-user no-effect
safety gate
KB policy gate
raw KB exposure gate
internal_only exposure gate
not_for_direct_quote gate
prompt bloat gate
constraint conflict gate
trace sanitization
artifact encoding hygiene
no-mutation proof
```

### 7.4. `eval_harness`

PRD-specific deterministic runners / harnesses, которые не участвуют в runtime path, но нужны для future regression/eval.

Правила:

```text
- не удалять в этом PRD;
- пометить как eval_harness;
- определить, какие стоит оставить permanent, а какие можно архивировать позже.
```

### 7.5. `prd_artifacts`

Исторические logs/reports/scorecards по PRD.

Правила:

```text
- не удалять;
- можно создать compact index;
- можно пометить heavy artifacts как archive_candidates;
- raw/private artifacts не архивировать и не коммитить.
```

### 7.6. `archive_candidates`

Файлы, которые потенциально можно убрать из активного пути.

Критерии:

```text
- не импортируются runtime code;
- не нужны active regression gate;
- не содержат unique acceptance evidence;
- не содержат private/raw data;
- имеют safe rollback path.
```

В рамках этого PRD разрешено только:

```text
- пометить archive_candidate;
- создать archive_manifest;
- не удалять физически, если есть сомнения.
```

### 7.7. `docs_living_state`

Living docs:

```text
docs/PROJECT_STATE.md
docs/ROADMAP.md
docs/PRD_INDEX.md
docs/DECISIONS.md
```

Правила:

```text
- обновить согласно текущему stage;
- сократить шум только если безопасно;
- не терять critical decisions.
```

### 7.8. `do_not_touch`

Файлы, которые нельзя менять:

```text
KB all_blocks_merged
registry
Chroma DB
config defaults
.env
provider secrets
raw private logs
```
