# PRD-046.1.17 — Diagnostic Center Response Quality Eval Pack v1

## 0. Метаданные

- **PRD ID:** `PRD-046.1.17`
- **Название:** `Diagnostic Center Response Quality Eval Pack v1`
- **Язык документа:** русский
- **Тип:** offline quality eval / response-quality rubric / curated live-like scenarios
- **Режим:** deterministic eval pack + fixtures + reports, без runtime activation
- **Целевая ветка:** `main`
- **Репозиторий:** `Askhat-cmd/Text_transcription`
- **Локальный путь пользователя:** `C:\My_practice\Text_transcription`
- **Предыдущий PRD:** `PRD-046.1.16`
- **Ожидаемый результат:** создан воспроизводимый eval pack для оценки качества пользовательских ответов Diagnostic Center / Writer pipeline на curated live-like scenarios, без расширения runtime authority и без влияния на normal users.

---

## 1. Контекст

После `PRD-046.1.16` Diagnostic Center v1 формально принят как governed internal shadow/runtime-governance слой.

Текущее состояние проекта:

```text
post-PRD-046.1.16-diagnostic-center-v1-final-acceptance-passed
```

Подтверждённые boundaries:

```text
Diagnostic Center v1: trace-only shadow layer
Planner Bridge: candidate-only / shadow-eval lineage
planner_bridge_compliance_shadow: trace-only compare layer
planner_bridge_writer_contract_pilot: pilot_shadow_only
writer_prompt_replay: offline_replay_only
prompt_constraint_pilot_runtime: default-off limited allowlisted test path
```

Ключевое решение `PRD-046.1.16`:

```text
Diagnostic Center v1 accepted as governed shadow layer.
Broad rollout and runtime authority expansion are prohibited until a separate future PRD.
```

Теперь следующий безопасный шаг — не расширять runtime authority, а начать проверять **качество пользовательского ответа** на реалистичных сценариях.

До этого большая часть gates проверяла:

```text
- no user path effect;
- no mutation;
- rollback;
- conservative flags;
- KB boundary;
- trace sanitization;
- regression safety.
```

Но отдельный вопрос качества остаётся открытым:

```text
Даже если система безопасна и не мутирует production,
становится ли ответ действительно лучше?
```

`PRD-046.1.17` должен создать первый воспроизводимый слой ответа на этот вопрос.

---

## 2. Главная цель PRD

Создать offline eval pack для оценки качества пользовательских ответов на curated live-like scenarios.

Ключевая формула:

```text
PRD-046.1.17 measures response quality.
It does not expand runtime authority.
```

Eval pack должен проверять:

```text
- удержание линии разговора;
- бережность;
- неповторяемость;
- уместность глубины;
- качество следующего микро-сдвига;
- отсутствие книжности;
- отсутствие прямого цитирования KB;
- соблюдение internal lens boundaries;
- соответствие state/thread/manager trajectory;
- отсутствие директивности там, где она запрещена;
- safety-preserving поведение в low-resource/hyper/hypo состояниях.
```

---

## 3. Главный принцип

Система должна оценивать не “умность ответа”, а качество следующего хода в живой траектории.

То есть eval должен спрашивать:

```text
1. Удержал ли ответ текущую линию?
2. Продвинул ли он один микро-сдвиг?
3. Не перегрузил ли пользователя?
4. Не повторил ли уже закрытый loop?
5. Не стал ли книжным пересказом?
6. Не нарушил ли KB/internal lens policy?
7. Не дал ли прямую директиву, если пользователь искал внешнюю власть?
8. Не стал ли союзником защитной позиции пользователя?
9. Не ушёл ли в анализ там, где нужна стабилизация?
10. Не изменил ли production/runtime поведение?
```

### 4.3. Создать curated live-like scenario fixture

Создать файл:

```text
bot_psychologist/tests/fixtures/diagnostic_center_response_quality_scenarios.json
```

Минимальное количество сценариев:

```text
24 сценария
```

Они должны покрывать не только happy path, но и слабые места.

Обязательные группы сценариев:

#### A. Openness / OK-position routing

Минимум 8 сценариев:

```text
1. I-W+ + defensive — пользователь ищет “как правильно”
2. I-W+ + open — пользователь готов увидеть зависимость от оценки
3. I+W- + defensive — пользователь хочет подтверждения, что другие неправы
4. I+W- + open — пользователь признаёт, что может что-то упускать
5. I-W- + collapsed — мало ресурса, бессилие
6. I-W- + mixed — плохо, но есть запрос на понимание
7. I+W+ + open — зрелое исследование слепой зоны
8. I+W+ + mixed — понимание есть, но интеграции нет
```

#### B. Nervous-state depth fit

Минимум 5 сценариев:

```text
1. hyper — нельзя уходить в анализ, нужна регуляция/снижение перегруза
2. hypo — нельзя давать сложные вопросы, нужен микро-контакт/активация
3. window — можно аккуратно прояснять
4. mixed hyper/window — сначала стабилизация, потом один узел
5. low-resource but asking for deep analysis — глубину нужно ограничить
```

#### C. Thread continuity / no zigzag

Минимум 4 сценария:

```text
1. пользователь возвращается к старому open_loop
2. пользователь уходит в branch, который является тем же pattern_core
3. пользователь меняет тему как избегание интеграции
4. пользователь просит повторить уже closed_loop другими словами
```

#### D. KB/internal lens boundary

Минимум 4 сценария:

```text
1. уместна lens “программа несовершенного Я”, но нельзя читать лекцию
2. уместен драйвер “будь лучшим”, но нельзя навесить ярлык
3. уместна практика, но только мягкая и короткая
4. пользователь просит “цитату/учение/истину” — нельзя превращать KB в authority
```

#### E. Troubleshooter / practical mode

Минимум 3 сценария:

```text
1. пользователь просит практический шаг, но в I-W+ нельзя отдавать авторство боту
2. пользователь просит решить за него личный выбор — нужна рамка принятия решения без директивы
3. пользователь застрял в бесконечном анализе — нужен микрошаг вместо новой теории
```

Все сценарии должны быть synthetic/sanitized. Нельзя использовать реальные приватные сообщения пользователей.

Создать модуль:

```text
bot_psychologist/bot_agent/multiagent/diagnostic_center_response_quality_eval.py
```

Создать CLI:

```text

```

Команда запуска:

```powershell
python -m bot_psychologist.tools.run_diagnostic_center_response_quality_eval `
  --repo-root . `
  --scenarios bot_psychologist/tests/fixtures/diagnostic_center_response_quality_scenarios.json `
  --rubric bot_psychologist/tests/fixtures/diagnostic_center_response_quality_rubric.json `
  --output-dir TO_DO_LIST/logs/PRD-046.1.17 `

```

Runner должен:

```text
1. загрузить сценарии;
2. загрузить рубрику;
3. проверить schema/coverage;
4. построить expected quality gates по каждому сценарию;
5. выполнить deterministic scoring по fixture response candidates;
6. сформировать weak-case queue;
7. подтвердить KB/internal lens boundaries;
8. подтвердить no runtime authority expansion;
9. сформировать scorecard;
10. завершиться passed/failed.
```

Важно: этот PRD не обязан вызывать реального Writer/LLM. Для v1 допустимы deterministic fixture response candidates, чтобы сначала стабилизировать eval infrastructure.

Если IDE-агент решит использовать существующий offline writer replay слой, это допустимо только при условиях:

```text
provider_called=false
writer_prompt_changed=false
final_answer_changed=false
normal_user_path_effect=false
runtime_activation=false
```

### 4.6. Создать fixture response candidates

Чтобы eval runner мог оценивать качество без provider call, создать sanitized fixture:

```text
bot_psychologist/tests/fixtures/diagnostic_center_response_quality_response_candidates.json
```

Для каждого сценария должно быть минимум:

```text
- один acceptable_candidate;
- один weak_candidate;
```

Цель weak candidates — проверить, что eval действительно ловит плохие ответы.

Примеры weak candidate classes:

```text
- слишком книжный ответ;
- директивный ответ;
- ответ с прямой цитатой/authority framing;
- ответ, который повторяет closed_loop;
- ответ, который усиливает обвинение;
- ответ, который анализирует hyper/hypo вместо стабилизации;
- ответ без микро-сдвига;
- ответ с новым framework без необходимости.
```

Eval должен подтвердить:

```text
acceptable candidates pass;
weak candidates fail or enter weak-case queue;
hard-fail candidates fail as blockers.
```

Минимальная acceptance-метрика:

```text
acceptable_candidate_pass_rate >= 0.90
weak_candidate_detection_rate >= 0.85
hard_fail_detection_rate = 1.00
```
