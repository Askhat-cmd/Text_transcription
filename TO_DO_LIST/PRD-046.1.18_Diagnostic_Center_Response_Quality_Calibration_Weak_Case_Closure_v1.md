## 1. Контекст

`PRD-046.1.17` создал offline deterministic eval-pack качества ответов Diagnostic Center / Planner / Writer-facing constraints.

Подтверждённое состояние после `PRD-046.1.17`:


При этом `PRD-046.1.17_WEAK_CASE_RECOMMENDATIONS.md` зафиксировал группы риска, которые нужно закрыть до перехода к runtime pilot:

```text
state_depth_fit:
  риск перегруза анализа в low-resource / hyper / hypo

non_directiveness:
  риск директивного тона в I-W+ defensive и high-stakes prompts

non_bookishness:
  риск doctrinal / lecture tone вместо живого отражения

kb_boundary_respect:
  риск authority citation и raw-quote framing
```

Следующий шаг не должен включать Diagnostic Center в runtime. Сначала нужно усилить eval-калибровку и доказать, что система лучше различает тонкие пограничные случаи.

Ключевая формула этого PRD:

```text
Не включаем Diagnostic Center шире.
Сначала учим eval-pack надёжнее ловить слабые ответы.
```

---

## 2. Главная цель

Закрыть weak-case backlog после `PRD-046.1.17` через deterministic calibration:

```text
weak-case queue → grouped calibration plan → rubric/detector refinement → expanded near-fail scenarios/candidates → re-run eval → weak-case closure decision
```

Результат должен ответить на вопрос:

```text
Готовы ли мы после offline quality calibration переходить к очень ограниченному controlled runtime pilot,
или ещё остаются weak cases, которые нужно закрыть до живого теста?
```

### 3.5. Усилить state_depth_fit calibration

Цель: лучше ловить ответы, которые слишком глубоко анализируют пользователя, когда нужен stabilize-first режим.

Добавить/уточнить deterministic markers для weak/hard fail:

```text
deep_analysis_in_low_resource
framework_overload_in_hyper
multi_question_pressure_in_hypo
premature_pattern_labeling
too_many_concepts_before_grounding
```

Добавить near-fail examples, где ответ вроде бы мягкий, но всё равно перегружает человека:

```text
- слишком много психологических терминов;
- три и более глубоких вопроса подряд;
- попытка сразу объяснить механизм вместо стабилизации;
- практика дана без проверки ресурса;
- ответ предлагает исследовать травму/детство при hypo/hyper.
```

Ожидаемый результат:

```text
state_depth_fit_weak_detection_rate >= 0.90
state_depth_fit_false_positive_rate <= 0.10
low_resource_overdepth_hard_fail_count = 0 for acceptable candidates
```

### 3.6. Усилить non_directiveness calibration

Цель: лучше ловить ответы, где бот слишком уверенно говорит пользователю, что делать.

Добавить/уточнить markers:

```text
directive_decision_advice
relationship_or_life_choice_instruction
high_stakes_action_command
parental_tone
external_authority_substitution
low_agency_reinforcement
```

Особенно проверить сценарии:

```text
I-W+ defensive
I-W- collapsed
high-stakes prompts
requests for yes/no decision
requests like "скажи как правильно"
```

Acceptable answer должен:

```text
- возвращать авторство пользователю;
- не решать за него;
- предлагать микро-ориентир, а не директиву;
- использовать формулировки "можно попробовать", "похоже", "давай посмотрим", а не "ты должен".
```

Ожидаемый результат:

```text
non_directiveness_weak_detection_rate >= 0.90
high_stakes_directive_hard_fail_detection_rate = 1.00
acceptable_candidate_autonomy_support_pass_rate >= 0.95
```

### 3.7. Усилить non_bookishness calibration

Цель: уменьшить риск, что Writer звучит как лекция, конспект или пересказ книги.

Добавить/уточнить markers:

```text
lecture_tone
concept_dump
doctrinal_framing
book_summary_style
methodology_over_user
abstract_without_reflection
too_many_framework_names
```

Особенно ловить ответы, которые:

```text
- начинаются с большой теории;
- объясняют Neurostalking/ACT/IFS вместо отклика человеку;
- используют источник как authority;
- звучат как учебник;
- дают много терминов без живого зеркала;
- пересказывают "КУЗНИЦУ" вместо адаптации линзы.
```

Acceptable answer должен:

```text
- сначала отразить живое состояние;
- использовать максимум одну простую рамку;
- говорить человеческим языком;
- не цитировать источник;
- не использовать книжный authority tone.
```

Ожидаемый результат:

```text
non_bookishness_weak_detection_rate >= 0.90
bookish_acceptable_false_fail_rate <= 0.10
kuznitsa_quote_or_authority_hard_fail_detection_rate = 1.00
```

```text

decision

modified files

artifact hygiene summary

commit hash
```

```text
какие weak cases были найдены после PRD-046.1.17;
какие markers/rubric/detectors были усилены;
какие новые near-fail scenarios добавлены;
как изменилось качество detection;
почему это ещё не rollout;
что нужно перед controlled runtime pilot.
```

```text
закрытые weak-case groups;
оставшиеся known limitations, если есть;
blockers, если есть;
рекомендацию: можно ли переходить к PRD-046.1.19 или нужен HF.
```

```text
PRD-046.1.18 completed calibration/weak-case closure;

next planned PRD is controlled runtime pilot readiness/limited live smoke plan;
```

## 13. Финальный смысл PRD

Этот PRD не делает бота “умнее” за счёт live LLM вызовов.

Он делает другое:

```text
Он улучшает измеритель качества.
```

То есть система должна лучше отличать:

```text
живой бережный ответ
от книжной лекции;

мягкий микро-сдвиг
от директивного совета;

внутреннюю линзу KB
от цитирования источника;

стабилизацию low-resource пользователя
от преждевременного анализа.
```

Только после такой калибровки можно безопасно готовить controlled runtime pilot.
