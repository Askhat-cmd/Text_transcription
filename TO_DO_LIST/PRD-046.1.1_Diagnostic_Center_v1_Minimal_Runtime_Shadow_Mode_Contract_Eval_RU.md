## 0. Краткое резюме для IDE-агента

`PRD-046.1` создал внутренний контракт Diagnostic Center v1:

```text
DiagnosticCenterInput
DiagnosticCenterOutput
DiagnosticHypothesis
DiagnosticLensSignal
NextMicroShift
DiagnosticCenterTrace
deterministic dry-run builder
contract audit
10 eval cases
```

Теперь нужно сделать следующий маленький шаг: **подключить Diagnostic Center v1 в runtime только как shadow trace**.

Главное:

```text
Diagnostic Center должен строиться на реальном runtime-снимке,
но не должен влиять на ответ пользователю.
```

То есть:

```text
State Analyzer
→ Thread Manager
→ Memory Retrieval
→ Context Assembly
→ Diagnostic Card
→ Diagnostic Center Shadow Output   # trace-only
→ Writer Move / Writer / Validator  # без изменения поведения
```

В этом PRD запрещено:

```text
- менять Writer prompt;
- передавать Diagnostic Center output в WriterContract;
- менять финальный ответ;
- включать Diagnostic Center как active planning layer;
- делать provider calls;
- делать Chroma reindex;
- мутировать KB/registry/config;
- расширять Diagnostic Center в большой LLM-агент.
```

## 2. Почему нужен этот PRD

Контракт уже есть, но он пока проверяется на fixture/eval данных. Следующий риск — подключить его сразу в планирование ответа и незаметно сломать runtime.

Правильный следующий шаг:

```text
реальный runtime signal
→ shadow DiagnosticCenterOutput
→ сравнение с Diagnostic Card / trace
→ divergence metrics
→ no user-path effect
```

Это позволит увидеть:

```text
- как Diagnostic Center ведёт себя на реальных State/Thread/Context данных;
- совпадает ли safety priority с текущим Diagnostic Card;
- не конфликтует ли next_micro_shift с suggested_writer_move;
- не создаёт ли Diagnostic Center лишнюю глубину;
- не пытается ли KB использоваться как direct quote source;
- не ломает ли shadow path latency / trace / debug.
```

## 3. Goals

### 3.1. Добавить shadow adapter

Создать небольшой модуль-адаптер, который превращает уже существующие runtime-сущности в `DiagnosticCenterInput`.

Адаптер должен принимать:

```text
query / user_message
state_snapshot
updated_thread
context_package
memory_bundle
diagnostic_card
thread_debug
```

И возвращать:

```text
DiagnosticCenterOutput
shadow_trace
divergence_metrics
```

### 3.2. Встроить shadow call в orchestrator без влияния на ответ

В `MultiAgentOrchestrator.run()` добавить вызов Diagnostic Center shadow после сборки:

```text
context_package
diagnostic_card
```

Но до или после Writer — только если это не меняет Writer input.

Рекомендуемый вариант:

```text
context_package = ...
diagnostic_card = ...
diagnostic_center_shadow = build_shadow(...)
writer_contract = WriterContract(...)
draft_answer = await writer_agent.write(writer_contract)
...
return debug + diagnostic_center_shadow
```

Важно:

```text
Diagnostic Center shadow output не должен попадать в WriterContract.
Diagnostic Center shadow output не должен менять diagnostic_card.
Diagnostic Center shadow output не должен менять writer prompt.
Diagnostic Center shadow output не должен менять validation_result.
```

### 3.3. Добавить debug trace

В `debug` ответа добавить отдельный блок:


## 6. Архитектурные правила shadow mode

### 6.1. Shadow mode не влияет на пользователя

Hard invariant:

```text
final_answer_with_shadow == final_answer_without_shadow
```

Если exact equality трудно проверить из-за LLM nondeterminism, тест должен использовать stub/fake writer или deterministic writer path.

Минимальное доказательство:

```text
writer_contract_changed = false
writer_prompt_changed_by_shadow = false
diagnostic_center_output_passed_to_writer = false
final_answer_changed_by_shadow = false
```

### 6.2. Shadow failure is non-blocking

Если Diagnostic Center shadow падает:

```text
- ответ пользователю всё равно генерируется старым путём;
- debug получает diagnostic_center_shadow.status = "error";
- error sanitized;
- raw payload/private data не пишется в git.
```

### 6.3. Safety priority must match

Если:

```text
state_snapshot.safety_flag = true
thread_state.safety_active = true
nervous_state = hyper|hypo
intent = crisis
```

shadow output должен:

```text
status = safety_first
depth_allowed = none|low
response_goal = safety_redirect|ground|contain
```

И divergence metric:

```text
safety_priority_match = true
```

### 6.4. KB boundary must remain strict

Diagnostic Center shadow может видеть только governed/sanitized KB metadata.

Запрещено:

```text
- raw full KB text in shadow trace;
- source direct quote;
- content_full;
- long unsafe source fragment;
- обход knowledge_policy.
```

Hard metric:

```text
raw_kb_text_exposed = false
must_not_quote_source = true
kb_usage_mode = internal_lens_only|disabled
```

### 6.5. Diagnostic Center is still not Planner

Shadow output может иметь `next_micro_shift`, но он не должен исполняться.

То есть:

```text
next_micro_shift = observed recommendation
not runtime command
not writer instruction
not active planner output
```

## 8. Mapping runtime → DiagnosticCenterInput

Адаптер должен аккуратно маппить существующие поля.

### 8.1. StateSnapshot

```text
state_snapshot.nervous_state → nervous_state
state_snapshot.intent → intent
state_snapshot.openness → openness
state_snapshot.ok_position → ok_position
state_snapshot.safety_flag → safety_flag
state_snapshot.confidence → context_signals.state_confidence
```

Если поля отсутствуют:

```text
use "unknown"
do not fail
add trace warning
```

### 8.2. ThreadState

```text
thread_state.relation_to_thread → relation_to_thread
thread_state.phase → phase
thread_state.response_mode → response_mode
thread_state.pattern_core → pattern_core
thread_state.thread_action → thread_action
thread_state.active_frame → context_signals.active_frame
thread_state.open_loops → context_signals.open_loops
thread_state.closed_loops → context_signals.closed_loops
thread_state.continuity_score → context_signals.continuity_score
```

### 8.3. ContextAssemblyPackage

```text
knowledge_rag_hits → sanitized knowledge_hits
semantic_memory_hits count only unless sanitized
trace flags → context_signals.context_trace_flags
recent_turns count → context_signals.recent_turns_count
```

### 8.4. MemoryBundle / KB hits

Diagnostic Center shadow should not receive raw content.

Allowed KB fields:

```text
lens_family
score
source_id
block_id
allowed_use
safety_flags
summary/safe_preview only if already sanitized and short
```

Forbidden KB fields:

```text
content_full
raw_text
full_text
text
source_text
page_content
```

If forbidden fields are present in input, adapter must strip them.

## 9. Divergence scoring

Create deterministic scoring helper:

```python
def build_diagnostic_center_shadow_divergence_v1(
    *,
    diagnostic_card: DiagnosticCard,
    diagnostic_center_output: DiagnosticCenterOutput,
    thread_state: ThreadState,
    state_snapshot: StateSnapshot,
) -> dict[str, Any]:
    ...
```

Minimum output:

```json
{
  "safety_priority_match": true,
  "depth_compatible": true,
  "writer_move_compatible": true,
  "response_mode_compatible": true,
  "relation_to_thread_match": true,
  "phase_match": true,
  "pattern_core_present": true,
  "kb_boundary_ok": true,
  "raw_kb_text_exposed": false,
  "thread_risk": "none",
  "warnings": []
}
```

### 9.1. Hard blockers

These must fail the eval:

```text
safety_priority_match = false
raw_kb_text_exposed = true
kb_boundary_ok = false
shadow exception blocks final answer
Diagnostic Center output passed to Writer
Writer prompt changed by shadow
no_mutation_proof failed
```

### 9.2. Soft warnings

These should be reported, but not necessarily fail:

```text
writer_move_compatible = false
response_mode_compatible = false
phase_match = false
pattern_core_present = false
thread_risk = zigzag|repeat|premature_depth
```

Reason: in shadow mode we are learning where Diagnostic Center differs from current Diagnostic Card.

## 15. Acceptance Criteria

PRD считается выполненным, если:

```text
1. Diagnostic Center shadow output появляется в debug trace.
2. Shadow output строится на реальных runtime-сущностях.
3. Shadow output не попадает в WriterContract.
4. Shadow output не меняет Writer prompt.
5. Shadow output не меняет final answer.
6. Shadow failure не блокирует ответ.
7. Safety priority match rate = 1.0.
8. KB boundary violations = 0.
9. Raw KB text exposure count = 0.
10. Contract/eval/shadow tests passed.
11. No-mutation proof passed.
12. Нет лишних debug-директорий вне TO_DO_LIST/logs/PRD-046.1.1/.
13. Orchestrator изменён минимально, основная логика вынесена в отдельные модули.
14. docs/PROJECT_STATE.md, docs/ROADMAP.md, docs/PRD_INDEX.md обновлены.
15. Implementation report обновлён после финального push и содержит commit hash + push status.
```

## 18. Рекомендуемый final_status

Возможные статусы:

```text
passed
done_with_shadow_divergence_warnings
done_with_shadow_blocker
failed_safety_violation
```

Правила:

```text
passed:
  no hard blockers, eval passed, no-mutation passed, user path unchanged.

done_with_shadow_divergence_warnings:
  hard blockers отсутствуют, но есть soft warnings по alignment/phase/writer_move.

done_with_shadow_blocker:
  shadow работает, но есть hard blocker в eval/trace/user-path proof.

failed_safety_violation:
  shadow повлиял на user-facing answer, writer prompt, KB/governance, Chroma, registry, config или safety priority.
```
