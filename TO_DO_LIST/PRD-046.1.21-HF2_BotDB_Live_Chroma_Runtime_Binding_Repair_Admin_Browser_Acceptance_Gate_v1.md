## 2. Главная цель PRD

Закрыть именно live-проблему, которую видит пользователь в браузере и бот в runtime.

PRD-HF2 считается успешным только если:

```text
Dashboard visible ChromaDB = 247
Dashboard visible Chroma status = ok
/api/dashboard returns chroma.count=247 and chroma.status=ok

/api/query/ returns HTTP 200 and hits > 0
bot retrieval smoke shows semantic_fallback_used=false

registry focus source remains protected
KB text/governance not mutated
```

Важно:

```text
Нельзя принимать PRD как passed, если direct diagnostic показывает 247,
но browser/admin/API всё ещё показывают Chroma unavailable.
```

## 4. Root-cause hypothesis

HF1 показал противоречие:

```text
direct diagnostic:
  Chroma count = 247
  source_ids = ["123__кузница_духа"]
  query_smoke hits_count = 1

live dashboard / registry stats:
  chroma.status = unavailable
  chroma.count = 0
  chroma_error = object of type 'int' has no len()

live query:
  /api/query/ = 503
```

Рабочие гипотезы:

```text
1. live PipelineRunner держит stale/corrupted Chroma client/collection;
2. dashboard/registry/query используют старый singleton runner, который не refresh/rebind после reindex;
3. direct diagnostic создаёт fresh client и поэтому видит 247;
4. live API процесс после reindex не перезапускается или не обновляет Chroma binding;
5. query route fallback не активируется в реальном live server или работает не в том процессе;
6. Dashboard использует runner.chroma_manager.get_stats(), который всё ещё падает на stale collection.
```

HF2 должен не гадать, а доказать артефактами, какая из гипотез верна.

## 10. Required code repair

### 10.1. Chroma runtime health adapter

Создать `Bot_data_base/storage/chroma_runtime_health.py`.

Он должен давать единый безопасный способ получить live Chroma state:

```python
get_chroma_runtime_health(config_path: str) -> dict
```

Ожидаемые поля:

```text
status=ok|unavailable
count=<int>
source_ids=[...]
count_by_source_id={}
persist_directory=<path>
collection_name=<name>
fresh_client_used=true|false
error_code=<code|null>
error_message_sanitized=<string|null>
```

Важно:

```text
- не возвращать traceback в API;
- если stale manager падает, попробовать fresh client probe;
- если fresh client успешен, API должен использовать fresh result для dashboard/stats.
```

### 10.2. ChromaManager rebind

В `ChromaManager` добавить безопасный метод:

```python
refresh_collection_binding() -> dict
```

И/или:

```python
get_stats_safe(refresh_on_error=True) -> dict
```

Логика:

```text
1. попробовать current _collection;
2. если ошибка object of type 'int' has no len или похожая Chroma segment error:
   - пересоздать client или collection binding;
   - повторить count/get один раз;
3. если direct/fresh path работает — вернуть ok;
4. если не работает — вернуть unavailable без падения API.
```

### 10.3. Dashboard

`/api/dashboard` не должен показывать `Chroma unavailable`, если fresh runtime health видит `247`.

Успешное API:

```json
{
  "chroma": {
    "status": "ok",
    "count": 247,
    "indexed_source_ids": ["123__кузница_духа"]
  }
}
```

### 10.4. Registry stats

`/api/registry/stats` должен вернуть:

```json
{
  "chroma_total": 247,
  "chroma_status": "ok",
  "chroma_error_code": null
}
```

Degraded mode допустим только как failed/blocker state, не как passed acceptance.

### 10.5. Query route

`POST /api/query/` не должен возвращать 503, если:

```text
- all_blocks_merged.json доступен;
- direct/fresh Chroma count = 247;
- или можно выполнить safe fallback to all_blocks for recovery mode.
```

Acceptance для query:

```text
query_http_200=true
query_hits_count>0
query_error_chromadb_unavailable=false
semantic_fallback_used=false
botdb_circuit_open=false
```

Если используется emergency all_blocks fallback внутри BotDB query route, это должно быть явно отражено как:

```text
botdb_query_route_fallback_used=true
```

Но это не должно открывать Bot Agent semantic fallback. Бот должен получать HTTP 200 от BotDB.
