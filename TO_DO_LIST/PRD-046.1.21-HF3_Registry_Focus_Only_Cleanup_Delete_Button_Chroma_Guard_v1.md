## 6. Исправить delete/source_exists robustness

### 6.1. Проблема

Сейчас delete route делает:

```python
chroma_deleted = runner.chroma_manager.delete_source(source_id)
removed = runner.registry.delete_source(source_id)
```

Если Chroma падает на `source_exists` или `delete_source`, UI получает ошибку, даже если source является registry-only мусором и в Chroma фактически нет его записей.

### 6.2. Нужно сделать

В `Bot_data_base/api/routes/registry.py` и/или helper module добавить safe delete layer:

```text
_safe_chroma_delete_source(runner, source_id, expected_no_chroma_records)
```

Поведение:

```text
1. Если source_id пустой → deleted=0, status=skipped_empty_source_id.
2. Если source_id = focus source → hard block.
3. Если Chroma health ok и source_ids only ["123__кузница_духа"] → для non-focus source считать chroma_records=0 и не падать.
4. Если runner.chroma_manager.delete_source(source_id) падает:
   - не пробрасывать raw exception в UI;
   - вернуть sanitized warning;
   - разрешить registry delete только если independent proof говорит, что source отсутствует в Chroma.
5. Если Chroma source presence unknown → не удалять, вернуть controlled 409 с понятным сообщением, но без traceback/source_exists int len.
```

UI не должен показывать:

```text
source_exists_error: object of type 'int' has no len()
```

Вместо этого:

```text
Удаление недоступно: не удалось безопасно проверить Chroma
```

или, если proof есть:

```text
Источник удалён из реестра; записей Chroma для него не найдено
```
