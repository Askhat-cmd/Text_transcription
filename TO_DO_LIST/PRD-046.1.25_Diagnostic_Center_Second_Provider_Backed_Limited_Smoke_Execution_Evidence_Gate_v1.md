# PRD-046.1.25 — Diagnostic Center Second Provider-Backed Limited Smoke Execution & Evidence Gate v1

## 0. Краткий обзор PRD

Этот PRD выполняет **второй controlled provider-backed limited smoke** для Diagnostic Center, но без broad rollout и без включения Diagnostic Center для normal users.

PRD не является production rollout. Его задача — аккуратно расширить доказательную базу после успешного PRD-046.1.23/046.1.24: проверить ещё один ограниченный provider-backed прогон на allowlisted synthetic operator, подтвердить rollback-first поведение, normal-user no-effect, safety/KB boundary, sanitized traces, BotDB stability и no-mutation.

Главный принцип этого PRD: **не делать отдельный planning-only шаг без необходимости**, потому что предыдущие gates зелёные. Поэтому PRD объединяет source gates, preflight, controlled execution и evidence capture в одном безопасном цикле.

## 2. Цель

Выполнить второй provider-backed limited smoke для Diagnostic Center в controlled режиме:

```text
source gates
→ BotDB/live preflight
→ rollback-first precheck
→ allowlisted synthetic operator execution
→ normal-user controls
→ provider budget gate
→ quality/micro-shift evidence
→ safety/KB boundary evidence
→ sanitized trace evidence
→ rollback postcheck
→ no-mutation proof
→ docs/report sync
```

Цель — получить дополнительное runtime/provider evidence без расширения normal-user path.

---

## 3. Почему это следующий безопасный шаг

`PRD-046.1.23` уже выполнил первый provider-backed limited smoke.

`PRD-046.1.24` отдельно подтвердил результаты, rollback, normal-user no-effect, BotDB stability, sanitization, safety/KB boundary и no-mutation.

Следующий шаг не должен быть broad rollout и не должен объявлять production-ready. Но проект уже готов к **ещё одному ограниченному provider-backed smoke**, потому что:

- первый smoke прошёл;
- results gate прошёл;
- blockers/warnings нет;
- BotDB live dependency зелёная;
- normal-user no-effect доказан;
- rollback и hard-stop gates уже существуют;
- Diagnostic Center остаётся под conservative governance.

```text

- добавлять новые production KB sources;
- мутировать all_blocks_merged.json / registry.json / config.yaml;
- удалять runtime/eval modules;
- просить IDE-агента самому проектировать архитектуру.
```

## 6. Обязательные gates

### 6.1. Source gate

Runner должен подтвердить, что источники зелёные:

```text
PRD-046.1.23 final_status=passed
PRD-046.1.23 decision=provider_backed_limited_smoke_execution_passed

PRD-046.1.24 final_status=passed
PRD-046.1.24 decision=continue_limited_candidate
PRD-046.1.24 blockers=false
PRD-046.1.24 warnings=false
```

Если source artifacts отсутствуют или невалидны — hard-stop.

### 6.2. BotDB live preflight

Перед execution runner обязан проверить live BotDB:

```text
GET /api/status
GET /api/registry
GET /api/registry/
GET /api/dashboard
GET /api/dashboard/

```

Acceptance:

```text
dashboard_chroma_count=247

registry_sources=1

query_status=200
semantic_fallback_used=false

```

Если BotDB недоступен или query endpoint не `200` — hard-stop.

### 6.3. Rollback-first precheck

До provider execution подтвердить:

```text
PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=true baseline работает;
test_apply не применяется к normal users;
rollback switch способен полностью отключить apply path;
stale apply после force-disabled отсутствует.
```

### 6.4. Provider budget gate

Allowed:

```text
target_user_provider_calls <= 6
normal_user_provider_calls = 0
raw_provider_payload_committed = false
provider_payload_sanitized = true
```

Если provider calls превысили budget — hard-stop.

### 6.5. Normal-user no-effect gate

Для `normal_user_control_001` и `normal_user_control_002`:

```text
diagnostic_center_apply_count=0
prompt_constraint_apply_count=0
provider_call_count=0
writer_prompt_changed_by_pilot=false
final_answer_changed_by_pilot=false
normal_user_trace_has_apply=false
```

### 6.6. Quality / micro-shift gate

Для target scenarios:

```text
scenario_count=6
provider_backed_scenarios_completed=6
candidate_weaker_than_baseline_count=0
hard_fail_count=0
forbidden_move_violation_count=0
micro_shift_present_count >= 5

```

Важно: качество оценивается не как “умность ответа”, а как соответствие состоянию, линии, safety и микро-сдвигу.

### 6.7. Safety / KB boundary gate

Acceptance:


`КУЗНИЦА ДУХА` может использоваться только как internal lens library.



## 10. Decision logic

### Passed

Можно ставить:

```json
{
  "final_status": "passed",
  "decision": "continue_limited_candidate",
  "broad_rollout_allowed": false,
  "production_ready": false,
  "next_recommended_prd": "PRD-046.1.26 - Diagnostic Center Limited Provider-Backed Smoke Consolidation / Expansion Decision Gate v1"
}
```

Только если все gates зелёные.

### Blocked

Если любой hard-stop сработал:

```json
{
  "final_status": "blocked",
  "decision": "hotfix_required",
  "broad_rollout_allowed": false,
  "production_ready": false
}
```

Implementation report должен явно указать blocker и recommended HF.

### Warning-only

Warnings допустимы только если:

```text
normal_user_no_effect=passed
rollback=passed
safety_kb_boundary=passed
provider_payload_sanitized=true
no_mutation=passed
BotDB query=200
```

Если warning касается safety, KB boundary, provider raw payload, normal-user path, rollback или BotDB query — это blocker, не warning.
