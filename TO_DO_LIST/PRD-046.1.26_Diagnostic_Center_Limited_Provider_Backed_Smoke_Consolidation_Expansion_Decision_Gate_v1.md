# PRD-046.1.26 — Diagnostic Center Limited Provider-Backed Smoke Consolidation / Expansion Decision Gate v1

## 0. Краткий обзор PRD

Этот PRD выполняет крупный, но безопасный шаг после двух provider-backed limited smoke циклов Diagnostic Center.

PRD должен:

- собрать и проверить cumulative evidence по `PRD-046.1.23`, `PRD-046.1.24`, `PRD-046.1.25`;
- подтвердить, что два provider-backed limited smoke цикла дают устойчивую картину качества, rollback, safety/KB boundary, trace sanitization, BotDB stability и normal-user no-effect;
- создать decision gate: можно ли переходить к следующему controlled limited cohort expansion PRD;
- подготовить expansion readiness package для следующего шага;
- зафиксировать future cleanup/stabilization requirement, чтобы после финализации Diagnostic Center проект был очищен от временного шума без удаления нужных regression/eval tools.

PRD не выполняет новый provider-backed smoke, не расширяет cohort, не включает broad rollout, не объявляет production-ready и не меняет normal-user path.

---

## 1. Контекст

После `PRD-046.1.25` проект имеет два последовательных provider-backed limited smoke execution цикла:

1. `PRD-046.1.23`
   - allowlisted operator: `pilot_runtime_operator_001`
   - provider-backed scenarios: `5`
   - normal-user controls: `2`
   - rollback-first / hard-stop / KB-boundary / trace-sanitization / no-mutation gates

2. `PRD-046.1.25`
   - allowlisted operator: `pilot_runtime_operator_002`
   - provider-backed scenarios: `6`
   - provider budget: `6/6`
   - normal-user controls: `2`
   - normal-user apply: `0`
   - normal-user provider calls: `0`
   - rollback / safety-KB / trace / BotDB / no-mutation / artifact hygiene gates passed

`PRD-046.1.24` уже выполнил post-run results/quality/rollback gate по первому provider-backed cycle и принял решение `continue_limited_candidate`.

Теперь нужно не запускать broad rollout, а выполнить consolidation / expansion decision gate:

```text
2 provider-backed smoke cycles
→ cumulative evidence consolidation
→ expansion readiness decision
→ next controlled cohort expansion PRD, если gates зелёные
```

---

## 2. Цель

Цель PRD — подтвердить, что Diagnostic Center provider-backed limited path готов к следующему controlled expansion step, но ещё не готов к broad rollout или production-ready status.

Итоговый gate должен вернуть одно из решений:

```text
blocker_requires_hotfix
repeat_single_operator_limited_smoke
ready_for_controlled_cohort_expansion_prd
ready_for_final_acceptance_candidate_prd
```

Ожидаемое решение при зелёных gates:

```text
ready_for_controlled_cohort_expansion_prd
```

`ready_for_final_acceptance_candidate_prd` разрешается только если evidence показывает очень сильную устойчивость, но PRD всё равно не должен сам включать production-ready и broad rollout.

---

## 3. Non-goals

Этот PRD НЕ должен:

- выполнять новый provider call;
- запускать новый provider-backed smoke;
- расширять cohort фактически;
- включать Diagnostic Center для normal users;
- включать broad rollout;
- объявлять production-ready;
- менять production defaults;
- менять `chunk_type`, `allowed_use`, `safety_flags`;
- мутировать KB governance authority;
- делать Chroma reindex;
- удалять runtime/eval modules;
- удалять historical PRD artifacts;
- коммитить raw provider payload, private logs, secrets, `.env`, sqlite/db/cache, `.venv`, `node_modules`;
- подменять deterministic governance LLM-решениями.

## 8. Future cleanup/stabilization requirement

Добавить в output отдельный артефакт:

```text
TO_DO_LIST/logs/PRD-046.1.26/future_cleanup_stabilization_requirement.json
TO_DO_LIST/reports/PRD-046.1.26_FUTURE_CLEANUP_STABILIZATION_REQUIREMENT.md
```

Смысл: заранее закрепить, что после финализации Diagnostic Center нужен отдельный cleanup/stabilization PRD.

Важно: этот PRD не должен удалять файлы. Он только фиксирует будущую обязанность и первичную классификацию cleanup scope.

Будущий cleanup/stabilization PRD должен разделить проект на зоны:

```text
1. Production runtime
2. Permanent quality/eval/regression tools
3. Historical archive
4. Temporary/debug artifacts eligible for cleanup
```

Предварительный cleanup scope:

```text
- классифицировать Diagnostic Center temporary modules;
- определить permanent regression gates;
- определить historical PRD artifacts, которые остаются архивом;
- определить debug/trace samples, которые можно архивировать или скрыть;
- определить obsolete fixtures, если они дублируют новые;
- убрать навигационный шум из active docs;
- не удалять safety/eval/no-mutation/encoding gates;
- не удалять контракты, ставшие частью архитектуры;
- не ломать воспроизводимость прошлых PRD evidence.
```

Этот пункт обязателен, потому что Diagnostic Center строится через цепочку временных симуляций, shadow/eval/replay/gate layers, и после завершения этапа нужна архитектурная уборка без потери safety-regression покрытия.
