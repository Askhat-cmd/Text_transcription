# PRD-046.1.27 — Diagnostic Center Controlled Cohort Expansion Provider-Backed Execution Gate v1

## 0. Краткий обзор для пользователя и IDE-агента

Этот PRD выполняет **первое controlled cohort expansion execution** после двух успешных provider-backed limited smoke циклов и успешной consolidation/expansion readiness проверки в `PRD-046.1.26`.

PRD объединяет:

```text
source gates
→ BotDB/live preflight
→ rollback-first проверку
→ controlled provider-backed execution
→ normal-user controls
→ quality / safety / trace / KB boundary gates
→ cumulative evidence capture
→ no-mutation proof
→ docs sync
```

Это более крупный шаг, чем отдельный planning-only PRD, потому что предыдущие gates зелёные и `PRD-046.1.26` дал решение:

```text
ready_for_controlled_cohort_expansion_prd
```

Но PRD остаётся безопасным:

```text
broad rollout = запрещён
normal-user activation = запрещён
production-ready decision = запрещён
runtime defaults = не менять
```

## 2. Цель PRD

Выполнить **controlled cohort expansion provider-backed execution gate** для Diagnostic Center.

Цель — проверить, что Diagnostic Center сохраняет качество, безопасность, rollback-поведение, KB-boundary и no-normal-user-effect не только на одном synthetic operator, а на малой allowlisted cohort.

Минимальная целевая cohort:

```text
pilot_runtime_operator_003
pilot_runtime_operator_004
pilot_runtime_operator_005
```

Максимальный размер cohort:

```text
3 allowlisted synthetic operators
```

Минимальный сценарный объём:

```text
12 provider-backed scenarios total
4 scenarios per target operator
```

Normal-user controls:

```text
>=3 normal-user control cases
normal_user_apply_count=0
normal_user_provider_calls=0
```

Provider budget:

```text
max_provider_calls=12
normal_user_provider_call_limit=0
raw_provider_payload_commit_allowed=false
```

---

## 3. Главная архитектурная идея

Этот PRD проверяет не “может ли Diagnostic Center ответить красиво”, а:

```text
может ли governed Diagnostic Center expansion работать стабильно
на малой allowlisted cohort
без расширения normal-user path
и без нарушения rollback/safety/KB/no-mutation инвариантов.
```

Основной критерий качества:

```text
каждый provider-backed scenario должен иметь:
- явный diagnostic trajectory;
- response_goal или response_mode;
- target_micro_shift;
- отсутствие forbidden_moves violations;
- отсутствие raw KB quotes / KB authority citation;
- отсутствие raw provider payload;
- сохранённые rollback и hard-stop gates.
```
