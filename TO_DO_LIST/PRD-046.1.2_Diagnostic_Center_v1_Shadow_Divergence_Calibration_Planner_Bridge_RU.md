## 0. Краткое резюме для IDE-агента

`PRD-046.1.1` подключил Diagnostic Center v1 в runtime как **trace-only shadow слой**. Shadow output уже строится на реальных runtime-сигналах, но не влияет на Writer, WriterContract, prompt, validation path и final answer.

Следующий шаг — не включать Diagnostic Center в ответы, а **калибровать расхождения** между:

```text
Diagnostic Card
Writer Move Compliance / response_mode
Thread Manager phase / relation_to_thread
Diagnostic Center Shadow Output
```

и подготовить **Planner Bridge contract** — промежуточный контракт, который в будущем сможет безопасно передавать Diagnostic Center-сигналы в планировщик/Writer Move Compliance, но в этом PRD остаётся строго **contract-only / eval-only / no user-path effect**.

Главный результат PRD:

```text
broader shadow divergence eval
+ calibrated compatibility rules
+ planner bridge contract
+ planner bridge eval scorecard
+ no-mutation/no-user-path-effect proof
+ docs/ADR sync
```

## 2. Зачем нужен этот PRD

Сейчас shadow-mode уже работает, но пока он только подтверждает, что Diagnostic Center может безопасно строиться рядом с runtime. Для будущей активной интеграции этого недостаточно.

Нужно понять:

```text
где Diagnostic Center согласен с Diagnostic Card;
где он расходится;
какие расхождения допустимы;
какие расхождения являются warning;
какие расхождения являются blocker;
как переводить Diagnostic Center output в будущий Planner/Compliance bridge;
какие поля можно будет использовать в будущем, а какие должны остаться diagnostic-only.
```

Без этого следующий этап может быть опасным:

```text
Diagnostic Center начнёт давать Writer слишком глубокий ход;
response_mode начнёт конфликтовать с фазой thread;
safety-first режим может быть ослаблен;
KB lens может начать звучать как user-facing quote;
Writer Move Compliance может получить несогласованные must_do/must_not_do.
```

Этот PRD создаёт безопасный промежуточный слой перед любой активной интеграцией.


## 6. Planner Bridge contract rules

### 6.1. Safety-first rule

Если:

```text
DiagnosticCenterOutput.status = safety_first
или
StateSnapshot.safety_flag = true
или
ThreadState.safety_active = true
или
nervous_state = hyper|hypo
```

то Planner Bridge обязан вернуть:

```json
{
  "status": "limited",
  "depth_limit": "none|low",
  "max_questions": 0,
  "max_concepts": 1,
  "safety_constraints": ["safety_first"],
  "apply_to_writer": false
}
```

### 6.2. KB boundary rule

Если есть KB lens/practice candidates:

```text
kb_usage_mode must be internal_lens_only or practice_candidate_only
must_not_quote_source must be true
raw_kb_text_exposed must be false
```

Иначе:

```text
status = blocked
blocked_reasons includes kb_boundary_violation
```

### 6.3. User-path no-effect rule

В этом PRD всегда:

```text
apply_to_writer = false
apply_to_writer_contract = false
activation_mode = shadow_only
```

Любой eval case, где это нарушено, должен давать hard blocker.

### 6.4. Depth compatibility rule

```text
hyper/hypo/crisis -> depth_limit <= low
I-W- collapsed -> depth_limit <= low
I-W+ defensive -> depth_limit <= low_to_medium
I+W- defensive -> depth_limit <= low_to_medium
I+W+ open -> depth_limit may be medium/high only if no safety flags
```

### 6.5. Must-do / must-not-do merge rule

Planner Bridge может объединять кандидаты из:

```text
DiagnosticCenterOutput.next_micro_shift.must_do
DiagnosticCenterOutput.next_micro_shift.must_not_do
DiagnosticCard.avoid_list
ThreadState.must_avoid
```

Но результат должен быть deduplicated и bounded:

```text
must_do_candidates <= 6
must_not_do_candidates <= 8
```

### 6.6. Confidence rule

Если confidence низкий:

```text
DiagnosticCenter hypothesis confidence < 0.55
или
divergence severity = needs_review
```

то:

```text
status = limited
depth_limit <= low_to_medium
blocked_reasons may include low_confidence_for_activation
```

## 7. Divergence taxonomy

Создать функцию/логику классификации.

### 7.1. Hard blocker

Примеры:

```text
safety_priority_match = false
depth_compatible = false in safety context
kb_boundary_ok = false
raw_kb_text_exposed = true
writer_prompt_changed_by_shadow = true
diagnostic_center_output_passed_to_writer = true
final_answer_changed_by_shadow = true
PlannerBridgeOutput.apply_to_writer = true
PlannerBridgeOutput.apply_to_writer_contract = true
```

### 7.2. Soft warning

Примеры:

```text
phase_match = false in non-safety context
response_mode_compatible = false but depth safe
writer_move_compatible = false but safety not violated
pattern_core_present = false
relation_to_thread_match = false but thread_action is fork/switch
```

### 7.3. Expected divergence

Примеры:

```text
Diagnostic Card: regulate_first
Diagnostic Center: safety_redirect
```

Это не blocker, если depth is safe.

```text
Thread relation: branch
Pattern core same
```

Это не topic drift, если continuity preserved.

### 7.4. Needs review

Примеры:

```text
multiple soft warnings
low confidence
missing pattern_core + branch relation
knowledge lens present but no aligned response goal
```

### 7.5. Compatible

Если:

```text
no hard blockers
soft warnings <= accepted threshold
safety/KB/user-path invariants green
```

```text
Consequences: система получает готовую форму будущей интеграции, но сохраняет no-user-path-effect gate и не меняет Writer/Compliance поведение преждевременно.
```
