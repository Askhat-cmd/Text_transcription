# PRD-046.1.30 — Diagnostic Center Controlled Rollout Planning v1

## 0. Краткое назначение PRD

Этот PRD подготавливает **controlled rollout plan** для Diagnostic Center после завершения provider-backed limited smoke, controlled cohort expansion, final runtime governance acceptance и cleanup/stabilization.

PRD-046.1.30 должен создать безопасный, проверяемый и rollback-first план следующего шага: **ограниченное controlled rollout execution в отдельном будущем PRD**, без фактического включения broad rollout и без активации Diagnostic Center для normal users в этом PRD.

---

## 2. Цель PRD

Подготовить controlled rollout planning package, который отвечает на вопросы:

1. **Кому можно включать Diagnostic Center в следующем execution PRD?**
2. **Какие флаги, allowlist и rollback условия должны быть включены?**
3. **Какие preflight gates обязательны до execution?**
4. **Какие hard-stop условия немедленно отменяют rollout?**
5. **Какие permanent gates из `docs/DIAGNOSTIC_CENTER_EVAL_HARNESS.md` должны блокировать execution?**
6. **Какие evidence artifacts должны быть собраны в будущем execution PRD?**
7. **Как доказать, что normal-user path остаётся без эффекта?**

---

## 3. Non-goals

В этом PRD строго запрещено:

```text
- не выполнять provider-backed execution;
- не вызывать provider API;
- не включать Diagnostic Center для normal users;
- не делать broad rollout;
- не объявлять production-ready;
- не менять production defaults;
- не менять runtime authority boundaries;
- не мутировать KB governance authority поля;
- не делать Chroma reindex;
- не удалять cleanup candidates;
- не удалять eval/replay/no-mutation/safety gates;
- не коммитить raw provider payload/private logs/secrets/.env/sqlite/cache/.venv/node_modules.
```

Этот PRD — **planning/readiness only**.

---

## 4. Rollout boundary для планирования

План должен быть консервативным.

Допустимый следующий execution PRD может планировать только:

```text
rollout_type: controlled_limited_rollout_execution
max_target_users: 3
target_user_type: allowlisted_internal_or_synthetic_operators_only
normal_user_activation_allowed: false
broad_rollout_allowed: false
production_ready_allowed: false
default_flags_must_remain_conservative: true
rollback_first_required: true
hard_stop_required: true
```

Если IDE-агент считает, что нужны реальные пользователи вне internal/synthetic allowlist, он обязан остановиться и записать blocker:

```text
blocker = human_product_governance_required_before_real_user_rollout
```

## 8. Controlled rollout plan requirements

### 8.1. Cohort policy

Cohort policy должен содержать:

```json
{
  "max_target_users": 3,
  "target_user_type": "allowlisted_internal_or_synthetic_operators_only",
  "normal_user_activation_allowed": false,
  "broad_rollout_allowed": false,
  "production_ready_allowed": false,
  "requires_explicit_allowlist": true,
  "requires_test_prefix_or_operator_marker": true
}
```

### 8.2. Toggle matrix

Toggle matrix должен включать минимум:

```text
force_disabled=true -> total disabled
enabled=false -> total disabled
mode=shadow -> no apply
mode=controlled_limited_apply -> apply only allowlisted target users
normal_user -> never apply
hard_stop_triggered=true -> forced rollback
provider_budget_exceeded=true -> forced rollback
safety_kb_boundary_violation=true -> forced rollback
trace_sanitization_failure=true -> forced rollback
botdb_unstable=true -> forced rollback
```

### 8.3. Preflight gates

Минимум:

```text
BotDB status ok
Registry focus source only
Chroma count ok
Query endpoint 200
semantic_fallback_used=false
botdb_circuit_open=false
runtime defaults conservative
allowlist explicitly set
rollback switch tested before execution
artifact output path clean
provider budget configured
normal-user controls configured
```

### 8.4. Hard-stop criteria

Минимум:

```text
any normal-user apply > 0
any normal-user provider call > 0
rollback failure > 0
raw provider payload persisted
raw KB text exposed as authority
KB quote/citation boundary violation
provider budget exceeded
BotDB circuit open
semantic fallback used unexpectedly
runtime defaults changed
production data mutation detected
artifact encoding/hygiene blocker
```

### 8.5. Evidence capture plan

Должен описывать, что future execution PRD обязан собрать:

```text
execution_manifest
provider_budget_report
sanitized_provider_trace
normal_user_no_effect_report
quality_micro_shift_report
safety_kb_boundary_report
trace_sanitization_report
rollback_precheck_postcheck_report
botdb_preflight_postcheck_report
no_mutation_proof
artifact_hygiene_report
operator_summary
next_prd_recommendation
```
