# PRD-046.1.31 — Diagnostic Center Controlled Rollout Execution Gate v1

## 0. Назначение PRD

Этот PRD выполняет первый **controlled rollout execution** после успешного planning-пакета `PRD-046.1.30`.

Цель — не расширить Diagnostic Center на обычных пользователей, а провести строго ограниченное, allowlisted, rollback-first исполнение по уже подготовленному rollout plan и собрать доказательства:

- controlled rollout выполняется без нарушения governance boundaries;
- normal-user path остаётся без эффекта;
- provider budget, rollback, hard-stop, safety/KB, trace sanitization, BotDB stability и no-mutation gates остаются зелёными;
- следующий шаг может быть только отдельным results / rollback / quality gate PRD.

## 1. Краткий контекст

Предыдущая цепочка уже прошла:

```text
PRD-046.1.23 — first provider-backed limited smoke execution
PRD-046.1.24 — post-run results gate
PRD-046.1.25 — second provider-backed limited smoke execution
PRD-046.1.26 — consolidation / expansion decision gate
PRD-046.1.27 — controlled cohort expansion execution
PRD-046.1.28 — final runtime governance acceptance / stabilization readiness
PRD-046.1.29 — stabilization / cleanup / eval harness consolidation
PRD-046.1.30 — controlled rollout planning
```

`PRD-046.1.30` должен быть источником rollout policy, cohort policy, toggle matrix, preflight gates, hard-stop criteria, rollback plan, normal-user no-effect plan и evidence capture plan.

## 2. Главный принцип

```text
Controlled rollout execution ≠ broad rollout.
Controlled rollout execution ≠ production-ready.
Controlled rollout execution ≠ normal-user activation.
```

Этот PRD разрешает только ограниченное execution-окно для allowlisted internal/synthetic operators.

## 6. Controlled rollout boundaries

### 6.1. Cohort

Использовать только allowlisted internal/synthetic operators из planning-пакета PRD-046.1.30.

Default maximum:

```text
max_target_operator_count = 3
```

Если PRD-046.1.30 содержит более строгий лимит, использовать более строгий лимит.

Запрещено:

```text
normal_user_activation_allowed=false
broad_rollout_allowed=false
production_ready=false
```

### 6.2. Provider budget

Default maximum:

```text
provider_call_budget_max = 12
```

Если PRD-046.1.30 содержит более строгий бюджет, использовать более строгий бюджет.

Hard stop:

```text
provider_calls_total > provider_call_budget_max
```

### 6.3. Normal-user controls

Минимум:

```text
normal_user_control_count >= 3
```

Обязательное условие:

```text
normal_user_apply_count == 0
normal_user_provider_calls == 0
normal_user_prompt_delta_count == 0
normal_user_final_answer_changed_by_rollout_count == 0
```

### 6.4. Scenario coverage

Execution должен включать минимум 12 controlled rollout scenarios, если PRD-046.1.30 не задаёт другой строгий минимум.

Покрыть группы:

```text
- supportive / low-risk clarity request
- I-W+ / defensive / directive-seeking
- I+W- / defensive / blame-seeking
- I-W- / low resource / collapsed or near-collapsed
- mixed openness / thread continuity
- KB lens boundary case
- retrieval/no-retrieval routing case
- rollback simulation case
- normal-user control case
```

## 7. Runtime / flag discipline

Runner должен работать в controlled execution harness.

Если используются runtime flags, они должны быть применены только внутри execution harness/process environment и не должны менять committed defaults.

Baseline must remain:

```text
default broad activation disabled
normal-user activation disabled
rollback switch available
force-disable path available
```

Любое изменение committed defaults — blocker.

```text

forbidden_moves_violation_count == 0

thread_continuity_hard_fail_count == 0
```
