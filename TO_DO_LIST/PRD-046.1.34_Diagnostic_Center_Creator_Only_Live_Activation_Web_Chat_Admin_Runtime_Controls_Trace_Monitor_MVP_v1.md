# PRD-046.1.34 — Diagnostic Center Creator-Only Allowlisted Live Activation / Web Chat + Admin Runtime Controls / Trace Monitor MVP v1

## 0. Назначение PRD

Этот PRD продолжает проект **Bot Psychologist / Neo MindBot** после `PRD-046.1.33`.

`PRD-046.1.33` завершён как readiness-only boundary gate:

```text
final_status=passed
decision=ready_for_allowlisted_limited_live_activation_prd
required pytest set=15/15 passed
strict runner=passed
blockers=none
warnings=none
no_mutation_proof_passed=true
runtime_defaults_changed=false
kb_registry_chroma_config_mutated=false
docs_consistency_gate_passed=true
```

Следующий шаг по `docs/PROJECT_STATE.md`, `docs/ROADMAP.md` и `TO_DO_LIST/reports/PRD-046.1.33_NEXT_PRD_RECOMMENDATION.md` — `PRD-046.1.34`.

Но важное уточнение архитектора:

```text
В проекте пока нет реальных внешних пользователей.
Единственный живой пользователь сейчас — создатель проекта.
```

Поэтому этот PRD НЕ должен трактовать `allowlisted limited live activation` как тест на 1–3 внешних людях.

Правильная специализация текущего шага:

```text
creator-only live activation
```

То есть Diagnostic Center можно включить в реальном runtime только для создателя проекта как единственного allowlisted live user, а для всех прочих возможных user_id путь должен остаться прежним и безопасным.

---

## 1. Главная цель

Дать создателю проекта возможность тестировать бота “по-живому” через Web Chat / Web UI, при этом:

```text
Diagnostic Center реально влияет на creator-only path,
но НЕ включается для всех пользователей,
НЕ объявляется production-ready,
НЕ становится broad rollout,
НЕ мутирует KB,
НЕ меняет runtime defaults для normal users.
```

Итоговое решение PRD должно быть одним из:

```text
creator_live_activation_passed
creator_live_activation_blocked_fix_required
creator_live_activation_stopped_by_hard_stop
```

---

## 2. Почему этот шаг сейчас безопасен

Основание:

1. `PRD-046.1.33` подтвердил готовность к allowlisted limited live activation.
2. Broad rollout по-прежнему запрещён.
3. Normal-user activation по-прежнему запрещён.
4. Production-ready по-прежнему false.
5. Реальных внешних пользователей нет.
6. Единственный live tester — creator/owner.
7. Runtime уже имеет Web UI / Admin Config Panel и вкладку `Runtime`, где логично управлять runtime-флагами.
8. Скриншот текущей админки показывает, что вкладка `Runtime` уже содержит runtime-параметры, system status и controls. Поэтому Diagnostic Center controls нужно развивать там, а не создавать отдельную несвязанную панель.

## 4. Термины

### 4.1. Broad rollout

```text
Функция включена для всех пользователей по умолчанию.
Любой пользователь пишет боту → Diagnostic Center активен → влияет на ответ.
```

В этом PRD broad rollout запрещён.

### 4.2. Controlled live testing

```text
Бот работает в реальном runtime.
Diagnostic Center может влиять на ответ.
Но только для явно разрешённого user_id/chat_id/role.
Есть rollback, hard-stop, trace, monitoring и post-run results gate.
```

### 4.3. Creator-only live activation

```text
Частный случай controlled live testing.
Единственный разрешённый live user — creator/owner проекта.
Все прочие user_id считаются normal/unknown users и не получают active Diagnostic Center influence.
```

Ввести управляемый runtime mode:

```text
diagnostic_center_runtime_mode:
  disabled
  shadow_only
  creator_only
  allowlist_live
  all_users_locked
```

```text

```

`allowlist_live` можно реализовать как режим/структуру данных для будущего, но фактически в этом PRD allowlist должен содержать только creator.

`all_users_locked` может отображаться в админке, но не должен быть включаемым.

## 10. Diagnostic Center active influence boundary

Для creator-only path разрешается активное влияние Diagnostic Center на ответ в пределах уже существующей governed architecture.

Разрешено:

```text
- использовать Diagnostic Center output как internal map;
- использовать Planner/Prompt-Constraint governed path, если он уже прошёл gates;
- передавать Writer controlled constraints / next_micro_shift / forbidden_moves;
- сохранять sanitized trace;
- показывать sanitized monitor.
```

Запрещено:

```text
- Diagnostic Center не пишет финальный ответ пользователю;
- Diagnostic Center не цитирует KB напрямую;
- Diagnostic Center не меняет chunk_type/allowed_use/safety_flags;
- Diagnostic Center не обходит Writer;
- Diagnostic Center не становится user-facing agent;
- Diagnostic Center не включает all_users;
- Diagnostic Center не влияет на unknown/normal users.
```

## 12. Sanitized Diagnostic Trace Storage

### 12.1. Цель

Нужно сохранить историю работы Diagnostic Center так, чтобы её можно было мониторить, но без утечки raw private content.

### 12.2. Что сохранять

Минимальный sanitized trace schema:

```json
{
  "schema_version": "diagnostic_center_sanitized_trace_v1",
  "turn_id": "string",
  "conversation_id": "string",
  "user_id_hash": "string",
  "is_creator": true,
  "timestamp": "iso8601",
  "runtime_mode": "creator_only",
  "diagnostic_center_active": true,
  "nervous_state": "window|hyper|hypo|unknown",
  "intent": "string",
  "openness": "open|mixed|defensive|collapsed|unknown",
  "ok_position": "I+W+|I-W+|I+W-|I-W-|unknown",
  "thread_action": "continue|branch|new_thread|return|unknown",
  "selected_lens_family": [],
  "selected_lens_ids": [],
  "working_hypothesis_summary": "sanitized short summary",
  "target_micro_shift": "sanitized short summary",
  "forbidden_moves": [],
  "retrieval_used": false,
  "kb_boundary_status": "passed|blocked|not_used",
  "safety_flags": [],
  "hard_stop_triggered": false,
  "rollback_status": "not_triggered|triggered",
  "validator_result": "passed|weak|blocked",
  "final_answer_hash": "sha256",
  "trace_sanitized": true
}
```

### 12.3. Что НЕ сохранять

```text
- raw provider payload;
- chain-of-thought;
- secrets;
- .env;
- полный raw user message внутри Diagnostic Center trace;
- raw content_full из КУЗНИЦЫ;
- приватные runtime logs;
- полный финальный ответ, если он уже хранится в обычной message history;
- любые ключи API.
```

Допускается хранить короткий sanitized preview только если он проходит sanitizer и не содержит sensitive/private/raw KB content.

## 16. Hard-stop criteria

PRD должен реализовать hard-stop для creator live smoke.

Hard-stop должен срабатывать, если:

```text
- Diagnostic Center влияет на non-creator user;
- all_users mode включился;
- force_disabled не отключает Diagnostic Center;
- raw provider payload попал в artifact;
- raw content_full из KB попал в user-facing/monitor trace;
- Writer начал цитировать КУЗНИЦУ как authority;
- Diagnostic Center начал писать final answer;
- safety/KB boundary failed;
- provider budget exceeded;
- web chat live smoke не смог доказать creator-only boundary;
- trace sanitizer failed;
- rollback failed;
- BotDB dependency failed;
- normal-user control failed.
```

Если hard-stop сработал:

```text
diagnostic_center_force_disabled=true
final_status=creator_live_activation_stopped_by_hard_stop
next_prd_recommendation=hotfix_required
```
