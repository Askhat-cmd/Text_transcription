# PRD-046.1.35-HF1 — Creator Live Evidence + BotDB/RAG-to-Writer Repair v1

## 0. Краткое назначение

`PRD-046.1.35-HF1` — это hotfix/evidence PRD после `PRD-046.1.35`.

`PRD-046.1.35` честно зафиксировал:

```text
final_status=evidence_incomplete
decision=evidence_incomplete_hotfix_required
actual_live_turn_evidence_count=0
warnings=[actual_live_turn_evidence_missing_or_weak]
next_prd_recommendation=PRD-046.1.35-HF1
```

Главная задача текущего PRD — не расширять rollout, а получить **реальное доказательство живого creator-only turn** и одновременно починить/доказать цепочку:

```text
Bot_data_base /api/query
  → bot_psychologist retriever
  → MemoryRetrievalAgent
  → knowledge_policy
  → Context Assembly
  → WriterContract
  → Writer prompt
  → final answer
  → Web Trace / debug UI
```

Цель — чтобы создатель проекта мог уже в пилотном режиме тестировать бота “по-живому” через Web Chat и видеть в trace, почему ответ был именно таким.

## 1. Контекст проблемы

### 1.1. Что показал PRD-046.1.35

Предыдущий gate подтвердил, что safety/rollback/no-mutation/docs gates в целом зелёные, но не доказал реальный live-turn.

Проблема не в том, что проект “провалился”. Наоборот, gate сработал правильно: он не принял simulated/runtime-probe evidence за полноценное доказательство живой работы.

Теперь нужен HF1, который должен создать proof:

```text
реальный creator пишет сообщение
→ backend реально обрабатывает turn
→ Diagnostic Center работает в creator_only boundary
→ BotDB/RAG путь либо доставляет знания до Writer, либо честно объясняет в trace, где они отфильтрованы
→ final answer возвращается в Web Chat
→ sanitized trace сохраняется и виден
```

### 1.2. Что видно по текущим ручным скриншотам

В Web Trace по вопросу `что такое нейросталкинг` видно:

```text
Memory Agent:
  context_turns: 6
  semantic_hits: 0
  has_relevant_knowledge: false
  rag_query: что такое неосталкинг? что такое нейросталкинг

Чанки в Writer:
  Нет релевантных чанков

Writer user prompt:
  ЗНАНИЙ ИЗ БАЗЫ:
  нет релевантных знаний

Session Dashboard:
  BLOCKS: 0
```

При этом в терминальных логах Bot_data_base ранее наблюдалось:

```text
[QUERY] Chroma query failed: object of type 'int' has no len()
[QUERY] Chroma fallback failed: object of type 'int' has no len()
[QUERY] fallback all_blocks_merged path activated after fallback exception
POST /api/query/ 200 OK
```

А в логах `bot_psychologist` ранее встречался признак, что API retrieval может возвращать блоки:

```text
[RETRIEVAL] API search: 4 блоков
[MRA] rag hits=4 ... retrieval_source_used=api circuit_open=False
```

Значит, проблема может быть в одном или нескольких местах:

1. BotDB Chroma query падает и отдаёт fallback вместо нормального semantic search.
2. BotDB API возвращает chunks, но они имеют низкие scores и отфильтровываются по `RAG_MIN_SCORE`.
3. `MemoryRetrievalAgent` получает hits, но `knowledge_policy` не допускает их в Writer.
4. `Context Assembly` получает `knowledge_rag_hits`, но выкидывает их при budget/dedup.
5. `WriterContract` не прокидывает `knowledge_rag_hits` в prompt.
6. Web Trace показывает не тот слой или не показывает policy/drop reasons.
7. Runtime Diagnostic Center creator-only evidence остаётся simulated/probe, а не actual live-turn proof.

Этот PRD должен найти и устранить минимальную причину, не ломая архитектуру.

## 7. Диагностическая матрица причин

HF1 должен классифицировать причину отсутствия чанков в Writer.

### 7.1. botdb_empty

```text
BotDB /api/query вернул 200, но chunks=[].
```

Действие:

- проверить registry/dashboard/source count;
- проверить query text;
- проверить source `123__кузница_духа`;
- зафиксировать, что проблема на стороне BotDB retrieval.

### 7.2. chroma_query_failed_with_fallback

```text
Chroma query падает, но fallback возвращает candidates.
```

Действие:

- воспроизвести `object of type 'int' has no len()`;
- найти безопасную минимальную причину;
- исправить query/fallback path без reindex;
- если невозможно без reindex — поставить blocker для отдельного Chroma PRD.

### 7.3. filtered_by_score

```text
BotDB/retriever raw hits > 0, но после RAG_MIN_SCORE semantic_hits=0.
```

Действие:

- не занижать глобальный порог вслепую;
- добавить trace: raw scores, threshold, filtered count;
- если нужно, предложить отдельный Retrieval Governance PRD;
- для HF1 допускается локальный safe fix только если очевидно, что score mapping bug.

### 7.4. governance_blocked

```text
knowledge_policy получил hits, но included_writer_count=0.
```

Действие:

- не обходить governance;
- показать policy_action/reasons/risk_flags;
- если hits предназначены только как diagnostic_lens, доказать diagnostic path, а не пихать raw content в Writer;
- если для пользовательского вопроса нужен writer_context, но governance source запрещает writer use, поставить `hf1_passed_creator_live_ready_rag_governance_blocked` или рекомендовать отдельный KB governance/readiness PRD.

### 7.5. context_dropped

```text
knowledge_policy included_writer_count > 0, но context_assembly_knowledge_hits_count=0.
```

Действие:

- проверить dedup/budget;
- добавить drop reasons;
- исправить очевидный баг context assembly.

### 7.6. writer_prompt_missing

```text
context_assembly_knowledge_hits_count > 0, но Writer prompt показывает “ЗНАНИЙ ИЗ БАЗЫ: нет”.
```

Действие:

- исправить WriterContract/to_prompt_context/formatting;
- добавить тест, что `KNOWLEDGE RAG HITS` реально попадают в prompt.

### 7.7. ui_trace_mismatch

```text
Backend имеет knowledge hits, но Web Trace показывает 0.
```

Действие:

- исправить debug payload/API/UI mapping;
- не менять runtime response behavior;
- добавить trace alignment test.

### 16.2. hf1_passed_creator_live_ready_rag_governance_blocked


```text

registry/dashboard не подтверждают готовность
```

## 22. Важная архитектурная позиция

Главная цель HF1 — не “заставить Writer любой ценой использовать КУЗНИЦУ”.

Главная цель:

```text
сделать цепочку знаний наблюдаемой, честной и рабочей.
```

Если знания есть, разрешены governance и должны быть полезны Writer’у — они должны доходить.

Если знания есть, но governance говорит “internal_only / diagnostic_lens only” — это должно быть видно в trace, и это не надо обходить.

Если BotDB падает в fallback — это должно быть видно.

Если fallback возвращает нерелевантные chunks — это должно быть видно.

Если UI показывает 0, а backend имеет hits — это баг trace alignment, а не обязательно баг retrieval.
