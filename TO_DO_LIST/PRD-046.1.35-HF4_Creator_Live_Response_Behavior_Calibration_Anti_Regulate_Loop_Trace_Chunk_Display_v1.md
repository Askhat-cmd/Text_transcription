# PRD-046.1.35-HF4 — Creator Live Response Behavior Calibration / Anti-Regulate Loop / Trace Chunk Display v1

## 0. Назначение PRD

`PRD-046.1.35-HF4` продолжает `PRD-046.1.35-HF3`.

HF3 доказал техническую цепочку:

```text
BotDB → Memory Retrieval → Knowledge Policy → Context Assembly → Writer Prompt → Web Trace
```

Но живой пользовательский тест показал новый blocker качества:

```text
RAG работает, чанки доходят до Writer,
но бот отвечает слишком коротко, отстранённо,
часто навязывает дыхательные/телесные практики,
и Web Trace визуально не показывает содержимое чанков в блоке "Чанки в Writer".
```

Главная цель HF4 — не расширять Diagnostic Center и не делать KB Context v2, а привести creator-live поведение к минимально приемлемому уровню для ручного пилотного тестирования:

```text
1. Не предлагать дыхание/телесную практику по умолчанию.
2. Уважать прямой запрос пользователя "не практику, а пример".
3. Правильно маршрутизировать запросы на объяснение/пример.
4. Сохранять живость, конкретность и человеческую связность ответа.
5. Починить Web Trace display: если Writer получает чанки, UI должен показывать безопасный preview, а не пустой "-".
```


## 6. Проблема 1 — anti-regulate loop

### 6.1. Текущее неверное поведение

Система слишком легко выбирает:

```text
response_mode=regulate
suggested_writer_move=regulate_first
must_do=offer_one_simple_body_action
```

Даже когда пользователь не в кризисе и просит пример.

### 6.2. Требуемое поведение

Добавить deterministic guard:

```text
anti_regulate_loop_v1
```

Он должен учитывать:

```text
current user message
recent turns
thread_state.response_mode
diagnostic_card.suggested_writer_move
diagnostic_card.risk_flags
writer_move_instructions
safety_active
```

Guard должен подавлять `regulate_first`, если выполняется одно из условий:

```text
1. user_message содержит явный запрос примера:
   "пример", "приведи пример", "из жизни", "с родителем", "с партнёром", "покажи на примере"

2. user_message или recent turns содержат rejection практики:
   "не практику", "не упражнение", "не технику", "просто объясни",
   "не дыхание", "хватит дыхания", "без практик"

3. В последних N=3 ответах бота уже была дыхательная/телесная практика.

4. response_mode=regulate, но:
   safety_active=false
   и user intent/request_type=example_request|explain_request
```

Исключение:

```text
regulate_first можно оставить,
если safety_active=true,
или явные crisis/emergency markers,
или пользователь прямо просит успокоиться/дыхание/стабилизацию.
```

## 7. Проблема 2 — example request routing

### 7.1. Новый request type

Добавить или вывести в trace поле:

```text
request_type:
  example_request
  explain_request
  practice_request
  support_request
  safety_regulation_request
  unknown
```

Это может быть deterministic helper, без нового LLM-агента.

### 7.2. Routing policy

Если:

```text
request_type=example_request
safety_active=false
```

то запрещено:

```text
response_mode=regulate
suggested_writer_move=regulate_first
writer_move_must_do contains offer_one_simple_body_action
```

Предпочтительное поведение:

```text
response_mode=reflect|explore|practice? no, лучше example
suggested_writer_move=give_concrete_example
writer style=concrete_contextual_example
max_sentences=5-8
max_questions=0 или 1
```

### 7.3. Новый writer move

Добавить writer move:

```text
give_concrete_example
```

Минимальный контракт:

```json
{
  "move": "give_concrete_example",
  "max_sentences": 8,
  "max_questions": 1,
  "style": "concrete_contextual_example",
  "must_do": [
    "give_one_concrete_life_example",
    "use_user_requested_context",
    "avoid_practice_instruction",
    "avoid_breathing_instruction",
    "connect_example_to_current_thread"
  ],
  "must_not_do": [
    "do_not_offer_body_action",
    "do_not_turn_into_lecture",
    "do_not_open_multiple_threads",
    "do_not_ignore_user_rejected_practice"
  ]
}
```

Если новый move слишком широк для текущей архитектуры, допустимо реализовать как:

```text
clarify_one_point + extra guard example_request=true
```

Но в trace должно быть видно:

```text
request_type=example_request
practice_suppressed=true
reason=user_requested_example
```

## 8. Проблема 3 — короткость и отстранённость

HF4 должен не делать ответы длинными всегда, а добавить минимум живости и контекстности.

Для live creator-pilot допустимый ответ на пример должен быть:

```text
4–8 предложений
без списка
без "сделайте вдох"
с конкретной сценой
с человеческой связкой с предыдущими ходами
без сухой инструкции
без чрезмерной лекции
```

Добавить quality flags:

```text
too_short_for_example_request
detached_tone_risk
ignored_user_requested_example
practice_forced_after_rejection
```

Эти flags могут быть deterministic в live-smoke artifacts. Блокировать ответ в runtime пока не нужно, но strict runner HF4 должен считать их blocker для acceptance.

## 9. Проблема 4 — Web Trace "Чанки в Writer" показывает пустые карточки

### 9.1. Наблюдение

На скрине видно:

```text
Контекст памяти | turns: 5 | hits: 3
Чанки в Writer (3)
book
-
score: 0.052
book
-
score: 0.040
book
-
score: 0.038
```

То есть count есть, но content preview в UI пустой или заменён на "-".

HF3 проверил count alignment:

```text
backend_context_knowledge_hits_count=3
backend_writer_prompt_knowledge_hits_count=5
web_trace_chunks_in_writer_count=5
mismatch_detected=false
```

Но это count-only gate. Оно не проверяет, что UI показывает содержимое.

### 9.2. Требуемый fix

Добавить trace display quality gate:

```text
writer_chunks_display_gate_v1
```

Он должен проверять:

```text
writer_chunks_count >= 1
non_empty_preview_count >= 1
empty_preview_count == 0 для displayed writer chunks
each chunk has:
  source/title fallback
  score
  content_preview или sanitized_content_preview
```

Если backend уже отдаёт content, но UI показывает "-", чинить frontend mapping.

Если backend не отдаёт safe preview для UI, добавить безопасное поле в debug trace:

```json
{
  "writer_chunks_detail": [
    {
      "chunk_id_hash": "sha256:...",
      "source": "book",
      "score": 0.052,
      "content_preview": "safe sanitized preview ...",
      "preview_len": 160,
      "governance": {
        "policy_action": "...",
        "allowed_for_writer": true
      }
    }
  ]
}
```

Запрещено:

```text
- отдавать raw content_full в UI;
- обходить knowledge_policy sanitization;
- показывать unsafe/internal_only raw content.
```
