# PRD-046.1.37-HF1 — Actual Live Runtime Timeout / Evidence Repair / Docs State Correction v1

## 0. Назначение PRD

`PRD-046.1.37-HF1` — hotfix после `PRD-046.1.37`.

Цель HF1 — не добавлять новые функции Diagnostic Center и не переходить в настройку мультиагентной системы, а закрыть конкретный финальный blocker:

```text
PRD-046.1.37 final_status=blocked
decision=blocked_runtime_readiness

Причина:
backend health/admin/BotDB/Web UI доступны,
но actual-live adaptive/debug path падает по TimeoutError,
из-за чего final completion gate не может доказать живой backend path.
```

HF1 должен:

```text
1. Найти реальную причину TimeoutError на /api/v1/questions/adaptive и debug trace.
2. Сделать runner устойчивым к нормальной live-latency.
3. Дать IDE-агенту право самостоятельно пробовать несколько стратегий восстановления.
4. Получить actual-live evidence: HTTP 200 + answer_received + trace_received.
5. Повторить rollback/hard-stop и normal-user no-effect уже на реальном live path.
6. Исправить living docs: не писать completed, пока scorecard blocked.
7. Если evidence проходит — закрыть Diagnostic Center v1 как creator-live / developer-pilot layer.
```

## 3. Главная гипотеза HF1

Основная гипотеза:

```text
TimeoutError вызван слишком коротким timeout в final runner.
```

В `PRD-046.1.37` runner использовал около `8 sec` для actual-live adaptive/debug calls.

Для реального LLM/Web path это может быть недостаточно. По живым ручным тестам ответы могли занимать больше 8 секунд.

HF1 не должен просто “увеличить timeout и забыть”. Он должен доказать:

```text
- какой timeout реально нужен;
- на каком этапе возникает задержка;
- можно ли получить answer и trace при 30/45/60/90 sec;
- не зависает ли backend бесконечно;
- не ломается ли rollback/normal-user boundary;
- нет ли скрытой ошибки endpoint/payload/auth.
```

## 9. Adaptive timeout retry strategy

HF1 должен явно поддерживать self-repair behavior при TimeoutError.

### 9.1. Timeout ladder

Для каждого actual-live adaptive case runner обязан пробовать:

```text
attempt_1: timeout_sec=15
attempt_2: timeout_sec=30
attempt_3: timeout_sec=45
attempt_4: timeout_sec=60
attempt_5: timeout_sec=90
```

Можно остановиться раньше, если case прошёл:

```text
http_status=200
answer_received=true
trace_received=true
```

### 9.2. Backoff

Между попытками:

```text
sleep 2 sec after 15 sec timeout
sleep 4 sec after 30 sec timeout
sleep 6 sec after 45 sec timeout
sleep 8 sec after 60 sec timeout
```

### 9.3. Warmup

Перед основными кейсами выполнить warmup:

```text
1. health/admin checks
2. BotDB query
3. one short adaptive warmup:
   "короткий runtime warmup"
```

Warmup не должен считаться actual-live proof, но должен помочь прогреть runtime/provider.

### 9.4. Trace polling after adaptive timeout

Если adaptive call timed out, runner НЕ должен сразу считать trace missing.

Он должен:

```text
1. сохранить session_id;
2. poll debug trace endpoint 5 раз:
   after 3 sec
   after 6 sec
   after 10 sec
   after 15 sec
   after 20 sec
3. если trace появился, записать:
   adaptive_timeout_but_trace_arrived=true
   trace_latency_after_timeout_sec=...
```

Если trace появился, но answer не получен — case всё равно failed, но это важно для диагностики.

### 9.5. Payload variants

Если все timeout attempts падают, runner должен проверить, не ошибся ли формат payload.

Пробовать варианты:

```json
{
  "query": "...",
  "user_id": "...",
  "session_id": "...",
  "debug": true
}
```

```json
{
  "question": "...",
  "user_id": "...",
  "session_id": "...",
  "debug": true
}
```

```json
{
  "message": "...",
  "user_id": "...",
  "session_id": "...",
  "debug": true
}
```

Если endpoint возвращает validation error, runner должен записать это как `payload_schema_mismatch`, а не timeout.

### 9.6. Auth variants

Если есть 401/403, runner должен проверить:

```text
- X-API-Key
- Authorization: Bearer
- dev key from config/test env
```

Запрещено коммитить actual secret values.

### 9.7. Timeout classification

Каждый failed case должен получить classification:

```text
adaptive_timeout_before_first_byte
adaptive_completed_but_trace_timeout
trace_poll_timeout_after_answer
payload_schema_mismatch
auth_failure
backend_error_5xx
provider_latency
provider_error
session_store_blocked
debug_trace_route_missing
unknown_timeout
```
