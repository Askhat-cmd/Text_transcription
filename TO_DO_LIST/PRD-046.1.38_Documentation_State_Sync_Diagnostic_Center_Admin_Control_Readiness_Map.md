# PRD-046.1.38 — Documentation State Sync + Diagnostic Center Admin Control Readiness Map

## 0. Краткое резюме

Цель `PRD-046.1.38` — привести сопроводительную документацию и Web Admin control surface в соответствие с фактическим состоянием проекта после закрытия текущей фазы Diagnostic Center v1.

После `PRD-046.1.37-HF1` Diagnostic Center v1 считается закрытым для текущей creator-only / developer-pilot фазы. Следующий большой фокус проекта — `Multiagent Quality & Tuning Track`.

Но перед переходом к дальнейшей настройке качества ответов нужно закрыть промежуточный слой:

1. Синхронизировать документацию:
   - верхний слой `docs/`;
   - `bot_psychologist/docs/`;
   - `Bot_data_base/docs/`;
   - `TO_DO_LIST` как исторический архив;
   - operational state / roadmap / decisions / PRD index.

2. Реализовать в Web Admin понятный `Diagnostic Center Control` surface:
   - текущий статус Diagnostic Center;
   - матрицу режимов;
   - переключатели runtime-режимов;
   - kill switch / rollback;
   - developer / creator / allowlist / all-users controls;
   - отображение governance-boundary;
   - audit-friendly trace состояния.

Важное уточнение от владельца проекта: сейчас единственный пользователь проекта — сам разработчик. Поэтому этот PRD разрешает реализовать **полный набор переключателей в Web Admin**, включая будущий all-users toggle. Но all-users toggle должен быть реализован как **developer-local governed mode**, а не как production-ready broad rollout.

## 1. Контекст

### 1.1. Текущее состояние проекта

Фактическое состояние после `PRD-046.1.37-HF1`:

```text
Diagnostic Center v1:
  status: completed_for_current_creator_only_phase

Runtime:
  active_runtime: multiagent_adapter
  pipeline_version: multiagent_v1
  legacy_fallback: disabled

Current allowed scope:
  developer / creator-only / allowlisted testing

Blocked as production decision:
  broad rollout
  production_ready declaration
  uncontrolled normal-user activation
  KB governance mutation
  Chroma reindex
```

### 1.2. Почему этот PRD нужен сейчас

Diagnostic Center прошёл длинный путь разработки, доработок, hotfix и evidence gates. Исторический путь отражён в `TO_DO_LIST`, но компактная operational documentation должна ясно показывать:

- что сейчас действительно активно;
- что закрыто;
- что разрешено;
- что заблокировано;
- где находится Web Admin control surface;
- какие режимы есть у Diagnostic Center;
- какие переключатели меняют runtime;
- какие переключатели только отображают future-governance intent;
- какой следующий трек после Diagnostic Center.

Кроме того, Web Admin уже имеет базовую multiagent control surface, но Diagnostic Center как управляемая подсистема пока недостаточно явно представлен в UI.

## 2. Главный продуктовый принцип

Проект находится в single-developer режиме. Это значит:

```text
Пользователь = разработчик = operator = creator.
```

Следовательно, можно реализовать полный набор переключателей в Web Admin уже сейчас, потому что они нужны для быстрой локальной настройки, экспериментов и будущего тестирования.

Но это НЕ отменяет governance:

```text
UI may expose all switches.
Runtime must still enforce safety boundaries.
```

То есть:

- Web Admin может показывать и давать переключать `all_users`;
- backend обязан ясно маркировать такой режим как `developer_local_all_users`, если нет production-ready gate;
- `production_ready=false` должен оставаться false;
- `broad_rollout_allowed=false` должен оставаться false;
- `normal_user_activation_allowed=false` должен оставаться false как production-governance flag;
- если проект позже получит внешних пользователей, all-users toggle должен быть закрыт отдельным PRD/gate.

## 5. Термины и режимы

### 5.1. Diagnostic Center runtime modes

Реализовать следующий enum:

```text
disabled
shadow
developer
creator_only
allowlist
developer_local_all_users
```

### 5.2. Значение режимов

#### `disabled`

Diagnostic Center не оказывает runtime-влияния. Trace может показывать:

```json
{
  "diagnostic_center": {
    "mode": "disabled",
    "active": false
  }
}
```

#### `shadow`

Diagnostic Center работает только как trace/evidence layer.

```text
No writer prompt mutation.
No final answer mutation.
No user-path authority expansion.
```

#### `developer`

Режим для локального developer testing. Может включать расширенный trace и test apply для developer identity.

#### `creator_only`

Режим для текущего владельца проекта как единственного creator/operator.

#### `allowlist`

Режим для ограниченного списка user_id.

```json
{
  "allowlist_user_ids": ["creator", "pilot_runtime_operator_001"]
}
```

#### `developer_local_all_users`

Режим, позволяющий в локальной разработке применять Diagnostic Center ко всем текущим пользователям системы.

Важно:

```text
Это НЕ production broad rollout.
Это developer-local all-users mode.
```

Backend должен возвращать:

```json
{
  "mode": "developer_local_all_users",
  "production_ready": false,
  "broad_rollout_allowed": false,
  "normal_user_activation_allowed": false,
  "scope_note": "developer-local only; not production broad rollout"
}
```

## 6. Boundary flags

Во всех effective payload должны быть явные flags:

```json
{
  "broad_rollout_allowed": false,
  "production_ready": false,
  "normal_user_activation_allowed": false,
  "single_developer_project": true,
  "developer_local_all_users_available": true,
  "external_users_allowed": false
}
```

`single_developer_project=true` разрешает UI показывать полный набор переключателей.

Но production flags остаются false.

## 7. Backend requirements

### 7.1. Новый модуль runtime policy

Создать небольшой модуль, например:

```text
bot_psychologist/bot_agent/diagnostic_center_control.py
```

или, если структура проекта уже содержит похожее место, использовать его.

Модуль должен содержать:

```python
DiagnosticCenterMode
DiagnosticCenterControlState
load_diagnostic_center_control_state()
save_diagnostic_center_control_state()
build_diagnostic_center_effective_payload()
apply_diagnostic_center_control_update()
reset_diagnostic_center_control_state()
```

### 7.2. Effective payload schema

`GET /api/admin/diagnostic-center/effective` должен возвращать примерно:

```json
{
  "schema_version": "diagnostic_center_control_v1",
  "status": "completed_for_current_creator_only_phase",
  "current_mode": "creator_only",
  "effective_active": true,
  "force_disabled": false,
  "kill_switch_available": true,
  "single_developer_project": true,
  "developer_identity": {
    "creator_user_id": "creator",
    "developer_user_ids": ["creator"],
    "allowlist_user_ids": []
  },
  "available_modes": [
    "disabled",
    "shadow",
    "developer",
    "creator_only",
    "allowlist",
    "developer_local_all_users"
  ],
  "boundary_flags": {
    "broad_rollout_allowed": false,
    "production_ready": false,
    "normal_user_activation_allowed": false,
    "external_users_allowed": false
  },
  "scope": {
    "runtime_scope": "developer_local",
    "external_rollout_scope": "blocked",
    "note": "all-users switch is available only as developer-local governed mode"
  },
  "last_evidence": {
    "last_prd": "PRD-046.1.37-HF1",
    "diagnostic_center_track_status": "closed_for_current_phase",
    "recommended_runner_timeout_sec": 42
  },
  "warnings": []
}
```

### 7.3. Control update endpoint

`POST /api/admin/diagnostic-center/control`

Пример request:

```json
{
  "mode": "developer_local_all_users",
  "force_disabled": false,
  "allowlist_user_ids": ["creator"],
  "reason": "single developer local testing"
}
```

Backend requirements:

- requires dev key;
- validates enum;
- sanitizes allowlist;
- writes audit metadata;
- returns new effective payload;
- never writes `.env`;
- never changes KB/Chroma;
- never sets `production_ready=true`.

### 7.4. Reset endpoint

`POST /api/admin/diagnostic-center/reset`

Должен вернуть safe default:

```json
{
  "mode": "creator_only",
  "force_disabled": false,
  "allowlist_user_ids": []
}
```

или, если безопаснее:

```json
{
  "mode": "shadow",
  "force_disabled": false,
  "allowlist_user_ids": []
}
```

Выбор safe default зафиксировать в implementation report.

### 7.5. Kill switch priority

`force_disabled=true` имеет абсолютный приоритет.

Даже если mode = `developer_local_all_users`, effective payload должен показывать:

```json
{
  "mode": "developer_local_all_users",
  "force_disabled": true,
  "effective_active": false,
  "disabled_reason": "force_disabled"
}
```

## 8. Frontend Web Admin requirements

### 8.1. Новая вкладка или блок

Добавить отдельную вкладку в Web Admin:

```text
Diagnostic Center
```

Если текущая UI-структура проще, допустим отдельный блок внутри `Runtime` или `Diagnostics`, но предпочтительно отдельная вкладка.

### 8.2. UI sections

Вкладка должна иметь секции:

1. **Current Status**
   - статус;
   - current mode;
   - effective active;
   - last PRD;
   - production ready false;
   - broad rollout false.

2. **Runtime Mode**
   - radio/select:
     - disabled;
     - shadow;
     - developer;
     - creator_only;
     - allowlist;
     - developer_local_all_users.

3. **Kill Switch**
   - checkbox/toggle `force_disabled`.

4. **Allowlist**
   - textarea или input для user IDs;
   - простая comma/newline parsing.

5. **Developer-local all-users warning**
   - если выбран `developer_local_all_users`, UI должен показать жёлтое предупреждение:

```text
Этот режим доступен для single-developer local testing.
Он не означает production-ready broad rollout.
```

6. **Boundary Flags**
   - read-only карточки:
     - production_ready=false;
     - broad_rollout_allowed=false;
     - normal_user_activation_allowed=false;
     - external_users_allowed=false.

7. **Actions**
   - Save;
   - Reset to safe default;
   - Refresh effective state.

### 8.3. UX requirement

Все переключатели можно показывать и делать активными, потому что проект single-developer.

Но UI обязан визуально отделять:

```text
developer-local controls
```

от:

```text
production rollout decision
```

## 9. Runtime integration requirement

Этот PRD не обязан полностью менять user-path behavior, если текущий Diagnostic Center control уже живёт через feature flags. Но он должен создать единый effective control contract.

Минимальное требование:

- admin payload показывает effective state;
- control state сохраняется;
- trace/debug может включить diagnostic_center_control snapshot;
- runtime не ломается;
- existing tests remain green.

Если агент видит, что можно безопасно подключить effective mode к уже существующему runtime gating без изменения behavior, он может это сделать.

Но если подключение требует риска/широкого изменения orchestrator path, нужно:

- ограничиться admin/readiness surface;
- явно зафиксировать в report как deferred follow-up.
