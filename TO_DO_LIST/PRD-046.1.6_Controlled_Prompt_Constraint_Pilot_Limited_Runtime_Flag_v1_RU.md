## 1. Краткое резюме

В `PRD-046.1.5` был реализован offline replay/eval слой `writer_prompt_replay`: система научилась сравнивать baseline prompt-context и candidate prompt-context с pilot constraints без влияния на реальный Writer/user-path.

Следующий безопасный шаг — добавить **ограниченный runtime-флаг** для prompt-constraint pilot.

Важно: это **не широкое включение Diagnostic Center в ответы**.

В этом PRD нужно реализовать controlled runtime flag, который по умолчанию выключен и может включать prompt constraints только в узком test/apply режиме:

```text
default production path:
WriterContract -> Writer -> Validator -> final answer
без изменений

test pilot path only when explicitly enabled + allowlisted:
WriterContract -> Prompt Constraint Pilot Decision -> Writer prompt with constrained section -> Validator -> final answer
```

Главная задача PRD — создать безопасную инфраструктуру для ограниченного эксперимента, где:

- по умолчанию нет влияния на user-path;
- pilot может применяться только при явном feature flag;
- pilot может применяться только для allowlisted/test users;
- rollback мгновенно возвращает поведение к baseline;
- safety/KB/quality gates запрещают применение в рискованных случаях;
- все применения фиксируются в trace/eval artifacts;
- roadmap заранее фиксирует будущий `cleanup / stabilization` этап, чтобы временные shadow/eval системы не превратились в шум в проекте.

## 3. Главная цель

Добавить безопасный runtime-механизм, который может в ограниченном тестовом режиме передать pilot constraints в Writer prompt, но только если выполнены все условия:

```text
feature flag enabled
+ mode=test_apply
+ user_id/test context is explicitly allowlisted
+ rollback flag is not active
+ replay/pilot quality passed
+ no KB boundary violation
+ no safety hard blocker
+ no prompt bloat blocker
+ no contract conflict
```

При любом нарушении условий система обязана вернуться в baseline/shadow mode.

## 5. Архитектурный принцип

Сейчас система должна перейти от:

```text
offline replay only
```

к:

```text
default-off limited runtime flag
```

Но не к:

```text
production-wide prompt influence
```

Правильная схема:

```text
User message
  -> State Analyzer
  -> Thread Manager
  -> Memory Retrieval
  -> Context Assembly
  -> Diagnostic Card
  -> Diagnostic Center Shadow
  -> Planner Bridge Candidate
  -> Compliance Shadow
  -> Writer Contract Pilot
  -> Writer Prompt Replay
  -> Prompt Constraint Pilot Runtime Decision
       -> disabled / shadow_only / test_apply
  -> Writer
  -> Validator
  -> final answer
```

По умолчанию `Prompt Constraint Pilot Runtime Decision = disabled`.

## 6. Требования к feature flags

Добавить новые feature flags в `bot_psychologist/bot_agent/feature_flags.py`.

### 6.1 Boolean flags

```python
PROMPT_CONSTRAINT_PILOT_ENABLED = False
PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED = True
```

Пояснение:

- `PROMPT_CONSTRAINT_PILOT_ENABLED=false` — default-off.
- `PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=true` — аварийный rollback, который имеет приоритет над всеми другими настройками.

Если `FORCE_DISABLED=true`, pilot не применяется даже при `ENABLED=true`.

### 6.2 String flags

```python
PROMPT_CONSTRAINT_PILOT_MODE = "shadow"
PROMPT_CONSTRAINT_PILOT_ALLOWED_USER_IDS = ""
PROMPT_CONSTRAINT_PILOT_TEST_USER_PREFIX = "pilot_"
PROMPT_CONSTRAINT_PILOT_MAX_PROMPT_DELTA_CHARS = "2500"
PROMPT_CONSTRAINT_PILOT_MAX_PROMPT_DELTA_RATIO = "0.35"
```

Допустимые значения `PROMPT_CONSTRAINT_PILOT_MODE`:

```text
shadow
test_apply
```

Правила:

- `shadow` — только trace/eval, не влияет на Writer prompt.
- `test_apply` — можно добавить constrained prompt section, но только для allowlisted/test users и только при passed gates.

## 7. Новые модули

Создать отдельные модули. Не складывать логику в `orchestrator.py`.

### 7.1 Contract

Файл:

```text
bot_psychologist/bot_agent/multiagent/contracts/prompt_constraint_pilot_runtime_v1.py
```

Контракты:

```python
PromptConstraintPilotRuntimeInput
PromptConstraintPilotRuntimeDecision
PromptConstraintPilotRuntimeTrace
```

`PromptConstraintPilotRuntimeDecision` должен содержать:

```python
schema_version: str = "prompt_constraint_pilot_runtime_decision_v1"
activation_mode: str  # disabled | shadow_only | test_apply
apply_to_writer_prompt: bool
apply_to_writer_contract: bool
apply_to_final_answer: bool
rollback_active: bool
eligible_user: bool
feature_flag_snapshot: dict
candidate_constraints: dict
blocked_reasons: list[str]
warnings: list[str]
trace: PromptConstraintPilotRuntimeTrace
```

Жёсткие инварианты:

- `apply_to_writer_contract` всегда `False` в этом PRD.
- `apply_to_final_answer` всегда `False` в этом PRD.
- `apply_to_writer_prompt` может быть `True` только при `activation_mode="test_apply"` и только если все gates passed.
- `rollback_active=True` всегда приводит к `activation_mode="disabled"` и `apply_to_writer_prompt=False`.

### 7.2 Runtime decision builder

Файл:

```text
bot_psychologist/bot_agent/multiagent/prompt_constraint_pilot_runtime.py
```

Функция:

```python
build_prompt_constraint_pilot_runtime_decision_v1(
    *,
    user_id: str,
    writer_prompt_replay_result: dict,
    writer_contract_pilot: dict,
    state_snapshot: StateSnapshot,
    thread_state: ThreadState,
    feature_flags_snapshot: dict | None = None,
) -> PromptConstraintPilotRuntimeDecision
```

Builder должен:

1. Читать feature flags.
2. Проверять rollback.
3. Проверять allowlist/test-user eligibility.
4. Проверять replay quality.
5. Проверять safety/KB/prompt-bloat/conflict blockers.
6. Возвращать `disabled`, `shadow_only` или `test_apply`.
7. Никогда не мутировать `WriterContract`.
8. Никогда не вызывать provider.
9. Никогда не изменять final answer напрямую.

### 7.3 Prompt section builder

Файл:

```text
bot_psychologist/bot_agent/multiagent/prompt_constraint_section.py
```

Функция:

```python
format_prompt_constraint_section_v1(decision: PromptConstraintPilotRuntimeDecision | dict) -> str
```

Возвращает короткий текстовый блок для Writer prompt только если:

```text
decision.activation_mode == "test_apply"
decision.apply_to_writer_prompt == true
```

Иначе возвращает пустую строку.

Пример формата:

```text
PILOT PROMPT CONSTRAINTS:
- depth_limit: low
- max_questions: 0
- max_concepts: 1
- must_do: keep_language_short; validate_current_state
- must_not_do: do_not_analyze_deeply; do_not_offer_multiple_steps
- kb_usage_mode: internal_lens_only
- must_not_quote_source: true
```

Ограничения:

- не включать raw KB text;
- не включать internal diagnostic labels, которые могут попасть пользователю;
- не включать длинные списки;
- максимум 8 `must_do`;
- максимум 12 `must_not_do`;
- только sanitized values.

## 8. Изменения в Writer path

### 8.1 Нельзя ломать текущий WriterAgent

Текущий baseline path должен остаться прежним.

Допускается добавить в `WriterAgent` новый optional path:

```python
async def write(
    self,
    contract: WriterContract,
    *,
    prompt_constraint_decision: dict | None = None,
) -> str:
```

или отдельный метод:

```python
async def write_with_prompt_constraints(
    self,
    contract: WriterContract,
    *,
    prompt_constraint_decision: dict,
) -> str:
```

Предпочтительнее минимальный backward-compatible вариант: optional аргумент с default `None`.

Если `prompt_constraint_decision is None` или `apply_to_writer_prompt=False`, Writer должен вести себя идентично baseline.

### 8.2 Изменение prompt template

Не менять `WRITER_SYSTEM`.

Допускается добавить отдельный optional блок в user prompt только для test_apply path.

В baseline path user prompt должен оставаться идентичным прежнему.

Требование:

- в `writer_debug` добавить:
  - `prompt_constraint_pilot_activation_mode`
  - `prompt_constraint_pilot_applied`
  - `prompt_constraint_pilot_blocked_reasons`
  - `prompt_constraint_pilot_prompt_section_chars`
- при baseline/default-off:
  - `prompt_constraint_pilot_applied=false`
  - `prompt_constraint_pilot_prompt_section_chars=0`
