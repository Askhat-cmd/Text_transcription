## 1. Краткий контекст

После `PRD-046.1.6` в проекте появился controlled runtime decision слой `prompt_constraint_pilot_runtime`.

Текущее состояние после `PRD-046.1.6`:

```text
Diagnostic Center v1: trace-only shadow layer
Planner Bridge: candidate-only / shadow-eval lineage
planner_bridge_compliance_shadow: trace-only compare layer
planner_bridge_writer_contract_pilot: pilot_shadow_only
writer_prompt_replay: offline_replay_only
prompt_constraint_pilot_runtime: default-off limited allowlisted test path
```

`PRD-046.1.6` подтвердил:

```text
cases_total=50
```

Но это означает только то, что limited runtime-флаг технически готов. Это **не означает**, что можно расширять Diagnostic Center / Prompt-Constraint Pilot на широкий runtime.

Следующий шаг должен быть не добавлением новой функциональности, а проверкой evidence:

```text
собрать runtime/test_apply evidence
→ проверить rollback-поведение при повторных переключениях
→ сравнить baseline vs test_apply quality deltas
→ проверить safety/KB/conflict/bloat gates
→ сформировать решение: stay_limited | hotfix_required | supervised_rollout_candidate
```

## 2. Цель PRD

Цель `PRD-046.1.7` — создать воспроизводимый audit/eval слой, который анализирует результаты limited `test_apply` режима Prompt-Constraint Pilot и принимает безопасное архитектурное решение о следующем шаге.

Главная задача:

```text
Не расширить runtime-влияние.
А доказать, что controlled prompt constraints безопасны, откатываемы и качественно полезны в ограниченном test path.
```

## 3. Главный принцип

Этот PRD должен работать как quality gate, а не как rollout PRD.

Запрещено:

```text
- включать Prompt-Constraint Pilot по умолчанию;
- расширять allowlist автоматически;
- делать broad rollout;
- менять WriterContract как production authority;
- менять final answer path для обычных пользователей;
- менять KB governance authority;
- делать Chroma reindex;
- делать production KB mutation;
- использовать provider calls для обязательных acceptance gates;
- переносить Diagnostic Center из shadow в active runtime.
```

Разрешено:

```text
- собирать и анализировать existing/runtime-like trace artifacts;
- создавать synthetic runtime evidence cases;
- повторно запускать deterministic eval/smoke;
- моделировать rollback toggle scenarios;
- сравнивать baseline и test_apply prompt-context на уровне quality metrics;
- генерировать safe/sanitized JSON reports;
- обновлять docs/PROJECT_STATE.md, docs/ROADMAP.md, docs/PRD_INDEX.md;
- добавить ADR только если будет принято новое архитектурное решение.
```

## 4. Non-goals

В рамках этого PRD не делать:

1. **Не делать broad rollout.**  
   Даже если quality gate зелёный, результатом может быть только рекомендация к следующему отдельному PRD.

2. **Не менять default flags.**  
   Текущая безопасная позиция сохраняется:

   ```text
   PROMPT_CONSTRAINT_PILOT_ENABLED=false
   PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=true
   ```

3. **Не делать Diagnostic Center active authority.**  
   Diagnostic Center остаётся internal map-making / trace lineage.

4. **Не давать LLM authority над safety/governance.**  
   Deterministic governance остаётся authority.

5. **Не делать provider-dependent acceptance.**  
   Eval должен быть воспроизводимым без реального provider-call.

6. **Не менять KB/chunk/source data.**  
   `all_blocks_merged.json`, registry, Chroma и source content не должны мутироваться.
