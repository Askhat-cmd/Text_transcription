# PRD-046.1.8 — Supervised Prompt-Constraint Pilot Rollout Plan v1

## 0. Краткое название

**Supervised Prompt-Constraint Pilot Rollout Plan v1**

## 1. Контекст

После `PRD-046.1.6` в проект добавлен controlled runtime decision слой `prompt_constraint_pilot_runtime`:

```text
PROMPT_CONSTRAINT_PILOT_ENABLED=false
PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=true
PROMPT_CONSTRAINT_PILOT_MODE=shadow|test_apply
PROMPT_CONSTRAINT_PILOT_ALLOWED_USER_IDS=
PROMPT_CONSTRAINT_PILOT_TEST_USER_PREFIX=pilot_
PROMPT_CONSTRAINT_PILOT_MAX_PROMPT_DELTA_CHARS=2500
PROMPT_CONSTRAINT_PILOT_MAX_PROMPT_DELTA_RATIO=0.35
```

После `PRD-046.1.7` evidence/rollback/quality gate прошёл успешно:

```text
final_status=passed
decision=supervised_rollout_candidate
evidence_quality=sufficient
rollback_failure_count=0
candidate_weaker_than_baseline_count=0
raw_kb_text_exposure_count=0
normal_user_apply_count=0
provider_called_by_eval_count=0
production_mutation_detected=false
```

Это означает, что prompt-constraint pilot можно готовить к **узкому supervised rollout**, но это ещё не означает broad rollout и не означает включение Diagnostic Center как полноценного runtime authority.

Текущий baseline остаётся:

```text
Diagnostic Center v1: trace-only shadow layer
Planner Bridge: candidate-only / shadow-eval lineage
planner_bridge_compliance_shadow: trace-only compare layer
planner_bridge_writer_contract_pilot: pilot_shadow_only
writer_prompt_replay: offline_replay_only
prompt_constraint_pilot_runtime: default-off limited allowlisted test path
```

## 2. Цель PRD

Подготовить **строгий, проверяемый и откатываемый supervised rollout plan** для Prompt-Constraint Pilot.

PRD должен создать не “включение для всех”, а план и технический readiness layer, который отвечает на вопросы:

```text
кто может попасть в rollout;
какие флаги и условия нужны;
какие gates должны пройти до test_apply;
как быстро отключить pilot;
какие метрики считать;
что является blocker;
что является warning;
когда rollout запрещён;
какой следующий PRD нужен для execution.
```

## 3. Главный принцип

```text
PRD-046.1.8 планирует controlled supervised rollout,
но не расширяет production activation автоматически.
```

После выполнения этого PRD обычный пользовательский путь должен остаться прежним:

```text
PROMPT_CONSTRAINT_PILOT_ENABLED=false
PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=true
normal_user_apply_count=0
default_off_user_path_effect_count=0
```

## 4. Non-goals

В рамках этого PRD запрещено:

1. Включать broad rollout.
2. Включать `PROMPT_CONSTRAINT_PILOT_ENABLED=true` как новый default.
3. Менять `PROMPT_CONSTRAINT_PILOT_FORCE_DISABLED=true` как baseline default.
4. Добавлять auto-enable логику.
5. Давать Diagnostic Center прямую authority-роль.
6. Менять `chunk_type`, `allowed_use`, `safety_flags` через LLM или overlay.
7. Мутировать production KB, registry, Chroma или config.
8. Делать provider-dependent acceptance.
9. Менять final answer для обычных пользователей.
10. Удалять или ослаблять rollback-first policy.
11. Делать supervised rollout execution без отдельного следующего PRD.

```text

```

Warning conditions:

```text

operator runbook missing optional human-readable md
```

Warnings не должны автоматически блокировать PRD, если hard abort = 0.

### 5.6. Toggle matrix

Создать supervised toggle matrix, которая фиксирует ожидаемое поведение:

| ENABLED | FORCE_DISABLED | MODE | user | expected |
| --- | --- | --- | --- | --- |
| false | true | shadow | normal | disabled/no apply |
| true | true | test_apply | allowlisted | disabled/no apply |
| true | false | shadow | allowlisted | shadow_only/no apply |
| true | false | test_apply | normal | shadow_only/no apply |
| true | false | test_apply | non-allowlisted | shadow_only/no apply |
| true | false | test_apply | `pilot_*` | apply only if all gates pass |
| true | false | test_apply | explicit allowlist | apply only if all gates pass |
| true | false | test_apply | allowlisted + safety gate fail | downgrade/block |
| true | false | test_apply | allowlisted + KB gate fail | downgrade/block |
| true | false | test_apply | allowlisted + bloat/conflict fail | downgrade/block |

Матрица должна быть машинно проверяемой и сохраняться в JSON.

### 5.7. Operator runbook

Создать runbook для будущего supervised execution PRD.

Runbook должен включать:

```text
1. Preflight перед включением.
2. Какие env flags выставляются вручную.
3. Какой минимальный cohort разрешён.
4. Как собрать trace после каждого supervised run.
5. Как проверить normal user no-effect.
6. Как проверить rollback.
7. Как отключить pilot.
8. Какие artifacts сохранить.
9. Какие метрики сравнить.
10. Какие условия автоматически запрещают следующий шаг.
```

Важно: runbook не должен просить агента включать флаги прямо сейчас. Он должен описывать будущую процедуру.
