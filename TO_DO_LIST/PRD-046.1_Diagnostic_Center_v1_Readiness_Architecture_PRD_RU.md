## 0. Краткое резюме для IDE-агента

Этот PRD начинает фазу **Diagnostic Center v1**, но **не включает Diagnostic Center в активный пользовательский runtime**.

Задача этого шага — создать безопасный архитектурный фундамент:

```text
контракты Diagnostic Center
→ схемы входа/выхода
→ deterministic dry-run builder
→ golden/eval cases
→ audit runner
→ no-mutation proof
→ docs/ADR sync
```

Diagnostic Center в этом PRD должен работать как **read-only / dry-run архитектурный слой**, который на основе уже существующих сигналов (`State Analyzer`, `Thread Manager`, `Context Assembly`, `KB/retrieval metadata`) собирает диагностическую карту хода.

Главный принцип:

```text
Diagnostic Center не пишет пользователю.
Diagnostic Center не заменяет Writer.
Diagnostic Center не принимает governance authority решения.
Diagnostic Center собирает карту текущего хода и предлагает следующий микро-сдвиг.
```

## 2. Архитектурный смысл Diagnostic Center v1

Diagnostic Center v1 — это не “ещё один большой психологический агент”.

Это **слой восприятия и сборки карты текущего хода**.

Он должен помочь системе отвечать не только на текст вопроса, а на более точный вопрос:

```text
Где сейчас находится пользователь,
в какой линии разговора мы уже идём,
какой внутренний механизм может быть активен,
и какой следующий микро-сдвиг сейчас уместен?
```

Diagnostic Center должен поддерживать главную формулу проекта:

```text
Не дать человеку готовую истину.
А помочь ему увидеть механизм, из которого родился его вопрос.
```

И одновременно соблюдать ограничения:

```text
не лечить;
не диагностировать клинически;
не становиться духовным авторитетом;
не давать директивных жизненных советов;
не превращать КУЗНИЦУ ДУХА в цитатник;
не ломать защиту пользователя;
не вести глубже, чем человек может выдержать.
```

---

## 3. Главная проблема, которую решает PRD

Сейчас система уже умеет:

```text
читать состояние;
держать thread;
собирать context package;
подмешивать governed KB hits;
формировать Diagnostic Card;
ограничивать Writer через compliance;
валидировать ответ.
```

Но эти сигналы пока не объединены в отдельный устойчивый слой:

```text
Diagnostic Center Output
```

Из-за этого дальнейшее развитие может пойти опасным путём:

```text
Writer снова начнёт сам “додиагностировать”;
новая логика расползётся по prompt-ам;
KB-линзы могут использоваться без явного контракта;
thread continuity и diagnostic hypothesis могут конфликтовать;
сложно будет тестировать качество диагностического хода отдельно от текста ответа.
```

Этот PRD создаёт промежуточный слой, чтобы **сначала стабилизировать диагностическую карту**, а уже потом подключать её к runtime.

## 4. Goals

### 4.1. Создать контракт Diagnostic Center v1

Нужно определить строгие схемы:

```text
DiagnosticCenterInput
DiagnosticCenterOutput
DiagnosticHypothesis
DiagnosticLensSignal
NextMicroShift
DiagnosticCenterTrace
```

Контракт должен быть машинно-проверяемым и расширяемым.

### 4.2. Создать dry-run Diagnostic Center builder

Builder должен работать в безопасном режиме:

```text
read-only
deterministic-first
no provider calls
no production mutation
no runtime activation
```

Он должен принимать уже существующие данные/сигналы и возвращать структурированный output.

### 4.3. Зафиксировать диагностические оси

Минимальные оси:

```text
nervous_state: hyper | window | hypo | unknown
intent: contact | vent | clarify | validation | directive | explore | practical_step | crisis | unknown
openness: open | mixed | defensive | collapsed | unknown
ok_position: I+W+ | I-W+ | I+W- | I-W- | unknown
relation_to_thread: continue | branch | new_thread | return_to_old | unknown
phase: stabilize | clarify | explore | integrate | unknown
```

Важно:

```text
ok_position — это не ярлык на человека.
Это временная позиция текущего хода.
```

### 4.4. Ввести рабочую диагностическую гипотезу

Diagnostic Center должен возвращать не “истину”, а hypothesis:

```text
working_hypothesis
confidence
evidence
counter_evidence
risk_flags
```

Пример:

```json
{
  "working_hypothesis": {
    "pattern_candidate": "confirmation_seeking_under_uncertainty",
    "mechanism_summary": "пользователь может искать внешнее подтверждение вместо внутреннего авторства",
    "blind_spot_candidate": "форма вопроса выглядит как анализ, но внутри может быть запрос на разрешение",
    "confidence": 0.68
  }
}
```

### 4.5. Ввести Next Micro-Shift

Diagnostic Center должен предлагать не готовый ответ, а следующий ход для Writer/Planner:

```text
response_goal
response_mode
target_micro_shift
must_do
must_not_do
depth_allowed
max_questions
```

Пример:

```json
{
  "next_micro_shift": {
    "response_goal": "stabilize_authorship",
    "response_mode": "reflect_then_one_question",
    "target_micro_shift": "вернуть пользователю часть авторства без давления",
    "must_do": ["validate_feeling", "avoid_directive_answer", "name_one_mechanism_softly"],
    "must_not_do": ["give_final_advice", "quote_kb_directly", "introduce_new_framework"],
    "depth_allowed": "low_to_medium",
    "max_questions": 1
  }
}
```

### 4.6. Создать eval/golden cases

Нужно создать набор тестовых случаев, которые проверяют:

```text
I-W+ + defensive
I-W+ + open
I+W- + defensive
I+W- + open
I-W- + collapsed
I-W- + mixed
I+W+ + open
hyper/hypo safety-first
branch that looks new but shares same pattern_core
KB lens is useful but direct quote is forbidden
```

Минимум: **10 cases**.

### 4.7. Обновить docs

Обновить:

```text

docs/ROADMAP.md
docs/PRD_INDEX.md
docs/DECISIONS.md
```

Добавить новое ADR:

## 6. Архитектурные правила

### 6.1. Diagnostic Center не пишет пользователю

Diagnostic Center output — это internal contract.

Он может дать Writer/Planner:

```text
что держать;
чего избегать;
какой микро-сдвиг нужен;
какая глубина допустима;
какая гипотеза вероятна.
```

Но он не должен формировать финальный ответ.

### 6.2. Diagnostic Center не заменяет существующие модули

Он не должен заново выполнять роль:

```text
State Analyzer
Thread Manager
Context Assembly
Writer Move Compliance
Validator
```

Он должен использовать их сигналы и объединять их в карту.

### 6.3. Safety и nervous_state имеют приоритет

Если вход показывает:

```text
nervous_state = hyper
nervous_state = hypo
intent = crisis
safety flags present
```

то Diagnostic Center должен ограничить глубину:

```text
depth_allowed = none | low
response_goal = stabilize | ground | safety_redirect
```

И запретить:

```text
deep_pattern_interpretation
existential_reframe
complex KB lens
multi-question response
```

### 6.4. Openness и ok_position не являются диагнозом

Они должны трактоваться как:

```text
temporary turn-position
probabilistic signal
routing hint
```

Запрещено:

```text
помечать пользователя как “дефенсивный человек”;
писать это пользователю;
сохранять как клинический диагноз;
использовать как неизменный профиль.
```

### 6.5. KB используется только как internal lens library

Diagnostic Center может выбрать:

```text
lens_family_candidate
practice_candidate
avoid_when
use_when
```

Но не должен:

```text
подставлять длинные фрагменты книги;
цитировать source напрямую;
делать source authority поверх состояния пользователя;
обходить retrieval governance.
```

### 6.6. Code-first там, где можно кодом

Следовать принципу:

```text
LLM — только там, где нужна интерпретация.
Код — для контрактов, enum, validation, routing guardrails, continuity checks, no-mutation proof.
```

В этом PRD Diagnostic Center v1 builder должен быть deterministic/dry-run.

## 10. Базовые routing rules

### 10.1. `I-W+ + defensive`

Сигнал:

```text
ищет правильный ответ;
передаёт авторство наружу;
боится ошибиться;
просит директиву.
```

Diagnostic Center должен выбирать:

```text
response_goal = stabilize_authorship
depth_allowed = low_to_medium
must_not_do = give_directive_certainty
```

### 10.2. `I+W- + defensive`

Сигнал:

```text
обвинительная рамка;
проверка бота;
поиск союзника против других.
```

Diagnostic Center должен выбирать:

```text
response_goal = decenter_without_shaming
must_not_do = join_accusation
must_do = validate_feeling_without_confirming_frame
```

### 10.3. `I-W- + collapsed`

Сигнал:

```text
низкий ресурс;
бессилие;
нет пространства для анализа.
```

Diagnostic Center должен выбирать:

```text
response_goal = ground_and_reduce_load
depth_allowed = none|low
must_not_do = deep_analysis
```

### 10.4. `I+W+ + open`

Сигнал:

```text
человек готов видеть шире;
есть ресурс;
есть метапозиция.
```

Diagnostic Center может выбирать:

```text
response_goal = deepen_and_integrate
depth_allowed = medium|high
must_do = offer_alternative_perspective
```

### 10.5. Hyper/Hypo priority

Любое состояние:

```text
nervous_state = hyper|hypo
```

должно ограничивать сложность:

```text
depth_allowed <= low
max_questions <= 1
max_concepts <= 1
```

Если есть crisis/safety flags, Diagnostic Center output:

```text
status = safety_first
response_goal = safety_redirect
```

```text
## ADR-030 - Diagnostic Center is a map-making layer, not a user-facing agent

Status: accepted

Context:
Проект переходит к Diagnostic Center после readiness gates. Есть риск сделать Diagnostic Center большим LLM-агентом, который начнет писать пользователю, заменять Writer или принимать authority-решения по KB/governance.

Decision:
Diagnostic Center v1 является internal map-making layer. Он собирает DiagnosticCenterOutput из существующих сигналов State Analyzer / Thread Manager / Context Assembly / governed retrieval metadata и предлагает working_hypothesis + next_micro_shift. Он не пишет финальный ответ, не цитирует KB, не меняет governance authority и не включается в active runtime без отдельного PRD.

Consequences:
Диагностическая логика становится тестируемой отдельно от Writer. Runtime сохраняет multiagent дисциплину. Следующий PRD может подключать Diagnostic Center только в shadow mode или controlled integration.
```

## 21. Критерий качества

Хорошая реализация этого PRD — это не “умный Diagnostic Center”.

Хорошая реализация — это когда появился **ясный, тестируемый и безопасный контракт**, по которому следующий PRD сможет подключать Diagnostic Center в shadow-mode без риска сломать текущий runtime.

Главная проверка:

```text
Можно ли теперь отдельно тестировать:
что система увидела,
почему выбрала такой ход,
какую глубину разрешила,
какую линзу допустила,
и какой следующий микро-сдвиг предложила?
```

Если да — PRD выполнен правильно.
