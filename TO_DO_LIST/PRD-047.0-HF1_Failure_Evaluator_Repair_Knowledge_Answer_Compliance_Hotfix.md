# PRD-047.0-HF1 — Failure Evaluator Repair + Knowledge Answer Compliance Hotfix

## 0. Краткое резюме

`PRD-047.0` сделал важную основу: добавил `Knowledge Answer Routing Guard`, `Practice Gate v0`, dataset и direct/live runner. Но аудит GitHub после реализации показал, что PRD нельзя считать полностью принятым по качеству поведения.

Главная проблема: отчёт `live_failure_cases_direct.json` формально показывает `5/5 passed`, но сами ответы всё ещё содержат старые ошибки:

1. Greeting case:
   - `practice_gate.practice_allowed=false`;
   - но ответ всё равно предлагает дыхательную практику:
     ```text
     Положи ладонь на грудь, медленно вдохни...
     ```
   - это должно было считаться fail.

2. Known concept challenge:
   - guard срабатывает;
   - но ответ всё ещё говорит о двух смыслах, включая внешнее отслеживание, и снова спрашивает:
     ```text
     О каком из этих двух смыслов вы говорите?
     ```
   - это нарушает смысл `do_not_ask_user_to_define_term_before_answering`.

3. Relation case:
   - `knowledge_answer.needed=true`;
   - но `kb_grounding_available=false`;
   - ответ снова уходит в технический мониторинг / ЭЭГ / нейроинтерфейсы;
   - runner всё равно засчитывает case как passed.

4. Evaluation runner проверяет в основном guard-флаги, но недостаточно проверяет сам текст ответа.

Цель HF1 — не расширять архитектуру, а закрыть этот разрыв:

```text
trace/guard says correct
but final answer is still wrong
and evaluator incorrectly passes it
```

### 1.2. Почему нужен HF1

`PRD-047.0` выполнил structural contract, но acceptance quality пока ложноположительный.

Это опасно для следующего трека: если runner считает слабые ответы passed, мы начнём улучшать Active Frame / Planner на неверной диагностике.

Поэтому перед `PRD-047.1 — Active Frame / Response Planner v1` нужен короткий hotfix:

```text
PRD-047.0-HF1
```

Он должен:

- исправить evaluator;
- заставить runner падать на реальных слабых ответах;
- усилить runtime compliance там, где guard уже говорит `practice_allowed=false` или `knowledge_answer_first=true`;
- перепрогнать failure cases;
- добиться честного pass только когда финальный ответ действительно соответствует ожиданиям.

## 2. Главная цель

Сделать так, чтобы `PRD-047.0` failure-case baseline проверял не только внутренние флаги, но и итоговый текст ответа.

Финальный критерий:

```text
Case считается passed только если:
1. routing/trace flags корректны;
2. final answer не нарушает запреты;
3. final answer содержит ожидаемый смысл;
4. final answer не просит пользователя определить известный внутренний термин;
5. final answer не предлагает практику, когда practice_allowed=false.
```

## 5. Required runtime fixes

### 5.1. Enforce Practice Gate in Writer Contract

Если:

```json
"practice_gate": {
  "practice_allowed": false
}
```

то Writer Contract / Writer Prompt должен явно получить запреты:

```text
- Не предлагай упражнение.
- Не предлагай дыхание.
- Не предлагай телесную практику.
- Не давай микрошаг как действие, если пользователь не просил шаг и нет safety/regulate необходимости.
```

Важно: это не должно ломать `safe_override` и явный low-resource/safety path.

Для greeting:

```text
Привет! Меня зовут Асхат!
```

ожидаемый ответ должен быть простым контактом:

```text
Привет, Асхат. Рад знакомству. Можем спокойно начать: принеси любой вопрос или тему, с которой хочешь разобраться.
```

Недопустимо:

```text
Положи руку...
Сделай вдох...
Выдохни...
Почувствуй тело...
```

если пользователь не просил практику и нет safety/low-resource признаков.

### 5.2. Knowledge Answer First must override clarification question

Если:

```json
"knowledge_answer": {
  "needed": true,
  "should_answer_directly": true,
  "should_ask_definition_first": false,
  "kb_grounding_available": true
}
```

то Writer не должен завершать ответ вопросом типа:

```text
Что ты вкладываешь в этот термин?
О каком варианте ты говоришь?
Ты имеешь в виду внешнее или внутреннее?
```

Допустим мягкий финал после ответа:

```text
Если ты используешь термин иначе, можно уточнить.
```

Но не как основной ход и не как вопрос, который перекладывает определение обратно на пользователя.

### 5.3. Internal concept beats external surveillance interpretation

Для `нейросталкинг`, если есть internal KB grounding, запрещено предлагать внешний смысл как равноправный вариант:

Недопустимые фразы:

```text
внешнее слежение
отслеживание чужих нейроданных
биофидбек
ЭЭГ
нейроинтерфейсы
технический мониторинг
цифровые следы
```

если пользователь явно не спрашивал именно про внешний/технический смысл.

Правильный ответ должен опираться на internal meaning:

```text
НейроСталкинг — это наблюдение за паттернами, триггерами и автоматическими реакциями: как включается программа, где запускается страдание, как не сливаться с реакцией полностью.
```

### 5.4. Relation case grounding fallback

Для вопроса:

```text
Как самореализация коррелируется с нейросталкингом?
```

HF1 должен добиться:

```json
"knowledge_answer": {
  "needed": true,
  "matched_concepts": ["нейросталкинг", "самореализация"],
  "kb_grounding_available": true,
  "should_answer_directly": true
}
```

Если обычный RAG не приносит lexical hit по `нейросталкинг`, нужно добавить bounded fallback:

- либо concept-focused retrieval query;
- либо использование raw semantic candidates до фильтрации, если они безопасны;
- либо deterministic concept summary fallback только для уже известного governed concept alias map.

Предпочтение:

```text
Сначала concept-focused retrieval / lexical override.
Fallback summary — только если реализуется безопасно и явно помечается как internal concept fallback.
```

Не делать Chroma reindex.

## 8. Expected direct answers

### 8.1. Greeting

Input:

```text
Привет! Меня зовут Асхат!
```

Expected style:

```text
Привет, Асхат. Рад знакомству. Можем спокойно начать: принеси любой вопрос или тему, с которой хочешь разобраться.
```

No practice.

### 8.2. Known concept challenge

Input:

```text
Ты вообще понимаешь что такое "Нейросталкинг"?
```

Expected style:

```text
Да, Асхат, теперь точнее: в нашей внутренней рамке НейроСталкинг — это не внешнее слежение и не биофидбек, а наблюдение за тем, как внутри включаются паттерны, триггеры и автоматические реакции. Это работа с первым уровнем: увидеть программу в моменте, не сливаться с ней полностью и постепенно перестать запускаться на старый автоматизм.
```

Допускается признать неточность:

```text
Раньше я сдвинулся в неверную сторону, когда стал говорить про внешнее отслеживание.
```

### 8.3. Relation case

Input:

```text
Как самореализация коррелируется с нейросталкингом?
```

Expected style:

```text
Если держаться внутреннего смысла термина, НейроСталкинг связан с самореализацией через снятие автопилота. Самореализация требует проявляться и выбирать своё направление, а НейроСталкинг помогает увидеть паттерны, триггеры и автоматические реакции, которые это проявление блокируют. То есть он не “создаёт самореализацию”, а помогает заметить внутренние программы, из-за которых человек снова уходит в страх, обесценивание или откладывание.
```

No external surveillance, no EEG, no биофидбек.

## 16. Архитектурная позиция

Этот HF1 важнее, чем кажется.

Он не просто чинит один вопрос про НейроСталкинг. Он вводит принцип:

```text
Внутренние trace-флаги не являются доказательством качества.
Качество подтверждается только итоговым ответом.
```

Без этого весь дальнейший Multiagent Quality & Tuning Track будет строиться на ненадёжных метриках.
