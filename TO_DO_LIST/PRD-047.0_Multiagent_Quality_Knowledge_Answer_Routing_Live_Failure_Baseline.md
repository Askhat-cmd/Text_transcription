# PRD-047.0 — Multiagent Quality & Tuning Start: Knowledge Answer Routing + Live Dialogue Failure Baseline v1

## 0. Краткое резюме

Цель `PRD-047.0` — начать трек `Multiagent Quality & Tuning Track` с самой заметной живой проблемы: бот не умеет уверенно отвечать на вопросы по известным внутренним понятиям из базы знаний, даже когда RAG уже принёс релевантные чанки.

Реальный симптом из живого чата:

```text
Пользователь: Ты вообще понимаешь что такое "Нейросталкинг"?
RAG hits: содержат определение НейроСталкинга из базы.
Writer response: снова просит пользователя уточнить, что он вкладывает в термин.
```

Это не просто слабый Writer. Это сбой связки:

```text
State Analyzer
→ Thread Manager / response_mode
→ Memory Retrieval / RAG
→ Context Assembly
→ Writer Contract
→ Writer
→ Validator / Quality Trace
```

В загруженном `Полотно ЛЛМ_%.txt` видно, что Writer получил релевантный KB-фрагмент про НейроСталкинг, но одновременно получил `response_mode=reflect` и обязательство задать уточняющий вопрос. Поэтому он выбрал уточнение вместо ответа.

Этот PRD не должен копировать стиль Wake_up. Он должен перенести полезные механизмы:

- держать линию разговора;
- не начинать каждый ответ с нуля;
- отвечать из известной внутренней карты понятий;
- не вставлять практики без основания;
- использовать RAG как внутреннюю линзу, а не как цитатник;
- показывать в trace, почему был выбран именно такой ход.

## 1. Контекст

После `PRD-046.1.38` Diagnostic Center v1 имеет Web Admin control surface и закрыт для текущей creator/developer-local фазы. Следующий большой фокус проекта — `Multiagent Quality & Tuning Track`.

Текущий runtime:

```text
User message
  ↓
State Analyzer
  ↓
Thread Manager
  ↓
Memory Retrieval
  ↓
Context Assembly
  ↓
Diagnostic Card
  ↓
Diagnostic Center / Planner Bridge / Prompt-Constraint layers
  ↓
Writer Move Compliance
  ↓
Writer
  ↓
Validator / Trace
  ↓
Memory Update
```

Живой чат выявил несколько классов проблем:

1. **Greeting / first contact**
   - пользователь: `Привет! Меня зовут Асхат!`
   - бот сразу предлагает дыхательную практику;
   - проблема: practice/regulate появляется без запроса и без признаков low-resource.

2. **Known concept question**
   - пользователь: `Что такое самореализация?`
   - бот отвечает терпимо, но довольно общо.

3. **Known internal concept relation**
   - пользователь: `Как самореализация коррелируется с нейросталкингом?`
   - бот трактует `нейросталкинг` как внешнее слежение / отслеживание мозговой активности, хотя в базе есть внутреннее определение.

4. **User correction / challenge**
   - пользователь: `Ты вообще понимаешь что такое "Нейросталкинг"?`
   - RAG приносит релевантные чанки:
     - `НейроСталкинг и НеоСталкинг — это два уровня одного процесса...`
     - `НейроСталкинг — это работа с первым этажом...`
   - Writer снова просит пользователя определить термин.

Это критичный failure case для доверия к системе.

## 2. Главная цель PRD

Сделать первый управляемый шаг настройки качества:

```text
Если пользователь спрашивает о понятии, которое есть во внутренней базе знаний,
и RAG/lexical match даёт прямое попадание,
бот должен сначала ответить по внутреннему смыслу понятия,
а не просить пользователя снова объяснить термин.
```

Параллельно нужно заложить baseline для живых failure cases:

```text
live_dialogue_failure_cases_v1
```

чтобы следующие PRD улучшали качество по измеримым примерам, а не по ощущению.

10. Не менять модель Writer по умолчанию.
11. Не коммитить `.env`, secrets, raw provider payload, raw private logs, sqlite/db/cache.

## 4. Product principles

### 4.1. Knowledge answer beats empty clarification

Если система уже имеет релевантный внутренний материал, Writer не должен отвечать так:

```text
Что ты вкладываешь в это слово?
```

если можно сначала дать рабочее внутреннее определение:

```text
В нашей внутренней рамке НейроСталкинг — это наблюдение за паттернами, триггерами и автоматическими реакциями: как включается программа, где запускается страдание, как не сливаться с реакцией целиком.
```

Допустимо добавить уточнение после ответа, но не вместо ответа.

### 4.2. KB is internal lens library, not quote source

Ответ должен быть живым и адаптированным, а не дословным пересказом книги.

Правильно:

```text
Если говорить в логике нашей базы, НейроСталкинг — это первый уровень наблюдения: человек учится видеть свои автоматические реакции, триггеры и паттерны до того, как они полностью захватят поведение.
```

Неправильно:

```text
В книге сказано: "НейроСталкинг и НеоСталкинг — это два уровня одного процесса..."
```

### 4.3. Practice is not default help

Практика появляется только если:

```text
- пользователь просит конкретный шаг;
- response planning выбрал integrate/action;
- состояние требует регуляции;
- практика прямо продолжает текущую линию;
- или пользователь явно хочет упражнение.
```

Greeting, concept question, clarification, correction и disagreement не должны автоматически вести к practice.

### 4.4. Wake_up is a mechanical reference, not style target

Из Wake_up берём:

- удержание линии;
- опору на предыдущий ответ пользователя;
- один следующий смысловой ход;
- не начинать каждый раз заново;
- работать с паттерном, а не только с текстом вопроса.

Не берём:

- жёсткую драматизацию;
- давление;
- “щелчки” как обязательную манеру;
- авторитарные интерпретации;
- стиль “я вижу тебя насквозь”.

## 5. Required design change

### 5.1. Новый lightweight слой: Knowledge Answer Routing Guard

Добавить небольшой deterministic guard, который работает после Memory Retrieval / Context Assembly и до Writer Contract.

Возможные имена:

```text
knowledge_answer_routing.py
knowledge_answer_guard.py
rag_answer_planner.py
```

Рекомендуемое место:

```text
bot_psychologist/bot_agent/multiagent/
```

или место, соответствующее текущей структуре runtime.

Guard должен определить:

```json
{
  "knowledge_answer": {
    "needed": true,
    "reason": "known_internal_concept_question",
    "concept": "нейросталкинг",
    "concept_match_type": "exact_or_near_exact",
    "kb_grounding_available": true,
    "should_answer_directly": true,
    "should_ask_definition_first": false,
    "practice_allowed": false,
    "preferred_response_mode": "inform",
    "writer_instruction": "Answer from internal KB meaning first; do not ask the user to define the term before giving the known meaning."
  }
}
```

Если в системе уже есть близкий компонент, не создавать дубликат, а расширить существующий planning/diagnostic bridge.

### 5.2. Concept detection

Минимально покрыть следующие типы запросов:

```text
Что такое X?
Что значит X?
Как X связано с Y?
Как X коррелирует с Y?
Ты понимаешь, что такое X?
Ты вообще понимаешь X?
Ты неправильно понял X.
```

Для этого PRD обязательно покрыть:

```text
нейросталкинг
неосталкинг / нео-сталкинг
самореализация
```

Но реализация не должна быть жёстко зашита только под эти три слова. Нужно сделать простую расширяемую структуру:

```json
{
  "concept_aliases": {
    "нейросталкинг": ["нейросталкинг", "нейро сталкинг", "нейро-сталкинг", "neurostalking"],
    "неосталкинг": ["неосталкинг", "нео сталкинг", "нео-сталкинг", "neostalking"],
    "самореализация": ["самореализация", "самореализоваться"]
  }
}
```

Если есть более правильное место для словаря/настроек — использовать его.

### 5.3. RAG evidence threshold

Не полагаться только на semantic score, потому что живой trace показывает низкие scores при фактически релевантном exact concept hit.

Нужен combined signal:

```text
semantic_score
+ lexical concept match
+ phrase presence in hit preview/content
+ source allowed for internal lens
```

Если semantic score низкий, но в hit preview есть точное понятие и определяющая фраза — считать `kb_grounding_available=true`.

### 5.4. Writer Contract adjustment

Если `knowledge_answer.needed=true` и `kb_grounding_available=true`, Writer prompt должен получить отдельный блок:

```text
KNOWLEDGE ANSWER ROUTING:
- Пользователь спрашивает о понятии, которое найдено во внутренней базе.
- Сначала дай краткое рабочее объяснение по базе.
- Не проси пользователя определить термин до ответа.
- Не превращай ответ в лекцию.
- Не цитируй базу как внешний авторитет.
- Не давай практику, если пользователь не просил шаг.
- Если есть неоднозначность, сначала дай внутренний смысл, затем мягко уточни.
```

### 5.5. Response mode compatibility

Если текущий pipeline пока не поддерживает `inform`, допустимы два варианта:

Вариант A — добавить новый `response_mode=inform`.

Вариант B — оставить `reflect/explore`, но добавить higher-priority writer instruction:

```text
knowledge_answer_first=true
```

и запретить `ask_one_specific_question` как обязательный ход в этом сценарии.

Предпочтение:

```text
Сначала B, если это меньше ломает архитектуру.
A — только если mode enum уже расширяем безопасно.
```

### 5.6. Practice Gate v0

Добавить минимальный guard:

```json
{
  "practice_gate": {
    "practice_allowed": false,
    "reason": "concept_question_or_greeting_not_action_request"
  }
}
```

Покрыть сценарии:

```text
Привет! Меня зовут Асхат!        -> no practice
Что такое нейросталкинг?         -> no practice
Как X связано с Y?               -> no practice by default
Дай один конкретный шаг          -> practice allowed
Мне плохо, помоги успокоиться    -> regulate/practice allowed
```

Важно: Practice Gate v0 не должен ломать safety/regulate для кризиса или явного low-resource.

## 8. Expected answer quality

Для вопроса:

```text
Ты вообще понимаешь что такое "Нейросталкинг"?
```

Допустимый ответ должен быть примерно такого типа:

```text
Да, Асхат, теперь вижу точнее: в нашей рамке НейроСталкинг — это не внешнее слежение и не биофидбек, а наблюдение за тем, как внутри включаются паттерны, триггеры и автоматические реакции. Это работа с “первым этажом”: увидеть программу в моменте, не сливаться с ней полностью и постепенно перестать запускаться на старый автоматизм. Если связывать это с самореализацией, то НейроСталкинг помогает убрать не внешнюю преграду, а внутренний автопилот, который мешает человеку проявляться.
```

Критерии:

```text
- отвечает прямо;
- признаёт прежнюю неточность без самоунижения;
- использует внутреннюю базу;
- не просит пользователя заново определить термин;
- не даёт практику;
- не делает длинную лекцию;
- не копирует текст базы дословно;
- не утверждает себя как духовный авторитет.
```

## 16. Архитектурная позиция

Этот PRD — не финальная настройка личности бота.

Это первый практический шаг:

```text
бот должен перестать выглядеть так,
будто он не знает собственные внутренние понятия.
```

После этого можно переходить к более глубокой настройке:

```text
Active Frame
Response Planner
Practice Gate v1
Quality Validator
Trace-as-eval
```

Главный принцип:

```text
Сначала — правильно отвечать на известное.
Потом — тонко вести линию.
```
