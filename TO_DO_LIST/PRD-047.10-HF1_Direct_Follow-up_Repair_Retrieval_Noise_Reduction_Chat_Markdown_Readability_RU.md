# PRD-047.10-HF1 — Direct Follow-up Repair + Context-Aware Retrieval Noise Reduction + Chat Markdown Readability

**Проект:** Bot Psychologist / Neo MindBot  
**Язык PRD:** русский  
**Статус:** ready for IDE agent  
**Тип:** hotfix после PRD-047.10  
**Главный фокус:** живой диалог, корректная работа коротких follow-up реплик, умное включение/исключение RAG, читаемый Web Chat  
**Важный принцип:** это не новый слой ограничений. Это очистка противоречий, чтобы Writer мог свободнее и точнее отвечать.

---

## 0. Краткое резюме

`PRD-047.10` успешно добавил `human_like_answer_policy` и сделал Writer заметно живее в `mvp_free_dialogue`.

Живое тестирование владельцем проекта показало, что качество стало лучше, но остался важный класс сбоев:

1. Бот плохо обрабатывает короткие реплики пользователя:
   - `да`
   - `да, предложи`
   - `да конечно`
   - `конечно`
   - `ок`
   - `приведи`
   - `покажи`

   В таких случаях система иногда воспринимает короткий ответ как новый слабый/низкоресурсный запрос, а не как согласие на предыдущее предложение бота.

2. Старые advisory-слои всё ещё могут подавлять прямой ответ:
   - `practice_policy=forbidden`
   - `practice_suppression_active=true`
   - `validate_briefly`
   - `max_sentences`
   - `avoid_multiple_steps`
   - `deepen_mechanism`

   Это приводит к ответам-заглушкам вроде:

   ```text
   Сфокусируюсь на разборе, без практик по умолчанию...
   ```

   вместо того чтобы дать обещанную фразу, пример или простой способ.

3. RAG / KB chunks иногда попадают в Writer prompt как шум, даже когда текущий ход требует прежде всего продолжить локальный диалог.

4. При этом нельзя делать грубое правило “если пользователь сказал да — не давать RAG”.
   Это неверно.

   Правильная логика:

   ```text
   Короткая реплика сама по себе не решает, нужен ли RAG.
   Нужно смотреть на предыдущий ход, активную тему и обещание бота.
   ```

   Например:

   ```text
   Бот: Хочешь объясню это на примере?
   Пользователь: Да.
   ```

   Здесь RAG может быть уместен, если пример относится к концепту из базы знаний.

5. Web Chat отображает ответы как plain text и делает длинные ответы менее читаемыми. Нужно безопасное markdown-отображение: абзацы, списки, жирный, курсив.

### 1.2. Живой failure-кластер

Пример из живого чата:

```text

Пользователь: да, предложи
Бот: Сфокусируюсь на разборе, без практик по умолчанию...
```

Другой пример:

```text
Бот: Хочешь, я предложу одну короткую фразу‑реплику для внутренней пометки?

Бот: Сфокусируюсь на разборе, без практик по умолчанию...
```

```text

**«Сейчас во мне включилась реакция: тело напряглось, мысль появилась, импульс пошёл. Я это вижу — и не обязан сразу действовать».**

Короткая версия: **«Тело напряглось — я замечаю — пауза».**
```

## 2. Главный продуктовый принцип HF1

Система должна быть не набором жёстких правил, а умным диалоговым harness.

Нельзя делать так:

```text
если user_message == "да" -> всегда не использовать RAG
если user_message короткий -> low_resource / regulate
если practice_policy=forbidden -> никогда не давать короткий способ
```

Нужно делать так:

```text
короткая реплика пользователя
→ посмотреть предыдущий assistant turn
→ понять, на что пользователь согласился
→ восстановить ожидаемый intent
→ решить, нужен ли recent context / semantic memory / KB grounding
→ дать прямой живой ответ
```

---

## 3. Цель

Добавить в unified runtime поддержку контекстных коротких follow-up реплик и более умный retrieval-gating, чтобы:

1. `да`, `конечно`, `предложи`, `покажи`, `ок`, `давай` не считались ошибкой или пустым запросом.
2. Бот понимал, что короткая реплика может быть согласием на предыдущее предложение.
3. Writer получал восстановленный смысл текущего хода:
   - user accepted example offer;
   - user accepted phrase offer;
   - user accepted one-step offer;
   - user accepted explanation offer;
   - user accepted practice/micro-observation offer.
4. RAG попадал в Writer prompt только если он уместен **в контексте текущего продолжения**, а не по простому словарному правилу.
5. Writer мог игнорировать RAG, если он не помогает текущему ответу.
6. Web Chat стал читать markdown-ответы как нормальный чат: абзацы, списки, жирный, курсив.

## 6. Required runtime changes

### 6.1. Добавить Dialogue Pragmatics Layer v1 без нового агента

Создать или расширить модуль, например:

```text
bot_psychologist/bot_agent/multiagent/dialogue_pragmatics.py
```

Это не LLM-agent. Это deterministic helper для восстановления смысла коротких follow-up реплик.

Минимальный публичный контракт:

```python
def build_dialogue_pragmatics_v1(
    *,
    user_message: str,
    conversation_context: str,
    previous_assistant_message: str | None = None,
    thread_state: Any | None = None,
    active_frame: dict | None = None,
    dialogue_policy: dict | None = None,
) -> dict:
    ...
```

Возвращаемый payload:

```json
{
  "version": "dialogue_pragmatics_v1",
  "is_short_utterance": true,
  "short_utterance_type": "affirmation",
  "is_contextual_followup": true,
  "followup_relation": "accepts_previous_assistant_offer",
  "previous_assistant_offer_type": "short_phrase|example|one_step|explanation|practice_observation|unknown",
  "inherited_user_intent": "give_short_phrase",
  "inherited_topic": "нейросталкинг / неосталкинг / current active concept",
  "should_answer_directly": true,
  "should_not_ask_confirmation_again": true,
  "allow_concrete_step_or_phrase": true,
  "retrieval_need_hint": "contextual_optional|kb_grounding_useful|memory_only|none",
  "reason": "user_affirmed_previous_offer"
}
```

### 6.2. Short utterance must never be treated as bug by default

Short utterances include:

```text
да
давай
конечно
ок
угу
ага
предложи
покажи
приведи
да, предложи
да конечно
давай пример
покажи пример
```

Acceptance:

- API must not return `422`.
- Streaming must not fail.
- State Analyzer / Planner must not treat short utterance as invalid input.
- If no context exists, bot answers naturally:

  ```text
  Да, могу. Только напомни, о чём именно — и я продолжу.
  ```

- If context exists, bot continues the prior offer directly.

### 6.3. Previous assistant offer extraction

The helper must detect previous assistant offers from recent context, especially phrases like:

```text
Хочешь, я предложу...
Хочешь, чтобы я показал...
Могу привести пример...
Хочешь пример...
Хочешь, я объясню...
Хочешь короткую фразу...
Хочешь один простой способ...
Хочешь, я покажу на твоей ситуации...
```

Offer types:

```text
short_phrase
example
one_step
explanation
practice_observation
summary
application
unknown
```

This should be robust enough for Russian MVP testing. English support may be basic but should not regress.

### 6.4. Direct follow-up must override stale suppression

If:

```json
{
  "is_contextual_followup": true,
  "followup_relation": "accepts_previous_assistant_offer",
  "should_answer_directly": true
}
```

then in `mvp_free_dialogue`:

- `practice_suppression_active` must become advisory, not blocking.
- `practice_policy=forbidden` must not prevent a concrete answer if the bot itself offered a phrase/example/simple method.
- `validate_briefly` must not cause the bot to ask confirmation again.
- `max_sentences` must not force a useless stub.
- `regulate` must not override direct answer unless safety/crisis is active.

Important: this does not mean “always give a practice”.
It means:

```text
If bot offered a concrete phrase/example/simple observation method and user accepted,
the system should fulfill that offer.
```

## 7. Context-aware Retrieval Gating v1

### 7.1. Core principle

Do not use a simplistic rule:

```text
short answer -> no RAG
```

Use contextual decision:

```text
short answer + previous offer type + active topic + knowledge need + available hits
→ retrieval_action
```

### 7.2. Add retrieval decision payload

Create/extend retrieval decision metadata:

```json
{
  "retrieval_decision_version": "contextual_retrieval_gating_v1",
  "retrieval_action": "none|recent_context_only|memory_only|kb_grounding|concept_answer|practice_catalog",
  "rag_candidates_count": 3,
  "rag_included_count": 1,
  "rag_included_reason": "previous_offer_needs_concept_grounding",
  "rag_suppressed_reason": "",
  "writer_can_ignore_rag": true,
  "rag_relevance_to_current_turn": "high|medium|low|unknown",
  "inherited_topic": "нейросталкинг",
  "inherited_offer_type": "example"
}
```

### 7.3. When RAG should be included

RAG may be included when:

1. Previous assistant offer was about:
   - explaining a concept;
   - showing an example of a concept;
   - giving a practice/observation method grounded in KB;
   - distinguishing two concepts;
   - applying a named concept to life.

2. Current short utterance confirms the offer.

3. Active concept or previous turn had a known concept, for example:
   - нейросталкинг;
   - программа несовершенного Я;
   - драйверы;
   - метанаблюдение;
   - работа с триггером;
   - observation / pattern / automatic reaction.

4. Retrieved chunks are relevant enough.

Example:

```text
Assistant: Хочешь объясню это на примере?
User: Да.
```

Expected:

```json
{
  "retrieval_action": "kb_grounding",
  "rag_included_reason": "accepted_example_offer_for_known_concept",
  "writer_can_ignore_rag": true
}
```

### 7.4. When RAG should be excluded or not forwarded

RAG should not be forwarded to Writer when:

1. It is unrelated to the accepted follow-up.
2. It only appears because semantic search matched generic words.
3. Current turn is purely local:
   - “да, предложи фразу”
   - “дай коротко”
   - “повтори”
   - “сформулируй одним предложением”
   and the needed content is already in recent turns.
4. `knowledge_answer_needed=false`, no inherited concept, no previous concept offer, and RAG relevance is low.

Expected:

```json
{
  "retrieval_action": "recent_context_only",
  "rag_candidates_count": 3,
  "rag_included_count": 0,
  "rag_suppressed_reason": "low_relevance_to_contextual_followup",
  "writer_can_ignore_rag": true
}
```

### 7.5. Writer prompt should separate RAG candidates from included RAG

Do not dump all candidate chunks into Writer prompt.

Separate:

```text
RAG DECISION:
retrieval_action=...
rag_included_reason=...
writer_can_ignore_rag=true

RAG INCLUDED FOR WRITER:
- only selected chunks

RAG CANDIDATES DEBUG:
- safe trace only, not necessarily in writer prompt
```

If `rag_included_count=0`, Writer prompt should not include big KB chunks.

### 7.6. Writer must be allowed to ignore RAG

Add to Writer prompt in MVP mode:

```text
Если блок знаний передан, используй его только если он помогает текущему ответу.
Не пересказывай базу знаний ради самой базы.
Если текущий ход требует простого продолжения, опирайся на контекст разговора.
```

## 8. Repair of repeated bot mistake

If user says:

```text
ты снова не ответил
ты опять заглючил
ты не ответил на свой вопрос
ты обещал предложить
ты сам это предложил
```

System must classify it as:

```json
{
  "repair_user_dissatisfaction": true,
  "should_answer_directly": true,
  "should_apologize_briefly": true,
  "should_not_ask_confirmation_again": true
}
```

Expected style:

```text
Да, ты прав — я снова ушёл не туда. Исправляю.

Фраза может быть такой:

**«Сейчас во мне включилась реакция. Тело напряглось, мысль появилась, импульс пошёл. Я это вижу — и беру паузу».**
```

Not acceptable:

```text
Хочешь, чтобы я сейчас дал конкретную фразу?
```

after user already asked for it.

## 9. Web Chat Markdown Readability

### 9.1. Goal

Make bot responses in Web Chat readable like a modern chat interface.

Support:

```markdown
**bold**
*italic*
- bullet lists
1. numbered lists
paragraph breaks
inline `code`
```

### 9.2. Safety

Do not render raw HTML unsafely.

Use a safe markdown renderer and sanitizer, for example:

- `react-markdown`
- `remark-gfm`
- sanitization layer if HTML is enabled
- or keep raw HTML disabled

### 9.3. UI requirements

1. Bot messages should preserve paragraph spacing.
2. Lists should render as actual lists.
3. Bold and italic should work.
4. Long answers should not become a wall of text.
5. Trace/debug panels should remain readable and should not break.
6. User messages may remain plain text or use minimal markdown; main priority is assistant message rendering.
7. Streaming messages should render progressively without causing UI crash.

### 9.4. Acceptance example

Assistant answer:

```markdown
Да. Короткая фраза:

**«Тело напряглось — я замечаю — беру паузу».**

Расширенная версия:

1. **Триггер:** начальник сказал что-то спорное.
2. **Тело:** появилось напряжение.
3. **Пауза:** я вижу реакцию и не обязан сразу действовать.
```

Should display as formatted markdown, not raw markdown text.
