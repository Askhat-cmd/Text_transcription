# PRD-047.10-HF2 — Live Turn Evidence Export + Follow-up Reliability Repair + Markdown Smoke Verification

**Проект:** Bot Psychologist / Neo MindBot  
**Дата:** 2026-06-01  
**Язык PRD:** русский  
**Статус:** ready for IDE-agent implementation  
**Основание:** live-тест после `PRD-047.10-HF1` показал, что формальные dry/direct/live runners зелёные, но реальные multi-turn сценарии всё ещё ломаются на коротких follow-up репликах, offer-fulfillment, stale regulate-stub и визуальном markdown-рендеринге.

---

## 0. Краткое резюме

`PRD-047.10-HF1` технически добавил:

```text
dialogue_pragmatics_v1
contextual_retrieval_gating_v1
writer_can_ignore_rag
safe Web Chat markdown rendering
```

Но живой пользовательский прогон показал, что ключевые проблемы сохраняются:

1. Короткая реплика пользователя типа `да` иногда приводит к `Streaming error: 422 Unprocessable Entity`.
2. Реплики `да`, `да хочу`, `да конечно`, `Да, конечно, дай пример!` не всегда наследуют предыдущее предложение бота.
3. Бот всё ещё может отвечать старой заглушкой:

```text
Сфокусируюсь на разборе, без практик по умолчанию. Здесь ключевой узел в том, как автоматический контроль включает перегруз ещё до действия.
```

4. При просьбе разобрать конкретный пример с начальником бот может не отвечать по сути, а уходить в abstract mechanism / regulation frame.
5. Web Chat markdown визуально не подтверждён в реальном чате: build прошёл, но пользователь не видит качества форматирования как в ChatGPT.
6. Архитектору не хватает полных машиночитаемых live-turn evidence logs в `TO_DO_LIST`, чтобы анализировать не по скриншотам, а по структуре trace/prompt/runtime.

Цель HF2 — не добавлять новые жёсткие guards и не сжимать Writer. Цель — **сделать видимым каждый проблемный ход, воспроизвести реальные сбои, починить именно подтверждённые узлы и доказать, что Writer получает правильную сцену для ответа**.

Главная формула HF2:

```text
Не строить новую клетку для Writer.
Сначала дать полное evidence-полотно по каждому live-turn.
Потом точечно починить прагматику follow-up, 422 и markdown smoke.
```

## 1. Контекст текущего состояния

### 1.1. Что уже есть

После `PRD-047.10` и `PRD-047.10-HF1` в проекте уже есть:

```text
mvp_free_dialogue
human_like_answer_policy
constraint_resolution
dialogue_pragmatics_v1
contextual_retrieval_gating_v1
react-markdown + remark-gfm + skipHtml в Web Chat
```

Система формально остаётся в правильных границах:

```text
один runtime path
один Writer
один Planner
без нового LLM-agent
без broad rollout
без normal-user activation
без KB governance mutation
```

### 1.2. Что показал live-тест пользователя

Реальный диалог выявил сценарии, которые runners HF1 не поймали:

```text
Assistant: Хочешь, я покажу, как это может выглядеть в конкретной твоей ситуации?
User: да
Result: Streaming error: 422 Unprocessable Entity
```

```text
User: Да, конечно, дай пример!
Assistant: Сфокусируюсь на разборе, без практик по умолчанию...
```

```text
User: давай разберем это на примере:
Например на работе, когда начальник не прав, но я зажимаю себя...
Assistant: Сфокусируюсь на разборе, без практик по умолчанию...
```

```text
User: ты не ответил мне на вопрос!
Assistant: частично извиняется, но снова не даёт полноценный direct repair.
```

```text
User: спасибо
Trace: rag_candidates_count=3, rag_included_count=1, retrieval_action=memory_only
```

Последний случай не критичен по ответу, но показывает, что retrieval-gating ещё не полностью чистый: если `retrieval_action=memory_only`, RAG не должен попадать как включённый KB-блок без явного contextual reason.

### 1.3. Важное уточнение архитектора

Нельзя вводить примитивное правило:

```text
если пользователь сказал "да" → RAG не нужен
```

Это неверно.

Правильное поведение:

```text
если пользователь сказал "да" после предложения:
- объяснить концепт;
- привести пример;
- связать с базовой философией;
- дать практический обзор;
- показать применение нейросталкинга;

то RAG может быть нужен и уместен.
```

Но если пользователь сказал `да` после предложения обычного follow-up вопроса или пользователь просто закрывает контакт (`спасибо`, `понял`, `ок`), RAG чаще всего не нужен.

Система должна быть **умной и контекстной**, а не набором запретов.

Не начинать кодинг до создания task list.

### 4.2. Обязательный autonomous reproduce → fix → rerun loop

Агент должен самостоятельно запустить бота и прогнать вопросы, чтобы выявить баги.

Допускается до 3 циклов:

```text
cycle_1: reproduce current failures and save evidence
cycle_2: implement targeted fix and rerun
cycle_3: if still failing, do one bounded additional fix and rerun
```

Если после 3 циклов остаётся failure, агент не должен маскировать его. Нужно:

```text
- сохранить failing evidence;
- указать blocker/warning в implementation report;
- не заявлять full passed.
```

## 7. Follow-up Reliability Repair v2

### 7.1. Проверить, не слишком ли поздно строится dialogue_pragmatics

Сейчас `dialogue_pragmatics` может строиться после State Analyzer / Thread Manager, когда короткая реплика уже ошибочно классифицирована как `new_thread`, `hypo`, `regulate`.

Агент обязан проверить pipeline order.

Если pragmatics строится поздно, добавить лёгкий deterministic pre-thread pragmatics pass:

```text
pre_thread_dialogue_pragmatics
```

Он должен использовать:

```text
current user_message
previous assistant turn
previous user turn
current active thread
recent conversation tail
```

и передавать результат в:

```text
State Analyzer input context / previous_thread hint
Thread Manager update hint
Response Planner
WriterContract
Trace
```

Это не новый агент. Это deterministic helper.

### 7.2. Offer-followup detection

Добавить или усилить определение предыдущего предложения ассистента.

Примеры assistant offers:

```text
Хочешь, я покажу пример?
Хочешь, я покажу, как это может выглядеть в конкретной твоей ситуации?
Хочешь, я предложу одну короткую фразу?
Хочешь, я свяжу это с твоей ситуацией?
Хочешь, я разберу это на примере?
Хочешь, я дам один простой способ?
Хочешь, я объясню подробнее?
```

Типы offer:

```text
example_offer
concrete_situation_offer
phrase_offer
one_step_offer
practice_offer
concept_explanation_offer
application_offer
summary_offer
unknown_offer
```

Affirmative replies:

```text
да
давай
да хочу
да конечно
конечно
ок
ага
предложи
покажи
да, предложи
да, конечно, дай пример
да, покажи
```

Если current user message является affirmative follow-up и previous assistant turn содержит offer, то:

```json

}
```

### 7.3. 422 reliability guard for short messages

Найти источник `Streaming error: 422 Unprocessable Entity` на коротком `да`.

Проверить:

```text
request payload schema
empty/too short message validation
streaming adapter
frontend request body
backend chat route
API key/header handling
session/thread id handling
```

Acceptance:

```text
"да" must never produce 422.
"ок" must never produce 422.
"спасибо" must never produce 422.
"?" must not crash; it may ask for clarification.
emoji-only input must not crash; it may respond briefly.
```

If source is frontend streaming handling, fix frontend.  
If source is backend validation, fix backend.  
If source is adapter payload, fix adapter.  
Do not hide 422 by retrying silently without evidence.

Evidence for every 422 reproduction/fix must be saved.

### 7.4. Repair after user dissatisfaction

If user says:

```text
ты не ответил
ты опять заглючил
ответь на вопрос
ты ушёл не туда
не это спрашивал
```

Then response should:

1. briefly acknowledge miss;
2. infer the last unanswered question from recent turns;
3. answer it directly;
4. not ask “что именно пошло не так?” unless the missing question cannot be inferred.

Example expected answer for concrete boss scenario:

```text
Да, ты прав — я не ответил на пример.

В твоей ситуации нейросталкинг выглядел бы так:

1. Триггер: начальник говорит то, с чем ты не согласен.
2. Тело: появляется напряжение, сжатие, злость.
3. История: «если я возражу, я буду выглядеть неуважительно».
4. Импульс: промолчать и зажать себя.
5. Наблюдение: «сейчас во мне конфликт между правдой и безопасностью в иерархии».

Смысл не в том, чтобы сразу спорить с начальником, а в том, чтобы увидеть цепочку до автоматического зажима.
```

Не обязательно дословно, но смысл должен быть direct repair.

### 7.5. Stale regulate-stub removal / quarantine

Найти источник фразы:

```text
Сфокусируюсь на разборе, без практик по умолчанию. Здесь ключевой узел в том, как автоматический контроль включает перегруз ещё до действия.
```

Проверить:

```text
writer static fallback
response_planner fallback
prompt_constraint_pilot
legacy compliance template
active_line fallback
validator fallback
```

В `mvp_free_dialogue` эта фраза не должна появляться в ответах на:

```text
да
да хочу
да конечно
дай пример
давай разберем на примере
```


Если фраза остаётся допустимой в других безопасных сценариях, ограничить её только там. Не удалять без понимания источника.

## 8. Context-aware retrieval decision v2

### 8.1. Не отключать RAG примитивно

Запрещено правило:

```text
short utterance => no RAG
```

Нужно правило:

```text
short utterance + inherited offer context => retrieval depends on offer type/topic
```

### 8.2. Retrieval action taxonomy

Уточнить `retrieval_action`:

```text
none
recent_context_only
memory_only
kb_optional
kb_grounding
concept_answer
practice_catalog
example_grounding
```

Meaning:

```text
none: не искать и не включать RAG.
recent_context_only: достаточно последних ходов.
memory_only: semantic/user memory, но без KB chunks.
kb_optional: KB chunks можно дать Writer’у как optional context, Writer may ignore.
kb_grounding: KB нужен для понятия/философии/практики.
concept_answer: known concept answer needs grounding.
practice_catalog: пользователь просит практики/обзор.
example_grounding: пользователь просит пример, и KB может дать стиль/рамку, но recent turns важнее.
```

### 8.3. Expected retrieval behavior

| Scenario | Expected retrieval |
|---|---|
| `спасибо` | `none` or `recent_context_only`; `rag_included_count=0` |
| `да` after “хочешь пример?” | `example_grounding` or `recent_context_only`; RAG allowed only if topic/concept relevant |
| `да` after “хочешь фразу?” | usually `recent_context_only`; no large KB unless phrase is concept-specific |
| `да, конечно, дай пример` after concept explanation | `example_grounding` allowed |
| `давай разберем это на примере начальника...` | `recent_context_only` + optional `kb_optional`; recent situation must dominate |
| `что такое нейросталкинг` | `concept_answer` / `kb_grounding` |
| `какие практики в нейросталкинге` | `practice_catalog` |

### 8.4. RAG inclusion must match action

If:

```text
retrieval_action=memory_only
```

then:

```text
rag_included_count should be 0
```

If RAG is included anyway, evidence must contain explicit reason:

```json
{
  "rag_included_reason": "...",
  "why_memory_only_still_included_rag": "..."
}
```

But preferred HF2 fix: do not include KB chunks under `memory_only`.

## 9. Writer prompt / contract requirements

### 9.1. Writer must receive pragmatics as scene, not command noise

Writer prompt should include compact block:

```text
DIALOGUE PRAGMATICS:
- current utterance is affirmative follow-up to previous assistant offer
- offer type: example_offer / phrase_offer / concrete_situation_offer
- answer directly now
- do not ask confirmation again
```

### 9.2. Soft-overrule old constraints when direct follow-up is clear

If:

```text
should_answer_directly=true
should_not_ask_confirmation_again=true
```

then old constraints should be softened:

```text
max_sentences
validate_briefly
regulate by default
practice_policy=forbidden
avoid_multiple_steps
```

But do not remove safety boundaries.

Add to `constraint_resolution`:

```json
{
  "overruled_constraints": [
    "diagnostic_card.max_sentences",
    "writer_move.validate_briefly",
```
