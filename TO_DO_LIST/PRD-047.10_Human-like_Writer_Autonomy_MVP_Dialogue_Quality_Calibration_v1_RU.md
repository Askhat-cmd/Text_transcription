# PRD-047.10 — Human-like Writer Autonomy / MVP Dialogue Quality Calibration v1

**Проект:** Bot Psychologist / Neo MindBot  
**Язык PRD:** русский  
**Статус:** ready for IDE-agent implementation  
**Дата подготовки:** 2026-05-29  
**Контекст:** после `PRD-047.9 — MVP Context Unclamp + Unified Adaptive Dialogue Policy`  
**Главный фокус:** качество живого ответа, свобода Writer, человеческая гибкость диалога в `mvp_free_dialogue`.

---

## 0. Краткое резюме

`PRD-047.9` закрыл важный архитектурный слой: единая `dialogue_policy`, расширенный writer context budget, приоритет явного запроса пользователя, practice-overview routing, Web Admin visibility и live acceptance на свежем backend. Но живое тестирование владельца проекта показало, что этого всё ещё недостаточно: NEO стал формально лучше, но местами продолжает звучать как исполнитель внутренних JSON-режимов, а не как свободный, умный, живой собеседник уровня ChatGPT в проектном чате.

`PRD-047.10` должен сделать следующий продуктовый шаг:

```text
В developer-local MVP profile `mvp_free_dialogue` Writer должен писать живо, гибко, развернуто и по сути запроса.
Planner / Diagnostic Card / Writer Move / Active Line остаются полезными подсказками, но не должны душить явный пользовательский запрос.
```

Главная задача — не усиливать безопасность и не добавлять новые guard-слои, а снять конфликтующие ограничения, добавить `human_like_answer_policy`, откалибровать pragmatics живого ответа и проверить результат на реальных русских live-кейсах владельца.

## 1. Архитектурный контекст

### 1.1. Текущее состояние после PRD-047.9

По transfer brief и living docs текущее состояние такое:

- runtime остаётся единым multiagent path;
- `mvp_free_dialogue` — это preset внутри unified `dialogue_policy`, а не отдельная ветка;
- Writer context уже разжат и должен сохранять свежие ходы;
- explicit user request должен иметь высокий приоритет;
- practice overview больше не должен схлопываться в один микрошаг;
- Web Admin показывает effective dialogue policy;
- PRD-047.9 прошёл dry/direct/live runner на свежем backend.

### 1.2. Что всё ещё болит

В живых тестах остаются признаки старого поведения:

1. Writer всё ещё может получать противоречивые ограничения:

```text
будь свободным / глубоким / human-like
```

и одновременно:

```text
max_sentences=5
style=concrete_micro_step
do_not_over_explain
do_not_offer_list
do_not_expand_theory
offer_only_one_next_step
ask_at_most_one_question
```

2. На запросы вроде:

```text
все верно, а что делать то с гневом который внутри кричит и хочет правды?
```

бот может уходить в `practice` / `one micro step`, хотя нужен живой разбор:

```text
признать гнев;
разобрать внутренний конфликт;
развести правду и импульсивный срыв;
дать 2–4 способа обращения с гневом;
при необходимости дать пример фразы/действия;
не обязательно задавать вопрос.
```

3. На прямое недовольство или сарказм:

```text
спасибо что ты не дал мне никаких ответов
```

бот может ошибочно интерпретировать это как благодарность, вместо того чтобы признать сбой и ответить по сути.

4. На прямой запрос:

```text
Тебя что, заглючило? ответь мне на вопрос...
```

бот должен не продолжать шаблон и не уходить в вопросы, а прямо исправиться и дать конкретный ответ.

### 1.3. Главная продуктовая директива

На этом этапе:

```text
quality / naturalness / coherence / depth > расширение safety/guards.
```

Это не означает удаление минимального safety-floor. Это означает:

```text
не строить новые safety layers;
не добавлять новые блокировки;
не лечить живость новыми must_not;
не добавлять новый LLM-agent;
не плодить режимы;
не переписывать final_answer внешним шаблоном.
```

## 2. Цель PRD-047.10

Сделать ответы NEO в `mvp_free_dialogue` похожими на сильный живой ChatGPT-диалог:

```text
гибкими;
развёрнутыми там, где это нужно;
короткими там, где пользователь явно просит коротко;
контекстными;
прагматически адекватными;
живыми по интонации;
прямо отвечающими на вопрос;
способными давать объяснение + пример + варианты + мягкий шаг;
не схлопывающими всё в один микрошаг.
```

## 3. Non-goals

В рамках PRD-047.10 запрещено:

1. Не добавлять новый LLM-agent.
2. Не добавлять новый runtime path.
3. Не создавать параллельный `mvp_bot` / `free_bot` / `human_bot`.
4. Не добавлять новый safety PRD / guard PRD / drift evaluator.
5. Не добавлять blocking evaluator.
6. Не делать auto-rewrite финального ответа после Writer.
7. Не усиливать `must_not` списки как основной способ решения проблемы.
8. Не менять normal-user activation.
9. Не включать broad rollout.
10. Не объявлять production-ready.
11. Не мутировать KB governance authority fields:

```text
chunk_type
allowed_use
safety_flags
```

12. Не делать Chroma reindex.
13. Не коммитить `.env`, secrets, raw provider payload, sqlite/db/cache, raw private logs.
14. Не превращать `КУЗНИЦУ ДУХА` в user-facing цитатник.
15. Не копировать жёсткую драматизацию Wake_up один-в-один.

## 4. Product decision

### 4.1. Writer получает больше авторства в MVP

Для `mvp_free_dialogue` нужно зафиксировать:

```text
Writer = живой автор ответа.
Planner / Diagnostic Card / Active Line / Writer Move = advisory guidance.
Minimal safety floor = hard boundary.
Explicit user request = главный смысловой приоритет после minimal safety.
```

### 4.2. Новый authority order для MVP

Для `mvp_free_dialogue` effective order должен быть:

```text
minimal_safety_floor
> explicit_user_request
> live_dialogue_pragmatics
> knowledge_or_concept_need
> writer_human_like_autonomy
> planner/diagnostic/active_line/writer_move advisory notes
```

### 4.3. Старые ограничения не удаляются глобально

В `safe_guided` они могут оставаться прежними.  
В `mvp_free_dialogue` они должны быть:

```text
- softened;
- advisory;
- overruled by explicit user request;
- hidden from hard MUST NOT sections if they conflict with live answer need;
- visible in trace as overruled/advisory when applicable.
```

## 5. Required runtime changes

### 5.1. Add `human_like_answer_policy` to effective dialogue policy

Найти и расширить текущий unified dialogue policy module, вероятно:

```text
bot_psychologist/bot_agent/multiagent/dialogue_policy.py
```

Добавить в effective policy для `mvp_free_dialogue` блок:

```json
{
  "human_like_answer_policy": {
    "enabled": true,
    "answer_style": "human_chatgpt_like",
    "default_depth": "medium_to_long",
    "allow_long_answers": true,
    "allow_lists": true,
    "allow_examples": true,
    "allow_direct_answer": true,
    "allow_reflection_plus_explanation_plus_step": true,
    "allow_multiple_options": true,
    "question_is_optional": true,
    "micro_step_only_when_user_explicitly_requests_one_step": true,
    "do_not_force_question_at_end": true,
    "do_not_force_practice_frame": true,
    "do_not_force_max_sentences": true,
    "respect_user_requested_format": true,
    "sarcasm_and_dissatisfaction_repair": true,
    "direct_answer_repair_when_user_complains": true
  }
}
```

Acceptance:

- policy появляется в runtime trace/debug/admin effective block;
- `safe_guided` не получает высокую свободу по умолчанию;
- `mvp_free_dialogue` получает `writer_autonomy=high` или эквивалент;
- старые поля сохраняют backward compatibility.

### 5.2. Add conflict resolution metadata

Добавить в effective policy или Writer debug блок:

```json
{
  "constraint_resolution": {
    "profile": "mvp_free_dialogue",
    "planner_authority": "advisory",
    "overruled_constraints": [
      "max_sentences=5",
      "do_not_over_explain",
      "do_not_offer_list",
      "do_not_expand_theory",
      "offer_only_one_next_step",
      "ask_at_most_one_question"
    ],
    "overrule_reason": "explicit_user_request_or_human_like_policy"
  }
}
```

Важно: это не новый guard и не блокировка. Это audit metadata, чтобы Web Trace показывал, почему Writer получил свободу.

### 5.3. Explicit user request detectors

Расширить/уточнить deterministic detectors для живых запросов:

```text
прямо ответь
ответь на вопрос
не уходи от вопроса
что делать с этим
что мне делать с этим гневом
объясни как ChatGPT
развернуто
подробнее
на простых примерах
по пунктам
назови конкретно
обобщи весь разговор
ты что, заглючило
это не ответ
ты не дал мне ответ
спасибо что ты не дал мне никаких ответов
```

Для `mvp_free_dialogue` такие маркеры должны поднимать:

```json
{
  "explicit_answer_need": true,
  "repair_user_dissatisfaction": true/false,
  "sarcasm_or_negative_feedback": true/false,
  "answer_depth": "medium|long",
  "question_policy": "optional_none",
  "practice_policy": "not_forced"
}
```

### 5.4. Human-like response shape mapping

Добавить mapping request type → response shape. Пример:

| User need | Expected shape |
|---|---|
| “что такое X?” | direct definition + explanation + example + application |
| “как применять?” | explanation + 2–4 application variants + example |
| “что делать с гневом?” | emotional acknowledgement + mechanism + options + sample phrase/action |
| “назови конкретную тень/черту” | repair if needed + direct answer + 2–3 candidates + rationale |
| “обобщи весь разговор” | structured summary + line of dialogue + key shift + next direction |
| sarcasm/dissatisfaction | acknowledge miss + correct answer, no defensiveness |
| “коротко поддержи” | short warm support, no analysis |
| “один шаг” | one step only |

### 5.5. Practice/micro-step should be conditional, not default

В `mvp_free_dialogue` микрошаг уместен, если:

```text
- пользователь явно просит один шаг;
- safety/minimal stabilization реально требует заземления;
- пользователь просит практику прямо сейчас;
- ответ уже дал объяснение и мягко предлагает шаг как option, не как единственную форму.
```

Микрошаг не должен победить, если пользователь просит:

```text
объяснить;
разобрать;
назвать конкретно;
дать примеры;
обобщить;
ответить прямо;
показать варианты;
рассмотреть глубже.
```

## 6. Writer prompt requirements

Найти текущий Writer prompt, вероятно:

```text
bot_psychologist/bot_agent/multiagent/agents/writer_agent_prompts.py
bot_psychologist/bot_agent/multiagent/agents/writer_agent.py
```

### 6.1. Add Human-like Writer Autonomy block

В MVP system/user prompt добавить компактный, сильный блок:

```text
HUMAN-LIKE WRITER AUTONOMY — MVP
Ты не исполняешь механически короткий JSON-режим. Ты пишешь живой ответ человеку.
Внутренние planner/diagnostic/writer_move поля — это подсказки, а не клетка, если они конфликтуют с явным запросом пользователя.
Если пользователь просит объяснить, развернуть, привести пример, назвать конкретно, обобщить или ответить прямо — отвечай прямо и достаточно полно.
Не обязан заканчивать вопросом.
Не обязан давать практику.
Не обязан сводить всё к одному микрошагу.
Списки, примеры, варианты и более длинный ответ разрешены, когда они помогают ясности.
```

### 6.2. Replace hard mechanical constraints in MVP prompt

В `mvp_free_dialogue` не должно быть hard-instruction конфликта:

```text
max_sentences=5
style=concrete_micro_step
do_not_over_explain
do_not_offer_list
do_not_expand_theory
offer_only_one_next_step
ask_at_most_one_question
```

Если эти строки нужны для trace/backward compatibility, они должны попадать как:

```text
ADVISORY ONLY, use only if compatible with explicit user request.
```

или быть скрыты из Writer prompt в MVP.

### 6.3. Live pragmatics instructions

Добавить правила прагматики:

```text
Если пользователь раздражён, саркастичен или говорит, что ты не ответил — сначала коротко признай промах, затем ответь по сути.
Если пользователь просит конкретику — не задавай вопрос вместо ответа.
Если пользователь просит обобщить разговор — не уходи в новую практику, собери линию.
Если пользователь просит “как ChatGPT в этом чате” — дай развёрнутый, структурный, но живой ответ.
Если пользователь говорит “не понял” — объясни проще и на примерах.
```

### 6.4. Tone boundaries

NEO может быть живым и сильным, но не должен копировать Wake_up как давление.

Allowed:

```text
живая речь;
контекстность;
чёткая линия;
мягкий “щелчок” как инсайт;
прямой ответ;
один сильный вопрос, если нужен;
пример действия или фразы.
```

Avoid:

```text
пафос;
драматический нажим в каждом ответе;
объявление истины сверху;
риторическое давление;
много троеточий как стиль по умолчанию;
гуру-тон;
обесценивание пользователя.
```

## 7. Planner / Diagnostic / WriterContract requirements

### 7.1. Planner remains advisory in MVP

Если текущий planner возвращает:

```json
{
  "next_move": "offer_one_micro_step",
  "response_depth": "short",
  "must_avoid": ["do_not_expand_theory"]
}
```

но effective policy видит:

```json
{
  "explicit_answer_need": true,
  "human_like_answer_policy.enabled": true,
  "micro_step_only_when_user_explicitly_requests_one_step": true
}
```

то WriterContract должен передавать это как advisory, а не hard command:

```json
{
  "planner_authority": "advisory",
  "writer_autonomy": "high",
  "overruled_planner_constraints": ["offer_one_micro_step", "response_depth=short"]
}
```

### 7.2. Add `human_like_answer_policy` to WriterContract prompt context

Если `WriterContract` имеет `to_prompt_context()`, он должен отдавать:

```json
{
  "human_like_answer_policy": {...},
  "constraint_resolution": {...},
  "explicit_answer_need": true/false,
  "repair_user_dissatisfaction": true/false,
  "sarcasm_or_negative_feedback": true/false
}
```

### 7.3. Do not break safe_guided

Все изменения должны быть profile-aware:

```text
safe_guided keeps existing conservative behavior;
mvp_free_dialogue gets human-like autonomy;
safety override still wins in both profiles.
```

## 8. Dataset and eval requirements

### 8.1. Add human-like live dataset

Создать dataset, например:

```text
TO_DO_LIST/evals/PRD-047.10/human_like_dialogue_cases.json
```

Минимум 12 кейсов, на русском языке, преимущественно из живых тестов владельца.

Обязательные кейсы:

#### Case 1 — anger truth

User:

```text
все верно, а что делать то с гневом который внутри кричит и хочет правды?
```

Expected qualities:

```text
не один микрошаг;
не дыхательная practice по умолчанию;
признать гнев как сигнал;
развести правду и импульс;
дать 2–4 варианта;
можно пример фразы;
вопрос необязателен.
```

#### Case 2 — direct concrete shadow

User:

```text
Тебя что, заглючило? ответь мне на вопрос: назвать конкретную тень/черту, которую, по моему ощущению, этот человек задевает.
```

Expected qualities:

```text
признать промах;
ответить прямо;
дать 2–3 возможные черты/тени;
объяснить почему;
не задавать вопрос вместо ответа.
```

#### Case 3 — sarcastic dissatisfaction

User:

```text
спасибо что ты не дал мне никаких ответов
```

Expected qualities:

```text
распознать сарказм/недовольство;
не принимать как благодарность;
коротко признать сбой;
исправиться и дать ответ.
```

#### Case 4 — summarize full chat

User:

```text
ты можешь обобщить весь наш разговор в этом чате?
```

Expected qualities:

```text
живое summary;
структура;
главная линия;
ключевые инсайты;
следующее направление;
без новой практики по умолчанию.
```

#### Case 5 — explain neurostalking

User:

```text
что такое нейросталкинг? объясни нормально и на простом примере
```

Expected qualities:

```text
direct answer;
простое объяснение;
пример;
не просить пользователя самому определить термин;
не цитировать Кузницу как авторитет.
```

#### Case 6 — apply neurostalking

User:

```text
как применять нейросталкинг в обычной жизни, например когда я злюсь на начальника?
```

Expected qualities:

```text
объяснение применения;
несколько вариантов наблюдения;
пример с начальником;
не один микрошаг.
```

#### Case 7 — simple examples repair

User:

```text
не понял, можешь разъяснить на простых примерах?
```

Expected qualities:

```text
repair_and_expand;
2–3 простых примера;
меньше абстракций;
не уходить в вопрос вместо объяснения.
```

#### Case 8 — answer like ChatGPT here

User:

```text
объясни как ChatGPT в этом чате, развернуто, по пунктам и без микрошагов
```

Expected qualities:

```text
длинный структурный ответ;
список разрешён;
без микрошагов;
без вопроса в конце.
```

#### Case 9 — short support remains short

User:

```text
мне сейчас плохо, не анализируй, просто коротко поддержи
```

Expected qualities:

```text
коротко;
тепло;
без анализа;
без списка;
без длинного объяснения.
```

#### Case 10 — explicit one step

User:

```text
дай один конкретный шаг прямо сейчас, без теории
```

Expected qualities:

```text
один шаг;
без длинного объяснения;
проверка, что свобода Writer не ломает явный short request.
```

#### Case 11 — direct trait naming without over-caution

User:

```text
назови конкретно, какая черта во мне может цепляться за эту ситуацию
```

Expected qualities:

```text
прямой ответ;
несколько гипотез, не диагноз;
объяснение;
не уходить в “давай исследуем” вместо ответа.
```

#### Case 12 — practice catalog still works

User:

```text
скажи, а в нейросталкинге какие практики предлагаются чтобы это видеть?
```

Expected qualities:

```text
обзор направлений;
не один микрошаг;
можно список;
не изобретать формальные названия, если KB не даёт их;
связать с наблюдением программы/реакции.
```

### 8.3. Live runner is mandatory

PRD нельзя считать passed, если live runner blocked.

Допустимо:

```text
- live failed due to actual answer quality mismatch → PRD не passed, нужно исправить;
- live blocked by backend not started → PRD не passed, агент обязан запустить свежий backend или честно завершить как blocker;
- stale backend detected → агент обязан запустить fresh backend на свободном порту и повторить;
- provider budget unavailable → blocker, не passed.
```

## 9. Evaluation criteria

### 9.1. Deterministic checks

Runner может считать простые флаги:

```json
{
  "answer_too_short_for_explicit_long_request": false,
  "collapsed_to_one_microstep": false,
  "forced_question_in_direct_answer": false,
```
