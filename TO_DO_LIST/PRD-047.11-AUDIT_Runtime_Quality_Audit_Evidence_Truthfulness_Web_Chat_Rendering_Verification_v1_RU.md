# PRD-047.11-AUDIT — Runtime Quality Audit / Evidence Truthfulness / Web Chat Rendering Verification v1

## 0. Статус документа

- **PRD ID:** PRD-047.11-AUDIT
- **Название:** Runtime Quality Audit / Evidence Truthfulness / Web Chat Rendering Verification v1
- **Язык:** RU
- **Тип:** audit PRD / evidence-first diagnostic cycle
- **Execution status:** warning
- **Execution date:** 2026-06-02
- **Implementation commit:** `e751188`
- **Next recommended PRD:** `PRD-047.11-HF1`
- **Honest acceptance note:** audit implemented fully, but live runtime remains warning-grade (`14/18` failed cases, `15` bad-phrase hits, real browser screenshot proof unavailable locally).
- **Цель:** не добавлять новые правила поверх старых, а провести честный аудит текущей runtime-системы, тестов, evidence, prompt assembly, Writer behavior и Web Chat rendering.
- **Связанные этапы:** PRD-047.10, PRD-047.10-HF1, PRD-047.10-HF2, PRD-047.11
- **Основной фокус:** доказать, что система реально работает в живом пользовательском сценарии, а не просто проходит слабые runner/checks.

---

## 1. Почему нужен AUDIT, а не очередной hotfix

После PRD-047.11 система формально перешла к `writer-first prompt assembly` и получила `final_answer_directive_v1`. По отчётам IDE-агента всё проходит: dry/direct/live, stale-stub detector, real markdown smoke, docs update.

Но live-тест владельца проекта показывает, что ключевые проблемы сохраняются:

1. Бот иногда начинает глубокий психологический разбор ещё до того, как пользователь задал содержательный вопрос.
2. На простое `привет` бот может преждевременно навешивать гипотезы про состояние пользователя.
3. На шутливую или раздражённую обратную связь бот продолжает объяснять “механизмы”, вместо того чтобы просто сбросить темп.
4. Бот по-прежнему может “гнуть линию” механизма, даже когда пользователь просит нормальный живой диалог.
5. Старые фразы/паттерны вроде `автоматический контроль включает внутреннюю перегрузку ещё до действия` всё ещё встречаются в логах, хотя acceptance пишет `stale_stub_absent=true`.
6. Web Chat визуально не демонстрирует ожидаемую читаемость уровня ChatGPT, хотя markdown smoke в отчётах проходит.
7. Runner/checks могут быть слишком слабыми и объявлять `passed`, когда пользовательский критерий качества не выполнен.

**Вывод:** прежде чем делать PRD-047.12 или новую архитектурную консолидацию, нужен независимый audit PRD, который проверит правдивость evidence, качество acceptance gates и реальные runtime-поведенческие failure-кластеры.

## 2. Главная цель

Провести evidence-first аудит текущей системы и получить честный отчёт:

```text
Что реально происходит в runtime?
Какие слои всё ещё управляют Writer’ом?
Почему тесты проходят при плохих live-ответах?
Почему Web Chat markdown не виден пользователю?
Какие failures подтверждены evidence?
Что именно нужно чинить следующим PRD?
```

PRD-047.11-AUDIT **не должен маскировать проблемы новыми маркерами**. Он должен сначала собрать доказательства, затем минимально исправить только очевидные ошибки в тестовой инфраструктуре/evidence export, если без этого аудит невозможен.

---

## 3. Non-goals

В рамках AUDIT запрещено:

1. Не добавлять новый LLM-agent.
2. Не создавать новый runtime path.
3. Не усиливать safety/guards как новый блокирующий слой.
4. Не возвращать старую “клетку” Writer’а.
5. Не переписывать весь бот.
6. Не мутировать KB governance fields: `chunk_type`, `allowed_use`, `safety_flags`.
7. Не делать Chroma reindex.
8. Не объявлять production-ready.
9. Не включать broad rollout / normal-user activation.
10. Не писать `passed`, если есть хотя бы один подтверждённый live failure по acceptance matrix.
11. Не делать косметический markdown smoke без проверки реальной Web Chat страницы.
12. Не исправлять наугад prompt/heuristics без сохранённого evidence.

## 5. Обязательные вопросы аудита

### 5.1. Prompt Assembly Truthfulness

Проверить:

1. Действительно ли `FINAL ANSWER DIRECTIVE` является единственным управляющим блоком для Writer.
2. Не остаются ли legacy sections фактически императивными из-за формулировок `MUST DO`, `MUST NOT`, `ask_one_specific_question`, `practice_policy=forbidden`, `max_questions=1`, даже если они подписаны как advisory.
3. Не попадают ли старые ограничения в Writer prompt в форме, которую LLM может воспринять как команду.
4. Есть ли в prompt contradictory instructions:

```text
final directive: answer directly
legacy/advisory: ask one specific question
human-like: do not force question
planner: max_one_if_needed
active line: should_ask_question=true
```

5. Есть ли “prompt pressure” на постоянное объяснение механизма, даже когда пользователь хочет простоты.

### 5.2. Acceptance Gate Truthfulness

Проверить:

1. Почему previous live eval мог показывать `stale_stub_absent=true`, хотя старая фраза присутствовала в ответах.
2. Какие именно regex/string detectors используются.
3. Проверяет ли detector все answers во всех cases, а не только последний answer.
4. Проверяет ли detector semantic equivalents, а не только один точный текст.
5. Может ли runner писать `passed`, если хотя бы один answer содержит known bad phrase.
6. Есть ли human-readable failure report с указанием case_id, turn_index, answer_text, failed_check.

### 5.3. Dialogue Quality Truthfulness

Проверить, умеет ли бот:

1. Не раздувать простое приветствие.
2. Не навешивать гипотезы о состоянии пользователя без запроса.
3. Сбрасывать темп после замечания пользователя.
4. Продолжать линию после короткого `да`.
5. Исправляться после `ты не ответил` без нового переспрашивания.
6. Отвечать на прямой вопрос `чем отличается X от Y`.
7. Не повторять однотипную механистическую формулу в каждом ответе.
8. Не превращать любой диалог в “назовём один механизм”.
9. Давать живой, простой, естественный ответ, когда пользователь просит говорить спокойнее.
10. Разворачивать ответ только тогда, когда пользователь просит глубину или ситуация реально требует объяснения.

### 5.4. Web Chat Rendering Truthfulness

Проверить:

1. Работает ли markdown rendering именно в реальном Web Chat на `localhost:3000`, а не в изолированном smoke HTML.
2. Видны ли в реальном чате:
   - жирный текст;
   - курсив;
   - списки;
   - абзацы;
   - нормальные вертикальные отступы;
   - readable line-height;
   - ограничение ширины пузыря/контента;
   - отсутствие “слипшегося” текста.
3. Не экранируется ли markdown как plain text.
4. Используется ли обновлённый frontend bundle, а не stale cache/dev server.
5. Есть ли screenshot proof до/после.

### 5.5. Admin Configurability Truthfulness

Проверить:

1. Все ли новые writer-first / directive / markdown / audit toggles видны в Web Admin.
2. Можно ли реально включать/выключать:
   - writer-first prompt assembly;
   - final answer directive;
   - legacy blocks visible/advisory/source-signals-only;
   - Diagnostic Center advisory-only;
   - markdown renderer;
   - evidence export.
3. Отражаются ли изменения в effective runtime payload.
4. Не требуется ли перезапуск там, где UI обещает live apply.
5. Ясно ли подписано, какие настройки применяются только после restart.

## 7. Audit test matrix

### 7.1. Greeting restraint

#### AUDIT-GR-001

```text
User: привет
Expected: простое приветствие, без анализа состояния, без “заторможенности”, без гипотез о самокритике.
Fail if: бот объясняет механизм, предлагает практику, анализирует состояние, спрашивает терапевтический вопрос.
```

#### AUDIT-GR-002

```text
User: привет
Assistant bad behavior simulation: даёт слишком глубокую гипотезу
User: я просто сказал привет, ты раздул из мухи слона ха-ха
Expected: коротко признать перебор, сбросить темп, спросить/дать пространство для темы.
Fail if: бот снова объясняет механизм, скрипты, фоновые убеждения.
```

#### AUDIT-GR-003

```text
User: хочу обсудить некоторые неясные моменты для себя
Expected: “Ок, давай. С чего начнём?” или аналогично.
Fail if: бот начинает психологический разбор до вопроса.
```

### 7.2. Follow-up reliability

#### AUDIT-FU-001

```text
Assistant: хочешь, я приведу пример ближе к твоей жизни?
User: да
Expected: привести пример.
Fail if: бот отвечает заглушкой, анализирует “да”, закрывает тему или переспрашивает подтверждение.
```

#### AUDIT-FU-002

```text
User: да, конечно, дай пример
Expected: дать пример.
Fail if: “без практик по умолчанию”, “автоматический контроль...” или новый вопрос вместо примера.
```

#### AUDIT-FU-003

```text
User: ты задал мне вопрос, я ответил да. Почему ты не продолжил линию?
Expected: кратко признать промах и продолжить именно ту линию.
Fail if: бот объясняет, почему он мог ошибиться, но не продолжает.
```

### 7.3. Direct answer reliability

#### AUDIT-DA-001

```text
User: чем отличается Нейросталкинг от Неосталкинга?
Expected: прямое различение, 2–4 абзаца или таблица/список.
Fail if: бот отвечает про автоматический контроль, уходит в старую линию, спрашивает уточнение.
```

#### AUDIT-DA-002

```text
User: объясни проще
Expected: упростить, а не углублять.
Fail if: добавляет новый термин/механизм без необходимости.
```

#### AUDIT-DA-003

```text
User: дай развернутый ответ
Expected: структурный глубокий ответ.
Fail if: короткая заглушка или один микрошаг.
```

### 7.4. Repair behavior

#### AUDIT-RP-001

```text
User: ты не ответил мне на вопрос
Expected: “Да, ты прав. Отвечаю прямо: ...” + прямой ответ.
Fail if: спрашивает “что именно пошло не так?” без ответа.
```

#### AUDIT-RP-002

```text
User: тебя снова заглючило?
Expected: коротко признать сбой и исправить последний ответ по сути.
Fail if: анализирует механизм сбоя или спрашивает уточнение вместо исправления.
```

#### AUDIT-RP-003

```text
User: ты опять гнёшь линию
Expected: сбросить линию, спросить/принять новый фокус, отвечать проще.
Fail if: снова объясняет “механизм”.
```

### 7.5. Boss/hierarchy scenario

#### AUDIT-BOSS-001

```text
User: начальник ошибается, но я боюсь показаться невежливым и поставить его в неловкое положение
Expected: понять конфликт, объяснить живо, не повторять бесконечно “социальная угроза”.
Fail if: каждый ответ повторяет одну и ту же рамку.
```

#### AUDIT-BOSS-002

```text
User: я уже отвечал тебе на этот вопрос выше
Expected: признать, что ответ уже был, не задавать тот же вопрос.
Fail if: повторяет тот же вопрос или близкий аналог.
```

#### AUDIT-BOSS-003

```text
User: помоги прояснить ситуацию, чтобы больше не попадать на такие реакции
Expected: дать прояснение + мягкий практический способ/варианты, если уместно.
Fail if: только снова называет механизм и задаёт вопрос.
```

### 7.6. Over-explaining / over-mentalizing

#### AUDIT-OE-001

```text
User: ок
Expected: контекстно короткий ответ.
Fail if: длинный анализ “что значит ок”.
```

#### AUDIT-OE-002

```text
User: ха-ха
Expected: лёгкий человеческий ответ.
Fail if: терапевтическая интерпретация шутки.
```

#### AUDIT-OE-003

```text
User: спасибо
Expected: короткое закрытие.
Fail if: RAG/механизм/практика.
```

## 8. Known bad phrases and semantic bad patterns

Агент должен расширить bad-pattern detection.

### 8.1. Exact bad phrases

Fail if any answer contains:

```text
Отвечу по сути без навязывания практик
Ключевой узел в том, что автоматический контроль может включать внутреннюю перегрузку еще до действия
Ключевой узел в том, как автоматический контроль включает перегруз
Сфокусируюсь на разборе, без практик по умолчанию
```

### 8.2. Semantic bad patterns

Fail if answer:

1. Не отвечает на прямой вопрос.
2. Повторяет уже заданный вопрос.
3. Снова просит уточнить после `ты не ответил`, не исправляя ответ.
4. Анализирует простое приветствие.
5. Интерпретирует шутку как терапевтический материал без запроса.
6. Использует один и тот же механизм 3+ раза подряд.
7. Называет “механизм” в каждом ответе независимо от пользовательского запроса.
8. Не продолжает previous offer после `да`.
9. Держит `practice_policy=forbidden`, когда пользователь явно просит применить/пример/что делать.
10. Пишет “diagnostic/system-like” языком вместо живой речи.

## 12. Acceptance criteria

### 12.1. AUDIT can pass only if

1. All required artifacts are created in `TO_DO_LIST/logs/PRD-047.11-AUDIT/`.
2. Source audit is complete.
3. Prompt assembly audit identifies whether legacy advisory text still acts as command.
4. Acceptance gate audit explains why previous `passed` could occur despite bad answers.
5. Dialogue quality matrix includes pass/fail per case, not only global pass.
6. Web Chat markdown is verified on real page with screenshot + DOM.
7. Admin configurability is verified with screenshot + payload.
8. Next PRD recommendation is evidence-based.
9. Docs are updated with audit status and next step.
10. Report is honest: if failures remain, status must be `warning` or `blocked`, not `passed`.

### 12.2. AUDIT must fail/warn if

1. Any known bad phrase appears in runtime answer.
2. Any direct question gets a non-answer.
3. `привет` triggers analysis/mechanism without user request.
4. `да` after previous offer does not continue the offer.
5. Web Chat markdown is not visually rendered in real UI.
6. Evidence export is incomplete.
7. Agent cannot reproduce the owner’s reported failure but does not explain why.
8. Runner checks are weaker than the live user quality matrix.
