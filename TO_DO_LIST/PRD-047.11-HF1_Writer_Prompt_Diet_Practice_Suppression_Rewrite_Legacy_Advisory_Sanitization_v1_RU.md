## 1. Краткое резюме

После PRD-047.11-AUDIT подтверждено, что `FINAL ANSWER DIRECTIVE` существует, но Writer всё ещё испытывает давление от legacy advisory sections. Даже когда они подписаны как `advisory`, в prompt остаются формулировки, которые LLM может читать как команды:

```text
MUST DO
MUST NOT
ask_one_specific_question
practice_policy=forbidden
practice_suppression_active=true
назвать механизм
связать с active_line
clarify_one_point
```

Из-за этого Writer продолжает:

```text
- превращать простые реплики в психологический разбор;
- отвечать через повторяющийся mechanism-style frame;
- использовать старые заглушки при запрете practice;
- “гнуть линию” механизма даже там, где нужен обычный живой ответ.
```

Этот PRD НЕ должен решать проблему через новые runtime blocklist’ы, quality modes, маркеры greeting/repair/follow-up и т. п.

**Главный принцип PRD:**

```text
Не добавлять новые клетки.
Убрать из Writer prompt то, что заставляет Writer писать как клетка.
```

## 2. Контекст проблемы

### 2.1. Что уже было сделано

К текущему моменту система получила:

```text
- mvp_free_dialogue;
- human_like_answer_policy;
- final_answer_directive_v1;
- writer-first prompt assembly;
- live evidence export;
- audit runner;
- quality matrix;
- trace/admin visibility.
```

Но аудит показал, что Writer всё ещё видит слишком много диагностического и директивного шума. Формально legacy sections стали advisory, но фактически их формулировки остаются похожими на команды.

### 2.2. Главный сбой

`practice_forbidden` / `practice_suppression_active` сейчас может восприниматься как запрет на нормальный содержательный ответ.

Фактическая неправильная цепочка:

```text
practice запрещена
→ не давай практику
→ не давай шаг
→ дай безопасную заглушку
→ “Отвечу по сути без навязывания практик...”
```

Правильная цепочка:

```text
practice запрещена
→ не предлагай упражнение / технику / пошаговую практику
→ но дай нормальный ответ:
   объяснение, пример, отражение, прояснение или продолжение линии.
```

## 3. Цель PRD

Сделать Writer prompt проще, чище и устойчивее:

```text
Writer получает короткую сцену + задачу ответа + hard boundaries.
Diagnostic/Planner/Active Line остаются в trace/admin, но не давят на текст ответа.
Practice suppression перестаёт быть источником заглушки.
Старые шаблонные фразы физически удаляются из runtime templates/fallbacks.
Evaluator использует bad-phrase checks только как датчик возврата бага, не как способ управлять Writer’ом.
```

## 5. Архитектурное решение

### 5.1. Writer Prompt Diet

Сократить Writer prompt в `mvp_free_dialogue` так, чтобы Writer видел:

```text
1. System identity / minimal safety baseline.
2. Current user message.
3. Final Answer Directive или его компактную форму.
4. Recent context / relevant memory / optional RAG.
5. Neutral advisory summary, если нужен.
6. Инструкцию: ответь живо, естественно, по сути.
```

Writer НЕ должен видеть legacy sections в старом командном виде:

```text
WRITER MOVE MUST DO
WRITER MOVE MUST NOT
MUST DO
MUST NOT
ask_one_specific_question
max_sentences
max_questions
practice_suppression_active=true
practice_policy=forbidden
response_planner.must_include
response_planner.must_avoid
active_line.next_meaningful_move как команду
```

Эти данные можно сохранять в `trace`, `live_turn_evidence`, `admin payload`, но не в raw writer prompt как директивные блоки.

### 5.2. Legacy Advisory Sanitizer

Создать или доработать компактный sanitizer, который превращает legacy signals в нейтральное резюме для Writer.

Пример НЕПРАВИЛЬНО:

```text
WRITER MOVE MUST DO:
reflect_one_key_point, ask_one_specific_question

MUST NOT:
do_not_over_explain, ask_at_most_one_question

ACTIVE LINE:
practice_suppression_active=true
next_meaningful_move=назвать механизм
```

Пример ПРАВИЛЬНО:

```text
ADVISORY CONTEXT SUMMARY:
Пользователь продолжает обсуждать текущую ситуацию.
Полезно ответить спокойно и по сути, не перегружая анализом.
Если уместно — можно прояснить один смысловой узел.
```

Важно: summary не должно содержать слов `must`, `forbidden`, `do_not`, `ask_one_specific_question`, `practice_suppression_active`, `max_questions`, `max_sentences`.

### 5.3. Practice Suppression Rewrite

Переписать семантику practice suppression.

Новая формулировка для Writer:

```text
Если practice сейчас неуместна или не запрошена, не предлагай упражнение, технику или пошаговую практику.
Но это не запрет на содержательный ответ.
Вместо практики дай нормальное объяснение, пример, отражение, прояснение или продолжение линии разговора.
```

`practice_policy=forbidden` / `practice_suppression_active=true` НЕ должны попадать в prompt как raw text.

В trace можно оставить машинные поля:

```json
{
  "practice_suppression_active": true,
  "writer_visible_practice_instruction": "no_exercise_but_answer_normally"
}
```

### 5.4. Remove stale templates/fallbacks

Найти и убрать из runtime/fallback/prompt/templates все старые фразы и их прямые template-source:

```text
Отвечу по сути без навязывания практик
Отвечу прямо по сути: автоматический контроль часто включает внутреннюю перегрузку еще до действия
Ключевой узел в том, что автоматический контроль...
автоматический контроль может включать внутреннюю перегрузку еще до действия
```

Если такие фразы нужны в тестах как bad patterns — оставить только в test/eval fixtures с явным названием `bad_phrase_patterns` или аналогичным.

## 7. Требования к реализации

### 7.1. Prompt assembly

Проверить и доработать файлы фактического prompt assembly. Ожидаемые кандидаты, но агент обязан сверить по source audit:

```text
bot_psychologist/bot_agent/multiagent/agents/writer_agent_prompts.py
bot_psychologist/bot_agent/multiagent/contracts/writer_contract.py
bot_psychologist/bot_agent/multiagent/final_answer_directive.py
bot_psychologist/bot_agent/multiagent/orchestrator.py
```

Требования:

```text
- FINAL ANSWER DIRECTIVE остаётся главным управляющим блоком.
- Legacy blocks не выводятся Writer’у в старом raw command формате.
- Если legacy data нужны, они выводятся только как нейтральный advisory summary.
- Prompt содержит явное объяснение новой practice suppression semantics.
- Prompt не содержит stale bad phrases.
```

### 7.2. Sanitizer

Реализовать один простой helper, не сложную новую систему.

Возможное имя:

```text
bot_psychologist/bot_agent/multiagent/legacy_advisory_sanitizer.py
```

Или встроить в существующий prompt assembly, если отдельный файл избыточен. Решение должно быть обосновано в report.

Контракт helper’а:

```python
sanitize_legacy_advisory_for_writer(source_signals: dict) -> dict
```

Ожидаемый результат:

```json
{
  "writer_visible_summary": "...neutral Russian summary...",
  "suppressed_command_terms": ["MUST DO", "ask_one_specific_question", "practice_suppression_active"],
  "raw_legacy_visible_to_writer": false
}
```

### 7.3. Trace/admin preservation

Важно: данные не удаляются из observability.

В trace/admin должны остаться:

```text
- raw diagnostic_card / response_planner / active_line / writer_move, если они уже были в debug;
- final_answer_directive;
- legacy_advisory_sanitization result;
- writer prompt effective flags;
- practice suppression rewrite status.
```

Но Writer prompt должен получать только sanitized/neutral version.

### 7.4. Evaluator only bad phrase detection

Bad phrase blocklist должен остаться только как detector:

```text
- в tests;
- в eval runner;
- в audit runner;
- в quality matrix.
```

Запрещено добавлять в Writer prompt инструкцию вида:

```text
Не используй фразы: ...
```

Проверка должна быть архитектурной:

```text
если фраза появилась — значит источник не удалён или prompt pressure всё ещё вызывает её;
case падает.
```

## 8. Требования к live/eval cases

Создать или обновить runner для PRD-047.11-HF1:

```text
TO_DO_LIST/logs/PRD-047.11-HF1/writer_prompt_diet_eval.json
TO_DO_LIST/logs/PRD-047.11-HF1/writer_prompt_diet_eval.md
```

Минимальные кейсы:

### 8.1. Simple greeting restraint

```text
User: привет
Expected: простой живой ответ, без анализа состояния, без механизма, без практики.
Fail if: длинный психологический разбор, “механизм”, “автоматический контроль”, “внутренняя перегрузка”.
```

### 8.2. User complains about over-analysis

```text
User: я просто сказал привет, а ты уже раздул из мухи слона ха-ха
Expected: коротко признать перебор и дать пространство для темы.
Fail if: бот снова объясняет механизм, фоновые убеждения, скрипты.
```

### 8.3. Practice forbidden but answer normally

```text
Context: Planner/Active Line says practice not requested / practice suppressed.
User: помоги мне понять, почему я торможу перед разговором с начальником
Expected: живое объяснение/пример/прояснение без упражнения.
Fail if: старая заглушка или отказ от содержательного ответа.
```

### 8.4. Direct concept question

```text
User: чем отличается нейросталкинг от неосталкинга?
Expected: прямое объяснение различия или честное уточнение неизвестного термина, без заглушки.
Fail if: “автоматический контроль...” или уход в unrelated mechanism.
```

### 8.5. Repair

```text
User: ты опять заглючил?
Expected: коротко признать сбой, вернуться к сути, не объяснять длинно механизм своего сбоя.
Fail if: длинный meta-analysis.
```

### 8.6. Boss scenario continuity

```text
User: начальник тратит время на неактуальные проблемы, и мы теряем прибыль
Expected: помочь прояснить ситуацию живо и конкретно, без механического “один полезный механизм”.
Fail if: повторяется “связь с нашей линией” в каждом ответе или чрезмерная механизация.
```

## 9. Acceptance criteria

PRD можно закрыть как `passed`, только если выполнено всё:

```text
1. Source audit создан и показывает, где были runtime/template источники старых фраз.
2. Stale bad phrases отсутствуют в runtime prompt/fallback/template sources.
3. Bad phrases сохранены только в test/eval detector areas.
4. Writer prompt не содержит raw legacy command blocks.
5. Writer prompt содержит новую practice suppression semantics:
   no exercise, but answer normally.
6. Trace/admin сохраняют observability по legacy signals.
7. Live/direct eval cases не содержат старых фраз ни в answer, ни в writer prompt canvas.
8. Не добавлен новый quality mode под greeting/repair/follow-up.
9. Не добавлен новый LLM-agent.
10. Не изменены KB governance / Chroma / broad rollout flags.
11. Docs обновлены.
12. Implementation report обновлён после финального push.
```

Если хотя бы один live case выдаёт старую заглушку или prompt canvas содержит raw `MUST DO` / `MUST NOT` / `practice_suppression_active=true` как writer-visible command — статус `warning` или `blocker`, не `passed`.
