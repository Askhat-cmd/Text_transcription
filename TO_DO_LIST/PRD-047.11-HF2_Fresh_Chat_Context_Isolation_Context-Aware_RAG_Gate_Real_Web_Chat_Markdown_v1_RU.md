- **Язык:** RU

- **Предыдущий этап:** PRD-047.11-HF1 — Writer Prompt Diet / Practice Suppression Rewrite / Legacy Advisory Sanitization v1

## 1. Контекст

После PRD-047.11-HF1 runtime стал заметно лучше: legacy imperative blocks убраны из writer-visible prompt, practice suppression переписан как `no_exercise_but_answer_normally`, stale fallback source удалён.

Но live-тест владельца проекта показал оставшиеся проблемы:

1. На первый ход нового чата `привет, меня зовут Асхат!` бот снова дал объяснение про автоматический контроль.
2. В LLM canvas видно, что на greeting/contact turn система сформировала `answer_shape=mechanism_explanation`, advisory summary с фокусом “показать один механизм”, а также RAG/Knowledge candidates в Writer path.
3. В canvas одновременно видно противоречие: `CONTEXTUAL RETRIEVAL DECISION: rag_included_count=0`, но выше в prompt присутствует блок `KNOWLEDGE RAG HITS` с большими чанками из базы.
4. В Web Chat по-прежнему нет подтверждённого красивого markdown/rendering поведения: абзацы, списки, жирный текст, line-height, bubble width, screenshot proof.

Вывод: HF1 почистил Writer prompt от старых команд, но следующие проблемы находятся выше/рядом:

- старт нового чата и cross-session memory policy;
- единая точка принятия решения, что именно попадает Writer’у из RAG;
- реальная frontend-readable rendering proof.

---

## 2. Главная цель

Сделать начало нового чата чистым и спокойным, а RAG — действительно контекстно-уместным.

```text
Новый чат = новая локальная линия.
Старая память = background context, не active steering.
RAG = попадает Writer’у только через единый context-aware gate.
Web Chat = реально читабельный markdown/rendering в браузере.
```

## 5. Требование A — Fresh Chat Context Isolation

### 5.1. Новая политика

Ввести явную политику:

```text
fresh_chat_context_policy_v1
```

Смысл:

```text
Если это начало нового чата / новая session / первые 1–2 turn,
то система не должна автоматически продолжать старую тему,
если пользователь явно не просит продолжить прошлый разговор.
```

### 5.2. Разделить память на три уровня

Не нужно полностью переписывать память. Достаточно разделить поведение:

```text
recent_current_chat_turns
  можно использовать всегда;

cross_session_summary / user_profile
  background only, использовать только при явной релевантности;

rag/knowledge_base
  только через unified RAG gate.
```

### 5.3. Для начала нового чата

На первом turn нового чата:

```text
if user_message is greeting/contact/short intro:
  - do not infer previous topic
  - do not continue stale mechanism line
  - do not include cross-session assistant answer as active context
  - do not include KB/RAG chunks unless user asks concept/question
  - answer simply
```

Пример ожидаемого ответа:

```text
Привет, Асхат. Рад знакомству. О чём хочешь поговорить?
```

или:

```text
Привет, Асхат. Рад тебя видеть. Можем начать с любой темы.
```

Запрещено:

```text
- объяснять “автоматический контроль”;
- говорить “сейчас полезнее не упражнение...”;
- анализировать приветствие;
- тащить старую active line без запроса пользователя.
```

### 5.4. Когда можно использовать прошлую память

Разрешено использовать прошлую память, если пользователь явно пишет:

```text
продолжим прошлую тему
как мы обсуждали
помнишь, я говорил
вернёмся к тому разговору
то, что я вчера писал
```

Или если current user message явно совпадает с сохранённой темой и не является greeting/contact.

### 5.5. Trace fields

Добавить в trace:

```json
{
  "fresh_chat_context_policy": {
    "version": "fresh_chat_context_policy_v1",
    "is_new_chat": true,
    "turn_index": 1,
    "is_greeting_or_contact": true,
    "cross_session_memory_allowed": false,
    "cross_session_memory_reason": "fresh_greeting_no_explicit_continuation",
    "active_context_source": "current_chat_only"
  }
}
```

## 7. Требование C — Unified Context-Aware RAG Gate

### 7.1. Проблема

Сейчас live canvas показывает противоречие:

```text
CONTEXTUAL RETRIEVAL DECISION:
retrieval_action=memory_only
rag_included_count=0
```

но Writer prompt всё равно содержит:

```text
KNOWLEDGE RAG HITS:
...
```

Это недопустимо.

### 7.2. Новое правило

Должна быть одна финальная точка, которая решает, что попадёт Writer’у:

```text
writer_context_package_v1
```

Она должна включать:

```json
{
  "recent_turns_for_writer": [],
  "profile_for_writer": {},
  "rag_for_writer": [],
  "rag_candidates_for_trace": [],
  "rag_gate_decision": {}
}
```

### 7.3. Writer-visible RAG

Writer получает RAG только если:

```text
rag_gate_decision.include_rag_for_writer=true
```

Если false:

```text
semantic_hits_for_writer = "нет релевантных знаний"
```

или пустой безопасный блок, но без raw chunks.

### 7.4. Trace-visible RAG

RAG candidates можно и нужно сохранять в trace/admin:

```json
{
  "rag_candidates_for_trace": [
    {
      "score": 0.117,
      "source": "book",
      "reason_not_included": "fresh_greeting_no_knowledge_need"
    }
  ]
}
```

Но candidates не должны попадать в Writer prompt.

### 7.5. Gating conditions

Не включать RAG в Writer prompt, если:

```text
- is_new_chat=true and is_greeting_or_contact=true;
- knowledge_answer_needed=false;
- retrieval_action=memory_only/recent_context_only/none;
- top scores ниже configured threshold;
- current turn is repair about bot behavior, unless user asks conceptual explanation;
- current turn is simple social/contact phrase.
```

Включать RAG, если:

```text
- пользователь спрашивает known concept;
- пользователь просит объяснить термин/модель/практику;
- пользователь отвечает “да” на предложение объяснить концепт/пример, и предыдущая тема требовала БД;
- retrieval_action=kb_grounding/concept_answer/practice_catalog;
- gate reason явно указывает, почему RAG нужен.
```

Важно: не делать правило “короткая реплика = без RAG”. Короткая реплика может наследовать тему, где RAG нужен.

### 7.6. Trace fields

```json
{
  "writer_context_package": {
    "version": "writer_context_package_v1",
    "rag_candidates_count": 3,
    "rag_included_for_writer_count": 0,
    "rag_suppressed_reason": "fresh_greeting_no_knowledge_need",
    "knowledge_answer_needed": false,
    "writer_prompt_contains_rag_chunks": false
  }
}
```

## 9. Требование E — Real Web Chat Markdown Rendering

### 9.1. Реальный рендеринг

Проверить и при необходимости исправить компонент Web Chat message rendering.

Требования:

- markdown рендерится в сообщениях бота;
- `**жирный**` становится `<strong>`;

- абзацы разделяются вертикальными отступами;
- line-height читабельный;
- bubble/content max-width адекватный;
- markdown не отображается как plain text;

## 12. Acceptance criteria

PRD можно закрыть `passed`, только если:

1. HF1 не регрессировал.
2. New chat greeting даёт простой контактный ответ.
3. First-turn greeting не содержит RAG chunks in Writer prompt.
4. Trace shows:
   - `fresh_chat_context_policy_v1`;
   - `writer_context_package_v1`;
   - `writer_prompt_contains_rag_chunks=false` for greeting.
5. RAG включается только по explicit gate reason.
6. Short `да` после concept offer не ломается и может наследовать RAG контекст.
7. Web Chat markdown реально подтверждён browser screenshot + DOM JSON.
8. Admin effective/memory screenshots сохранены.
9. Docs обновлены.
10. No-mutation proof создан.
11. Commit + push выполнены.
12. Implementation report обновлён после финального push.

Если Playwright/browser proof не выполнен — статус максимум `warning`, не `passed`.

Если RAG chunks всё ещё попадают Writer’у при `rag_included_count=0` — статус `blocker`.

Если greeting снова отвечает mechanism explanation — статус `blocker`.
