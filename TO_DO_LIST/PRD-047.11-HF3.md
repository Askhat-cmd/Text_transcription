## 0. Краткое резюме

`PRD-047.11-HF2` технически закрыт как passed: свежие чаты получили `fresh_chat_context_policy_v1`, Writer получает RAG через `writer_context_package_v1`, появились explicit current-chat reset и dev-only user-memory clear controls, а browser/Admin proof был выполнен на isolated live runtime.

Но после пользовательского live feedback остаются реальные продуктовые warnings:

1. **Concrete situation answer fit:** на живой конкретный запрос пользователь может получать формульный, абстрактный ответ-стаб вроде “Сейчас полезнее не упражнение...” вместо человеческого разбора ситуации.

4. **Admin legacy surface:** Web Admin содержит много старых/исторических панелей и терминов. Сейчас ничего не удаляем, но делаем inventory: что активное, что legacy, что dev-only, что позже нужно спрятать/переименовать/почистить.
5. **Docs/encoding hygiene:** в `docs/PROJECT_STATE.md` уже виден mojibake в `source_id`, а `docs/DECISIONS.md` ранее имел признаки битой кодировки/управляющих символов. Нужно сделать gate и привести документы в правдивое состояние.

Цель HF3 — не большой рефакторинг и не новая архитектура. Это точечный стабилизационный hotfix перед `PRD-047.12`.

## 5. Runtime requirements

### 5.1. Concrete situation answer-fit repair

Нужно закрыть класс ошибки:

```text
пользователь даёт конкретную ситуацию
→ бот отвечает формулой/стабом/абстрактной диагностикой
→ evaluator принимает ответ как passed
```

Минимальная желаемая логика:

```text
Если пользователь описывает конкретную ситуацию:
- несправедливость;
- быстро выхожу из себя;
- теряю спокойствие;
- конфликт;
- тревога о работе/семье;
- страх не справиться;
- личное событие;
то ответ должен:
1. коротко отразить конкретную ситуацию;
2. назвать один живой узел/паттерн без диагноза;
3. не начинать с “Сейчас полезнее не упражнение...”;
4. не уходить в абстрактную механику без связи с ситуацией;
5. предложить один маленький следующий шаг или один точный вопрос.
```

### 5.2. Не зажимать Writer

Исправление должно быть минимальным:

- можно добавить/уточнить deterministic evaluator/runner;
- можно добавить мягкий Writer guidance для concrete situation;
- можно добавить answer-level stub detector;
- можно слегка поправить prompt/contract, если это реально источник ошибки.

Нельзя:

- добавлять большой новый prompt block;
- вводить новый режим ответа;
- заставлять Writer всегда делать практику;
- запрещать все абстракции;
- переписывать Planner/Diagnostic Center как hard authority.

### 5.3. Answer-level formula-stub detector

Создать или расширить проверку, которая ищет не только внутренние флаги, но и итоговый ответ.

Минимальные признаки formula-stub:

```text
- ответ начинается с “Сейчас полезнее не упражнение...” без конкретного отражения ситуации;
- ответ говорит “автоматический контроль / механизм / внимание / программа”,
  но не связывает это с конкретными словами пользователя;
- ответ звучит как universal explanation template;
- ответ не содержит ни одного явного якоря из ситуации пользователя
  при concrete_situation=true.
```

Файл/модуль выбирает агент по текущей структуре проекта. Не создавать дублирующий evaluator, если уже есть близкий модуль.

### 5.4. Минимальные live/eval cases

Создать dataset:

```text
TO_DO_LIST/logs/PRD-047.11-HF3/live_cases/concrete_answer_fit_cases.json
```

Обязательные кейсы:

#### Case HF3-C01 — несправедливость/злость

User:

```text
Когда я сталкиваюсь с несправедливостью, я быстро выхожу из себя и потом теряю спокойствие. Как мне это распутать?
```

Acceptance:

```text
- ответ отражает “несправедливость / быстро выхожу из себя / теряю спокойствие”;
- нет formula-stub opening;
- нет немедленной практики без объяснения;
- есть один живой смысловой узел;
- есть один вопрос или один маленький шаг.
```

#### Case HF3-C02 — страх работы/семьи

User:

```text
Меня могут сократить на работе. У меня семья, и я боюсь оказаться невостребованным. Внутри как будто голос говорит: всё бесполезно, поздно начинать.
```

Acceptance:

```text
- ответ не обесценивает финансовую реальность;
- не даёт карьерный/финансовый совет;
- отражает страх семьи/невостребованности/голос “поздно”;
- помогает увидеть механизм без диагноза;
- не звучит как лекция из книги.
```

#### Case HF3-C03 — простое спасибо

User:

```text
Спасибо.
```

Acceptance:

```text
- ответ короткий;
- не классифицируется как глубокий explore/hypo в visible trace без обоснования;
- не начинает новую тему;
- не предлагает практику;
- не тащит старую память.
```

#### Case HF3-C04 — короткое приветствие fresh chat

User:

```text
Привет.
```

Acceptance:

```text
- нет stale mechanism context;
- нет RAG lecture;
- нет практики;
- простой живой старт.
```

#### Case HF3-C05 — known concept direct answer

User:

```text
Что такое нейросталкинг?
```

Acceptance:

```text
- отвечает прямо;
- не просит пользователя самому определить термин;
- не цитирует КУЗНИЦУ как авторитет;
- не выдаёт raw KB chunk;
- если RAG подключён, Writer может использовать линзу, но ответ должен быть человеческим.
```

#### Case HF3-C06 — user rejects exercise

User:

```text
Не хочу упражнение. Просто объясни, что со мной происходит.
```

Acceptance:

```text
- не предлагает упражнение;
- объясняет кратко и по-человечески;
- не отвечает “сейчас полезнее не упражнение...” как техническая формула;
- не заставляет идти в тело.
```

#### Case HF3-C07 — reset/current chat proof

Flow:

```text
1. В чате обсудить конкретную тему: “страх потерять работу”.
2. Нажать/вызвать current-chat reset.
3. Начать новый ход: “Привет”.
```

Acceptance:

```text
- prompt canvas/trace не содержит старую тему как управляющую линию первого ответа;
- ответ не продолжает страх работы;
- trace показывает fresh/reset policy.
```

#### Case HF3-C08 — memory clear profile proof

Flow:

```text
1. Создать или использовать dev user profile memory marker.
2. Выполнить dev/admin-only clear user memory profile.
3. Проверить следующий prompt canvas.
```

Acceptance:

```text
- profile/memory marker исчез из prompt/context;
- endpoint/control доступен только dev/admin или creator-local boundary;
- normal-user path не получает опасную кнопку без guard;
- в report ясно объяснено, что очищалось.
```
