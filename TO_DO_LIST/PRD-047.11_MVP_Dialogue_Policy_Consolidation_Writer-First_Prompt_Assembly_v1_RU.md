# PRD-047.11 — MVP Dialogue Policy Consolidation / Writer-First Prompt Assembly v1

**Проект:** Bot Psychologist / Neo MindBot  
**Язык документа:** RU  
**Статус:** draft for IDE-agent implementation  
**Целевая ветка:** `main`  
**Фокус:** консолидация MVP dialogue policy, Writer-first prompt assembly, advisory-only Diagnostic Center, устранение старых конфликтующих директив, сохранение гибкости Web Admin и наблюдаемости.

---

## 0. Краткое резюме

После `PRD-047.10`, `PRD-047.10-HF1` и `PRD-047.10-HF2` в проекте появились важные элементы:

- `mvp_free_dialogue` как developer-local профиль;
- `human_like_answer_policy`;
- `dialogue_pragmatics_v1`;
- `contextual_retrieval_gating_v1`;
- `live_turn_evidence_v1`;
- markdown smoke artifacts;
- live evidence export.

Однако живые тесты владельца проекта показывают, что проблема не закрыта: Writer всё ещё получает противоречивые команды от старых слоёв — Diagnostic Card, Writer Move, Active Line, Response Planner, fallback/compliance paths. В результате в `mvp_free_dialogue` возникает старая заглушка вроде:

```text
Отвечу по сути без навязывания практик. Ключевой узел в том, что автоматический контроль может включать внутреннюю перегрузку еще до действия.
```

Она появляется даже там, где пользователь просит продолжить линию, ответить на конкретный вопрос, привести пример или исправить предыдущий сбой.

`PRD-047.11` должен не добавлять ещё один слой маркеров и не усиливать guards, а **пересобрать верхний слой управления ответом вокруг Writer-first prompt assembly**.

Целевая формула:

```text
Старые слои готовят сцену → единый Final Answer Directive решает конфликт → Writer пишет живой ответ.
```

## 1. Контекст

### 1.1. Что уже хорошо

Текущая система имеет ценные компоненты, которые нужно сохранить:

- один unified runtime path;
- один Writer;
- один Response Planner;
- один orchestrator;
- Diagnostic Center как мощный источник диагностики;
- RAG / Memory Retrieval;
- Thread Manager;
- Web Admin;
- trace/debug/live-turn evidence;
- no-mutation proof pipeline;
- модульную структуру PRD/log/report artifacts.

### 1.2. Что сломано концептуально

Система уже перешла в MVP-направление “развязать руки Writer’у”, но prompt всё ещё содержит старые директивные блоки:

```text
WRITER MOVE: clarify_one_point
max_sentences=5
max_questions=1
MUST DO: ask_one_specific_question
MUST NOT: do_not_over_explain
practice_suppression_active=true
response_depth=short
```

Эти блоки могут конфликтовать с новыми сигналами:

```text
dialogue_pragmatics.should_answer_directly=true
should_not_ask_confirmation_again=true
human_like_answer_policy.enabled=true
planner_authority=advisory
must_avoid=не уходить в заглушку
```

Итог: Writer видит не одну ясную задачу, а набор конкурирующих приказов.

### 1.3. Главная архитектурная проблема

Текущая система слишком часто работает так:

```text
Analyzer / Diagnostic / Planner диктуют Writer’у форму ответа.
```

Нужно перейти к модели:

```text
Analyzer / Diagnostic / Planner / Diagnostic Center / RAG готовят сцену.
Final Answer Directive собирает единое решение.
Writer пишет ответ как живой автор.
Hard boundaries остаются только для safety/governance.
```

```json
  }
}
```

## 4. Архитектурное решение

### 4.1. Writer-first Prompt Assembly

Добавить новый модуль:

```text
bot_psychologist/bot_agent/multiagent/final_answer_directive.py
```

Он должен строить единый `FinalAnswerDirective` из:

- user message;
- recent turns;
- dialogue policy;
- dialogue pragmatics;
- contextual retrieval decision;
- response planner;
- active line;
- diagnostic card;
- Diagnostic Center shadow/effective payload;
- knowledge answer guard;
- thread state;
- state analyzer output;
- human-like answer policy.

Главная задача модуля — не “добавить ещё один guard”, а **разрешить конфликт старых advisory-сигналов до попадания в Writer prompt**.

### 4.2. Final Answer Directive как единственный управляющий блок для Writer в MVP

В `mvp_free_dialogue` Writer prompt должен получать:

1. System prompt Writer’а.
2. User message.
3. Recent turns.
4. Thread summary / memory, если уместно.
5. RAG/context, если уместно.
6. NEO Philosophy Kernel как lens.
7. `FINAL ANSWER DIRECTIVE` как единственный итоговый управляющий блок.
8. Safe hard boundaries.

Writer prompt **не должен напрямую получать как команды**:

```text
WRITER MOVE INSTRUCTIONS
max_sentences
max_questions
MUST DO ask_one_specific_question
MUST NOT do_not_over_explain
Diagnostic Card avoid_list как приказ
Response Planner response_depth=short как приказ
Active Line should_ask_question=true как приказ
practice_suppression_active=true как приказ
```

Эти поля могут оставаться:

- в trace;
- в evidence export;
- в admin diagnostics;
- в Final Answer Directive `source_signals`;
- как advisory context, если не конфликтуют.

Но они не должны диктовать Writer’у форму ответа при `mvp_free_dialogue`.

### 4.3. Diagnostic Center остаётся, но advisory-only

Diagnostic Center должен сохраниться как важный источник обогащения контекста Writer’а.

В `mvp_free_dialogue` Diagnostic Center role:

```text
diagnostic_center_role = advisory_context_only
```

Разрешено:

- добавлять диагностическую линзу;
- подсвечивать возможный паттерн;
- предлагать осторожную гипотезу;
- давать background для Writer;
- попадать в trace/evidence/admin.

Запрещено:

- принудительно задавать форму ответа;
- заставлять Writer задавать вопрос;
- принудительно запрещать развернутый ответ;
- перезаписывать явную просьбу пользователя;
- инициировать практику/антипрактику как жёсткую команду;
- блокировать direct repair, если safety не активен.

### 4.4. Hard boundaries vs advisory signals

Разделить в коде и prompt:

#### Hard boundaries

```text
minimal_safety_baseline
privacy
no raw secrets
no raw private logs
no diagnosis
no medical/legal/financial directives
no spiritual authority
crisis routing, если safety действительно активен
```

#### Advisory signals

```text
state label
nervous_state
diagnostic_card
active_line
response_planner
writer_move
diagnostic_center
practice preference
rag hints
style hints
```

В `mvp_free_dialogue` advisory signals должны быть материалом для Writer’а, а не клеткой.

## 5. Runtime requirements

### 5.1. Новый FinalAnswerDirective payload

Добавить dataclass / TypedDict / Pydantic-compatible структуру:

```python
class FinalAnswerDirective:
    version: str
    profile: str
    user_intent: str
    answer_obligation: str
    must_answer: str
    answer_shape: str
    depth: str
    style: str
    question_policy: str
    practice_policy: str
    rag_policy: str
    diagnostic_center_role: str
    planner_role: str
    active_line_role: str
    diagnostic_card_role: str
    writer_autonomy: str
    hard_boundaries: list[str]
    advisory_context: dict
    suppressed_legacy_constraints: list[str]
    source_signals: dict

```

### 5.2. Conflict resolution rules

Implement deterministic conflict resolver:

```text
minimal safety > explicit user request > repair/dissatisfaction > accepted previous offer > knowledge/concept need > conversation continuity > writer freedom > planner/diagnostic advisory
```

Examples:

#### Case A — user says “да” after assistant offered example

Expected:

```json
{
  "user_intent": "accepted_previous_offer",
  "answer_obligation": "fulfill_previous_offer",
  "must_answer": "give the promised example",
  "question_policy": "do_not_ask",
  "depth": "medium",
  "rag_policy": "optional_if_relevant"
}
```

#### Case B — user says “ты не ответил”

Expected:

```json
{
  "user_intent": "repair_after_failed_answer",
  "answer_obligation": "answer_previous_question_directly",
  "question_policy": "do_not_ask_until_answered",
  "style": "brief_apology_then_direct_answer"
}
```

#### Case C — user asks “чем отличается Нейросталкинг от Неосталкинга?”

Expected:

```json
{
  "user_intent": "concept_comparison",
  "answer_obligation": "compare_two_concepts_directly",
  "answer_shape": "definition_then_difference_then_example",
  "depth": "medium",
  "question_policy": "optional_none"
}
```

#### Case D — user says “спасибо”

Expected:

```json
{
  "user_intent": "close_ack",
  "answer_obligation": "close_gently",
  "depth": "very_short",
  "question_policy": "none",
  "rag_policy": "none"
}
```

### 5.3. Stale regulate stub removal

Search entire repository for variants:

```text
Отвечу по сути без навязывания практик
Ключевой узел в том, что автоматический контроль может включать внутреннюю перегрузку еще до действия
Сфокусируюсь на разборе, без практик по умолчанию
```

Requirements:

- remove hardcoded stale fallback from `mvp_free_dialogue` path;
- if phrase remains for historical tests, isolate it under explicit test fixture only;
- final-answer evaluator must fail if this phrase appears in any live/direct/dry answer;
- no PRD may be marked `passed` if stale stub appears in any answer field, not only final answer.

### 5.4. Writer prompt cleanup

In `mvp_free_dialogue`, update Writer prompt assembly:

Allowed visible command blocks:

```text
SYSTEM PROMPT
USER MESSAGE
RECENT TURNS
RELEVANT CONTEXT
OPTIONAL RAG / KB GROUNDING
NEO PHILOSOPHY KERNEL
FINAL ANSWER DIRECTIVE
HARD BOUNDARIES
```

Legacy blocks should either be omitted or moved into non-command advisory source blocks:

```text
SOURCE SIGNALS / advisory only, do not obey as command
```

Acceptance: full LLM canvas must clearly show:

```text
FINAL ANSWER DIRECTIVE
legacy_constraints_suppressed=[...]
diagnostic_center_role=advisory_context_only
planner_role=advisory_context_only
writer_autonomy=high
```

### 5.5. Response Planner role change

Response Planner remains useful but must not command Writer in MVP.

In `mvp_free_dialogue`:

```text
response_planner.role = advisory_next_move_selector
```

Planner may suggest:

- answer shape;
- likely user intent;
- possible next move;
- warning about overloading;
- suggested depth.

Planner must not force:

- `max_sentences`;
- `ask_one_specific_question`;
- `practice_forbidden`, unless minimal safety requires it;
- stale compact answer;
- answer rewrite.

### 5.6. Active Line role change

Active Line remains for continuity, but in MVP:

```text
active_line.role = advisory_continuity_signal
```

It may suggest what line is active, but Final Answer Directive decides whether to continue, branch, repair, compare, or close.

### 5.7. Diagnostic Card / Writer Move role change

Diagnostic Card and Writer Move should not appear in Writer prompt as imperative blocks in MVP.

They must be represented as:

```json
"diagnostic_advisory": {
  "situation_label": "...",
  "possible_need": "...",
  "confidence": 0.67,
  "role": "advisory_context_only"
}
```

If they conflict with user intent, Final Answer Directive must list them in:

```json
"suppressed_legacy_constraints": [
  "writer_move.max_questions=1",
  "diagnostic_card.ask_at_most_one_question",
  "response_planner.response_depth=short"
]
```
