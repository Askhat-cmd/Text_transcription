# PRD-047.12-HF1 — Final Answer Acceptance Gate / Real Live Loop / Web Markdown Rendering / Docs Truth Sync

**Проект:** Bot Psychologist / Neo MindBot  
**Версия PRD:** 047.12-HF1  
**Тип:** blocker hotfix поверх `PRD-047.12`  
**Статус:** готово для передачи IDE-агенту  
**Главный принцип:** не добавлять новые заглушки, не латать фразы, а заставить уже созданную unified dialogue system реально отвечать живо и корректно на пользовательском пути.

## 0. Краткое резюме

`PRD-047.12` создал важный архитектурный каркас:

```text
unified_dialogue_policy_v2
profile presets
Dialogue Act Resolver
Last Assistant Offer Tracker
Unanswered Question Tracker
Style Preference State
Answer Obligation Resolver
Final Answer Directive
Writer-first prompt assembly
```

Но live-проверка после реализации показала blocker:

```text
1. Бот всё ещё может отвечать stale-stub фразой вместо живого ответа.
2. Acceptance runner пропустил плохой concrete situation answer как passed.
3. Web Chat markdown/readability фактически не работает красиво на пользовательском экране.
4. Вопрос может считаться answered по намерению системы, а не по реальному качеству final_answer.
5. Negative feedback / goodbye / insult могут уходить в continue_thread.
6. Документация должна быть приведена к фактическому состоянию системы.
7. Нужно проверить, насколько реально убрано раздвоение safe_guided / mvp_free_dialogue, а не только задекларировано.
```

HF1 должен закрыть эти blocker-классы.

Главная формула HF1:

```text
Не создавать заглушки.
Не писать fake-pass отчёты.
Не принимать mocked/dry доказательства за реальную работу.
Не считать obligation ответом.
Не считать DOM fixture доказательством пользовательского markdown.

Система должна пройти реальные live cycles на localhost:3000/chat + backend runtime.
Если падает — агент исправляет, перезапускает, повторяет цикл.
Если не добился passed — не пушит в main как passed.
```

## 1. Почему нужен HF1

После `PRD-047.12` были обнаружены фактические несоответствия:

### 1.1. False-positive live acceptance

`T08 - concrete situation` был отмечен как passed, хотя ответом была stale-stub фраза:

```text
Сейчас полезнее прямое объяснение механизма: автоматический контроль может перегружать внимание еще до действия...
```

Это значит:

```text
Acceptance runner проверяет наличие pipeline/output, но не проверяет answer quality gate.
```

### 1.2. Stale-stub повторяется в реальном чате

На реальные реплики пользователя:

```text
здравствуй, мой дорогой бот!
Хочу разобраться с одной непростой темой...
ты снова проигнорировал мой вопрос
повторю заново свои вопросы...
спасибо что не сумел ответить...
до свидания глупый бот!
```

бот повторял или удерживал одну и ту же неадекватную фразу про автоматический контроль.

Это значит:

```text
final_answer не проходит смысловую валидацию;
плохой assistant turn не помещается в quarantine;
следующие turns заражаются старым плохим ответом;
repair recovery не восстанавливает настоящий unanswered question.
```

### 1.3. Web Markdown визуально не работает

Скриншот пользователя показывает: в Web Chat ответ идёт обычной простынёй текста, без нормального видимого markdown-рендера:

```text
- нет устойчивого жирного выделения;
- нет красивых структурных списков;
- нет ChatGPT-like spacing;
- DOM smoke мог быть passed, но пользовательский визуальный путь всё ещё poor.
```

Это значит:

```text
browser smoke должен проверять не только DOM existence, но и computed style + real assistant bubble visual path.
```

### 1.4. Документация отстаёт от runtime

`bot_psychologist\docs` должен отражать фактическую систему после PRD-047.12-HF1:

```text
какая policy реально активна;
какие presets реально существуют;
как работает Writer-first assembly;
как работает acceptance gate;
как запускать реальные live cycles;
какие legacy элементы остались advisory-only;
что НЕ считается production-ready.
```

## 2. Цель HF1

HF1 должен сделать пять вещей:

```text
1. Ввести Final Answer Acceptance Gate, который проверяет не только intent/obligation, но реальный final_answer.
2. Ввести Stale Stub Quarantine: плохой ответ не считается answered, не становится active focus, last offer или advisory seed.
3. Ввести Real Live Acceptance Loop: реальные запросы, реальные ответы, перезапуск/исправление до passing system.
4. Исправить Web Chat markdown/readability на фактическом пользовательском пути.
5. Обновить bot_psychologist\docs и провести pre-main audit unified policy vs old split.
```

Ключевой критерий:

```text
Если пользователь задаёт живой конкретный вопрос, бот должен дать живой конкретный ответ.
Не шаблон. Не заглушку. Не повтор старой диагностической фразы.
```

Запрещённые deterministic элементы:

## 5. Runtime requirement: Final Answer Acceptance Gate v1

Добавить слой:

```text
final_answer_acceptance_gate_v1
```

Он должен работать после Writer + Validator, но до:

```text
- marking unanswered_question_state as answered;
- saving assistant response as healthy recent turn;
- saving last_assistant_offer;
- live acceptance pass;
- final API status OK без warning.
```

### 5.1. Input

Gate получает:

```json
{
  "user_message": "...",
  "final_answer": "...",
  "dialogue_act_resolution": {},
  "answer_obligation_resolution": {},
  "unanswered_question_state_before": {},
  "last_assistant_offer_before": {},
  "dialogue_style_state": {},
  "final_answer_directive": {},
  "writer_debug": {},
  "validator_result": {}
}
```

### 5.2. Checks

Gate обязан проверять:

```text
1. stale_stub_detected;
2. answer_too_generic_for_concrete_situation;
3. answer_does_not_address_direct_question;
4. answer_repeats_previous_bad_answer;
5. repair_failed_to_answer_recovered_question;
6. negative_goodbye_not_closed;
7. greeting_answered_with_mechanism_explanation;
8. self_intro_answered_with_lecture;
9. markdown_requested_but_plain_text_only;
10. writer_error_or_empty_answer.
```

### 5.3. Output

```json
{
  "version": "final_answer_acceptance_gate_v1",
  "status": "passed | warning | failed",
  "severity": "none | low | medium | high | blocker",
  "failed_checks": [],
  "stale_stub_detected": false,
  "answer_considered_real": true,
  "can_mark_question_answered": true,
  "can_save_as_healthy_context": true,
  "must_quarantine_answer": false,
  "retry_allowed": true,
  "retry_reason": "",
  "notes": []
}
```

### 5.4. Hard fail rules

Hard fail если:

```text
- stale_stub_detected=true;
- concrete_situation_question получил generic mechanism-only answer;
- repair_complaint не ответил на recovered question;
- greeting/contact получил old mechanism explanation;
- answer повторяет один и тот же bad answer 2+ turns;
- final_answer пустой или technical fallback без реальной ошибки;
- user explicitly says “повторю вопрос” and answer ignores repeated question.
```

Если hard fail:

```text
can_mark_question_answered=false
can_save_as_healthy_context=false
must_quarantine_answer=true
live_case_status=failed
```

## 6. Runtime requirement: Stale Stub Quarantine v1

### 6.1. Existing detector must become blocking

В проекте уже есть `stale_stub_detector.py`. HF1 должен сделать его частью runtime acceptance.

Любой final_answer, содержащий semantic stale stub:

```text
автоматический контроль
внутренняя перегрузка
до действия
энергия уходит в внутренний спор
подготовку вместо самого шага
```

не должен проходить как нормальный ответ на:

```text
- greeting;
- self_intro;
- concrete_situation_question;
- repair_complaint;
- repeated_direct_question;
- negative_goodbye;
- thanks with complaint;
- direct knowledge question.
```

### 6.2. Quarantine behavior

Если ответ попал в quarantine:

```text
- не помечать unanswered_question_state as answered;
- не использовать этот answer как last_assistant_offer;
- не использовать этот answer как active_line focus;
- не использовать этот answer как advisory_context_summary seed;
- не использовать этот answer как здоровый recent turn для Writer;
- сохранять в trace как quarantined_answer для диагностики;
- live runner должен считать case failed.
```

### 6.3. Retry policy

Если final_answer failed gate, runtime может сделать **один controlled retry** на том же turn:

```text
retry_writer_once_with_repair_prompt=true
```

Retry prompt должен сказать Writer:

```text
Предыдущий ответ не принят acceptance gate: reason=...
Ответь заново на фактический пользовательский запрос.
Не повторяй stale-stub.
Не делай заглушку.
```

Если retry снова failed:

```text
return final answer with warning status in debug,
case failed,
question remains unanswered,
do not mark passed.
```

Запрещено:

```text
- делать бесконечные runtime retry внутри одного пользовательского хода;
- скрывать failed gate как OK;
- подставлять заранее написанный психологический ответ.
```

## 7. Runtime requirement: Unanswered Question Truth State v2

Текущий дефект: вопрос может считаться answered по `answer_obligation`, даже если final_answer был плохим.

Нужно изменить принцип:

```text
question answered only if final_answer_acceptance_gate.can_mark_question_answered=true
```

### 7.1. New statuses

Добавить статусы:

```text
pending
answered
partially_answered
failed_answer
quarantined_answer
repaired
```

### 7.2. Update rule

После ответа:

```text
if gate.status == passed and gate.can_mark_question_answered == true:
    answer_status=answered/repaired
else:
    answer_status=failed_answer or quarantined_answer
    answer_required=true
```

### 7.3. Repair recovery

Если пользователь пишет:

```text
ты не ответил
ты снова проигнорировал
повторю вопрос
это не ответ
спасибо что не сумел ответить
```

система должна искать:

```text
last question with answer_required=true
or answer_status in failed_answer/quarantined_answer/partially_answered
```

А не последний раздражённый meta-turn.

## 10. Real Live Acceptance Loop

Агент должен прогонять не только unit tests, а реальные циклы.

### 10.1. Required modes

```text
dry: structural tests without provider
mock: only for isolated unit tests, cannot be final proof
direct: backend local runtime call
live: real provider/runtime call
browser: localhost:3000/chat real UI path
```

Final pass требует:

```text
unit=passed
direct=passed
live=passed
browser=passed
acceptance_gate=passed
architecture_audit=passed
docs_sync=passed
```

### 10.2. Iterative repair loop

Агент обязан выполнить цикл:

```text
1. Start backend + frontend clean.
2. Reset test session/thread state.
3. Run real live acceptance suite.
4. Run browser markdown proof.
5. If any mandatory case failed:
   a. inspect logs/traces;
   b. fix root cause;
   c. restart services;
   d. reset test session;
   e. rerun full suite.
6. Repeat until passed.
```

Чтобы избежать бесконечного локального зависания, в automation script можно поставить:

```text
max_cycles=6
```

Но если после `max_cycles` есть failures:

```text
final_status=blocker
push_to_main=false
create blocker report
stop
```

Запрещено:

```text
- снижать критерии, чтобы получить green;
- удалять failing case;
- менять expected результат под плохой ответ;
- писать passed при failed live transcript;
- пушить main с blocker status.
```

### 10.3. Cycle artifacts

Создать:

```text
TO_DO_LIST/logs/PRD-047.12-HF1/cycles/cycle_01/**
TO_DO_LIST/logs/PRD-047.12-HF1/cycles/cycle_02/**
...
TO_DO_LIST/logs/PRD-047.12-HF1/live_acceptance_cycles.json
TO_DO_LIST/logs/PRD-047.12-HF1/live_acceptance_cycles.md
```

Каждый cycle должен содержать:

```text
- started_at;
- git_head;
- services status;
- session_id;
- cases run;
- failed checks;
- trace refs;
- screenshots refs;
- fix summary before next cycle.
```
