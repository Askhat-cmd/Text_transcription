# PRD-047.12 — Unified Dialogue Profile Simplification / Safe Guided + MVP Consolidation v1

## 0. Краткое резюме

`PRD-047.12` — это не очередной точечный hotfix под отдельные фразы пользователя.  
Это архитектурная консолидация диалогового слоя NEO, чтобы система перестала бесконечно латать частные случаи вроде:

```text
"да"
"ты не ответил"
"отвечай спокойнее"
"спасибо"
"повторю вопрос"
"я просто познакомился"
```

Главная цель PRD:

```text
Сделать единый, простой и наблюдаемый диалоговый профиль,
который понимает функцию реплики пользователя,
держит последний оффер ассистента,
помнит последний неотвеченный прямой вопрос,
учитывает стилевые предпочтения пользователя
и передаёт Writer'у ясное answer obligation без жёсткого командования.
```

Текущая проблема не в отдельных словах, а в отсутствии цельного слоя:

```text
Dialogue Act Resolver
+ Last Assistant Offer Tracker
+ Unanswered Question Tracker
+ Dialogue Style Preference State
+ Answer Obligation Resolver
+ Writer-first unified profile
```

Этот PRD должен объединить уже существующие элементы:

```text
mvp_free_dialogue
safe_guided / guided remnants
dialogue_pragmatics_v1
final_answer_directive_v1
human_like_answer_policy
writer_context_package_v1
fresh_chat_context_policy_v1
planner/diagnostic advisory summaries
```

в одну понятную operational map.

Важно:

```text
Diagnostic Center НЕ удалять.
Planner НЕ удалять.
Active Line НЕ удалять.
Response Planner НЕ удалять.
Они остаются advisory-only слоями.
Writer остаётся живым автором ответа.
```

PRD-047.12 должен закрыть класс ошибок системно, а не через набор `if user_text == ...`.

## 1. Контекст

После `PRD-047.11-HF3` проект находится в warning-grade стабилизационном состоянии.

HF3 закрыл часть runtime-багов:

```text
- concrete situation answers больше не должны падать в formula-stub;
- bare gratitude turns калиброваны ближе к contact/window;
- memory reset / current-chat reset proof появился;
- localhost:3000 снова попал в browser proof;
- admin legacy inventory собран;
- encoding hygiene выполнен.
```

Но HF3 также честно зафиксировал нерешённые проблемы:

```text
- browser_status=warning;
- Web Chat markdown/readability неполный;
- система всё ещё может неверно продолжать короткие follow-up реплики;
- repair complaint может отвечать не на последний вопрос;
- простое знакомство может провоцировать лекцию;
- style preference может не становиться устойчивым стилевым состоянием;
- acceptance runner может ловить отдельные фразы, но не проверять общий механизм.
```

Пользователь справедливо указал архитектурный риск:

```text
Если мы будем чинить "да", потом "ты не ответил", потом "отвечай спокойнее",
то получим бесконечный цикл мелких правок.
```

Поэтому `PRD-047.12` должен перейти от phrase-specific hotfixes к универсальному диалоговому механизму.

## 2. Цель

Создать единый диалоговый слой `unified_dialogue_profile_v1`, который:

1. Определяет **функцию текущей реплики пользователя**, а не только её текст.
2. Сохраняет и использует **последний оффер ассистента**.
3. Сохраняет и использует **последний прямой вопрос пользователя, который требовал ответа**.
4. Учитывает **стилевые предпочтения пользователя** в рамках текущего диалога.
5. Формирует Writer'у не клетку, а ясное **answer obligation**.
6. Сохраняет Diagnostic Center / Planner / Active Line как advisory-only.
7. Не добавляет новый LLM-agent.
8. Не создаёт новый runtime path.
9. Делает Web Trace понятным для аудита.
10. Исправляет Web Chat markdown/readability как часть user-path acceptance.

Целевая формула:

```text
User message
→ dialogue act resolution
→ dialogue state update
→ answer obligation resolution
→ advisory context assembly
→ Writer-first response
→ validator/evaluator checks
→ trace proof
```

### 5.2. Dialogue Act Resolver v1

Создать или расширить существующий слой `dialogue_pragmatics_v1`, чтобы он определял не фразы, а диалоговые функции.

Допустимые dialogue acts v1:

```text
greeting
self_intro
contact_open
close_ack
direct_question
knowledge_question
concrete_situation_question
confirmation_to_last_offer
rejection_of_last_offer
repair_complaint
style_preference_update
continuation_request
summary_request
practice_request
clarification_request
topic_shift
meta_system_feedback
smalltalk
unknown
```

Важно:

```text
Нельзя реализовывать это как exact string mapping.
```

Допускаются эвристики по смысловым признакам, например:

```text
- короткая реплика + предыдущий assistant offer ожидает yes/no → confirmation_to_last_offer;
- "ты не ответил", "не ответил на вопрос", "мимо" → repair_complaint;
- "отвечай спокойнее/короче/подробнее/без воды" → style_preference_update;
- "что такое", "объясни", "как это связано" → knowledge_question;
- "я столкнулся с..." + конкретная ситуация → concrete_situation_question;
- "спасибо" / "благодарю" → close_ack.
```

Но acceptance должен проверять, что логика работает и на вариантах фраз, а не только на одном exact string.

Trace example:

```json
{
  "dialogue_act_resolution": {
    "version": "dialogue_act_resolver_v1",
    "dialogue_act": "confirmation_to_last_offer",
    "confidence": 0.82,
    "evidence": [
      "short_affirmative_reply",
      "last_assistant_offer_open"
    ],
    "not_exact_match_rule": true
  }
}
```

### 5.3. Last Assistant Offer Tracker

Добавить или нормализовать состояние последнего оффера ассистента.

Минимальная структура:

```json
{
  "last_assistant_offer": {
    "version": "last_assistant_offer_v1",
    "offer_id": "offer_turn_8_001",
    "turn_index": 8,
    "offer_type": "explain_examples | adapt_technique | continue_topic | summarize | shorten | expand | practice | none",
    "offer_text_summary": "показать, как адаптировать одну технику нейросталкинга под три уровня Спиральной Динамики",
    "expected_user_reply": "yes_no_or_request",
    "action_on_confirmation": "answer_last_offer",
    "is_open": true
  }
}
```

Требование:

```text
Если пользователь отвечает коротко "да", "ага", "можно", "покажи", "давай",
и last_assistant_offer.is_open=true,
то система должна продолжить последний оффер,
а не возвращаться к старой фразе или старому prompt canvas.
```

После исполнения оффера:

```text
last_assistant_offer.is_open=false
```

или создаётся новый offer, если ответ завершился новым предложением продолжения.

### 5.4. Unanswered Question Tracker

Добавить или нормализовать состояние последнего прямого вопроса пользователя, требовавшего ответа.

Минимальная структура:

```json
{
  "unanswered_question_state": {
    "version": "unanswered_question_tracker_v1",
    "last_direct_user_question": "что такое нейросталкинг, и как его можно применять в жизни?",
    "turn_index": 4,
    "answer_required": true,
    "answer_status": "pending | answered | partially_answered | repaired",
    "reason": "direct_knowledge_question"
  }
}
```

Требование:

```text
Если пользователь пишет repair_complaint:
"ты не ответил"
"ты не ответил на вопрос"
"ответь на вопрос"
"не уходи от вопроса"
"мимо"
```

система должна:

```text
1. найти last_direct_user_question со статусом pending/partially_answered;
2. сформировать answer_obligation=repair_and_answer_last_question;
3. Writer должен коротко признать промах;
4. Writer должен ответить именно на этот вопрос;
5. не должен генерировать случайную "полезную фразу" из старого контекста.
```

Если pending-вопроса нет:

```text
answer_obligation=repair_ask_to_repeat_briefly
```

но это fallback, не основной путь.

### 5.5. Dialogue Style Preference State

Добавить состояние стилевых предпочтений текущего диалога.

Минимальная структура:

```json
{
  "dialogue_style_state": {
    "version": "dialogue_style_state_v1",
    "tone": "calm | warm | direct | neutral",
    "length_preference": "short | medium | detailed | adaptive",
    "complexity_preference": "simple | normal | deep",
    "avoid": [
      "overexplaining",
      "unrequested_practice",
      "patronizing_tone"
    ],
    "updated_by_turn_index": 3,
    "source": "style_preference_update"
  }
}
```

Требование:

```text
"отвечай спокойнее" → tone=calm
"короче" → length_preference=short
"подробнее" → length_preference=detailed
"без воды" → avoid includes overexplaining
"без упражнений" → avoid includes unrequested_practice
```

Это не должно становиться жёсткой клеткой.  
Если пользователь позже просит подробно, explicit request должен временно победить short preference.

### 5.6. Answer Obligation Resolver v1

Минимальные answer obligations:

```text

repair_and_answer_last_question
acknowledge_style_preference_then_answer
close_gently
acknowledge_self_intro

```

Пример:

```json

    "answer_obligation": "answer_last_offer",
    "answer_shape": "structured_explanation",
    "question_policy": "optional_none",
    "depth": "medium",
    "style_overrides": {
      "tone": "calm",
      "avoid_overexplaining": true

    "source": [
      "dialogue_act=confirmation_to_last_offer",
      "last_assistant_offer.is_open=true"
    ]
  }
}
```

### 5.7. Writer Contract / Prompt Assembly

Writer должен получать unified block:

```text
UNIFIED DIALOGUE PROFILE
DIALOGUE ACT RESOLUTION
LAST ASSISTANT OFFER
UNANSWERED QUESTION STATE
DIALOGUE STYLE STATE
ANSWER OBLIGATION
FINAL ANSWER DIRECTIVE
ADVISORY CONTEXT SUMMARY
```

Приоритеты:

```text
1. Minimal safety / hard boundaries
2. Explicit user request
3. Answer obligation
4. Last offer / unanswered question recovery
5. Style preference
6. Knowledge/context need
7. Planner/Diagnostic advisory
8. General writer freedom
```

Запрет:

```text
Writer prompt не должен снова превращаться в длинную клетку правил.
```

Нужно сохранить Writer freedom:

```text
Writer пишет естественно.
Система только сообщает, что сейчас нужно сделать:
- ответить на вопрос;
- продолжить последний оффер;
- исправить промах;
- учесть стиль;
- закрыть контакт.
```

### 5.8. Diagnostic Center Boundary

Обязательно сохранить:

```text
diagnostic_center_role=advisory_context_only
planner_role=advisory_context_only
active_line_role=advisory_context_only

```

Acceptance должен проверить, что в trace и docs нет:

```text

```

## 7. Web Chat / UI requirements

### 7.1. Markdown rendering contract

На `http://localhost:3000/chat` assistant messages должны рендерить:

```text
- абзацы как отдельные paragraphs;

- списки как <ul>/<ol> + <li>;
- нумерованные списки как <ol>;
- переносы не должны сливаться в одну простыню;
- raw markdown ** не должен быть виден;
- trace panel не должен ломать bubble readability.
```

## 13. Acceptance criteria

PRD считается `passed`, только если:

```text
1. Создан TASK_LIST до кодинга.
2. Source gates выполнены.
3. Unified dialogue profile реализован без нового runtime path.
4. Dialogue Act Resolver работает как function-level classification, не exact phrase patch.
5. Last Assistant Offer Tracker работает.
6. Unanswered Question Tracker работает.
7. Style Preference State работает.
8. Answer Obligation Resolver работает.
9. Writer prompt получает unified block и не получает конфликтующих legacy hard commands.
10. Diagnostic Center / Planner / Active Line остаются advisory-only.
11. Self-intro не вызывает лекцию.
12. "отвечай спокойнее + direct question" отвечает на вопрос по сути.
13. "ты не ответил" восстанавливает последний direct question.
14. "да" продолжает последний assistant offer.
15. "спасибо" закрывает контакт коротко.
16. concrete situation не падает в formula-stub.
17. Web Chat markdown на localhost:3000/chat проходит DOM proof.
18. Browser smoke status=passed.
19. Unit/integration tests passed.
20. Encoding hygiene issue count for changed files = 0.
21. No-mutation proof passed.
22. Docs synced.
23. Implementation report обновлён после финального push.
```

Если всё, кроме browser markdown, passed:

```text
final_status=warning
```

Если dialogue act / repair / last offer не работают:

```text
final_status=blocker
```
