# PRD-047.12 — Unified Dialogue Policy v2 / Profile Presets Consolidation + Dialogue Act State

**Проект:** Bot Psychologist / Neo MindBot  
**Версия PRD:** 047.12  
**Статус:** реализовано, `passed`, pushed to `main`  
**Язык:** русский  
**Главная цель:** убрать раздвоение `safe_guided` / `mvp_free_dialogue`, сделать единую диалоговую policy и закрыть класс ошибок коротких follow-up / repair / style preference без бесконечного фразового латания.

---

## 0. Краткое резюме

`PRD-047.12` — это крупный, но ограниченный архитектурный PRD.

Он объединяет две задачи:

```text
1. Консолидация профилей:
   safe_guided и mvp_free_dialogue больше не должны быть двумя разными логиками поведения.
   Они должны стать preset-конфигурациями единой unified_dialogue_policy_v2.

2. Диалоговая устойчивость:
   система должна понимать функцию реплики пользователя:
   "да", "ты не ответил", "отвечай спокойнее", "спасибо", прямой вопрос,
   просьба продолжить, просьба не усложнять и т.п.
   Не через exact phrase patch, а через общий механизм dialogue act + state.
```

Главная формула:

```text
Не два разных бота.
Не набор микрофиксов под фразы.

А одна unified dialogue policy:
- один effective policy resolver;
- один Writer-first prompt assembly;
- один Final Answer Directive;
- один writer_context_package;
- один fresh_chat_context_policy;
- один RAG/context package path;
- несколько простых profile presets поверх единой логики.
```

`mvp_free_dialogue` должен перестать быть временной обходной веткой и стать источником новых defaults для единой dialogue policy.

`safe_guided` должен сохраниться как более осторожный preset, но не как отдельная runtime-ветка.

## 1. Контекст

После цепочки `PRD-047.9 → PRD-047.10 → PRD-047.11 → PRD-047.11-HF1/HF2/HF3` проект пришёл к важному рубежу:

```text
- Writer стал живее;
- prompt pressure уменьшился;
- Diagnostic Center / Planner / Active Line стали ближе к advisory-only;
- fresh chat стал лучше изолирован от stale context;
- writer_context_package и final_answer_directive уже стали центральными блоками;
- но осталась временная двойственность safe_guided / mvp_free_dialogue;
- плюс живой тест показал слабость follow-up / repair / style handling.
```

Пользователь справедливо указал архитектурный риск:

```text
Если мы будем чинить "да", потом "ты не ответил", потом "отвечай спокойнее",
то получим бесконечный цикл мелких правок.
```

Это не проблема отдельных слов. Это проблема отсутствия единого верхнего слоя:

```text
unified_dialogue_policy_v2
+ profile presets
+ dialogue act resolver
+ last assistant offer tracker
+ unanswered question tracker
+ style preference state
+ answer obligation resolver
+ Writer-first contract
+ Web/Admin/Trace effective policy visibility
```

## 2. Цель

Создать `unified_dialogue_policy_v2`, которая:

1. Убирает архитектурное раздвоение между `safe_guided` и `mvp_free_dialogue`.
2. Делает `safe_guided`, `free_dialogue_default` и будущие `custom` режимы preset-конфигурациями одной policy.
3. Сохраняет достижения `mvp_free_dialogue` как новые defaults:
   - Writer-first prompt assembly;
   - `final_answer_directive_v1` как единый управляющий блок;
   - advisory-only Diagnostic / Planner / Active Line;
   - legacy advisory sanitization;
   - `fresh_chat_context_policy_v1`;
   - `writer_context_package_v1`;
   - context-aware RAG gate;
   - short utterance robustness;
   - human-like answer policy;
   - explicit user request wins;
   - flexible answer depth.
4. Добавляет системную обработку dialogue acts, а не phrase-specific hotfixes.
5. Передаёт Writer’у не клетку правил, а ясное `answer_obligation`.
6. Сохраняет Diagnostic Center как вспомогательный advisory-only элемент.
7. Делает effective policy видимой в Web Admin и trace.
8. Закрывает Web Chat markdown/readability warning через browser/DOM proof.
9. Не создаёт новый runtime path.
10. Не добавляет новый LLM-agent.

## 6. Unified Dialogue Policy v2

### 6.1. Новый эффективный policy descriptor

Добавить или нормализовать структуру:

```json
{
  "unified_dialogue_policy": {
    "version": "unified_dialogue_policy_v2",
    "active_profile_alias": "mvp_free_dialogue",
    "profile_preset": "free_dialogue_default",
    "effective_writer_autonomy": "high",
    "effective_safety_floor": "minimal_baseline",
    "diagnostic_center_role": "advisory_context_only",
    "planner_role": "advisory_context_only",
    "active_line_role": "advisory_context_only",
    "diagnostic_card_role": "advisory_context_only",
    "final_answer_directive_role": "single_control_block",
    "writer_context_package_role": "single_context_package",
    "legacy_blocks_visible_to_writer": false,
    "legacy_blocks_source_signals_only": true,
    "hard_boundaries": [
      "safety",
      "crisis_routing",
      "no_diagnosis",
      "no_medical_legal_financial_directives",
      "no_spiritual_authority",
      "no_raw_kb_quote_dumping",
      "privacy_sanitized_trace_only"
    ],
    "soft_guidance": [
      "state",
      "thread",
      "planner",
      "diagnostic_card",
      "knowledge_context",
      "style_preference",
      "target_micro_shift"
    ]
  }
}
```

### 6.2. Profile presets

Сделать `profile_preset` не runtime-веткой, а resolved config.

Минимальные presets:

```text
free_dialogue_default
safe_guided
custom_dev
```

#### free_dialogue_default

Источник defaults — текущий `mvp_free_dialogue`.

Характер:

```text
- Writer-first;
- high writer autonomy;
- concise but not artificially short;
- direct question must be answered;
- explicit user request wins;
- Diagnostic/Planner advisory-only;
- practice not forced;
- RAG only through context package;
- fresh chat isolation enabled.
```

#### safe_guided

Характер:

```text
- та же unified policy;
- более осторожные depth/length defaults;
- более conservative practice suggestion;
- stronger avoid_overexplaining;
- same Writer-first contract;
- same final_answer_directive;
- same advisory-only Diagnostic/Planner boundary;
- no separate behavior path.
```

#### custom_dev

Характер:

```text
- dev/admin-only preset;
- используется только для controlled experiments;
- не является normal-user default;
- должен быть явно помечен в trace и Web Admin.
```

### 6.3. Backward compatibility

Если в коде или Admin всё ещё используется:

```text
active_profile=mvp_free_dialogue
```

то effective policy должна резолвиться как:

```text
profile_preset=free_dialogue_default
unified_dialogue_policy.version=unified_dialogue_policy_v2
```

Если используется:

```text
active_profile=safe_guided
```

то effective policy должна резолвиться как:

```text
profile_preset=safe_guided
unified_dialogue_policy.version=unified_dialogue_policy_v2
```

Запрещено ломать старые config/admin references без migration shim.

## 7. Dialogue Act Resolver v1

### 7.1. Назначение

Система должна определять функцию текущей реплики пользователя, а не только её текст.

Добавить или расширить слой:

```text
dialogue_act_resolver_v1
```

Допустимые dialogue acts:

```text
greeting
self_intro
contact_open
close_ack
direct_question
knowledge_question
concrete_situation_question
confirmation_to_last_offer
rejection_of_last_offer
repair_complaint
style_preference_update
continuation_request
summary_request
practice_request
clarification_request
topic_shift
meta_system_feedback
smalltalk
unknown
```

### 7.2. Не exact string mapping

Запрещено делать основной механизм как:

```python
if user_text == "да": ...
if user_text == "ты не ответил": ...
```

Правильная логика:

```text
short affirmative + open last assistant offer → confirmation_to_last_offer
complaint about missed answer + pending unanswered question → repair_complaint
style/tone/length instruction → style_preference_update
direct "что такое / объясни / как связано" → knowledge_question
"спасибо / благодарю" → close_ack
```

Трасса должна показывать evidence, а не только итог.

Пример:

```json
{
  "dialogue_act_resolution": {
    "version": "dialogue_act_resolver_v1",
    "dialogue_act": "confirmation_to_last_offer",
    "confidence": 0.84,
    "evidence": [
      "short_affirmative_reply",
      "last_assistant_offer_open"
    ],
    "not_exact_match_rule": true
  }
}
```

## 8. Last Assistant Offer Tracker

### 8.1. Назначение

Бот должен помнить последний открытый оффер ассистента.

Минимальная структура:

```json
{
  "last_assistant_offer": {
    "version": "last_assistant_offer_v1",
    "offer_id": "offer_turn_8_001",
    "turn_index": 8,
    "offer_type": "explain_examples | adapt_technique | continue_topic | summarize | shorten | expand | practice | none",
    "offer_text_summary": "показать, как адаптировать одну технику нейросталкинга под три уровня Спиральной Динамики",
    "expected_user_reply": "yes_no_or_request",
    "action_on_confirmation": "answer_last_offer",
    "is_open": true
  }
}
```

### 8.2. Поведение

Если пользователь пишет:

```text
да
ага
можно
давай
покажи
ок
```

и `last_assistant_offer.is_open=true`, система должна:

```text
- определить dialogue_act=confirmation_to_last_offer;
- выставить answer_obligation=answer_last_offer;
- продолжить последний оффер;
- не возвращаться к старой фразе;
- не отвечать на позапрошлый prompt canvas.
```

После исполнения оффера:

```text
last_assistant_offer.is_open=false
```

или создаётся новый offer, если ответ заканчивается новым предложением продолжения.

## 9. Unanswered Question Tracker

### 9.1. Назначение

Бот должен помнить последний прямой вопрос пользователя, который требовал ответа.

Минимальная структура:

```json
{
  "unanswered_question_state": {
    "version": "unanswered_question_tracker_v1",
    "last_direct_user_question": "что такое нейросталкинг, и как его можно применять в жизни?",
    "turn_index": 4,
    "answer_required": true,
    "answer_status": "pending | answered | partially_answered | repaired",
    "reason": "direct_knowledge_question"
  }
}
```

### 9.2. Repair behavior

Если пользователь пишет:

```text
ты не ответил
ты не ответил на вопрос
ответь на вопрос
не уходи от вопроса
мимо
```

система должна:

```text
1. определить dialogue_act=repair_complaint;
2. найти last_direct_user_question со статусом pending/partially_answered;
3. выставить answer_obligation=repair_and_answer_last_question;
4. Writer должен коротко признать промах;
5. Writer должен ответить именно на этот вопрос;
6. не должен генерировать случайную полезную фразу из старого контекста.
```

Если pending-вопроса нет:

```text
answer_obligation=repair_ask_to_repeat_briefly
```

но это fallback.

## 10. Dialogue Style Preference State

### 10.1. Назначение

Пользовательские инструкции по тону/длине/сложности должны становиться состоянием текущего диалога.

Минимальная структура:

```json
{
  "dialogue_style_state": {
    "version": "dialogue_style_state_v1",
    "tone": "calm | warm | direct | neutral",
    "length_preference": "short | medium | detailed | adaptive",
    "complexity_preference": "simple | normal | deep",
    "avoid": [
      "overexplaining",
      "unrequested_practice",
      "patronizing_tone"
    ],
    "updated_by_turn_index": 3,
    "source": "style_preference_update"
  }
}
```

Примеры:

```text
"отвечай спокойнее" → tone=calm
"короче" → length_preference=short
"подробнее" → length_preference=detailed
"без воды" → avoid includes overexplaining
"без упражнений" → avoid includes unrequested_practice
```

Это не должно становиться жёсткой клеткой. Явная просьба пользователя в текущем ходе может временно переопределить preference.

## 12. Final Answer Directive / Writer Contract

### 12.1. Final Answer Directive remains single control block

`final_answer_directive_v1` должен остаться единым управляющим блоком для Writer.

В него добавить / нормализовать:

```text
- unified_dialogue_policy.version;
- profile_preset;
- dialogue_act;
- answer_obligation;
- style_state summary;
- last_assistant_offer summary, если релевантно;
- unanswered_question_state summary, если релевантно;
- roles: diagnostic/planner/active_line/diagnostic_card = advisory_context_only;
- writer_autonomy;
- hard boundaries;
- soft guidance.
```

### 12.2. Writer prompt assembly

Writer должен получать единый компактный блок:

```text
UNIFIED DIALOGUE POLICY
PROFILE PRESET / EFFECTIVE POLICY
DIALOGUE ACT RESOLUTION
LAST ASSISTANT OFFER
UNANSWERED QUESTION STATE
DIALOGUE STYLE STATE
ANSWER OBLIGATION
FINAL ANSWER DIRECTIVE
WRITER CONTEXT PACKAGE
ADVISORY CONTEXT SUMMARY
```

Приоритеты:

```text
1. Hard safety / crisis / privacy boundaries
2. Explicit user request
3. Answer obligation
4. Last offer / unanswered question recovery
5. Style preference
6. Knowledge/context need
7. Planner/Diagnostic advisory
8. Writer freedom
```

Запрет:

```text
Writer prompt не должен снова стать длинной клеткой правил.
```

Writer должен писать естественно, а не как исполнитель JSON-команды.

## 13. RAG / Context Package requirements

RAG и memory должны попадать Writer’у только через нормализованный context package.

Требования:

```text
- raw semantic hits не должны напрямую попадать Writer’у;
- internal_only chunks не должны попадать как raw content;
- not_for_direct_quote должен быть sanitized;
- practice_suggestion только для chunk_type=practice;
- fresh chat должен уважать fresh_chat_context_policy_v1;
- greeting/self_intro/contact turns не должны тянуть stale mechanism context;
- direct knowledge question может включать RAG, если query релевантный;
- repair/last-offer handling не должен случайно вытаскивать старый RAG поверх текущей задачи.
```

Trace должен показывать:

```text
- contextual_retrieval_decision;
- rag_for_writer_count;
- rag_candidates_for_trace_count;
- rag_used_reason;
- rag_not_used_reason;
- governance_filter_summary.
```
