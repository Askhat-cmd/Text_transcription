# PRD-047.14-HF1.1 — Hardcoded User-Facing Reply Inventory / No-Stub Boundary Audit v1

## 0. Краткое резюме

Этот PRD является промежуточным audit-only этапом после `PRD-047.14-HF1`.

`PRD-047.14-HF1` успешно закрыл конкретный blocker: активный template-family leakage из `concrete_answer_fit.py` был удалён, `template_active_hits_after=0`, а `template_family_guard_v1` теперь работает как detector/quarantine layer, не создавая user-facing replacement.

Но после HF1 остаётся более широкий архитектурный риск:

```text
в active runtime всё ещё могут существовать другие hardcoded user-facing replies,
которые не относятся к найденной template-family,
но по сути являются заглушками / canned replies / static fallback answers.
```

Цель этого PRD — **не удалять их вслепую**, а провести строгий аудит и классификацию:

```text
какие hardcoded фразы являются допустимыми safety/minimal-contact guardrails,
а какие являются недопустимыми заглушками, подменяющими живой Writer.
```

Главный принцип:

```text
NEO не должен отвечать через вшитые шаблонные ответы.
Writer должен быть живым автором финального ответа.
Runtime может детектировать, ограничивать, маршрутизировать и карантинировать,
но не должен производить canned user-facing answer вместо Writer.
```

## 1. Контекст

### 1.1. Что уже принято

`PRD-047.14` был audit-only циклом и завершился со статусом `blocker`, потому что нашёл активную template-family leakage в:

```text
bot_psychologist/bot_agent/multiagent/concrete_answer_fit.py
```

`PRD-047.14-HF1` закрыл этот blocker:

```text
template_active_hits_before: 6
template_active_hits_after: 0
hardcoded_answer_factories_before: build_contextual_no_practice_answer
hardcoded_answer_factories_after: removed
final_status: passed
```

После HF1:

- `build_contextual_no_practice_answer()` удалён;
- `concrete_answer_fit.py` больше не создаёт user-facing fallback answer;
- `template_family_guard_v1` детектирует exact/fuzzy leakage;
- `final_answer_acceptance_gate_v1` может отправить contaminated answer в retry/quarantine;
- contaminated answer не должен попадать в healthy memory / summary source / last assistant offer.

### 1.2. Почему нужен HF1.1

Во время архитектурного аудита после HF1 обнаружено, что в `writer_agent.py` всё ещё есть отдельные hardcoded repair/contact ответы. Примеры классов риска:

```text
- greeting repair returns static text;
- knowledge/direct-answer repair returns static explanation;
- low-resource contact returns static support phrase;
- close/thanks returns static closing phrase;
- answer_last_offer repair may return static explanatory blocks.
```

Это не обязательно blocker HF1, потому что HF1 был направлен на конкретную template-family. Но это противоречит новому продуктово-архитектурному принципу:

```text
никаких заглушек, canned replies и вшитых пользовательских ответов,
если они не являются минимально необходимой safety/minimal-contact границей.
```

### 1.3. Почему нельзя сразу всё удалить

Некоторые hardcoded фразы могут быть допустимыми:

```text
- crisis/safety minimal response;
- пустой ответ Writer -> безопасный fallback только если иначе пользователь получит пустоту;
- короткое закрытие close/thanks, если это действительно non-generative deterministic acknowledgement;
- системные technical/debug messages, которые не идут пользователю.
```

Но другие фразы могут быть вредными:

```text
- статичные объяснения понятий;
- статичные repair answers;
- статичные “разборы” ситуации;
- фразы, которые заменяют Writer вместо retry/contract/gate;
- hidden answer factories, которые звучат как живой ответ, но одинаково повторяются.
```

Поэтому этот PRD — audit/classification, а не массовый refactor.


## 6. Обязательное правило: no-stub boundary

IDE-агент должен явно зафиксировать в отчёте:

```text
В ходе PRD-047.14-HF1.1 не создано ни одной новой user-facing заглушки.
```

Запрещённые сущности:

```text
- canned reply;
- static fallback answer;
- hardcoded final answer;
- hardcoded explanation answer;
- hardcoded repair answer;
- “temporary” user-facing text;
- answer factory;
- fixed support phrase outside safety/minimal-contact boundary;
- deterministic user-facing replacement вместо Writer.
```

Разрешённые сущности, но только после классификации:

```text
- safety/crisis minimal grounding phrase;
- empty-answer emergency fallback;
- non-user-facing debug label;
- detector marker constants;
- test fixture strings;
- historical log evidence;
- admin/help text not used as final assistant answer.
```

## 8. Классификация

Для каждого найденного hardcoded фрагмента создать запись:

```json
{
  "path": "...",
  "line_start": 123,
  "line_end": 130,
  "symbol": "function_or_variable_name",
  "snippet_preview": "...",
  "is_user_facing_possible": true,
  "call_site_count": 2,
  "call_sites": ["..."],
  "runtime_path_classification": "active_runtime|test_fixture|historical_artifact|docs|admin_help|detector_only|unknown",
  "reply_type": "safety_minimal|empty_answer_fallback|greeting|repair|knowledge_explanation|direct_answer|close_ack|debug|detector_marker|unknown",
  "classification": "allowed_minimal_boundary|allowed_safety_boundary|allowed_non_user_facing|warning_needs_targeted_refactor|blocker_stub_user_facing|unknown_requires_architect_decision",
  "reason": "...",
  "recommended_action": "keep|document|convert_to_writer_retry|convert_to_contract_signal|move_to_detector_only|remove_in_targeted_prd|needs_architect_decision"
}
```

### 8.1. Критерии `blocker_stub_user_facing`

Ставить blocker, если:

```text
- строка может стать финальным ответом пользователю;
- она объясняет понятие, ситуацию, механизм или практику вместо Writer;
- она является repair answer для конкретного вопроса;
- она содержит готовую смысловую интерпретацию;
- она может повторяться в разных диалогах;
- она не является safety/minimal-contact fallback;
- она создаёт user-facing replacement вместо retry/quarantine/gate feedback.
```

### 8.2. Критерии `allowed_safety_boundary`

Разрешено только если:

```text
- это минимальная фраза для safety/crisis/empty response;
- нет содержательной интерпретации;
- нет практики, если она не safety-critical;
- фраза короткая;
- она используется только при сбое/кризисе/пустом ответе;
- есть trace/debug evidence, что это fallback.
```

### 8.3. Критерии `warning_needs_targeted_refactor`

Ставить warning, если:

```text
- фраза user-facing, но её безопасное удаление требует понимания call sites;
- она относится к close/greeting/contact, но может звучать шаблонно;
- она используется редко, но всё равно вшита;
- замена может сломать acceptance tests;
- нужно отдельное targeted repair PRD.
```

## 14. Acceptance criteria

PRD можно считать `passed`, если:

```text
- audit tool создан и запускается;
- все hardcoded user-facing reply candidates найдены и классифицированы;
- no_stub_boundary_report создан;
- нет unknown active candidates;
- no-runtime-mutation proof passed;
- encoding gate passed;
- тесты guard/final answer gate/concrete answer fit прошли;
- implementation report создан и обновлён после push;
- docs синхронизированы;
- next PRD recommendation создана.
```

PRD должен быть `warning`, если:

```text
- есть warning_needs_targeted_refactor, но нет blocker_stub_user_facing;
- есть допустимые safety/minimal-contact фразы, требующие будущей документации;
- есть большой объём hardcoded contact/repair candidates, но они классифицированы.
```

PRD должен быть `blocker`, если:

```text
- найден хотя бы один active blocker_stub_user_facing;
- есть unknown active user-facing candidate;
- audit tool не может покрыть writer_agent.py / final_answer_acceptance_gate.py;
- no-runtime-mutation proof failed;
- encoding gate failed;
- implementation report не обновлён после push.
```
