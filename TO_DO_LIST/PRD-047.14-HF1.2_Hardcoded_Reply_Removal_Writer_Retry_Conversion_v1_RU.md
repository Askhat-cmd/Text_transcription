# PRD-047.14-HF1.2 — Hardcoded Reply Removal / Writer Retry Conversion v1

## 0. Краткое резюме

`PRD-047.14-HF1.2` — targeted repair PRD после `PRD-047.14-HF1.1`.

`PRD-047.14-HF1.1` выполнил audit-only no-stub boundary inventory и честно завершился со статусом `blocker`:

```text
hardcoded_candidates_total: 1501
active_user_facing_candidates: 131
blocker_stub_user_facing_count: 55
warning_needs_targeted_refactor_count: 41
unknown_active_candidates_count: 0
runtime_mutation_status: passed
```

Главный вывод HF1.1:

```text
В системе остались active hardcoded user-facing replies.
Они могут подменять живой ответ Writer.
Это противоречит целевой архитектуре NEO.
```

Цель HF1.2:

```text
Убрать high-confidence user-facing hardcoded replies,
не ломая мультиагентную систему,
не уничтожая advisory/diagnostic signals,
не заменяя старые заглушки новыми.
```

Архитектурная формула HF1.2:

```text
Writer пишет финальный ответ.
Runtime не пишет смысловой ответ за Writer.
Runtime может:
- классифицировать;
- сигнализировать;
- направлять;
- дать Writer contract/directive;
- отправить на retry;
- quarantine contaminated/bad answer.
```

Runtime не должен:

```text
- возвращать готовый смысловой ответ пользователю;
- чинить плохой ответ новой canned reply;
- держать hidden answer factories;
- превращать safety в автора ответа;
- превращать КУЗНИЦУ / NEO-концепты в статичные объяснения.
```

## 1. Зачем нужен этот PRD

Проект изначально строился вокруг живой смысловой системы на основе `КУЗНИЦА ДУХА v.2` и `Neo MindBot КОНСПЕКТ`.

Ранняя реализация ушла в сторону:

```text
жёсткие правила
+ маршруты
+ safety-first control
+ canned fallback answers
+ dry bot behavior
```

Поздние PRD-047.x итерации начали обратный архитектурный разворот:

```text
Writer-first
multiagent as helpers
Diagnostic Center advisory-only
Final Answer Acceptance Gate as quarantine/validation
no template leakage
no hardcoded final answers
```

HF1.2 продолжает этот разворот.

Цель — вернуть авторство финального ответа Writer, оставив другим агентам роли:

```text
- State Analyzer: видеть состояние;
- Thread Manager: держать линию;
- Context Assembly / retrieval: давать релевантный контекст;
- Diagnostic layers: advisory signals only;
- Planner / Active Line: next-move hints only;
- Gate: validation / retry / quarantine;
- Memory: сохранять только healthy accepted context.
```

## 3. Главный принцип HF1.2

HF1.2 не должен “улучшить тексты”.

HF1.2 должен убрать architectural smell:

```text
static user-facing final-answer factories
```

Запрещено:

```text
старый hardcoded answer → новый hardcoded answer
```

Разрешено:

```text
старый hardcoded answer → retry signal / gate feedback / writer directive / contract metadata
```

Пример неправильной правки:

```python
if answer_bad:
    return "Ты прав, я ушёл мимо вопроса. Нейросталкинг — это..."
```

Пример правильной правки:

```python
if answer_bad:
    self.last_debug["compliance_repair_signal"] = "answer_missed_known_concept"
    self.last_debug["retry_recommended"] = True
    return text
```

Дальше `final_answer_acceptance_gate_v1` должен провалить ответ, передать feedback в `final_answer_directive`, и Writer должен написать новый ответ сам.

## 7. Strategy: three repair passes

HF1.2 должен идти не по принципу “удалить всё”, а по трём проходам.

### 7.1. Pass A — high-confidence Writer user-facing stubs

В первую очередь работать с:

```text
bot_psychologist/bot_agent/multiagent/agents/writer_agent.py
```

И только с high-confidence user-facing returns, которые:

```text
- возвращают готовый финальный ответ;
- находятся внутри `_enforce_answer_compliance` или `_enforce_mvp_free_dialogue_compliance`;
- являются static repair / knowledge / direct-answer blocks;
- могут подменить Writer output;
- не являются safety/crisis/minimal-contact boundary.
```

Примеры классов, которые нужно конвертировать:

```text
knowledge_direct_answer_repair
repair_answer_last_question_repair
answer_last_offer_repair
style_preference_direct_answer_repair
acceptance_gate_greeting_repair — только если это не safety/minimal boundary
MVP repair branches that return fixed explanation
```

### 7.2. Pass B — convert to retry/contract/gate signal

Каждый удаляемый static return должен быть заменён не текстом, а сигналом.

Допустимые replacement mechanisms:

```text
- self.last_debug["no_stub_repair_signal"] = ...
- self.last_debug["retry_recommended"] = True
- self.last_debug["compliance_failed_checks"] append signal
- final_answer_directive.acceptance_gate_feedback
- Final Answer Acceptance Gate failed_checks
- quarantine reason
- Writer retry through existing orchestrator retry path
```

Если нет безопасного existing retry path — не создавать новый user-facing fallback. Вместо этого:

```text
- оставить original Writer text;
- пометить его как failed via gate;
- quarantine from healthy memory/summary/last offer;
- написать warning artifact.
```

### 7.3. Pass C — leave safety/minimal boundaries narrow

Разрешено временно оставить:

```text
- crisis/safety minimal grounding;
- empty answer emergency fallback;
- explicit close acknowledgement;
- minimal greeting contact;
```

Но только если:

```text
- фраза короткая;
- не объясняет понятие;
- не делает диагностику;
- не даёт практику;
- не содержит смысловой интерпретации;
- classified as allowed_safety_boundary или allowed_minimal_boundary;
- covered by tests / audit exception.
```

## 8. Что НЕ считать stub

Не удалять автоматически:

```text
- detector marker constants;
- enum labels;
- trace/debug reason strings;
- planner/active-line internal labels;
- admin/help labels;
- test fixtures;
- historical logs;
- prompt instruction text, если оно не возвращается как final answer;
- KB source text;
- safety flags.
```

Важно: `active_line.py`, `diagnostic_center.py`, `final_answer_directive.py`, `response_planner.py` могут содержать static advisory text. Это может быть архитектурным долгом, но HF1.2 не должен удалять advisory text, пока не доказано, что он становится user-facing final answer.

```text

- проверять, что target stubs removed/converted;
- проверять, что новые hardcoded final-answer factories не появились;
- проверять, что detector/test/admin exceptions не считаются blockers;
- проверять, что `\x00` / UTF-16-like NUL logs не проходят encoding gate.
```

### 9.2. Add runtime no-stub detector

Если нужно, добавить:

```text
bot_psychologist/bot_agent/multiagent/no_stub_guard.py
```

Но этот модуль не должен писать ответы.

Он может только:

```text
- detect hardcoded-looking final answer;
- classify answer as suspicious;
- produce guard payload;
- contribute failed_check to gate;
- expose debug/trace.
```

Обязательное поле:

```json
{
  "user_facing_replacement_created": false
}
```

### 9.3. Update Writer compliance

В `writer_agent.py`:

1. Найти high-confidence static returns.
2. Удалить/конвертировать их в signals.
3. Не добавлять новые static replacement strings.
4. Сохранить Writer output как источник текста.
5. Если output плохой — пусть Gate / retry path обрабатывает.

Пример допустимой структуры:

```python
if should_repair_known_concept:
    self._set_final_answer_shape_debug("knowledge_direct_answer_retry_signal")
    self.last_debug["no_stub_repair_signal"] = {
        "reason": "knowledge_answer_missed",
        "recommended_action": "writer_retry",
        "must_answer": "known_concept_question",
    }
    return text
```

### 9.4. Update Final Answer Acceptance Gate

Если writer debug содержит no-stub repair signal, gate должен:

```text
- failed_checks += appropriate check;
- retry_recommended = true;
- must_quarantine_answer = true unless retry succeeds;
- can_save_as_healthy_context = false;
- can_use_as_summary_source = false;
- can_save_last_assistant_offer = false;
```

Не создавать replacement answer.

### 9.5. Orchestrator retry path

Использовать только existing Writer retry mechanism from `final_answer_acceptance_gate_v1`.

Запрещено:

```text
- добавлять новый LLM agent;
- добавлять новый runtime path;
- создавать static fallback after failed retry.
```

Если retry exhausted:

```text
- return best available non-stub Writer output only if gate accepts;
- otherwise quarantine state effects;
- record blocker/warning artifact.
```

## 16. Acceptance criteria

HF1.2 can be `passed` only if:

```text
- all high-confidence writer_agent static repair/knowledge/direct-answer returns are removed or converted;
- no new canned user-facing replies are introduced;
- Writer remains final answer author;
- Gate/contract/retry handle repair instead of static answer returns;
- no-stub acceptance tool passes for target scope;
- targeted tests pass;
- regression tests pass;
- encoding gate catches and prevents NUL/UTF-16-like logs;
- runtime mutation scope proof passes;
- implementation report is created and updated after push;
- docs are synchronized.
```

HF1.2 must be `warning` if:

```text
- writer_agent high-confidence stubs are removed,
  but remaining blocker_stub_user_facing candidates exist in out-of-scope/advisory modules;
- summary routing warning remains;
- low-confidence candidates require later PRD;
- safety/minimal-contact boundaries remain and are documented.
```

```text
- any high-confidence writer_agent final-answer stub remains;
- a new hardcoded user-facing answer is introduced;
- old stubs are replaced by new stubs;
- tests fail;
- encoding gate misses \x00;
- runtime mutation touches disallowed paths;
- implementation report is not post-push updated.
```
