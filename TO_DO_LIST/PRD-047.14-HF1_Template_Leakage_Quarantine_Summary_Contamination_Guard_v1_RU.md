**Статус документа:** ready for IDE agent  
**Язык:** русский

**Тип PRD:** targeted hotfix после audit-only blocker  
**Главный принцип:** убрать активные вшитые шаблонные ответы без создания новых заглушек и без поломки уже улучшившегося живого поведения Writer.

## 0. Краткое резюме

`PRD-047.14` завершился честным audit-only blocker: активная template-family leakage найдена в runtime-файле:

```text
bot_psychologist/bot_agent/multiagent/concrete_answer_fit.py
```

Проблема: в активном runtime есть hardcoded fallback/helper, который возвращает готовый пользовательский ответ с повторяющимся skeleton:

```text
в твоем описании важно не свести все к одному общему механизму...
Сначала отдели факты от вывода...
Затем найди центральное убеждение...
После этого проверь...
Практический смысл распутывания...
```

Это нельзя лечить новым hardcoded ответом.  
Цель HF1 — **удалить активный источник шаблонного поведения** и заменить его архитектурно корректной логикой:

```text
bad final answer detected
→ mark as template-family leakage
→ use existing Writer retry / existing final answer gate mechanics
→ do not write contaminated answer into healthy memory / summary / last assistant offer
→ expose trace/debug evidence
→ no canned replacement answer
```

## 1. Контекст

### 1.1. Что установил PRD-047.14

По отчёту `PRD-047.14_IMPLEMENTATION_REPORT.md`:

```text
final_status: blocker
runtime_mutation_status: passed
template_leakage_status: blocker
template_active_hits_count: 6

memory_contamination_status: passed
runtime_modularity_status: warning
next_prd: PRD-047.14-HF1 - Template Leakage Quarantine / Summary Contamination Guard v1
```

Ключевой blocker:

```text
active template-family leakage remains in:
bot_psychologist/bot_agent/multiagent/concrete_answer_fit.py
```

### 1.2. Почему это срочно

У пользователя уже были live-диалоги, где один и тот же skeleton повторялся в разных ходах. Это ломает главный продуктовый принцип NEO:

```text
бот должен быть живым собеседником,
а не системой вшитых шаблонов, заготовок и универсальных ответов.
```

### 1.3. Почему нельзя просто заменить текст другим текстом

Запрещено заменить старый шаблон на новый “более красивый” hardcoded fallback.

Нам не нужен новый статичный ответ вида:

```text
Понимаю, это непросто...
Давай посмотрим...
Вот три шага...
```

Это будет та же ошибка, только с другой формулировкой.

Правильная архитектура:

```text
runtime может детектировать плохой/шаблонный ответ;
runtime может давать Writer короткий internal repair feedback;
runtime не должен сам писать финальный пользовательский ответ вместо Writer.
```

## 2. Цель PRD

### 2.1. Главная цель

Устранить активный runtime-источник template-family leakage и заблокировать попадание таких ответов в healthy memory / summary / last assistant offer.

### 2.2. Конкретные цели

1. Удалить или безопасно преобразовать `build_contextual_no_practice_answer()` так, чтобы активный runtime больше не возвращал готовый hardcoded пользовательский ответ.
2. Найти все call sites этой функции или аналогичных hardcoded answer builders.
3. Убедиться, что в active runtime/prompt/user-facing paths не осталось exact/fuzzy template-family markers.
4. Добавить targeted regression tests против:
   - exact template skeleton;
   - formulaic fallback openings;
   - active hardcoded final-answer factories;
   - memory/summary/last-offer contamination.
5. Добавить/расширить guard, который:
   - обнаруживает template-family leakage в финальном ответе;
   - помечает ответ как contaminated;
   - не даёт contaminated answer попасть в healthy memory / summaries / last assistant offer;
   - не создаёт новый canned answer.
6. Сохранить Writer живым автором финального ответа.
7. Сохранить single runtime path и advisory-only Diagnostic Center.

## 4. Жёсткие архитектурные правила HF1

### 4.1. Абсолютный запрет на заглушки

IDE-агент обязан соблюдать:

```text
Запрещено создавать новые user-facing заглушки, canned replies, static fallback answers,
готовые многострочные ответы, “заготовки” и шаблонные универсальные тексты.
```

Запрещённые паттерны реализации:

```python
return "Понимаю, это непросто..."
return "Давай разберем это по шагам..."
return f"В твоем описании важно..."
DEFAULT_FALLBACK_ANSWER = "..."

```

Запрещено создавать функции, которые по сути являются фабрикой пользовательского финального ответа:

```python
build_*_answer(...)
make_*_reply(...)
fallback_*_response(...)
static_*_answer(...)
```

Исключение: короткие internal-only repair hints для Writer/trace допустимы, если они:

```text
- не являются пользовательским финальным ответом;
- не содержат готовой структуры ответа;
- не содержат numbered skeleton;
- не могут быть напрямую отправлены пользователю;
- явно помечены как internal repair feedback.
```

### 4.2. Что делать с существующими заглушками

```text
1. Не удалять вслепую.

5. Если не используется — удалить мёртвый код и обновить __all__ / imports / tests.
6. Если удаление ломает тесты — тесты надо обновить под правильную архитектуру, а не вернуть заглушку.
```

## 7. Требования к реализации

### 7.1. Удалить активный hardcoded template fallback

В файле:

```text
bot_psychologist/bot_agent/multiagent/concrete_answer_fit.py
```

Нужно устранить активный user-facing hardcoded ответ.

Текущий проблемный участок:

```python
def build_contextual_no_practice_answer(...):
    return (
        "... в твоем описании важно не свести все к одному общему механизму ..."
        "... Сначала отдели факты от вывода ..."
        "... Затем найди центральное убеждение ..."
        "... После этого проверь ..."
        "... Практический смысл распутывания ..."
    )
```

Разрешённые варианты:

#### Вариант A — функция не используется

Если call sites отсутствуют:

```text
- удалить функцию полностью;
- убрать её из __all__;
- удалить связанные тесты/ожидания, если они проверяли старый static answer;
- добавить тест, что функция/символ отсутствует или не экспортируется.
```

#### Вариант B — функция используется

Если call sites есть:

```text
- не возвращать пользовательский ответ;
- заменить функцию на internal repair guidance / structured signal;
- изменить call sites так, чтобы они использовали existing Writer retry / gate feedback;
- не отправлять результат этой функции пользователю напрямую.
```

Пример допустимого направления:

```python
def build_concrete_answer_repair_feedback(...) -> dict[str, Any]:
    return {
        "repair_type": "template_leakage_or_concrete_answer_fit",
        "writer_instruction": "Answer naturally to the user's concrete situation using their actual details. Do not use a fixed step-by-step skeleton.",
        "user_facing": False,
    }
```

Даже этот internal hint должен быть коротким и не должен содержать готовую структуру финального ответа.

### 7.2. Добавить template-family detector / quarantine

Нужно добавить или расширить существующий detector, который ловит template-family leakage в final answer.

Допустимые места:

```text
existing stale_stub_detector
existing final_answer_acceptance_gate
new small module: bot_psychologist/bot_agent/multiagent/template_leakage_guard.py
```

Если создаётся новый модуль, он должен быть маленьким и deterministic.

Detector должен ловить exact/fuzzy family:

```text
"в твоем описании важно не свести все к одному общему механизму"
"Сначала отдели факты от вывода"
"Затем найди центральное убеждение"
"После этого проверь"
"Практический смысл распутывания"
"сейчас полезнее не упражнение"
"сейчас полезнее прямое объяснение"
"исправляю прошлую заготовку"
"автоматический контроль"
```

Важно:

```text
Detector не должен писать новый ответ.
Detector только ставит flags / status / reason.
```

### 7.3. Интеграция с Final Answer Acceptance Gate

Если в текущей архитектуре уже есть final answer acceptance gate, нужно подключить template-family detector туда.

Желаемая логика:

```text
final_answer generated
→ template_family_guard checks answer
→ if no leak: normal path
→ if leak:
    - mark final_answer_acceptance failed or warning according to current gate semantics;
    - trigger existing one Writer retry if available;
    - pass short internal repair feedback to Writer;
    - do not create static fallback answer;
    - if retry still leaks, quarantine from healthy memory/summary/last offer.
```

Запрещено:

```text
if leak:
    return "Извини, я отвечу иначе..."
```

Это новая заглушка. Не делать.

### 7.4. Summary / memory contamination guard

HF1 должен гарантировать:

```text
ответ, помеченный template_family_leakage=true,
не попадает в healthy context memory,
не становится last_assistant_offer,
не закрывает unanswered question as healthy,
не используется как clean summary source.
```

Если уже есть похожий quarantine mechanism после Final Answer Acceptance Gate, нужно расширить его template-family flag.  
Если такого механизма нет — добавить минимальный guard только для template-family contaminated answers.

Trace/debug должен показывать:

```json
{
  "template_family_guard": {
    "version": "template_family_guard_v1",
    "checked": true,
    "leak_detected": true,
    "markers": ["..."],
    "action": "retry|quarantine_from_memory|passed",
    "user_facing_replacement_created": false
  }
}
```

Важно: `markers` не должны включать длинные пользовательские тексты или приватные данные. Только detector markers.

### 7.5. Summary request routing warning

PRD-047.14 нашёл `summary_routing_status=warning_static_risk`, но не доказал active misroute.

В HF1 нужно сделать только минимальную защиту от contamination:

```text
- если summary/recap generation берёт previous assistant answers,
  она не должна использовать answers flagged template_family_leakage.
```

Не нужно в HF1 полностью чинить `summary_request` routing, если это требует изменения Dialogue Act Resolver / Answer Obligation.  
Если при реализации видно, что маленькая точечная правка summary routing безопасна, она допустима только как отдельный clearly documented mini-fix, но не должна разрастаться.

Если routing risk остаётся — указать следующий PRD:

```text
PRD-047.14-HF2 — Summary Request Routing / Answer Obligation Repair v1
```

4. Проверять generic hardcoded final-answer factory patterns:

## 17. Acceptance criteria

HF1 считается passed только если:

```text
1. active template-family hits in runtime/prompt/user-facing paths = 0;
2. `concrete_answer_fit.py` больше не содержит hardcoded user-facing template answer;
3. не создан новый canned/static fallback answer;
4. exact/fuzzy template detector работает;
5. final answer gate или equivalent guard ловит template-family leakage;
6. contaminated answer не попадает в healthy memory / summary / last assistant offer;
7. regression tests passed или failed tests честно классифицированы как blocker с причиной;
8. no runtime path added;
9. no LLM agent added;
10. no Writer prompt overhaul;
11. no KB governance/Chroma mutation;
12. production_ready=false;
13. broad_rollout_allowed=false;
14. normal_user_activation_allowed=false;
15. docs обновлены;
16. implementation report обновлён после финального push;
17. commit и push выполнены в main.
```

HF1 считается blocker, если:

```text
- active template-family hits остаются;
- найден новый hardcoded final answer template;
- тесты показывают, что contaminated answer может попасть в memory/summary/last offer;
- runtime behavior был расширен вне scope;
- implementation report не обновлён после push;
- агент удалил historical evidence без manifest;
- агент “починил” проблему новым шаблонным ответом.
```
