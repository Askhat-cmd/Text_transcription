# PRD-047.14-HF2 — Summary Request Routing / Answer Obligation Repair v1

## 0. Краткое резюме

`PRD-047.14-HF2` — targeted runtime repair после:

```text
PRD-047.14       — Residual Legacy / Runtime Modularity / Template Leakage Audit v1
PRD-047.14-HF1   — Template Leakage Quarantine / Summary Contamination Guard v1
PRD-047.14-HF1.1 — Hardcoded User-Facing Reply Inventory / No-Stub Boundary Audit v1
PRD-047.14-HF1.2 — Hardcoded Reply Removal / Writer Retry Conversion v1
```

После HF1.2 основной high-confidence Writer-level no-stub blocker закрыт:

```text
writer_agent_stub_blockers: 31 -> 0
total_blocker_inventory: 55 -> 24
converted_to_retry_signal_count: 31
new_user_facing_stub_created: false
known_blockers: none
final_status: warning
```

Теперь нужно закрыть ранее найденный routing risk из `PRD-047.14`:

```text
summary/recap request может ошибочно классифицироваться как confirmation_to_last_offer
и получать answer_obligation=answer_last_offer
вместо dedicated summary obligation.
```

Цель HF2:

```text
Явный пользовательский запрос на итог / резюме / summary текущей беседы
должен получать dedicated route:

dialogue_act = summary_request
answer_obligation = summarize_current_conversation

и не должен перехватываться confirmation_to_last_offer / answer_last_offer.
```

Главное архитектурное правило:

```text
Runtime определяет обязанность ответа.
Writer пишет само резюме.
Runtime не создаёт canned summary.
Runtime не вставляет готовый шаблон итога.
```

---

## 1. Архитектурный контекст

Целевая архитектура NEO:

```text
Writer = главный автор финального ответа.

Остальные агенты:
- держат линию разговора;
- формируют память;
- диагностируют состояние;
- собирают релевантный контекст;
- помогают Writer понять текущую обязанность ответа;
- валидируют/карантинируют плохой ответ;
- но не пишут финальный смысловой ответ вместо Writer.
```

HF2 должен продолжить этот принцип.

Нельзя возвращаться к старой архитектуре:

```python
if user_asks_summary:
    return "Вот краткий итог нашей беседы..."
```

Правильная архитектура:

```text
if user_asks_summary:
    dialogue_act = summary_request
    answer_obligation = summarize_current_conversation
    writer_context includes compact conversation summary material
    Writer writes final summary
    Gate validates that it is really a summary
```

---

## 2. Проблема

В `PRD-047.14` была зафиксирована routing-risk находка:

```text
User: "Подведи краткий итог нашей беседы"

Observed internal risk:
dialogue_act = confirmation_to_last_offer
answer_obligation = answer_last_offer

Expected:
dialogue_act = summary_request
answer_obligation = summarize_current_conversation
```

Почему это опасно:

```text
1. Внешне ответ может выглядеть приемлемым.
2. Но внутреннее состояние закрывает не ту обязанность.
3. Last assistant offer может быть ошибочно принят как подтверждённый.
4. Summary может быть не сохранён/не классифицирован корректно.
5. Следующие turns могут наследовать неправильную линию.
6. Writer получает неверный directive.
```

HF2 должен устранить именно routing/obligation problem, а не полировать стиль summary answer.

## 7. Detection requirements

### 7.1. Summary request markers

Добавить/укрепить detector `detect_summary_request(...)`.

Он должен распознавать русские и английские формулировки:

```text
подведи итог
краткий итог
итог нашей беседы
итог разговора
сделай резюме
резюмируй
суммируй
к чему мы пришли
что мы поняли
собери кратко
собери всё вместе
дай summary
summary
recap
summarize
summarise
brief recap
conversation summary
what did we cover
what have we concluded
```

Также:

```text
"Итог?"
"Кратко итог"
"Дай вывод"
```

но только если context указывает на текущий диалог, а не на внешний документ.

### 7.2. Negative / not-summary cases

Не считать summary request:

```text
"Итого 5 тысяч?"                         — arithmetic/confirmation
"В итоге что мне делать прямо сейчас?"    — direct step request
"Сделай вывод из этой цитаты"             — external text analysis, not conversation summary
"Подведи итоги проекта за год"            — external report request
"Да, хорошо"                              — contextual confirmation, not summary by itself
"Хорошо, давай"                           — confirmation to last offer
"Итог: я согласен"                        — user statement, not request
```

### 7.3. Conflict priority

If explicit summary markers are present, summary must outrank:

```text
confirmation_to_last_offer
answer_last_offer
continue_previous_offer
followup_confirmation
practice_request
clarification_request
```

Unless user explicitly asks for a step/action:

```text
"Кратко итог и один следующий шаг"
```

Then allowed combined obligation:

```text
answer_obligation = summarize_current_conversation_then_offer_one_next_step
```

HF2 may implement only base route and classify combined cases as warning/next scope if needed.

## 9. Answer obligation requirements

Add/repair obligation:

```text
answer_obligation = summarize_current_conversation
```

Expected payload:

```json
{
  "answer_obligation": "summarize_current_conversation",
  "answer_required": true,
  "must_answer": "summarize current conversation",
  "must_not": [
    "ask for confirmation before summarizing",
    "answer last offer instead of summary",
    "start a new practice unless explicitly requested",
    "write a canned summary"
  ],
  "writer_should": [
    "summarize the current conversation",
    "keep it concise unless user asked otherwise",
    "preserve live context and user goal",
    "avoid inventing facts not present in conversation"
  ]
}
```

Important:

```text
This is metadata/directive, not final text.
```

## 10. Final Answer Directive requirements

`final_answer_directive_v1` must expose summary obligation to Writer.

Expected directive fields:

```json
{
  "final_answer_directive": {
    "answer_obligation": "summarize_current_conversation",
    "must_answer": "summary of current conversation",
    "summary_request": true,
    "summary_scope": "current_conversation",
    "no_confirmation_needed": true,
    "no_practice_unless_requested": true
  }
}
```

Do not insert a fixed summary template.

Allowed instruction:

```text
Write a concise summary of the conversation so far, using only available conversation context.
```

Forbidden instruction:

```text
Return:
1. Мы обсудили...
2. Главный вывод...
3. Следующий шаг...
```

That would become a static answer shape.

## 11. Writer contract / prompt context requirements

Writer should receive:

```text
- user asked for conversation summary;
- summary scope;
- recent conversation context;
- memory/context package if already available;
- do not answer last offer;
- do not ask whether user wants summary;
- do not add practice unless user requested;
- no canned summary template.
```

Writer must remain free to phrase the summary naturally.

---

## 12. Last assistant offer / unanswered state requirements

When route is `summary_request`:

```text
- do not treat user turn as confirmation_to_last_offer;
- do not close last offer as accepted unless user explicitly confirms it separately;
- do not seed new last assistant offer from summary unless summary itself contains an offer;
- do not mark unrelated previous direct question as answered unless summary satisfies it;
- summary final answer can be saved as healthy context only if gate passes.
```

## 13. Final Answer Acceptance Gate requirements

Add summary-specific checks without creating summary text.

Gate should check:

```text
if answer_obligation == summarize_current_conversation:
    - final answer should not be empty;
    - final answer should not ask "Хотите, чтобы я подвёл итог?";
    - final answer should not answer an old offer instead of summary;
    - final answer should include at least one conversation/context anchor if context exists;
    - final answer should not introduce unsupported new facts;
```

Possible failed checks:

```text
summary_request_not_answered
summary_request_reconfirmed_instead_of_answered
summary_answered_last_offer_instead
summary_too_generic_without_context_anchor
```

On failure:

```text
retry_recommended = true
must_quarantine_answer = true
can_save_as_healthy_context = false
can_use_as_summary_source = false
can_save_last_assistant_offer = false
```

No replacement text.
