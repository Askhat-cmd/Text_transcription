## 0. Краткое резюме

Этот PRD запускает **audit-only этап** перед любыми новыми улучшениями живого поведения бота.

После `PRD-047.13-HF1` активный cleanup noise закрыт, единая policy surface зафиксирована, runtime behavior не менялся, а пользователь сообщил, что текущее живое поведение бота стало заметно лучше. Поэтому следующий шаг **не должен сразу менять Writer, prompts, orchestrator, Dialogue Act Resolver, Final Answer Gate, RAG, Diagnostic Center или runtime path**.

Главная задача PRD-047.14:

```text
найти и классифицировать остаточные источники:
- template-family leakage;
- summary/recap request misrouting;
- residual legacy/runtime split noise;
- runtime modularity / God-object risk;
- docs/trace/admin mismatch;
- stale prompt/test/evidence artifacts,
без исправления runtime behavior.
```

Итогом PRD должен стать не “починенный бот”, а **честный audit pack**, по которому архитектор сможет выбрать следующий targeted PRD:

```text
PRD-047.14-HF1 — если найден active template leakage;
PRD-047.14-HF2 — если подтверждён summary request routing bug;
PRD-047.15 — если активных blockers нет и можно идти в live dialogue polish;
или отдельный docs/runtime-modularity PRD, если проблема не в ответах, а в структуре.
```

### 1.3. Главная новая находка: template-family leakage

В пользовательских live-диалогах после cleanup обнаружен повторяющийся шаблон ответа:

```text
в твоем описании важно не свести все к одному общему механизму...
Сначала отдели факты от вывода...
Затем найди центральное убеждение...
После этого проверь...
Практический смысл распутывания...
```

Этот шаблон мог появляться:

```text
- после первичного запроса про страх сокращения;
- после уточнения про страх потерять работу;
- после фразы про событие → мысль → тело;
- в recent summarized turns;
- в last assistant offer / context;
- как заражающий fragment для следующих ответов.
```

Архитектурная трактовка:

```text
это не cleanup bug;
это не доказательство провала PRD-047.13-HF1;
это residual runtime quality issue:
Template-family leakage / repeated answer skeleton / summary contamination.
```

### 1.4. Вторая находка: summary/recap request routing risk

Есть риск, что запросы вида:

```text
Подведи краткий итог нашей беседы.
Кратко резюмируй, к чему мы пришли.
Собери выводы из нашего диалога.
```

могут быть внутренне классифицированы не как summary request, а как:

```text
dialogue_act=confirmation_to_last_offer
answer_obligation=answer_last_offer
```

При этом финальный ответ может выглядеть приемлемо, но routing будет неправильным. Это опасно, потому что система может внешне “отвечать нормально”, но внутренне двигаться не той answer obligation.
