## 1. Архитектурный контекст

Целевая система NEO:

```text
Writer пишет основной ответ пользователю.

Остальные агенты:
- держат линию разговора;
- помогают видеть состояние пользователя;
- формируют память;
- диагностируют текущий контекст;
- решают, нужен ли RAG/БД;
- отсекают шум;
- валидируют ответ;
- но не пишут финальный user-facing ответ вместо Writer.
```

Composer добавлен как внутренний retrieval-brain:

```text
current user turn + dialogue context
→ retrieval intent
→ contextual query plan
→ suppress/include RAG decision
→ compact writer context package
→ Writer writes final answer
```

Composer v1 сейчас работает не через LLM, а через deterministic heuristics.

HF1 должен ответить на вопрос:

```text
Достаточно ли такой эвристики для живых диалогов,
или нужен гибридный/LLM-assisted слой?
```

---

## 2. Главный вопрос HF1

Нужно проверить не только “тесты зелёные”, а живое качество:

```text
1. Composer понимает короткие согласия?
2. Composer правильно наследует тему из last assistant offer?
3. Composer не ищет literal "Да, хорошо"?
4. Composer не тащит КБ в summary request?
5. Composer не подавляет RAG там, где он нужен?
6. Composer не перегружает Writer лишними chunks?
7. Composer виден в trace так, чтобы владелец проекта мог понять решение?
8. Composer помогает Writer, а не мешает ему?
```

---

## 3. Важное решение: HF1 не добавляет LLM вызовы

В этом PRD запрещено добавлять:

```text
- новый LLM agent;
- LLM call inside Composer;
- LLM reranker;
- LLM query rewriting;
- LLM judge;
- model/provider config changes;
- new runtime path;
- new user-facing fallback text.
```

Причина:

```text
Сначала нужен owner-trace evidence.
Без evidence мы не поймём, что именно должен делать LLM:
- распознавать inherited topic?
- переписывать query?
- оценивать noise?
- выбирать chunks?
- решать include/suppress?
- или только объяснять uncertainty?
```

HF1 должен подготовить решение, а не внедрять его.

## 12. LLM / Hybrid future decision brief

Create:

```text
TO_DO_LIST/logs/PRD-047.15-HF1/llm_hybrid_decision_brief.md
TO_DO_LIST/logs/PRD-047.15-HF1/llm_hybrid_decision_brief.json
```

This is mandatory.

It must compare three architectures:

### 12.1. Option A — Heuristic-only Composer

Description:

```text
Current deterministic v1 with rule improvements.
```

Pros:

```text
- predictable;
- cheap;
- no latency;
- easy to test;
- stable trace;
- low risk of hallucinated retrieval intent;
- no extra provider dependency.
```

Cons:

```text
- brittle on subtle context;
- needs many rules;
- weak on ambiguous mixed cases;
- may miss implicit user intent;
- manual maintenance grows over time.
```

Best use:

```text
- clear routing;
- summary suppression;
- greetings/close/support;
- obvious knowledge questions;
- obvious short follow-up with strong last offer.
```

### 12.2. Option B — Hybrid Heuristics + LLM-assisted Composer

Description:

```text
Heuristics handle high-confidence cases.
LLM is called only for ambiguous/mixed/low-confidence cases.
```

Pros:

```text
- keeps deterministic safety for simple cases;
- improves ambiguous context;
- controls cost/latency;
- allows better query rewriting;
- can output structured JSON only;
- easier to gate than LLM-first.
```

Cons:

```text
- more complex architecture;
- needs JSON validation;
- needs timeout/fallback;
- must prevent user-facing text;
- needs cost/latency monitoring;
- possible model drift.
```

Best use:

```text
- mixed cases;
- "да, но на моём примере";
- implicit topic shifts;
- weak inherited topic;
- query rewrite;
- KB vs memory decision.
```

### 12.3. Option C — LLM-first Composer

Description:

```text
LLM decides retrieval need/query/inclusion for most turns.
```

Pros:

```text
- highest flexibility;
- better semantic interpretation;
- fewer brittle rules;
- can understand complex implicit context.
```

Cons:

```text
- higher cost;
- higher latency;
- less deterministic;
- harder to test;
- possible hallucinated query;
- provider dependency;
- more safety/validation required;
- may become hidden authority over Writer.
```

Best use:

```text
- only after trace evidence proves deterministic/hybrid insufficient.
```

### 12.4. Required recommendation logic

Decision brief must recommend one of:

```text
keep_heuristic_v1_and_tune
build_hybrid_llm_assist_for_low_confidence_cases
design_llm_first_composer_experiment
defer_llm_until_more_live_data
```

Default expected recommendation:

```text
build_hybrid_llm_assist_for_low_confidence_cases
```

But final recommendation must be evidence-based from trace results.

## 13. Future hybrid architecture sketch

If decision is hybrid, sketch future architecture:

```text
Composer deterministic pre-pass
→ if high confidence:
      use heuristic decision
→ if low confidence / mixed:
      call LLM Query Composer with strict JSON schema
→ validate JSON
→ apply no-user-facing-text gate
→ merge into retrieval_decision
→ Writer receives compact context
```

LLM output schema must be only metadata:

```json
{
  "retrieval_need": "...",
  "retrieval_action": "...",
  "query_source": "...",
  "composed_query": "...",
  "query_terms": [],

  "writer_can_ignore_rag": true,
  "include_for_writer_if_found": false,
  "reason": "...",
  "no_user_facing_text_created": true
}
```

```text
final answer
advice to user
therapy phrase
canned explanation
full summary text
```

## 16. No-stub rule

HF1 must preserve no-stub rule.

Forbidden:

```text

- any canned Composer explanation;
- any static final answer;
- any LLM prompt that asks model to write user-facing text;
- any owner review artifact that is later consumed by runtime as answer text.
```

Allowed:

```text

- labels;
- expected categories;
- owner notes;
- query terms;
```
