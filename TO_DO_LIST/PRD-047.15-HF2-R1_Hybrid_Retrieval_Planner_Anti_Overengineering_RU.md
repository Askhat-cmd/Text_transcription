## 0. Краткое резюме

Этот PRD реализует узкий и безопасный этап `PRD-047.15-HF2-R1`: **Hybrid Retrieval Planner / Query-Before-RAG**.

Главная цель — изменить порядок retrieval так, чтобы система сначала понимала **нужно ли искать**, **что именно искать**, **какой тип знания нужен**, **какая глубина допустима**, и только потом выполняла RAG/Memory retrieval.

Ключевой принцип после нового Strategic Plan v2:

```text
Planner не становится новым “мозгом” бота.
Planner не пишет ответ пользователю.
Planner не командует Writer’ом.
Planner только готовит retrieval metadata и мягкий контекст.
Writer остаётся единственным автором финального ответа.
```

HF2-R1 должен прекратить рост доменных эвристик в Composer и заменить их на:

```text
Minimal Universal Heuristics
+ optional metadata-only LLM Retrieval Planner for complex/mixed cases
+ approved query before RAG
+ trace-visible chunk routing
+ Writer freedom preserved
```

## 1. Контекст

### 1.1. Почему этот PRD нужен

В текущей архитектуре retrieval может выполняться слишком рано: `MemoryRetrievalAgent` строит RAG query из `user_message`, `thread_state` и частичных сигналов до того, как semantic planner/composer успел сформировать более точный запрос.

В результате:

```text
- RAG может искать по буквальной фразе вместо живого механизма;
- короткие ответы “да”, “давай”, “на моём примере” могут искать мусор;
- сложные запросы требуют семантического планирования, но решаются эвристиками;
- Composer может красиво описать query в trace, когда реальный RAG уже выполнен по другому query;
- Writer получает слабые или нерелевантные chunks;
- слабый retrieval маскируется как проблема Writer’а.
```

HF2-R1 должен исправить именно это место: **query planning должен происходить до фактического RAG-запроса**.

### 1.2. Почему это R1, а не старый HF2

После обсуждения риска переусложнения проекта был принят новый курс:

```text
- не добавлять бесконечные эвристики;
- не превращать диагностику в систему команд;
- не делать Planner “начальником Writer’а”;
- не строить сложную внутреннюю ветвистость без evidence;
- сохранять простоту, модульность, trace-аудит и настраиваемость.
```

Поэтому этот PRD является revision-версией HF2: он должен быть уже, чище и безопаснее.

## 2. Цель

Реализовать metadata-only Hybrid Retrieval Planner в active multiagent runtime так, чтобы:

```text
1. Простые universal cases решались без LLM.
2. Сложные/mixed/low-confidence cases могли вызвать LLM Planner.
3. Planner возвращал только strict JSON retrieval_plan.
4. approved retrieval_plan управлял первичным RAG query.
5. MemoryRetrievalAgent выполнял RAG по composed_query из retrieval_plan, если он есть.
6. Writer получал компактный, governance-filtered retrieval context.
7. Writer мог игнорировать retrieval, если живой ответ лучше.
8. Все решения были видны в trace/API/Web Trace surfaces.
```

Главный acceptance result:

```text
Для query_kb/query_memory/query_kb_and_memory executed_rag_query должен совпадать с approved retrieval_plan.composed_query, а не строиться legacy _build_rag_query до Planner.
```

## 5. Архитектурное решение HF2-R1

### 5.1. Новый целевой поток

```text
User message
→ lightweight context signals
→ Universal Retrieval Gates
→ if simple/high-confidence: deterministic retrieval_plan
→ if complex/mixed/low-confidence: metadata-only LLM Retrieval Planner
→ strict JSON validation
→ approved retrieval_plan
→ MemoryRetrievalAgent.assemble(..., retrieval_plan=retrieval_plan)
→ RAG/Memory execution по approved composed_query
→ score/policy/chunk gate
→ Writer Context Package
→ Writer
→ Validator / quality_trace
→ API / Web Trace
```

### 5.2. Главное правило query-before-RAG

Если есть approved `retrieval_plan` с action:

```text
query_kb
query_memory
query_kb_and_memory
```

то `MemoryRetrievalAgent` обязан использовать:

```text
retrieval_plan.composed_query
```

как первичный query.

Legacy `_build_rag_query(user_message, thread_state)` разрешён только как fallback:

```text
- если retrieval_plan отсутствует;
- если retrieval_plan invalid;
- если feature flag выключен;
- если action требует fallback_if_invalid=legacy_query;
- если тест явно проверяет backwards compatibility.
```

### 5.3. Planner не является Writer

Planner не может возвращать:

```text
- user-facing answer;
- совет пользователю;
- готовую фразу ответа;
- терапевтическую интерпретацию для прямой вставки;
- “скажи пользователю...”;
- “ответ должен быть...” как текст ответа.
```

Planner возвращает только machine-readable metadata.

## 6. Universal Retrieval Gates

Universal gates — deterministic, content-neutral, не доменные. Они нужны, чтобы не вызывать LLM и RAG там, где это не нужно.

### 6.1. Допустимые universal gates

| Gate | Пример | Action | LLM |
| --- | --- | --- | --- |
| greeting/contact | “привет”, “здравствуй” | `suppress_rag` | no |
| thanks/farewell | “спасибо”, “пока” | `suppress_rag` | no |
| explicit no/reject | “нет”, “не хочу” | `use_current_context_only` | no/rare |
| formatting request | “короче”, “списком”, “без воды” | `use_current_context_only` | no |
| summary request | “подведи итог” | `use_current_context_only` | no |
| short support request | “просто поддержи” | `suppress_rag` or `use_current_context_only` | no |
| no-theory/no-practice constraint | “без теории”, “без практик” | constraint only | no/rare |
| safety/crisis | самоповреждение/опасность | safety path | existing safety |
| clear explicit KB ask | “что такое X?” | may `query_kb` with literal cleaned query | usually no, if high-confidence |

### 6.2. Простые случаи без LLM

Простые случаи должны решаться без LLM Planner:

```text
- приветствие;
- спасибо/пока;
- короткое “да” после однозначного last assistant offer;
- короткое “нет”;
- просьба сократить/переформатировать;
- подведение итога текущего диалога;
- просьба “без теории / без практик”;
- очевидный current-context-only follow-up.
```

### 6.3. Сложные случаи с возможным LLM Planner

LLM Planner допустим только если deterministic gates не дают уверенного решения.

Примеры:

```text
- “да, но на моём примере”;
- “давай, только без теории” после сложного предложения ассистента;
- короткое “да” после ambiguous previous offer;
- “а как это связано с тем, что я говорил раньше?”;
- запрос одновременно про практику, объяснение и личный контекст;
- запрос требует выбора между KB и memory;
- literal query будет плохим, потому что нужны mechanism hints;
- user message поверхностно про одно, а active thread указывает на другое;
- low confidence в universal gate;
- mixed constraints: “коротко, но с примером из моей ситуации”.
```

## 7. Mechanism-aware retrieval hints без перестройки БД

HF2-R1 должен добавить в retrieval_plan поля, которые помогут будущей БД, но не требуют сейчас полной перестройки Bot_data_base.

### 7.1. Почему нужны mechanism-aware hints

Для психологического бота поиск по словам недостаточен.

Правильная цепочка:

```text
слова пользователя
→ эмоциональный маркер
→ возможный механизм
→ допустимая глубина
→ тип нужного чанка
→ безопасная дозировка
→ Writer context
```

Пример:

```text
“Если я перестану контролировать, то умру”
```

Не должен искать только `контролировать` и `умру`.

Он должен распознать possible mechanism:

```text
control_as_safety
loss_of_control_as_threat
helplessness_shame
```

### 7.2. Новые поля retrieval_plan

Добавить в schema retrieval_plan поля:

```json
{
  "needed_chunk_types": ["concept", "mechanism", "dialogue_move", "practice", "safety", "diagnostic_lens", "source_fragment"],
  "avoided_chunk_types": [],
  "mechanism_hints": [],
  "depth_level_hint": 0,
  "safety_layer_required": false,
  "allowed_use_filter_hint": ["writer_support", "diagnostic_hint", "practice_suggestion"],
  "diagnostic_hints_used": false,
  "writer_can_ignore_rag": true,
  "retrieval_gap_reason": ""
}
```

### 7.3. Chunk type vocabulary v0

Не менять БД, но зафиксировать vocabulary v0 для Planner/Trace:

```text
concept          — понятие / короткая смысловая рамка
mechanism        — психологический механизм / паттерн
dialogue_move    — возможный ход ответа / способ отражения
practice         — упражнение / практика / микрошаг
safety           — safety boundary / grounding / медицинские/кризисные ограничения
diagnostic_lens  — внутренняя линза для диагностического слоя
source_fragment  — обычный фрагмент источника / raw-ish content after governance
general_text     — fallback, если тип неизвестен
```

### 7.4. Retrieval gap visibility

Если Planner запросил `needed_chunk_types`, а найденные chunks не имеют таких типов или метаданных, trace должен показать:

```text
retrieval_gap_reason = "missing_chunk_type_metadata"
```

Другие допустимые причины:

```text
no_hits
low_score_only
policy_rejected_all
budget_truncated
missing_mechanism_chunk
missing_dialogue_move_chunk
planner_suppressed_rag
legacy_fallback_used
```

## 8. LLM Retrieval Planner contract

### 8.1. Версия

Ввести версию:

```python
HYBRID_RETRIEVAL_PLANNER_VERSION = "hybrid_retrieval_planner_v1_r1"
```

### 8.2. Feature flags / modes

Добавить безопасные modes:

```text
off        — Planner отключён, legacy behavior
shadow     — Planner строит retrieval_plan и trace, но не управляет RAG
apply      — Planner управляет RAG query для allowlisted/test/local режимов
```

Default должен быть безопасным. Если текущая проектная политика требует no production change, использовать:

```text
HYBRID_RETRIEVAL_PLANNER_MODE=shadow
```

Для tests/runner разрешить явно включать `apply`.

### 8.3. Strict JSON schema

Planner output должен быть strict JSON.

Минимальный contract:

```json
{
  "version": "hybrid_retrieval_planner_v1_r1",
  "mode": "off|shadow|apply",
  "planner_called": true,
  "llm_called": false,
  "universal_gate": "none|greeting|thanks|farewell|explicit_yes|explicit_no|summary_request|formatting_request|short_support|safety|clear_kb_ask|other",
  "retrieval_need": "none|conversation_context|knowledge_context|memory_context|kb_and_memory|trace_only",
  "retrieval_action": "suppress_rag|use_current_context_only|query_kb|query_memory|query_kb_and_memory|trace_only",
  "query_source": "current_user_message|last_assistant_offer|recent_turns|thread_state|active_line|mixed_context|fallback",
  "composed_query": "",
  "query_terms": [],
  "query_language": "ru|en|mixed|unknown",
  "needed_chunk_types": [],
  "avoided_chunk_types": [],
  "mechanism_hints": [],
  "depth_level_hint": 0,
  "safety_layer_required": false,
  "allowed_use_filter_hint": [],
  "needs_kb": false,
  "needs_memory": false,
  "needs_recent_turns": true,
  "include_for_writer_if_found": false,
  "writer_can_ignore_rag": true,
  "max_chunks_for_writer": 0,
  "max_chars_per_chunk": 0,
  "confidence": 0.0,
  "reason": "metadata-only reason",
  "risk_flags": [],
  "fallback_if_invalid": "suppress_rag|use_current_context_only|legacy_query|trace_only",
  "no_user_facing_text_created": true
}
```

### 8.4. Validation rules

Reject Planner output if:

```text
- not valid JSON;
- missing required fields;
- no_user_facing_text_created != true;
- composed_query contains advice-like prose;
- reason contains user-facing answer text;
- retrieval_action unknown;
- confidence not numeric 0..1;
- max_chunks_for_writer > configured hard max;
- max_chars_per_chunk > configured hard max;
- needed_chunk_types contains unknown type without mapping to general_text;
- safety_layer_required=false while existing safety classifier indicates safety path.
```

### 8.5. Fallback behavior

On invalid/timeout/low-confidence:

```text
- do not fail user response;
- record planner_error / fallback_used in trace;
- use universal deterministic decision if available;
- otherwise use_current_context_only or legacy_query based on fallback_if_invalid;
- never use invalid Planner text as answer.
```

## 9. LLM Planner prompt requirements

Если LLM Planner вызывается, prompt должен быть компактным и metadata-only.

### 9.1. Planner system instruction

Использовать смысл такого системного сообщения:

```text
Ты Hybrid Retrieval Planner для психологического диалогового бота.
Ты НЕ отвечаешь пользователю.
Ты НЕ пишешь финальный ответ.
Ты НЕ даёшь совет.
Ты НЕ ставишь диагноз.
Ты НЕ являешься терапевтом.

Твоя задача — вернуть только JSON retrieval_plan:
- нужен ли retrieval;
- какой retrieval_action выбрать;
- какой composed_query выполнить до RAG;
- какие chunk types желательны;
- какие mechanism hints могут помочь поиску;
- какая глубина допустима;
- какие ограничения передать Writer’у как metadata.

Writer остаётся единственным автором ответа и может игнорировать retrieval.
Верни только JSON. Никакого markdown, никаких пояснений вне JSON.
```

### 9.2. Planner input должен быть compact

Передавать только:

```json
{
  "user_message": "...",
  "recent_turns_compact": [],
  "last_assistant_offer": {},
  "thread_state_compact": {},
  "state_snapshot_compact": {},
  "dialogue_pragmatics": {},
  "fresh_chat_policy": {},
  "constraints": [],
  "available_chunk_type_vocab": []
}
```

Не передавать:

```text
- полный raw chat;
- raw private logs;
- raw provider payload;
- full KB chunks;
- hidden chain-of-thought;
- secrets;
- API keys;
- длинные prompt canvases целиком.
```

## 10. MemoryRetrievalAgent changes

### 10.1. Backwards-compatible signature

Изменить сигнатуру без ломки старых вызовов:

```python
assemble(..., retrieval_plan: dict | None = None, ...)
```

Или эквивалентно через dataclass/contract, если в проекте уже есть подходящий contract layer.

### 10.2. Execution rules

Внутри retrieval execution:

```text
if retrieval_plan.mode == "apply" and retrieval_action in query_*:
    executed_rag_query = retrieval_plan.composed_query
elif retrieval_plan.retrieval_action == "suppress_rag":
    skip external KB retrieval
elif retrieval_plan.retrieval_action == "use_current_context_only":
    use recent turns/profile/thread only; skip external KB
elif retrieval_plan.mode == "shadow":
    run current legacy behavior but record planned_query vs executed_rag_query
else:
    use legacy _build_rag_query fallback
```

### 10.3. Trace proof fields

Memory bundle / debug must include:

```json
{
  "retrieval_plan_applied": true,
  "retrieval_plan_mode": "shadow|apply|off",
  "planned_composed_query": "...",
  "executed_rag_query": "...",
  "legacy_rag_query": "...",
  "query_before_rag_proof": true,
  "rag_skipped_reason": "...",
  "retrieval_gap_reason": "..."
}
```
