## 0. Краткое резюме

`PRD-047.15-HF2-R1` добавил `hybrid_retrieval_planner_v1_r1`: metadata-only Hybrid Retrieval Planner, query-before-RAG, режимы `off|shadow|apply`, trace/debug/API поля, `MemoryRetrievalAgent.assemble(..., retrieval_plan=...)`, writer boundary и no-mutation proof.

Текущий разрыв: backend/debug/API уже умеют отдавать поля Hybrid Retrieval Planner, но Web Admin / Web Trace ещё не показывают их в понятном виде разработчику. Из-за этого владелец проекта не может быстро увидеть:

- почему Planner решил искать или не искать;
- какой `retrieval_action` выбран;
- какой `planned_composed_query` был утверждён;
- какой `executed_rag_query` реально ушёл в RAG;
- чем он отличается от `legacy_rag_query`;
- был ли `query_before_rag_proof=true`;
- был ли `fallback_used`;
- какие `needed_chunk_types`, `mechanism_hints`, `depth_level_hint` и `writer_can_ignore_rag` были переданы Writer’у;
- какая модель используется для optional LLM Planner path.

Цель HF2-R2 — **синхронизировать Backend ↔ Web Admin ↔ Web Trace по видимости Hybrid Retrieval Planner** и одновременно убрать/изолировать очевидный visual noise в Admin UI, который мешает видеть актуальный runtime. Смысловую логику Planner/RAG/Writer менять нельзя.

## 1. Главный принцип

Этот PRD не должен превращаться в новый backend hotfix и не должен менять поведение бота.

Формула:

```text
HF2-R1 сделал query-before-RAG.
HF2-R2 должен сделать query-before-RAG видимым.
```

То есть задача HF2-R2:

```text
не улучшать Planner,
не менять Writer,
не перестраивать БД,
а показать в UI/Web Trace то, что backend уже знает.
```

## 2. Ответ на вопрос о LLM Planner

### 2.1. Текущее состояние

Optional LLM path для Hybrid Retrieval Planner уже реализован в HF2-R1:

- universal deterministic gates решают простые случаи без LLM;
- для complex/low-confidence cases Planner может вызвать LLM;
- LLM должна вернуть strict JSON retrieval_plan;
- invalid JSON / unsafe payload / error → safe fallback;
- Writer остаётся единственным автором пользовательского ответа.

Но live smoke HF2-R1 проверял в основном universal-gate path и query-before-RAG visibility, а не полноценный real-provider complex LLM path. Поэтому в HF2-R2 нужно **не переписывать LLM Planner**, а сделать его состояние, модель и trace видимыми.

### 2.2. Модель для LLM Planner

Hybrid Retrieval Planner должен использовать модель:

```text
gpt-5-nano
```

как и другие лёгкие классификаторы/служебные reasoning-компоненты проекта.

Требования:

- default value: `HYBRID_RETRIEVAL_PLANNER_MODEL=gpt-5-nano`;
- Web Admin должен показывать effective planner model;
- runtime effective API должен отдавать `hybrid_retrieval_planner.model`;
- Web Trace должен показывать, был ли LLM вызван (`llm_called`) и, если доступно, какой planner model использовался;
- HF2-R2 не должен менять основную Writer model;
- HF2-R2 не должен добавлять отдельный новый LLM-agent;
- HF2-R2 не должен делать LLM Planner обязательным для всех turns.

## 3. Non-goals / запреты

В рамках HF2-R2 запрещено:

1. Менять алгоритм Hybrid Retrieval Planner.
2. Менять смысл universal gates.
3. Менять `_build_rag_query` fallback-логику, кроме read-only отображения legacy query.
4. Менять `MemoryRetrievalAgent` retrieval execution semantics.
5. Включать `apply` как новый production/default режим.
6. Делать broad rollout / normal-user activation.
7. Менять Writer prompt philosophy или Writer authority.
8. Добавлять новый LLM agent.
9. Добавлять новый runtime path.
10. Мутировать DB schema, KB governance, Chroma index или chunk storage.
11. Перестраивать БД или делать LLM enrichment чанков.
12. Добавлять domain-specific hardcoded psychological heuristics в frontend/backend visibility layer.
13. Показывать raw unsafe `content_full` или приватные prompts обычному пользователю.
14. Удалять или переименовывать исторические docs без отдельного cleanup PRD.
15. Исправлять все старые UI проблемы админки за пределами bounded admin-surface hygiene, описанной в разделе 7.3.
16. Агрессивно удалять backend config/runtime code для legacy-переменных без отдельного proof. В HF2-R2 допускается скрытие, read-only deprecation или перенос в Compatibility, но не слепое удаление.

## 7. Web Admin требования

На текущем скриншоте Web Admin открыта вкладка `Расширенные элементы управления → LLM`, где видны основная модель GPT и модель классификатора, но не видно отдельного статуса Hybrid Retrieval Planner.

HF2-R2 должен добавить в Web Admin read-only блок:

```text
Hybrid Retrieval Planner
```

Рекомендуемое размещение:

- либо во вкладке `LLM` как read-only service model block;
- либо во вкладке `Извлечение` / `Retrieval`, если там логически уже живёт RAG;
- либо в отдельном компактном блоке в `Расширенные элементы управления`, если архитектура UI это поддерживает.

### 7.1. Что показывать в Admin UI

```text
Version: hybrid_retrieval_planner_v1_r1
Mode: shadow / apply / off
Default safe mode: shadow
Model: gpt-5-nano
Max tokens: 320
Metadata-only: yes
Query-before-RAG: supported
Writer final author: preserved
Allowed modes: off, shadow, apply
LLM path: optional for complex/low-confidence cases
Production ready: false
Broad rollout allowed: false
Normal user activation allowed: false
```

### 7.2. Редактирование

По умолчанию HF2-R2 не должен добавлять редактирование режима Planner из UI.

Допустимо показать read-only effective values. Если в проекте уже есть безопасный runtime config control pattern, можно добавить read-only + warning, но не полноценное управление режимом без отдельного PRD.

Запрещено случайно дать normal user activation или сделать apply default.


### 7.3. Owner admin-surface hygiene addendum

На owner screenshots перед передачей HF2-R2 в IDE выявлен дополнительный риск: Admin UI уже содержит устаревшие или дублирующиеся элементы управления, из-за которых новый Hybrid Retrieval Planner будет плохо виден и владелец проекта будет путаться между актуальными и историческими слоями.

HF2-R2 должен включить **ограниченную UI-hygiene правку**, но только в границах visibility/admin-surface sync. Это не backend cleanup PRD и не удаление runtime logic.

2. `Knowledge Graph`
   - если Knowledge Graph действительно удалён/неактивен, убрать editable checkbox из Runtime;

#### 7.3.4. Где показывать Hybrid Retrieval Planner после cleanup

Так как Advanced Retrieval может быть скрыт как устаревший/дублирующий UI, новый блок Hybrid Retrieval Planner должен быть размещён в одном из актуальных мест:

Preferred:

```text

```

Alternative:

```text
Memory -> Retrieval Runtime / Hybrid Retrieval Planner
```

Не рекомендуется помещать новый Planner только внутрь старого `Advanced Controls -> Retrieval`, если этот раздел одновременно признаётся устаревшим.

## 8. Web Trace требования

В Web Trace / Debug Trace для каждого turn должен появиться видимый раздел:

```text
Hybrid Retrieval
```

### 8.1. Минимальная структура отображения

```text
Hybrid Retrieval
- Version: hybrid_retrieval_planner_v1_r1
- Mode: shadow/apply/off
- Universal gate: greeting / clear_kb_ask / safety_signal / complex_or_low_confidence / ...
- LLM called: yes/no
- LLM reason: ...
- Fallback used: yes/no
- Plan valid: yes/no
- Retrieval action: suppress_rag / use_current_context_only / query_kb / query_memory / query_kb_and_memory / trace_only
- Planned query: ...
- Executed RAG query: ...
- Legacy RAG query: ...
- Query-before-RAG proof: true/false
- Needed chunk types: mechanism, dialogue_move, safety, ...
- Mechanism hints: control_as_safety, panic_regulation, ...
- Depth level hint: 0/1/2/3
- Writer can ignore RAG: true/false
- Retrieval gap reason: ...
```

### 8.2. Visual status rules

UI должен подсвечивать:

```text
passed / ok:
query_before_rag_proof=true

warning:
mode=shadow and planned_query != executed_query
fallback_used=true
plan_valid=false
retrieval_gap_reason not empty
llm_called=true but planner model unknown

info:
rag suppressed by universal gate
use_current_context_only
```

Важно: `mode=shadow` + `planned_query != executed_query` не является ошибкой. Это нормальная semantics shadow mode. UI должен объяснить:

```text
Shadow mode: planned query is observed but legacy query is executed.
```

### 8.3. Privacy / debug boundary

Web Trace является dev/debug surface. Но всё равно запрещено:

- показывать raw unsafe `content_full` там, где раньше был только preview;
- расширять доступ обычным пользователям;
- добавлять вывод полного Writer system prompt в пользовательский UI;
- смешивать user-facing chat bubble и debug trace.
