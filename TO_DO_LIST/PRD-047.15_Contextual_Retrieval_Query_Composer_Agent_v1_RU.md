# PRD-047.15 — Contextual Retrieval Query Composer Agent v1

## 0. Краткое резюме

`PRD-047.15` вводит новый внутренний слой:

```text
Contextual Retrieval Query Composer Agent v1
```

Рабочее имя:

```text
contextual_retrieval_query_composer_v1
```

Назначение:

```text
Понять, нужен ли в текущем turn поиск по БД / RAG / memory,
и если нужен — сформировать правильный смысловой запрос не только из текущей реплики,
а из всего живого контекста диалога.
```

Главная идея:

```text
User: "Да, хорошо!"

Система не должна искать в БД по фразе "Да, хорошо".
Она должна понять:
- на что пользователь согласился;
- что было предложено предыдущим ответом;
- какой answer obligation сейчас активен;
- нужна ли для следующего ответа база знаний;
- какой запрос к БД реально нужен.
```

Цель PRD-047.15:

```text
Сделать retrieval умным контекстным помощником Writer,
а не механическим поиском по последней фразе пользователя.
```

Архитектурная роль:

```text
Composer не пишет ответ пользователю.
Composer не заменяет Writer.
Composer не вставляет canned text.
Composer не становится hard authority.
Composer решает retrieval intent / query / inclusion policy.
Writer остаётся автором финального ответа.
```

---

## 1. Почему этот PRD нужен сейчас

После PRD-047.14-HF2 закрыт важный routing risk:

```text
summary_request теперь не перехватывается confirmation_to_last_offer;
answer_obligation=summarize_current_conversation работает;
Writer получает summary metadata;
Gate валидирует summary behavior;
no canned summary replies.
```

До этого PRD-047.14-HF1/HF1.1/HF1.2 закрыли основной no-stub blocker:

```text
hardcoded template-family leakage removed;
Writer static repair/knowledge/direct-answer stubs converted to retry/gate signals;
Writer target blockers: 31 -> 0;
new user-facing stub: false.
```

Теперь следующий bottleneck — retrieval/context.

Сейчас retrieval может работать слишком прямолинейно:

```text
current user utterance -> search query
```

Но в живом диалоге часто нужен другой механизм:

```text
current user utterance
+ previous assistant offer
+ answer obligation
+ dialogue act
+ thread line
+ recent turns
+ memory state
+ current concept
+ knowledge need
→ contextual retrieval decision
→ semantic query
→ writer context package
```

---

## 2. Продуктовый замысел

NEO должен быть живой мультиагентной системой, а не набором жёстких маршрутов.

Целевая архитектура:

```text
Writer — главный автор ответа.

Остальные агенты:
- помогают видеть состояние пользователя;
- держат линию разговора;
- формируют память;
- диагностируют текущий узел;
- решают, нужен ли контекст из БД;
- отсекают шум;
- валидируют ответ;
- но не пишут финальный смысловой ответ вместо Writer.
```

`Contextual Retrieval Query Composer Agent` — это не “поисковик по словам”.
Это смысловой агент, который отвечает на вопросы:

```text
1. Нужно ли сейчас вообще идти в БД?
2. Если нужно — какую тему искать?
3. Достаточно ли текущего контекста без RAG?
4. Не будет ли RAG шумом для Writer?
5. Нужно ли искать по предыдущему предложению, а не по текущей короткой реплике?
6. Какие chunks реально дать Writer?
7. Можно ли Writer игнорировать RAG?
8. Нужно ли retrieval только для trace, но не для Writer?
```

## 3. Пример целевого поведения

### 3.1. Контекстный follow-up

Предыдущий ответ Writer:

```text
Могу показать это через нейросталкинг: как формируется автоматизм, где появляется защита и какой внутренний шаг его ослабляет.
```

Пользователь:

```text
Да, хорошо!
```

Неправильно:

```text
query = "Да, хорошо"
retrieval_action = none или мусорный поиск
```

Правильно:

```json
{
  "contextual_followup": true,
  "inherited_topic": "нейросталкинг как автоматизм / защита / внутренний шаг",
  "retrieval_need": "knowledge_context",
  "composed_query": "нейросталкинг автоматизм защитный механизм внутренний шаг наблюдение",
  "retrieval_action": "include_relevant_rag_if_high_confidence",
  "writer_can_ignore_rag": true
}
```

### 3.2. Короткое согласие без knowledge need

Предыдущий ответ:

```text
Хочешь, я просто поддержу тебя коротко, без разбора?
```

Пользователь:

```text
Да.
```

Правильно:

```json
{
  "contextual_followup": true,
  "inherited_offer_type": "short_support",
  "retrieval_need": "none",
  "retrieval_action": "suppress_rag",
  "reason": "short support does not need KB"
}
```

### 3.3. Summary request

Пользователь:

```text
Подведи краткий итог нашей беседы.
```

Правильно:

```json
{
  "retrieval_need": "conversation_context",
  "retrieval_action": "use_current_conversation_context_only",
  "external_kb_needed": false,
  "reason": "summary_request should summarize current conversation, not fetch new KB chunks"
}
```

### 3.4. Knowledge question

Пользователь:

```text
Что такое нейросталкинг?
```

Правильно:

```json
{
  "retrieval_need": "knowledge_context",
  "composed_query": "нейросталкинг определение внутреннее наблюдение автоматизм программа",
  "retrieval_action": "include_relevant_rag",
  "writer_can_ignore_rag": false
}
```

## 4. Главный архитектурный принцип

Composer — это retrieval brain, не answer brain.

Запрещено:

```text
Composer пишет ответ пользователю.
Composer возвращает готовое объяснение.
Composer вставляет canned фразы в Writer.
Composer решает за Writer, как именно формулировать ответ.
Composer цитирует КУЗНИЦУ как внешний авторитет.
Composer перегружает Writer всеми найденными chunks.
```

Разрешено:

```text
Composer создаёт contextual query.
Composer объясняет retrieval decision.
Composer выбирает include / suppress / trace-only.
Composer отдаёт compact context package.
Composer помечает Writer can ignore RAG.
Composer сохраняет trace/debug.
```

---

## 5. Relation to КУЗНИЦА ДУХА / Neo MindBot

`КУЗНИЦА ДУХА` и `Neo MindBot КОНСПЕКТ` должны оставаться внутренним lens library, а не user-facing цитатником.

Composer может использовать БД для поиска:

```text
- диагностических линз;
- понятий;
- практик;
- внутренних смысловых моделей;
- use_when / avoid_when;
- коротких grounding fragments.
```

Но Writer должен получать их как:

```text
internal context / optional lens / compact knowledge support
```

а не как:

```text
готовый текст, который надо пересказать.
```

## 7. Non-goals

В PRD-047.15 запрещено:

```text
- менять Writer prompt philosophy;
- создавать canned replies;
- создавать hardcoded final answers;
- добавлять static explanation fallback;
- добавлять новый user-facing answer factory;
- менять модель/температуру/provider;
- менять Chroma/DB schema без отдельного PRD;
- делать массовый KB cleanup;
- менять Diagnostic Center authority;
- делать большой orchestrator refactor;
- включать production/broad rollout/normal users;
- добавлять full LLM retrieval agent, если можно решить v1 deterministic/heuristic composer;
- делать UI/frontend changes;
- закрывать remaining no-stub candidates HF1.2;
- чинить все edge cases contextual follow-up сразу.
```

---

## 8. Важное решение: v1 не должен быть новым LLM-агентом

В v1 Composer должен быть deterministic/heuristic agent-like layer.

Причина:

```text
- система только что очищена от hardcoded answer factories;
- новый LLM agent увеличит вариативность и runtime drift;
- сначала нужно сделать observable deterministic contract;
- потом можно добавить LLM-assisted query composer в PRD-047.15-HF/047.16.
```

То есть:

```text
new logical agent layer: allowed
new LLM agent: not allowed in v1
new runtime path: not allowed
```

## 11. Retrieval actions

### 11.1. suppress_rag

Use when:

```text
- greeting/contact;
- close/thanks;
- short support;
- safety grounding;
- user is overloaded;
- current answer obligation does not require external knowledge;
- RAG would be noise.
```

Expected:

```json
{
  "retrieval_action": "suppress_rag",
  "writer_can_ignore_rag": true,
  "include_for_writer_if_found": false,
  "max_chunks_for_writer": 0
}
```

### 11.2. use_current_context_only

Use when:

```text
- summary_request;
- repair over previous answer;
- user asks to summarize current chat;
- answer should use conversation context, not KB.
```

### 11.3. query_kb

Use when:

```text
- user asks concept definition;
- user asks about КУЗНИЦА / NEO / нейросталкинг terms;
- previous assistant offered to explain a concept and user accepted;
- practice overview request needs knowledge grounding.
```

### 11.4. query_memory

Use when:

```text
- user asks “что я говорил раньше?”;
- user references earlier personal pattern;
- current chat context is insufficient but memory may contain relevant user-specific material.
```

### 11.5. query_kb_and_memory

Use when:

```text
- user asks to apply a concept to their recurring pattern;
- both conceptual lens and personal memory could help.
```

### 11.6. trace_only

Use when:

```text
- Composer can infer a useful query,
  but confidence is too low to send chunks to Writer.
```

## 12. Contextual follow-up rules

Composer must handle short/ambiguous replies:

```text
да
да, хорошо
хорошо
ок
ага
давай
можно
покажи
продолжай
```

It must not search these strings directly.

Instead:

```text
if short reply and last_assistant_offer.is_open:
    inherit topic and offer_type from last_assistant_offer

if short reply and unanswered_question_state.answer_required:
    inherit last_direct_user_question

if short reply and dialogue_pragmatics.is_contextual_followup:
    inherit previous assistant offer / active topic

if no inherited topic:
    suppress_rag
```

Example:

```json
{
  "user_message": "Да, хорошо",
  "last_assistant_offer": {
    "is_open": true,
    "offer_type": "explain_concept",
    "offer_text_summary": "показать через нейросталкинг"
  }
}
```

Expected:

```json
{
  "retrieval_need": "knowledge_context",
  "query_source": "last_assistant_offer",
  "composed_query": "нейросталкинг объяснение автоматизм наблюдение внутренний механизм",
  "retrieval_action": "query_kb",
  "include_for_writer_if_found": true
}
```

## 13. Summary request rules

For:

```text
dialogue_act=summary_request
answer_obligation=summarize_current_conversation
```

Composer should normally produce:

```json
{
  "retrieval_need": "conversation_context",
  "retrieval_action": "use_current_context_only",
  "query_source": "summary_request",
  "composed_query": "",
  "writer_can_ignore_rag": true,
  "include_for_writer_if_found": false,
  "max_chunks_for_writer": 0,
  "reason": "summary of current conversation should not introduce external KB"
}
```

Exception:

```text
If user explicitly asks:
"Подведи итог и свяжи это с нейросталкингом"
```

Then:

```text
retrieval_need=mixed
retrieval_action=query_kb_and_memory or query_kb
query_source=mixed_context
```

HF047.15 v1 may classify this as warning / future scope if too complex.

## 14. Knowledge question rules

For knowledge/concept questions:

```text
что такое нейросталкинг
что значит самореализация
объясни Кузницу Духа
какие есть практики
как работает автоматизм
```

Composer should build a conceptual query.

Example:

```json
{
  "retrieval_need": "knowledge_context",
  "retrieval_action": "query_kb",
  "query_source": "current_user_message",
  "composed_query": "нейросталкинг определение наблюдение автоматизм программа внутренний механизм",
  "query_terms": ["нейросталкинг", "автоматизм", "наблюдение", "внутренний механизм"],
  "writer_can_ignore_rag": false,
  "include_for_writer_if_found": true,
  "max_chunks_for_writer": 3,
  "max_chars_per_chunk": 700
}
```

## 15. Practice request rules

For:

```text
дай практику
какие практики есть
что сделать прямо сейчас
один шаг
```

Composer should distinguish:

```text
practice_catalog / practice_overview -> possible KB context
direct microstep in current situation -> often no KB needed
safety grounding -> suppress RAG
```

Expected examples:

```json
{
  "practice_overview_requested": true,
  "retrieval_need": "practice_context",
  "retrieval_action": "query_kb",
  "composed_query": "практики наблюдения автоматизма мягкое распутывание use_when avoid_when"
}
```

```json
{
  "answer_shape": "one_step",
  "retrieval_need": "none",
  "retrieval_action": "suppress_rag",
  "reason": "one-step direct response should not be overloaded by KB"
}
```

## 16. Noise suppression rules

Composer must suppress RAG when:

```text
- answer_obligation=close_gently;
- dialogue_act=greeting/contact_open/self_intro without knowledge need;
- planner_next_move=give_short_support;
- planner_next_move=stabilize_safety;
- user appears overloaded and current need is support;
- current request is formatting-only;
- Writer already has enough recent context;
- previous RAG was low relevance.
```

This is as important as query composition.

## 18. Interaction with existing retrieval_decision

If existing module already has:

```text
retrieval_decision
contextual_retrieval_gating_v1
writer_context_package_v1
```

Composer must not replace it blindly.

Instead:

```text
Composer creates query plan.
Existing retrieval layer executes/suppresses retrieval based on query plan.
Writer context package receives final included chunks and composer trace.
```

Possible contract:

```json
{
  "retrieval_decision": {
    "composer": {},
    "retrieval_action": "...",
    "rag_candidates_for_trace": [],
    "rag_included_for_writer": [],
    "rag_included_count": 0,
    "rag_suppressed_reason": "...",
    "writer_can_ignore_rag": true
  }
}
```

## 19. Chunk inclusion policy

Composer must enforce small context budgets.

Defaults:

```text
max_chunks_for_writer:
- suppress_rag: 0
- trace_only: 0
- query_kb: 2-3
- query_memory: 2
- query_kb_and_memory: 3 total

max_chars_per_chunk:
- default: 500-700
- hard max: 900
```

Writer should not receive raw large chunks if a compact extract is enough.

---

## 20. No-stub and no-answer-factory guard

Composer must not introduce user-facing text.

Forbidden patterns inside Composer:

```text
return "Вот объяснение..."
return "Нейросталкинг — это..."
return "Главный вывод..."
return "Сделай такой шаг..."
```

Allowed strings:

```text
enum labels
query terms
debug reasons
trace descriptions
test fixtures
```

Add tests that scan Composer for hardcoded final-answer-like returns.

## 27. Acceptance criteria

PRD-047.15 can be `passed` if:

```text
- contextual_retrieval_query_composer_v1 exists;
- short contextual replies do not search literal "Да, хорошо";
- short acceptance can inherit topic from last assistant offer;
- summary request suppresses external KB retrieval by default;
- knowledge questions compose KB queries;
- greeting/close/support suppress RAG;
- Composer payload is visible in trace/retrieval_decision/writer context;
- Writer receives only compact relevant chunks;
- no canned user-facing text is introduced;
- no new LLM agent is added;
- no new runtime path is added;
- runtime mutation scope proof passes;
- encoding gate passes with explicit NUL fields;
- tests pass;
- docs and implementation report are updated after push.
```

PRD-047.15 must be `warning` if:

```text
- v1 deterministic rules pass core cases,
  but some mixed cases are deferred;
- live endpoint not available but static/direct tests pass;
- remaining out-of-scope static candidates from HF1.2 remain;
- query quality needs future LLM-assisted composer calibration.
```

PRD-047.15 must be `blocker` if:

```text
- Composer searches literal short replies like "Да, хорошо";
- Composer includes KB chunks for summary_request by default;
- Composer creates user-facing text;
- Writer receives noisy chunks despite suppress_rag;
- no-stub scan fails;
- tests fail;
- runtime mutation touches disallowed DB/governance/frontend files;
- new LLM agent is added;
- implementation report not post-push updated.
```
