## 0. Краткое резюме

Следующий шаг после `PRD-047.15-HF2-R2` — не новая логика Writer, не новый агент и не очередное расширение domain-specific эвристик.

Цель `PRD-047.16` — подготовить базу знаний к mechanism-aware retrieval через единый контракт metadata и безопасный dry-run/audit слой.

Нужно сделать так, чтобы chunks в БД перестали быть просто кусками текста и стали управляемыми карточками знания:

```text
source material
→ semantic atoms
→ chunk_type
→ mechanism_hints
→ user_markers_examples
→ depth_level
→ allowed_use
→ avoid_when / contraindications
→ quote_policy
→ source_trace
→ audit report
```

Но это должно быть сделано без переусложнения и без превращения metadata в rule engine.

Главная формула:

```text
Чанк помогает Planner / Retrieval / Chunk Gate / Writer Context Package видеть возможный механизм.
Чанк НЕ командует Writer’ом.
Чанк НЕ маршрутизирует ответ жёстким if/else.
Чанк НЕ является готовым пользовательским ответом.
```

## 1. Контекст

### 1.1. Что уже закрыто предыдущим этапом

`PRD-047.15-HF2-R2` закрыл visibility parity по Hybrid Retrieval Planner:

- backend runtime fields;
- Admin Runtime visibility;
- Web Trace visibility;
- compact trace visibility;
- browser screenshot proof;
- без изменения Writer authority;
- без изменения KB / DB / Chroma;
- без production rollout;
- без нового runtime path.

Следующая рекомендация после `HF2-R2` прямо указывает на:

```text
PRD-047.16 — Mechanism-Aware Knowledge Base Preparation / Chunk Metadata v1
```

Причина:

```text
Hybrid Retrieval Planner уже стал видимым,
но качество retrieval дальше будет ограничено качеством самих chunks и metadata.
```

### 1.2. Почему это нужно сейчас

На текущем этапе проблема уже не только в том, как строится query-before-RAG.

Если в БД лежат обычные текстовые chunks, Planner может построить хорошую retrieval-гипотезу, но получить слабый материал:

```text
- слишком общий chunk;
- chunk без механизма;
- chunk без depth_level;
- практика без противопоказаний;
- линза, которую нельзя отдавать Writer’у напрямую;
- source fragment, ошибочно попавший как direct_to_writer;
- raw author text без quote_policy;
- материал, который требует предыдущей главы;
- смешанный chunk: теория + практика + кейс + safety.
```

Поэтому следующий слой качества — подготовка БД как библиотеки механизмов, а не как склада текстов.

### 1.3. Главная архитектурная граница

Этот PRD не должен снова сделать систему жёсткой.

Правильный смысл metadata:

```text
semantic guidance, not deterministic routing.
```

Плохой путь:

```python
if "мама" in user_message:
    depth_level = 3
    include_chunks("inner_child")
```

Правильный путь:

```json
{
  "mechanism_hints": ["parental_gaze", "shame_as_wrong_visibility"],
  "confidence": 0.62,
  "reason": "context suggests old shame / external gaze theme",
  "writer_can_ignore": true
}
```

## 2. Цель PRD

Создать foundation для mechanism-aware подготовки БД:

1. Единый metadata schema contract для chunks.
2. Backward-compatible adapter для существующих chunks/blocks.
3. Dry-run enrichment/audit pipeline без Chroma reindex.
4. Validation rules для governance-critical полей.
5. Отчёты по metadata completeness и quality risks.
6. Тестовые fixtures на ключевые типы chunks.
7. Документация и trace-ready vocabulary для будущих PRD.

Итог PRD должен дать проекту не новую user-facing функцию, а проверяемый слой подготовки знания:

```text
Можно посмотреть chunk и понять:
- что это за тип знания;
- какой механизм он поддерживает;
- когда его можно использовать;
- когда нельзя;
- можно ли показывать Writer’у;
- можно ли цитировать;
- какая глубина уместна;
- откуда это взято;
- какие риски есть при неправильном использовании.
```

## 5. Runtime requirements

### 5.1. Новый metadata contract

Добавить или расширить существующий schema layer так, чтобы существовал единый contract:

```text
MechanismAwareChunkMetadata v1
```

Точное место определить по source audit. Предпочтение:

```text
использовать существующие Bot_data_base / knowledge_governance modules,
а не создавать параллельную новую систему.
```

Если подходящего места нет, допустимо создать маленький модуль, например:

```text
Bot_data_base/knowledge_governance/mechanism_metadata.py
```

Но в implementation report нужно объяснить, почему выбран именно этот путь.

### 5.2. Обязательные базовые поля

Schema должна поддерживать минимум:

```json
{
  "metadata_schema_version": "mechanism_aware_v1",
  "chunk_id": "",
  "source_id": "",
  "source_doc": "",
  "source_type": "book | transcript | note | architecture_note | manual | unknown",
  "source_span": "",
  "heading_path": [],
  "parent_section_id": "",
  "chunk_type": "",
  "title": "",
  "core_thesis": "",
  "mechanism_hints": [],
  "user_markers_examples": [],
  "emotional_markers": [],
  "body_markers": [],
  "underlying_need": [],
  "protective_parts": [],
  "use_when": [],
  "avoid_when": [],
  "contraindications": [],
  "depth_level": 0,
  "intensity": "low | medium | high | unknown",
  "allowed_use": [],
  "visibility": "writer_allowed | internal_only | source_only",
  "quote_policy": "can_quote_short | paraphrase_only | internal_only | do_not_use",
  "response_intent": [],
  "recommended_moves": [],
  "forbidden_moves": [],
  "writer_instruction": "",
  "writer_can_ignore": true,
  "related_chunks": [],
  "safety_links": [],
  "practice_links": [],
  "derived_from": [],
  "confidence": 0.0,
  "quality_flags": [],
  "governance_flags": []
}
```

Не все поля обязаны быть заполнены для каждого chunk, но schema должна различать:

```text
required fields
optional fields
quality warning fields
invalid fields
```

### 5.3. Controlled vocabulary v1

Использовать controlled vocabulary как общий язык слоёв, не как route engine.

#### `chunk_type`

Минимальный набор:

```text
source_fragment
concept
mechanism
diagnostic_lens
dialogue_move
practice
anti_pattern
safety
case_example
style_voice
```

Допустимо сохранить существующие legacy типы через compatibility mapping, но нельзя ломать старые chunks.

#### `allowed_use`

Минимальный набор:

```text
direct_to_writer
internal_lens
retrieval_seed
diagnostic_hint
practice_suggestion
not_for_direct_quote
internal_only
```

Правила:

```text
- practice_suggestion допустим только если chunk_type == practice;
- direct_to_writer несовместим с quote_policy == do_not_use;
- internal_only должен блокировать direct_to_writer;
- source_fragment по умолчанию не должен быть direct_to_writer;
- diagnostic_lens по умолчанию должен быть internal_lens / diagnostic_hint, а не direct_to_writer;
- style_voice не должен всплывать как обычный semantic content;
- safety chunks могут иметь priority, но не должны становиться user-facing diagnosis.
```

#### `depth_level`

В `PRD-047.16` использовать компактную шкалу adapted reference:

```text
0 — stabilization / basic support / бытовой ответ
1 — practical reflection / факт-интерпретация / простая рамка
2 — mechanism / diagnostic lens / защитная часть
3 — deeper pattern / history-sensitive / shame-parental-gaze-inner-child
```

Если в старых материалах есть глубина 4/5, PRD должен не внедрять её в runtime, а сделать compatibility warning:

```text
extended_depth_detected: true
mapped_depth_level: 3
quality_flag: extended_depth_needs_manual_review
```

### 5.4. Anti-heuristic guard

Добавить audit/check, который подтверждает:

```text
- user_markers_examples не используются как deterministic routing rule;
- mechanism_hints не создают hardcoded branch;
- depth_level не назначается напрямую по одной подстроке;
- metadata generation не добавляет domain-specific if/else в runtime.
```

Это может быть реализовано как:

```text
- code search scan по новым файлам;
- unit test на отсутствие forbidden pattern в новых modules;
- human-readable anti_heuristic_report.md.
```

Запрещённые примеры для новых runtime-файлов:

```text
if "мама" in user_message
if "контроль" in user_message
if "стыд" in user_message
if "паника" in user_message: depth_level = 3
```

Допустимо в test fixtures и reference examples, если это не runtime route.

### 5.5. Backward-compatible adapter

Сделать адаптер, который умеет взять существующий chunk/block metadata и привести к `MechanismAwareChunkMetadata v1` в dry-run режиме.

Требования:

```text
- не терять existing metadata;
- unknown поля складывать в legacy_metadata / extra_metadata;
- не менять source file на диске;
- не писать в Chroma;
- не менять DB;
- сохранять warnings;
- возвращать normalized preview.
```

Пример результата:

```json
{
  "chunk_id": "...",
  "metadata_schema_version": "mechanism_aware_v1",
  "chunk_type": "mechanism",
  "allowed_use": ["internal_lens", "retrieval_seed"],
  "depth_level": 2,
  "mechanism_hints": ["control_as_safety"],
  "quality_flags": ["missing_avoid_when"],
  "legacy_metadata": {
    "old_lens_family": "control"
  }
}
```

## 6. Mechanism-aware metadata by chunk type

### 6.1. `source_fragment`

Назначение:

```text
trace/source attribution, backing source for derived chunks, audit.
```

Default:

```text
allowed_use: retrieval_seed или internal_only
quote_policy: paraphrase_only или internal_only
visibility: source_only / internal_only
```

Не должен автоматически становиться `direct_to_writer`.

### 6.2. `concept`

Поля особой важности:

```text
title
core_thesis
use_when
avoid_when
quote_policy
writer_instruction
```

Пример concept:

```text
карта не территория
факт vs интерпретация
программа несовершенного Я
защитная часть
автоматизм
```

### 6.3. `mechanism`

Это центральный тип для PRD-047.16.

Поля особой важности:

```text
mechanism_hints
core_thesis
mechanism_formula, если есть
user_markers_examples
underlying_need
protective_parts
recommended_moves
forbidden_moves
depth_level
```

Механизм — это не тема. Пример:

```text
Тема: паническая атака.
Механизм: контроль как попытка вернуть безопасность.
```

### 6.4. `diagnostic_lens`

Линза — это не диагноз.

Default:

```text
allowed_use: internal_lens, diagnostic_hint
visibility: internal_only
quote_policy: internal_only / paraphrase_only
writer_can_ignore: true
```

Обязательные safety notes:

```text
safe_user_translation
risk_if_exposed
allowed_writer_use
```

Если этих полей нет, выставить quality warning.

### 6.5. `dialogue_move`

Ход ответа, не готовый ответ.

Поля:

```text
response_intent
recommended_moves
forbidden_moves
writer_instruction
depth_level
avoid_when
```

Пример:

```text
сначала признать защитную функцию Контролёра;
потом мягко показать цену контроля;
не говорить “просто отпусти”.
```

### 6.6. `practice`

Практика — это интервенция, поэтому metadata строже.

Обязательные practice поля:

```json
{
  "goal": "",
  "target_mechanisms": [],
  "duration": "",
  "intensity": "low | medium | high",
  "steps_short": [],
  "preconditions": [],
  "avoid_when": [],
  "contraindications": [],
  "follow_up_question": "",
  "risk_if_wrong_timing": "",
  "not_first_response": true,
  "max_one_practice_per_answer": true
}
```

Validation rules:

```text
- practice_suggestion только для chunk_type == practice;
- practice без steps_short получает warning/blocker в зависимости от текущих governance rules;
- practice без avoid_when получает warning;
- high intensity practice без contraindications получает blocker/manual_review;
- practice не должна быть direct_to_writer, если нет preconditions/use_when.
```

### 6.7. `anti_pattern`

Anti-pattern должен помогать не вредить.

Поля:

```text
risk
do_instead
forbidden_moves
use_when
writer_instruction
```

Примеры:

```text
не спорить с паникой;
не давать глубокую практику в остром состоянии;
не превращать практику в плётку самосовершенствования;
не называть защитную часть врагом.
```

### 6.8. `safety`

Safety выше авторской философии.

Поля:

```text
risk_domain
required_response_elements
forbidden_moves
priority
refer_out_hint
medical_uncertainty_hint
```

Важно:

```text
safety chunk не должен ставить диагноз;
safety chunk может требовать стабилизацию / refer-out / emergency guidance;
safety metadata не может быть финально назначена LLM без validator.
```

### 6.9. `case_example`

Кейс — это аналогия, не основной ответ.

Default:

```text
quote_policy: paraphrase_only
allowed_use: internal_lens / retrieval_seed
```

Поля:

```text
mechanism_links
use_when
avoid_when
writer_instruction
risk_of_identification_or_shame
```

### 6.10. `style_voice`

Style chunks формируют голос, но не являются semantic answer material.

Default:

```text
allowed_use: internal_lens
visibility: internal_only
quote_policy: internal_only
```

Запрещено:

```text
style_voice chunk попадает как обычный RAG hit для direct answer.
```

## 8. LLM enrichment policy

В `PRD-047.16` LLM не должен быть обязательным.

Правило:

```text
No provider-backed LLM enrichment by default.
```

Допустимо создать LLM-ready interface / placeholder contract только если:

```text
- он не вызывает provider;
- он не является stub, который притворяется успешной enrichment;
- он явно disabled;
- в report написано: offline LLM enrichment вынесен в будущий PRD.
```

LLM в будущих PRD может предлагать:

```text
summary;
lens candidates;
tags;
use_when / avoid_when candidates;
self-containedness warnings;
split/merge suggestions.
```

Но LLM не должен финально решать:

```text
do_not_use;
allowed_use;
safety-critical flags;
```

## 11. Tests

IDE-агент обязан добавить/обновить тесты.

Точное расположение определить по текущей структуре. Возможные группы:

```text
Bot_data_base/tests/**
bot_psychologist/tests/**
tests/**
```

### 11.1. Schema validation tests

Проверить:

```text
- valid minimal concept chunk;
- valid mechanism chunk;
- valid diagnostic_lens chunk;
- valid dialogue_move chunk;
- valid practice chunk;
- valid safety chunk;
- valid anti_pattern chunk;
- valid source_fragment chunk;
- unknown legacy fields preserved;
- missing required fields produce warnings/errors;
- depth_level outside range handled safely;
- extended depth mapped with warning;
- invalid allowed_use conflict rejected or warninged according to severity.
```

### 11.2. Governance invariant tests

Проверить:

```text
- practice_suggestion cannot be assigned to non-practice chunk;
- internal_only cannot also be direct_to_writer;
- source_fragment direct_to_writer is blocked by default;
- diagnostic_lens direct_to_writer is blocked by default;
- quote_policy do_not_use blocks direct quote/direct_to_writer;
- high intensity practice requires contraindications;
- safety chunk cannot be converted to diagnosis text;
- derived chunk requires source_trace / derived_from.
```

### 11.3. Adapter tests

Проверить:

```text
- existing metadata → normalized mechanism-aware metadata;
- missing optional fields become quality warnings, not crash;
- old fields preserved in legacy_metadata;
- no source mutation;
- sanitized preview truncates content;
- content_full not present in audit output.
```

### 11.4. Dry-run script tests

Проверить:

```text
- fixture mode completes;
- report files created;
- metrics present;
- no raw content in JSON/MD reports;
- no Chroma/DB write;
- no runtime config mutation.
```

### 11.5. Anti-heuristic tests

Добавить минимальный scan/test:

```text
- новые runtime modules не содержат hardcoded semantic routing по “мама/стыд/контроль/паника”;
- markers only appear in fixtures/examples/reference docs, not live routing logic;
- mechanism_hints are stored as metadata, not used for direct if/else routing.
```

### 11.6. Regression tests

Запустить релевантные существующие тесты:

```powershell
pytest
```

Если полный `pytest` слишком долгий или в проекте есть стандартный subset, запустить:

```text
- knowledge governance tests;
- multiagent retrieval / trace tests;
- API contract tests, если metadata touched;
- encoding hygiene tests;
- existing no-stub/no-mutation tests, если есть.
```

В `test_command_output.txt` обязательно записать команды и результат.
