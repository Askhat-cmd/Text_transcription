## 0. Краткое резюме

`PRD-047.16` создал foundation:

```text
MechanismAwareChunkMetadata v1
→ legacy adapter
→ dry-run audit
→ completeness report
→ anti-heuristic report
→ no-mutation proof
```

Но audit показал, что реальная БД пока бедна смысловой metadata:

```text
mechanism_hints_present_ratio ≈ 0.48
missing_mechanism_hints_count = 28 / 54
contraindications_present_ratio = 0.0
practice metadata gaps remain
use_when / avoid_when частично формальные, а не смысловые
```

`PRD-047.17` должен сделать следующий безопасный шаг:

```text
не включать новую metadata в live Writer path,
не делать Chroma reindex,
не менять ответы бота,
а создать offline enrichment candidate layer над реальными chunks книги «КУЗНИЦА ДУХА v.2.md».
```

Главная задача:

```text
Для каждого выбранного chunk сделать enriched candidate pack:
- summary / core_thesis candidate;
- mechanism_hints candidates;
- use_when candidates;
- avoid_when candidates;
- contraindications candidates;
- safe_user_translation для diagnostic_lens;
- risk_if_exposed;
- allowed_writer_use;
- recommended_moves / forbidden_moves candidates;
- confidence / grounding / manual_review flags.
```

Важно:

```text
Enrichment candidate — это НЕ применение metadata.
Enrichment candidate — это НЕ изменение live DB.
Enrichment candidate — это НЕ команда Writer’у.
Enrichment candidate — это материал для будущей ручной/автоматизированной валидации и следующего PRD.
```

## 1. Контекст

### 1.1. Почему этот PRD нужен

После `PRD-047.16` у проекта появился стабильный metadata-контракт, но реальное качество chunks зависит от того, насколько осмысленно заполнены поля.

Сейчас schema есть, но многие поля либо отсутствуют, либо заполнены слишком общими значениями.

Особенно важно улучшить:

```text
mechanism_hints
use_when
avoid_when
contraindications
safe_user_translation
risk_if_exposed
allowed_writer_use
practice safeguards
source-grounded summary
```

Без этого Hybrid Retrieval Planner сможет видеть metadata structure, но не будет получать достаточно богатый semantic signal.

### 1.2. Главный источник данных

Главная книга, которая уже загружена в БД и является источником chunks:

```text
КУЗНИЦА ДУХА v.2.md
```

В локальном пространстве файл содержит крупную книгу с главами и практикумами. В source audit `PRD-047.16` уже было зафиксировано, что в проекте существуют:

```text
Bot_data_base/data/uploads/books/КУЗНИЦА ДУХА v.2.md
Bot_data_base/data/processed/books/123__кузница_духа_blocks.json
Bot_data_base/data/processed/all_blocks_merged.json
Bot_data_base/data/registry.json
Bot_data_base/data/chroma_db
```

Значит `PRD-047.17` должен работать не абстрактно, а по реальному источнику.

### 1.3. Предварительный профиль книги для понимания агентом

Книга содержит блоки, которые особенно важны для mechanism-aware разметки:

```text
Глава 1 — Программа «несовершенное Я», драйверы выживания, автопилот, практики.
Глава 2 — химия клетки, дофамин/кортизол, freeze, реконсолидация, DMN, практики.
Глава 3 — карта не территория, удаление/обобщение/искажение, факт vs история, практики.
Глава 4 — спиральная динамика, уровни, переходы, практики.
Глава 5 — тень, проекция, вытеснение, тело, отношения, практики.
Глава 6 — инстинкты, самосохранение, социальный/сексуальный инстинкт, перехваты, практики.
Глава 7 — страх, тело, режимы нервной системы, страх идентичности, практики.
Глава 8 — информационный метаболизм, непереваренный опыт, тело, интеграция, практики.
Глава 10 — НейроСталкинг, присутствие, ментальный суверенитет, прямое самосознание, практики.
```

Для PRD это означает:

```text
- chunk_type должен различать concept / mechanism / practice / case_example / diagnostic_lens / dialogue_move / anti_pattern / safety;
- practices требуют отдельной safety-разметки;
- concepts нельзя автоматически превращать в user-facing ответ;
- deeper chapters, особенно тень/страх/нейросталкинг, требуют depth и avoid_when;
- source_fragment должен сохранять provenance, но не становиться прямым ответом.
```

## 2. Цель PRD

Создать offline enrichment pipeline поверх `MechanismAwareChunkMetadata v1`, который:

1. Берёт chunks из реального processed source книги `КУЗНИЦА ДУХА v.2.md`.
2. Нормализует их через механизм `PRD-047.16`.
3. Создаёт enrichment candidates для слабых/неполных metadata полей.
4. Сохраняет candidates в отдельные artifacts.
5. Не применяет candidates к live DB/Chroma/runtime.
6. Создаёт reports для архитектора и будущего PRD.
7. Делает отдельный manual review pack для owner/developer review.

Главный результат:

```text
После PRD-047.17 мы должны увидеть:
- какие chunks готовы к enrichment;
- какие chunks нуждаются в manual review;
- какие practices опасны без contraindications;
- какие diagnostic_lens нельзя отдавать Writer’у напрямую;
- какие mechanism_hints действительно можно использовать как semantic retrieval signal;
- какие chapters/sections требуют source cleanup before enrichment.
```

## 7. Enrichment modes

### 7.1. Deterministic baseline mode — обязательный

Команда должна работать без внешнего LLM:

```powershell
python Bot_data_base/tools/run_mechanism_metadata_enrichment.py --mode deterministic --source kuznica --limit 80
```

Deterministic mode может использовать:

```text
- existing MechanismAwareChunkMetadata;
- heading_path;
- chunk_type;
- existing summary/core_thesis;
- source preview;
- practice step extraction;
- simple source-profile chapter mapping;
- PRD-047.16 validation warnings.
```

Но deterministic mode не должен претендовать на глубокую semantic truth.

Он должен честно выставлять:

```text
confidence <= 0.65
manual_review_required = true
safe_to_apply_automatically = false
```

### 7.2. Optional LLM candidate mode — разрешён, но disabled by default

Допустимо реализовать provider-backed offline enrichment, если в проекте уже есть безопасный provider client и env configured.

Но по умолчанию режим выключен.

Команда:

```powershell
python Bot_data_base/tools/run_mechanism_metadata_enrichment.py --mode llm-candidate --source kuznica --limit 20 --confirm-provider
```

Требования:

```text
- без --confirm-provider LLM mode не запускается;
- если API key/model недоступны, режим gracefully skipped, а не fake-passed;
- raw provider payload не коммитить;
- prompt/response сохранять только sanitized;
- budget ограничить;
- batch limit по умолчанию не больше 20 chunks;
- timeout/retry bounded;
- LLM output только JSON candidate, не user-facing answer;
- LLM не решает final allowed_use / safety / visibility;
- все LLM candidates manual_review_required = true.
```

Если LLM mode не реализован в этом PRD, это допустимо только если deterministic pipeline и schema полностью готовы. В implementation report нужно честно указать:

```text
LLM candidate mode deferred to future PRD
```

Но предпочтительно реализовать хотя бы безопасный interface + skipped proof.

### 7.3. Hybrid mode — optional, не обязателен

Можно создать hybrid mode только если это не усложняет реализацию:

```text
deterministic prefill → optional LLM candidate → deterministic validator
```

Но не делать сложный orchestration.

Enrichment должен уметь фильтровать именно книгу:

```text

source_trace.source_title
```

В отчёте указать, какой exact filter использован.

### 9.2. Chunk selection strategy

Для первого enrichment pass выбрать:

```text
- chunks with missing_mechanism_hints;
- practice chunks with missing steps / avoid_when / contraindications;
- diagnostic_lens chunks missing safe_user_translation/risk_if_exposed/allowed_writer_use;
- source_fragment chunks that may be better represented by derived concept/mechanism candidates;
- high-value chapters: 1, 3, 5, 6, 7, 8, 10.
```

Default limit:

```text
80 chunks deterministic
20 chunks optional LLM candidate
```

Если real chunks меньше — работать со всеми.
