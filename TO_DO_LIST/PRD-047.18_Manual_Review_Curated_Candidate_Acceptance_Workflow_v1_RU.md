# PRD-047.18 — Manual Review / Curated Candidate Acceptance Workflow v1

**Проект:** Bot Psychologist / Neo MindBot  
**Дата:** 2026-06-16  
**Статус:** PRD для IDE-агента  
**Предыдущий этап:** PRD-047.17 — Offline Summary / Lens / Use-When Enrichment over Mechanism Metadata v1  
**Главная цель:** превратить offline-кандидаты enrichment из PRD-047.17 в управляемую очередь ручной проверки и безопасный curated overlay preview, не активируя их в runtime, Writer, Chroma или live metadata.

---

## 0. Краткое резюме

PRD-047.17 создал 80 offline-кандидатов смысловой разметки по реальной книге `КУЗНИЦА ДУХА v.2.md`:

```text
247 processed source blocks
→ 80 selected real-source enrichment candidates
→ manual_review_pack
→ quality report
→ no runtime activation
```

Следующий шаг — не генерировать ещё больше кандидатов и не применять их в live БД, а создать **workflow ручной приемки**:

```text
candidate_only artifacts
→ review queue
→ decision template
→ decision validator
→ curated overlay preview
→ curation status report
→ no live apply
```

В результате проекта должен появиться проверяемый путь, по которому человек/архитектор позже сможет принять, отклонить или отложить отдельные candidate fields:

```text
summary_candidate
mechanism_hints_candidates
use_when_candidates
avoid_when_candidates
contraindications_candidates
safe_user_translation_candidate
risk_if_exposed_candidate
allowed_writer_use_candidate
recommended_moves_candidates
forbidden_moves_candidates
quote_policy_suggestion
depth_level_suggestion
chunk_type_review_suggestion
```

**Важно:** PRD-047.18 не должен автоматически принимать реальные кандидаты. Он должен создать безопасную систему принятия решений и доказать её на fixture/sample decisions.

## 1. Контекст

### 1.1. Что уже есть после PRD-047.16

PRD-047.16 добавил `MechanismAwareChunkMetadata v1` и read-only audit layer.

Ключевая идея PRD-047.16:

```text
metadata = semantic guidance, not deterministic routing
```

Механизм-aware metadata не является командой Writer'у, не является hard route и не является live KB truth без отдельной приемки.

### 1.2. Что уже есть после PRD-047.17

PRD-047.17 добавил:

```text
Bot_data_base/knowledge_governance/offline_enrichment.py
Bot_data_base/tools/run_mechanism_metadata_enrichment.py
mechanism_metadata_enrichment_candidate_v1
source profile по Кузнице
chapter coverage report
manual review pack
quality report
anti-runtime-activation proof
no-mutation proof
```

Фактический результат PRD-047.17:

```text
source blocks: 247
selected blocks: 80
candidate_count: 80
risk distribution: high=11, medium=50, low=19
practice candidates: 21
validation blockers: none
known warning: practice_current_metadata_missing_contraindications=21
```

### 1.3. Почему нужен PRD-047.18

Сейчас есть кандидаты, но нет полноценного workflow:

```text
кто и как принимает candidate field;
как фиксируется отклонение;
как отличить pending от accepted;
как собрать accepted overlay без live mutation;
как проверить, что никто не применил кандидаты автоматически;
как подготовить следующий шаг к retrieval eval / metadata apply dry-run.
```

Без такого workflow следующий шаг легко ошибочно превратится в:

```text
candidate → live metadata
```

Это запрещено.

Правильная траектория:

```text
candidate → review decision → curated overlay preview → eval → explicit apply PRD later
```

## 2. Цель

Создать offline/manual-review слой для PRD-047.17 enrichment candidates:

1. Сформировать review queue из `manual_review_pack.json` / `enrichment_candidates_deterministic.json`.
2. Создать стабильный schema contract для review decisions.
3. Создать decision template, удобный для человека.
4. Создать validator для review decisions.
5. Создать curated overlay preview generator, который применяет только явно accepted decisions из отдельного decisions-файла.
6. Доказать работу workflow на fixture/sample decisions, не принимая реальные candidate fields автоматически.
7. Сохранить все артефакты в `TO_DO_LIST/logs/PRD-047.18/`.
8. Обновить docs и ADR.
9. Подтвердить no runtime mutation / no Chroma reindex / no Writer path change.

## 3. Non-goals

В этом PRD запрещено:

```text
- применять кандидаты в live metadata;
- изменять processed blocks;
- менять Chroma или делать reindex;
- менять embeddings;
- менять Writer prompt;
- менять WriterContract live path;
- менять State Analyzer / Thread Manager / Hybrid Retrieval Planner runtime;
- добавлять новый LLM-agent;
- делать provider-backed review;
- генерировать user-facing тексты;
- добавлять Web Admin UI;
- считать decisions от IDE-агента человеческой ручной приемкой;
- автоматически принимать low-risk candidates;
- превращать mechanism_hints в deterministic routing;
- использовать candidate fields в runtime retrieval path.
```

Этот PRD создаёт **workflow и preview**, а не live-применение.



### 5.4. Hard validation rules

Validator must reject decision files when:

```text
- schema_version invalid;
- candidate_id missing;
- candidate_id not found in source candidate set;
- review_status outside vocabulary;
- field_decision outside vocabulary;
- accepted field has empty value;
- accepted field value type mismatches candidate field type;
- depth_level not in 0..3;
- quote_policy outside controlled vocabulary;
- chunk_type outside controlled vocabulary;
- safe_to_apply_to_live_metadata is true;
- reviewer_role claims human approval but reviewer_id/reviewed_at missing;
- any raw provider payload exists;
- any raw full source text is copied into decision artifacts;
- high-risk practice accepted without contraindications decision.
```

### 5.5. Important governance rule

Even when a decision is accepted, the output remains:

```text
curated_overlay_preview
```

It is still not live DB truth.

## 6. Review queue requirements

Runner must generate a review queue grouped by priority:

1. high-risk practice candidates;
2. practice candidates with missing contraindications;
3. diagnostic_lens requiring safe translation;
4. mechanism candidates with missing/generic mechanism_hints;
5. source_fragment requiring derived candidate review;
6. low-risk concept/style/case candidates.

Each queue item must include only safe/sanitized information:

```json
{
  "candidate_id": "...",
  "queue_priority": 1,
  "chunk_type": "practice",
  "risk_level": "high",
  "heading_path": [],
  "content_preview": "<=300 chars",
  "candidate_fields_preview": {},
  "manual_review_reasons": [],
  "validation_warnings": [],
  "recommended_reviewer_action": "review_contraindications_first"

```
