# PRD-047.20 — Curated Review Batch 1 + Retrieval Evaluation over Accepted Overlay v1

**Проект:** Bot Psychologist / Neo MindBot  
**Дата:** 2026-06-16  
**Статус:** PRD для IDE-агента  
**Предыдущий этап:** PRD-047.19-HF1 — Evidence Integrity / Regression Rerun / Artifact Encoding Repair v1  
**Главная цель:** перейти от инфраструктурной подготовки metadata к проверке реальной пользы: вручную/кураторски собрать первый ограниченный curated batch accepted decisions для offline evaluation, построить accepted overlay preview и проверить, помогает ли он retrieval/eval сценариям — без live apply, без Chroma reindex и без изменения Writer/runtime.

---

## 0. Краткое резюме

Мы закрыли инфраструктурную петлю:

```text
PRD-047.16 — MechanismAwareChunkMetadata foundation
PRD-047.17 — offline enrichment candidates
PRD-047.18 — manual review workflow
PRD-047.19 — dry-run apply preflight
PRD-047.19-HF1 — clean evidence repair
```

Теперь нельзя продолжать бесконечно строить preflight поверх preflight.

Следующий шаг должен дать **product evidence**:

```text
если взять маленькую партию реально полезных metadata decisions,
становится ли retrieval/evaluation точнее?
```

PRD-047.20 создаёт первый ограниченный curated batch:

```text
review_queue 80 candidates
→ batch_1 selection 10–20 candidates
→ batch_1 decisions pack
→ batch_1 accepted overlay preview
→ apply preflight over batch_1 overlay
→ retrieval eval dataset
→ baseline vs overlay shadow eval
→ recommendation: полезна ли metadata для следующего шага
```

Важная граница:

```text
Batch 1 accepted overlay = accepted for offline evaluation only.
Это НЕ live KB truth.
Это НЕ Chroma metadata.
Это НЕ Writer/runtime authority.
```

## 5. Важная терминология

### 5.1. Не использовать ложное “human approved”

IDE-агент не должен писать:

```text

human_approved = true
```

если реального ручного файла от пользователя/архитектора не было.

### 5.2. Допустимый статус для Batch 1

Использовать:

```json
{
  "reviewer_role": "offline_curator_batch_1",
  "human_final_approval": false,
  "evaluation_only": true,
  "live_apply_allowed": false
}
```

То есть:

```text
accepted = accepted for offline eval only

```

PRD должен поддерживать опциональный input:

```text

```

Но по умолчанию PRD-047.20 работает как offline curated eval batch.

## 7. Batch 1 selection requirements

Создать модуль:

```text
Bot_data_base/knowledge_governance/curated_batch_eval.py
```

Batch 1 должен выбрать от 10 до 20 candidates из PRD-047.18 queue.

### 7.1. Приоритет отбора

Отбирать так, чтобы batch давал реальную проверку пользы:

```text
- 3–5 mechanism/concept candidates;
- 3–5 diagnostic_lens candidates;
- 3–5 practice candidates, но осторожно;
- 2–4 source_fragment candidates, если нужен derived-review/reject/defer пример;
- не более 2 high-risk practice accepted fields.
```

### 7.2. Что нельзя принимать в Batch 1

Запрещено accepted для:

```text
- separator-only content_preview: "***", "---", empty;
- source_fragment как direct writer-facing content;
- high-risk practice без avoid_when + contraindications;
- diagnostic_lens без safe_user_translation;
- fields, где нет source grounding;
- fields, которые звучат как diagnosis или identity claim;
- generic summary типа "материал для компактного объяснения" без конкретного смысла.
```

Такие candidates можно:

```text
reject
defer
needs_source_context
```

### 7.3. Conservative accept policy

Для практик в Batch 1 разрешено принимать в первую очередь:

```text
avoid_when_candidates
contraindications_candidates
allowed_writer_use_candidate
```

Не принимать practice steps, если они отсутствуют или не проверены.

Для diagnostic_lens разрешено принимать:

```text
safe_user_translation_candidate
risk_if_exposed_candidate
allowed_writer_use_candidate
forbidden_moves_candidates
```

Для concept/mechanism разрешено принимать:

```text
summary_candidate
core_thesis_candidate
mechanism_hints_candidates
use_when_candidates
avoid_when_candidates
allowed_writer_use_candidate
```

Но только если source preview не пустой и не separator-only.

## 11. Retrieval evaluation requirements

Создать offline eval dataset:

```text
TO_DO_LIST/logs/PRD-047.20/retrieval_eval_dataset.json
TO_DO_LIST/logs/PRD-047.20/retrieval_eval_dataset.md
```

Dataset size: 15–25 cases.

Cases must cover:

```text
- control_as_safety / потеря контроля;
- shame / страх чужого взгляда;
- fact_vs_interpretation / карта не территория;
- imperfect_self_program / я недостаточен;
- practice timing / когда не давать практику;
- diagnostic_lens safe translation;
- source_fragment should not go direct-to-writer;
- low-resource / avoid depth;
- user asks practical short answer;
- user asks mechanism explanation.
```

Case schema:

```json
{
  "case_id": "eval_001",
  "user_query": "...",
  "expected_candidate_ids": [],
  "expected_chunk_types": [],
  "expected_mechanism_hints": [],
  "expected_allowed_use": [],
  "forbidden_chunk_types": [],
  "notes": "..."
}
```

### 11.1. Eval modes

Runner must compare at least two modes:

```text
baseline_botdb_retrieval
batch_1_overlay_shadow_lookup
```

Optional third mode:

```text
combined_baseline_plus_overlay_shadow
```

### 11.2. Baseline mode

If BotDB is available, query `/api/query/` read-only.

Save:

```text
TO_DO_LIST/logs/PRD-047.20/baseline_retrieval_results.json
```

If BotDB unavailable, mark baseline as skipped with reason and run overlay shadow eval only. Skipped BotDB is warning, not blocker.

### 11.3. Overlay shadow lookup mode

Do not use Chroma.

Use Batch 1 accepted overlay fields and source refs to determine:

```text
- does overlay contain a relevant accepted candidate for this case?
- does accepted metadata contain expected mechanism/chunk_type/allowed_use?
- would overlay help retrieval planner choose better chunk type/depth?
```

This is **shadow eval**, not runtime behavior.

### 11.4. Eval scoring

Create simple metrics:

```json
{
  "case_count": 20,
  "baseline_hit_rate": 0.0,
  "overlay_shadow_hit_rate": 0.0,
  "combined_expected_help_rate": 0.0,
  "unsafe_overlay_hit_count": 0,
  "separator_preview_accepted_count": 0,
  "practice_without_safety_count": 0
}
```

Acceptance expectations:

```text
separator_preview_accepted_count = 0
practice_without_safety_count = 0
unsafe_overlay_hit_count = 0
overlay_shadow_hit_rate >= 0.35 OR honest warning explaining batch too small
```

## 21. Warning conditions

Accept with warning if:

```text
- BotDB unavailable and baseline retrieval mode skipped;
- overlay_shadow_hit_rate below 0.35 but explained by small batch;
- batch has few accepted diagnostic_lens because safe translations are missing;
- many candidates are deferred due to source_preview weakness;
- eval proves metadata not yet useful enough for next integration.
```

These are useful product findings, not failure.
