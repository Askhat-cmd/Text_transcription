# PRD-047.21 — Overlay-Aware Retrieval Shadow Integration / Trace-Only v1

**Проект:** Bot Psychologist / Neo MindBot  
**Дата:** 2026-06-16  
**Статус:** PRD для IDE-агента  
**Предыдущий этап:** PRD-047.20 — Curated Review Batch 1 + Retrieval Evaluation over Accepted Overlay v1  
**Главная цель:** впервые подключить Batch 1 accepted overlay к основному bot runtime только как **trace-only shadow signal**, чтобы видеть в Web/API trace, помог бы overlay текущему retrieval, но не менять executed retrieval, WriterContract, Writer prompt, final answer, Chroma или live metadata.

---

## 0. Краткое резюме

PRD-047.20 показал полезный offline signal:

```text
Batch 1 selection: 16 candidates
Accepted overlay items: 12
Accepted fields: 139
baseline_hit_rate: 0.1111
overlay_shadow_hit_rate: 0.7778
combined_expected_help_rate: 0.7778
unsafe_overlay_hit_count: 0
```

Это уже product evidence: curated metadata может помогать retrieval видеть смысловые механизмы лучше, чем текущий baseline.

Но следующий шаг НЕ должен применять overlay к live retrieval.

PRD-047.21 должен сделать только trace-only shadow integration:

```text
User message
→ normal State Analyzer / Thread Manager / Retrieval / Writer path без изменений
→ параллельно overlay shadow matcher смотрит Batch 1 overlay
→ результат пишется только в trace/debug/evidence
→ Writer не видит overlay
→ final answer не меняется
```

Цель — получить живые trace-доказательства:

```text
в каких реальных запросах overlay мог бы помочь retrieval,
а где он шумит / опасен / бесполезен.
```

## 1. Почему это следующий правильный шаг

Мы договорились ускоряться, но не прыгать сразу в live apply.

Правильная последовательность:

```text
offline eval passed
→ trace-only runtime shadow
→ allowlisted shadow live evidence
→ only then possible controlled integration PRD
```

PRD-047.21 — это первый runtime-touching PRD после metadata/eval цепочки, поэтому он должен быть особенно строгим:

```text
trace only
feature-flagged
default safe
easy rollback
no Writer influence
no retrieval mutation
```

Key constraints from PRD-047.20:

```text

ready_for_runtime_visibility=false in apply sense
```

Для PRD-047.21 мы разрешаем **только trace visibility**, не runtime authority.

### 2.2. Что значит trace-only

Trace-only означает:

```text
- overlay может быть прочитан runtime'ом только для debug/shadow trace;
- overlay не меняет executed retrieval query;
- overlay не меняет semantic_hits;
- overlay не меняет Context Assembly;
- overlay не попадает в WriterContract;
- overlay не попадает в Writer prompt;
- overlay не попадает в final answer;
- overlay не становится Chroma metadata;
- overlay не меняет BotDB registry/processed blocks.
```

## 3. Цель

Добавить в основной multiagent runtime безопасный trace block:

```json
{
  "overlay_shadow": {
    "schema_version": "overlay_shadow_trace_v1",
    "enabled": true,
    "mode": "trace_only",
    "overlay_source": "batch_1",
    "used_for_writer": false,
    "used_for_retrieval_execution": false,
    "would_help": true,
    "matched_candidates": [],
    "safety_flags": [],
    "blockers": [],
    "warnings": []
  }
}
```

Этот block должен быть виден в debug/API trace surfaces, но не влиять на ответ.

## 4. Non-goals

Запрещено:

```text
- live apply accepted overlay to metadata;
- Chroma reindex;
- embedding changes;
- changing executed retrieval query;
- changing retrieved semantic hits;
- passing overlay to WriterContract;
- changing Writer prompt;
- changing final answer logic;
- changing State Analyzer / Thread Manager decisions based on overlay;
- adding Web Admin configuration UI;
- broad rollout;
- user-facing mention of overlay;
- treating overlay as human approved;
- using overlay as deterministic routing.
```

Add a minimal config surface, preferably environment/config based, not Web Admin UI yet.

Required defaults:

```text
OVERLAY_SHADOW_TRACE_ENABLED=false
OVERLAY_SHADOW_TRACE_MODE=trace_only
OVERLAY_SHADOW_OVERLAY_FILE=TO_DO_LIST/logs/PRD-047.20/batch_1_accepted_overlay_preview.json
OVERLAY_SHADOW_MAX_MATCHES=5
OVERLAY_SHADOW_MIN_SCORE=0.0
```

If project already has a runtime config registry, use it without broad refactor.

### 5.1. Default-off behavior

When flag is absent or false:

```json
{
  "overlay_shadow": {
    "enabled": false,
    "reason": "disabled_by_config"
  }
}
```

or omit field if current trace style prefers omission.

### 5.2. Enabled behavior

When flag is true:

```text
load overlay file read-only
run shadow matching after normal retrieval planning/execution
emit trace only
```

If overlay file missing:

```json
{
  "enabled": true,
  "status": "skipped",
  "reason": "overlay_file_missing",
  "used_for_writer": false,
  "used_for_retrieval_execution": false
}
```

Missing overlay is warning, not runtime failure.

## 6. New module requirements

Create small module:

```text
bot_psychologist/bot_agent/multiagent/overlay_shadow_trace.py
```

If current project layout differs, place it near retrieval/trace helpers, but keep it small and isolated.

Responsibilities:

```text
- load Batch 1 overlay safely;
- normalize accepted overlay items into lightweight trace cards;
- run simple shadow matching against current user message + optional planned/executed retrieval query;
- return structured trace block;
- never return text to Writer;
- never mutate memory/retrieval state.
```

### 6.1. Input contract

```python
build_overlay_shadow_trace(
    *,
    user_message: str,
    retrieval_query: str | None,
    state_snapshot: dict | None,
    thread_state: dict | None,
    overlay_file: str,
    enabled: bool,
    max_matches: int = 5,
) -> dict
```

### 6.2. Output contract

```json
{
  "schema_version": "overlay_shadow_trace_v1",
  "enabled": true,
  "mode": "trace_only",
  "overlay_source_prd": "PRD-047.20",
  "batch_id": "batch_1",
  "overlay_item_count": 12,
  "used_for_writer": false,
  "used_for_retrieval_execution": false,
  "used_for_final_answer": false,
  "would_help": true,
  "match_count": 3,
  "matched_candidates": [
    {
      "candidate_id": "...",
      "chunk_type": "mechanism",
      "risk_level": "medium",
      "score": 4.2,
      "matched_terms": ["контроль", "страх"],
      "focus_tags": [],
      "safe_user_translation_preview": "...",
      "allowed_writer_use_preview": "...",
      "trace_only": true
    }
  ],
  "safety_flags": [],
  "warnings": [],
  "blockers": []
}
```

### 6.3. Sanitization

Trace cards must not include:

```text
- full raw source text;
- raw provider payload;
- secrets;
- full private conversation history;
- full accepted overlay field dump;
- full practice steps.
```

Use previews only:

```text
safe_user_translation_preview <= 180 chars
allowed_writer_use_preview <= 180 chars
summary_preview <= 220 chars
```

---

## 7. Integration point requirements

Integrate after retrieval execution / memory bundle assembly, but before final debug payload is returned.

Allowed:

```text
- add overlay_shadow block into debug payload / trace object;
- add timing/cost metadata if trace already supports it;
- expose overlay_shadow in API debug JSON.
```

Forbidden:

```text
- appending overlay cards to semantic_hits;
- appending overlay cards to memory_bundle;
- appending overlay text to Writer context;
- changing retrieval_plan;
- changing executed query;
- changing final answer directive;
- changing Validator behavior based on overlay.
```

If there is an existing `DebugTrace` / `debug_payload` structure, add a new optional field:

```text
debug_payload["overlay_shadow"] = overlay_shadow_trace
```

Do not break old clients if field absent.
