# PRD-047.22-HF1 — Live API Writer KB Payload Trace Path Repair v1

**Проект:** Bot Psychologist / Neo MindBot  
**Дата:** 2026-06-17  
**Статус:** Hotfix PRD для IDE-агента  
**Основание:** аудит PRD-047.22 обнаружил разрыв между deterministic contract/smoke и live HTTP API path  
**Главная цель:** добиться, чтобы при `WRITER_KB_PAYLOAD_ENABLED=true` реальный live HTTP запрос `что такое Нейросталкинг?` возвращал в debug/API/Web Trace непустой `writer_kb_payload_trace` и `payload_chunk_count>=1`, без изменения retrieval ranking, Chroma, registry, processed blocks или live metadata.

---

## 0. Краткое резюме

PRD-047.22 сделал важную работу:

```text
- добавил writer_kb_payload_v1;
- deterministic Neurostalking smoke прошёл;
- tests прошли;
- no-mutation proof зелёный;
- Writer prompt меняется только в KB payload section;
- retrieval query/ranking не менялись.
```

Но live HTTP smoke показал критически важный warning:

```json
{
  "query": "что такое Нейросталкинг?",
  "request_status_code": 200,
  "debug_status_code": 200,
  "writer_kb_payload_trace": {},
  "payload_chunk_count": 0,
  "reason": "trace_missing_or_payload_empty"
}
```

Это означает:

```text
контракт работает в deterministic smoke/tests,
но в live API path Writer KB Payload не появляется в trace для ключевого сценария.
```

До исправления этого разрыва нельзя переходить к PRD-047.23 live evidence/evaluation, потому что мы снова будем собирать evidence на неполном канале доставки chunks.

---

## 1. Контекст

### 1.1. Что принято по PRD-047.22

Сохраняем как выполненное:

```text
- writer_kb_payload.py exists;
- structured payload contract exists;
- no blind mid-sentence truncation in deterministic smoke;
- tests pass;
- fallback available;
- no Chroma/live metadata mutation.
```

### 1.2. Что не принято

Не принимается как clean live readiness:

```text
api_writer_kb_payload_smoke.status=warning
writer_kb_payload_trace={}
payload_chunk_count=0
query="что такое Нейросталкинг?"
```

Это не просто warning. Для главного acceptance-сценария PRD-047.22 live path должен показывать payload trace.

## 2. Цель HF1

1. Найти причину, почему live HTTP path не возвращает `writer_kb_payload_trace`.
2. Исправить минимально.
3. Не менять retrieval ranking/query.
4. Не менять Chroma/metadata/registry.
5. Не менять Writer authority.
6. Добиться live HTTP smoke:

```text

writer_kb_payload_trace.enabled=true
payload_chunk_count>=1
mid_sentence_cut_count=0
bad_suffix_present=false
internal_payload_leak_in_answer=false
```

7. Сохранить root cause report и clean evidence.

## 15. Blocker conditions

Block if:

```text
- live API smoke still has payload_chunk_count=0;
- writer_kb_payload_trace still missing/empty;
- fix only changes deterministic smoke but not live path;
- retrieval query/ranking changes;
- Chroma or live metadata mutates;
- internal payload appears in final answer;
- tests fail;
- implementation report says passed while live smoke says warning/fail.
```
