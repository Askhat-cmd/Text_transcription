# PRD-047.22-HF2 — Manual Web Chat Runtime Parity / Writer KB Payload Effective Config v1

**Проект:** Bot Psychologist / Neo MindBot  
**Дата:** 2026-06-17  
**Статус:** Hotfix PRD для IDE-агента  
**Предыдущие этапы:**
- PRD-047.22 — Writer KB Payload Contract / Structured Chunk Delivery v1
- PRD-047.22-HF1 — Live API Writer KB Payload Trace Path Repair v1

**Главная цель:** убрать двойственность между managed runner path и обычным ручным Web Chat path: в пилотном Web Chat structured `writer_kb_payload_v1` должен быть **каноническим основным путём доставки KB chunks Writer’у**, а legacy semantic hits fallback должен остаться только аварийным fallback с явным warning в trace.

---

## 0. Краткое резюме

После PRD-047.22-HF1 managed live smoke доказал, что structured Writer KB Payload работает:

```text
writer_kb_payload_trace.enabled=true
payload_chunk_count=1
mid_sentence_cut_count=0
```

Но ручной Web Chat trace пользователя после перезапуска показал другую картину:

```text
WRITER KB PAYLOAD:
enabled=false
payload.version=legacy_semantic_hits_fallback_v1
reason=writer_kb_payload_disabled
```

Это означает, что новая система работает в управляемом тестовом запуске, но обычный ручной Web Chat может идти по старому пути.

Для пилотного проекта это слишком медленно и создаёт вредную двойственность.

HF2 должен сделать structured Writer KB Payload **единой нормальной системой для manual Web Chat / pilot dev runtime**:

```text
manual Web Chat
→ effective config visible
→ WRITER_KB_PAYLOAD_ENABLED=true for pilot/dev Web Chat
→ writer_kb_payload_v1 is primary
→ legacy fallback only on explicit builder failure / disabled mode
→ trace clearly says what happened
```

## 1. Почему нужен HF2

Мы не хотим бесконечно плодить двойные системы:

```text
old fallback
new payload
managed smoke path
manual browser path
API path
Web Trace path
```

На данном этапе правильный ход — не очередной shadow-only слой, а **консолидация обсуждаемого вопроса в один основной путь**.

В рамках Writer KB Payload этим основным путём должен стать:

```text
writer_kb_payload_v1
```

Legacy semantic hits fallback должен остаться только как:

```text
- аварийный fallback;
- явно trace-visible;
- warning condition;
- не нормальный путь для пилотного Web Chat.
```

---

## 2. Диагноз проблемы простыми словами

Проблема не в том, что система не умеет передавать полный chunk Writer’у.

Проблема в том, что разные способы запуска backend дают разные effective flags:

```text
managed HF1 runner:
  WRITER_KB_PAYLOAD_ENABLED=true
  writer_kb_payload_v1 работает

manual Web Chat:
  WRITER_KB_PAYLOAD_ENABLED=false или не применился
  включился legacy_semantic_hits_fallback_v1
```

То есть нужно чинить не `len`, а:

```text
1. single source of truth для runtime config;
2. startup parity;
3. trace visibility effective config;
4. manual Web Chat smoke;
5. canonical primary path для Writer KB Payload.
```

---

## 3. Цель HF2

Сделать так, чтобы при обычном пилотном запуске Web Chat:

```text
query: "что такое Нейросталкинг?"
```

в реальном LLM canvas / Writer prompt / trace было:

```text
WRITER KB PAYLOAD:
enabled=true
payload.version=writer_kb_payload_v1
payload_chunk_count >= 1
legacy_semantic_hits_fallback_v1 not primary
mid_sentence_cut_count=0
```

И чтобы в Web Trace было невозможно спутать:

```text
Writer получил полный structured payload
```

с:

```text
UI показывает только preview payload
```

## 6. Runtime config: single source of truth

### 6.1. Required behavior

Сейчас нужно убрать ситуацию, где managed runner и Web Chat видят разные effective config.

Добавить или укрепить единый config resolver для Writer KB Payload:

```text
bot_psychologist/.../runtime_config.py
```

или использовать уже существующий config module, если он есть.

Config resolver должен возвращать не только значение, но и источник:

```json
{
  "key": "WRITER_KB_PAYLOAD_ENABLED",
  "effective_value": true,
  "source": "env|config_file|default|test_override",
  "raw_value": "true",
  "default_value": true,
  "runtime_mode": "local|dev|test|production|unknown"
}
```

### 6.2. Pilot/dev default

Для пилотного локального Web Chat допускается ускорение:

```text
WRITER_KB_PAYLOAD_ENABLED=true by default for local/dev/pilot runtime
```

Но с guard:

```text
production/broad rollout default must not be silently changed without explicit APP_ENV gate.
```

Рекомендуемая логика:

```text
if APP_ENV in ["local", "dev", "pilot", "test"]:
    default WRITER_KB_PAYLOAD_ENABLED=true
else:
    default WRITER_KB_PAYLOAD_ENABLED=false unless explicit env/config enables it
```

Если в проекте нет `APP_ENV`, добавить безопасное определение:

```text
APP_ENV=local by default only for local startup scripts / dev runner
```

Не делать production-ready declaration.

### 6.3. .env / startup script hygiene

Обновить dev startup docs/scripts так, чтобы обычный запуск backend для Web Chat явно включал:

```text
WRITER_KB_PAYLOAD_ENABLED=true
OVERLAY_SHADOW_TRACE_ENABLED=true optional only if needed for trace; not required for HF2
DEBUG_TRACE_ENABLED=true
APP_ENV=local
```

Если есть `run_web_chat_dev.ps1` / `start_backend.ps1` / README command — обновить.

Если таких скриптов нет, создать минимальный safe startup script:

```text
bot_psychologist/tools/start_pilot_web_chat_backend.ps1
```

или `.bat`/`.ps1` по текущему стилю проекта.

## 7. Canonical Writer KB Payload path

### 7.1. Primary path

В manual Web Chat / local pilot:

```text
writer_kb_payload_v1 is primary when KB hits exist.
```

Если payload builder успешно собрал хотя бы один chunk:

```text
legacy_semantic_hits_fallback_v1 must not be primary.
```

### 7.2. Fallback path

Legacy fallback разрешён только если:

```text
- WRITER_KB_PAYLOAD_ENABLED=false explicitly;
- builder failed;
- no eligible KB chunks after policy;
- emergency compatibility mode.
```

В любом из этих случаев trace обязан содержать:

```json
{
  "writer_kb_payload_trace": {
    "enabled": false,
    "status": "fallback_used",
    "fallback_reason": "disabled_by_config|builder_failure|no_eligible_chunks|compat_mode",
    "fallback_is_primary": true,
    "warning": "legacy semantic hits fallback used instead of writer_kb_payload_v1"
  }
}
```

## 8. Trace requirements

### 8.1. Effective config block

В Web Trace/API debug добавить блок:

```json
{
  "runtime_config_trace": {
    "schema_version": "runtime_config_trace_v1",
    "app_env": "local",
    "backend_pid": 12345,
    "backend_start_time": "...",
    "writer_kb_payload_enabled": true,
    "writer_kb_payload_enabled_source": "env|default_local|config_file|test_override",
    "overlay_shadow_trace_enabled": true,
    "debug_trace_enabled": true
  }
}
```

### 8.2. Payload display vs sent payload

Добавить в `writer_kb_payload_trace` различение:

```json
{
  "payload_sent_to_writer_char_count": 1800,
  "payload_display_preview_char_count": 500,
  "payload_display_is_preview": true,
  "payload_full_text_sent_to_writer": true,
  "payload_full_text_exposed_in_web_trace": false
}
```

Важно: полный сырой текст не обязательно показывать в Web Trace. Но trace должен ясно говорить:

```text
Writer получил больше, чем UI preview.
```

## 16. Acceptance criteria

HF2 accepted if:

1. Source gates pass.
2. Effective runtime config is visible in API/Web Trace.
3. Manual Web Chat / same API path as Web Chat returns `writer_kb_payload_v1` for Neurostalking query.
4. `WRITER_KB_PAYLOAD enabled=false` no longer appears in pilot manual Web Chat unless explicitly disabled.
5. `legacy_semantic_hits_fallback_v1` is not primary when builder succeeds.
6. Trace distinguishes sent payload from display preview.
7. Prompt canvas proves Writer sees `writer_kb_payload_v1`.
8. Final answer does not leak internal payload metadata.
9. Retrieval ranking/query unchanged.
10. Chroma/live metadata/registry unchanged.
11. Tests pass.
12. Encoding hygiene passes.
13. Docs/ADR updated.
14. Commit and push complete.
15. Implementation report updated after final push.

## 17. Warning conditions

Accept with warning if:

```text
- browser screenshot could not be captured, but API/Web Chat path smoke passed;
- Web Trace panel still shows preview only, but trace explicitly marks display_is_preview=true and sent_char_count is correct;
- production default remains disabled by gate, while local/dev/pilot is enabled.
```

---

## 18. Blocker conditions

Block if:

```text
- manual Web Chat still shows WRITER_KB_PAYLOAD enabled=false in pilot/dev mode;
- manual Web Chat still uses legacy_semantic_hits_fallback_v1 as primary for KB expected query;
- Writer prompt canvas lacks writer_kb_payload_v1;
- final answer leaks internal payload JSON/metadata;
- retrieval ranking/query changes;
- Chroma/live metadata is mutated;
- production default is silently broad-enabled without APP_ENV gate;
- tests fail;
- implementation report says passed while manual parity smoke fails.
```
