# PRD-047.22 — Writer KB Payload Contract / Structured Chunk Delivery v1

**Проект:** Bot Psychologist / Neo MindBot  
**Дата:** 2026-06-16  
**Статус:** PRD для IDE-агента  
**Предыдущий этап:** PRD-047.21 — Overlay-Aware Retrieval Shadow Integration / Trace-Only v1  
**Главная цель:** убрать blind 300-char truncation как основной способ передачи знаний Writer’у и заменить его структурированным Writer KB Payload, чтобы выбранные retrieval chunks доходили до Writer’а как полноценные смысловые карточки, без обрыва на середине фразы и без изменения retrieval ranking / Chroma / live metadata.

---

## 0. Краткое резюме

PRD-047.21 успешно вывел accepted overlay в trace-only shadow. Но параллельно выявлена более базовая проблема live-path:

```text
retrieval находит правильный чанк,
но Writer получает не полноценный structured chunk,
а обрезанный semantic_hits-фрагмент примерно на 300 символов.
```

В prompt canvas по вопросу:

```text
что такое Нейросталкинг?
```

Writer получил в блоке `ЗНАНИЯ ИЗ БАЗЫ` текст, который оборвался так:

```text
**НеоСталкинг** — это в
```

То есть retrieval сработал, но канал доставки знания до Writer’а испортил материал.

PRD-047.22 должен исправить именно этот слой:

```text
selected semantic/RAG hit
→ Writer KB Payload builder
→ structured KB card
→ Writer receives meaning, excerpt, metadata, truncation trace
→ Writer remains final author
```

Важно:

```text
Этот PRD НЕ меняет retrieval ranking.
Этот PRD НЕ применяет overlay к live metadata.
Этот PRD НЕ делает Chroma reindex.
Этот PRD НЕ даёт overlay retrieval authority.
Этот PRD меняет только способ упаковки уже выбранных chunks для Writer’а.
```

---

## 1. Почему это нужно сделать сейчас

Если продолжить собирать allowlisted live evidence по overlay, но Writer всё ещё получает обрубки chunks, мы будем тестировать не качество БД и retrieval, а испорченный канал доставки.

Правильная последовательность теперь:

```text
PRD-047.21 — overlay visible in trace only
PRD-047.22 — selected chunks delivered to Writer correctly
PRD-047.23 — allowlisted live evidence / trace review with reliable Writer KB payload
```

То есть PRD-047.22 — не откат и не отвлечение. Это закрытие bottleneck между retrieval и Writer.

---

## 2. Контекст проблемы

Текущий проблемный путь:

```text
selected chunk
→ rag_for_writer
→ semantic_hits_trimmed
→ первые ~300 символов
→ Writer
```

Проблема проявляется так:

```text
ЗНАНИЯ ИЗ БАЗЫ:
- НейроСталкинг и НеоСталкинг — это два уровня одного процесса.

**НейроСталкинг** — это работа с первым этажом...

**НеоСталкинг** — это в
```

Writer вынужден достраивать смысл сам. В удачном случае он ответит неплохо. Но архитектурно это ошибка: если KB найден, он должен доходить до Writer’а не как случайный обрезок.

### 5.1. Overlay metadata

`WRITER_KB_PAYLOAD_USE_OVERLAY_METADATA=false` по умолчанию.

В v1 можно подготовить plumbing для overlay metadata enrichment, но нельзя делать overlay metadata обязательным источником Writer payload.

Если флаг включён в tests/smoke:

```text
- использовать overlay metadata только если chunk_id/block_id совпадает с уже выбранным retrieval hit;
- не использовать overlay для выбора нового chunk;
- не добавлять overlay-only chunks Writer’у;
- не использовать overlay fields with quote_policy=internal_only as user-facing quote;
- trace must show overlay_metadata_used=true/false.
```

## 7. Truncation strategy requirements

Запрещено:

```python
content[:300]
content[:N]
```

как основной способ формирования Writer KB text.

Допустимо:

```text
1. Preserve whole content if <= EXCERPT_MAX_CHARS.
2. Prefer heading + first complete paragraphs.
3. If too long, cut at paragraph boundary.
4. If no paragraph boundary, cut at sentence boundary.
5. If no sentence boundary, cut at word boundary with explicit trace warning.
6. Never cut in the middle of markdown heading/bold marker if avoidable.
```

Required fields:

```text
original_char_count
sent_char_count
content_truncated
truncation_strategy
truncation_reason
truncated_mid_sentence
```

### 7.1. Known bad regression

This must fail tests:

```text
content_excerpt.endswith("**НеоСталкинг** — это в")
```

or any equivalent mid-sentence/mid-word cut for the Neurostalking concept case.

## 8. Writer prompt / contract integration

Replace current flat block:

```text
ЗНАНИЯ ИЗ БАЗЫ:
- <trimmed raw hit>
```

with structured block, for example:

```text
WRITER KB PAYLOAD:
version=writer_kb_payload_v1
chunk_count=1

[KB-1]
chunk_id=...
source_doc=Кузница Духа
chunk_type=concept
quote_policy=paraphrase_only
allowed_use=direct_to_writer
core_thesis=...
content_excerpt:
...
writer_instruction=Use as grounding. Paraphrase. Do not dump raw quote.
content_truncated=true
truncation_strategy=paragraph_then_sentence_boundary
```

Important:

```text
- Writer remains final author.
- Writer can ignore KB if irrelevant.
- Payload is grounding, not command.
- Keep existing hard boundaries.
```

### 8.1. Backward compatibility

If `WRITER_KB_PAYLOAD_ENABLED=false`, keep existing behavior unchanged.

If enabled but payload builder fails:

```text
- fall back to existing KB block;
- trace warning writer_kb_payload_failed;
- do not crash response.
```
