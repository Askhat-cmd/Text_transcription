# PRD-047.23 — Bot_data_base Chunk Boundary / Retrieval-to-Writer Payload Audit v1

**Проект:** Bot Psychologist / Neo MindBot  
**Дата:** 2026-06-17  
**Статус:** PRD для IDE-агента  
**Предыдущий этап:** PRD-047.22-HF2 — Manual Web Chat Runtime Parity / Writer KB Payload Effective Config v1  
**Тип PRD:** evidence/audit-first, no-mutation  
**Главная цель:** понять, где именно режется знание: в Bot_data_base chunking, в retrieval/query assembly, в Writer KB Payload, в prompt canvas или только в Web Trace preview. Не чинить вслепую.

---

## 0. Краткое резюме

После PRD-047.22-HF2 транспорт до Writer исправлен:

```text
manual Web Chat → writer_kb_payload_v1 → Writer
```

Новый ручной trace пользователя показывает, что structured payload действительно включён:

```text
WRITER KB PAYLOAD:
enabled=true
payload.version=writer_kb_payload_v1
fallback_used=false
```

Но trace выявил новые вопросы:

```text
1. Иногда payload выглядит полным, но content_excerpt всё равно заканчивается обрывом.
2. Иногда content_truncated=true и truncation_strategy=sentence_boundary — это допустимое runtime-сокращение.
3. Иногда content_truncated=false, но сам source chunk, похоже, уже заканчивается посреди фразы.
4. По запросу “Программа несовершенное Я” retrieval может подтягивать chunk про НейроСталкинг.
5. По запросу про пять драйверов retrieval подтягивает не сами пять драйверов, а соседние chunks: “Шаг 1: Триггер”, пример и происхождение программы.
6. В trace встречаются несогласованности: rag_for_writer_count=0, но Writer KB Payload содержит chunk_count>0.
7. RAG query может дублировать / тащить предыдущий вопрос и становиться слишком длинным.
```

Поэтому следующий шаг — не overlay live evidence, а точный аудит цепочки:

```text
Bot_data_base source text
→ chunking
→ UniversalBlock / stored block
→ Chroma / query API
→ Memory Retrieval semantic_hits
→ Writer Context Package
→ Writer KB Payload
→ prompt canvas
→ Web Trace display preview
```


### 1.2. Что показал новый trace пользователя

Новый файл ручного запуска Web Chat показывает, что payload теперь включён. Это хорошо.

Но он также показывает, что проблема “обрезки” может находиться не в Writer KB Payload, а раньше — в подготовке или хранении chunks.

Пример подозрительного кейса:

```text
content_excerpt заканчивается:
“Она становится тем, за чем можно наблюдать, а не тем, изнутри чего т”
content_truncated=false
truncation_strategy=none
```

Такой случай означает:

```text
runtime payload builder не считает, что он обрезал текст,
но сам stored chunk может уже быть обрезанным или сохранённым с плохой boundary.
```

Другой кейс:

```text
content_truncated=true
truncation_strategy=sentence_boundary
total_sent_char_count=1575
```

Это может быть нормальным runtime-сокращением, если оно завершает текст по sentence/paragraph boundary и не ломает смысл.

### 1.3. Почему нельзя сразу чинить chunker

Пока неизвестно, где источник проблемы:

```text
- исходный markdown уже плохой;
- Bot_data_base chunker режет по char limit, а не по section/sentence boundary;
- stored UniversalBlock содержит неполный текст;
- Chroma metadata/content сохраняет короткий preview вместо full content;
- runtime query вытаскивает не тот chunk;
- Writer KB Payload берёт preview вместо full block;
- Web Trace показывает preview и визуально кажется, что Writer получил обрыв;
- query assembly смешивает текущий вопрос с предыдущей темой.
```

Этот PRD должен дать точный ответ.

1. Source gates pass.
2. Chunking code map identifies exact files/functions responsible for chunk splitting/storage/query/runtime payload/Web Trace preview.

4. Chunk boundary audit classifies every observed cut.
5. The “изнутри чего т” case is explained as source boundary cut, stored preview issue, runtime truncation, or unknown with evidence.
6. Query assembly audit explains previous-topic pollution / duplication / truncation risks.
7. Writer payload consistency audit explains rag_for_writer_count vs payload chunk_count mismatch.
8. Retrieval relevance audit explains why expected sections were or were not selected.
9. No Bot_data_base source/chunk/Chroma mutation occurs.
10. Tests pass.
11. Encoding hygiene passes.
12. Implementation report gives clear next PRD recommendation.
13. Commit and push complete.
