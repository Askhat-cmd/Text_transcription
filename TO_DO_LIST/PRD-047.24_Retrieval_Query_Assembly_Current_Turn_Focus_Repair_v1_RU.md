# PRD-047.24 — Retrieval Query Assembly / Current-Turn Focus Repair v1

**Проект:** Bot Psychologist / Neo MindBot  
**Дата:** 2026-06-17  
**Статус:** PRD для IDE-агента  
**Предыдущий этап:** PRD-047.23 — Bot_data_base Chunk Boundary / Retrieval-to-Writer Payload Audit v1  
**Тип PRD:** targeted runtime repair + trace proof  
**Главная цель:** исправить сборку retrieval query так, чтобы KB-поиск опирался прежде всего на текущий вопрос пользователя, не загрязнялся предыдущей темой, не дублировался и не обрезался посреди слова. Это должно вернуть retrieval к точным sections/chunks без Chroma reindex и без изменения Bot_data_base.

---

## 0. Краткое резюме

PRD-047.23 доказал, что проблема сейчас не в базовом Writer KB Payload path и не в подтверждённом дефекте исходного Bot_data_base chunking.

Главные найденные проблемы:

```text
C23-002:
raw_user_query = а что такое "Программа несовершенное Я"?
executed_query = а что такое "Программа несовершенное Я"? + предыдущий вопрос про самореализацию и Нейросталкинг
retrieval_focus_status = polluted
expected section exists, but retrieval ушёл в старый/соседний topic

C23-003:
raw_user_query = расскажи о Пяти драйверах выживания...
executed_query = raw_user_query + duplicated raw_user_query, обрезанный mid-word
retrieval_focus_status = polluted
expected five-driver definition blocks exist, but selected chunks are neighbor/mechanism chunks
```

Поэтому PRD-047.24 должен починить именно:

```text
retrieval query assembly
current-turn focus
dedupe
safe truncation
planned/executed query consistency
trace proof
```

Не нужно reindex, не нужно переписывать chunker, не нужно менять Writer prompt как основную меру.

## 4. Архитектурное решение

### 4.1. Canonical current-turn query builder

Создать или выделить модуль:

```text
bot_psychologist/bot_agent/multiagent/retrieval_query_builder.py
```

Если похожий модуль уже есть — не дублировать, а расширить существующий.

Основная функция:

```python
build_retrieval_query(
    *,
    user_message: str,
    previous_user_message: str | None,
    thread_state: dict | None,
    dialogue_act: dict | None,
    planner_query: str | None,
    inherited_topic: str | None,
    max_chars: int,
    mode: str = "current_turn_focus_v1",
) -> RetrievalQueryBuildResult
```

Output contract:

```json
{
  "schema_version": "retrieval_query_build_v1",
  "raw_user_query": "...",
  "canonical_query": "...",
  "executed_query": "...",
  "planned_query": "...",
  "planner_query_used": true,
  "previous_user_query_included": false,
  "previous_user_query_inclusion_reason": "none|elliptical_followup|explicit_reference|not_used",
  "inherited_topic_used": false,
  "inherited_topic_reason": "...",
  "dedupe_applied": false,
  "duplicate_fragment_count": 0,
  "truncation_applied": false,
  "truncation_strategy": "none|word_boundary|sentence_boundary",
  "query_truncated_mid_word": false,
  "current_turn_focus_status": "clean|elliptical_contextualized|polluted_blocked|fallback",
  "fallback_used": false,
  "fallback_reason": ""
}
```

### 4.2. No more naive previous-question concatenation

Default rule:

```text
Do NOT concatenate previous user question into retrieval query.
```

Previous user question may be included only if:

```text
1. current user message is elliptical/short follow-up;
2. it cannot stand alone;
3. previous topic is explicitly referenced or required;
4. inclusion is compact, not full previous question dump;
5. trace states the reason.
```

Examples where previous context may be used:

```text
“а второй уровень?” → include inherited_topic=Нейросталкинг / НеоСталкинг
“а как это связано с этим?” → include compact current topic from thread_state
“да, расскажи подробнее” → include last assistant offer summary, not full previous user query
```

Examples where previous context must NOT be used:

```text
“а что такое Программа несовершенное Я?” after self-realization question
“расскажи о пяти драйверах...” after previous imperfect-self question
```

These are standalone knowledge questions.

### 4.3. Use planned query safely

If a deterministic/hybrid planner produces a concise `planned_query`, it may be used when:

```text
- universal gate = clear_kb_ask OR knowledge_question;
- planned_query is non-empty;
- planned_query is not stale;
- planned_query is not broader than current user query;
- planned_query passes query safety validation.
```

For C23-002, if planner gives:

```text
planned_query = программа несовершенное
```

then executed query should prefer:

```text
программа несовершенное Я
```

or a safe expanded variant:

```text
программа несовершенное Я OS Я недостаточен
```

but must not append the previous self-realization query.

### 4.4. Dedupe + safe truncation

Add deterministic cleaning:

```text
- normalize whitespace;
- collapse repeated exact fragments;
- collapse repeated quoted topic phrases;
- strip accidental duplicated raw query;
- apply max chars at word boundary;
- never truncate in the middle of a word;
- record truncation strategy.
```

If truncation needed:

```text
prefer word_boundary or sentence_boundary
query_truncated_mid_word=false
```
