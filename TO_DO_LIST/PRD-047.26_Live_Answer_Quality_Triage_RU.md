**Проект:** Bot Psychologist / Neo MindBot

**Тип PRD:** короткий evidence/triage PRD перед DB-track  
**Главный принцип:** сначала сделать причину live-деградации видимой, затем переходить к mechanism-aware DB/chunk metadata.

## 0. Краткое резюме

После `PRD-047.25` техническая труба знаний стала стабильнее:

- `writer_kb_payload_v1` является canonical для Writer KB delivery;
- `current_turn_focus_v1` остаётся primary для retrieval query assembly;
- legacy query builder не является primary;
- overlay остаётся trace-only и не имеет права управлять Writer’ом;
- live evidence показал стабильность KB payload path, но также выявил `overlay_false_positive_count=6`;
- `direct_answer_success_rate=0.85`, то есть часть live-ответов всё ещё может быть слабее целевого уровня.

Следующий шаг **не должен** переписывать DB, Writer prompt, retrieval ranking или runtime.  
Задача этого PRD — сделать короткий, честный triage live-качества:

```text
живой ответ стал слабее / суше / учебниковее
→ где именно причина?
→ dialogue_act?
→ answer_obligation?
→ response_mode?
→ overlay shadow noise?
→ Writer KB payload?
→ retrieval/query?
→ evaluator пропустил слабый ответ?
```

Результат PRD-047.26 должен дать архитектурный ответ:

```text
1. Что реально ломает live-качество.
2. Какие кейсы требуют HF.
3. Можно ли безопасно идти в следующий DB-track.
4. Какой следующий PRD должен быть: DB-track или targeted HF.
```

## 1. Контекст

### 1.1. Что уже принято

`PRD-047.24` починил current-turn retrieval query assembly:

- standalone knowledge question больше не загрязняется предыдущей темой;
- duplicate fragments убраны;
- mid-word truncation убран;
- `retrieval_query_build_trace_v1` виден в trace.

`PRD-047.25` собрал live evidence по cleaned retrieval baseline + overlay + Writer KB payload:

- `kb_payload_primary_rate=1.0`;
- `current_turn_focus_clean_rate=1.0`;
- `legacy_query_builder_primary_count=0`;
- `overlay_apply_detected_count=0`;
- `internal_leak_count=0`;
- `raw_kb_dump_count=0`;
- `unsafe_practice_count=0`;
- `diagnostic_overclaim_count=0`;
- `direct_answer_success_rate=0.85`;
- `overlay_false_positive_count=6`.

### 1.2. Почему нельзя сразу идти в DB-track

DB-track действительно следующий стратегический большой шаг. Но если сейчас сразу улучшать chunk metadata, мы можем улучшить входящие чанки, не заметив, что live-ответ портится раньше:

- State Analyzer мог неправильно поставить `dialogue_act`;
- Thread Manager мог оставить не тот `open_loop`;
- `answer_obligation` мог быть слишком слабым;
- `response_mode=reflect` мог быть выбран там, где нужен прямой answer-first;
- overlay shadow мог шуметь и загрязнять trace/evaluator;
- Writer мог получать правильный KB payload, но неправильную задачу;
- evaluator мог принять формально безопасный, но слабый ответ.

Поэтому PRD-047.26 — это короткая проверка качества перед DB-track.

## 2. Цель

Создать **read-only live-quality triage harness**, который:

1. Запускает/проверяет живой backend локально или использует уже запущенный backend.

## 5. Runtime requirements

### 5.1. Общие требования

PRD-047.26 не должен менять runtime-поведение.

Все новые инструменты должны быть read-only:

```text
bot_psychologist/tools/prd_047_26_live_quality_triage.py
bot_psychologist/eval/prd_047_26_cases.json
bot_psychologist/eval/prd_047_26_schema.py
```

Если в проекте уже есть подходящие директории для eval/tools, использовать существующую структуру и не плодить новую.

### 5.2. Runner modes

Runner должен поддерживать два режима.

#### `--mode dry-run`

Не вызывает backend/LLM.  
Проверяет:

- валидность cases;
- валидность expected signals;
- наличие нужных artifact paths;
- доступность отчётов PRD-047.25;
- source gate.

#### `--mode live`

Запускает live-прогоны через локальный backend или подключается к уже запущенному backend.

Runner должен:

- не требовать внешнего браузера;
- использовать API/Web Chat path, который реально близок к live manual Web Chat;
- корректно передавать кириллицу;
- сохранять request/response/trace без raw secrets;
- не сохранять `.env`, keys, provider raw payload с секретами.

### 5.3. Backend lifecycle

Runner должен аккуратно работать с backend:

- если backend уже запущен — использовать его;
- если backend не запущен и в проекте уже есть managed-start pattern — использовать существующий managed runner;
- если backend стартовать нельзя — live mode должен завершиться `warning`, а не подделывать evidence.

В отчёте обязательно указать:

```json
{
  "backend_mode": "existing|managed_start|unavailable",
  "backend_url": "...",
  "live_mode_status": "passed|warning|blocked",
  "reason": "..."
}
```

## 6. Triage case set

Создать компактный набор из **12 live-кейсов**.

Файл:

```text
bot_psychologist/eval/prd_047_26_live_quality_cases.json
```

Каждый кейс должен иметь:

```json
{
  "case_id": "Q26-001",
  "title": "...",
  "turns": [
    {"role": "user", "content": "..."}
  ],
  "case_type": "direct_kb_answer|elliptical_followup|summary_request|practice_request|support|overlay_noise_probe",
  "expected_answer_shape": {
    "should_answer_directly": true,
    "should_not_reask_definition": true,
    "should_not_open_new_topic": true,
    "should_not_dump_raw_kb": true,
    "should_preserve_living_tone": true
  },
  "expected_trace_signals": {
    "writer_kb_payload_expected": true,
    "overlay_apply_must_be_false": true,
    "legacy_query_builder_must_be_false": true,
    "current_turn_focus_expected": true
  },
  "failure_class_candidates": [
    "dialogue_act_error",
    "answer_obligation_error",
    "overlay_shadow_noise",
    "evaluator_false_pass"
  ]
}
```

### 6.1. Обязательные типы кейсов

Минимум:

1. **Direct KB answer / known concept**
   - вопрос про `Программа несовершенное Я`;
   - вопрос про `пять драйверов выживания`;
   - вопрос про `Нейросталкинг`.

2. **Answer-first vs reflective mode**
   - пользователь задаёт прямой терминологический вопрос;
   - бот должен ответить, а не спрашивать пользователя “что вы имеете в виду”.

3. **Elliptical follow-up**
   - пользователь спрашивает “а подробнее про второй?” после ответа со списком;
   - retrieval может наследовать тему только если это реально follow-up.

4. **Summary request**
   - пользователь просит “суммируй наш разговор”;
   - бот должен суммировать текущий conversation context, а не идти в KB.

5. **Practice request**
   - пользователь просит практику;
   - бот может дать короткую практику, но без raw quote dump и без перегруза.

6. **Support / low-resource**
   - пользователь пишет эмоционально, без явного KB ask;
   - бот не должен тащить теорию и overlay.

7. **Overlay noise probe**
   - кейс, где overlay раньше мог дать false positive;
   - overlay должен оставаться trace-only и не загрязнять final answer.

## 14. Acceptance criteria

### 14.1. Passed

Можно ставить `passed`, если:

```text
- source gate passed;
- dry-run passed;
- live-run executed at least 12 cases;
- all required artifacts created;
- no mutation proof passed;
- tests passed;
- trace contains enough evidence to classify failures;
- overlay_apply_detected_count=0;
- internal_leak_count=0;
- raw_kb_dump_count=0;
- unsafe_practice_count=0;
- diagnostic_overclaim_count=0;
- db_track_readiness=ready или ready_with_warning;
- next PRD recommendation is explicit.
```

### 14.2. Passed with warning

Ставить `passed_with_warning`, если:

```text
- live backend unavailable, но dry-run/source/tests passed и report честно фиксирует warning;
- часть trace fields отсутствует, но это зафиксировано как trace_missing_evidence;
- direct_answer_success_rate ниже 1.0, но failure classes понятны;
- overlay_false_positive_count > 0, но overlay_apply_detected_count=0;
- DB-track можно начинать только с warning.
```

### 14.3. Blocked

Ставить `blocked`, если:

```text
- source gate failed;
- runner не может выполнить dry-run;
- artifacts отсутствуют;
- no-mutation proof failed;
- overlay_apply_detected_count > 0;
- есть internal payload leak;
- есть raw KB dump в final answer;
- live evidence нельзя доверять из-за transport/encoding corruption;
- невозможно понять, какие слои участвовали в ответе.
```
