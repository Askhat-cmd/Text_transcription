**Главная цель:** добиться единой runtime-правды между Admin, Web Chat и backend для semantic cards, а также исправить два route-багa без создания новых параллельных подсистем.

## 0. Краткое резюме простыми словами

После `PRD-047.27-HF1` агент доказал в автоматическом runner’е, что semantic cards видны и выбираются.

Но ручной чат владельца проекта показал другое:

```text
Semantic Cards Pilot | disabled | cards 0
STATUS = disabled
ENABLED SOURCE = default
LOADED CARDS = 0
```

При этом агент утверждает, что backend был поднят с:

```text
SEMANTIC_CARDS_PILOT_ENABLED=true
```

Значит, проблема не в самих semantic cards, а в **runtime parity**:

```text
автоматический proof смотрит на один runtime,
а реальный Web Chat владельца проекта — на другой runtime или другую effective-config правду.
```

Кроме того, в ручном `Чат_4` обнаружены два route-багa:

1. `работе` ошибочно может ловиться как meta marker `бот` → `dialogue_act=meta_system_feedback`.
2. Фраза `мне не нужна практика, я просто хочу понять причину...` проваливается в `unknown/continue_thread` и даёт нерелевантный ответ.

Этот HF2 должен исправить именно это.

## 1. Очень важный архитектурный принцип HF2

Владелец проекта отдельно подчеркнул риск:

```text
проект начинает усложняться;
слишком много переключателей, режимов, fallback’ов и параллельных подсистем;
нужно двигаться к единой инфраструктуре бота.
```

Поэтому HF2 запрещено использовать как повод добавить ещё один режим или ещё один флаг.

Главное правило:

```text
не плодить варианты;
свести существующие поверхности к одной runtime-правде.
```

HF2 не должен создавать:

```text
- новый runtime path;
- новый semantic cards mode;
- новый owner-only flag;
- новый admin-only override;
- новый LLM-agent;
- новый routing subsystem;
- новую параллельную схему payload;
- ещё один fallback для обхода текущей проблемы.
```

Если возникает желание добавить новый переключатель — остановиться и использовать уже существующий effective config.

## 5. Runtime parity requirements

### 5.1. Единственная runtime-правда

Все поверхности должны использовать один effective config resolver:

```text
Admin Runtime
Web Chat trace
Debug API
Managed verification runner
Manual owner Web Chat
```

Запрещено, чтобы runner выставлял env/config отдельно и доказывал то, чего нет в owner Web Chat.

### 5.2. Runtime identity trace

В Admin Runtime и каждом Web Chat trace должен быть виден один блок runtime identity:

```json
{
  "backend_pid": 22496,
  "backend_start_time": "...",
  "backend_port": 8001,
  "app_env": "local",
  "debug_trace_enabled": true,
  "semantic_cards_pilot_enabled": true,
  "semantic_cards_pilot_enabled_source": "env",
  "semantic_cards_pilot_raw_value": "true",
  "semantic_cards_pack_id": "semantic_cards_pilot_v1",
  "semantic_cards_loaded_count": 12
}
```

Если backend port нельзя надёжно определить, report должен объяснить почему и дать альтернативный proof через endpoint URL / process identity.

### 5.3. Manual owner Web Chat proof

HF2 считается успешным только если **обычный Web Chat владельца проекта** показывает:

```text
Semantic Cards Pilot | enabled | cards 12
STATUS = ready/enabled
ENABLED SOURCE = env
LOADED CARDS = 12
SELECTED COUNT > 0 на knowledge-вопросах
```

Не засчитывается proof, если он снят только с managed runner и не совпадает с owner Web Chat.

### 5.4. No new switch

Запрещено добавлять:

```text
OWNER_WEB_CHAT_SEMANTIC_CARDS_ENABLED
SEMANTIC_CARDS_WEB_TRACE_FORCE_ENABLE
SEMANTIC_CARDS_ADMIN_OVERRIDE
SEMANTIC_CARDS_RUNTIME_PARITY_MODE
```

Использовать только существующий:

```text
SEMANTIC_CARDS_PILOT_ENABLED
```

и существующий effective config mechanism.

## 7. Route repair requirements

### 7.1. Meta-system false positive repair

Bug:

```text
“Я разговариваю на работе...”
```

was routed as:

```text
dialogue_act=meta_system_feedback
```

Likely reason:

```text
работе contains substring бот
```

Required repair:

```text
- meta-system markers must use word boundaries / tokenized matching;
- “бот” matches only as standalone word or clear bot/system reference;
- “работа”, “работе”, “работаю”, “разработчик” must not trigger meta_system_feedback.
```

Expected route for this user message:

```text
dialogue_act=concrete_situation_question or direct_question
answer_obligation=answer_concrete_situation or answer_direct_question
must_answer=current user message
```

### 7.2. Explicit no-practice + cause-understanding repair

Bug message:

```text
мне не нужна практика, я просто хочу понять как быть с самой причиной гнева а не с ее последствиями!
```

Bad observed route:

```text
dialogue_act=unknown
answer_obligation=continue_thread
must_answer=previous question
answer became unrelated “open task for 5 minutes”
```

Required route:

```text
dialogue_act=concrete_situation_question or knowledge_question
answer_obligation=answer_concrete_situation or answer_knowledge_question
answer_shape=mechanism_explanation / contextual_explanation
practice_instruction=no_exercise_but_answer_normally
question_policy=optional_none
must_answer=current user message
```

Required answer behavior:

```text
- explain cause of anger toward perceived lying;
- do not offer practice;
- do not answer with task/productivity template;
- do not use previous stale must_answer;
- answer current user request directly.
```

### 7.3. Must-answer current turn rule

For current user messages that contain any of these explicit signals:

```text
не нужна практика
не хочу практику
хочу понять причину
не последствия
почему меня так злит
что с причиной
как быть с причиной
```

`final_answer_directive.must_answer` must be the current user message, not a previous user question.

No new dialogue act required unless absolutely necessary. Prefer existing labels:

```text
concrete_situation_question
knowledge_question
direct_question
```

## 13. Acceptance criteria

HF2 is `passed` only if:

```text
- owner Web Chat trace and Admin Runtime show same semantic cards enabled state;
- semantic_cards_pilot.enabled=true when SEMANTIC_CARDS_PILOT_ENABLED=true;
- loaded_card_count=12 in owner Web Chat runtime;
- selected_card_ids visible for expected knowledge cases;
- semantic_card_id visible in Writer KB Payload chunk summaries;
- no-theory case suppresses cards with explicit reason;
- “работе” no longer triggers meta_system_feedback;
- no-practice/cause request routes to direct/concrete/knowledge answer;
- must_answer=current user message in repaired cases;
- no new semantic cards/runtime feature flags added;
- runtime consolidation note created;
- tests pass;
- no DB/Chroma/registry/processed_blocks mutation;
- no Writer prompt rewrite;
- no new runtime path;
- no new LLM-agent.
```

HF2 is `passed_with_warning` only if:

```text
- semantic cards parity is fixed;
- route bugs are fixed;
- but Thread Manager still shows confusing new_thread labels without breaking behavior;
- or screenshots are unavailable but JSON/trace proof is complete.
```

HF2 is `blocked` if:

```text
- owner Web Chat still shows semantic cards disabled/cards 0 while env true;
- runner proof and owner Web Chat proof still disagree;
- route bug with “работе” remains;
- no-practice/cause request still gives practice/stale answer;
- must_answer remains stale in repaired cases;
- new runtime path or new flag added;
- DB/Chroma mutated;
- tests fail.
```

```text

one effective config;
one Writer KB payload path;

Admin and Web Chat show the same reality.
```
