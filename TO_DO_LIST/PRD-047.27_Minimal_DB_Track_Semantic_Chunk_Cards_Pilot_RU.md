## 0. Краткое резюме простыми словами

После `PRD-047.26-HF1` бот стал лучше понимать прямые запросы пользователя: где нужно объяснить, где дать одну практику, где ответить на конкретный вопрос.

Теперь можно осторожно двигаться к БД.

Но не нужно сразу переделывать всю базу, reindex Chroma, строить большой механизм metadata и раздувать архитектуру.

Этот PRD должен сделать маленький пилот:

```text
взять ограниченный набор важных знаний
→ оформить их как semantic chunk cards
→ подключить их к существующему Writer KB payload в pilot mode
→ проверить, что ответы стали глубже, точнее и живее.
```

Semantic chunk card — это не просто кусок текста. Это карточка, которая объясняет Writer’у:

```text
о чём этот материал,
какой механизм он описывает,
когда его можно использовать,
когда нельзя,
насколько глубоко идти,
можно ли давать практику,
как сказать это живым языком.
```

Главное ограничение:

```text
карточка не командует Writer’ом.
Она только помогает ему видеть материал точнее.
```

## 1. Контекст

### 1.1. Почему сейчас можно идти в DB-track

`PRD-047.26-HF1` закрыл route/obligation/evaluator debt:

```text
direct_answer_success_rate = 1.0
dialogue_act_error_count = 0
answer_obligation_error_count = 0
evaluator_false_pass_count = 0
db_track_readiness = ready_with_warning
```

Остался warning:

```text
overlay_false_positive_count = 8
```

Но overlay сейчас trace-only и не применяется к финальному ответу. Поэтому он не должен блокировать маленький DB-track pilot.

### 1.2. Почему не делаем HF2 прямо сейчас

Проект пилотный. Реальных пользователей пока нет, кроме владельца/разработчика проекта.

Поэтому мы не должны бесконечно полировать trace-слои, если они не ломают живой ответ.

Правильный следующий шаг:

```text
не идеальный overlay,
а первый ощутимый прирост качества знаний в живом боте.
```

## 2. Цель PRD

Сделать первый минимальный механизм-aware DB pilot.

А именно:

1. Выбрать небольшой набор важных материалов из текущей базы/контекста.
2. Создать pilot-набор semantic chunk cards.
3. Не менять всю БД.
4. Не reindex всё подряд.
5. Не делать новый runtime path.
6. Не добавлять LLM-agent.
7. Подключить карточки к существующему Writer KB payload / context package только в pilot/dev режиме.
8. Проверить на live-кейсах, что ответы стали содержательнее и не стали более “методичными”.

## 3. Non-goals

В рамках PRD запрещено:

```text
- делать полный DB migration;
- менять всю Bot_data_base schema;
- reindex Chroma globally;
- превращать markers в if/else rule engine;
- делать relation graph;
- добавлять новый LLM enrichment agent;
- добавлять новый user-facing runtime path;
- менять Writer prompt глобально;
- переписывать retrieval ranking;
- включать overlay apply;
- строить Web Admin UI для редактирования карточек;
- пытаться покрыть всю Кузницу Духа;
- делать универсальный ingestion pipeline для всех будущих книг.
```

Разрешено:

```text
- создать маленький pilot pack карточек;
- создать schema / loader / validator для карточек;
- создать adapter, который превращает semantic cards в Writer KB payload-friendly формат;
- включить pilot mode через env/config flag, default off;
- использовать pilot cards в local/dev/live smoke;
- создать отчёты, tests, no-mutation proof и live evidence.
```

## 4. Главный продуктовый критерий

После PRD бот должен не просто “найти похожий кусок текста”, а лучше отвечать по смыслу.

Пример:

Пользователь спрашивает:

```text
Что такое программа «Несовершенное Я»?
```

Бот должен получить не только фрагмент книги, а компактную смысловую опору:

```text
это не диагноз;
это устойчивый паттерн “со мной что-то не так”;
программа не равна человеку;
объяснять мягко, без давления;
не давать практику без запроса;
глубина 1–2.
```

И ответить живо, прямо и понятно.

## 5. Pilot scope

Сделать только маленький pilot pack.

Минимальный состав:

```text
10–20 semantic chunk cards
```

Обязательные темы:

1. `program_imperfect_self` — программа «Несовершенное Я».
2. `five_survival_drivers` — пять драйверов выживания.
3. `be_strong_driver` — драйвер «Будь сильным».
4. `be_best_driver` — драйвер «Будь лучшим».
5. `please_others_driver` — драйвер «Радуй других».
6. `try_harder_driver` — драйвер «Старайся сильнее».
7. `hurry_up_driver` — драйвер «Спеши».
8. `control_as_safety` — контроль как безопасность.
9. `fact_vs_interpretation` — факт / интерпретация.
10. `panic_control_support` — почему при панике контроль усиливается.
11. `one_bounded_practice_not_self_improvement_whip` — практика не как плётка самосовершенствования.
12. `neurostalking_basic_lens` — Нейросталкинг как наблюдение механизмов, не духовный авторитет.

Если есть ресурс, можно добавить до 20 карточек, но не больше.

## 6. Semantic Chunk Card contract

Создать JSON schema / dataclass, например:

```text
bot_psychologist/bot_agent/knowledge/semantic_chunk_card.py
```

Минимальные поля карточки:

```json
{
  "card_id": "program_imperfect_self_v1",
  "schema_version": "semantic_chunk_card_v1",
  "title": "Программа «Несовершенное Я»",
  "chunk_type": "concept|mechanism|practice|diagnostic_lens|dialogue_move|anti_pattern|safety|source_fragment",
  "source_ref": {
    "source_doc": "КУЗНИЦА ДУХА v.2",
    "section": "Глава 1",
    "source_block_id": "optional"
  },
  "core_thesis": "Краткая смысловая суть.",
  "mechanism_hints": ["imperfect_self", "shame", "self_fixing_loop"],
  "user_markers_examples": ["со мной что-то не так", "я недостаточно хорош"],
  "allowed_use": ["direct_to_writer", "internal_lens"],
  "avoid_when": ["acute_crisis", "user_asked_no_theory"],
  "depth_level": 1,
  "quote_policy": "paraphrase_only|can_quote_short|internal_only|do_not_use",
  "practice_policy": "no_practice_unless_requested|practice_allowed_if_explicit|not_a_practice",
  "writer_instruction": "Объяснять как механизм, не как диагноз.",
  "safety_notes": [],
  "writer_can_ignore": true
}
```

Обязательные правила:

```text
- writer_can_ignore всегда true;
- card не является командой;
- card не создаёт deterministic routing;
- user_markers_examples не используются как жёсткие if/else;
- depth_level — подсказка дозировки, не приказ;
- allowed_use / quote_policy обязательны;
- practice cards не должны попадать как практика без explicit request.
```

## 9. Adapter to Writer KB payload

Создать adapter:

```text
bot_psychologist/bot_agent/knowledge/semantic_card_payload_adapter.py
```

Назначение:

```text
semantic chunk card
→ compact Writer KB payload item
```

Writer-visible item должен содержать только безопасные поля:

```json
{
  "chunk_id": "semantic_card:program_imperfect_self_v1",
  "source_doc": "КУЗНИЦА ДУХА v.2",
  "chunk_type": "concept",
  "core_thesis": "...",
  "content_excerpt": "...",
  "allowed_use": ["direct_to_writer"],
  "quote_policy": "paraphrase_only",
  "writer_instruction": "...",
  "depth_level": 1,
  "writer_can_ignore": true
}
```

Не передавать Writer’у:

```text
- raw long source text;
- internal-only diagnostics;
- hidden evaluator notes;
- raw trace details;
- private/debug data.
```

## 10. Pilot mode integration

Добавить feature flag:

```text
SEMANTIC_CARDS_PILOT_ENABLED=false
```

Default:

```text
false
```

Разрешить включать только local/dev/test:

```text
APP_ENV=local|dev|test
```

Если `SEMANTIC_CARDS_PILOT_ENABLED=true`, existing retrieval / context assembly может дополнительно подмешивать подходящие semantic cards в Writer KB payload.

Важно:

```text
- не создавать новый runtime path;
- не менять retrieval ranking globally;
- не заменять обычные semantic hits;
- не трогать Chroma;
- не делать карточки главнее Writer’а;
- не делать overlay apply.
```

Пилотная логика matching должна быть простой:

```text
current user message + current_turn_focus
→ keyword/semantic-lite match по title/core_thesis/user_markers_examples/mechanism_hints
→ максимум 1–3 cards в context package
```

Но запрещено превращать это в rule engine.

Допустимо сделать simple scorer:

```text
score = title/core_thesis/marker overlap + existing retrieval context hint
```

Сохранить trace:

```json
{
  "semantic_cards_pilot": {
    "enabled": true,
    "selected_card_count": 2,
    "selected_card_ids": ["program_imperfect_self_v1"],
    "selection_reason": "title/core_thesis/current_turn_overlap",
    "writer_can_ignore": true,
    "applied_as_authority": false
  }
}
```

## 17. Acceptance criteria

PRD считается `passed`, если:

```text
- source_gate passed;
- semantic cards schema/load/adapter tests passed;
- pilot selection tests passed;
- regression subset passed;
- dry-run passed;
- live pilot executed;
- direct_answer_success_rate >= 0.9;
- internal card leak count = 0;
- raw source dump count = 0;
- practice overpush count = 0;
- no DB/Chroma/registry mutation;
- feature flag default off;
- Writer remains final author;
- no new runtime path;
- no new LLM-agent.
```

`passed_with_warning`, если:

```text
- technical gates pass;
- cards work;
- but answers are still too textbook / tone warning appears;
- or selected_card_count too often 0 on expected knowledge cases;
- or overlay warning remains but does not affect output.
```

`blocked`, если:

```text
- карточки попали в ответ как raw internal metadata;
- Writer начал цитировать internal-only fields;
- practice card предлагается без explicit practice request;
- DB/Chroma были мутированы без разрешения;
- появился новый runtime path;
- feature flag default on;
- tests fail;
- live pilot corrupt / encoding broken.
```
