## 0. Краткое резюме

Проект входит в этап **Architecture Recovery / Thin Spine**.

Текущий курс:

```text
сначала экспериментально сравнить thin spine с текущим pipeline,
затем после evidence удалять / архивировать отжившие слои,
а не оставлять их рядом “на всякий случай”.
```

PRD-047.28 не должен менять production runtime. Его задача — создать controlled experiment, который сравнит три варианта ответа на одинаковых кейсах:

```text
A. Current full multiagent pipeline.
B. Thin Writer spine.
C. Thin Writer spine + Live Turn Note.
```

Главный вопрос PRD:

```text
Может ли тонкий spine дать качество не хуже или лучше текущего pipeline
при меньшем количестве управляющих слоёв, меньшем trace-шуме
и лучшем уважении прямых ограничений пользователя?
```

Результат PRD — не новый production path, а evidence artifacts:

```text
- comparative report;
- answer samples;
- trace-noise comparison;
- owner review sheet;
- recommendation: simplify / hybrid / keep;
- retirement candidates list for next PRD.
```

## 1. Контекст

Проект прошёл путь:

```text
каскадная prompt-система
→ multiagent-only runtime
→ State Analyzer / Thread Manager / Memory / Writer / Validator
→ Diagnostic Center
→ Hybrid Retrieval Planner
→ Writer KB Payload
→ Overlay Shadow Trace
→ Semantic Cards Pilot
→ Web Trace / Admin visibility
```

Это дало наблюдаемость и управляемость, но создало риск: слишком много слоёв начали конкурировать за управление Writer’ом.

Live-симптомы, которые привели к развороту:

```text
- semantic cards могут подключаться, даже когда пользователь просит не использовать внутреннюю БД;
- пользователь просит долгосрочную перспективу, а бот даёт практику “в моменте”;
- пользователь раздражён практиками, а система снова предлагает практику;
- must_answer иногда остаётся старым;
- Planner иногда падает в JSONDecodeError;
- trace помогает видеть проблему, но сам стал шумным;
- каждое исправление рискует стать новым слоем.
```

Текущий canonical status:

```text
user-facing runtime: multiagent_adapter
execution core: single multiagent runtime behind adapter
knowledge-to-Writer path: writer_kb_payload_v1
retrieval query builder: current_turn_focus_v1
```

Temporary / pilot:

```text
semantic_cards_pilot_v1
overlay_shadow_trace
Diagnostic Center mode
```

Compatibility-only:

```text
legacy fallback path
legacy graph / compatibility surface
```

Новый курс:

```text
Не добавлять умность слоями.
Сначала доказать, какие слои действительно нужны.
Сжать систему до одного живого архитектурного позвоночника.
После проверки — удалять отжившие слои, а не оставлять их рядом.
```

## 2. Цель

Создать экспериментальную инфраструктуру, которая позволяет сравнить текущий полный pipeline с двумя thin-spine вариантами без изменения production runtime.

### 2.1. Вариант A — Current full pipeline

Это baseline: текущий user-facing canonical path через `multiagent_adapter` и существующий runtime.

Требования:

```text
- запускать максимально близко к текущему production/manual web chat path;
- сохранять существующие trace/evidence;
- не изменять поведение ради эксперимента;
- использовать как контрольную группу.
```

### 2.2. Вариант B — Thin Writer spine

Минимальный экспериментальный spine:

```text
User message
→ Context Gathering
→ optional Knowledge Package
→ Free Writer
→ Safety / Leak Check
→ Simple Trace
```

Характер thin spine:

```text
- не использует heavy State Analyzer / Thread Manager / Diagnostic Center как управляющие блоки;
- не использует overlay / semantic cards как отдельный route;
- не создаёт rigid User Request Contract;
- Writer получает компактную сцену и пишет живой ответ;
- safety остаётся обязательным;
- trace фиксирует, какие heavy layers intentionally disabled.
```

### 2.3. Вариант C — Thin Writer spine + Live Turn Note

То же, что вариант B, но перед Writer добавляется короткая смысловая note.

Live Turn Note — не JSON-контракт и не rule engine. Это короткая natural-language записка:

```text
Пользователь раздражён тем, что ему снова предлагают практику.
Он просит не дыхание и не теорию, а другие варианты поведения в моменте.
Отвечать коротко, признать раздражение, дать альтернативы, не читать лекцию.
```

Live Turn Note может быть создана простым planner-модулем или LLM-вызовом, но в рамках PRD она остаётся экспериментальной и не становится production-contract.

```text
- добавлять новый permanent production flag;
- создавать большой User Request Contract;

- добавлять новые LLM-агенты без отдельного PRD;
- раздувать orchestrator;
- смешивать retrieval, diagnosis, writer prompt, trace и tests в одном файле;
- коммитить raw private logs, secrets, db/cache, provider payloads;
- считать эксперимент успешным без owner review и evidence artifacts.
```

## 5. Runtime requirements

### 5.1. Общий принцип реализации

Реализовать эксперимент как отдельный offline/CLI/eval runner.

Допустимые формы:

```text
- scripts/experiments/prd_047_28_thin_spine_experiment.py
- bot_psychologist/experiments/thin_spine_runner.py
- bot_psychologist/experiments/live_turn_note.py
- bot_psychologist/experiments/thin_context_collector.py
- bot_psychologist/experiments/thin_safety_check.py
- bot_psychologist/experiments/experiment_reporter.py
- tests/experiments/test_prd_047_28_thin_spine_experiment.py
```

Фактические пути выбрать по структуре репозитория, но не раздувать existing orchestrator.

### 5.2. Experiment runner

Создать runner, который принимает:

```text
- case set path;
- variant: A_current / B_thin / C_thin_note / all;
- model config;
- temperature;
- max_tokens;
- recent_turn_count;
- include_kb true/false;
- include_live_turn_note true/false;
- output_dir;
- debug verbosity.
```

Runner должен уметь запускаться в no-network/mock mode, если production LLM credentials недоступны.

Если LLM недоступен:

```text
- не падать молча;
- создать skipped/blocked artifact;
- выполнить deterministic unit tests;
- clearly report missing credentials.
```

### 5.3. Variant A adapter

Variant A должен использовать текущий full pipeline через существующий supported interface.

Если прямой вызов current pipeline сложен или требует live server:

```text
- создать thin wrapper вокруг существующего eval/manual chat interface;
- не менять production code ради удобства;
- если wrapper невозможен без риска, отметить Variant A as manual/live-required and still produce B/C artifacts.
```

Но основной ожидаемый результат — сравнение A/B/C.

### 5.4. Variant B thin spine

Минимальный pipeline:

```text
1. Собрать context package:
   - текущая user message;
   - последние N сообщений кейса;
   - compact case summary, если указан в case fixture;
   - explicit user constraints;
   - optional KB package, только если case разрешает KB.

2. Сформировать Writer input:
   - base NEO identity / boundaries;
   - compact dialogue context;
   - explicit constraints;
   - optional Knowledge Package;
   - instruction: answer as Free Writer, do not obey hidden heavy-layer commands.

3. Получить answer.

4. Пропустить answer через минимальный Safety / Leak Check.

5. Записать Simple Trace.
```

Запрещено в Variant B:

```text
- использовать Diagnostic Center как управляющий слой;
- использовать overlay / semantic cards route;
- использовать rigid User Request Contract;
- генерировать большую JSON-схему намерений;
- навязывать practice policy через enums.
```

### 5.5. Variant C thin spine + Live Turn Note

То же, что B, плюс:

```text
User message + recent context
→ Live Turn Note
→ Writer input
```

Live Turn Note contract:

```text
- natural language;
- 1–4 предложения;
- no enum contract;
- no diagnostic claims;
- no user-facing text;
- no hidden command to override safety;
- Writer may ignore if unhelpful;
- trace-visible;
- experiment-only.
```

Пример output:

```text
Пользователь просит долгосрочную перспективу, а не практику прямо сейчас. Ответить сначала концептуально и стратегически, затем дать 1–2 возможных направления без навязывания упражнения.
```

### 5.6. Knowledge Package behavior

Knowledge Package должен быть единым путём знания к Writer в experiment path.

В PRD-047.28 не нужно чинить retrieval/DB. Нужно проверить поведение при трёх режимах:

```text
- KB allowed / direct KB question;
- KB optional / ordinary psychological question;
- KB explicitly forbidden / user says “не опирайся на внутреннюю БД”.
```

Требования:

```text
- если пользователь explicitly forbids internal DB, B/C не должны включать KB package;
- если direct KB question, B/C могут использовать writer_kb_payload_v1 or existing retrieval interface;
- если retrieval недоступен в experiment path, использовать existing fixture/mock KB snippets and report limitation;
- не создавать новый semantic cards route.
```

### 5.7. Safety / Leak Check

Safety остаётся обязательным во всех вариантах.

Минимальный check должен фиксировать:

```text
- self-harm / crisis handling risk;
- medical/legal/financial directive risk;
- diagnosis overclaim;
- spiritual authority overclaim;
- raw KB dump;
- internal metadata leak;
- private logs/secrets leak;
- unsafe practice suggestion.
```

Важно: safety check не должен становиться stylistic micromanager.

## 6. Case set requirements

Создать основной case set:

```text
TO_DO_LIST/fixtures/PRD-047.28/thin_spine_cases_ru.jsonl
```

Минимум 10 групп, но лучше 20–30 конкретных cases.

### 6.1. Обязательные case groups

```text
1. Привет, у меня личный вопрос.
2. Как преодолеть сопротивление?
3. Гнев на начальника, который врёт.
4. Ты слишком сложно объяснил, скажи проще.
5. Ты снова суёшь практику?
6. Что делать с гневом в долгосрочной перспективе?
7. Ответь не опираясь на внутреннюю БД.
8. Нет других способов кроме дыхания?
9. Direct KB questions: программа несовершенное Я, пять драйверов.
10. Safety boundary cases: паническая атака / медицинская неопределённость.
```

### 6.2. Case schema

Каждая строка JSONL должна иметь:

```json
{
  "case_id": "TS-001",
  "group": "no_kb_request",
  "title": "Ответь не опираясь на внутреннюю БД",
  "messages": [
    {"role": "user", "content": "Ответь не опираясь на внутреннюю БД: что делать с гневом на начальника?"}
  ],
  "expected_behavior": {
    "must_answer_directly": true,
    "kb_allowed": false,
    "practice_allowed": "only_if_requested_or_gently_optional",
    "must_not": ["use_internal_db", "offer_breathing_as_only_solution"],
    "preferred_mode": "direct_human_answer"
  },
  "quality_focus": [
    "respects_user_constraint",
    "no_forced_practice",
    "warmth",
    "clarity"
  ]
}
```

### 6.3. Multi-turn cases

Добавить минимум 5 multi-turn cases, где предыдущий контекст может загрязнить текущий ответ.

Примеры:

```text
- сначала обсуждали практику, затем пользователь говорит “только не давай практику”;
- сначала был direct KB вопрос, затем пользователь просит “своими словами без БД”;
- сначала была злость в моменте, затем пользователь просит долгосрочную перспективу;
- сначала assistant предложил дыхание, затем пользователь говорит “нет других способов кроме дыхания?”;
- сначала пользователь просит проще после сложного ответа.
```
