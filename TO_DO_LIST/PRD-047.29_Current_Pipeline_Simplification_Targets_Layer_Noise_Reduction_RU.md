## 0. Простое описание для владельца проекта

Сейчас эксперимент показал: полностью упрощённый thin spine пока рано включать. Старый текущий pipeline отвечает стабильнее. Но в нём остался шум: иногда система снова предлагает практику, когда пользователь прямо просит не давать практику; иногда плохо слышит просьбу “скажи проще” или “дай не только дыхание”; trace стал слишком большим и мешает быстро понимать, где сломался ответ.

Этот PRD нужен, чтобы не строить нового бота рядом, а улучшить уже работающий текущий pipeline: сделать его живее, прямее и менее шумным. Результат, который нужен владельцу: интерактивный пилотный бот, который в Web Chat лучше слышит последнее сообщение пользователя, не навязывает практики, уважает ограничения и даёт понятный trace без огромной простыни.


## 3. Цель

Сделать текущий pipeline лучше для живого пилотного использования владельцем проекта.

После PRD-047.29 бот должен:

```text

7. Сохранять current canonical runtime.

```

## 6. Основная задача 1 — Repair latest-turn constraint respect

### 6.1. Проблема

Пользователь может прямо сказать:

```text
- “не давай практику”;
- “не снова дыхание”;
- “скажи проще”;
- “мне нужна долгосрочная перспектива”;
- “ответь без внутренней БД”.
```

Текущий pipeline иногда слышит это слабее, чем должен, потому что на Writer давят старые слои, прошлые obligations или advisory-шум.

### 6.2. Требование

Не добавлять новый агент. Исправить внутри существующего current pipeline место, где уже собирается final directive / answer obligation / WriterContract.

Добавить или усилить компактную структуру текущего хода:

```text
latest_turn_constraints_v1
```

Это не большой User Request Contract. Это короткий список явно сказанных ограничений последнего пользовательского сообщения.

Пример:

```json
{
  "latest_turn_constraints_v1": {
    "no_practice": true,
    "no_breathing_only": true,
    "simplify": false,
    "long_term_perspective": false,
    "no_internal_db": false,
    "source": "latest_user_turn_explicit_text"
  }
}
```

Важно:

```text
- только явные ограничения;
- не диагностировать пользователя;
- не делать enum-систему на всё;
- не использовать как новый routing brain;
- передавать Writer’у как короткое обязательное ограничение текущего ответа.
```

### 6.3. Acceptance

Для live/replay cases:

```text
- если пользователь сказал “не хочу практику” → финальный ответ не предлагает практику;
- если пользователь сказал “не дыхание” → ответ не сводится к дыханию;
- если пользователь сказал “скажи проще” → ответ становится проще и короче;
- если пользователь сказал “долгосрочно” → ответ не ограничивается моментной стабилизацией;
- если пользователь сказал “без внутренней БД” → KB/semantic cards не должны попасть в Writer-visible payload.
```

## 7. Основная задача 2 — Practice pushback hard stop

### 7.1. Проблема

PRD-047.28 показал, что forced-practice остаётся главным раздражающим классом. Для пилота это критично: если владелец пишет “ты снова суёшь практику?”, бот должен не спорить и не выдавать новую практику.

### 7.2. Требование

В existing current pipeline добавить/усилить hard stop на уровне final directive / WriterContract:

```text
Если latest user turn содержит явный отказ от практики,
то Writer должен сначала ответить по сути без практики.
```

Не надо добавлять сложные виды практик. Нужна простая граница:

```text
no_practice_explicit = true
→ no new exercise / breathing / grounding / homework in this answer
→ answer conceptually, emotionally, practically, but without “сделай упражнение”.
```

### 7.3. Разрешённый ответ

Можно:

```text
- признать раздражение;
- объяснить механизм коротко;
- дать варианты поведения без оформления как практика;
- дать прямой человеческий ответ;
- задать один точный вопрос, если он реально нужен.
```

Нельзя:

```text
- “попробуй прямо сейчас...”;
- “сделай упражнение...”;
- “подыши...”;
- “положи руку на грудь...”;
- “давай начнём с практики...”;
- пять пунктов домашнего задания.
```

## 8. Основная задача 3 — Alternatives to breathing repair

### 8.1. Проблема

Кейс `TS-008` показал слабость: если пользователь спрашивает “нет других способов кроме дыхания?”, ответ должен дать другие способы, а не снова возвращаться к дыханию.

### 8.2. Требование

В existing pipeline добавить распознавание явного запроса:

```text
no_breathing_only = true
```

При этом Writer должен дать 3–5 альтернатив, но без превращения ответа в методичку.

Примеры допустимых альтернатив:

```text
- пауза перед ответом;
- выйти из переписки / разговора на короткое время;
- записать резкую фразу в черновик, не отправляя;
- назвать границу одной фразой;
- сменить физическое действие: пройтись, умыться, выйти из комнаты;
- заранее подготовить одну спокойную формулировку.
```

Важно:

```text
дыхание можно упомянуть только как один из вариантов, если оно не является центром ответа.
```

## 9. Основная задача 4 — Simplify-after-complexity repair

### 9.1. Проблема

Если пользователь говорит “слишком сложно, скажи проще”, бот должен не продолжать объяснять слой за слоем, а сжать ответ.

### 9.2. Требование

Если `simplify=true`, WriterContract / final directive должен получить:

```text
- коротко;
- простыми словами;
- без лекции;
- 1 главная мысль;
- максимум 1 уточняющий вопрос;
- без длинной структуры.
```

### 9.3. Acceptance

Replay/live case должен показать:

```text
- ответ короче предыдущего;
- нет длинной теории;
- есть прямое объяснение;
- нет новой практики, если она запрещена.
```

## 10. Основная задача 5 — Long-term perspective respect

### 10.1. Проблема

Если пользователь просит “что делать в долгосрочной перспективе?”, бот не должен отвечать только про “переждать всплеск прямо сейчас”.

### 10.2. Требование

Если `long_term_perspective=true`, Writer должен:

```text
- сначала дать долгосрочную рамку;
- объяснить повторяющийся механизм;
- показать 2–3 направления работы;
- не сводить ответ к моментной практике.
```

Пример направления:

```text
- замечать, где злость долго копится;
- учиться выражать границу раньше, чем накопится взрыв;
- отделять факт нарушения от старой внутренней реакции;
- выбирать форму действия, а не только разрядку эмоции.
```

## 11. Основная задача 6 — Compact trace summary for pilot use

### 11.1. Проблема

PRD-047.28 показал огромный trace у current pipeline. Это полезно для глубокого debug, но для пилотного владельца мешает быстро понять ответ.

Пример проблемы:

```text
A_current может давать trace_char_count порядка сотен тысяч символов.
B/C дают короткий trace, но хуже по качеству ответа.
```

Значит, нужно не заменять pipeline, а дать compact summary поверх него.

### 11.2. Требование

Добавить compact top-level trace summary:

```text
runtime_trace_summary_v1
```

Он должен быть доступен в debug/API/Web Trace там, где уже есть trace surfaces.

Минимальные поля:

```json
{
  "runtime_trace_summary_v1": {
    "entrypoint": "multiagent_adapter",
    "latest_turn_constraints": ["no_practice", "no_breathing_only"],
    "kb_visible_to_writer": false,
    "semantic_cards_visible_to_writer": false,
    "overlay_apply_detected": false,
    "final_directive_mode": "answer_directly_without_practice",
    "practice_blocked_by_user_request": true,
    "warnings": [],
    "full_trace_available": true
  }
}
```

### 11.3. Важно

Не удалять full trace. Просто сделать верхний слой, который владелец может читать за 10 секунд.

Запрещено:

```text
- выкинуть существующий debug;
- скрыть evidence;
- заменить trace summary красивой, но ложной строкой;
- добавлять frontend-сложность, если backend summary достаточно.
```
