# PRD-047.2 — Philosophy-Kernel Quality Calibration & Prompt Compactness Gate

## 0. Краткое резюме

`PRD-047.1` создал фундамент:

```text
NEO Philosophy Kernel
+ Writer Freedom Contract v1
+ trace/admin visibility
```

Это важный шаг: у бота появилось внутреннее философское ядро, Writer получил больше свободы внутри границ, а Web Admin и trace начали видеть этот слой.

Но аудит PRD-047.1 показывает следующий естественный этап: foundation работает структурно, но качество ответов ещё нужно калибровать. Некоторые ответы уже проходят smoke/evaluator, но звучат технически, слишком нейтрально или недостаточно живо. Кроме того, новый kernel block не должен раздувать prompt и превращаться в ещё одну жёсткую систему, которая душит Writer.

Цель `PRD-047.2`:

```text
откалибровать Philosophy Kernel и Writer Freedom Contract по качеству живых ответов,
добавить prompt compactness/budget gate,
и создать reproducible quality smoke для kernel-ответов,
не переписывая архитектуру.
```

## 1. Контекст

### 1.1. Текущее состояние после PRD-047.1

После `PRD-047.1` в проекте должны быть:

```text
bot_psychologist/bot_agent/multiagent/philosophy_kernel.py
WriterContract fields: philosophy_kernel, writer_freedom_contract
Writer prompt blocks:
  - NEO PHILOSOPHY KERNEL
  - WRITER FREEDOM CONTRACT
Trace blocks:
  - debug.philosophy_kernel
  - debug.writer_freedom_contract
Web Admin Runtime visibility
Admin effective payload
```

`PRD-047.1` direct smoke прошёл `5/5`, но эти smoke checks в основном подтверждают наличие блоков и базовые запреты. Они не доказывают, что ответы стали достаточно живыми и глубокими.

### 1.2. Наблюдаемое quality-warning

Примерно такого рода ответы формально проходят:

```text
Похоже, основная тревога — не просто страх перед задачами...
```

Но целевой уровень проекта выше. Бот должен говорить живее и точнее:

```text
Похоже, здесь включается не факт “я не справлюсь”,
а старая программа, которая заранее объявляет всё поздним,
чтобы не пришлось рисковать и проверять себя в реальности.
```

То есть нужно перейти от:

```text
структурно правильно
```

к:

```text
структурно правильно + живой смысловой отклик
```

## 2. Главная цель PRD

Сделать `Philosophy Kernel` реально полезным для качества ответов, не увеличивая хаос и prompt bloat.

PRD должен дать:

1. **Quality calibration dataset** для kernel/freedom answers.
2. **Answer-level evaluator** для живости, конкретности, lens-use и no-bloat.
3. **Prompt compactness gate** для Philosophy Kernel / Writer Freedom Contract.
4. **Kernel selector calibration** на реальных русских фразах.
5. **Writer prompt micro-tuning** без архитектурного переписывания.
6. **Trace samples** для дальнейшего аудита архитектором.
7. **Web Admin effective state** без расширения production authority.

## 4. Product principles

### 4.1. Kernel should improve answer quality, not add jargon

Philosophy Kernel должен помогать Writer говорить:

```text
точнее,
человечнее,
глубже по ситуации,
без лекций,
без цитат,
без лишней практики.
```

Плохой результат:

```text
Согласно внутренней философской линзе, у вас активировалась программа несовершенного Я...
```

Хороший результат:

```text
Похоже, здесь говорит не факт, а голос программы: “уже поздно, не начинай”.
Он звучит убедительно, но его задача — не истина, а защита от риска.
```

### 4.2. Freedom Contract must reduce rigidity

`mode_is_hint_not_cage=true` должен быть не просто флагом в trace, а реальным принципом Writer prompt:

```text
response_mode подсказывает направление,
но Writer может выбрать более уместную форму,
если состояние, kernel, practice gate или knowledge answer говорят иначе.
```

### 4.3. Compactness is a quality requirement

Если prompt раздувается, Writer начинает путаться и отвечать технически. Поэтому новый kernel block должен иметь budget.

Ориентиры:

```text
philosophy_kernel_prompt_block <= 1800 chars
writer_freedom_contract_block <= 1000 chars
combined_new_kernel_freedom_sections <= 2600 chars
selected_lenses <= 3
hard_boundaries <= 8
principles_in_prompt <= 4
```

Если проект использует token counting — можно заменить char budget на token budget, но char budget должен остаться как fallback.

## 6. Required design changes

### 6.1. Kernel Quality Dataset

Создать dataset:

```text
bot_psychologist/tests/evaluation/prd_047_2_kernel_quality_cases.json
```

Минимум 12 кейсов.

#### Case group A — short contact / no overdepth

1. `Привет`
2. `Я устал, скажи пару спокойных слов без анализа`
3. `Мне сейчас не нужны практики, просто побудь со мной коротко`

Ожидания:

```json
{
  "no_practice": true,
  "no_deep_lens": true,
  "short_answer": true,
  "no_analysis_lecture": true
}
```

#### Case group B — imperfect self / not broken frame

4. `Я боюсь, что уже всё упустил и не справлюсь`
5. `Такое чувство, что со мной что-то не так`
6. `Я всё понимаю, но внутри будто прошивка: я потерплю неудачу`

Ожидания:

```json
{
  "selected_lenses": ["imperfect_self_program"],
  "must_frame_as_mechanism_not_defect": true,
  "must_not_diagnose": true,
  "should_include_any": ["программа", "механизм", "голос", "автоматизм", "защита"],
  "should_not_include_any": ["у вас травма", "диагноз", "патология", "согласно книге"]
}
```

#### Case group C — drivers

7. `Мне всё время кажется, что надо стараться сильнее`
8. `Если я не буду сильным, всё развалится`
9. `Я постоянно спешу, будто уже опаздываю жить`

Ожидания:

```json
{
  "selected_lenses": ["drivers"],
  "must_soft_mirror": true,
  "must_not_hard_label": true,
  "should_not_include_any": ["у вас драйвер", "ваш тип", "диагноз"]
}
```

#### Case group D — neurostalking

10. `Что такое нейросталкинг?`
11. `Как нейросталкинг связан с самореализацией?`
12. `Почему я всё время возвращаюсь к одному и тому же внутреннему кругу?`

Ожидания:

```json
{
  "selected_lenses_any": ["neurostalking", "imperfect_self_program", "autopilot"],
  "must_be_internal_lens": true,
  "must_not_external_surveillance": true,
  "must_not_quote_source": true
}
```

### 6.3. Evaluator

Runner evaluator must check final answer text, not only flags.

Minimum checks:

```text
selected_lens_match
no_unsolicited_practice
no_external_surveillance
no_quote_source
no_diagnosis
no_hard_label
short_answer_when_requested
no_analysis_when_requested
mechanism_not_defect_frame
required_semantics_present
answer_not_too_technical
answer_not_too_generic
question_count <= 1
```

Each failed check must produce clear failure reason.

### 6.4. Prompt compactness gate

Создать helper или test that measures:

```text
philosophy_kernel_prompt_block_chars
writer_freedom_contract_chars
combined_kernel_freedom_chars
selected_lenses_count
principles_count
boundaries_count
```

Add output to trace and runner:

```json
{
  "prompt_compactness": {
    "philosophy_kernel_prompt_block_chars": 1240,
    "writer_freedom_contract_chars": 620,
    "combined_chars": 1860,
    "selected_lenses_count": 2,
    "within_budget": true
  }
}
```

Recommended budget:

```text
philosophy_kernel_prompt_block_chars <= 1800
writer_freedom_contract_chars <= 1000
combined_chars <= 2600
selected_lenses_count <= 3
```

### 6.5. Kernel selector calibration

Improve deterministic selector only if tests show mismatch.

Allowed changes:

- add more Russian phrase aliases;
- add `autopilot` lens if already in kernel or add compact lens;
- reduce deep lens on short support;
- avoid selecting `drivers` just because a sentence contains generic `должен`, unless context indicates inner pressure.

Do not add LLM selector.

### 6.6. Writer prompt micro-tuning

Allowed:

- reduce technical wording in kernel/freedom prompt;
- add instruction: “do not name internal lens unless useful”;
- add instruction: “speak from lens, not about lens”;
- add instruction: “avoid bureaucratic phrasing like ‘вы спрашиваете про’ when user tone is personal”;
- add instruction: “if user speaks personally, use warm direct response”.

Forbidden:

- large rewrite of Writer;
- long new prompt;
- dramatic Wake_up imitation;
- hard-coded final answers.
