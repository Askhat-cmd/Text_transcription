# PRD-047.30 — Writer Input Authority Audit / KB Influence Throttle v1

**Проект:** Bot Psychologist / Neo MindBot  
**Репозиторий:** `Askhat-cmd/Text_transcription`  
**Локальный путь пользователя:** `C:\My_practice\Text_transcription`  
**Тип PRD:** runtime simplification / Writer authority recovery / pilot usability  
**Статус:** готово для IDE-агента  
**Дата:** 2026-06-24  
**Основание:** PRD-047.29 + архитектурное решение владельца проекта: не добавлять новые подсистемы, а ревизовать и правильно расставить власть существующих входов Writer’а.

---

## 0. Простое описание для владельца проекта

Этот PRD нужен, чтобы бот снова стал больше похож на живого собеседника, а не на систему, которую тянут в разные стороны база знаний, semantic cards, diagnostic hints, retrieval и старые advisory-блоки.

Главная проблема: после улучшения БД и смысловой разметки бот технически стал умнее, но местами начал отвечать хуже — промахиваться, звучать как методичка, использовать внутренние знания там, где пользователь просит живой человеческий ответ.

Главный результат, который нужен: **Writer должен снова стать главным автором ответа**. Внутренние системы должны помогать, но не командовать. БД и semantic cards должны попадать в Writer-visible prompt только когда они действительно нужны: прямой вопрос по термину, запрос к БД, концепт, источник, safety-grounding. На обычных эмоциональных/support turn’ах Writer должен сначала отвечать живо, без скрытого давления БД.

Это не PRD на новую подсистему. Это PRD на ревизию и упрощение уже существующих входов Writer’а.

## 1. Краткое резюме

После PRD-047.29 текущий pipeline стал лучше уважать явные ограничения последнего сообщения:

```text
- не давать практику;
- не сводить к дыханию;
- сказать проще;
- дать долгосрочный ракурс;
- не использовать внутреннюю БД.
```

Но осталась более глубокая архитектурная проблема:

```text
на обычных non-KB эмоциональных/support turns
Writer-visible KB payload и semantic cards всё ещё могут попадать в Writer,
даже когда пользователь не просил знание из БД.
```

Это совпадает с наблюдением владельца проекта:

```text
бот начал отвечать хуже после фазы “умной БД / смысловых чанков”.
```

Гипотеза PRD-047.30:

```text
БД и semantic cards не нужно усиливать.
Их влияние на Writer нужно поставить на правильное место.
```

Внутренняя формула:

```text
knowledge_policy.py решает: можно ли чанк использовать безопасно.
writer_context_package.py решает: нужно ли сейчас показывать его Writer’у.
Writer решает: использовать это в ответе или проигнорировать.
```

## 2. Главный архитектурный принцип

Не создавать новую подсистему.

Не добавлять нового агента.

Не расширять rule-engine.

Не лечить каждый новый пользовательский вопрос отдельным паттерном.

Нужно провести ревизию существующих входов Writer’а и закрепить простую иерархию:

```text
1. Safety floor — всегда обязателен.
2. Latest user intent — главный источник текущего ответа.
3. Conversation context — помогает не терять линию.
4. KB / semantic cards / diagnostic hints — optional, Writer can ignore.
```

Эта иерархия должна быть отражена не как новый большой JSON-контракт, а как порядок сборки Writer-visible prompt и trace-summary.

## 3. Что уже есть и что нужно использовать

### 3.1. `knowledge_policy.py`

Использовать как существующий safety/governance gate для чанков.

Его задача:

```text
- можно ли чанк вообще использовать;
- internal_only или нет;
- do_not_use или нет;
- можно ли дать Writer’у;
- можно ли использовать как diagnostic_lens;
- можно ли использовать как practice_candidate;
- есть ли mixed_intent / clinical / spiritual authority / quote risk.
```

Важно: `knowledge_policy.py` не должен становиться главным решателем “нужна ли БД в этом ответе”. Он решает безопасность и допустимость чанка, а не уместность KB-grounding для конкретного живого turn’а.

### 3.2. `writer_context_package.py`

Использовать как главную точку Writer-visible packaging.

Именно здесь должен решаться вопрос:

```text
показывать ли Writer’у KB payload / semantic cards сейчас,
или оставить их только в trace.
```

После PRD-047.29 там уже есть правильная логика для `no_internal_db=true`. PRD-047.30 должен расширить этот принцип на обычные non-KB эмоциональные/support/repair/simplify turns, но без создания новой production-подсистемы.

### 3.3. `final_answer_directive.py`

Использовать как источник текущей цели ответа и latest-turn constraints.

Но не перегружать его новой логикой чанков. Он должен сказать, что сейчас важно в ответе, а не выбирать конкретные KB items.

### 3.4. Semantic cards / Writer KB payload

Оставить как optional grounding.

Запрещено трактовать их как command layer для Writer’а.

## 6. Обязательный артефакт до кода — Writer Input Authority Map

Перед изменениями создать:


Формат таблицы:


Authority level использовать только такой:


```text

latest user message                 -> MUST_OBEY
compact conversation context        -> CONTEXT
Writer KB payload                   -> OPTIONAL_GROUNDING
semantic cards                      -> OPTIONAL_GROUNDING
diagnostic hints                    -> OPTIONAL_GROUNDING or TRACE_ONLY
retrieval traces                    -> TRACE_ONLY
overlay shadow                      -> TRACE_ONLY
legacy duplicate advisory summaries -> REMOVE_LATER_CANDIDATE or TRACE_ONLY
```

Важно: если агент обнаружит, что какой-то advisory layer фактически командует Writer’ом, он обязан это зафиксировать как conflict.

## 8. Основная задача 2 — KB / semantic cards visibility throttle

### 8.1. Проблема

Сейчас обычный эмоциональный/support turn может получить Writer-visible KB payload и semantic cards, даже если пользователь не просил знания из БД.

Это создаёт риск:

```text
- ответ звучит как методичка;
- Writer начинает пересказывать найденный материал;
- последняя живая реплика пользователя теряет приоритет;
- бот становится “умным комментатором”, а не собеседником.
```

### 8.2. Требование

В существующем `writer_context_package.py` добавить/усилить небольшую deterministic decision-функцию для Writer-visible grounding.

Это НЕ новый агент и НЕ новая подсистема.

Допустимые варианты реализации:

```text
Вариант A: private helper functions внутри `writer_context_package.py`.
Вариант B: маленький pure helper module рядом, если так проще тестировать.
```

Запрещено:

```text
- новый LLM call;
- новый planner;
- новый feature flag;
- новый route;
- новый long-lived runtime service.
```

### 8.3. Decision output

Добавить в writer context package и debug trace:

```json
{
  "writer_grounding_visibility_v1": {
    "kb_visible_to_writer": false,
    "semantic_cards_visible_to_writer": false,
    "reason": "non_kb_emotional_support_turn",
    "direct_kb_question": false,
    "safety_grounding_allowed": false,
    "no_internal_db": false,
    "trace_only_grounding_available": true
  }
}
```

Название можно скорректировать, но смысл должен быть сохранён.

### 8.4. Когда KB / semantic cards можно показывать Writer’у

Разрешить Writer-visible KB / semantic cards, если есть хотя бы одно:

```text
- прямой вопрос по термину / концепту / механизму из БД;
- пользователь явно просит материал из внутренней базы;
- запрос явно knowledge/concept/source/explanation, а не эмоциональная поддержка;
- safety boundary требует проверенного grounding;
- existing knowledge_answer_guard / retrieval plan явно показывает knowledge need;
- direct KB question из live pilot должен остаться grounded.
```

### 8.5. Когда KB / semantic cards должны быть trace-only

Подавлять Writer-visible KB / semantic cards, если turn относится к:

```text
- обычная эмоциональная жалоба / support request;
- “мне тяжело”, “я запутался”, “меня бесит”, “я злюсь”, без запроса на термин;
- repair after bad answer;
- simplify request;
- practice pushback;
- no-breathing-only request;
- no_internal_db request;
- greeting / contact / thanks / close.
```

В этих случаях:

```text
- KB/semantic evidence можно оставить в trace;
- Writer-visible chunks должны быть пустыми или минимальными;
- Writer должен получить живую сцену и последние ограничения, а не БД-подсказку.
```

## 9. Основная задача 3 — Writer prompt authority wording

### 9.1. Требование

В существующей сборке Writer prompt / WriterContract добавить короткое authority wording, без раздувания prompt.

Смысл:

```text
Safety and explicit latest-user request are mandatory.
Conversation context is supportive.
KB, semantic cards, retrieval notes, and diagnostic hints are optional grounding.
Use them only if they help the current answer.
Never let grounding change the user's actual request.
Do not sound like you are reciting internal knowledge.
```

Русский смысл для Writer:

```text
Сначала ответь на последнее сообщение пользователя.
Не позволяй внутренней БД, semantic cards или diagnostic hints менять цель ответа.
Используй знания только если они помогают, но не пересказывай их.
Если это живой эмоциональный запрос — отвечай живо и прямо.
```

### 9.2. Acceptance

```text
- prompt length не должен заметно вырасти;
- нет нового большого блока правил;
- optional grounding явно помечено как optional / can ignore;
- latest user intent и safety остаются выше KB.
```

## 10. Основная задача 4 — Duplicate advisory collapse

### 10.1. Проблема

Некоторые advisory-блоки могут повторять одно и то же:

```text
final_answer_directive говорит “не давать практику”;
legacy advisory повторяет;
dialogue policy повторяет;
writer context снова повторяет.
```

Это делает Writer prompt шумным.

### 10.2. Требование

Провести минимальное сжатие Writer-visible advisory summary:

```text
- если final_answer_directive_v1 уже несёт effective signal, не дублировать его длинной advisory-простынёй;
- оставить подробности в trace;
- Writer-visible summary сделать коротким.
```

Не удалять подсистемы физически в этом PRD, если нет безопасного proof.

### 10.3. Acceptance

```text
- prompt canvas показывает меньше дублирующих advisory-блоков;
- full trace остаётся доступным;
- behavior не ухудшается;
- next_retirement_targets.md обновлён конкретными кандидатами на удаление.
```

## 18. Важное предупреждение IDE-агенту

Не решай эту задачу как “добавить ещё один умный фильтр”.

Решай её как восстановление правильной власти:

```text
Safety обязателен.
Последнее сообщение пользователя главное.
Контекст помогает.
БД и semantic cards — справка, не начальник.
Writer пишет ответ.
```

Если возникает выбор между “добавить новый слой” и “уменьшить шум существующего слоя”, выбирай уменьшение шума.
