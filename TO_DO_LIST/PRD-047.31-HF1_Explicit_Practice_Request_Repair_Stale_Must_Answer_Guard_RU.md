## 0. Простыми словами

После PRD-047.30 бот стал лучше защищён от лишнего влияния БД: KB и semantic cards больше не лезут в обычные человеческие ответы. Это было правильное направление.

Но после чистого перезапуска владелец проекта проверил тот же сценарий и нашёл новый критичный провал:

```text
Пользователь прямо попросил практику, чтобы тренировать навык не быть реактивным,
а бот дал случайный совет про “открой задачу и сделай 5 минут”.
```

Это не просто слабый ответ. Это значит, что система потеряла главный текущий запрос и не привязала практику к живой линии разговора:

```text
сопротивление
→ гнев на начальника, который врёт
→ хочу практику, чтобы не быть реактивным
```

Этот PRD должен быстро починить именно это:

```text
если пользователь явно просит практику,
бот должен дать одну практику по текущей теме,
а не случайную универсальную рекомендацию.
```

Нам не нужен новый большой слой. Нужно исправить приоритет последней реплики, stale `must_answer`, распознавание explicit practice request и проверку релевантности практики.

## 1. Архитектурный контекст

PRD-047.30 ввёл правильный принцип:

```text
Writer остаётся главным автором ответа.
KB / semantic cards / diagnostic hints — optional grounding, а не начальники Writer’а.
```

Но обновлённый живой чат показал, что освобождённый от KB Writer всё ещё может получить неправильный фокус от старых подсистем:

```text
- State Analyzer / intent может не увидеть explicit practice request;
- Thread Manager может ошибочно открыть new_thread;
- answer_obligation / must_answer может остаться на предыдущей реплике;
- writer_grounding_visibility может слишком жёстко подавить practice/dialogue_move grounding;
- acceptance gate видит generic answer, но не заставляет ответ вернуться к текущей теме.
```

Главный диагноз:

```text
Мы уменьшили давление БД,
но теперь нужно гарантировать, что latest user request управляет ответом сильнее старого must_answer.
```

## 3. Цель PRD

Починить live Web Chat regression:

```text
explicit practice request должен побеждать stale must_answer
и приводить к одной contextual practice по текущей линии разговора.
```

Цель не в том, чтобы бот всегда давал практики. Цель точнее:

```text
если пользователь сам просит практику / упражнение / тренировку навыка,
бот должен дать практику, связанную с текущим контекстом,
если это безопасно.
```

## 5. Главная исправляемая ошибка

### 5.1. Фактический провал

Проверочный сценарий:

```text
User: привет
Assistant: приветствие

User: Как преодолеть свое сопротивление? Некоторые вещи я вообще не хочу делать. Например есть люди, которые мне неприятны, но я вынужден с ними общаться
Assistant: объясняет сопротивление

User: А если на меня накатывает гнев во время разговора, когда я вижу что мне нагло врут, но это мой начальник!
Assistant: объясняет гнев / начальник / риск последствий

User: дай мне какую нибудь практику. Чтобы я мог выработать в себе навык не быть реактивным
Assistant: Сделай один шаг прямо сейчас: открой задачу и выполни первый минимальный фрагмент в течение 5 минут.
```

Проблема:

```text
Ответ про “открой задачу” может быть уместен для прокрастинации,
но он не отвечает на запрос о нереактивности при гневе в разговоре с начальником.
```

### 5.2. Expected behavior

Ожидаемый тип ответа:

```text
- признать, что пользователь просит именно практику;
- связать практику с текущей ситуацией: гнев + ложь + начальник + риск реакции;
- дать одну короткую практику на зазор между импульсом и ответом;
- не давать длинную лекцию;
- не подсовывать случайную productivity-практику;
- не навязывать БД;
- если используются practice/dialogue_move chunks, использовать их как optional support.
```

Пример допустимого ответа:

```text
Да. Попробуй практику “Пауза перед ответом”.

Когда в разговоре с начальником ты чувствуешь вспышку гнева, твоя задача не подавить гнев, а выиграть 3–5 секунд между импульсом и словами.

Шаг:
мысленно скажи себе: “Я сейчас злюсь, потому что вижу несоответствие. Но мне важно не взорваться, а сохранить позицию”.

Потом задай один нейтральный вопрос:
“Можете уточнить, на какие факты вы опираетесь?”

Смысл практики: ты не споришь с гневом, а переводишь его из атаки в ясность. Это и есть тренировка нереактивности.
```

## 6. Required implementation behavior

### 6.1. Explicit practice request detector

Использовать существующий слой latest-turn / dialogue pragmatics / final directive. Не создавать отдельную большую подсистему.

Добавить или уточнить компактный signal:

```json
{
  "explicit_practice_request": true,
  "practice_request_target": "non_reactivity_in_current_thread",
  "practice_requested_by_user": true,
  "practice_required_because_user_requested": true
}
```

Этот signal должен включаться не только по слову “практика”, а по смыслу:

```text
- дай практику;
- дай упражнение;
- хочу потренировать;
- как выработать навык;
- что делать прямо в моменте;
- как тренировать нереактивность;
- как научиться не взрываться.
```

Но не создавать длинный список частных фраз. Это должен быть небольшой универсальный recognizer, встроенный в существующий latest-turn focus.

### 6.2. Latest user request must override stale must_answer

Если `explicit_practice_request=true`, то:

```text
must_answer = latest_user_message
answer_obligation = provide_contextual_practice
answer_shape = one_contextual_practice
practice_instruction = practice_required_because_user_requested
```

Запрещено оставлять `must_answer` на предыдущей реплике, если последняя реплика явно просит практику/упражнение.

### 6.3. Thread continuation guard

Если последняя реплика является коротким запросом практики после обсуждения конкретной проблемы, Thread Manager / dialogue pragmatics не должен автоматически открывать `new_thread`.

Для сценария из `ЧАТ_С_БОТОМ2.txt` thread должен быть:

```json
{
  "relation": "continuation",
  "inherited_topic": "anger_at_boss_lying / non_reactivity",
  "should_answer_directly": true
}
```

Допустимое правило:

```text
Explicit practice request after recent concrete emotional/problem discussion
inherits recent concrete thread unless user explicitly changes topic.
```

### 6.4. writer_grounding_visibility adjustment

PRD-047.30 не откатывать.

Но `writer_grounding_visibility_v1` должен различать:

```text
ordinary support / repair / simplify / pushback turn
→ KB and semantic cards hidden from Writer by default;

explicit practice request
→ allow narrow practice/dialogue_move/anti_pattern grounding if useful and safe.
```

Не включать всю БД обратно.

Разрешить Writer-visible grounding только для узких типов:

```text
- practice
- dialogue_move
- anti_pattern
- safety, если нужен safety floor
```

Запрещать как Writer-visible для этого hotfix, если нет прямого knowledge request:

```text
- long theory
- source_fragment raw text
- broad concept dump
- diagnostic_lens with high depth
- unrelated productivity chunks
```

### 6.5. Practice relevance guard

Добавить в acceptance gate или existing validator lightweight check:

```text
practice_irrelevant_to_user_context
```

Он должен срабатывать, если:

```text
- пользователь просит практику по текущей проблеме;
- ответ содержит практику, но она не связана с current_thread_focus;
- ответ уходит в generic productivity / task-start / 5-minute task when thread is anger/non-reactivity;
- ответ не упоминает текущий контекст, где это требуется для релевантности.
```

Если check fails, выполнить один retry с tightened directive:

```text
Give exactly one practice for the current thread: anger when speaking with a boss who lies, training non-reactivity. Do not give a generic productivity practice.
```

Это не должно становиться стилистическим micromanager. Это защита от явной нерелевантности.

## 7. Trace requirements

В trace для explicit practice request должен быть виден короткий блок:

```json
{
  "explicit_practice_request_v1": {
    "present": true,
    "source": "latest_user_message",
    "practice_target": "non_reactivity_in_current_thread",
    "inherited_topic": "anger_at_boss_lying",
    "must_answer_overridden": true,
    "old_must_answer": "...",
    "new_must_answer": "latest_user_message",
    "answer_shape": "one_contextual_practice",
    "allowed_grounding_types": ["practice", "dialogue_move", "anti_pattern", "safety"]
  }
}
```

Если old_must_answer не был stale, trace должен честно показывать:

```json
"must_answer_overridden": false
```

### 7.1. Trace summary

В пользовательском/админском trace summary добавить кратко:

```text
Detected explicit practice request. Inherited topic from recent thread. Forced contextual practice answer. KB remains optional/narrow.
```

Не раскрывать лишние внутренние raw chunks пользователю.

### 8.3. Live smoke

После реализации выполнить live Web Chat smoke после полного перезапуска backend/frontend, желательно после killing old ports.

Минимум 1 обязательный сценарий:

```text
1. привет
2. Как преодолеть свое сопротивление? Некоторые вещи я вообще не хочу делать. Например есть люди, которые мне неприятны, но я вынужден с ними общаться
3. А если на меня накатывает гнев во время разговора, когда я вижу что мне нагло врут, но это мой начальник!
4. дай мне какую нибудь практику. Чтобы я мог выработать в себе навык не быть реактивным
```

Live smoke считается пройденным, если на 4-й ход бот:

```text
- даёт практику;
- практика связана с гневом/начальником/ложью/нереактивностью;
- не отвечает generic productivity/task advice;
- trace показывает explicit_practice_request=true;
- trace показывает must_answer_latest=true или equivalent;
- writer_grounding_visibility не включает broad KB by default;
- no safety regression.
```

Дополнительно выполнить 3 коротких smoke-cases:

```text
A. Ordinary support turn without practice request → practice not forced.
B. User explicitly says “не давай практику” → practice suppressed.
C. Direct KB/source question → KB/source path preserved.
```
