## 0. Краткое описание простым языком

Сейчас внутренний runtime стал намного честнее и управляемее: видно, какие знания реально доходят до Writer, broad KB не давит на обычные support-turn’ы, а явный запрос практики получает узкую полезную опору.

Оставшаяся проблема уже не в том, что БД ломает ответ, и не в том, что практика игнорируется. Проблема тоньше: **некоторые ответы всё ещё звучат слишком методично, длинно или учебно, когда пользователь ждёт живой человеческий ответ**.

Этот PRD должен сделать следующий пилотный шаг: **калибровать форму ответа**, не переписывая архитектуру, не меняя БД, не добавляя style-agent и не строя новый rule engine.

Ожидаемый результат: после перезапуска владелец проекта задаёт небольшой набор разноплановых вопросов, раскрывает указанные вкладки trace, и видно, что бот отвечает живее, короче и точнее, при этом сохраняет safety, direct KB/source path, no-internal-db, explicit practice и runtime truth trace.

## 1. Контекст

### 1.1. Что уже сделано

PRD-047.30 вернул Writer’у больше авторства: KB / semantic cards / diagnostic hints стали optional grounding, а не начальником ответа.

PRD-047.31-HF1 починил срочный провал: явный запрос практики теперь перебивает stale `must_answer`, сохраняет current thread и допускает только узкое practice-grounding без broad KB.

PRD-047.32 сделал owner trace честнее: разделены retrieval candidates, trace-only/filtered candidates и actual Writer-visible payload; Hybrid Planner shadow failures больше не выглядят как production failure; UI/trace показывает, что реально ушло Writer’у.

### 1.2. Что показал новый owner live chat

В owner chat после PRD-047.32 ответы стали лучше, но остаётся качество формы:

- первый ответ на “Как преодолеть сопротивление?” всё ещё длинноват и похож на мини-лекцию;
- ответ на гнев к начальнику стал адекватным, но всё ещё структурно методичный;
- explicit long-term practice работает хорошо: даёт одну короткую практику и правильный Writer payload;
- trace подтверждает, что на ordinary support/direct explanation turn’ах Writer payload может быть `0`, значит длиннота возникает не от БД, а от answer-shape / prompt guidance / Writer behavior.

### 1.3. Архитектурная линия

Не создавать новый style-agent.  
Не вводить жёсткий User Request Contract.  
Не делать бесконечную настройку под конкретные фразы.  
Не менять Bot_data_base, Chroma, semantic-card content или retrieval architecture.

Работа должна идти в рамках текущего pipeline:

```text
multiagent_adapter
→ dialogue_act / final_answer_directive
→ writer_grounding_visibility
→ writer_context_package
→ Writer prompt / Writer freedom contract
→ runtime_truth_trace
```

## 2. Цель PRD

Стабилизировать форму ответов в owner pilot:

```text
1. ordinary support / explanation answers — короче, живее, меньше “механизм → список → пример → вывод”;
2. direct question — сначала прямой ответ, затем 1–2 опоры, без длинной лекции;
3. explicit practice request — одна bounded practice, привязанная к текущему thread;
4. no-practice request — объяснение без упражнения и без скрытой домашки;
5. no-internal-db request — Writer payload 0, но ответ живой;
6. direct KB/source request — grounded answer можно длиннее, но без raw KB dump;
7. safety/panic — безопасность важнее компактности.
```

## 7. Minimal Runtime Calibration

### 7.1. Не новый слой, а уточнение существующих подсказок

Можно изменять только существующие места:

```text
- dialogue_policy answer-shape hints;
- final_answer_directive shape/depth guidance;
- Writer prompt compactness wording;
- runtime trace fields for answer shape;
- acceptance gate soft checks where already exists.
```

Нельзя добавлять новый style-agent или новый route.

### 7.2. Ввести / уточнить compact answer-shape profiles как мягкие подсказки

Это не должен быть жёсткий rule engine. Это должны быть **soft answer-shape hints**, которые Writer может нарушить при safety, direct KB/source, explicit detail request или сложном user context.

Минимальные профили:

```text
ordinary_support_compact:
  target: 1–3 коротких абзаца
  shape: contact → one clear meaning → optional one next step/question
  avoid: lecture, numbered list unless user asks, long theory
  writer_payload_expected: 0 by default

ordinary_explanation_compact:
  target: 600–1400 chars soft
  shape: direct answer → 1 mechanism in simple words → 1 practical implication
  avoid: “что происходит / почему важно / что делать” as default template

explicit_practice_one_bounded:
  target: one practice only
  shape: name practice → steps → why it helps in one sentence
  must reuse current user anchors
  writer_payload_expected: 0 or 1 narrow practice/dialogue_move/anti_pattern/safety

direct_kb_grounded:
  target: can be longer
  shape: answer concept/source directly → short grounding → no raw dump
  writer_payload_expected: 1+ allowed if grounded

no_internal_db_compact:
  target: normal live answer
  writer_payload_expected: 0
  avoid: references to internal DB/source

safety_clear:
  target: clarity over compactness
  shape: immediate safety/medical/crisis guidance if needed
```

### 7.3. Снизить methodical default

Для ordinary support/explanation turns Writer prompt должен мягко запрещать дефолтную структуру:

```text
“механизм → список подходов → пример → что помнить”
```

Если пользователь задал широкий вопрос, Writer может дать структуру, но не должен автоматически превращать каждый ответ в мини-методичку.

Рекомендуемая формулировка для prompt/directive:

```text
For ordinary support or explanation turns, do not default to a teaching article. Prefer a human answer: one main recognition, one simple explanation, one useful next move. Use lists only when they make the answer clearer or the user asks for options.
```

На русском в prompt можно формулировать так:

```text
На обычных support/explanation ходах не отвечай как методичка. Сначала дай живое попадание в смысл, затем одно простое объяснение и один возможный следующий ход. Не делай список по умолчанию, если пользователь не просит варианты.
```

```text

```

Ожидание: не мини-лекция. Ответ живой, с одним главным механизмом и одним следующим ходом. Не больше 1 короткого списка, если список вообще нужен.



Ожидание: сохранить контекст, дать рабочий ответ, не уходить в длинную теорию. Можно дать одну-две фразы поведения.


#### D. Explicit long-term practice

Ожидание: `practice_request`, одна bounded practice, current thread anchors preserved: гнев / враньё / начальник / нереактивность. Narrow Writer payload 0 или 1; если 1 — только allowed practice/dialogue_move/anti_pattern/safety.

Открыть trace:

```text

Final Answer Directive

Runtime Truth Trace
Writer KB Payload
Semantic Cards Pilot
Полотно LLM
```

```text

```

Ожидание: объяснение без упражнения, без скрытой домашки, компактнее старых ответов.

Открыть trace:

```text
Latest Turn Constraints
Final Answer Directive
Practice Note
Human-Like Answer Policy
Runtime Truth Trace
Полотно LLM
```

#### F. No internal DB

```text
Ответь своими словами, без внутренней БД: что мне делать, если я злюсь, но не могу прямо спорить с начальником?
```

Ожидание: Writer payload `0`, no internal DB/source references, живой ответ.

Открыть trace:

```text

Runtime Truth Trace
Writer KB Payload
Полотно LLM
```

#### G. Direct KB/source request


Ожидание: direct KB/source path работает, Writer payload `1+`, но без raw dump.


Ожидание: человеческий короткий ответ, без анализа механизмов, без практики, без БД.

