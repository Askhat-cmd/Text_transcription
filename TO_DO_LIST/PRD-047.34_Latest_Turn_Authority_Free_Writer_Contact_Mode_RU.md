**Проект:** Bot Psychologist / Neo MindBot

## 0. Краткое описание простым языком

После PRD-047.30–047.33 бот стал заметно лучше: БД больше не командует обычным разговором, явный запрос практики получает узкую полезную опору, trace стал честнее, а ответы стали короче и живее.

Но последний owner-chat показал важный архитектурный сбой: когда пользователь после вопроса по внутренней БД сказал:

```text
Мне сейчас просто тяжело. Не хочу разбирать, просто скажи что-нибудь по-человечески.
```

бот продолжил старый KB-разбор про программу «несовершенное Я» и пять драйверов. Это значит, что старый `must_answer` снова оказался сильнее последней живой реплики пользователя.

Этот PRD должен исправить не частную фразу, а общий принцип:

```text
Writer отвечает на последнее сообщение пользователя.
Вся предыдущая история, БД, карточки, маршруты, open loops и старый must_answer — только смысловая база для ответа на последний ход.
```

Ожидаемый результат: если пользователь меняет способ общения, устал, просит без анализа, просит проще, просит поддержку, отказывается от текущего формата или просто делает новый живой поворот — Writer отвечает именно на это. Предыдущая тема остаётся фоном, но перестаёт быть задачей.

## 1. Контекст

### 1.1. Что уже работает

PRD-047.30:

```text
KB / semantic cards / diagnostic hints стали optional grounding, а не управляющими блоками.
```

PRD-047.31-HF1:

```text
явный запрос практики перебивает stale must_answer и получает узкое practice-grounding.
```

PRD-047.32:

```text
runtime truth trace показывает, что найдено, что filtered/trace-only, и что реально ушло Writer’у.
```

PRD-047.33:

```text
answer-shape calibration улучшил краткость и живость ответов, сохранив explicit practice, no-practice, no-internal-db, direct KB/source и runtime truth boundaries.
```

### 1.2. Что ещё сломано

В живом чате после PRD-047.33 пользователь сказал, что ему тяжело и он не хочет разбирать. Система выставила:

```text
dialogue_act = unknown
answer_obligation = continue_thread
must_answer = предыдущий KB-вопрос про программу «несовершенное Я» и пять драйверов
answer_shape = structured_explanation
```

Это неверно. Когда последняя реплика явно меняет режим контакта, старый `must_answer` должен быть демотирован в контекст.

### 1.3. Главная архитектурная поправка

Не добавлять очередной список маркеров. Не создавать новый LLM-agent. Не строить новый route.

Нужно изменить принцип власти внутри текущего pipeline:

```text
latest user turn = primary task
previous conversation = context
KB / semantic cards = optional meaning base
old must_answer = only previous open loop, not a command
Writer = final author of the human response
```

## 2. Цель PRD

Сделать так, чтобы текущий canonical pipeline всегда начинал с последнего сообщения пользователя.

Минимальные цели:

```text
1. Последняя непустая реплика пользователя по умолчанию становится current_user_request / must_answer.
2. Старый must_answer не может командовать ответом, если latest turn меняет тему, режим, тон, формат или эмоциональную потребность.
3. Dialogue act `unknown` больше не должен автоматически вести к continue_thread со старым must_answer.
4. Writer получает более свободную сцену: “ответь последнему человеку, а прошлое используй как фон”.
5. Existing hard boundaries сохраняются: safety, no_internal_db, no_practice, explicit_practice, direct KB/source, privacy/no raw KB dump.
6. Не добавлять новую подсистему и не расширять marker/rule-engine подход.
```

## 4. Главный принцип реализации

### 4.1. Последний ход — задача, история — контекст

Нужно привести runtime к простой формуле:

```text
answer_target = latest_user_message
conversation_history = context_for_interpreting_latest_user_message
retrieved_knowledge = optional meaning base for latest_user_message
previous_must_answer = previous open loop, not current command
```

### 4.2. Старый must_answer может сохраниться только при явном продолжении

Старый `must_answer` может остаться активным только если последняя реплика явно продолжает прошлую задачу.

Примеры явного продолжения:

```text
да, продолжи
расскажи подробнее про это
объясни второй пункт
что насчёт пяти драйверов подробнее
приведи пример к прошлому
```

Но если последняя реплика меняет режим контакта, отказывается от анализа, просит поддержки, просит проще, просит без БД, просит без практики или задаёт новый вопрос — старый `must_answer` должен стать контекстом.

Важно: реализация не должна превращаться в длинный список фраз. Агент должен выразить это как общий принцип latest-turn authority в существующих слоях.

### 4.3. Writer должен сам выбирать живой ответ в мягкой зоне

Жёстко остаются только настоящие границы:

```text
safety/crisis
no_internal_db
no_practice
explicit_practice_request
direct KB/source request
privacy/no raw KB dump
```

Всё остальное должно быть мягкой сценой для Writer’а, а не клеткой.

## 9. Minimal Runtime Repair

### 9.1. Current user request должен быть явным

В существующем pipeline добавить/уточнить компактное поле в existing trace/directive structures:

```text
current_user_request = latest non-empty user message
must_answer_source = latest_turn | explicit_continue_previous | safety | direct_source | explicit_practice
previous_must_answer_demoted = true/false
previous_must_answer = <short sanitized text or none>
```

Это не новая подсистема. Это clarification внутри existing directive/trace.

### 9.2. Default rule: latest turn wins

В `final_answer_directive_v1` и/или `unanswered_question_tracker`:

```text
By default, must_answer = latest_user_message.
```

Исключение:

```text
latest_user_message explicitly asks to continue the previous answer/open loop.
```

Если latest turn не является явным продолжением, previous must_answer должен быть демотирован в context.

### 9.3. Unknown dialogue act не должен продолжать старое задание

Если `dialogue_act=unknown`, но latest user message содержит нормальную человеческую реплику, не ставить автоматически:

```text
answer_obligation=continue_thread
must_answer=previous question
answer_shape=structured_explanation
```

Вместо этого:

```text
answer_obligation=answer_latest_turn
must_answer=latest_user_message
answer_shape_profile=free_writer_contact / writer_choose_soft
previous_must_answer_demoted=true
```

### 9.4. Free Writer Contact Mode

Добавить мягкий режим в existing answer-shape/profile/directive logic:

```text
free_writer_contact
```

Смысл:

```text
User’s latest turn is not asking for a technical/KB/practice explanation; it changes the desired human contact mode. Answer the latest turn directly. Use prior context only to understand what this is about. Do not continue the previous internal task.
```

Prompt wording для Writer:

```text
Отвечай на последнее сообщение пользователя. Всё, что было выше, — только контекст, а не задание. Если последняя реплика меняет способ общения, отказывается от разбора или просит человеческий отклик, не продолжай предыдущую тему: ответь живо и прямо на текущую реплику.
```

### 9.5. Старые open loops не удалять, а снижать власть

Не надо ломать open loop system. Нужно изменить её роль:

```text
old open loop = context hint
latest user turn = answer target
```

Если нужно, old loop может быть visible in trace as:

```text
previous_open_loop_available=true
previous_open_loop_role=context_only
```

### 9.6. Сохранить hard boundaries

Нельзя сломать:

```text
no_internal_db → Writer payload 0
no_practice → no practice/exercise/homework
explicit_practice_request → one bounded contextual practice
direct KB/source → compact grounded answer with Writer payload allowed
safety/crisis → safety first
privacy/no raw KB dump
```
