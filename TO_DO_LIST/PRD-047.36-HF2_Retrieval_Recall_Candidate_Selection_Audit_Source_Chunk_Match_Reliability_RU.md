# PRD-047.36-HF2 — Retrieval Recall & Candidate Selection Audit / Source Chunk Match Reliability v1

**Проект:** Bot Psychologist / Neo MindBot  
**Репозиторий:** `Askhat-cmd/Text_transcription`  
**Локальный путь:** `C:\My_practice\Text_transcription`  
**Дата:** 2026-06-30  
**Статус:** ACCEPTED WITH WARNING
**Тип:** retrieval reliability / candidate-selection audit / hidden-competence hardening PRD after PRD-047.35  
**Главный курс:** не добавлять словари, rule-engine или новый runtime path; доказать, где именно теряется релевантный source chunk: raw DB query → runtime candidates → hybrid/ranking/gating → Writer KB Payload.

---

## 0. Краткое описание простым языком

После PRD-047.35 бот стал лучше скрывать внутреннюю БД от обычного пользователя: он больше не должен говорить про чанки, semantic cards, внутренние источники и trace. Это правильно.

Но появился следующий, более важный вопрос:

```text
Если в БД уже есть нужное знание, бот точно умеет его найти и дать Writer'у?
```

Последний live-чат владельца показывает проблему. На вопрос:

```text
Что такое анестетическая депрессия?
```

бот сначала ответил в медицинском смысле: про действие анестетиков, подавление ЦНС, дыхание и сердечную активность.

Затем на уточнение:

```text
анестетическая депрессия это ведь еще и психологический термин, что ты знаешь об этом?
```

бот дал общий психологический ответ про эмоциональное онемение.

И только после прямой поправки:

```text
Так стоп! Что такое анестетическая депрессия в Нейросталкинге?
```

система перешла в более подходящий нейросталкинговый фрейм и в trace стало видно `Writer-visible 2`.

Это нельзя чинить добавлением ручного словаря:

```text
анестетическая депрессия → нейросталкинг
```

Такой путь быстро вернёт проект к rule-engine и бесконечным частным правкам.

**Цель PRD-047.36-HF2:** сделать диагностически видимым и надёжным путь relevant source chunk до Writer. Если источник содержит прямое или близкое совпадение, trace должен показать, был ли этот source chunk найден, как ранжирован, почему отфильтрован или почему отправлен в Writer KB Payload.

## 1. Почему этот PRD нужен именно сейчас

PRD-047.35 успешно сдвинул продукт к hidden knowledge competence:

```text
БД не публичная тема.
БД — скрытая компетенция Writer.
```

Но скрытая компетенция должна быть не только скрытой, но и рабочей.

Сейчас есть риск:

```text
source material содержит нужную основу ответа,
но retrieval/candidate selection не доставляет её Writer'у,
а Writer отвечает из общей модели или по неподходящему фрейму.
```

В live-следе Chat 8 виден типичный сбой:

```text
user: Что такое анестетическая депрессия?
retrieval_action: query_kb
Runtime Truth Trace: Writer-visible 1 | filtered 3
payload_ids: semantic_card:program_imperfect_self_v1
memory_semantic_hit chunks: filtered out with filter_reason=direct_knowledge_question
final answer: medical/anesthesia frame
```

Затем после явного указания `в Нейросталкинге`:

```text
Runtime Truth Trace: Writer-visible 2 | filtered 3
payload_ids: 99c31e75-b401-40df-96e5-c82c5cb11569, d527d369-beb2-4113-b370-cb2738e76b55
final answer: closer to internal psychological frame
```

Это показывает не только answer-quality проблему. Это показывает missing proof проблему:

```text
мы не знаем точно, где потерялся правильный source chunk:
- raw search его не нашёл?
- Memory Agent нашёл, но низко ранжировал?
- hybrid merge вытеснил?
- gate отфильтровал?
- Writer KB Payload получил не тот chunk?
- Writer получил, но проигнорировал?
```

Этот PRD должен дать ответ на эти вопросы.

## 2. Архитектурный принцип PRD-047.36-HF2

Главный принцип:

```text
Не усиливать Writer догадками.
Сначала доказать путь source chunk.
```

Нужно проверить четыре уровня:

```text
1. Raw source / BotDB query layer
   Что реально есть в базе и что находит прямой запрос?

2. Runtime retrieval candidate layer
   Какие кандидаты получает MemoryRetrievalAgent / current_turn_focus_v1 / hybrid path?

3. Candidate selection / gating layer
   Какие кандидаты были сохранены, отфильтрованы, по какой причине и с какими scores?

4. Writer KB Payload layer
   Что фактически дошло до Writer, с каким chunk_id/source_doc/chunk_type/quote_policy?
```

Trace должен перестать отвечать только на вопрос:

```text
сколько chunks было visible?
```

И начать отвечать на вопрос:

```text
нашёлся ли ожидаемый source chunk и где он потерялся?
```

## 8. Minimal repair logic after audit

После read-only audit агент должен создать:

```text
TO_DO_LIST/logs/PRD-047.36-HF2/fix_decision.md
```

В нём указать одну из веток:

### Branch A — source chunk есть в raw search, но теряется до Writer

Разрешён узкий generic fix:

```text
Для explicit knowledge question сохранять exact / near-exact source candidate как candidate-preservation item до gate/payload,
если он проходит existing governance policy.
```

Запрещено делать fix по конкретному слову.

### Branch B — source chunk есть в runtime candidates, но gate фильтрует без ясной причины

Разрешён fix в candidate inclusion / gate reason:

```text
- сделать filter_reason точным;
- не фильтровать лучший direct source match только потому, что он general_text,
  если вопрос является direct knowledge question и allowed_use разрешает direct_to_writer.
```

При этом нельзя обходить:

```text
- safety;
- do_not_use / excluded;
- internal_only;
- not_for_direct_quote;
- no_internal_db;
- no_raw_kb_dump.
```

### Branch C — payload получает chunk, но Writer отвечает из внешней общей модели

Разрешён мягкий Writer grounding repair:

```text
Для direct knowledge question с payload_match=true Writer должен начинать с grounded internal meaning,
но не раскрывать БД/чанки.
```

Запрещено:

```text
- заставлять Writer цитировать raw source;
- писать "в Нейросталкинге говорится" без уверенного grounding;
- добавлять public source dump.
```

### Branch D — raw source не содержит прямого/близкого материала

Не делать runtime fix.

Нужно честно зафиксировать:

```text
source_match_missing_or_unproven
```

И рекомендовать отдельный будущий DB/source preparation PRD, но не выполнять его здесь.

### Branch E — audit inconclusive

Не делать behavior fix.

Сначала расширить read-only trace/report until loss_stage becomes known.


```text

- можно сказать человечески: "я бы понимал это так..." или "если говорить в этой логике...";
- не говорить обычному пользователю про БД, чанки, trace или отсутствие chunk;
- не просить пользователя заново объяснить известный контекст, если retrieval/trace уже показывает active_concept.
```

Owner/debug режим может видеть proof в trace, но user-facing ответ остаётся чистым.
