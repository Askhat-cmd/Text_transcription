## 0. Кратко простыми словами

PRD-047.36-HF1 закрыл главный класс drift:

```text
visible bubble больше не должен раскрывать trace/canvas другого turn.
```

Но owner live file `ЧАТ_С_БОТОМ_11.txt` показывает новый остаточный класс:

```text
trace/canvas теперь не всегда подменяется чужим,
но у части сообщений он вообще не появляется после reload.
```

Это не Writer-quality issue. Это owner/debug observability issue.

Нужна гарантия:

```text
если assistant answer был delivered и сохранён как normal chat turn,
то Web UI в dev/owner mode должен уметь показать trace/canvas для этого же turn
или явно показать owner/debug причину, почему trace unavailable.
```

## 1. Current evidence

Owner after HF1 reports:

```text
После перезагрузки веб страницы trace появляется только у некоторых сообщений.
У некоторых сообщений trace так и не проявляется.
```

В `ЧАТ_С_БОТОМ_11.txt` виден mixed state:

```text
- некоторые bot answers имеют Pipeline NEO block;
- некоторые bot answers визуально идут без Pipeline/trace block;
- один expanded trace показывает memory_written assistant empty even when visible answer exists;
- trace for turn 5 exists in canvas, but saved-memory preview shows assistant empty;
- this means HF1 fixed turn identity binding, but not trace availability/completeness for every delivered turn.
```

Important owner example:

```text
Visible turn:
user: да, например один из смыслов это моя семья, то что я делаю для них
assistant visible answer exists
trace/canvas turn: 5 exists
but memory_written preview contains:
assistant:
<empty>
```

Also code audit shows a likely residual source:

```text
useMultiAgentTrace now asks backend by explicit turnNumber.
If backend has no exact trace for that turn, debug route can fall back to latest trace.
Frontend then suppresses mismatched trace instead of rendering wrong canvas.
This prevents wrong trace display, but can produce silent missing trace cards.
```

HF3 must prove whether missing trace is caused by:

```text
A. trace was never persisted for that delivered turn;
B. trace was persisted under another session/user/conversation key;
C. history hydration has turnNumber but no trace cache key;
D. debug endpoint exact lookup returns latest fallback/mismatch;
E. frontend suppresses mismatch without visible owner/debug status;
F. streaming done/trace event ordering drops trace payload;
G. saved memory write happened before final assistant answer was available.
```

## 4. Required investigation

### 4.1. Backend trace store audit

Inspect how multiagent debug traces are stored and keyed:

```text
session_id
conversation_id
runtime_user_scope
request_user_id
user_id
turn_index / turn_number
```

For each delivered turn in a live session, prove:

```text
- was trace stored? yes/no
- under which key?
- with which turn_index?
- with which user_input?
- with which final answer hash?
- does it map to saved history turn_number?
```

### 4.2. Debug endpoint exact lookup

Current endpoint behavior includes fallback-to-latest when requested turn is missing. HF3 must make this owner/debug-safe:

```text
- exact turn lookup should not silently fall back to latest when turn_index is explicitly requested;
- if fallback is kept for legacy endpoint, response must expose exact_match=false and fallback_reason;
- frontend must not hide this silently.
```

Preferred target:

```text
GET /api/debug/session/{session_id}/multiagent-trace?turn_index=N
returns exact trace for N or structured 404 with available_turn_indices.
```

### 4.3. Frontend trace availability state

In owner/dev mode, if trace is missing for a bot message, UI should show a small owner/debug status instead of no trace block:

```text
Trace unavailable for this turn
reason: not_found | mismatched_turn | legacy_turn | no_debug_trace_saved | auth/dev_key_missing
turn_number: N
session_id: ...
available_turn_indices: [...]
```

This must be owner/debug only. Do not show debug noise in normal public mode.

### 4.4. SSE trace and done event order

Audit whether trace can be lost when:

```text
- trace event arrives after done event;
- done payload contains answer but no trace;
- EOF flush happens without trailing blank line;
- retry path duplicates answer but loses trace;
- frontend finalizes bot message before tracePayload is available.
```

Acceptance target:

```text
final bot message stores trace if any trace was delivered by SSE,
and backend trace endpoint can still recover trace after reload if inline trace was absent.
```

### 4.5. Memory write timing

Investigate `memory_written` preview and saved history timing.

If trace shows:

```text
memory_written.assistant is empty
```

while visible answer exists, classify why:

```text
- expected debug preview timing issue;
- real saved-history gap;
- acceptance gate pending/quarantine remnant;
- final answer written after trace snapshot and not reflected in trace.
```

Target:

```text
Owner trace should distinguish final saved assistant answer from early memory-write preview.
```

## 8. Acceptance Gate

ACCEPTED only if:

```text
- every delivered assistant turn in the Chat 11 replay has either exact trace available or explicit owner/debug unavailable reason;
- frontend no longer silently hides trace absence in owner/dev mode;
- explicit turn lookup no longer returns latest trace as if it were exact;
- reload history can request trace for every assistant message by stable turnNumber;
- no mismatched canvas is rendered;
- saved history answer and trace answer are either hash-aligned or discrepancy is explicitly explained;
- no Writer/retrieval/DB/source behavior changed;
- targeted backend/frontend tests pass;
- reports complete;
- docs synced;
- git clean and pushed.
```

ACCEPTED_WITH_WARNING only if:

```text
- some legacy historical turns legitimately have no trace, but UI labels them as legacy/no_debug_trace_saved;
- browser-level visual capture is unavailable, but API + component tests prove behavior;
- full pytest still fails only on known unrelated `_build_llm_prompts` import error.
```

BLOCKED if:

```text
- new delivered turns still silently miss trace cards;
- exact turn lookup can still silently serve wrong/latest turn;
- trace/canvas can still render under the wrong message;
- reload loses trace availability for fresh turns;
- memory_written assistant empty corresponds to real saved-history answer loss;
- fix changes retrieval or DB/source behavior.
```
