## 1. Почему HF3 был сдан неправильно

По отчёту HF3 live smoke проверил слишком узкий случай:

```text
backend health 200
frontend 200
one adaptive test turn 200
exact trace lookup turn_index=1 -> 200
missing turn_index=999 -> 404 requested_turn_missing
```

Это не доказывает, что trace работает в реальном Web Chat.

Недостаточные проверки:

```text
- не доказан многоходовый новый чат в браузере;
- не доказано, что каждый delivered assistant turn имеет trace;
- не доказан reload этого же нового чата;
- не доказан старый чат после reload;
- не доказано, что UI показывает реальные trace/canvas, а не только unavailable-state;
- не доказано, что available_turn_indices покрывает все delivered turns;
- не доказано, что requested_turn_missing отсутствует для свежих delivered turns.
```

В HF4 запрещено сдавать работу на основании только backend/API smoke.

## 2. Главная цель HF4

Восстановить цепочку:

```text
fresh Web Chat user turn
→ assistant answer delivered
→ backend saves exact multiagent debug trace for this turn
→ history stores same turn_number
→ frontend requests exact trace by that turn_number
→ Web UI renders actual Pipeline NEO / LLM canvas for that assistant message
→ after browser reload the same trace is still available
```

Owner/debug unavailable-state остаётся полезным только для legacy/no-debug cases.

Для новых delivered turns `Trace unavailable` не является успехом.

## 6. Required root-cause audit

### 6.1. Trace persistence coverage

For a fresh web session of 5 turns, prove:

```text
delivered_turn_numbers = [1,2,3,4,5]
saved_history_turn_numbers = [1,2,3,4,5]
stored_debug_turn_indices = [1,2,3,4,5]
available_turn_indices = [1,2,3,4,5]
```

If stored_debug_turn_indices only contains `[1]` or `[1,3]`, HF4 must find why.

Inspect:

```text
- where save_multiagent_debug() is called;
- whether it is called for every adaptive-stream request;
- which id is used: web session id, chat session id, user_id, conversation_id, runtime_user_scope;
- whether turn_index increments correctly across frontend/backend/session history;
- whether certain answer types skip debug save;
- whether acceptance/quarantine path skips debug save;
- whether restart clears in-memory traces for old sessions and how UI labels that.
```

### 6.2. Session key mapping

Prove for every turn:

```text
frontend activeChatId
request body session_id
X-Session-Id header
backend conversation/session id
multiagent debug store key
history API session id
trace endpoint requested session id
```

HF3 narrowed candidate lookup with `include_all_keys=False`. If this removed the actual key where traces are stored, repair the mapping without reintroducing wrong cross-session trace drift.

### 6.3. Turn number mapping

Prove there is no off-by-one or regenerated numbering:

```text
frontend user message turnNumber
frontend bot message turnNumber
SSE done turn_number
history turn_number
trace debug turn_index
trace endpoint requested turn_index
trace response turn_index
```

### 6.4. Old sessions after backend restart

If old traces are in-memory and legitimately gone after backend restart, UI may show unavailable, but reason must be precise:

```text
reason_code = debug_trace_expired_after_backend_restart | legacy_no_debug_trace_saved
```

For new chats after restart, this excuse is invalid.
