## 0. Простыми словами

Post-HF readiness gate показал важную вещь:

```text
бот уже в целом отвечает правильно,
но trace не всегда доказывает, что важные границы были распознаны и применены.
```

Два blocker-кейса:

```text
G5: пользователь просит без внутренней БД / без Нейросталкинга.
Видимое поведение нормальное, Writer Payload = 0,
но boundary_flags не показывает no_internal_db.

G6: пользователь просит без практик и упражнений.
Видимое поведение нормальное,
но boundary_flags не показывает no_practice.
```

Это не задача сделать бота умнее.  
Это задача сделать runtime trace честным и доказательным:

```text
граница распознана
→ граница применена
→ Writer получил корректное ограничение
→ payload/practice были подавлены именно по этой причине
→ owner/debug trace это явно показывает.
```

Пока это не доказано, readiness gate не может быть принят.

## 2. Important note about old chats after backend restart

Owner screenshots show:

```text
fresh/new Web Chat: trace works correctly.
old chat after backend restart: Trace unavailable with reason debug_trace_expired_after_backend_restart.
```

Decision for HF6:

```text
Do not turn HF6 into persistent historical trace storage repair.
```

Reason:

```text
Old debug traces can be in-memory runtime artifacts.
After backend restart, they may legitimately expire.
HF4 already established this as an accepted warning if UI labels it honestly.
```

HF6 must preserve and clarify this behavior:

```text
- old pre-restart chats may show debug_trace_expired_after_backend_restart;
- this is not a blocker if reason is explicit and not confused with fresh-turn failure;
- fresh turns after restart must still have real trace;
- boundary flags must be visible for new G5/G6 test turns.
```

If IDE-agent discovers that old traces are actually persistable with a tiny safe fix, it may report a recommendation, but must not bundle persistent trace storage into HF6.

If owner later wants old-chat trace persistence across backend restarts, create separate PRD:

```text
PRD-047.36-HF7 — Persistent Debug Trace Store / Historical Trace Recovery v1
```

Do not implement HF7 inside HF6.

## 10. Acceptance Gate

HF6 is ACCEPTED only if:

```text
- G5 no_internal_db visible behavior remains correct;
- G5 trace explicitly includes no_internal_db boundary flag;
- G5 Writer Payload = 0 and semantic cards not writer-visible;
- G6 no_practice visible behavior remains correct;
- G6 trace explicitly includes no_practice boundary flag;
- G6 answer contains no practice/homework/step routine;
- combined no_internal_db + no_practice sets both flags;
- HF5 direct concept follow-up still works when no boundary applies;
- fresh trace works after backend restart;
- old chat post-restart trace expiry is labelled clearly and not confused with fresh failure;
- no Writer/retrieval/DB/Chroma/source/semantic-card-authority mutation outside scope;
- targeted and preservation tests pass or unrelated historical blocker is recorded;
- reports complete;
- docs synced;
- git status clean and pushed.
```

ACCEPTED_WITH_WARNING only if:

```text
- old pre-HF6 chats still cannot recover in-memory debug traces after backend restart,
  but they show explicit debug_trace_expired_after_backend_restart and fresh chats are clean;
- greeting/contact wording remains slightly therapeutic;
- full suite fails only on known unrelated historical import blocker.
```

BLOCKED if:

```text
- no_internal_db boundary flag is still missing;
- no_practice boundary flag is still missing;
- Writer Payload > 0 under explicit no_internal_db;
- semantic cards become writer-visible under explicit no_internal_db;
- answer still gives practice/homework/step routine under explicit no_practice;
- fresh turns after backend restart show Trace unavailable;
- HF5 concept follow-up regresses to Writer Payload 0;
- public answer exposes DB/chunks/cards/trace;
- implementation changes retrieval ranking, DB, Chroma, source docs, or adds dictionary/alias/new agent/new runtime path;
- reports are incomplete.
```
