## 0. Краткое описание простым языком

PRD-047.36-HF2 показал, что качество ответа заметно выросло: бот стал лучше использовать БД как скрытую компетенцию, а не как публичный справочник. На кейсе `анестетическая депрессия` бот уже не ушёл в медицинскую трактовку и дал психологический ответ через внутреннюю пустоту, эмоциональное онемение, исчерпание старого дефицитарного топлива и переходное состояние.

Теперь не нужно снова сразу чинить runtime.

Нужно сделать gate:

```text
готов ли бот к owner pilot freeze на ограниченном наборе сценариев?
```

Этот PRD не должен улучшать всё подряд. Его задача — проверить текущую систему как продукт:

```text
- держит ли latest-turn authority;
- не светит ли БД/чанки/trace обычному пользователю;
- работает ли retrieval proof;
- не обрезается ли финальный ответ;
- не теряется ли ключевой смысл при payload truncation;
- не путает ли trace владельца;
- сохраняется ли safety;
- остаётся ли Writer живым финальным автором.
```

Главная формула:

```text
Не улучшаем бесконечно.
Сначала проверяем готовность.
```

Если gate находит blocker — готовим один узкий HF.  
Если blocker нет — фиксируем pilot-ready baseline и переносим мелкие warnings в backlog.

## 3. Product invariants to freeze

Gate должен проверить и зафиксировать следующие инварианты.

### 3.1. Latest Turn Authority

```text
Последняя реплика пользователя — главный target ответа.
Прошлое — только контекст, если пользователь явно не просит продолжить старую тему.
```

### 3.2. Hidden Knowledge Competence

```text
БД / semantic cards / chunks / source proof / trace не являются темой обычного ответа.
Они могут быть видны владельцу в trace, но не пользователю.
```

Запрещены в обычном user-facing answer:

```text
- в базе;
- по чанкам;
- semantic card;
- source proof;
- trace;
- в загруженных материалах;
- внутренняя БД;
- Runtime Truth Trace;
- Writer KB Payload.
```

### 3.3. Writer Final Authorship

```text
Writer остаётся финальным автором.
Retrieval, semantic cards, Diagnostic Center, Planner, Trace — advisory/support only.
```

### 3.4. Retrieval Proof Reliability

```text
Если source chunk найден, trace показывает raw/runtime/payload path.
Если source chunk не найден, это честно классифицируется как source-missing/top-k/source-preparation issue.
```

### 3.5. Delivery Integrity

```text
Raw Writer final answer = API answer = visible chat bubble = saved memory assistant message
по смыслу, длине и ключевым абзацам.
```

### 3.6. Payload Excerpt Integrity

```text
Если chunk_truncated=true, ключевой matched span / определение / source span должен попадать в content_excerpt, который реально дошёл Writer'у.
```

### 3.7. Safety Floor

```text
Safety сильнее глубины, БД, Wake-like depth и красивого ответа.
```

## 7. Delivery & Payload Integrity Mini-Gate

This is mandatory inside PRD-047.36.

### 7.1. Payload Excerpt Integrity Check

For every scenario with Writer KB Payload:

If any payload item has:

```text
chunk_truncated=true
TRUNCATED_CHUNKS > 0
content_truncated=true
```

then verify:

```text
- matched phrase / focus terms are inside content_excerpt;
- key source span / definition is inside content_excerpt;
- original_char_count is recorded;
- sent_char_count is recorded;
- content_truncated is recorded;
- truncation_reason is recorded;
- matched_span_in_excerpt=true/false is recorded;
- key_definition_in_excerpt=true/false is recorded if detectable;
- if matched span is cut off, scenario is BLOCKER_payload_excerpt_lost_match.
```

Required trace/report fields:

```json
{
  "payload_excerpt_integrity_v1": {
    "enabled": true,
    "payload_id": "...",
    "chunk_id": "...",
    "original_char_count": 0,
    "sent_char_count": 0,
    "content_truncated": false,
    "truncation_reason": "none | char_budget | sentence_boundary | unknown",
    "matched_terms_or_phrase": ["..."],
    "matched_span_in_excerpt": true,
    "key_definition_in_excerpt": "true | false | unknown",
    "blocker": false,
    "blocker_reason": "..."
  }
}
```

This may be implemented as a read-only report over existing payload data. Do not mutate chunks.

### 7.2. Final Answer Delivery Spot Check

On at least 3 long or medium answers, compare:

```text
A. raw Writer final answer
B. API response final answer
C. visible chat bubble content if accessible through live/web trace/manual capture
D. saved memory assistant message
```

Acceptance:

```text
- same key paragraphs;
- no preview-only save;
- no missing tail;
- no UI-only truncation;
- no saved-memory truncation that changes future context.
```

If mismatch is detected:

```text
BLOCKER_delivery_integrity
```

Then next PRD must be:

```text
PRD-047.36-HF1 — Final Answer Delivery Integrity / Chat Bubble Truncation Repair
```

Do not fix delivery in this gate unless the issue is a trivial report/capture bug.

### 7.3. Trace Consistency Check

For source proof scenarios, check:

```text
- if payload_match.near_exact_match=true and sent_to_writer=true, loss_stage must not look like a raw_source failure without explanation;
- if loss_stage remains raw_source due to taxonomy nuance, loss_reason must clearly explain the distinction;
- owner trace must distinguish:
  - raw source search miss;
  - runtime top-k miss;
  - payload included despite raw/source mismatch;
  - payload included with near-exact runtime match;
  - source absent vs source found but truncated.
```

If trace is confusing but behavior is correct:

```text
WARNING_trace_consistency_cleanup_needed
```

If trace hides a real payload/source loss:

```text
BLOCKER_trace_truth_gap
```
