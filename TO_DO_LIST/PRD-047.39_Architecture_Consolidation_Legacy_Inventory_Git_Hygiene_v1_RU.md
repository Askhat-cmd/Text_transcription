## 1. Почему это нельзя сделать за одну итерацию

`TO_DO_LIST/logs/PRD-047.37/cleanup_retirement_candidates.md` уже заканчивается разделом **Retirement Discipline**:

```text
"Do not delete major runtime code in a freeze PRD.
A future cleanup PRD must prove that a candidate is inactive or safely replaceable,
include rollback steps, and preserve trace/debug visibility where it affects owner evidence."
```

Исторический паттерн проекта тоже всегда разделяет audit и repair:

```text
PRD-047.11-AUDIT -> PRD-047.11-HF1
PRD-047.14 (audit-only) -> PRD-047.14-HF1
PRD-046.0.4 (audit) -> PRD-046.0.5 (apply)
PRD-047.36 (gate, BLOCKER) -> PRD-047.36-HF1..HF6
```

Причина — атрибуция причин при регрессии:

```text
Если классификация и удаление происходят в одном PRD и что-то ломается,
невозможно понять: ошиблись ли в классификации (код не был мёртв)
или удаление задело нелинейную зависимость
(динамический импорт, admin UI роут, string-based dispatch).
```

Дополнительно: часть находок лежит в разных слоях системы одновременно — git-инфраструктура, runtime config (~90 флагов), writer-контракт, admin UI. Правка всех слоёв за один PRD означала бы проверку live smoke по 4-5 независимым осям сразу — тот же паттерн, который в PRD-047.36 потребовал шесть хотфиксов подряд.

**Вывод: минимум 2 стадии, реалистично 4-5 узких PRD. Этот документ — Стадия 1.**

### 4.5. Logs classification manifest + forward-looking policy

Перед любым untrack-действием над `TO_DO_LIST/logs` runner обязан построить манифест:

```text
TO_DO_LIST/logs/PRD-XXX/*.md                  -> tier: evidence_of_record (keep)
TO_DO_LIST/logs/PRD-XXX/*_manifest.md         -> tier: evidence_of_record (keep)
TO_DO_LIST/logs/PRD-XXX/live_turn_exports/*   -> tier: raw_artifact (untrack candidate)
TO_DO_LIST/logs/PRD-XXX/prompt_canvases/*     -> tier: raw_artifact (untrack candidate)
TO_DO_LIST/logs/PRD-XXX/*_triage_report.json  -> tier: raw_artifact if >100KB
TO_DO_LIST/logs/PRD-XXX/*.png                 -> tier: raw_artifact (untrack candidate)
```

Правило по умолчанию: **если файл referenced по имени (не по папке целиком) в PROJECT_STATE.md/PRD_INDEX.md/DECISIONS.md — он остаётся tracked независимо от размера и типа.**

Манифест — это только отчёт. Owner утверждает untrack-список перед выполнением, ровно как с ветками.

**Forward-looking policy (чтобы бардак не вырос заново за следующие 40 PRD):** начиная с PRD-047.40, новые raw-артефакты (`live_turn_exports/`, `prompt_canvases/`, полные json-дампы) кладутся в `TO_DO_LIST/logs/PRD-XXX/raw/` — эта подпапка сразу в `.gitignore` по паттерну. `*.md` отчёты и лёгкие summary/status json (<50KB) остаются в корне `PRD-XXX/` и трекаются как обычно. Разделение становится соглашением об именовании на будущее, а не разовой уборкой, которую придётся повторять.

## 12. Simple explanation

Это не PRD «почистить проект». Это PRD «составить точную карту того, что можно чистить, и убрать руками только то, что не является кодом — ветки, backups, сырые (не-отчётные) части логов, один мёртвый тест». Отчёты, на которые ссылается PROJECT_STATE/PRD_INDEX, не трогаем — они и есть context continuity для Architect-сессий. Всё, что реально меняет поведение бота (удаление старых pipeline, флагов, разбиение файлов) — это уже следующие 4 отдельных PRD, каждый со своим гейтом. S7-warning из PRD-047.38 — тоже не сюда, это backlog. Именно так этот проект уже показал, что работает надёжно: audit, потом repair, не одновременно.
