## 0. Короткое решение

```text
WRITER_USER_TEMPLATE.format(...) — один Python-вызов на ~170 именованных
аргументов, не последовательность операторов. Резать сам вызов нельзя —
он должен остаться одним. Поэтому режем ПО-ДРУГОМУ, чем slice1/2/3:
вычисление части именованных аргументов выносим в датакласс через
helper-функцию, а сам вызов format() остаётся один и тот же, просто
часть его kwargs теперь ссылается на поля датакласса вместо инлайн-
вычисления ctx.get(...).

Этот PRD берёт первые 5 смысловых семей (39 аргументов из ~170):
unified_dialogue_policy, dialogue_act_and_offer_state,
dialogue_style_and_answer_obligation, diagnostic_card_and_writer_move,
context_budget_and_profile (без conversation_context — тот остаётся
прямым проходом formatted_context, без изменений).

Границы в текущем файле: строки 303-342 (40 строк, из них 1 строка —
conversation_context=formatted_context на строке 336 — НЕ трогаем).
```

## 1. Почему в одном PRD сразу 5 семей, а не одна

Внутри giant-method-исключения (`_call_llm` формирует финальный текст
для LLM, сам _call_llm режем не одним PRD) — этот конкретный набор из
5 семей состоит ИСКЛЮЧИТЕЛЬНО из однотипных `str(ctx.get(key, default)
or default)` — чтения с дефолтом, без ветвления и без принятия решений
(в отличие от request_detectors_and_mvp_override_block в APPLY-9, где
была реальная условная логика). Поведенческий риск такой строки не
растёт от соседства с другими такими же строками — поэтому 5 семей
объединены в один PRD вместо пяти отдельных. Единственное исключение
внутри этого набора — diagnostic_card_summary (статический вызов
_format_diagnostic_summary), тоже детерминированный, без ветвления.

## 2. Проверенная архитектором зависимость входов

```text
Помимо ctx, helper-функции нужны ещё два уже вычисленных значения:

  dialogue_profile -> используется как ДЕФОЛТ на строке 304
    (unified_active_profile_alias=str(ctx.get(..., dialogue_profile)))
    — это slice1_inputs.dialogue_profile, уже доступен в _call_llm
    к моменту этого блока.

  context_meta -> используется в семье context_budget_and_profile
    (context_budget_chars, context_truncated,
    preserved_recent_turns_count, older_context_omitted_chars берутся
    из context_meta, не из ctx) — это slice1_inputs.context_meta,
    тоже уже доступен.

diagnostic_card_summary использует статическую функцию
_format_diagnostic_summary. В классе WriterAgent это @staticmethod,
который сам всего лишь делегирует в модульную функцию
_format_diagnostic_summary из writer_agent_fallback_helpers.py (это
проверено архитектором построчно — self не нужен, можно импортировать
модульную функцию напрямую в новый helper-файл, поведение будет то же).

writer_move_instruction_summary в оригинале — ЕДИНСТВЕННОЕ поле этого
набора БЕЗ обёртки str(): `ctx.get("writer_move_instruction_summary")
or "нет"`. Это не опечатка, которую нужно исправлять, — это
существующее поведение, переносить ТОЧНО как есть, не добавлять str().
```

## 7. Simple explanation (для Асхата)

```text

подсказку для нейросети, подставив туда примерно 170 значений (что
сейчас происходит в разговоре, какое у бота настроение действий, что

кусками по 30-40 значений в отдельные, маленькие файлы — сама "форма"
(вызов) остаётся одна, просто перестаёт быть настолько захламлённой
внутри. Это первый такой кусок из шести запланированных.

Сейчас впервые меняется место, откуда берутся значения ВНУТРИ самого
текста-подсказки, поэтому дополнительно сверяем сам текст подсказки
(не только финальный ответ) — построчно, один в один.
```
