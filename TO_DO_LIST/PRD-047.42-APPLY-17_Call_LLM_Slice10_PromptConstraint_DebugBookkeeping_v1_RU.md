## 1. Три важные особенности — прочитать перед реализацией

```text
ОСОБЕННОСТЬ 1 — этот кластер МУТИРУЕТ user_prompt.
Строки 547-548: если prompt_section непустой, user_prompt дописывается
(f"{user_prompt}\n\n{prompt_section}"). Поэтому строгий критерий серии
(SHA1 user_prompt построчно, все 3 сценария) СОХРАНЯЕТСЯ для этого PRD
в полном объёме, хотя серия format() закрыта. Helper возвращает
итоговый user_prompt полем датакласса, вызывающий код перезаписывает
локальную переменную.

ОСОБЕННОСТЬ 2 — порядок ключей last_debug влияет на байты snapshot.
Snapshot сериализуется json.dumps БЕЗ sort_keys — порядок вставки
ключей словаря попадает в файл. Патч-словарь ОБЯЗАН собираться строго
в исходном порядке присваиваний (18 ключей, порядок задан в коде
раздела 2). Одно применение self.last_debug.update(patch) эквивалентно
по порядку последовательным присваиваниям — менять порядок ключей,
"группировать по смыслу" или сортировать НЕЛЬЗЯ.

ОСОБЕННОСТЬ 3 — snapshot-харнесс НЕ покрывает ветку дописывания.
Проверено архитектором: во всех трёх сценариях snapshot
prompt_constraint_decision = None (activation_mode: disabled,
applied: false). Значит побайтовая идентичность snapshot до/после
подтвердит только ветку "без дописывания". Ветку С дописыванием
ОБЯЗАТЕЛЬНО покрыть прямыми тестами helper'а (см. Шаг 6): непустой
prompt_section -> user_prompt оканчивается на "\n\n" + prompt_section,
applied=True, prompt_section_chars=len(...), activation_mode и
blocked_reasons извлечены из decision.

Мелочь вдогонку: overruled_constraints кладётся в патч ПО ССЫЛКЕ
(как в оригинале self.last_debug["overruled_constraints"] =
overruled_constraints) — НЕ оборачивать в list(...), не копировать.
```
