## 1. Три важные особенности — прочитать перед реализацией

```text
ОСОБЕННОСТЬ 1 — порядок правил это семантика.
Карта ОБЯЗАНА нумеровать правила в порядке исполнения и для каждого
фиксировать: (а) условие срабатывания, (б) что делает с text,
(в) есть ли ранний return. Кластеры нарезать ТОЛЬКО по границам
целых правил, никогда — внутри правила. Будущая декомпозиция сможет
переносить только непрерывные последовательности правил с сохранением
порядка.

ОСОБЕННОСТЬ 2 — вход метода это WriterContract, не ctx.
Метод первым делом делает ctx = contract.to_prompt_context().
Харнесс обязан строить детерминированные WriterContract-фикстуры.
Разрешено переиспользовать подход построения контрактов из
существующего харнесса APPLY-6 (три сценария), но их НЕ ХВАТИТ:
нужен набор case'ов, целенаправленно бьющий по семьям правил
(answer_obligation-семья, mvp_free-ветка, planner_*-семья,
question_policy-семья, practice-семья, revoicing и т.д.).
Ориентир: 12-20 case'ов. Каждый case — пара (response_text, contract)
с говорящим именем.

ОСОБЕННОСТЬ 3 — честность покрытия важнее его величины.
Не гнаться за 100%: часть правил требует экзотических состояний.
Обязательное требование — ЯВНЫЙ список непокрытых правил в журнале
покрытия с кратким "почему сложно" по каждому. Этот список станет
входом для решения архитектора о рисках декомпозиции. Приукрашивание
покрытия — прямое нарушение (самоаттестация задокументирована как
структурная уязвимость проекта).

Гигиена snapshot'ов (новое правило, действует с этого PRD):
оба snapshot-файла генерировать ПОСЛЕ установки финального значения
NORMALIZED_TIMESTAMP в runner'е — метка в файлах обязана совпадать
с константой (закрываем хроническое замечание из v4.23).
```
