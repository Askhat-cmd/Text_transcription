## 1. Особенности — прочитать перед реализацией

```text
ОСОБЕННОСТЬ 1 — исход "not_matched" НЕ является 19-м обычным исходом,
это сигнал "код группы 1-6 не сработал, работай дальше по методу".
Вызывающий код в writer_agent.py НЕ пишет `if outcome == "not_matched":
pass` — просто НЕ добавляет никакой `if` для этого случая, и управление
естественно проваливается в СЛЕДУЮЩИЙ код метода (группы 7-12,
существующий, без изменений). Это прямой аналог того, как уже
устроены slice3/slice4 (например `if slice3_result.outcome ==
"be_strong": ...` — а если ни один outcome не совпал, код просто идёt
дальше без явного else).

ОСОБЕННОСТЬ 2 — единственная прямая last_debug-запись (строка 894).
Оригинал:
    self.last_debug["template_leakage_repair_deferred_to_gate"] = True
    self._set_final_answer_shape_debug("template_repair_deferred_to_gate")
    return self._strip_optional_followup_invitation(text) or text
helper возвращает outcome="practice_forbidden_unsolicited_repair" и
last_debug_patch={"template_leakage_repair_deferred_to_gate": True} —
БЕЗ вызова self-методов внутри себя. Вызывающий код делает СНАЧАЛА
self.last_debug.update(patch), ПОТОМ self._set_final_answer_shape_debug(...),
ПОТОМ return self._strip_optional_followup_invitation(text) or text —
ровно в этом порядке, byte-for-byte как в оригинале. Это единственная
last_debug-ловушка Block B, и она проще, чем в APPLY-22 (там было ветвление
после return; здесь — плоская прямая последовательность из трёх строк).

ОСОБЕННОСТЬ 3 — повторяющиеся исходные сигналы внутри этого среза.
"known_concept_correlation_repair" и "known_concept_neurostalking_repair"
встречаются КАЖДЫЙ по 2 раза внутри строк 804-896 (группа 1 и вложенное
условие группы 2) — как РАЗНЫЕ физические условия кода. Классификатор
обязан различать их как РАЗНЫЕ теги исхода (known_concept_prefirst_*
для группы 1, no_question_known_concept_* для группы 2) — это нужно
для тестируемости и трассировки (прямой прецедент APPLY-26, где
несколько исходов вели к одному тексту, но оставались разными тегами).
На call site эти теги ОБЪЕДИНЯЮТСЯ в один return (одинаковый self-вызов
с одинаковыми аргументами) — это осознанное объединение уровня вывода,
а не подмена логики.

ОСОБЕННОСТЬ 4 — маркерные tuple в группах 3 и 5 ("что именно"/"почему"/...
и "практик"/"упражн"/"таймер"/"шаг") — это ЛИТЕРАЛЬНЫЕ inline-tuple в
оригинале (не module-level константы, в отличие от _LOW_RESOURCE_NO_
PRACTICE_MARKERS из APPLY-26). Циклического импорта нет — просто
переносятся as-is внутрь helper'а как inline-литералы, параметром их
делать не нужно (в отличие от Особенности 3 из APPLY-22, где маркеры
БЫЛИ module-level именами в writer_agent.py).

ОСОБЕННОСТЬ 5 — has_repair_forbidden (строка 885) — чисто внутренняя
локальная переменная группы 5, используется только внутри одного if на
следующей строке и нигде больше в оригинале. В датаклассе результата
НЕ нужна — вычисляется и используется целиком внутри helper'а.
```
