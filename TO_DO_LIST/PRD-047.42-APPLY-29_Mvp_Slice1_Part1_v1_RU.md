## 0. Короткое решение

```text
Метод — другая сигнатура (явные keyword-параметры, не (self,
response_text, contract)), другая структура, ~35 параметров. НЕ
наследует допущения от _enforce_answer_compliance.

Разведка нашла 2 находки, которых не было в прошлом методе:
1. offer_repair_context (строка 1018) — вычисляется, нигде не
   используется. Унаследованный мёртвый код (прецедент — 2 недостижимых
   исхода из APPLY-26). НЕ чинится, НЕ переносится в helper,
   документируется.
2. Одна группа (B, "pragmatics_contextual_followup") мутирует text/
   lowered_text В МЕСТЕ, и если группа НЕ возвращает — мутация
   переживает группу и видна ВСЕМ последующим группам. Новый паттерн,
   не встречавшийся в прошлой декомпозиции. Решение: результат
   классификатора ВСЕГДА несёт updated_text/updated_lowered_text
   (даже когда outcome не not_matched — дёшево, безопасно), вызывающий
   код в writer_agent.py делает `text = result.updated_text; lowered_text
   = result.updated_lowered_text` СРАЗУ после вызова классификатора,
   до проверки outcome.

Метод режется на 2 PRD по объёму (225 строк ~ втрое больше Block A) —
см. ADR-126 для критерия. Этот PRD — Part 1 (группы A-J, строки
1019-1126, 108 строк). Part 2 (группы K-P + финальный fallback метода) —
следующий PRD (APPLY-30), уже спроектирован и провалидирован в этой же
сессии.

last_debug: ОДНА точка прямой записи в этой части (группа J, конец,
2 разных физических условия пишут одинаковую пару ключей — прямой
прецедент APPLY-27/28: патч применяется ПЕРЕД self-вызовом).

Дифференциальный фаззинг-тест (20 000 комбинаций против дословно
извлечённого кода строк 1019-1126, включая мутацию text/lowered_text
как часть проверки) — 0 расхождений.

Новый модуль: writer_agent_mvp_slice1.py.
```

## 1. Особенности

```text
1. Мутация text/lowered_text (группа B) переносится через
   updated_text/updated_lowered_text — ВСЕГДА возвращаются, вызывающий
   код переприсваивает их СРАЗУ после вызова классификатора, вне
   зависимости от outcome.
2. Группа A вычисляет `target` (последний вопрос или сообщение
   пользователя) — нужен на call site для must_answer. Возвращается как
   computed_target.
3. last_debug (группа J, конец): две ветки (1118-1122 и 1123-1126)
   пишут одинаковую пару ключей ({"answer_fit_repair_applied": True,
   "template_leakage_repair_deferred_to_gate": True}) при РАЗНЫХ
   условиях — остаются разными исходами классификатора (тестируемость),
   патч одинаковый.
4. detect_stale_stub — модульная функция (from ..stale_stub_detector),
   не self, безопасна для прямого вызова внутри helper'а.
```
