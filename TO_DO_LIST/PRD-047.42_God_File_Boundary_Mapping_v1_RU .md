## 0. Короткое решение

```text
Полный scope §5.3 ("writer_agent.py + admin_routes.py + writer_contract.py
+ 19 production diagnostic_center_* файлов") — это ~18 000 строк за один
PRD. Это нарушает собственное правило проекта "каждый PRD маленький"
(Transfer Brief §6) даже без единой строки кода, изменяющей поведение.

Этот PRD делает только одну вещь: строит ТОЧНУЮ карту границ для 3
явно названных god-файлов (writer_agent.py, admin_routes.py,
writer_contract.py, вместе 5311 строк) — без единого изменения кода.
19 diagnostic_center-файлов (~12700 строк) — отдельный будущий
PRD-047.42b, у них другой, более однородный паттерн (gate-модули),
разумнее разбирать отдельно со своим шаблоном.

Реальное разбиение (перенос кода по новым файлам) — НЕ в этом PRD.
Это Stage 1 (карта). Stage 2 (резка) — отдельный PRD-047.42-APPLY,
gated на ревью этой карты владельцем/архитектором. Тот же паттерн,
что PRD-047.39 -> PRD-047.40/041.
```

## 2. Scope PRD-047.42

```text
A. Построчная карта границ для writer_agent.py, admin_routes.py,
   writer_contract.py — какой код относится к какой ответственности,
   с точными диапазонами строк (не общими словами "логика Writer'а").
B. Предложение конкретных новых модулей/файлов под каждую группу
   ответственности — имя файла, что в него переезжает, что остаётся.
C. Contract-test набор "поведенческий снимок ДО" — для каждого из 3
   файлов зафиксировать текущий вход/выход на реальных или
   синтетических кейсах, чтобы Stage 2 (PRD-047.42-APPLY) имел с чем
   сравнивать после переноса кода.
D. Явный список зависимостей и вызывающего кода для каждого из 3
   файлов (кто их импортирует извне) — чтобы Stage 2 знал, что нельзя
   менять по внешнему контракту (имена классов/публичных методов).
```

## 3. Non-goals

```text
- НЕ переносит ни одной строки кода между файлами.
- НЕ меняет ни одну сигнатуру публичного метода/класса.
- НЕ трогает 19 diagnostic_center_* production-файлов — отдельный
  PRD-047.42b.
- НЕ трогает 68 файлов защищённой зоны (contracts/_v1, tests/, tools/)
  ни в каком виде.
- НЕ трогает Writer prompt содержимое, retrieval ranking, safety-логику,
  Bot_data_base, Chroma, registry.
- НЕ продолжает PRD-047.43 (admin UI dedup) или PRD-047.44/045/046.
- НЕ является "предложением поправить попутно" — если Исполнитель видит
  явный баг при чтении кода, он фиксирует его в отчёте как находку,
  не исправляет тут же.
```

## 4. Required implementation

```text
Repo-level:
tools/run_prd_047_42_god_file_boundary_mapping.py

Для КАЖДОГО из 3 файлов (writer_agent.py, admin_routes.py,
writer_contract.py) Исполнитель обязан:

1. Прочитать файл целиком (не полагаться на предварительную разведку
   архитектора из раздела 1 — она черновая и может быть неточной;
   перепроверить границы методов, номера строк, реальный размер каждого
   блока).
2. Для каждого метода/группы функций длиннее ~50 строк — определить
   доминирующую ответственность (parsing/validation/prompt_assembly/
   trace_debug/legacy_compat/retrieval/streaming/persistence/writer —
   таксономия унаследована из PRD-047.39 god_file_inventory.md) и
   зафиксировать точный диапазон строк.
3. Явно пометить legacy_compat-фрагменты (мёртвый/обратно-совместимый
   код) отдельно от активной логики — это кандидаты для отдельной,
   более лёгкой последующей чистки, не для Stage 2 переноса.
4. Предложить итоговую структуру файлов: имя нового модуля -> что в
   него входит -> примерный итоговый размер в строках. Сумма итоговых
   файлов должна давать явно более мелкие файлы (цель: ни один
   результирующий файл не должен сам оказаться god-file'ом, то есть
   быть ощутимо меньше 500 строк, где это разумно).
5. Собрать contract-test снимок: для writer_agent.py и
   writer_contract.py — набор реальных или синтетических input/output
   пар через _resolve_runtime_settings / _enforce_answer_compliance /
   to_prompt_context (насколько это возможно без живых provider-вызовов
   — с моками); для admin_routes.py — HTTP response snapshot по каждому
   из ~45 route-хендлеров (уже есть похожий паттерн в
   test_admin_runtime_contract.py, использовать как образец).
6. Собрать список внешних вызывающих мест (grep + import graph) для
   каждого из 3 файлов — что импортирует WriterAgent, WriterContract,
   что импортирует функции из admin_routes.py.

Explicitly НЕ обязан:
- предлагать финальные имена файлов как нечто неизменное — это
  предложение для ревью, не решение;
- запускать реальные provider-вызовы (OpenAI и т.п.) для contract-test
  снимков — использовать существующие моки/фикстуры проекта.
```

## 6. Acceptance decision

```text
ACCEPTED               -> карта построена для всех 3 файлов с точными
                           диапазонами строк, contract-test снимок
                           зелёный и покрывает основные пути (не 100%
                           веток, но явно больше одного happy-path на
                           метод), внешние зависимости перечислены,
                           no_mutation_proof подтверждает 0 изменений
                           в самих 3 файлах.
ACCEPTED_WITH_WARNINGS -> карта построена, но contract-test снимок
                           покрывает не все крупные методы (например
                           только 2 из 3 giant-методов writer_agent.py)
                           — тогда недостающее фиксируется как
                           конкретный gap, не общими словами.
BLOCKED                 -> обнаружено, что no_mutation_proof показал
                           хоть одно изменение в исходных 3 файлах.
INCONCLUSIVE             -> при построении карты выяснилось, что
                           границы ответственности физически нельзя
                           разделить без изменения поведения (например
                           общее замыкание/shared state между тем, что
                           предполагалось раскидать по разным модулям)
                           — тогда PRD-047.42-APPLY нужно переосмыслить
                           до старта, не резать вслепую.
```

## 7. Risk register

```text
Risk O — Карта строится по старой/неточной разведке архитектора
  Symptom: Исполнитель копирует диапазоны строк из раздела 1 этого PRD
           не перепроверяя.
  Response: раздел 4 п.1 explicitly требует перечитать файлы заново.
            Разведка архитектора — ориентир, не источник истины.

Risk P — Contract-test снимок создаёт иллюзию полноты
  Symptom: 1-2 happy-path теста на 900-строчный метод выглядят как
           "поведение зафиксировано", но реально покрывают малую долю
           веток; PRD-047.42-APPLY потом резёт код и "тесты проходят",
           хотя сломана непокрытая ветка.
  Response: отчёт должен явно указывать % покрытых веток/условий по
            каждому giant-методу, не только "N тестов зелёные".

Risk Q — writer_contract.to_prompt_context() (890 строк, 91% файла)
  Symptom: это прямая сериализация в промпт для LLM; любая ошибка
           здесь меняет то, что бот говорит пользователю. Самый
           чувствительный код из всех god-файлов проекта.
  Response: для этого метода contract-test снимок обязателен, не
            best-effort — INCONCLUSIVE, если покрытие для него хуже,
            чем для двух других файлов.
```

## 8. Simple explanation

```text
Три файла в проекте — самые большие и самые запутанные: один пишет
финальный ответ бота (writer_agent.py), один превращает этот ответ в
то, что реально уходит в языковую модель (writer_contract.py), один —
админка (admin_routes.py). Их давно пора разложить на файлы поменьше,
но резать вслепую 5300+ строк рискованно — можно случайно поменять,
что бот говорит людям.

Поэтому сначала — только карта: что где лежит, что от чего зависит,
и "фотография" текущего поведения (тесты, которые докажут потом, что
после разбиения бот отвечает точно так же). Резать сам код будем
только следующим PRD, после того как я и ты посмотрим на карту.
```
