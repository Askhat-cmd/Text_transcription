# PRD-047.4 — Response Planner v1 / Next Meaningful Move Calibration

## 0. Краткое резюме

`PRD-047.4` добавляет слой **Response Planner v1** — лёгкий deterministic layer между `Active Line / Dialogue Continuity` и `WriterContract`.

Цель слоя: не писать ответ пользователю, а выбрать **следующий смысловой ход** для Writer:

```text
что сейчас полезнее всего сделать в ответе:
ответить прямо,
углубить механизм,
починить ошибку,
дать короткую поддержку,
дать один шаг,
закрыть контакт,
стабилизировать safety,
или задать один точный вопрос.
```

Главная задача PRD-047.4 — сделать ответы бота менее случайными по траектории: чтобы Writer не просто получал `response_mode`, `active_line` и общие правила, а имел компактный план конкретного следующего хода.

Важно: это **не новый LLM-агент**. Это deterministic planner, построенный на уже существующих сигналах:

```text
state_snapshot
thread_state
active_line
knowledge_answer
practice_gate
philosophy_kernel
writer_freedom_contract
diagnostic_card
context_package_summary
```

### 1.2. Принятый warning после PRD-047.3

В `PRD-047.3` есть известное ограничение:

```text
Live runner may rely on local active-line fallback when adaptive API returns compatibility trace without embedded debug.active_line field.
```

Это не блокировало принятие PRD-047.3, но для PRD-047.4 вводится более строгий стандарт:

```text
Live acceptance для response_planner запрещено засчитывать через runner-side fallback.
```

Если live API response не содержит настоящий `debug.response_planner`, live acceptance должен быть `failed`, даже если локальный runner может построить expected planner decision самостоятельно.

### 1.3. Почему следующий шаг именно Response Planner

Active Line решает вопрос:

```text
какую линию разговора держим,
что пользователь сейчас хочет,
можно ли предлагать практику,
можно ли переозвучивать,
нужно ли repair.
```

Но Active Line ещё не отвечает достаточно полно на вопрос:

```text
какая форма ответа сейчас лучше всего продвинет диалог на один смысловой шаг?
```

Например:

```text
active_line.user_intent = understand_mechanism
active_line.should_offer_practice = false
active_line.next_meaningful_move = показать механизм
```

Этого достаточно, чтобы запретить неуместную практику, но недостаточно, чтобы Writer всегда понимал:

```text
ответить коротко или средне;
дать объяснение или один вопрос;
углубить через механизм или прямо назвать вывод;
закрыть диалог или продолжить;
связать known concept с текущей линией или дать самостоятельное определение;
какую именно ошибку нельзя повторить.
```

`Response Planner v1` должен заполнить этот разрыв.

## 4. Product principles

### 4.1. Один ответ = один следующий смысловой ход

Planner должен защищать Writer от расползания:

```text
не объяснение + практика + новая теория + вопрос + домашнее задание,
а один точный ход.
```

### 4.2. Active Line — источник линии, Planner — источник формы хода

`Active Line` отвечает:

```text
что за линия разговора сейчас активна.
```

`Response Planner` отвечает:

```text
как именно продолжить эту линию в следующем ответе.
```

### 4.3. Practice is never default

Практика допустима только если:

```text
- пользователь явно просит практику / шаг;
- safety/regulation требует стабилизации;
- practice_gate разрешил практику;
- active_line не запрещает практику;
- planner выбрал practice_policy != forbidden.
```

### 4.4. Revoicing is not understanding

Если `active_line.revoicing_allowed=false`, planner должен выставлять:

```json
"revoicing_policy": "suppressed"
```

и добавить соответствующий `must_avoid`.

### 4.5. Writer Freedom Contract сохраняется

Planner не должен превращать Writer в исполнителя жёсткого скрипта.

Правильная роль planner:

```text
дать ясный вектор и ограничения,
но оставить Writer живым автором ответа внутри hard boundaries.
```

### 4.6. Safety beats planner

Если есть safety/crisis/safe_override, planner не спорит с safety.

Он должен выбрать:

```text
next_move = stabilize_safety
answer_shape = safety_grounding
practice_policy = required_for_safety_or_grounding
response_depth = short
```

и не углублять философию.

## 5. Required architecture

### 5.1. Новый модуль

Создать файл:

```text
bot_psychologist/bot_agent/multiagent/response_planner.py
```

Минимальная структура:

```python
RESPONSE_PLANNER_VERSION = "response_planner_v1"

@dataclass(frozen=True)
class ResponsePlannerDecision:
    version: str
    enabled: bool
    next_move: str
    answer_shape: str
    response_depth: str
    target_micro_shift: str
    should_answer_directly: bool
    question_policy: str
    practice_policy: str
    revoicing_policy: str
    continuity_policy: str
    safety_priority: bool
    must_include: list[str]
    must_avoid: list[str]
    source_signals: dict[str, Any]
    confidence: float
    rationale: str

    def to_dict(self) -> dict[str, Any]: ...
```

Допускается pydantic или dataclass — использовать стиль, который лучше соответствует текущему проекту.

### 5.2. Основная функция

Добавить функцию:

```python
def build_response_planner_decision(
    *,
    user_message: str,
    state_snapshot: Any,
    thread_state: Any,
    diagnostic_card: Any | None,
    active_line: dict[str, Any],
    knowledge_answer_guard: dict[str, Any],
    philosophy_kernel: dict[str, Any],
    context_package: Any | None = None,
) -> ResponsePlannerDecision:
    ...
```

Функция должна быть deterministic и не должна вызывать LLM.

## 7. Decision matrix v1

Planner должен реализовать минимум такую матрицу приоритетов.

### 7.1. Safety / crisis / safe override

Если:

```text
thread_state.safety_active == true
или state_snapshot.safety_flag != none
или thread_state.response_mode == safe_override
```

то:

```json
{
  "next_move": "stabilize_safety",
  "answer_shape": "safety_grounding",
  "response_depth": "short",
  "question_policy": "none",
  "practice_policy": "required_for_safety_or_grounding",
  "revoicing_policy": "suppressed",
  "safety_priority": true
}
```

### 7.2. Correction / repair

Если:

```text
active_line.user_intent == correction_of_bot
или active_line.continuity_mode == repair_and_continue_line
или active_line.repair_mode != null
```

то:

```json
{
  "next_move": "repair_misalignment",
  "answer_shape": "repair_acknowledgement",
  "response_depth": "short",
  "question_policy": "none",
  "practice_policy": "forbidden",
  "revoicing_policy": "suppressed",
  "continuity_policy": "repair_and_continue"
}
```

Must include:

```text
признать сдвиг / ошибку;
вернуться к active_line;
не предлагать новую практику.
```

### 7.3. Thanks / close

Если:

```text
active_line.user_intent == thanks_close
```

то:

```json
{
  "next_move": "close_gently",
  "answer_shape": "gentle_close",
  "response_depth": "very_short",
  "question_policy": "none",
  "practice_policy": "forbidden",
  "continuity_policy": "close_without_new_loop"
}
```

### 7.4. Short support

Если:

```text
active_line.user_intent == short_support
```

то:

```json
{
  "next_move": "give_short_support",
  "answer_shape": "short_support",
  "response_depth": "very_short",
  "question_policy": "none",
  "practice_policy": "forbidden",
  "revoicing_policy": "suppressed"
}
```

### 7.5. Known concept / knowledge answer

Если:

```text
knowledge_answer.should_answer_directly == true
или active_line.user_intent == known_concept_question
```

то:

```json
{
  "next_move": "answer_known_concept",
  "answer_shape": "compact_direct",
  "response_depth": "medium",
  "should_answer_directly": true,
  "question_policy": "none",
  "practice_policy": "forbidden"
}
```

Must avoid:

```text
не спрашивать пользователя, что он вкладывает в известный термин;
не уходить в практику;
не цитировать KB как авторитет;
не давать книжную лекцию.
```

### 7.6. Explicit practice / direct step

Если:

```text
active_line.user_intent == ask_for_practice
или active_line.user_intent == ask_for_direct_step
```

и:

```text
practice_gate.practice_allowed == true
```

то:

```json
{
  "next_move": "give_direct_step" или "give_practice",
  "answer_shape": "one_step",
  "response_depth": "short",
  "question_policy": "none",
  "practice_policy": "one_micro_step_allowed"
}
```

Если `practice_gate.practice_allowed == false`, planner должен запретить практику и выбрать либо `deepen_mechanism`, либо `clarify_one_point`.

### 7.7. Understand mechanism

Если:

```text
active_line.user_intent == understand_mechanism
```

то:

```json
{
  "next_move": "deepen_mechanism",
  "answer_shape": "mechanism_explanation",
  "response_depth": "medium",
  "practice_policy": "forbidden",
  "revoicing_policy": "suppressed"
}
```

Question policy:

```text
none — если active_line.should_ask_question == false;
max_one_if_needed — если active_line.should_ask_question == true.
```

### 7.8. Low resource / hypo / collapsed

Если:

```text
state_snapshot.nervous_state == hypo
или thread_state.openness == collapsed
или thread_state.ok_position == I-W-
```

то planner должен снижать нагрузку:

```text
response_depth <= short
question_policy = none или max_one_if_needed
must_avoid += ["не философствовать", "не перегружать анализом", "не давать длинный список"]
```

### 7.9. Default

Если ни одно правило не сработало:

```json
{
  "next_move": "continue_active_line",
  "answer_shape": "compact_direct",
  "response_depth": "short",
  "question_policy": "max_one_if_needed",
  "practice_policy": "forbidden",
  "revoicing_policy": "suppressed"
}
```

## 9. Orchestrator integration

В `bot_psychologist/bot_agent/multiagent/orchestrator.py` добавить planner после построения:

```text
context_package
knowledge_answer_guard
philosophy_kernel_payload
active_line_state
diagnostic_card
```

и до:

```text
WriterContract(...)
```

Пример порядка:

```python
active_line_state = build_active_line_state(...).to_dict()
diagnostic_card = build_diagnostic_card_v1(...)

response_planner_decision = build_response_planner_decision(
    user_message=query,
    state_snapshot=state_snapshot,
    thread_state=updated_thread,
    diagnostic_card=diagnostic_card,
    active_line=active_line_state,
    knowledge_answer_guard=knowledge_answer_guard,
    philosophy_kernel=philosophy_kernel_payload,
    context_package=context_package,
).to_dict()

writer_contract = WriterContract(
    ...,
    active_line=active_line_state,
    response_planner=response_planner_decision,
)
```

Если planner падает, ответ не должен ломаться. Но fallback должен быть явным:

```json
"response_planner_error": "response_planner_failed:ValueError",
"response_planner": {
  "version": "response_planner_v1",
  "enabled": false,
  "next_move": "fallback_continue_active_line",
  ...
}
```

В live acceptance fallback не должен засчитываться как полноценный pass, если PRD-047.4 runner проверяет именно planner качество.

## 10. Writer prompt integration

Обновить:

```text
bot_psychologist/bot_agent/multiagent/agents/writer_agent_prompts.py
```

Добавить компактный блок:

```text
RESPONSE PLANNER
- next_move: {planner_next_move}
- answer_shape: {planner_answer_shape}
- response_depth: {planner_response_depth}
- target_micro_shift: {planner_target_micro_shift}

- revoicing_policy: {planner_revoicing_policy}

- must_avoid: {planner_must_avoid}
```

Правила для Writer:

```text
1. Safety hard rules выше planner.
2. Knowledge Answer hard rules выше planner.
3. Practice Gate hard rules выше planner.
4. Active Line и Response Planner — soft guidance, но их запреты по practice/revoicing/question должны уважаться.
5. Если planner.practice_policy=forbidden — не предлагай практику, упражнение, шаг, таймер, технику.
6. Если planner.question_policy=none — не заканчивай вопросом.
7. Если planner.revoicing_policy=suppressed — не начинай с пересказа вопроса пользователя.
8. Делай один следующий смысловой ход, указанный в target_micro_shift.
```

Не делать prompt слишком длинным. Planner block должен быть компактным.

## 12. Trace / API requirements

```json

"response_planner": { ... },
```

### 12.2. API response

Проверить, что `response_planner` не теряется в API layer.

Особенно проверить:

```text

/api/debug/session/{session_id}/multiagent-trace

```

Если есть Pydantic model для trace, добавить поля:

```python
response_planner_version: str | None = None
response_planner: dict[str, Any] | None = None
response_planner_error: str | None = None
```

### 12.3. Strict live evidence rule

Live runner PRD-047.4 должен читать planner только из API response/debug trace.

Запрещено:

```text
если API не вернул debug.response_planner,
runner строит local planner decision и засчитывает pass.
```

Разрешено:

```text
runner строит local expected decision для comparison,
но actual должен быть только из API trace.
```

Если actual отсутствует:

```json
{
  "passed": false,
  "failure_reason": "missing_live_api_debug_response_planner"
}
```

Если actual содержит fallback marker:

```json
{
  "passed": false,
  "failure_reason": "live_response_planner_fallback_not_allowed_for_acceptance"
}
```

### 14.2. Runner

Создать:

```text
bot_psychologist/scripts/run_prd_047_4_response_planner_cases.py
```

Режимы:

```text
--mode dry
--mode direct
--mode live
```

#### Dry mode

Проверяет `build_response_planner_decision()` на fixture signals без API и без provider.

#### Direct mode

Проверяет planner integration через local orchestrator / local assembled contract там, где это возможно без внешнего API.

Допускается использовать stubs для LLM-dependent writer outputs, но planner actual должен быть реальным output функции.

#### Live mode

Запускается против fresh temporary backend.

Требования:

```text
- actual response_planner берётся только из live API response/debug trace;
- local expected используется только для comparison;
- отсутствие debug.response_planner = fail;
- response_planner_error != null = fail;
- enabled=false = fail;
- fallback/source marker для acceptance = fail;
- live API freshness probe должен подтвердить response_planner block в /api/admin/runtime/effective.
```
