# PRD-047.5 — Planner Quality Calibration / Live Dialogue Answer Fit v1

## 0. Краткое резюме

`PRD-047.5` продолжает трек Multiagent Quality & Tuning после `PRD-047.4 Response Planner v1`.

`PRD-047.4` успешно добавил структурный слой `Response Planner v1`:

```text
Active Line → Response Planner → WriterContract → Writer → Validator / Trace
```

Но аудит live artifacts показал следующий качественный bottleneck:

```text
Planner уже стабильно присутствует в trace и API,
но часть live-кейсов получает формально валидный, но смыслово не лучший next_move.
```

Особенно это видно в группах:

```text
low_resource / hypo-like user text
soft distress / safety-adjacent user text
I+W- defensive / world-blaming user text
thanks / close
no-question explicit user request
explicit practice / direct-step request when Active Line misclassified upstream
```

Главная цель `PRD-047.5` — не добавлять новую архитектуру, а откалибровать `Response Planner v1` и runner/evaluator так, чтобы он оценивал не только наличие planner trace, но и соответствие финального ответа живой ситуации.

Формула PRD:

```text
PRD-047.4 доказал: planner встроен и виден.
PRD-047.5 должен доказать: planner выбирает уместный ход и Writer ему следует.
```

---

## 1. Контекст

### 1.1. Что принято после PRD-047.4

На момент старта PRD-047.5 должны быть приняты:

```text
PRD-047.0-HF1 — answer-level evaluator repair
PRD-047.1     — NEO Philosophy Kernel + Writer Freedom Contract
PRD-047.2     — Kernel Quality Calibration
PRD-047.3     — Active Line / Dialogue Continuity v1
PRD-047.4     — Response Planner v1 / Next Meaningful Move Calibration
```

`PRD-047.4` считается accepted with warning:

```text
accepted:    response_planner is integrated and traceable
warning:     some live cases pass structurally but reveal weak semantic fit
```

### 1.2. Почему нужен PRD-047.5

`PRD-047.4` runner проверяет, что:

```text
- planner present
- enabled true
- expected fields match
- no fallback source in live acceptance
- API trace contains response_planner
```

Но этого уже недостаточно. Теперь нужно проверять:

```text
- подходит ли next_move к живому состоянию пользователя;
- не слишком ли глубоко отвечает бот на low-resource запрос;
- не пропускает ли soft distress;
- не входит ли в коалицию с I+W- defensive framing;
- не задаёт ли вопрос, когда пользователь явно просит без вопросов;
- не открывает ли новый loop на thanks/close;
- не даёт ли practice там, где был запрос на понимание;
- не превращает ли direct-step request в очередной анализ;
- следует ли Writer planner contract в финальном answer text.
```

## 2. Главная цель

Добавить `Planner Quality Calibration / Live Dialogue Answer Fit v1`, который:

1. Расширяет PRD-047.4 dataset до живых answer-fit сценариев.
2. Калибрует deterministic `response_planner.py` на слабых live-кейсах.
3. Добавляет answer-level evaluator поверх planner-level evaluator.
4. Проверяет, что финальный ответ Writer соответствует planner decision.
5. Усиливает live evidence: planner trace должен приходить из API, а final answer должен проходить quality checks.
6. Не добавляет новых LLM-агентов.
7. Не меняет BotDB / Chroma / KB governance.
8. Не включает broad rollout / production-ready.

## 6. Response Planner calibration requirements

### 6.1. Add text-level override signals inside planner

`Response Planner` сейчас слишком зависит от upstream signals:

```text
active_line_user_intent
state_snapshot.nervous_state
thread_state.ok_position
thread_state.response_mode
```

PRD-047.5 должен добавить lightweight deterministic text-signal layer inside planner, не создавая нового агента.

В `response_planner.py` добавить internal helpers:

```python
def detect_low_resource_text(user_message: str) -> bool: ...
def detect_soft_distress_text(user_message: str) -> bool: ...
def detect_world_blame_defensive_text(user_message: str) -> bool: ...
def detect_no_question_request(user_message: str) -> bool: ...
def detect_close_text(user_message: str) -> bool: ...
def detect_explicit_practice_or_step_request(user_message: str) -> dict: ...
```

Названия можно адаптировать под стиль проекта.

### 6.2. Low-resource calibration

Текущая проблема:

```text
"Я как будто схлопнулся и пустой, не тяну длинные ответы."
```

может уйти в:

```text
next_move=deepen_mechanism
answer_shape=mechanism_explanation
response_depth=medium
```

Нужно калибровать:

```json
{
  "next_move": "give_short_support",
  "answer_shape": "short_support",
  "response_depth": "very_short",
  "question_policy": "none",
  "practice_policy": "forbidden",
  "revoicing_policy": "suppressed",
  "must_include": ["коротко", "без давления"],
  "must_avoid": ["без лекции", "без практики", "без глубокого анализа"]
}
```

Текстовые маркеры low-resource:

```text
схлопнулся
пустой
не тяну длинные ответы
нет сил
не могу думать
очень устал
без анализа
пару спокойных слов
просто поддержи
не грузи
коротко
```

### 6.3. Soft-distress / safety-adjacent calibration

Текущая проблема:

```text
"Мне сейчас очень плохо, не выдерживаю."
```

может уйти в обычный mechanism explanation, если upstream safety flag не сработал.

PRD-047.5 не должен заменять crisis routing, но planner должен иметь мягкий safety-adjacent guard.

Если есть soft distress markers:

```text
очень плохо
не выдерживаю
не вывожу
мне страшно
я разваливаюсь
не справляюсь прямо сейчас
```

и нет явного self-harm intent, planner должен выбрать:

```json
{
  "next_move": "stabilize_safety",
  "answer_shape": "safety_grounding",
  "response_depth": "short",
  "question_policy": "none",
  "practice_policy": "required_for_safety_or_grounding",
  "safety_priority": true,
  "must_include": ["короткая стабилизация", "здесь-и-сейчас", "простая опора"],
  "must_avoid": ["не уходить в философскую глубину", "не расширять анализ"]
}
```

Если уже существующий safety/crisis router сработал — его приоритет абсолютный.

### 6.4. Defensive / I+W- text calibration

Текущая проблема:

```text
"Мир тупит, люди тормозят, а мне надо всех дожать. Что с этим делать?"
```

может уйти в обычный mechanism explanation, если upstream `ok_position` не определил `I+W-`.

Добавить text-level detection:

```text
все тупят
люди тормозят
они идиоты
мир не понимает
мне надо всех дожать
почему они такие
все вокруг неадекватные
```

Planner должен выбирать:

```json
{
  "next_move": "clarify_one_point",
  "answer_shape": "one_question",
  "response_depth": "short",
  "target_micro_shift": "снять defensive-поляризацию через один decenter-вопрос",
  "should_answer_directly": false,
  "question_policy": "required_one_clarifying",
  "practice_policy": "forbidden",
  "must_include": ["один вопрос для декентрации"],
  "must_avoid": ["не входить в коалицию против мира", "без длинной лекции"]
}
```

### 6.5. Close / thanks calibration

Если пользователь пишет:

```text
спасибо
благодарю
понял, спасибо
ок, спасибо
```

Planner должен выбирать:

```json
{
  "next_move": "close_gently",
  "answer_shape": "gentle_close",
  "response_depth": "very_short",
  "question_policy": "none",
  "practice_policy": "forbidden",
  "continuity_policy": "close_without_new_loop",
  "must_avoid": ["не открывать новый шаг", "не давать практику", "не задавать вопрос"]
}
```

Даже если Active Line misclassified `thanks` as `understand_mechanism`, planner должен корректировать по тексту.

### 6.6. No-question request calibration

Если пользователь явно просит:

```text
без вопросов
не задавай вопросов
просто продолжи мысль
ответь без вопроса
```

то planner должен ставить:

```json
{
  "question_policy": "none",
  "must_avoid": ["не задавать вопрос"]
}
```

Это должно иметь приоритет над `active_line_should_ask_question=true`.

### 6.7. Explicit action / practice request calibration

Если пользователь явно просит действие:

```text
дай один шаг
что сделать прямо сейчас
дай практику
хочу действие
```

Planner должен учитывать текст, даже если Active Line ошибся.

Правила:

```text
если explicit action requested + practice_gate_allowed=true → one_micro_step_allowed
если explicit action requested + practice_gate_allowed=false → clarify_one_point / no practice
если user asked “один шаг” → answer_shape=one_step, no list
если user asked “практика” → one_micro_step_allowed only if gate allows
```

## 7. Writer answer-fit compliance requirements

### 7.1. Final answer must obey planner contract

Добавить или усилить answer-level checks:

```text
planner.next_move=give_short_support       → final answer very short, no analysis, no practice
planner.next_move=stabilize_safety         → final answer short, grounding/supportive, no deep theory
planner.next_move=clarify_one_point        → final answer has exactly one question or one concise decenter move
planner.next_move=close_gently             → final answer does not open new loop/question/practice
planner.practice_policy=forbidden          → final answer has no practice/step/timer markers
planner.question_policy=none               → final answer has no question mark and no implicit question opener
planner.revoicing_policy=suppressed        → final answer does not start with mechanical revoicing
planner.answer_shape=one_step              → final answer has exactly one step, not a list
```

### 7.2. Writer prompt update

Update `writer_agent_prompts.py` minimally:

Add a compact planner-obedience rule:

```text
Planner is not decorative. It defines the answer shape.
If planner says short_support / safety_grounding / gentle_close, do not expand into mechanism analysis.
If planner.question_policy=none, do not end with a question.
If planner.practice_policy=forbidden, do not give steps, exercises, timers, or practice-like instructions.
If planner.answer_shape=one_step, give exactly one step, not a list.
```

Keep prompt compact. Do not add a long manifesto.

### 7.3. Optional deterministic post-check, warning-only

Допустимо добавить warning-only function in validator or runner:

```python
build_planner_answer_fit_flags(final_answer, response_planner) -> dict
```

Но в runtime пока не блокировать ответ, если это требует риска. Runner/evaluator должен блокировать acceptance по dataset, runtime может только trace flags.

## 9. Runner requirements

Создать runner:

```text
bot_psychologist/scripts/run_prd_047_5_planner_answer_fit_cases.py
```

Runner должен поддерживать:

```bash
python scripts/run_prd_047_5_planner_answer_fit_cases.py --mode dry
python scripts/run_prd_047_5_planner_answer_fit_cases.py --mode direct
python scripts/run_prd_047_5_planner_answer_fit_cases.py --mode live --api-base-url http://127.0.0.1:8013/api/v1 --admin-runtime-url http://127.0.0.1:8013/api/admin/runtime/effective
```

Допустимые дополнительные флаги:

```text
--case-id <id>
--limit <n>
--output-dir TO_DO_LIST/logs/PRD-047.5
```

### 9.1. Dry mode

Dry mode проверяет:

```text
- dataset schema
- coverage by groups
- expected planner fields valid
- expected answer fit fields valid
```

### 9.2. Direct mode

Direct mode вызывает planner/writer path locally where possible.

Если прямой Writer provider call недоступен, direct mode должен честно отделять:

```text
planner_direct_checks
answer_fit_direct_checks_if_available
```

Не выдавать skipped за passed.

### 9.3. Live mode

Live mode должен:

```text
- запускаться на fresh temporary backend, например 127.0.0.1:8013;
- проверять /api/admin/runtime/effective на наличие response_planner;
- отправлять кейсы через real API path;
- извлекать debug.response_planner only from API response/trace;
- извлекать final answer text;
- запускать planner evaluator;
- запускать answer-fit evaluator;
- fail если planner missing / planner_error / fallback marker;
- fail если final answer violates expected_answer_fit;
- сохранять trace samples без raw private logs.
```

Запрещено:

```text
- достраивать planner locally для live acceptance;
- засчитывать кейс как passed, если API planner missing;
- засчитывать answer-fit как passed без final answer text;
- коммитить raw provider payload.
```
