# PRD-047.6 — Planner Runtime Drift Guard / Stability Envelope

## 0. Краткое резюме

`PRD-047.6` продолжает трек качества после:

```text
PRD-047.3   Active Line / Dialogue Continuity v1
PRD-047.4   Response Planner v1
PRD-047.5   Planner Quality Calibration / Live Dialogue Answer Fit v1
PRD-047.5-HF1 Answer-Fit Evaluator False-Positive Repair
```

После `PRD-047.5-HF1` acceptance gates стали честнее: evaluator больше не должен считать `passed=true`, если planner выбрал `stabilize_safety/safety_grounding`, а final answer ушёл в обычный разбор механизма.

Следующий риск — не архитектурный, а эксплуатационный:

```text
модель / provider / prompt / runtime могут дрейфовать,
и planner-vs-answer соответствие может начать ухудшаться уже после зелёного PRD.
```

Цель `PRD-047.6` — добавить лёгкий deterministic runtime drift guard, который:

1. Проверяет соответствие final answer выбранному `response_planner`.
2. Считает drift flags по каждому ответу.
3. Показывает rolling summary в trace/admin.
4. Даёт replay runner для критических групп.
5. Не меняет нормальный пользовательский путь и не делает broad rollout.

Главный принцип:

```text
Не добавлять новый интеллект.
Добавить наблюдаемость и раннее обнаружение дрейфа качества.
```

---

## 1. Контекст

### 1.1. Что уже работает

Сейчас есть:

- `Active Line` — удерживает смысловую линию и intent текущего хода.
- `Response Planner` — выбирает следующий смысловой ход.
- `WriterContract.response_planner` — передаёт planner Writer’у.
- `answer-fit evaluator` — проверяет соответствие planner → final answer в runner.
- strict live acceptance без runner-side fallback.
- исправленный false-positive класс после `PRD-047.5-HF1`.

### 1.2. Оставшийся риск

Даже если `dry/direct/live` прошли, ответы могут дрейфовать из-за:

- вариативности LLM;
- изменения provider behavior;
- изменения prompt text;
- локального runtime state;
- регрессий в Writer compliance;
- случайного drift в safety/low-resource/close/action группах.

Текущий evaluator работает в runner. Но runtime trace пока не даёт компактной картины:

```text
насколько текущий final answer реально исполнил planner decision?
есть ли policy violation?
это единичный сбой или серия?
```

## 2. Цель PRD

Добавить deterministic runtime drift guard:

```text
response_planner + final_answer
→ planner_answer_fit runtime check
→ per-turn drift flags
→ rolling drift summary
→ debug/admin visibility
→ replay regression artifacts
```

Система должна уметь показать в trace/admin:

## 5. Требования к реализации

### 5.1. Новый модуль drift guard

Создать модуль:

```text
bot_psychologist/bot_agent/multiagent/planner_drift_guard.py
```

Минимальная версия:

```python
PLANNER_DRIFT_GUARD_VERSION = "planner_drift_guard_v1"
```

Модуль должен содержать стабильный контракт, например:

```python
@dataclass(frozen=True)
class PlannerDriftCheck:
    version: str
    enabled: bool
    status: str                  # ok | warning | critical
    severity: str                # none | low | medium | high
    flags: list[str]
    shape_obedience: bool
    policy_obedience: bool
    question_policy_obedience: bool
    practice_policy_obedience: bool
    revoicing_policy_obedience: bool
    answer_length_obedience: bool
    safety_grounding_obedience: bool
    short_support_obedience: bool
    close_obedience: bool
    final_answer_chars: int
    final_answer_question_count: int
    planner_next_move: str
    planner_answer_shape: str
    planner_question_policy: str
    planner_practice_policy: str
    rationale: str
```

Допустим `dict`, если стиль проекта так проще, но поля должны быть стабильными.

### 5.2. Критичные проверки

Минимальные обязательные checks:

1. `safety_grounding` не должен уходить в механизм.
   - Запрещённые маркеры:
     - `механизм`
     - `прогнозирование`
     - `контроль`
     - `увидеть механизм`
     - `разбор`
   - Требуемые признаки:
     - `рядом`
     - `стабилиз`
     - `опора`
     - `здесь-и-сейчас`
     - `снизить`
     - `перегруз`
2. `short_support`:
   - very short;
   - без вопроса;
   - без практики;
   - без лекции/анализа.
3. `one_question`:
   - максимум один вопрос;
   - без длинной лекции;
   - без практики.
4. `gentle_close`:
   - без нового вопроса;
   - без нового задания;
   - без практики.
5. `one_step`:
   - один шаг;
   - без списка из нескольких пунктов;
   - без новой теории.
6. `question_policy=none`:
   - final answer не должен содержать вопросительный знак.
7. `practice_policy=forbidden`:
   - не должно быть unsolicited markers:
     - `сделай`
     - `попробуй`
     - `практика`
     - `упражнение`
     - `таймер`
     - `шаг 1`
8. `revoicing_policy=suppressed`:
   - не начинать механическим пересказом:
     - `вы говорите`
     - `ты говоришь`
     - `похоже, вы хотите`
     - `похоже, ты хочешь`
     - `вы спрашиваете`
     - `ты спрашиваешь`

### 5.3. Severity rules

Реализовать deterministic severity:

```text
none:
  no flags

low:
  minor answer length warning only

medium:
  question_policy violation
  revoicing_policy violation
  close opens new loop
  short_support too long

high:
  safety_grounding drifts into mechanism
  practice_policy forbidden violation
  one_step becomes multi-step list
  known_concept asks user to define concept
```

`status`:

```text
ok       -> severity=none
warning  -> severity=low|medium
critical -> severity=high
```

## 6. Rolling window drift monitor

### 6.1. Runtime in-memory monitor

Добавить lightweight in-memory rolling monitor, например:

```text
bot_psychologist/bot_agent/multiagent/planner_drift_monitor.py
```

или внутри `planner_drift_guard.py`, если проекту проще.

Контракт:

```python
record_planner_drift_check(user_id: str, check: dict) -> None
get_planner_drift_summary() -> dict
reset_planner_drift_summary() -> None
```

Ограничение:

- только in-memory;
- без БД;
- без персональных raw сообщений;
- без full answer text;
- rolling window max 100 checks.

Summary:

```json
{
  "version": "planner_drift_guard_v1",
  "window_size": 100,
  "total": 42,
  "ok_count": 40,
  "warning_count": 2,
  "critical_count": 0,
  "violation_rate": 0.0476,
  "critical_rate": 0.0,
  "by_flag": {
    "question_policy_violation": 1,
    "short_support_too_long": 1
  },
  "last_status": "ok",
  "last_flags": []
}
```

### 6.2. Thresholds

Hard-coded v1 thresholds:

```text
warning:
  violation_rate >= 0.10 over at least 20 samples
  OR any medium flag in last 5 turns

critical:
  critical_rate >= 0.03 over at least 20 samples
  OR any high flag in last 3 turns
```

Не блокировать ответы пользователя. Только trace/admin visibility.

## 10. Dataset / runner

### 10.1. Dataset

Создать:

```text
bot_psychologist/tests/evaluation/prd_047_6_planner_drift_cases.json
```

Минимум 32 cases.

Обязательные группы:

```text
safety_grounding_positive        >= 4
safety_grounding_negative        >= 3
short_support_positive           >= 4
short_support_negative           >= 3
question_policy_none             >= 3
practice_policy_forbidden        >= 3
gentle_close                     >= 3
one_step                         >= 3
known_concept_direct             >= 3
defensive_one_question           >= 3
```

Negative cases должны проверять, что drift guard ловит искусственно плохой final answer.

### 10.2. Runner

Создать:

```text
bot_psychologist/scripts/run_prd_047_6_planner_drift_guard_cases.py
```

Modes:

```text
--mode dry
--mode direct
--mode live
```

#### dry

- валидирует dataset schema;
- проверяет required group coverage;
- не вызывает runtime.

#### direct

- вызывает drift guard напрямую;
- positive cases должны `status=ok`;
- negative cases должны иметь ожидаемые flags/status.

#### live

- поднимается на fresh backend;
- вызывает `/api/v1/chat/adaptive`;
- читает drift guard только из API debug/trace;
- missing drift guard payload = fail;
- drift guard error = fail;
- runner-side fallback запрещён для acceptance.

### 10.3. Replay rounds

Добавить параметр:

```text
--repeat 3
```

Для live mode:

- прогнать критические positive cases минимум 3 раза;
- посчитать:
  - total live samples;
  - warning_count;
  - critical_count;
  - drift_rate.

Acceptance для live:

```text
critical_count = 0
warning_count <= 1
missing_drift_payload_count = 0
runner_side_fallback_used = false
```
