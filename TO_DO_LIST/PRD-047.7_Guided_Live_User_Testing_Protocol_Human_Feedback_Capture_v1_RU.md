# PRD-047.7 — Guided Live User Testing Protocol / Human Feedback Capture v1

## 0. Краткое резюме

`PRD-047.7` открывает следующий этап после:

```text
PRD-047.3     Active Line / Dialogue Continuity v1
PRD-047.4     Response Planner v1
PRD-047.5     Planner Quality Calibration / Live Dialogue Answer Fit v1
PRD-047.5-HF1 Answer-Fit Evaluator False-Positive Repair
PRD-047.6     Planner Runtime Drift Guard / Stability Envelope
```

После `PRD-047.6` система получила observe-only runtime drift guard: per-turn drift flags, rolling counters, trace/debug/admin visibility и replay artifacts без блокировки `final_answer`.

Теперь следующий главный шаг — не добавлять новый control layer, а подготовить **управляемое живое тестирование пользователем**.

Цель `PRD-047.7`:

```text
Дать владельцу проекта безопасный и структурированный способ тестировать Neo MindBot в живом диалоге,
сохранять human feedback,
связывать каждый отзыв с trace/debug,
и превращать реальные впечатления пользователя в следующие PRD без потери контекста.
```

Главный принцип:

```text
Не усиливать Writer hard constraints.
Не делать нового LLM-агента.
Не менять KB/governance.
Сфокусироваться на live testing protocol, feedback capture, evidence loop и review workflow.
```

## 1. Контекст

### 1.1. Что уже есть

На текущем этапе уже реализованы:

- Active Line / Dialogue Continuity;
- Response Planner;
- Planner Quality Calibration;
- strict answer-fit evaluator;
- false-positive repair;
- observe-only Planner Drift Guard;
- trace/debug/admin visibility;
- Web Admin runtime section;
- replay runners и sanitized artifacts.

### 1.2. Что ещё отсутствует

Система технически стала наблюдаемой, но пока нет удобного процесса:

1. как владельцу проекта тестировать бота как обычный пользователь;
2. какие сценарии проверять;
3. как фиксировать ощущение живости/попадания/зажатости;
4. как сохранить связь между human feedback и trace;
5. как на основе живой обратной связи принять следующий PRD;
6. как не потерять детали после сессии.

## 2. Цель

Создать `Guided Live User Testing Protocol v1`, который включает:

1. Набор guided live сценариев.
2. Формат фиксации human feedback.
3. Runtime/session artifact для каждой тестовой сессии.
4. Связку feedback ↔ trace/debug ↔ planner/active_line/drift_guard.
5. Web/Admin read-only или developer-local endpoint/utility для просмотра session summary.
6. Скрипт/runner для сборки sanitized live testing report.
7. Документацию для владельца проекта: как тестировать, что оценивать, как передавать замечания.
8. Acceptance gates, которые не оценивают “идеальность” ответов, а проверяют, что evidence loop работает.

## 4. Product principles

### 4.1. Human feedback is first-class evidence

Human feedback должен стать не “заметкой в чате”, а структурированным артефактом:

```json
{
  "session_id": "live_test_...",
  "turn_id": "turn_...",
  "user_rating": 4,
  "felt_alive": true,
  "felt_too_rigid": false,
  "felt_too_generic": false,
  "missed_context": false,
  "too_much_practice": false,
  "too_many_questions": false,
  "comment": "Ответ попал, но можно было теплее.",
  "linked_trace": {
    "active_line": "...",
    "response_planner_next_move": "...",
    "answer_shape": "...",
    "planner_drift_status": "ok"
  }
}
```

### 4.2. Feedback does not mutate runtime behavior directly

Feedback не должен автоматически менять prompt, routing или evaluator.

Правильный цикл:

```text
live test
→ feedback artifact
→ review summary
→ audit
→ next PRD / hotfix / no-op decision
```

### 4.3. Preserve Writer freedom

`PRD-047.7` не должен превращать живые замечания в новые жёсткие шаблоны.

Human feedback должен различать:

```text
нарушение политики/формы
vs
живой вкус/тон/стиль
```

### 4.4. Guided, not chaotic

Тестирование должно идти по сценариям, чтобы feedback был сравнимым.

## 5. Required architecture

### 5.1. Новый модуль live testing feedback

Добавить модуль:

```text

```

Минимальный контракт:

```python
LIVE_FEEDBACK_SCHEMA_VERSION = "live_feedback_v1"

class LiveFeedbackRecord:
    schema_version: str
    session_id: str
    turn_id: str
    created_at_utc: str
    scenario_id: str | None
    user_rating: int | None
    felt_alive: bool | None
    felt_understood: bool | None
    felt_too_rigid: bool | None
    felt_too_generic: bool | None
    felt_too_long: bool | None
    felt_too_short: bool | None
    too_much_practice: bool | None
    too_many_questions: bool | None
    missed_context: bool | None
    safety_concern: bool | None
    comment: str
    trace_summary: dict
```

### 5.2. Feedback storage

Для developer-local PRD достаточно файлового storage:

```text
TO_DO_LIST/live_feedback/PRD-047.7/sessions/<session_id>.json
```

Требования:

- хранить только sanitized summary;
- не сохранять raw private full dialogue by default;
- сохранять trace summary, но не raw provider payload;
- не сохранять secrets/API keys;
- session artifacts должны быть пригодны для commit только если sanitized.

### 5.3. Session metadata

Каждая live test session должна иметь:

```json
{
  "session_id": "live_test_2026...",
  "started_at_utc": "...",
  "ended_at_utc": "...",
  "tester_role": "project_owner",
  "runtime_mode": "developer_local",
  "backend_url": "http://127.0.0.1:....",
  "frontend_url": "http://localhost:3000",
  "scenario_set": "guided_live_v1",
  "turns_total": 0,
  "feedback_count": 0,
  "drift_warning_count": 0,
  "drift_critical_count": 0,
  "notes": []
}
```

### 5.4. Trace summary extraction

Для каждого feedback record прикреплять compact trace summary:

```json

    "severity": "...",
    "flags": []
  },
  "writer": {
    "model": "...",
    "fallback_used": false
  }
```

## 6. Guided live scenario set v1

Создать файл:

```text
bot_psychologist/tests/evaluation/prd_047_7_guided_live_scenarios.json
```

Минимум 18 сценариев:

- Ordinary understanding: застревание, откладывание старта, много мыслей без движения.
- Low resource: усталость, пустота, запрос “коротко и без анализа”.
- Soft distress / safety-adjacent: “мне плохо”, “не вывожу”, “паника накрывает”.
- Practice boundary: “без практик”, “дай один шаг”, “почему снова практика”.
- Defensive / I+W-: раздражение на людей, “все тормозят”, “надо всех дожать”.
- Known concept / KB: нейросталкинг, программа несовершенного Я, наблюдение без анализа.
- Close / continuation: “спасибо”, “продолжи без вопросов”, “глубже, но без практики”.

## 11. Trace / privacy requirements

Hard requirements:

1. No raw provider payload.
2. No secrets.
3. No `.env`.
4. No raw private full dialogue committed.
5. Feedback comment may be stored only as user-provided feedback text.
6. Trace summary must be compact and sanitized.
7. If raw answer snippets are stored, cap them:
   - user message preview <= 180 chars;
   - answer preview <= 240 chars.
8. Hash or omit user_id where possible.
