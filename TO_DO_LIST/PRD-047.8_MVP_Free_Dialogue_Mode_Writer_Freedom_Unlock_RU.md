# PRD-047.8 — MVP Free Dialogue Mode / Writer Freedom Unlock v1

## 0. Краткое резюме

После живого теста владельца проекта текущий Neo MindBot признан неудовлетворительным для MVP-диалога.

Проблема не в отсутствии очередного guard-layer, а в том, что runtime стал **слишком ограничительным**:

```text
- Writer зажат правилом “один ответ = один точный следующий ход”;
- Response Planner не умеет long/deep mode;
- low-resource / regulate / short_support слишком легко перехватывают обычный диалог;
- known concept follow-up не раскрывает понятие полноценно;
- explicit request “развернуто / подробно / полное объяснение” не имеет приоритета;
- Writer часто выполняет planner cage вместо человеческого ответа;
- токены экономятся там, где владелец проекта прямо хочет подробный ответ.
```

`PRD-047.8` вводит **developer-local MVP Free Dialogue Mode** — управляемый режим для одного владельца-разработчика, в котором бот может отвечать свободнее, подробнее, глубже и естественнее, не ломая базовую архитектуру.

Главная цель:

```text
Вернуть живой, полезный, развернутый диалог для MVP,
не удаляя архитектуру, но переводя большую часть ограничителей из hard-cage в advisory режим.
```

---

## 1. Основание для PRD

### 1.1. Реальный провал живого теста

В тестовом чате пользователь спросил:

```text
что такое нейросталкинг, и как его можно применять в жизни?
```

Бот ответил слишком кратко:

```text
В нашей внутренней рамке Нейросталкинг — это наблюдение за паттернами, триггерами и автоматическими реакциями, чтобы не сливаться с ними и возвращать себе выбор.
```

После просьбы:

```text
можешь дать более развернутый ответ?
```

бот ушёл в один узкий механизм и задал вопрос, вместо полного объяснения.

После прямой претензии пользователя:

```text
я совершенно не понимаю тебя, ты не дал мне понятного и полного разъяснения...
```

бот снова дал тот же короткий ответ.

После ещё одной просьбы:

```text
ответь более развернуто!
```

бот ответил:

```text
Я рядом. Сейчас не нужно ничего разбирать. Можно просто немного снизить внутреннее давление.
```

Это критический product failure.

### 1.2. Причина по trace/prompt

В прикрепленном LLM prompt видно, что текущий Writer получает слишком много сдерживающих правил:

```text
Один ответ = один точный следующий ход.
Не пытайся за один ответ объяснить всю ситуацию.
Ориентир длины: 4–7 предложений.
Никаких нумерованных списков.
Planner не декоративный: он определяет форму ответа.
Если planner says short_support/safety_grounding/gentle_close, не расширяй в механизм-лекцию.
```

А в конкретном turn runtime передал:

```text
РЕЖИМ ОТВЕТА: regulate
СОСТОЯНИЕ НЕРВНОЙ СИСТЕМЫ: hypo
WRITER MOVE: regulate_first
max_sentences=4
max_questions=0
MUST DO: offer_one_simple_body_action
RESPONSE PLANNER: next_move=give_short_support, answer_shape=short_support, response_depth=very_short
```

То есть система ошибочно решила, что пользователь в low-resource состоянии, хотя пользователь продолжал содержательный разбор и дал уточнение “в разговоре на месте”. Итог: бот вместо продолжения линии снова дал стабилизационный short-support.

## 2. Цель PRD

Создать **MVP Free Dialogue Mode v1**:

```text
developer_local_owner_only режим, который позволяет Writer’у отвечать полно, живо и развернуто,
если пользователь явно просит объяснение, детализацию, продолжение или демонстрирует нормальный исследовательский диалог.
```

В этом режиме:

1. explicit user request имеет приоритет над planner short-support;
2. `развернуто`, `подробно`, `полное объяснение`, `я не понял`, `объясни нормально` включают long/deep answer mode;
3. Planner становится advisory, кроме minimal safety baseline;
4. Writer получает разрешение на объемные ответы и структуру;
5. нумерованные списки разрешены, если они помогают ясности;
6. known concept должен раскрываться полно, а не одной строкой;
7. low-resource должен включаться только по явным признакам, а не по короткому сообщению или ошибочной hypo-классификации;
8. token budget увеличивается;
9. Web Admin получает переключатель режима.

## 3. Non-goals

В рамках PRD-047.8 запрещено:

1. Не удалять полностью safety baseline.
2. Не делать production rollout.
3. Не включать broad rollout.
4. Не включать normal-user activation для внешних пользователей.
5. Не менять KB governance authority fields:
   - `chunk_type`
   - `allowed_use`
   - `safety_flags`
6. Не делать Chroma reindex.
7. Не менять BotDB ingestion.
8. Не коммитить secrets, `.env`, raw provider payload, raw private logs.
9. Не делать необратимый отказ от Active Line / Response Planner / Drift Guard.
10. Не превращать режим в uncontrolled production mode.

Важно: `PRD-047.8` не “ломает безопасность навсегда”. Он вводит **MVP owner developer mode**, потому что текущий пользователь — владелец и разработчик проекта.

## 6. Writer freedom changes

### 6.1. Writer System Prompt v2 for MVP Free Dialogue

Создать отдельный prompt variant, не перетирать старый:

```text
WRITER_SYSTEM_MVP_FREE_DIALOGUE
```

Ключевые правила нового prompt:

```text
Ты — NEO, живой психологический бот-собеседник.
Ты можешь отвечать развернуто, если это нужно для ясности.
Не экономь токены искусственно.
Если пользователь просит объяснить подробно — дай полноценное объяснение.
Если пользователь говорит “я не понял” — сначала исправь ответ и объясни ясно, без защиты системы.
Planner — компас, не клетка.
Сохраняй один главный фокус, но можешь раскрывать его несколькими абзацами.
Разрешены списки, если они делают ответ понятнее.
Разрешены примеры.
Разрешены 2–4 смысловых блока, если пользователь просит развернуто.
Не обрывай known concept одной фразой.
Не уходи в short_support/regulate, если пользователь не просил поддержку или стабилизацию.
```

### 6.2. Remove over-hard constraints in MVP mode

В `mvp_free_dialogue` следующие правила становятся мягкими:

```text
- “Один ответ = один точный следующий ход” -> “один главный фокус, но можно раскрыть его полно”
- “Никаких нумерованных списков” -> списки разрешены для ясности
- “4–7 предложений” -> ориентир, но не потолок
- “Planner не декоративный: он определяет форму ответа” -> planner advisory unless minimal safety
- “не открыть новую теорию” -> не открывать лишнюю теорию, но объяснять концепт полно
```

## 7. Response Planner changes

### 7.1. Add long/deep depth

Расширить:

```python
RESPONSE_DEPTHS = {"very_short", "short", "medium", "long", "deep"}
```

### 7.2. Add answer shapes

Добавить:

```text
concept_explanation_full
expanded_explanation
example_driven_explanation
repair_and_expand
```

### 7.3. Explicit expansion request detector

Добавить detection:

```text
развернуто
подробно
полностью
полное разъяснение
нормально объясни
объясни подробнее
я не понял
не понимаю
что ты предлагаешь
дай полный ответ
```

Если такой marker есть, то в `mvp_free_dialogue`:

```json
{
  "next_move": "answer_known_concept" или "deepen_mechanism",
  "answer_shape": "expanded_explanation",
  "response_depth": "long",
  "question_policy": "none",
  "practice_policy": "allowed_if_explicit",
  "must_include": ["дать ясное объяснение", "дать пример", "показать применение"]
}
```

### 7.4. Low-resource override

В `mvp_free_dialogue` запрещено переводить в `short_support` только из-за:

```text
nervous_state=hypo
короткого ответа пользователя
одного уточнения
```

Low-resource должен включаться только если:

```text
- пользователь явно просит коротко / без анализа / просто поддержи;
- есть явный distress / panic / collapse;
- safety/minimal baseline требует стабилизации.
```

Пример:

```text
User: “в разговоре на месте”
```

Не является low-resource. Это continuation detail.

Expected planner:

```json
{
  "next_move": "continue_active_line",
  "answer_shape": "example_driven_explanation",
  "response_depth": "medium",
  "question_policy": "max_one_if_needed"
}
```

## 8. Knowledge answer / concept continuation

### 8.1. Known concept carry-over

Если предыдущий active concept был `нейросталкинг`, а пользователь пишет:

```text
можешь дать более развернутый ответ?
ответь более развернуто
я не понимаю тебя
что ты предлагаешь то?
```

то система должна продолжать known concept, а не терять concept.

Добавить в context/thread:

```json
"active_concept": "нейросталкинг"
```

и rule:

```text
follow-up expansion request inherits previous active_concept.
```

### 8.2. Required full concept answer structure

Для вопроса:

```text
что такое нейросталкинг, и как его можно применять в жизни?
```

в `mvp_free_dialogue` expected answer должен включать:

1. короткое определение;
2. простое объяснение человеческим языком;
3. чем это НЕ является;
4. как это работает в жизни;
5. пример на бытовой ситуации;
6. как применять без ухода в “упражнение ради упражнения”;
7. если уместно — один мягкий вопрос или предложение продолжить.

Ориентир: 700–1500 слов допустимо, если пользователь просит развернуто.

## 10. Drift Guard / Evaluator behavior

Drift Guard в `mvp_free_dialogue` должен оставаться observe-only.

Изменить thresholds/expectations:

```text
long answer is not drift if user explicitly requested expansion.
numbered list is not violation if clarity improves.
multi-paragraph answer is not violation if single main focus preserved.
```

Не считать violation:

```text
answer_length_long_when_expansion_requested
numbered_list_when_expansion_requested
multi_block_answer_when_concept_explanation_full
```

## 16. Acceptance criteria

PRD принимается только если:

1. В Web Admin можно включить `mvp_free_dialogue`.
2. Writer prompt в этом режиме реально отличается от safe-guided prompt.
3. Explicit expansion request вызывает long/deep answer mode.
4. `нейросталкинг` получает полноценное объяснение, а не одну строку.
5. Follow-up “развернуто” наследует previous active concept.
6. “я не понял / это не ответ” вызывает `repair_and_expand`, а не повторение старой фразы.
7. “в разговоре на месте” продолжает линию, а не уходит в short_support.
8. Нумерованные списки разрешены там, где они помогают ясности.
9. Token budget увеличен для Writer.
10. Minimal safety baseline остаётся.
11. Drift Guard остаётся observe-only.
12. dry/direct/live runner pass.
13. Live samples сохранены и показывают заметно более живой, полный ответ.
14. No-mutation proof зелёный.
15. Commit + push в main.
