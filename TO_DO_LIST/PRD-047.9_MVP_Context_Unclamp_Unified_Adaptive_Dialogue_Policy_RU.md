# PRD-047.9 — MVP Free Dialogue Context Unclamp + Unified Adaptive Dialogue Policy v1

## 0. Краткое резюме

`PRD-047.8` добавил `mvp_free_dialogue` как developer-local runtime profile и сделал первый шаг к развязыванию Writer’а. Живой тест владельца проекта показал улучшение, но не решил ключевую проблему: Writer всё ещё получает слишком противоречивый и частично обрезанный prompt.

Критичный пример из live test:

```text
Пользователь: "скажи, а в нейросталкинге какие практики предлагаются чтобы это видеть"
```

В Writer prompt при активном `mvp_free_dialogue` одновременно попали:

```text
SYSTEM: отвечай полно, глубоко и структурно
DIAGNOSTIC CARD: do_not_over_explain, ask_at_most_one_question
WRITER MOVE: clarify_one_point, max_sentences=5
ACTIVE LINE: ask_for_practice, дать короткую практику
RESPONSE PLANNER: give_direct_step, one_step, response_depth=short
CONTEXT: recent turns обрезаны до 2000 символов и потеряли самые свежие повороты
KB: semantic hit нерелевантен текущему вопросу о практиках
```

В результате Writer ответил не как свободный MVP-собеседник, а как исполнитель "одного микрошагa".

Цель `PRD-047.9` — не добавлять новый guard, а **разжать контекст и объединить режимы в единую адаптивную dialogue policy**, где `mvp_free_dialogue` является preset’ом параметров, а не параллельной архитектурой.

---

## 1. Контекст

### 1.1. Что PRD-047.8 сделал правильно

- Создал runtime profile `safe_guided | mvp_free_dialogue`.
- Оставил один orchestrator / writer / planner path.
- Добавил `dialogue_policy` в orchestrator и WriterContract.
- Добавил MVP Writer system prompt.
- Поднял writer max tokens до `>=2500` для MVP profile.
- Добавил active concept continuation.
- Добавил planner depths `long/deep`.
- Добавил answer shapes для expanded explanation.
- Добавил Web Admin переключатель.

### 1.2. Что PRD-047.8 не закрыл

1. Live acceptance был `blocked`, а не passed.
2. Writer prompt всё ещё содержит старые ограничения из Diagnostic Card, Writer Move, Active Line и Response Planner.
3. Conversation context режется через `[:2000]`, то есть сохраняется начало блока и теряются самые свежие ходы.
4. Knowledge routing не распознаёт вопрос "какие практики в нейросталкинге" как concept/practice overview.
5. Active Line может оставаться в старой теме и тащить stale intent.
6. Admin показывает `mvp_free_dialogue`, но соседние effective cards выглядят противоречиво: Writer Freedom может оставаться `guided`, хотя runtime фактически должен быть high-autonomy.
7. Writer still behaves like "one next move executor", not like adaptive human-like conversational agent in MVP mode.

---

## 2. Цель

Сделать `mvp_free_dialogue` реально свободным и полезным для creator-local MVP testing:

```text
User asks for explanation / overview / practices / examples
→ system recognizes this as an information-rich answer request
→ Writer receives enough recent context and relevant concept/practice grounding
→ Planner becomes advisory, not restrictive
→ final answer may be long, structured, example-rich and practical
```

При этом:

```text
No new LLM agent.
No duplicated runtime path.
No production rollout.
Minimal safety baseline remains.
Trace/Admin remain observable.
```

## 4. Product decision

### 4.1. Runtime profiles are presets, not branches

Ввести/зафиксировать единую модель:

```text
dialogue_policy = единый адаптивный слой
safe_guided = preset строгих параметров
mvp_free_dialogue = preset свободных developer-local параметров
```

Не должно быть архитектуры:

```text
if safe_guided: один бот
if mvp_free_dialogue: другой бот
```

Должно быть:

```json
{
  "dialogue_policy": {
    "profile": "mvp_free_dialogue",
    "writer_autonomy": "high",
    "planner_authority": "advisory",
    "diagnostic_card_authority": "advisory",
    "writer_move_authority": "advisory",
    "answer_depth_default": "medium",
    "answer_depth_when_requested": "long_or_deep",
    "allow_numbered_lists": true,
    "allow_examples": true,
    "allow_multi_paragraph": true,
    "context_budget_chars": 6000,
    "minimal_safety_baseline": true
  }
}
```

### 5.2. Explicit request wins in MVP profile

В `mvp_free_dialogue` явный запрос пользователя должен иметь приоритет над старыми ограничениями:

| User request | Expected behavior |
|---|---|
| "развернуто", "подробно", "полный ответ" | long/deep answer |
| "я не понял", "объясни нормально" | repair_and_expand |
| "какие практики предлагаются" | practice overview/catalog, not one microstep |
| "примеры" | examples allowed |
| "по пунктам" | numbered list allowed |
| "что такое X" | definition + explanation + example + application |
| "как применять" | application examples, not only one action |

### 5.3. Practice overview vs direct practice

Добавить различение:

```text
ask_for_practice_catalog
ask_for_practice_explanation
ask_for_direct_microstep
```

Фраза:

```text
"какие практики предлагаются"
```

не равна:

```text
"дай мне один шаг прямо сейчас"
```

Expected planner:

```json
{
  "next_move": "answer_practice_overview",
  "answer_shape": "practice_catalog_explanation",
  "response_depth": "long",
  "practice_policy": "overview_allowed",
  "question_policy": "optional_none",
  "must_avoid": ["не сводить к одному микрошагу"]
}
```

## 6. Prompt/context unclamp

### 6.1. Fix conversation context truncation

В `writer_agent.py` сейчас user prompt получает:

```python
conversation_context=(ctx["conversation_context"] or "нет")[:2000]
```

Это опасно, потому что если context ordered oldest→newest, Writer теряет последние ходы.

Нужно заменить на recency-preserving formatter:

```python
format_conversation_context_for_writer(
    conversation_context: str,
    profile: str,
    budget_chars: int,
) -> str
```

Rules:

```text
safe_guided: budget 2500–3000 chars
mvp_free_dialogue: budget 6000–9000 chars
preserve last turns first
never cut inside the last assistant/user turn pair if possible
mark truncation explicitly: [older context omitted]
```

Acceptance:

```text
Writer prompt for the test chat must include:
- question about neurostalking;
- request for more detailed answer;
- complaint "не понимаю";
- example with boss;
- "блин! а что делать то";
- "не понял, подробнее";
- current query about neurostalking practices.
```

### 6.2. Separate "recent full turns" from semantic memory

Writer should receive structured blocks:

```text
RECENT EXACT TURNS:
last 6–10 turns, recency-preserved

THREAD SUMMARY:
compact summary

RELEVANT MEMORY:
semantic hits

KB / CONCEPT GROUNDING:
concept chunks or generated internal grounding
```

Do not mash all into one 2000-char field.

### 6.3. Prompt canvas must show truncation clearly

In full LLM canvas, show:

```text
context_budget_chars=...
context_truncated=true/false
preserved_recent_turns_count=...
older_context_omitted_chars=...
```

## 7. Knowledge/RAG fix for concept-practice questions

### 7.1. Knowledge routing

Update `knowledge_answer_routing_guard` so that:

```text
"нейросталкинг какие практики"
"какие практики предлагаются"
"как это видеть"
"как применять нейросталкинг"
```

sets:

```json
{
  "knowledge_answer": {
    "needed": true,
    "concept": "нейросталкинг",
    "should_answer_directly": true,
    "answer_type": "practice_overview",
    "kb_grounding_available": true
  }
}
```

### 7.2. Retrieval quality

If user asks for practices, prefer chunks with:

```text
chunk_type=practice
lens_family=neurostalking
allowed_use=practice_suggestion or writer_context
```

If no good chunks available, Writer may answer from philosophy kernel and conversation context, but must say in plain language:

```text
"В нашей рамке это можно разложить на несколько практических направлений..."
```

not invent formal named practices.

### 7.3. Semantic hit budget

In MVP profile:

```text
semantic_hits for Writer: up to 4
chars per hit: 700–1000
prefer concept/practice relevant hits
```

Current `hits[:2]` and `h[:300]` is too small for concept/practice explanation.

## 8. Active Line and Planner staleness repair

### 8.1. Detect stale active line

If current message is a concept/practice overview request and active_line refers to unrelated older task:

```text
active_line="застревание на старте рабочей задачи..."
current="какие практики в нейросталкинге..."
```

then set:

```json
{
  "active_line_stale": true,
  "continuity_mode": "concept_branch_from_current_line",
  "active_line_authority": "low"
}
```

### 8.2. Planner should not force one_step for overview

For `mvp_free_dialogue`, if current request contains:

```text
какие практики
какие способы
какие варианты
расскажи подробнее
```

planner must not choose:

```text
answer_shape=one_step
response_depth=short
must_avoid=без списка
```

unless the user explicitly says:

```text
дай один шаг
что сделать прямо сейчас
один конкретный микро-шаг
```

## 9. Writer prompt changes

### 9.1. MVP prompt should include authority order

Update `WRITER_SYSTEM_MVP_FREE_DIALOGUE`:

```text
В mvp_free_dialogue:
1. Minimal safety baseline выше всего.
2. Явный запрос пользователя на глубину/обзор/примеры имеет высокий приоритет.
3. Planner / Diagnostic Card / Writer Move — ориентиры, а не ограничения.
4. Если они требуют "max_sentences=5", "do_not_over_explain", "one_step", но пользователь просит обзор или подробное объяснение — раскрывай ответ полно.
5. Не превращай обзорный вопрос в один микрошаг.
6. Если вопрос о концепте — дай определение, смысл, пример, применение.
7. Если вопрос о практиках — дай несколько направлений практик, объясни зачем каждая нужна и покажи пример.
```

### 9.2. Remove or soften conflicting user prompt fields

In MVP profile, transform old constraints before rendering Writer prompt:

```text
DIAGNOSTIC do_not_over_explain -> advisory unless user requested short
WRITER_MOVE max_sentences=5 -> ignore if answer_depth long/deep or overview request
MUST_DO ask_one_specific_question -> optional
RESPONSE_PLANNER one_step -> only if explicit direct microstep request
```

Add visible prompt block:

```text
MVP FREE DIALOGUE OVERRIDES:
- explicit_user_request_wins=true
- old_max_sentence_constraints_softened=true
- planner_advisory=true
- overview_questions_allow_lists=true
- target_answer_depth=long
```
