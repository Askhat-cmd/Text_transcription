# PRD-046.0.6-HF1 POLICY FIX REPORT

## Goal
Закрыть утечку `internal_only` в non-safety top-k retrieval после PRD-046.0.6 без деградации governance/semantic качества.

## Implemented
1. Added API-side retrieval policy module:
- `api/retrieval_policy.py`
- `is_safety_context_query(...)`
- `apply_retrieval_governance_policy(...)`

2. Integrated policy into `/api/query`:
- policy applied after scoring/rerank and before final top-k.
- non-safety context: `internal_only` hits excluded from returned top-k.
- safety context: `internal_only` can remain.

3. Increased raw fetch size before policy filtering:
- `raw_fetch_k = max(pre_filter_k, top_k*3, top_k+10)`.

4. Preserved governance passthrough and safety fields in API response.

## Policy Trace Contract
- `policy_version`: `retrieval_governance_safety_v1`
- `safety_context`: bool
- `raw_hits_count`
- `internal_only_filtered_count`
- `returned_hits_count`
- `filter_reasons`

## Validation Outcome
- baseline `internal_only_unsafe_exposure_count=4` -> HF1 `0`
- baseline weak cases (`KR-002`, `KR-013`, `KR-022`) no longer fail on internal_only leakage
- no raw leak introduced
- semantic threshold preserved (`top5=1.0`)
