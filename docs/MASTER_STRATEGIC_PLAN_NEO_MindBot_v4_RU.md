# ЧАСТЬ 0 — КАК ЧИТАТЬ ЭТОТ ДОКУМЕНТ

Новый архитектор, открывший проект впервые, читает по порядку: §1 (что за
проект), §2 (незыблемые принципы), §3 (роли и как устроена работа), §4
(где мы сейчас — единственное место, которое устаревает и обновляется),
затем ту Эпоху, в которой находится проект.

Проект движется через **две эпохи**, строго последовательно:

```text
ЭПОХА 1 — Консолидация (архитектурная чистота). Идёт сейчас.
          Цель: единая система без исторического мусора.
          Механизм: линейная серия PRD с жёстким Definition of Done.

ЭПОХА 2 — "Мозги" (интеллект бота). Не начата.
          Цель: retrieval/восприятие/голос по законам Fable-корпуса.
          Механизм: 5 тактов, стартует только после закрытия Эпохи 1
          И ратификации корпуса законов владельцем.
```

Между эпохами — **шлюз** (§8): два условия, оба обязательны, оба проверяемы.
Ни один архитектор не имеет права начать Эпоху 2, пока шлюз не пройден.

Слово **"пилот"** в этом проекте означает **внутреннюю готовность системы**
(целостность, чистая архитектура, модульность, документация), а НЕ запуск
на внешних людях. Единственный пользователь — владелец-разработчик, и это
надолго по сознательному решению. Safety, работа с реальными пользователями,
продвижение — отдельная, ещё не спланированная фаза за пределами обеих эпох.

# ЧАСТЬ 1 — ЧТО ЗА ПРОЕКТ

Neo MindBot — русскоязычный психологический бот-помощник с pipeline-
архитектурой поверх governed RAG:

```text
User turn -> State Analyzer -> Thread Manager -> Memory Retrieval
-> Context Assembly -> Diagnostic Card -> Writer Move Compliance
-> Writer -> Validator -> Memory Update
```

Скрытая философия бота (первоисточник, он же БД): корпус "Кузница Духа" и
"chuvstvo_do_slova". Идея — человек не сломан, программа не есть он сам,
страдание часто защитный механизм, практика — приглашение, не плеть. Эта
оптика должна жить как мягкая суть Writer'а, но по закону (см. Эпоха 2,
такт 3в), а не как отдельный слой поверх. Суждение об оптике корпуса и
утверждение идентичности Writer'а — не делегируемая человеческая точка
владельца (вердикт R05 §6 / R06); механическая пересборка идентичности в
такте 3(в) выполняется Executor'ом под этим суждением. [Если владелец решает
писать текст идентичности лично — это его решение сверх вердикта,
зафиксировать при ратификации.]

История: 755+ коммитов, несколько сотен PRD. Проект прошёл product-стадию
(latest-turn authority, Writer как финальный автор, БД как скрытая
компетенция) и вышел на frozen baseline (PRD-047.37). Известная боль
владельца — переусложнение через сотни PRD; выход из него и есть Эпоха 1.

# ЧАСТЬ 3 — РОЛИ И МЕХАНИКА РАБОТЫ

```text
Architect (chat-сессия, Claude/др.) — читает evidence (коммиты, отчёты,
    trace), решает следующий PRD, пишет PRD-документы, обновляет §4.
Executor (IDE-агент, Claude Code)   — читает PRD целиком, создаёт TASK_LIST,
    реализует, тестирует, коммитит, пишет отчёты в TO_DO_LIST/logs/PRD-XXX/.
Owner (владелец)                    — приносит Architect'у evidence после
    цикла, утверждает деструктивные шаги, ратифицирует законы Эпохи 2.
```

**Двусторонняя связь Architect <-> Executor** держится на трёх артефактах,
все в git: `docs/PROJECT_STATE.md`, `docs/PRD_INDEX.md`, и `.md`-отчёты в
`TO_DO_LIST/logs/PRD-XXX/`. Сырые артефакты (raw json, screenshots,
live_turn_exports) — НЕ в git (untracked с PRD-047.39; новые кладутся в
`TO_DO_LIST/logs/PRD-XXX/raw/`, эта подпапка в .gitignore). Architect
восстанавливает контекст из PROJECT_STATE + отчётов, не из сырья.

**Evidence hierarchy — что приносить Architect'у в начале сессии:**
```text
1. Commit hash после PRD (HEAD == origin/main, clean tree).
2. Путь к TO_DO_LIST/logs/PRD-XXX/ и список реально появившихся файлов.
3. next_recommendation.md последнего PRD.
4. Любые BLOCKER/WARNING из acceptance decision.
5. Для Эпохи 1 Phase B — diff по затронутым файлам.
```
Architect не принимает "всё прошло" на слово — только через артефакты.

```text
   сторон; рост словаря — только контрактным событием.
И2 знание — материал, не директива; императив разрешён только жёстким
   границам закрытого списка; Writer вправе игнорировать всё.
И3 каждое поле/метаданное имеет названного потребителя; write-only нет.
И4 рост схемы/словаря/слоёв — только против записанного провала-фикстуры.
И5 первоисточник всегда достижим: производное несёт путь к фрагменту
   корпуса; выборочная сверка — человеком.
В1 одна рамка, один производитель; вторых судей о человеке нет.
В2 рамка — уценённая гипотеза: confidence + reason-цитата слов человека;
   latest turn опровергает без переговоров; без свежего подтверждения —
   структурная уценка.
В3 словари восприятия закрыты, версионируются вместе со словарём механизмов.
В4 рамка декларативна и удаляема: без неё ответ валиден, лишь консервативнее.
Л1 лаборатория свидетельствует, не решает; ничего не блокирует сама.
Л2 видимый ответ — артефакт; согласованность проверяется по границе
   ДОСТАВКИ, не использования (Н1).
Л3 свидетельство несёт сорт [внутр./внешн.] и происхождение (живой провал /
   придуманный сценарий); сорт повышает только человек.
Л4 мера успеха evidence-машины — цена человеческого взгляда, не число проверок.
Л5 контракт — о поведении на видимой границе; ноль имён внутренних стадий.
Г1 механизменная грамматика лицензируется рамочной гипотезой этого хода;
   без лицензии — грамматика присутствия (обязательство идентичности Writer'а).
Г2 живость = след слов, введённых человеком самим, не эха бота (Н2);
   реплика, переносимая в любой диалог, — штамп.
Г3 у живости нет эталонного стиля; она определяется только негативно.
Статус Writer Move Compliance (R04 §4): либо читает рамку, либо жёсткая
граница закрытого списка — третьего статуса не существует.

3 исправленные нестыковки (R07): Н1 (согласованность по границе доставки,
не использования — защищает право Writer'а игнорировать знание),
Н2 (Г2 наследует происхожденческую оговорку Р7), Н3 (лицензия Г1
асимметрична, В4 цела).
```

## 7.3 Дорожная карта — 5 тактов (R07 §3)

```text
Такт 0 (во время Эпохи 1): только 2 акта (см. §6). Всё прочее ждёт.

Такт 1 — Контрактный акт. ОДИН PRD, точка слияния всей серии.
  Словарь механизмов v1 + словари восприятия + контракт рамки
  (поля->3 потребителя) + контракт пакета (4 отсека, бюджет дефицита)
  + Behavior Contract (страница-две на видимой границе, владелец
  утверждает построчно, ни одного имени внутренней стадии). Ноль кода.

Такт 2 — Baseline. Полный параллелизм. 3 вида фикстур руками владельца:
  (а) retrieval из живых чатов, (б) эталонные рамки на диалогах,
  (в) ручная лаборатория 3-5 сценариев в Web Chat. Продукт: измеренная
  отправная точка (защита от П20 — сначала измерь).

Такт 3 — Три переплавки.
  (а) Пилотная партия фабрики: один механизменный кластер в новую схему,
      ревью владельца, похороны general_text-предков в той же партии.
  (б) Переплавка восприятия: один производитель рамки, публичные выходы
      тройки (State Analyzer / Thread Manager / Diagnostic Card — см.
      pipeline Части 1) хоронятся, preservation зелёный.
  (в) Пересборка идентичности Writer'а: снятие осадка writer-контракта
      (это же плановая консолидация god-file), Г-обязательства внутрь.
  ПОРЯДОК: (в) СТРОГО после (б) — Г1 ссылается на рамку, которой до (б)
  нет. (а) и (б) при одном владельце безопаснее последовательно.

Такт 4 — Стыковка. Вторая точка слияния. Формирование запроса читает
  рамку и говорит словарём. Строго после (а) и (б). Здесь решается
  центральная ставка Г-1.

Такт 5 — Рутина контура обучения: цикл провал->фикстура->партия/правка
  ->замер. Сквозной прогонщик 3-5 сценариев только теперь.

ПРАВИЛО: полный reindex не является шагом ни на одном такте.
```

```text
    Решается тактами 3(а)+4. Нет измеримого выигрыша — центр дизайна R03
```

## 7.5 Определение прибытия Эпохи 2 (чтобы у эры была дверь, П13)

```text
Эра «мозгов» реализована, когда одновременно верны пять СВОЙСТВ (не шагов):
1. Обе стороны говорят словарём, и это видно на фикстурах против baseline.
2. Рамку производит один производитель; вторых судей о человеке — ноль.
3. Пакет живёт по контракту; идентичность Writer'а собрана без осадка,
   с Г-обязательствами внутри.
4. Контур обучения крутится как рутина, и фикстуры РЕАЛЬНО меняют
   приоритеты партий (если ни один приоритет не изменился — контур
   декоративен).
5. Сквозные фикстуры ловят расхождения доставки.
В момент объявления эры завершённой записывается условие входа в следующую
фазу (П13) — у любой отложенной фазы, включая safety, должна быть дверь.
Дальше — не новая эра чистки, а установившийся цикл.
```

# ЧАСТЬ 8 — ЕДИНЫЙ OWNER-ЧЕК-ЛИСТ (канон R07, предмет ратификации)

Владелец задаёт себе при каждом крупном решении в любой эпохе. Ответ
дольше десяти секунд — сам по себе найденный паттерн. Это канонический
чек-лист R07 дословно — он же предмет ратификации; три «архитектурных»
вопроса, которые могли бы сюда попасть, живут в теле плана (постоянный
модуль — конвенция Части 2; имя стадии в контракте — Л5 в такте 1 и §7.2;
живость числом — Г-карта в §7.1), не в каноне.

```text
ФОРМА:
1. Я даю этому имя, папку или модуль? Может ли это жить как данные+контракт?
2. Это контракт или контент? Контракт -> PRD; контент -> данные с журналом.
3. Если это «новый способ» — что хоронится в том же акте? (П1/П2)
ВЛАСТЬ:
4. Может ли Writer это проигнорировать, и останется ли ответ валидным?
5. Не появился ли второй судья о человеке — или читатель чужого
   промежуточного выхода?
6. Есть ли место, где слово пользователя, вес на чанке или список фраз
   решает напрямую? (rule engine во всех формах)
РОСТ:
7. Какой записанный провал это закрывает? Нет фикстуры — нет изменения.
8. Кто названный потребитель, и когда/от чего это умрёт?
9. Заплатил бы я за эту работу подрядчику по рыночной ставке?
ИСТИНА:
10. Это свидетельство — [внутр.] или [внешн.]? Что из отчётов я проверил
    руками на этой неделе?
11. Это измеримо на артефакте до Writer'а — или только по ощущению ответа?
12. Что в этой проверке способно меня удивить? Реплика/фикстура привязана
    к тому, что человек ввёл сам, — или к тому, чему его научил бот?
Еженедельная строка (П20): что система теперь умеет / что теперь нельзя
сломать — чего не было неделю назад. Пусто две недели подряд — остановка.
```

# ЧАСТЬ 9 — ДОЛГОСРОЧНЫЕ ПРЕДОСТЕРЕЖЕНИЯ (из R07, часть 2)

Не блокируют работу, но должны звучать в нужный момент:

```text
Б. "Вода из одного колодца": весь корпус учится на одном человеке
   (владелец = автор знаний + единственный собеседник + разметчик +
   читатель свидетельств). У Эпохи 2 есть ПОТОЛОК, заданный энтропией
   входа, не архитектурой. КОГДА метрики тактов 4-5 выйдут на плато —
   читать это в ПЕРВУЮ очередь как "потолок входа достигнут", не "нужен
   ещё закон/партия/поле". Строка в PROJECT_STATE (внести при такте 1):
   "плато внутренних метрик — кандидат-сигнал достигнутого потолка входа,
   не дефицита законов."

В. Формат внешнего аудита создаёт слепоту: внешний взгляд видит проект
   только глазами того, кого аудирует. Практика на будущее: иногда
   давать внешнему взгляду сырьё БЕЗ рамки вопроса ("вот материал, что
   видишь?"). Неоформленный аудит ловит то, что оформленный не может.

Г. Метрика вне всех законов: отношение часов строительства к часам
   использования. Раз в неделю — час без трейса и без фикстуры после,
   просто разговор с ботом как с ботом. Эра "мозгов" состоится не когда
   позеленеют фикстуры такта 5, а когда этот час перестанет быть
   дисциплиной. Это не пункт-задача — это напоминание о назначении.
```

# ЧАСТЬ 11 — ОДНОЙ ФРАЗОЙ

Проект идёт через две эпохи: сначала убрать исторический мусор до чистой
единой системы (Эпоха 1, линейная серия PRD с жёстким Definition of Done),
затем — только после шлюза с обязательной ратификацией законов — собрать
"мозги" бота по проверенному корпусу инвариантов (Эпоха 2, пять тактов).
Всё, что защищает от переусложнения, уже встроено: заранее зафиксированные
границы завершения, запрет на именованные органы и постоянные модули, и
единый чек-лист из двенадцати вопросов, ловящий уход в тупик до того, как
он стоит недели.
