## 0. КОНТЕКСТ И ФИЛОСОФИЯ ПРОЕКТА

### 0.1 Что такое Neo MindBot

Neo MindBot — это AI-сопровождающий в Telegram, который помогает пользователю работать с внутренними состояниями: мыслями, эмоциями, паттернами поведения. Бот не заменяет психотерапевта — он создаёт пространство честного самоисследования и предлагает структурированные практики.

**Методологическая база:** ACT, IFS, Polyvagal Theory, AQAL, Metacognitive Therapy.

### 0.2 Ключевые принципы

1. **Безопасность прежде всего** — блок `{{{AA}}}` имеет абсолютный приоритет над всем.
2. **Качество ответов важнее скорости** — v1 оптимизируется под качество диалога, не под latency.
3. **Управляемость** — каждый параметр бота доступен через admin-панель без деплоя.
4. **Универсальность** — бот работает в нескольких режимах (коучинг, информационный, кризисный), не только как психологический ассистент.
5. **Эволюционная миграция** — всё внедряется через feature-флаги поверх текущего `bot_psychologist`, без сломанного деплоя.

### 0.3 Чем Neo MindBot отличается от бота «Х»

| Параметр | Бот «Х» | Neo MindBot |
| :-- | :-- | :-- |
| Режимы работы | Только коучинг/терапия | Коучинг + Информационный + Кризисный |
| Управление | Фиксированные конфиги | Полная admin-панель (все промты, параметры, флаги) |
| SD-классификация | Не используется | Убрана из маршрутизации, метки в БД как пассивные данные |
| База знаний | Один источник | `Bot_data_base` по HTTP API (без fallback на `voice_bot_pipeline`) |
| Аудитория | Зелёный мем+ (неявно) | Любой осознанный пользователь без SD-фильтрации |

### 1.2 Целевой pipeline

```
User Input
    │
    ▼
[Feature Flag Check] ── NEO_MINDBOT_ENABLED? ──► [Legacy Pipeline (старый код)]
    │ YES
    ▼
[Mode Detector] ── Информационный / Коучинг / Кризисный
    │
    ▼
[StateClassifier] ── 1 LLM-вызов → 5 диагностических меток
    │
    ▼
[HybridQueryBuilder] ── строит запрос из меток (без SD-фильтра)
    │
    ▼
[Retriever → Bot_data_base HTTP API]
    │
    ▼
[VoyageReranker]
    │
    ▼
[DecisionGate] ── 1 LLM-вызов (финальный, после rerank)
    │
    ▼
[ResponseGenerator] ── применяет иерархию промтов + Output Layer
    │
    ▼
[MemoryUpdater] ── обновляет last_state_snapshot + current_summary в БД
    │
    ▼
Telegram HTML Response → User
```

## 2. ИЕРАРХИЯ ПРОМТОВ

### 2.1 Структура (порядок приоритетов)

```
{{{AA}}} SAFETY LAYER          ← Абсолютный приоритет. Никогда не переопределяется.
    ↓
{{{A}}}  SEASONAL FILTER       ← Весенний фильтр: постоянный префикс, адаптируется к состоянию.
    ↓
CORE     IDENTITY & MISSION    ← Роль (зеркало/навигатор/катализатор), миссия, границы.
    ↓
DIAG     DIAGNOSTIC ALGORITHM  ← 5 параметров диагностики → метки состояния.
    ↓
METHOD   REFLECTIVE METHOD      ← 4 шага (только в режиме коучинга).
    ↓
PROC     PROCEDURAL SCRIPTS     ← if-then workflow (онбординг, кризис, практика, трек).
    ↓
OUTPUT   OUTPUT LAYER           ← Telegram HTML, Anti-Bland Rule, Emoji Rule, Pre-Send Checklist.
```


### 2.2 Блок `{{{AA}}}` — Safety Layer

**Абсолютные запреты (нельзя нарушать ни при каких условиях):**

- Медицинские диагнозы и назначение лечения
- Финансовые советы
- Конкретные советы в ситуациях суицидального риска (только кризисный протокол)
- Оценка действий третьих лиц («он прав/неправ»)
- Имитация живого человека

**Кризисный протокол (при маркерах риска):**

1. Признать боль без анализа
2. Предложить горячую линию (номер телефона доверия)
3. Предложить связаться с близким человеком
4. Не продолжать стандартный диалог до стабилизации

### 2.3 Блок `{{{A}}}` — Весенний фильтр

- Постоянный префикс к каждому промту
- Адаптируется к состоянию НС: при HYPO/HYPER — сверхминималистичное проявление
- Ключевые принципы: анти-токсичная позитивность, присутствие без оптимизма, тепло без фальши
- Управляется через admin-панель (сезонный параметр: spring/summer/autumn/winter)


### 2.4 Блок DIAG — 5 параметров диагностики

```
P1: Состояние НС        HYPER / WINDOW / HYPO
P2: Функция запроса     Прожить / Понять / Решение / Подтверждение / Исследование / Просто контакт
P3: Дистанция к опыту   Реактивный / Наблюдатель / Мета-позиция
P4: Ведущая часть IFS   Критик / Защитник / Избегающий / Раненый ребёнок / Исследователь / Self
P5: Активный квадрант   I (внутреннее-Я) / IT (поведение) / WE (отношения) / ITS (контекст)
```

**Дополнительный параметр (пассивный):**

- `core_theme` — ключевая тема сессии (страх провала, отношения, выгорание и т.д.)


### 2.5 Блок METHOD — Reflective Method (4 шага)

Применяется **ТОЛЬКО** при функциях запроса: Прожить / Понять / Исследование / Просто контакт.

```
Шаг 1: Recognition    — Что происходит? Валидация без анализа.
Шаг 2: Decentering    — ACT-дефьюзия. Отделить наблюдателя от переживания.
Шаг 3: Perspective    — IFS/AQAL: взгляд с другой позиции. Один точный вопрос.
Шаг 4: Micro-action   — CTA: минимальное действие ≤10 минут. Не навязывать.
```


### 2.6 Режимы работы бота

| Режим | Триггеры | Что происходит |
| :-- | :-- | :-- |
| **Коучинг** | Личный опыт, «я чувствую», «у меня проблема» | Диагностика → Reflective Method → Практика |
| **Информационный** | «что такое X», «объясни», «расскажи про» | Экспертный ответ без диагностики и практик |
| **Кризисный** | Маркеры суицида, острой паники | Блок `{{{AA}}}` → кризисный протокол |

```json
    "active_quadrant": "i | it | we | its",

  },
  "engagement": {
    "active_track": null,
    "last_practice_offered": null,
    "last_practice_channel": "body | thinking | action | null",
    "insights_log": []
  },
  "system_flags": {
    "interaction_mode": "coaching | informational | crisis",
    "seasonal_filter": "spring",
    "communication_mode": "empathic | technical | minimal",
    "requires_context_continuity": true
  }

```


### 3.3 Схема `current_summary` (текст для промта)

Создаётся Summarizer Agent по шаблону:

```
1. Роль и цель пользователя: [кто, зачем пришёл]
2. Ключевые темы сессии: [3-5 тем]
3. Последний активный запрос: [на чём остановились]
4. Важные инсайты и договорённости: [что понял пользователь]
5. Тон и стиль общения: [характер диалога]
```

**Правила обновления:**

- `last_state_snapshot` → обновляется после каждого ответа бота
- `current_summary` → обновляется при завершении сессии (таймаут 30 мин) или каждые N=10 ходов
- `dialog_cache` → скользящее окно, 20-30 сообщений

- Просмотр `last_state_snapshot` любого пользователя
- Ручная коррекция `current_summary`
- Сброс `active_track`
- Настройка N (порог обновления summary)
- Настройка размера `dialog_cache`

## 4. БИБЛИОТЕКА ПРАКТИК

### 4.1 Структура записи (practices_db.json)

```json
{
  "id": "grounding_5_4_3_2_1",
  "name": "5-4-3-2-1 Заземление",
  "category": "body",
  "scientific_basis": "Polyvagal Theory, Somatic Experiencing",
  "triggers": ["hyperarousal", "panic", "anxiety", "window_of_tolerance"],
  "nervous_system_states": ["hyper", "window"],
  "time_required": 3,
  "instructions": "...",
  "micro_question": "Что изменилось в теле после?",
  "closure_question": "Какое слово описывает твоё состояние сейчас?",
  "channel": "body"
}
```


### 4.2 Каналы практик

| Канал | Основа | Примеры |
| :-- | :-- | :-- |
| **BODY** | Polyvagal, Somatic Experiencing | 5-4-3-2-1, сканирование тела, дыхание 4-7-8 |
| **THINKING** | ACT, CBT, MCT | Когнитивная дефьюзия, факт vs интерпретация, STOP-техника |
| **ACTION** | BJ Fogg Tiny Habits, CBT | Поведенческий эксперимент, микро-шаг, ритуал паузы |

### 4.3 Алгоритм подбора практики

```
Фильтр 1 (обязательный):  nervous_system_state совпадает с triggers
Фильтр 2 (обязательный):  channel ≠ last_practice_channel (запрет повтора канала)
Фильтр 3 (приоритетный):  dominant_part совпадает с triggers
Фильтр 4 (вторичный):     request_function не противоречит каналу

Ранжирование:
  → Точное совпадение dominant_part > Совпадение core_theme > Время (меньше = лучше)
  → Если равноценны: проверить историю (не повторять недавние) → random как last resort
```


### 4.4 Динамические треки (7 дней)

**Принципы сборки:**

- Чередование каналов: BODY → THINKING → ACTION → BODY...
- Прогрессия сложности: наблюдение (день 1) → интеграция (день 7)
- Привязка к `core_theme` пользователя

**Переход к следующему дню — Feedback-Based:**

```json
{
  "active_track": {
    "id": "track_anxiety_7d",
    "current_day": 3,
    "day_completed": [true, true, false],
    "current_practice_feedback": null
  }
}
```

Пока `current_practice_feedback == null` — день 4 не открывается.

**Таймаут выпадения:** >3 дней без активности → рестарт с диагностики.

### 4.5 Управление практиками через admin-панель

- CRUD для `practices_db.json`
- Редактирование триггеров и каналов
- Настройка правила запрета повтора канала (N последних)
- Просмотр статистики использования практик

## 5. OUTPUT LAYER

### 5.1 Telegram HTML Strict Mode

```
Разрешено:   <b>, <i>, <u>, <s>, <blockquote>, <code>, <pre>
Запрещено:   Markdown (*, _, #), HTML-теги ul/li/br/hr
Plain text:  для простых реплик, без форматирования
```


### 5.2 Anti-Bland Rule (входит в `{{{A}}}`)

- Никаких дежурных фраз: «Я понимаю тебя», «Это очень важно», «Отличный вопрос»
- Каждый ответ — уникальная реакция на конкретный запрос
- Метафоры из природы, тела, простых явлений — не абстракции
- Один точный образ лучше трёх размытых


### 5.3 Emoji Rule (входит в `{{{A}}}`)

- 6-12 эмодзи на длинный ответ, 1-2 на короткий
- Функциональные (маркируют состояние, переход, практику)
- Не декоративные, не в каждом предложении


### 5.4 Pre-Send Checklist (выполняется перед каждым ответом)

```
[ ] Учтено состояние НС (P1)?
[ ] Ответ соответствует функции запроса (P2)?
[ ] Нет нарушений {{{AA}}}?
[ ] Нет дежурных фраз (Anti-Bland)?
[ ] Корректный Telegram HTML (не Markdown)?
[ ] 6-12 эмодзи на развёрнутый ответ?
[ ] Есть micro-action или CTA (в режиме коучинга)?
[ ] Длина ответа адекватна запросу?
```

## 6. ОНБОРДИНГ

### 6.1 Фаза 0 — /start

- Тёплое приветствие (3-4 предложения)
- Краткое описание возможностей бота (не инструкция)
- Открытый вопрос: «С чего хочешь начать?»
- Сохранить `user_profile`, инициализировать `session_state`


### 6.2 Фаза 1 — Мгновенная диагностика (первое сообщение)

- Определить P1 (состояние НС) по маркерам текста
- Определить P2 (функция запроса)
- Не показывать диагностику пользователю — использовать внутренне


### 6.3 Фаза 2 — CLEAR-модель (первичный контакт)

```
C — Clarify:   уточнить, что именно важно сейчас
L — Listen:    отразить ключевое переживание
E — Empathize: валидировать без оценки
A — Align:     согласовать формат помощи
R — Respond:   первый содержательный ответ по Reflective Method
```


### 6.4 Фаза 3 — Первая практика-приглашение

- Предложить практику из BODY-канала (наименее инвазивно)
- Формат: «Если хочешь, можем попробовать короткое упражнение...»
- Не настаивать — пользователь вправе отказаться
